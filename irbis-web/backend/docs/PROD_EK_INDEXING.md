# Индексация боевого ЭК (EK) в own-индекс поиска

Эта инструкция описывает, как наполнить **собственный индекс поиска**
(`access.catalog.CatalogStore`) записями боевого электронного каталога **EK**
(~258 000 записей) с помощью индексатора `tools/index_catalog.py`, а затем
включить обслуживание поиска по этой базе из own-индекса (флаг `OWN_SEARCH_DBS`),
в обход сломанного индекса `K=` живого ИРБИС (#262, ср. #228/#229).

> **Безопасность.** Индексатор работает с ИРБИС **только на чтение**
> (`max_mfn` + `read_record`) и **ничего не пишет** в живой сервер. Записи EK —
> библиографические (не ПДн). В этом документе **нет секретов** (логины/пароли
> задаются через окружение и сюда не попадают).

---

## 1. Что делает индексатор

1. Подключается к ИРБИС по настройкам окружения (`config.Config`).
2. Обходит MFN базы `--db` от 1 (или от чекпойнта при `--resume`) до `max_mfn`.
3. Каждую запись читает (`read_record`), приводит к формату `CatalogStore`
   (переиспользуется сверенный маппинг `tools.migrate_irbis.map_catalog_record`:
   repeatable-поля → список инстансов, коды подполей → нижний регистр, канон
   `200^a` / `910^b`), и сохраняет через `catalog.save(db, rec, skip_warnings=True)`
   (валидация ФЛК + инверсия индекса).
4. Удалённые / пустые / нечитаемые записи и отклонённые ФЛК — **пропускает**
   (считает в `skipped`), не прерывая прогон.
5. Идемпотентен: исходный MFN штампуется в приватное подполе `907^_mfn`;
   повторный прогон по тем же MFN **перезаписывает** запись в тот же целевой MFN,
   не плодит дубли.
6. Пишет **checkpoint-файл** (последний обработанный MFN) после каждого батча —
   при обрыве прогон возобновляется с `--resume`.
7. Печатает **прогресс** в stderr: процент, скорость (rec/s), ETA. Итоговый
   **JSON-отчёт** идёт в stdout.

---

## 2. Окружение (подключение к ИРБИС и путь к стору)

Индексатор берёт те же переменные, что и обслуживающее приложение
(`config.Config`, читает `.env` рядом с backend). Задайте под боевой сервер:

| Переменная          | Назначение                                            | По умолчанию |
|---------------------|-------------------------------------------------------|--------------|
| `IRBIS_HOST`        | хост ИРБИС                                             | `127.0.0.1`  |
| `IRBIS_PORT`        | порт ИРБИС                                             | `6666`       |
| `IRBIS_USER`        | логин (или `IRBIS_READER_USER`)                       | `MASTER`     |
| `IRBIS_PASS`        | пароль (или `IRBIS_READER_PASS`)                      | пусто        |
| `IRBIS_WORKSTATION` | код АРМ (рабочей станции)                              | `A`          |
| `IRBIS_ENCODING`    | кодировка библиоданных сервера                         | `utf-8`      |
| `IRBIS_TIMEOUT`     | таймаут операции, c                                   | `8`          |
| `CATALOG_DB`        | путь к sqlite-стору CatalogStore (тот же, что у app)  | `backend/catalog.db` |
| `ACCESS_DB`         | путь к AccessStore (словари для ФЛК)                   | `backend/access.db`  |

> **Кодировка (#228) — для боевого EK СПб ГТБ оставляйте UTF-8.** Этот сервер
> **полностью UTF-8** (записи И словарь — проверено сырым двойным декодом), поэтому
> `IRBIS_ENCODING=utf-8` (дефолт). **НЕ ставьте `cp1251` на этот сервер — получите
> искажённую кириллицу** («индекс в CP1251» был диагностической ошибкой: кириллицу
> ломал PowerShell→ssh-конвейер, а не сервер). `IRBIS_ENCODING=cp1251` нужен ТОЛЬКО
> классическому ИРБИС с реальным CP1251-индексом (НЕ наш случай) — меняйте лишь
> после сверки сырого декода с боевым сервером.

> **`CATALOG_DB`.** Индексатор и обслуживающее приложение должны указывать на
> **один и тот же** файл `CATALOG_DB` — иначе `/api/search` не увидит
> наполненный индекс. Для боевого инстанса задайте устойчивый путь
> (напр. `/var/lib/biblio/catalog.db`), а не дефолтный файл рядом с кодом.

Пример (значения — плейсхолдеры, реальные секреты сюда не вписывайте):

```bash
export IRBIS_HOST=<боевой-хост>
export IRBIS_PORT=6666
export IRBIS_USER=<логин>
export IRBIS_PASS=<пароль>
export IRBIS_WORKSTATION=A
export IRBIS_ENCODING=utf-8           # EK СПб ГТБ — UTF-8 (НЕ ставьте cp1251, #228)
export CATALOG_DB=/var/lib/biblio/catalog.db
export ACCESS_DB=/var/lib/biblio/access.db
```

## 3. Запуск

### Сначала — короткий smoke (проверить связь, кодировку, маппинг)

```bash
cd irbis-web/backend
py -3.12 -m tools.index_catalog --db EK --limit 200
```

Проверьте отчёт: `read ≈ indexed` (минус удалённые/пустые), `errors == 0`.
Откройте любую запись в приложении / `CatalogStore` и убедитесь, что кириллица
читаемая (если «кракозябры» — поправьте `IRBIS_ENCODING` и перезапустите).

### Полный прогон EK

```bash
cd irbis-web/backend
py -3.12 -m tools.index_catalog --db EK --batch 1000
```

`--batch 1000` — чекпойнт и строка прогресса раз в 1000 записей. Прогресс идёт в
stderr; итоговый JSON — в stdout (можно перенаправить в файл для отчётности):

```bash
py -3.12 -m tools.index_catalog --db EK --batch 1000 > ek_index_report.json
```

### Оценка времени

При **~15–20 rec/s** (типичная скорость одиночного `read_record` по сети)
полный прогон **~258k** записей занимает **примерно 3,5–5 часов**. Реальную
скорость показывает строка прогресса (`rec/s` + `ETA`) уже на первом батче —
ориентируйтесь по ней. Если скорость заметно ниже, проверьте сеть/нагрузку на
сервер; индексатор не параллелит запросы намеренно (живой сервер общий — щадящий
режим).

> На боевом инстансе запускайте индексацию **в окно низкой нагрузки**: каждая
> запись — это round-trip к общему ИРБИС.

## 4. Обрыв и возобновление (checkpoint / resume)

Чекпойнт пишется после каждого батча в `.index_EK.checkpoint` (рядом с backend;
переопределяется `--checkpoint <путь>`). Хранит последний **успешно обработанный**
MFN.

Если прогон оборвался (разрыв соединения, перезапуск сервера, Ctrl+C) — просто
запустите с `--resume`: индексатор продолжит с `last_mfn + 1`.

```bash
py -3.12 -m tools.index_catalog --db EK --batch 1000 --resume
```

По **полному** прогону (дошли до `max_mfn` без `--limit`) чекпойнт-файл удаляется
автоматически — следующий запуск начнёт с начала (полная переиндексация,
идемпотентная: дубли не появятся).

Полезные флаги:

- `--start-mfn N` — начать строго с MFN `N` (приоритет над `--resume`); удобно
  переиндексировать диапазон.
- `--checkpoint <путь>` — свой путь чекпойнта (напр. на persistent-том).
- `--quiet` — без прогресса в stderr (итоговый JSON всё равно печатается).

> Чекпойнт привязан к коду БД: файл от другой базы игнорируется (старт с нуля),
> повреждённый JSON — тоже (не валит прогон).

## 5. Включение own-поиска по EK (база-за-базой)

Поиск из own-индекса включается флагом окружения `OWN_SEARCH_DBS` (список кодов
БД через запятую, регистронезависимо). По умолчанию пусто — весь поиск идёт через
ИРБИС, поведение прода не меняется.

**Порядок включения (осторожно, по одной базе):**

1. Убедитесь, что EK **полностью проиндексирована** (отчёт полного прогона:
   `indexed` сопоставимо с числом живых записей; `errors == 0`).
2. Сделайте EK **видимой** для OPAC, если ещё не сделана (read-доступ гостя/
   читателя к публичным БД): добавьте `EK` в `IRBIS_PUBLIC_DBS`
   (напр. `IRBIS_PUBLIC_DBS="IBIS,EK"`).
3. Включите own-поиск по EK:

   ```bash
   export OWN_SEARCH_DBS=EK
   ```

4. Перезапустите backend и проверьте: `/api/search?db=EK&expr=K=...` (одиночный
   префиксный запрос) должен вернуть `"source": "own"` и осмысленные результаты.
   Сравните выдачу с ожидаемой по нескольким контрольным терминам.

> Сейчас из own-индекса обслуживаются **одиночные префиксные** запросы
> (`PREFIX=term`, с усечением `$`) и поддержанные компаундные K-выражения; всё
> прочее по-прежнему уходит в ИРБИС. Поэтому включение own-поиска по EK не ломает
> остальные базы.

5. Когда EK проверена — тем же приёмом подключайте следующую базу: дополните
   `OWN_SEARCH_DBS` (`OWN_SEARCH_DBS=EK,PERIO`) **только после** её полной
   индексации (`--db PERIO`). Не включайте флаг для базы, которую ещё не
   проиндексировали, — own-поиск по ней вернёт пусто.

### Откат

Если по EK что-то не так — уберите EK из `OWN_SEARCH_DBS` (или очистите флаг) и
перезапустите backend: поиск по EK снова пойдёт через ИРБИС. Индекс при этом не
трогается; можно спокойно переиндексировать и включить снова.

## 6. Итоговый отчёт (stdout)

```json
{
  "db": "EK",
  "read": 257812,
  "indexed": 257640,
  "skipped": 172,
  "errors": 0,
  "from_mfn": 1,
  "to_mfn": 258000,
  "elapsed_sec": 15240.5,
  "rec_per_sec": 16.9,
  "checkpoint": "/.../.index_EK.checkpoint"
}
```

- `read` — прочитано записей; `indexed` — успешно сохранено в индекс;
  `skipped` — удалённые/пустые/нечитаемые/отклонённые ФЛК; `errors` — сбои
  сохранения (должно быть 0).
- `from_mfn`/`to_mfn` — пройденный диапазон MFN; `rec_per_sec` — фактическая
  скорость прогона (по ней оценивайте время следующего запуска).
