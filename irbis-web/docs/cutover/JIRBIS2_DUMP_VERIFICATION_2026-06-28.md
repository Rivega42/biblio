# Верификация turnkey-миграции на реальном дампе `jirbis2` (2026-06-28)

Узел 4 cutover (J-IRBIS / Joomla `jirbis2` → Biblio own-store) разблокирован:
получен боевой дамп. Этот отчёт фиксирует прогон turnkey
`jirbis_migrate.plan(parse_dump(sql))` на **реальных данных** — только агрегаты,
**без ПДн** (152-ФЗ: дамп и любые производные с ПДн в git не кладём).

## Источник

- Файл: `jirbis2_2026-06-28.sql.gz` (gzip, оригинал `jirbis2.sql` ≈ 97 МБ; ≈ 14 МБ
  сжатый). Распакован в локальный scratchpad **вне репозитория**, обработан
  in-process, не коммитился.
- Схема: стандартный Joomla-префикс `jos_` (все **179** таблиц). Все целевые
  таблицы миграции присутствуют — схема клиента совпала с допущениями модуля

  отличаться — модуль матчит по точным именам, при ином клиенте сверять список
  `CREATE TABLE` заранее).

## Что сделано в этом проходе (код, без записи ПДн / без записи в живой ИРБИС)

- **Подключена интеграционная точка #294** в `core.verify_reader_password`: третья
  ветка `phpass.verify_legacy_password` — перенесённый читатель с phpass-хэшем в
  поле пароля (RDR 130/100) входит своим существующим паролем. Чистая проверка в
  постоянном времени; ветки plaintext/MD5 не дублируются. Тест в
  `tests/test_reader_auth.py` (phpass верный/неверный + сохранена совместимость
  plaintext/MD5).
- **Upgrade-on-login persist** (пере-хэш в pbkdf2 и запись) — НЕ здесь: это
  супервизируемый шаг записи в поле пароля; запись в **живой ИРБИС сейчас не
  делаем** (posture #222).

## Что остаётся для исполнения cutover (записи — отдельным супервизируемым шагом)

1. **Применение плана**: завести 240 учёток в целевом хранилище пароля (поле RDR
   130/100 для читателей) с legacy phpass-хэшем, чтобы upgrade-on-login сработал.
   Сверка `jos_users` ↔ читатель ИРБИС по e-mail/билету. Реальная запись в боевой
   ИРБИС — вне read-only-окружения (#222), под надзором.
2. **Брони/соц/CMS** — применить по соответствующим own-store слоям (34 брони → holds,
   1 рейтинг → соц, CMS-перенос отдельно).
3. Данные формуляра/выдач/заказов читателя — в ИРБИС RDR/RQST (НЕ в Joomla), не
   мигрируются из дампа.

> Дамп `jirbis2.sql` и любые производные с ПДн (e-mail, хэши, формуляр) — **off-git**.
> В репозитории — только агрегаты и трансформирующий код (тестируется на синтетике).
