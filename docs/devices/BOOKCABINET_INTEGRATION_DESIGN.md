# BOOKCABINET_INTEGRATION_DESIGN — робо-шкаф книговыдачи/книгоприёма на Biblio

> **Статус:** проектный черновик (2026-07-01). Составлен по разбору репозитория
> `github.com/Rivega42/bookcabinet` (боевой путь Python aiohttp) + сверке с уже
> построенными нативными доменами Biblio.
> **Опора:** [DEVICES_NATIVE_ARCHITECTURE.md](DEVICES_NATIVE_ARCHITECTURE.md) (каноничная
> архитектура #272), [BIBLIO_DEVICE_INTEGRATION_DESIGN.md](BIBLIO_DEVICE_INTEGRATION_DESIGN.md)
> (ABIS-порт/`IAbis`, два режима), [DOMAIN_DEVICES_SPEC.md](specs/DOMAIN_DEVICES_SPEC.md).
> **Ключевое отличие от прежних доков:** BookCabinet — НЕ железо IDlogic. Это наше
> собственное устройство, поэтому оно становится **первым по-настоящему нативным
> устройством Biblio на чистом протоколе** — без SIP2 и без протокола IDlogic
> (`compat_devices`/`/easybookdll/*` здесь НЕ используются, они — только для
> бесшовного подхвата унаследованных станций IDlogic).

## 0. Резюме и вердикт

**Вопрос:** можно ли переиспользовать читательский портал Biblio как интерфейс шкафа
с функциями книгоприёма/выдачи?

**Вердикт: да, с высокой отдачей — но правильная форма не «портал поверх бэкенда
шкафа», а раскол шкафа на три слоя** с порталом сверху и бэкендом Biblio в роли
«мозга».

Причина: BookCabinet — ровно то «устройство», под которое в Biblio уже спроектирован
нативный домен (`access/devices.py` + `locker.py` + `circulation.py` + `holds.py` +
`readers.py` + `tag_codec.py`). «Мозг» шкафа (личности, ячейки, выдача/возврат,
синк ИРБИС) на ~90 % уже реализован в Biblio и **глубже**, чем в текущей
SQLite-надстройке шкафа (у нас есть реальные политики выдачи, штрафы, блокировка
должников, мультиарендность, own-store-поиск).

**Два зайца (формулировка владельца):**
- шкаф получает современный UI (портал) + настоящий библиотечный бэкенд вместо тонкой
  SQLite+ИРБИС-надстройки;
- Biblio получает физический «плацдарм захода» — самообслуживающее устройство как
  сильный вход с нашей АБИС и обкатку доменов `devices/locker/circulation` на железе.

## 1. Что такое BookCabinet (три слоя)

| Слой | Содержимое | Судьба |
|---|---|---|
| **Железо** | CoreXY-каретка + лоток, 3 RFID-ридера (NFC-билет ACR1281U-C · UHF-ЕКП IQRFID-5102 · UHF-книги RRU9816), 126 ячеек, сервоприводы/шторки/датчики | не трогаем |
| **Руки (механика+драйверы)** | `tools/*` (~40 скриптов, валидированы на железе), `mechanics/algorithms.py` (`take_shelf`/`give_shelf`/`wait_for_user`), `hardware/*`, `rfid/unified_card_reader.py`, калибровка. «Железные правила» (HOME=LEFT+BOTTOM, скорости хоминга, двухэтапная калибровка лотка) | **оставляем на RPi** |
| **Мозг (UI-agnostic)** | aiohttp: авторизация по карте, стейт-машины issue/return/load/unload (`business/*`), SQLite v2 (books/cells/users/operations), синк ИРБИС (`RI=`/`IN=`/поле 40/`910^A`) + офлайн-очередь `irbis/sync_queue.py` | **переносим в Biblio** (уже есть) |
| **UI** | React-киоск (ЧБ Material-«sunlight» тема): приветствие→роль→выдача/возврат→фидбек; библиотекарь load/extract; админ diagnostics/calibration | **заменяем порталом Biblio** (тема — сохранить как токен-набор) |

Контракт «мозга» уже чистый и фронт-agnostic: `POST /api/auth/card`, `POST /api/issue`,
`POST /api/return`, `GET /api/cells`, WebSocket `progress`/`operation_started|completed|failed`.
Именно это делает раскол дешёвым.

**Что «мозгу» нужно и что в Biblio уже есть:**

| Нужно шкафу | В Biblio |
|---|---|
| База пользователей | `readers.py`/`reader_registry.py` + ИРБИС RDR + own-store |
| Записи про ячейки | `locker.py` (ячейки, `busy/free/allocate`, мастер-ключ, битовая маска) + экран «Ячеистое хранение» (постамат/книгоприём) |
| Выдача/возврат/продление | `circulation.py` + `/api/circ/*` (политики, штрафы, блокировка должников) |
| Брони | `holds.py` + `/api/holds` |
| Реестр устройства + события | `devices.py` (`device_event` с `loan_ref`/`hold_ref`, health, KIND_STATION) |
| RFID-метка ↔ экземпляр | `tag_codec.py` (ISO 28560-2) |
| Синк каталога/читателей | ИРБИС-адаптер + own-store |

## 2. Целевая архитектура

```
                 ┌──────────────────────────────────────────────┐
   Читатель /    │  UI · Портал Biblio в режиме «станция/киоск»  │
   библиотекарь →│  (те же компоненты портала + ЧБ-тема-токены)  │
                 │  первый экран = портал (поиск/ЛК), выдача/     │
                 │  возврат — действия поверх                     │
                 └───────────────┬───────────────┬───────────────┘
             намерения+личность  │               │  физика+прогресс
             (HTTPS, портал↔ядро)│               │  (BDP: REST+WS, локально)
                                 ▼               ▼
        ┌──────────── МОЗГ: Biblio ────────┐  ┌──── РУКИ: device-agent (RPi) ────┐
        │ circulation/holds/readers/catalog │  │ CoreXY-движение, take/give_shelf, │
        │ devices/locker · тарифы · аудит   │  │ wait_for_user (детект RRU),       │
        │ ИРБИС-адаптер + own-store         │  │ шторки/замки/калибровка,          │
        │ политики выдачи/штрафы/должники   │  │ СВОИ ридеры/драйверы + экран,     │
        └───────────────┬───────────────────┘  │ локальный журнал ячеек + очередь  │
                        ▼                       └──────────────┬────────────────────┘
                 ИРБИС :6666 / own-store                       ▼
                                                       Железо (моторы, ридеры,
                                                       126 ячеек, шторки, датчики)
```

**Шов:** портал шлёт **намерения** («выдать экземпляр X читателю Y», «принять книгу»)
в Biblio; Biblio решает право/политику/бронь и синкает ИРБИС; device-agent исполняет
**физику** и отдаёт прогресс. Вся безопасность движения остаётся на RPi — Biblio
никогда не управляет моторами напрямую.

## 3. Граница ответственности (кто чем владеет)

### 3.1 Остаётся на RPi (device-agent, «руки»)
- **Драйверы и логика ридеров** (владелец — шкаф): NFC-билет (ACR1281U-C / PC/SC),
  UHF-ЕКП (IQRFID-5102), UHF-книги (RRU9816). Агент сам опрашивает ридеры, дедуплицирует
  метки, отдаёт «сырое» событие «приложена карта UID=…» / «в окне метка EPC=…».
- **Механика и безопасность:** homing, `take_shelf`/`give_shelf`, `wait_for_user`
  (книгу забрали — метка ушла из окна), шторки/замки, таймауты, аварийный стоп,
  калибровка. Ни одна из «железных правил» не переезжает.
- **Экран:** агент рендерит UI (портал-киоск) на локальном chromium-kiosk; он же —
  диспатчер команд «на руки».
- **Локальный журнал физики:** что в какой ячейке (источник правды о физическом мире —
  это принцип самого шкафа), очередь незасинканных операций.

### 3.2 Уходит в Biblio («мозг»)
- Личности читателей/персонала, права/гранты, категории.
- Каталог/холдинги, статусы экземпляров, EPC↔экземпляр.
- Циркуляция: политики срока, штрафы, блокировка должников, продление.
- Брони, формуляр, история, уведомления.
- Синк ИРБИС + own-store, аудит, мультиарендность/тарифы.

### 3.3 UI
- Портал Biblio в новом **режиме станции/киоск**: те же компоненты (поиск, ЛК,
  статус-чипы «В наличии/На руках/В постамате/Книгоприём» уже есть), плюс
  touch-раскладка и 3 флоу + режим библиотекаря.

## 4. Свой протокол Biblio ↔ device-agent (BDP)

> **Решение владельца (2026-07-01): не SIP2 и не протокол IDlogic.** SIP2 —
> легаси-хвост (тянет совместимость со станциями IDlogic); протокол IDlogic
> (`/easybookdll/*`, `compat_devices`) — только для бесшовного подхвата чужих станций,
> не для нашего устройства. Здесь проектируем **чистый нативный протокол** — назовём
> **BDP (Biblio Device Protocol)**.

### 4.1 Принципы
- **Намерение, а не RPC железа.** Biblio выражает бизнес-намерение; агент сам решает,
  как его исполнить механически. Обратный поток — доменные события + телеметрия.
- **Симметрия ролей.** Ядро Biblio ↔ агент — два равноправных узла: агент
  инициирует «читатель приложил карту», Biblio инициирует «выдать экземпляр».
- **Идемпотентность.** У каждой операции `op_id` (UUID агента); повтор с тем же
  `op_id` не создаёт вторую выдачу (критично для офлайн-replay).
- **Версионирование** (`bdp_version`) и **capability-дескриптор** (что это за
  устройство и что умеет) — Biblio не зашивает знание о конкретной модели шкафа.
- **Транспорт:** REST (команды/запросы) + WebSocket (события/прогресс/телеметрия),
  локально по TLS; аутентификация — **device-token** (выдаётся при регистрации
  устройства в Biblio, не Basic-legacy).
- **Термины — наши** (билет, экземпляр, ячейка, выдача, приём), не enum'ы IDlogic.

### 4.2 Регистрация и capability
`POST /api/devices/register` (Biblio) → device-token. Дескриптор:
```json
{
  "kind": "self_service_cabinet",
  "vendor": "bookcabinet", "model": "rpi3-corexy-126",
  "capabilities": ["issue", "return", "librarian_load", "librarian_extract",
                   "inventory", "cell_map", "progress_stream"],
  "cells": {"count": 126, "rows": ["FRONT","BACK"], "cols": 3, "shelves": 21},
  "readers": [
    {"role": "patron_card", "tech": "nfc",  "model": "ACR1281U-C"},
    {"role": "patron_ekp",  "tech": "uhf",  "model": "IQRFID-5102"},
    {"role": "item_tag",    "tech": "uhf",  "model": "RRU9816"}
  ]
}
```
`kind: self_service_cabinet` — **новый подвид device** (робо-станция выдачи/приёма),
рядом с существующими `station`/`safekeeper`/`smartshelf`.

### 4.3 Словарь операций (черновой)

**Агент → Biblio (что происходит физически / кто пришёл):**
| Операция | Тело | Смысл |
|---|---|---|
| `card.presented` | `{reader_role, uid}` | приложена карта — резолвить читателя |
| `item.detected` | `{epc}` | в окне метка книги (старт возврата) |
| `op.progress` | `{op_id, step, total, phase, msg}` | прогресс механической операции (WS) |
| `op.result` | `{op_id, status, error?}` | итог физической части |
| `device.health` | `{state, sensors, readers, queue_len}` | heartbeat + телеметрия |
| `sync.replay` | `{ops:[…offline ops…]}` | пакетная досдача офлайн-операций |

**Biblio → агент (намерения/команды):**
| Операция | Тело | Смысл |
|---|---|---|
| `session.begin` | `{reader, entitlements, screen}` | кого обслуживаем, что показать |
| `issue.execute` | `{op_id, item_epc, cell?, due_hint?}` | достать полку с книгой к окну и выдать |
| `return.accept` | `{op_id}` | принять книгу в любую свободную ячейку |
| `librarian.load` | `{op_id, item, cell?}` | загрузка экземпляра в ячейку |
| `librarian.extract` | `{op_id, cell}` | изъятие из ячейки |
| `inventory.run` | `{op_id, scope}` | обход ячеек + опрос меток |
| `cell.map.get` | `—` | текущее состояние ячеек |

**Разделение прогресса:** агент отдаёт свою «механическую шкалу» (шаги 1..N), а
портал маппит её в 3–6 понятных читателю фаз (готовим → везём → заберите).
Соответствие «механика→фазы UI» описывает агент в capability, чтобы Biblio/портал не
зашивали числа (в текущем шкафу это захардкожено 14→6 в `IssueProcess.tsx`).

### 4.4 Поток «выдача» (пример)
1. `card.presented{patron_card, uid}` → Biblio резолвит читателя, шлёт `session.begin`
   (портал показывает формуляр/выдаваемое/брони).
2. Читатель на портале выбирает экземпляр → портал → Biblio `POST /api/circ/issue`
   (проверка политики/долгов/брони, решение `Decision.ok/reasons`).
3. Если ok — Biblio → агент `issue.execute{op_id, item_epc}`; агент гонит механику,
   стримит `op.progress` (портал рисует фазы), затем `op.result{ok}`.
4. Biblio фиксирует выдачу (loan) + синкает ИРБИС (поле 40 / `910^A=1`); `device_event`
   с `loan_ref`.

Возврат — зеркально: `item.detected{epc}` → `POST /api/circ/return` → `return.accept`
→ книга в свободную ячейку → `awaiting_extraction` → позже библиотекарь `extract`.

## 5. Идентификация и единая личность

- **Карта → читатель:** UID (NFC-билет / UHF-ЕКП) резолвится в Biblio (RDR/own-store);
  привязки карт — как поля RDR 30/28/24 (main/ekp/extra). Разные ридеры — разные
  `reader_role`, но один читатель.
- **Книга → экземпляр:** метка UHF EPC сверяется по строке EPC ↔ `910^H`/`941^H`
  (тот же индекс `IN=`, что уже использует шкаф). См. §9 о крёстике HF/UHF.
- **Единая личность (требование владельца):** ЛК читателя на шкафу = ЛК портала.
  Тот же аккаунт, та же история/брони/формуляр. Шкаф — это терминал портала, который
  вдобавок умеет физическую выдачу/приём.
- **Портал — первый экран:** в режиме станции портал показывает обычный поиск/ЛК;
  выдача/возврат — действия поверх (кнопки/флоу), а не отдельное «киоск-приложение».
  Idle → приветствие «приложите карту»; после карты — персональный экран.

## 6. Офлайн-стратегия (когда шкаф без связи с Biblio)

> **Решение (уточнено владельцем 2026-07-01): НЕ «глухой режим».** В офлайне шкаф
> обязан (1) **идентифицировать читателя** и (2) **знать его формуляр** (какие книги,
> какие метки/EPC), а приём/выдачу **валидировать локально**. Возврат — НЕ «принимаем
> всегда»: иначе насуют книг не из этой библиотеки или вообще не книги, поэтому шкаф
> должен знать, КТО (идентификация) и ЧТО (метка книги) можно вернуть. Единственная
> функция, принципиально НЕ работающая офлайн, — **бронирование** (это обращение в
> Biblio или через Biblio в ИРБИС).

### 6.1 Что кэшируем на RPi
- **Идентификация:** реестр читателей — билет, UID карт (NFC-билет/ЕКП), ФИО, категория,
  флаг блокировки/долга, права. Обновляется дельтами (`updated-since` курсор).
- **Формуляр:** активные выдачи по каждому читателю — экземпляр + EPC + срок + заёмщик.
  Нужен, чтобы (а) показать формуляр офлайн, (б) валидировать возврат по метке.
- **Множество меток фонда:** ответ на «этот EPC — наш экземпляр?» — для отсева чужих
  меток и не-книг.

### 6.2 Сколько это данных и влезет ли в RPi 3 — ДА
Оценка по порядку величины:
| Данные | Размер записи | 50 тыс. | 500 тыс. / 1 млн |
|---|---|---|---|
| Читатели | ~300 Б | ~15 МБ | ~150 МБ |
| Активные выдачи (одновременных) | ~100 Б | ~10 МБ (100 тыс.) | — |
| Все метки фонда — точный индекс EPC | ~20 Б | — | ~20 МБ (1 млн) |
| Все метки фонда — **bloom-фильтр** (FP 0.1%) | ~1.8 бита | — | **~1.8 МБ (1 млн)** в RAM |

microSD RPi (десятки ГБ) держит полную копию с запасом; в RAM (1 ГБ) — горячий срез +
bloom-фильтр для мгновенного отсева «не наш EPC». Хранилище — локальный SQLite
(own-store), быстрый membership-тест — bloom. **Вывод: помещается уверенно.**

### 6.3 Две формы
- **Основная — edge-node Biblio у шкафа:** own-store держит полную копию
  читателей/выдач/меток; «офлайн» = edge без uplink к центру/ИРБИС, но шкаф работает
  против edge, тот синкает вверх дельтами. Переиспользуем готовую own-store/офлайн-
  машинерию Biblio (оси развёртывания on-prem).
- **Минимальный фолбэк — agent-local cache:** SQLite (читатели + активные выдачи +
  индекс EPC фонда) + bloom-фильтр; подписанный офлайн-журнал → `sync.replay`
  (идемпотентно по `op_id`).

### 6.4 Политика решений офлайн (валидированная, НЕ «принимаем всегда»)
- **Возврат:** принимаем ТОЛЬКО если EPC ∈ множестве меток фонда:
  - EPC = активная выдача → штатный возврат (заёмщик из кэша) → ячейка → `awaiting_extraction`;
  - EPC принадлежит фонду, но не на выдаче → принимаем + помечаем «незанятая книга» для персонала;
  - EPC не в фонде / метки нет / не книга → **отказ** («книга не распознана / не из нашей библиотеки»).
- **Выдача:** известный читатель (в кэше, не заблокирован/не должник) + экземпляр
  физически в шкафу → выдаём, ставим в очередь синка; иначе «связь восстанавливается».
- **Бронирование:** офлайн НЕДОСТУПНО — кнопку прячем/дизейблим, показываем «доступно при связи».
- **Идемпотентность replay** по `op_id`; на экране — статус синка («N операций ждут»),
  алерт при переполнении очереди.

## 7. UI: портал в режиме «станция/киоск»

- **Переиспользуем компоненты портала** (`reader/*`: поиск, ЛК, формуляр `HoldsTab`,
  история, статус-чипы) — новый full-screen touch-скин, не форк.
- **ЧБ-тема шкафа как «упрощённый» токен-набор** (требование владельца — цвета
  сохранить): чистый чёрный/белый, границы `2px solid` вместо теней, тач-цели ≥48px,
  `--radius` 0.75rem, Inter, статусы зелёный/красный/синий. Ложится **отдельной темой
  Biblio** «kiosk/high-contrast» поверх нашей системы тем/арендатор-брендинга — как
  ещё один набор `--*`-токенов, без форка портала. **Тема переключается из админки**
  (per-устройство/арендатор): «kiosk/high-contrast» — одна из выбираемых тем, а не
  жёстко зашитая; админ может назначить любую доступную тему конкретному экрану.
- **Экранное медиа из админки (digital signage), см. issue-идею:** админ управляет
  плейлистом медиа устройства; **нет медиа → показывается портал**; поверх — большие
  кнопки «Взять»/«Вернуть», тап → запрос авторизации → флоу выдачи/возврата.
- **3 флоу читателя** (приложи карту → выдача/возврат → результат) + **режим
  библиотекаря** (загрузка/изъятие/инвентаризация) как защищённые грантами экраны
  (surfaces composed by grants — см. [[irbis-product-terminology]]).
- **Прогресс** — из `op.progress` агента, фазы из capability-маппинга.

## 9. Швы и риски (честно)

1. **RFID разного класса.** Книги шкафа — UHF EPC Gen2 (RRU9816), сверка `book.rfid==EPC`
   строкой; наш `tag_codec` — HF ISO 28560-2 (ICODE SLIX). Крест — по строке EPC ↔
   `910^H/941^H` (индекс `IN=`), НЕ перекодированием метки. Слой маппинга, не стенка.
2. **Механика ≠ постамат.** `locker.py` моделирует *pickup* SafeKeeper (библиотекарь
   укомплектовал → читатель забрал). Шкаф — *робо-извлечение* самообслуживания.
   Модель ячеек совпадает, но флоу робота — новый device-flow поверх; `locker.py`
   дословно не переиспользуем (берём модель ячеек/`awaiting_extraction`).
3. **Безопасность движения — 100 % на RPi.** Biblio шлёт намерения; физика/таймауты/
   датчики/аварийный стоп — только агент. «Железные правила» из `CLAUDE.md` шкафа
   неприкосновенны.
4. **Полный цикл захвата полки на железе ещё не прогонялся** (в шкафу подтверждены
   только хоминг+калибровка лотка; бизнес-путь `move_tray` vs полевой
   `shelf_operations` расходятся). Это риск шкафа, не интеграции, но сухой цикл нужен
   до продакшена.
5. **Сеть/офлайн** — §6; спроектировать явно, не полагаться на «всегда online».
6. **Секрет-гигиена.** В `bookcabinet/docs/IRBIS_INTEGRATION.md` боевые креды ИРБИС
   открытым текстом (IP/логин/пароль). Рекомендация владельцу: убрать в env/секрет,
   вычистить историю. (Его репо — не наш; здесь только отмечаем.)

## 10.1 Ревью 2026-07-02 — блокеры и решения (панель HW · SW · продукт)

Проведено дотошное ревью тремя экспертами (аппаратка/RFID/safety/edge · софт/протокол/консистентность · продукт/лицензия/RBAC/медиа/метрики). Конвергенция сильная — несколько блокеров названы независимо.

**🔴 Блокеры до BDP-спайка (#2):**
1. **EPC→экземпляр резолвер отсутствует.** Циркуляция ключуется на `910^b` (инв. №), шкаф знает только UHF-EPC; «крест по `910^H`» в коде НЕТ (`catalog.find_exemplar` — только `910^b`). Нужен `find_exemplar_by_tag(db, epc)` (индекс `IN=` по 910^H/941^H); во всех device-flow оперировать EPC. Решить канонический ключ (инв. №) и совместимость фонда (HF ISO 28560 ≠ UHF EPC — какой фонд физически едет в шкаф).
2. **op_id-идемпотентность не реализована** ни в шкафу (`sync_queue` реплеит без ключа), ни в Biblio (`checkout` всегда INSERT) → двойная выдача при replay. Нужен `op_id UNIQUE` в loan + `checkout(op_id=)` с INSERT-OR-IGNORE.
3. **Offline-first «успех всегда» ⊥ Biblio-gate** → нужна двухфазная сага `reserve→execute→op.result→commit` + компенсация (rollback loan+`910^A`) при DENY/сбое механики; явная точка переключения online-gate / offline-local-gate.
4. **Самообслуживание невозможно:** `/api/circ/*` жёстко staff-only; хардкод карты `ADMIN99`=admin в `auth.py` шкафа перед всякой проверкой → device-actor/kiosk-делегирование (устройство действует от имени читателя) + удалить хардкод.
5. **Beachhead-лицензия невыразима + риск «окирпичить шкаф»:** нет оси канала (web/kiosk/device), `devices` фейл-опенится (нет в `entitlements.DEFAULT_MODULES`; section-гейт матрицы включён только для оцифровки). Нужны: lifeline-ядро (identify/return/extract/open-cell НИКОГДА не гейтится лицензией), ось канала, мультиаренда (`tenant` в device-таблицы + реальный device-token с ротацией).

**🟡 До прод / в ходе спайка:** книжный ридер в `main.py` поднят на ЧУЖОМ протоколе (IQRFID-драйвер на порту RRU9816; правильный `rru9816_driver.py` не подключён); задний ряд 63/126 ячеек не прогнан (capability объявляет реально доступные ячейки); нет персистентного «операция-в-полёте» (потеря питания → разъезд БД↔физика, `StartupRecovery` хомит вслепую); приём только по метке (нужен датчик присутствия/веса — муляж примут); edge-node НЕ на RPi 3 (отдельный мини-ПК); `sync_queue` растёт без предела и молча `failed` после 10 попыток; ПДн на edge без шифра/минимизации/TTL/remote-wipe + плейнтекст боевые креды ИРБИС и ФИО читателей в репо шкафа и git-истории → вычистить + сменить пароль; функций шкафа (тема/медиа-флот/статус/ячейки/self-service) нет в матрице/грантах/админ-UI; медиа-флота модели нет (`station_banner` — один BLOB без device/tenant) → `media_asset/playlist/schedule/device_playlist` per-tenant в DAM; метрики (cells/queue/per-reader) в `device_health` отсутствуют; deployment-ось не знает пресет `beachhead` (+ `entitlements.MODE_PRESETS` и `deployment._DEFAULT_MODULES` расходятся — нужен один SSOT); канонический `resolve_patron(role, uid)` с коллизиями UID (усечение UHF до 24 символов) и guest/unknown; запись поля 40 RDR только через Biblio с оптимистичной блокировкой (иначе last-writer-wins); кодировка ИРБИС cp1251 не подтверждена (риск порчи боевой RDR).

**Концептуальное решение (зафиксировать):** «ячейка» шкафа = pickup-локер (holds) ИЛИ полка фонда (catalog/inventory)? Дизайн смешивает; робо-извлечение по требованию ближе к smartshelf-инвентаризации, чем к holds-pickup (`locker.py` — модель STAFFED-заказа, не «книга просто лежит как фонд»).

**Правки, внесённые в модель (схема v2):** админка — часть Biblio (не отдельный блок); edge-node — отдельное железо у шкафа, RPi = только руки; шов BDP — двухфазная сага; lifeline-ядро вне лицензии.
