# DEVICE_CONNECTION_MAP — карта подключений и потоков запросов

> **Вопрос:** кто к кому и как подключается; кто кому какие запросы шлёт — для всего парка устройств (противокражные ворота, постаматы/станции самообслуживания, шкафы бронирования SafeKeeper, умные полки, СКУД, камеры/FaceDetect).
> **Источник:** статический разбор `C:\IRBIS64\Library Admin Server\` — дампы `_recon/_scratch_devices/str_*.txt`, конфиги, `manual.txt`, логи `Log\AppLog.txt`/`sdk_log\sdk_log.log`.
> **Достоверность:** каждое соединение помечено `[✓]` подтверждено строками/конфигом/логом, `[~]` инференс по архитектуре (протокол не в этих бинарниках — в прошивке станций/сервере).
> **Связанные:** [DEVICE_INVENTORY.md](DEVICE_INVENTORY.md) · [DEVICE_CONTROL_MAP.md](DEVICE_CONTROL_MAP.md) · [DEVICE_IRBIS_INTEGRATION.md](DEVICE_IRBIS_INTEGRATION.md) · [OPEN_QUESTIONS.md](OPEN_QUESTIONS.md)
> **⚠ Секреты:** реальные IP/креды — в конфигах на стенде; здесь хосты обозначены как `‹...›`, порты сохранены.

## 0. Легенда транспортов и портов

| Обозначение | Транспорт | Порт | Кодировка/протокол |
|---|---|---|---|
| **IRBIS-TCP** | TCP, нативный протокол ИРБИС64 | **6666** | ManagedClient.4 (`arm=B/C`), UTF-8 |
| **SIP2** | TCP, текстовый SIP2 | (настр.) | self-check клиент (альтернатива IRBIS-TCP) |
| **REST** | HTTP, JSON | **8005** | `api.svc`, `Authorization: Basic`, Newtonsoft.Json |
| **EBWEB** | HTTP, JSON | **8005** | `/easybookdll/*` (лицензия/health/кэш книг) |
| **Dahua** | TCP, Dahua NetSDK | **37777** | бинарный SDK (логин/видео/FaceDetect) |
| **COM** | RS-232/RS-485 через USB↔UART (CP210x) | — | 115200 8N1 / Andea 19200 8E1 |
| **USB/PCSC** | USB-HID / PC/SC (WinSCard) | — | смарт-карты, ЕКП |
| **SMB** | файловая шара | 445 | `\\SRV-RFID\...\LAS_Update` (обновление) |
| **SOCK** | локальный TCP-сокет | (настр.) | `TagService.SocketServer` (интеграция на ПК) |

**Три «звезды» подключений:**
1. **ИРБИС64 (`:6666`)** — источник истины по читателям/книговыдаче/заказам. К нему ходят LAS (модуль «Библиотекарь», заказы, регистрация ЕКП) и TagService.
2. **Центральный сервис IDlogic (`api.svc :8005`)** — хаб состояния всего парка устройств. LAS читает/пишет метаданные; устройства (станции/SafeKeeper/полки/СКУД/ворота/камеры) отчитываются ему `[~]`.
3. **Локальные устройства** — считыватели/ворота/камеры подключены к ближайшему ПК по COM/USB/PCSC/Dahua.

> **Главный нюанс достоверности:** все легенды `[~]` (устройство → сервер) — это **архитектурный вывод**: у LAS есть операции **чтения** их состояния с сервера, значит устройства это состояние серверу **отдают**. Но сам протокол «устройство → сервер» в данных бинарниках отсутствует — он в прошивке станций/контроллеров (см. [OPEN_QUESTIONS.md](OPEN_QUESTIONS.md) §1–3). Подтверждено «по проводу» только то, что отмечено `[✓]`.

## 3. Кто кому какие запросы шлёт (потоки)

### 3.1 Книговыдача в модуле «Библиотекарь» (LAS ↔ ИРБИС)
```
Читатель кладёт карту/книги на настольный считыватель
   считыватель ──код карты/меток──▶ LAS (EasyBook ядро)
LAS ──IRBIS-TCP:6666── irbis_server:
   1) Search   "V30:'<билет>' and V910^H <> ''"   → найти читателя (RDR)
   2) ReadRecord(RDR mfn)                          → ФИО, категория, долги (поля 10/11/50/40)
   3) Search   книгу по 910^A/^B (инв/штрихкод)    → доступность (910^H)
   4) WriteRecord(RDR)  + WriteRecord(KNIGA)        → поле 40 (выдано), 910^H, строка 691
   (опц.) LAS ──Dahua:37777── DetectFace            → идентификация читателя по лицу
```

### 3.2 Заказ SafeKeeper (бронирование) — LAS ↔ сервер ↔ ИРБИS ↔ камера
```
LAS ──REST:8005── api.svc:  OrderModify (создать) → OrderBodyModify (наполнить книгами)
LAS ──IRBIS-TCP:6666── irbis_server: CreateRequest (RQST) / пометить экземпляр OnOrder
[ Подготовлен ] → сотрудник комплектует на КАМЕРЕ мастер-ключом:
   мастер-ключ ──RFID локально──▶ контроллер SafeKeeper → ОТКРЫТЬ ячейку  [~ прошивка]
   контроллер ──[~]──▶ api.svc:  статус ячейки «занято», заказ «Укомплектован»
[ Выдан ] читатель прикладывает карту на камере → ячейка открыта →
   контроллер ──[~]──▶ api.svc → LAS ──IRBIS-TCP── Checkout (списать на читателя)
   (если ABIS-запись не прошла → статус «Выдан с ошибками», повтор из LAS)
```

### 3.3 Противокражные ворота (EAS-тревога)
```
Книга с активной EAS-меткой проходит через ворота
   считыватель ворот ──COM── RFID_ANTIVOR_ActiveX.dll:
      цикл: ReaderDataGet (инвентаризация) + RfidEASGet (бит EAS)
      EAS активен → EAS_Alarm() = IsAlarm=true → сирена/строб (реле SetOutput)
ворота/счётчик ──[~]──▶ api.svc:  GateEASEvent (дата, книга, RFID-код), счётчик вход/выход
LAS ──REST:8005── api.svc:  GateEASEventsGet / GateCountersGet  → показать статистику
```

### 3.4 SendChar (TagService без интеграции)
```
метка ──считыватель──▶ TagService: декод → строка кода
TagService:  ── печать в АКТИВНОЕ окно (эмуляция клавиатуры) + Enter   (АРМ Каталогизатор/OPAC)
        либо ──IRBIS-TCP:6666── подать код напрямую в ИРБИС (config.ini [IRBIS])
локальное ПО ──SOCK──▶ TagService.SocketServer (запросить чтение по сокету)
```

### 3.5 Мониторинг парка (Dashboard LAS → сервер)
```
LAS ──REST:8005── api.svc, периодически:
   StationsGet · SafeKeepersGet · SmartShelvesGet · ACSGet · GatesGet · VisitorsGet
   (+ info_*Get / info_*AdvancedGet для сводки и деталей)
   ← сервер отдаёт текущее состояние/здоровье узлов всех устройств
```

### 3.6 Лицензия и обновление
```
TagService/EasyBook ──EBWEB:8005── api.svc:
   /easybookdll/DeviceIsLicenseValid  (DeviceID из easybook_rfid.cfg3)
   /easybookdll/DeviceDataAdd  {deviceID, softOnlineCount, deviceOkCount, deviceErrorCount}
   /easybookdll/BooksCacheAddUpdate  (кэш прочитанных кодов)
LAS ──REST:8005── api.svc: upd_IsUpdateAvailable → upd_Update
LAS ──SMB:445── \\SRV-RFID\...\LAS_Update\LAS_Update.zip  (распаковка + сверка versions.inf)
```

## 4. Сводка по классам устройств: где «мозг»

| Класс | К чему физически подключён | Кто им «командует» | Управление «по проводу» в этих файлах? |
|---|---|---|---|
| Настольный считыватель | ПК сотрудника (COM/USB/TCP/PCSC) | EasyBook TagService / LAS | **Да** ([✓], DEVICE_CONTROL_MAP §2,§4) |
| Противокражные ворота | ПК ворот (COM) | RFID_ANTIVOR_ActiveX → EasyBook | **Да** ([✓], §5) |
| Камеры Dahua / FaceID | сеть (NVR :37777) | LAS напрямую (NetSDK) | **Да** ([✓], §6) |
| Постаматы / станции | свой контроллер киоска | прошивка киоска + api.svc | **Нет** — только метаданные [~] |
| SafeKeeper (ячейки) | свой контроллер + RFID мастер-ключ | прошивка камеры + api.svc | **Нет** — открытие ячеек в прошивке [~] |
| Умные полки | свой RFID-контроллер | прошивка полки + api.svc | **Нет** — только чтение состояния [~] |
| СКУД / турникеты | стац. считыватели/антенны | контроллер СКУД + api.svc | **Нет** — только метаданные [~] |
| Счётчик посетителей | встроен в ворота / видео | прошивка/Dahua + api.svc | **Нет** — только агрегаты [~] |

**Вывод по двум вопросам пользователя:**
1. **Кто к кому подключается** — три центра: ИРБИС64 `:6666` (читатели/выдача), сервис IDlogic `api.svc :8005` (парк устройств), и локальные шины устройств (COM/USB/PCSC/Dahua). Полная карта — §1, таблица — §2.
2. **Кто кому что шлёт** — LAS и TagService инициируют запросы к ИРБИС и к `api.svc`; устройства-станции отдают своё состояние центральному сервису (этот лег — `[~]`, протокол в прошивке). Сценарии запросов — §3.
