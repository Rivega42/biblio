# CONTROL_CAPABILITY — сможем ли мы управлять устройствами на основе полученных данных

> Прямой ответ на вопрос: «на основе полученных данных сможем ли управлять устройствами?» — по классам, с разделением **управление (актуация)** vs **учёт/мониторинг**.
> Опора: весь рекон + декомпиляция .NET (ilspycmd) и нативного ядра (Ghidra). См. [README.md](README.md).

## Короткий ответ

**Да — для всего, что связано с RFID и камерами (это и есть бесшовная замена JIRBIS/АРМ).** **Нет — для станционной механики** (открытие ячеек постаматов, конвейер/принтер киосков, контроллер умных полок): этот код в прошивке станций, которой в полученном ПО **нет вообще** — никакой объём реверса данного пакета её не добавит, нужен доступ к станции/её SDK.

## Матрица управляемости

| Класс устройства | Управление? | Чем располагаем (подтверждено) | Чего не хватает |
|---|---|---|---|
| **Настольный RFID-считыватель** | ✅ **полное** | нативный API ядра (`RfidReadDataByCfg/RfidDM15Read/Write/RfidEASGet/Set/ReaderPower`), кодек **ISO 28560-2** (декомпилирован), команды по вендорам (FEIG/ChaFon/NUR/Andea), сокет `:6001`. Сам `EasyBook_RFid.dll` у нас есть. | — (можно переиспользовать DLL либо реализовать SDK) |
| **Противокражные ворота** | ✅ **полное** | COM-интерфейс `RFID_ANTIVOR_ActiveX` (`ReaderOpen/ReaderDataGet/EAS_Set/Reset/Alarm`) → ядро; детект меток, EAS, тревога; релейный выход сирены (`FEISC_0x72_SetOutput`/`RDR_SetOutput`) | — |
| **Камеры Dahua / FaceID / счётчик** | ✅ **полное** | Dahua NetSDK (DLL у нас есть) + паттерн вызовов (`Login/DetectFace/RealPlay/Snapshot`), `:37777` | — (нужно Dahua-железо) |
| **Android ТСД (хранение фонда)** | ✅ **через SDK терминала** | модель данных + REST `BookKeeping*` + UHF-стек (Chainway R3 в наличии) | UHF-SDK конкретного терминала (даёт вендор; не блокер реверса) |
| **СКУД / турникеты** | ⚠️ **частично** | стационарные RFID-считыватели/антенны — драйвим тем же стеком; события прохода | актуация турникета (открыть/закрыть) — контроллер СКУД |
| **Станции самообслуживания** | ⚠️ **частично** | RFID + вся **логика книговыдачи** (наша) | механика: чековый принтер, конвейер/выдача — прошивка станции |
| **Постаматы / SafeKeeper** | ❌ **только учёт** | заказы/ячейки/состояния/мастер-ключ (метаданные через сервис) | **физическое открытие ячейки** — прошивка контроллера локера ([OPEN_QUESTIONS.md](OPEN_QUESTIONS.md) §1) |
| **Умные полки** | ❌ **только данные** | формат инвентаризации (DTO) | wire-протокол контроллера полки — прошивка |

## Почему именно так

- **RFID-уровень полностью реверснут и у нас есть бинарь ядра** → читать/писать метки (Danish/ISO 28560-2), ставить/снимать EAS, писать AFI, регулировать мощность — можем **прямо сейчас**. Это покрывает: книговыдачу/возврат на настольном считывателе, противокражку, инвентаризацию ТСД, идентификацию по карте/ЕКП.
- **Камеры** — стандартный Dahua NetSDK, есть и SDK, и паттерн → FaceID и счётчик посетителей воспроизводимы.
- **Станционная механика** (ячейки, конвейер, полки, турникеты) живёт в **контроллере станции**, а не в управляющем ПО. В полученном пакете (LAS + EasyBook + драйверы) этого кода нет — LAS лишь обменивается с центральным сервисом метаданными. Поэтому «открыть ячейку №N» мы **не можем** без протокола станционного контроллера (реле/RS485/соленоид) — это не пробел рекона, а отсутствующий артефакт.

## Что это значит для проекта

- **Фаза 1–2 Biblio реализуема полностью** на полученных данных: книговыдача/возврат/продление/регистрация (настольный считыватель), противокражные ворота, FaceID/камеры, мобильная инвентаризация (ТСД). Бесшовный подхват этих устройств — конфигом (см. [TZ_PHASE1_abis_device_core.md](TZ_PHASE1_abis_device_core.md) §6).
- **Фаза 3 (станции/постаматы/полки)** требует **доступа к станции или её SDK/прошивке** — это единственное, что нельзя получить из текущего ПО. Варианты: запросить протокол у поставщика станций (IDlogic/производитель локеров), снять сетевой/последовательный трейс с работающей станции, или получить станционный build.

> Итог: **управлять — да, всем RFID-парком и камерами; станционной механикой — только после получения протокола станций.** Программных блокеров в полученных данных больше не осталось; остаточный блок — исключительно аппаратно-прошивочный.
