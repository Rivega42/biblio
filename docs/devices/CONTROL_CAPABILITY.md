# CONTROL_CAPABILITY — сможем ли мы управлять устройствами на основе полученных данных

> Прямой ответ на вопрос: «на основе полученных данных сможем ли управлять устройствами?» — по классам, с разделением **управление (актуация)** vs **учёт/мониторинг**.
> Опора: весь рекон + декомпиляция .NET (ilspycmd) и нативного ядра (Ghidra). См. [README.md](README.md).

## Короткий ответ

- **RFID-парк и камеры** (настольный считыватель, ворота, FaceID/Dahua, ТСД) — **управляем напрямую сейчас**: у нас есть и протоколы, и сами бинари/SDK.
- **Станции, постаматы, SafeKeeper, умные полки** — **общаться для рабочих задач можно без перепрошивки**, заняв роль их центрального сервиса (см. раздел «Рабочие задачи… без перепрошивки»). Контракт сервиса у нас почти весь; недостаёт **только station-facing половины** протокола (что станция шлёт серверу) — добирается сетевым трейсом или станционным билдом, **не реверсом железа**.
- Чего реально нет в пакете: станционного ПО/прошивки — поэтому station→server вызовы пока не видели. Это единственный остаток, и он закрывается без перепрошивки.

| Класс устройства | Управление? | Чем располагаем (подтверждено) | Чего не хватает |
|---|---|---|---|

| **Противокражные ворота** | ✅ **полное** | COM-интерфейс `RFID_ANTIVOR_ActiveX` (`ReaderOpen/ReaderDataGet/EAS_Set/Reset/Alarm`) → ядро; детект меток, EAS, тревога; релейный выход сирены (`FEISC_0x72_SetOutput`/`RDR_SetOutput`) | — |
| **Камеры Dahua / FaceID / счётчик** | ✅ **полное** | Dahua NetSDK (DLL у нас есть) + паттерн вызовов (`Login/DetectFace/RealPlay/Snapshot`), `:37777` | — (нужно Dahua-железо) |
| **Android ТСД (хранение фонда)** | ✅ **через SDK терминала** | модель данных + REST `BookKeeping*` + UHF-стек (Chainway R3 в наличии) | UHF-SDK конкретного терминала (даёт вендор; не блокер реверса) |
| **СКУД / турникеты** | ⚠️ **частично** | стационарные RFID-считыватели/антенны — драйвим тем же стеком; события прохода | актуация турникета (открыть/закрыть) — контроллер СКУД |
| **Станции самообслуживания** | ⚠️ **частично** | RFID + вся **логика книговыдачи** (наша) | механика: чековый принтер, конвейер/выдача — прошивка станции |
| **Постаматы / SafeKeeper** | ⚙️ **через роль сервиса** (см. ниже) | заказы/ячейки/состояния/мастер-ключ (DTO + контракт api.svc) | **station-facing половина протокола** (что станция шлёт серверу и ждёт) — нужен трейс/станционный билд, **не перепрошивка** |
| **Умные полки** | ⚙️ **через роль сервиса** | формат инвентаризации (DTO) | station-facing вызовы контроллера полки |

## Почему именно так

- **RFID-уровень полностью реверснут и у нас есть бинарь ядра** → читать/писать метки (Danish/ISO 28560-2), ставить/снимать EAS, писать AFI, регулировать мощность — можем **прямо сейчас**. Это покрывает: книговыдачу/возврат на настольном считывателе, противокражку, инвентаризацию ТСД, идентификацию по карте/ЕКП.
- **Камеры** — стандартный Dahua NetSDK, есть и SDK, и паттерн → FaceID и счётчик посетителей воспроизводимы.
- **Станционная механика** (ячейки, конвейер, полки, турникеты) живёт в **контроллере станции**, а не в управляющем ПО. В полученном пакете (LAS + EasyBook + драйверы) этого кода нет — LAS лишь обменивается с центральным сервисом метаданными. Поэтому «открыть ячейку №N» мы **не можем** без протокола станционного контроллера (реле/RS485/соленоид) — это не пробел рекона, а отсутствующий артефакт.

## Рабочие задачи со станциями БЕЗ перепрошивки — как это вообще работает

Уточнение по сути вопроса: «общаться со станциями/постаматами/SafeKeeper для рабочих задач» (создать заказ → ячейка выделяется/открывается → читатель забирает), **не трогая прошивку**.

**Архитектура общения** (из рекона): станции/постаматы/SafeKeeper/полки общаются с **центральным сервисом** (`api.svc`), а НЕ напрямую с ИРБИС и НЕ с LAS. Рабочий цикл (на примере SafeKeeper): заказ создаётся в сервисе (`OrderModify`) → камера узнаёт о нём от сервиса → на локере читатель/мастер-ключ прикладывает карту → **прошивка камеры открывает ячейку локально** и рапортует обратно (`OrderBookProcessedSet`, статус `Issued`). То есть «открыть ячейку» инициирует сама станция по данным от сервиса — а не команда «open cell N» снаружи.

**Можем ли мы в этом участвовать без перепрошивки? — Да, заняв роль центрального сервиса.** Условия:
1. **Станция указывает на нас.** Устройства сетевые (у `SK_SafeKeeper` есть `IPAddress`, у ворот/СКУД — IP/порт). Система конфигурируема → URL сервера у станции, скорее всего, настраивается. Это **переконфигурация, не перепрошивка**.
2. **Мы предъявляем station-facing контракт.** LAS-facing контракт `api.svc` у нас **полностью** ([CENTRAL_SERVICE_API.md](CENTRAL_SERVICE_API.md), 151 операция). Часть station-facing вызовов, скорее всего, — те же маршруты (`OrdersGet`/`OrderBodyGet`/`OrderModify`/`MastersRFIDGet`/`SafeKeeperMasterRFIDModify`/`OrderBookProcessedSet`) + device-facing `/easybookdll/*` (по аналогии с настольным считывателем, где device→сервер канал мы подтвердили).

**Чего не хватает (и как добрать — без перепрошивки):** **точный набор вызовов «станция → сервис»** — что именно станция запрашивает и постит, и что ждёт в ответ. Станционного ПО в полученном пакете нет, поэтому этот «обратный» край контракта мы не видели. Добирается **тремя способами, ни один не требует перепрошивки**:
- **(a) сетевой трейс** одной живой сессии «станция ↔ сервис» (Wireshark/proxy) — самый быстрый;
- **(b) станционный билд/конфиг** (ПО, которое стоит на самом киоске/локере) — как у нас есть для настольного агента (TagService), но для станций его не передали;
- **(c) протокол от поставщика** станций/локеров.

**Прямое общение (peer-to-peer) со станцией по её IP** — теоретически возможно (устройства адресуемы), но LAS так не делает (только через сервис), и протокол прямого API станции нам неизвестен. Поэтому рабочий путь — **быть их сервисом**, а не ходить на их IP.

> **Точный ответ:** да, решать рабочие задачи со станциями/постаматами/SafeKeeper **без перепрошивки реально** — заняв позицию центрального сервиса (бóльшая часть контракта уже известна). Единственный недостающий артефакт — **station-facing половина протокола**, и она добирается трейсом/станционным билдом, а не реверсом железа. Это понижает §1–2 [OPEN_QUESTIONS.md](OPEN_QUESTIONS.md) с «нужна прошивка» до «нужен захват station↔server протокола».

## Что это значит для проекта

- **Фаза 1–2 Biblio реализуема полностью** на полученных данных: книговыдача/возврат/продление/регистрация (настольный считыватель), противокражные ворота, FaceID/камеры, мобильная инвентаризация (ТСД). Бесшовный подхват этих устройств — конфигом (см. [TZ_PHASE1_abis_device_core.md](TZ_PHASE1_abis_device_core.md) §6).
- **Фаза 3 (станции/постаматы/полки)** требует **доступа к станции или её SDK/прошивке** — это единственное, что нельзя получить из текущего ПО. Варианты: запросить протокол у поставщика станций (IDlogic/производитель локеров), снять сетевой/последовательный трейс с работающей станции, или получить станционный build.

> Итог: **RFID-парком и камерами — управляем напрямую сейчас.** Станциями/постаматами/SafeKeeper — **решаем рабочие задачи без перепрошивки**, заняв роль их центрального сервиса; недостаёт только station-facing половины протокола, которая добирается сетевым трейсом или станционным билдом (не реверсом железа). Перепрошивка не требуется нигде.
