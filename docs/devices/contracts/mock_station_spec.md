# Спецификация МОК‑станции (эмулятор SafeKeeper / постамата)

> **Зачем:** позволить разрабатывать и тестировать оркестрацию станций (заказ → ячейка → выдача) **без реального железа**. Мок играет роль станции/постамата, общаясь с Device Service Biblio так, как (по реконструкции) это делает реальная станция IDlogic. Одновременно служит **контракт‑тестом** для Device Service.
> **Опора:** реконструкция из кода‑потребителя LAS — см. [CONTROL_CAPABILITY.md](../CONTROL_CAPABILITY.md), [CENTRAL_SERVICE_API.md](../CENTRAL_SERVICE_API.md), [device_service_openapi.json](device_service_openapi.json).
> **Архитектура (#272):** мок тестирует **нативный поток Biblio** (holds #222 + домен `devices` + circulation) через **device-facing compat-шим**, а не «второй Device Service». См. [DEVICES_NATIVE_ARCHITECTURE.md](../DEVICES_NATIVE_ARCHITECTURE.md). Маршруты ниже = device-facing вызовы, которые шим принимает и транслирует в нативные сервисы.
> **Достоверность:** **[C]** — подтверждено клиентским кодом api.svc; **[I]** — инференс по реконструированной модели (направление «станция→сервис» в файлах не наблюдалось; уточняется позже трейсом/реальной станцией). Мок кодифицирует именно гипотезу [I], чтобы её можно было проверить.

## 1. Что эмулирует мок

Одна **камера бронирования SafeKeeper** (`StationTypeID` 3/4) или **постамат**: N ячеек, сетевой адрес, мастер‑ключи. Состояние: занятость ячеек (битовая маска), онлайн/здоровье, журнал событий. Драйвит «виртуальные» ячейки (open/занять/освободить) по данным от Device Service.

Параметры запуска (конфиг мока):
```
service_url   = https://<biblio-device-service>/        # наш Device Service
device_id     = <GUID станции>
auth          = Basic ServiceLogin:<legacy>             # унаследованный device-facing Basic [C]
cells_count   = 12
cell_shift    = 0                                        # CellNumberShifted = CellNo - CellShift [C]
ip_address    = 10.0.0.50                                # SK_SafeKeeper.IPAddress [C]
poll_interval = 5s
```

## 2. Модель состояния (реконструирована)

- **Ячейки:** `cells_state: long` — битовая маска, **бит i = ячейка i занята(1)/свободна(0)** (`BindingCells`) **[C]**. Свободные для заказа — `IsBusy==false && StationTypeID∈{3,4}` **[C]**.
- **Заказ** (`SK_Order`/`SK_OrderBody`): `StateId` `1`Создан→`2`Подготовлен→`3`Укомплектован→`4`Выдан→`6`Выдан‑с‑ошибками; отмена/удаление = opID `0` **[C]**. Ячейка в заказе передаётся как `cellNumber = CellNo - CellShift` **[C]**.
- **Здоровье:** `DeviceStateID` `1`неизв/`2`исправно/`3`диагностика/`4`не настроено — производное от `DeviceDataAdd{softOnlineCount,deviceOkCount,deviceErrorCount}` **[C]**.
- **Мастер‑ключ:** RFID‑карта сотрудника (`SK_MasterRFID`), привязана к станции (`SafeKeeperMasterRFIDModify`) **[C]**.

## 3. Поведение мока (циклы)

### 3.1 Heartbeat / здоровье — каждые `poll_interval` **[C для эндпоинта, I для частоты]**
```
POST /easybookdll/DeviceDataAdd
  { deviceID, softOnlineCount:1, deviceOkCount:<ok>, deviceErrorCount:<err> }
```
(в режиме мониторинга LAS читает это как почасовую исправность.)

### 3.2 Получение заказов на укомплектование **[I]**
Мок периодически запрашивает заказы своей станции и наполнение:
```
POST /las/OrdersGet                      → [SK_Order]   (фильтр по своей safeKeeperID)
POST /las/OrderBodyGet { orderID }       → [SK_OrderBody]  (какие книги класть)
POST /las/SafeKeeperInfoGet2 { safeKeeperID } → [SK_SafeKeeperInfo]  (карта ячеек)
```
> Гипотеза направления: станция **опрашивает** сервис (poll). Альтернатива — сервис пушит на `ip_address` станции; уточняется трейсом. Мок реализует poll (проще и совпадает с TagService‑паттерном).

### 3.3 Укомплектование (Подготовлен → Укомплектован) **[I]**
При появлении заказа в статусе `2` (Подготовлен): мок «открывает» свободную ячейку, кладёт книги, помечает ячейку занятой и переводит заказ в `3`:
```
для каждой книги: POST /las/OrderBookProcessedSet { bookCode, orderID }    [C: операция существует]
POST /las/OrderModify { opID:2, id:orderID, ..., stateID:3, safekeeperID, ... }   [C: формат]
обновить cells_state: set bit(cellNo)=1
```

### 3.4 Выдача (Укомплектован → Выдан) **[I]**
Симуляция «читатель приложил карту к локеру»: мок находит заказ читателя в статусе `3`, «открывает» ячейку, освобождает её и переводит заказ в `4`:
```
trigger: mock CLI/HTTP "reader_tap {readerRFID}"
POST /las/OrderModify { opID:2, id:orderID, ..., stateID:4, ... }
обновить cells_state: set bit(cellNo)=0
(при ошибке записи в АБИС со стороны сервиса — заказ остаётся 6)
```

### 3.5 Сервисное открытие мастер‑ключом **[I]**
```
trigger: mock CLI "master_tap {rfid}"
POST /las/MastersRFIDGet → проверить, что rfid привязан к станции
"открыть" указанную ячейку, записать событие истории
```

## 4. Тест‑сценарии (приёмочные для Device Service)

1. **Полный цикл брони:** оператор в Biblio создаёт заказ (`OrderModify opID:1` → state 1) и наполняет (`OrderBodyModify`) → мок видит заказ, укомплектовывает (state 3, ячейка занята) → `reader_tap` → выдача (state 4, ячейка свободна) + списание на читателя через ABIS‑порт. Проверка: статусы и `cells_state` согласованы на всех трёх сторонах (Biblio БД, мок, ABIS).
2. **Отмена:** заказ `opID:0` до укомплектования → ячейка не занималась.
3. **Выдача с ошибкой АБИС:** ABIS‑порт возвращает ошибку при выдаче → заказ переходит в `6`, ячейка освобождена, повтор из портала.
4. **Здоровье/оффлайн:** мок прекращает heartbeat → Device Service помечает станцию оффлайн (`IsOnline=0`), мониторинг это показывает.
5. **Мастер‑ключ:** `master_tap` валидного/невалидного ключа → открытие/отказ + запись в историю.

## 5. Реализация (рекомендация)

- Лёгкий сервис (Python/FastAPI или Node) с: HTTP‑клиентом к Device Service, in‑memory состоянием (cells_state, заказы), фоновым poll‑циклом, и CLI/HTTP‑триггерами `reader_tap`/`master_tap` для ручных сценариев.
- Конфиг — как в §1. Логировать все запросы/ответы (для сверки с будущим реальным трейсом).
- **Помечать [I]‑вызовы флагом** — когда появится реальный трейс/станция, заменить гипотезу на факт в одном месте.

## 6. Что мок НЕ заменяет
- Точный «станция→сервис» протокол реальных постаматов (эндпоинты/JSON/направление). Мок — **наша гипотеза**; сверка — позже одним сетевым трейсом или станционным билдом ([OPEN_QUESTIONS §1–2](../OPEN_QUESTIONS.md)). До тех пор мок полностью разблокирует разработку и тестирование оркестрации ячеек на стороне Biblio.
