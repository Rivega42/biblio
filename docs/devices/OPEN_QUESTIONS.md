## 1. 🔴 Привод ячеек постаматов / SafeKeeper — где и какой протокол
**Факт:** в `LibraryAdminServer.exe` и `IDLogicTagServiceLAS.exe` **нет** команд открытия ячеек/реле (грепом подтверждено отсутствие `OpenCell/CellOpen/OpenDoor/OpenLock/Unlock/solenoid/Relay/Modbus/SerialPort/System.IO.Ports/\\.\COM`). LAS лишь пишет метаданные заказа через `api.svc`; физическое открытие — мастер‑ключом на самой камере (`manual.txt:225,231`).
**Вывод:** реальный контроллер ячейки (релейная плата / RS485‑шина / соленоид) — в **прошивке станции SafeKeeper/постамата**, которой в этой поставке нет.
**Что нужно:**
- доступ к **контроллеру станции SafeKeeper/постамата** (отдельный модуль ПО / прошивка / её дистрибутив `TagServiceLAS`‑аналог на самом киоске);
- либо сетевой трейс «LAS/сервер → станция» при укомплектовании/выдаче;
- либо документация IDlogic на станционный протокол (релейная плата: тип, адресация ячеек, команда открытия).
**Зачем:** без этого Biblio не сможет **физически открыть ячейку** — только вести учёт заказов.

## 2. 🔴 Логика самой станции книговыдачи / киоска (Vending)
**Факт:** станция отдаёт серверу события и здоровье узлов (`IsPrinterOk`, `IsConveyorOk`), но книговыдача/печать чека/конвейер происходят **на киоске**; в LAS команд запуска нет.
**Что нужно:** ПО/прошивка киоска самообслуживания (вероятно отдельный «station»‑build IDlogic), его конфиг, протокол к чековому принтеру и механизму выдачи.
**Зачем:** воспроизвести сам сценарий самообслуживания, а не только статистику.

## 3. 🟡 Счётчик посетителей в воротах (вход/выход)
**Факт:** в `RFID_ANTIVOR_ActiveX.dll` счётчика нет; LAS читает агрегаты `MON_GateCounter` (`CounterIP`/`CounterPort`/направление) с сервера.
**Что нужно:** модель/вендор счётчика (встроен в ворота? отдельный ИК/видео‑датчик?), его сетевой протокол (IP/порт). Возможно FEIG people‑counter или Dahua‑видео.
**Зачем:** реплицировать вход/выходную статистику.

## 4. ✅ РАЗРЕШЕНО (клиентская сторона) — JSON‑контракт `api.svc`
Декомпиляция `WebService.cs` выполнена: **151 операция** (route → тело → ответ) и **106 DTO** извлечены целиком — [CENTRAL_SERVICE_API.md](CENTRAL_SERVICE_API.md). Остаётся неизвестной только **серверная** сторона: коды ошибок/валидаторы и точные тела ответов конкретных операций (тип-контракт известен, наполнение — нет). Для этого по-прежнему полезен один сетевой трейс LAS↔`api.svc`.

## 5. ✅ РАЗРЕШЕНО — Тег подполя RDR для RFID/ЕКП‑карты
**Было неясно**, теперь подтверждено декомпиляцией `EasyBookAbis.Irbis.Irbis`:
- **ЕКП → поле `28`** (`AddEkp`/`AddReader`, `Irbis.cs:286,294,333`);
- **карта/билет → поле `30`** (`AddReaderByCard`, `:381`);
- **доп. карта → поле `24`** (`AddExtraCard`, `:2367-2380`);
- **PIN → поле `103`** (`GetClientName`, `:626`); GUID → `12000`; хэши поиска → `12814/12815`.
Подробно — [EASYBOOKABIS_IRBIS_MAP.md](EASYBOOKABIS_IRBIS_MAP.md) §2,§3. → Для Biblio привязки хранить в этих же полях/эквивалентах.

## 6. 🟡 ЧАСТИЧНО — Семантика поля 691 и раскладка RQST
**Разрешено:** заказы в RQST ищутся `RI=` / `V30:'…' and V910^H<>''`, номер заказа = поле `903`; рекомендации — `V691^F/^C/^V/^A/^O` из поля `90` читателя; `Hold` через ИРБИС — **заглушка `return false`** (`Irbis.cs:2389`). См. [EASYBOOKABIS_IRBIS_MAP.md](EASYBOOKABIS_IRBIS_MAP.md) §3.
**Осталось инференсом:** точный смысл подполей `691^F/^C/^V/^O` и полная раскладка записи RQST (поля заказа).
**Что нужно:** PFT/FST БД KNIGA/RQST на стенде (`C:\IRBIS64\Datai\`) — сверить с реальными записями. Влияет на критерий миграции RQST в режиме 2 (см. [TZ_PHASE1_abis_device_core.md](TZ_PHASE1_abis_device_core.md) §8).

## 7. 🟢 Бинарные `.cfg/.cfg2/.ekp` и схема лицензирования
**Факт:** `EasyBook_rfid.cfg` (20 б) и `.ekp` (12 б) — бинарные (lightly‑encoded); декод — в `EasyBook_RFid.dll`/`LicenseChecker.dll`. Лицензия привязана к `DeviceID` (GUID) и проверяется веб‑вызовом `DeviceIsLicenseValid` + локально.
**Что нужно (если важно):** декомпиляция `LicenseChecker.dll`/`EkpMed*.dll`.
**Зачем:** понимать привязку лицензий считывателей (для миграции парка). Низкий приоритет — нашей системе своя лицензия.

## 8. 🟢 ISO 28560 / Danish — точная раскладка байт в блоках
**Факт:** сериализатор `%s=%d;%d;%d;%s;%s;%s` виден; точное смещение полей ISO 28560 внутри блоков ISO15693 — в коде.
**Что нужно:** дамп реальной метки (несколько блоков) со стенда + декомпиляция `IDLogic_Danish`; либо стандарт ISO 28560‑2/3.
**Зачем:** побайтовая совместимость при чтении уже промаркированного фонда. Для HF — обязательно при работе с существующими метками.

## 9. 🟢 AFI/EAS конкретные значения и пароли
**Факт:** `AFI ON/OFF` и пароль EAS — конфигурируемые строки (дефолт `00000000`); реальные значения вводятся в UniDll/LAS.
**Что нужно:** значения с боевого стенда (какой AFI/пароль EAS у фонда).
**Зачем:** корректно взводить/снимать противокражную подпись на существующих метках.

## 10. ✅ РАЗРЕШЕНО — SOAP vs REST у `api.svc`
Декомпиляция `WebService.cs` подтверждает: клиент использует **REST/JSON** (`HttpClient.PostAsync`, `application/json`, `Newtonsoft.JsonConvert`), Basic‑auth; никакого SOAP/WCF‑биндинга в клиенте нет. → строить как REST.
