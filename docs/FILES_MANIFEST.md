# Манифест файлов и раскладка репозитория (что где и для чего)

> Назначение: дать Claude Code точную карту комплекта — что каждый файл, для чего, **куда положить**
> в репозитории, какой у него статус (истина / источник / справка) и нужно ли его **поддерживать**
> или только **читать**. Используй этот файл, чтобы правильно организовать `docs/`. Дополняет
> `CLAUDECODE_PROJECT_BRIEF.md` (зачем/как) и `PROJECT_INDEX.md` (порядок чтения). Дата 20.06.2026.

## 3. Где что — истина по темам (куда смотреть при сомнении)

- **Протокол / команды:** `reference/irbis64_client_dll_TEXT.txt` → сводно `recon/RECON_FINDINGS_02.md`
  → привязка к функциям `design/FUNCTION_PROTOCOL_MAP.md`.
- **Доступ / права:** `design/ACCESS_MODEL_web-irbis.md`.
- **Контракты кода P0 (структура, БД, API, интерфейс протокола):** `build/P0_BUILDKIT_web-irbis.md`.
- **Что умеет продукт в целом:** `design/IRBIS_CAPABILITY_MAP_v2.md`.
- **Экраны/UX:** `design/SCREENMAP_web-*.md`.
- **Префиксы/форматы/связи конкретной базы:** `recon/RECON_FINDINGS_04.md` + `_05.md` (механика);
  значения боевых баз — снимать с сервера СПб ГТБ (`TODO(Проход Б)`).

## 4. Связи (что из чего выводится)

```
reference (протокол, мануал)
   └─> recon/FINDINGS (доказательная база, с источниками)
          └─> design (CAPABILITY_MAP, ACCESS_MODEL, FUNCTION_PROTOCOL_MAP, SCREENMAP)
                 └─> build (TZ + ADDENDUM + P0_BUILDKIT)
                        └─> код (backend/frontend)
PROJECT_BRIEF + INDEX + MANIFEST — навигация поверх всего.
```
Правило: при расхождении приоритет — `reference` > `recon` > `design` > `build`. Если код требует
факта, которого нет, — `TODO(...)`, а не догадка.

## 5. Чего в репозитории НЕТ (намеренно)

- Исходные файлы продукта ИРБИС (конфиги/доки/бинарь/данные) — они локально, в `.gitignore`. В репо —
  только наши производные (документация выше + код).
- Устаревшая `IRBIS_CAPABILITY_MAP.md` (v1) — удалена, канонична `_v2`. Не возвращать.
- Секреты, `*.ini` с кредами, `.env`, дампы, картинки-данные.

## 6. Что Claude Code должен сделать с этим комплектом

1. Создать структуру `docs/` по §1 и разложить файлы по папкам согласно §2.
2. Завести `docs/CHANGELOG.md` и корневой `README.md` (обзор + ссылка на `docs/CLAUDECODE_PROJECT_BRIEF.md`).
3. Поддерживать (режим «вести») документы статуса И/П синхронно с кодом; «читать» — не править,
   ссылаться. Незакрытое — `TODO(...)`, факты — с источником.
4. Не нарушать §5 (что не кладём в репо) и приоритет источников §4.
