## Физические файлы БД ИРБИС (MST/XRF/IFP/L01/N01/CNT и пр. — назначение каждого)

| Файл (расширение) | Назначение по документу |
|---|---|
| **MST** (файл документов) | Хранит записи переменной длины (лидер + справочник + поля). Новые/изменённые записи всегда дописываются в конец `Irbis64_struct.rtf (txt:5, 88, 97)`. Первая запись MST — управляющая (control record) `Irbis64_struct.rtf (txt:49)`. |
| **XRF** (файл перекрёстных ссылок) | Таблица ссылок на записи MST: ссылка №1 - MFN 1, №2 - MFN 2 и т.д. Хранит индекс позиции каждой записи `Irbis64_struct.rtf (txt:61, 63-65)`. |
| **N01** | Часть инверсного файла: узлы бинарного дерева словаря поисковых терминов `Irbis64_struct.rtf (txt:160-161, 164)`. Адрес корневой записи хранится как номер первой записи N01 `Irbis64_struct.rtf (txt:201)`. |
| **L01** | Часть инверсного файла: листья бинарного дерева. Записи-листья указывают на файл ссылок IFP `Irbis64_struct.rtf (txt:161, 186, 202)`. |
| **IFP** | Файл ссылок (постингов): список ссылок, соответствующих каждому термину словаря `Irbis64_struct.rtf (txt:160, 193, 252-253)`. |
| **BKP** | Файл копии, создаётся при реорганизации MST; служит для восстановления MST. Структура та же, что у MST, но без XRF (записи лежат последовательно без пропусков); удалённые записи в копию не пишутся `Irbis64_struct.rtf (txt:121)`. |
| **Lk1** | Промежуточный файл: формируется после отбора и сортировки терминов; из него загружается словарь, при этом выбирается формат размещения ссылок в IFP `Irbis64_struct.rtf (txt:254)`. |

Замечания по структуре инверсного файла: он состоит из 3 физических файлов — два (N01, L01) содержат словарь поисковых терминов в структуре бинарного дерева, третий (IFP) — список ссылок на каждый термин `Irbis64_struct.rtf (txt:160-161)`.

## Логическая структура записи (MFN, метка поля, подполя ^x, повторения, статусы записи)

### Запись MST = лидер + справочник + поля переменной длины
Запись файла документов — переменной длины, состоит из трёх частей: лидера фиксированной длины, справочника и полей переменной длины `Irbis64_struct.rtf (txt:7)`.

**Лидер (7 целых чисел)** `Irbis64_struct.rtf (txt:11-23)`:
- `MFN` (32 бита) — номер файла документов (номер записи);
- `MFRL` (32) — длина записи; всегда чётное число;
- `MFB_LOW` (32) — ссылка назад (младшее слово);
- `MFB_HIGH` (32) — ссылка назад (старшее слово). Смещение на предыдущий вариант записи имеет длину 8 байт = 4+4 `Irbis64_struct.rtf (txt:18)`;
- `BASE` (32) — смещение (базовый адрес) полей переменной длины, т.е. общая длина лидера и справочника в байтах;
- `NVF` (32) — число полей в записи (число входов в справочнике);
- `STATUS` (32) — индикатор записи: `BIT_LOG_DEL (1)` — логически удалённая, `BIT_LAST_REC (32)` — последний экземпляр записи, `BIT_ALL_ZERO (0)` — предыдущий вариант записи, `BIT_NOTACT_REC (8)` — неактуализированная (непроинвертированная) `Irbis64_struct.rtf (txt:21-22)`;
- `VERSION` (32) — номер версии записи `Irbis64_struct.rtf (txt:23)`.

**Справочник** `Irbis64_struct.rtf (txt:28-35)`: таблица из NVF входов, каждый вход — 3 целых числа:
- `TAG` (32) — метка поля;
- `POS` (32) — смещение первого символа поля в разделе полей переменной длины (смещения упорядочены по возрастанию, у первого поля POS = 0);
- `LEN` (32) — длина поля в байтах.

Длина справочника = `12*NVF` байт; `BASE = 26 + 12*NVF` `Irbis64_struct.rtf (txt:35)`.

**Поля переменной длины** `Irbis64_struct.rtf (txt:44-46)`: данные полей идут одно за другим в порядке справочника, без разделителей между полями.

### Управляющая запись MST (первая запись файла) `Irbis64_struct.rtf (txt:49-59)`
- `CTLMFN` (32) — резерв;
- `NXTMFN` (32) — номер, назначаемый следующей создаваемой записи;
- `NXT_LOW` (32) — младшее слово смещения на свободное место в файле (всегда указывает на конец MST);
- `NXT_HIGH` (32) — старшее слово того же смещения;
- `MFTYPE`, `RECCNT`, `MFCXX1`, `MFCXX2` (32 каждый) — резерв;
- `MFCXX3` (32) — индикатор блокировки базы данных (0 — нет, >0 — да).

### Флаги в XRF `Irbis64_struct.rtf (txt:66-76)`
Каждая ссылка XRF — 3 поля: `XRF_LOW` (32, младшее слово 8-байтового смещения на запись), `XRF_HIGH` (32, старшее слово), `XRF_FLAGS` — битовые флаги:
- `BIT_LOG_DEL (1)` — логически удалённая запись;
- `BIT_PHYS_DEL (2)` — физически удалённая;
- `BIT_ABSENT (4)` — несуществующая запись;
- `BIT_NOTACT_REC (8)` — неактуализированная запись;
- `BIT_LOCK_REC (64)` — заблокированная запись.

TODO(recon #1205: в документе `XRF_FLAGS` помечен как «33 бита» — `txt:71` — очевидная опечатка вместо 32; также в формате IFP «33 LOW» — `txt:281`).

### Техника актуализации
- **Создание**: запись добавляется в конец MST; номер берётся из `NXTMFN`, который затем +1. В XRF создаётся ссылка с флагами `BIT_NEW_REC + BIT_NOTACT_REC`; STATUS новой записи = `BIT_LAST_REC`. `BIT_NOTACT_REC` означает, что запись подлежит инвертированию `Irbis64_struct.rtf (txt:88-91)`.
- **Модификация**: новая версия пишется в конец MST со STATUS `BIT_LAST_REC + BIT_NOTACT_REC`; STATUS старой версии становится `BIT_ALL_ZERO(0) + BIT_NOTACT_REC`. Ссылка назад (`MFB_LOW/HIGH`) новой версии указывает на предыдущую. После инвертирования (в отличие от ISIS) ссылка назад НЕ обнуляется — для возможности отката; флаг `BIT_NOTACT_REC` снимается в XRF и MST `Irbis64_struct.rtf (txt:97-102)`.
- **Удаление**: трактуется как модификация; в XRF добавляются `BIT_LOG_DEL + BIT_NOTACT_REC`, в MST в STATUS добавляется `BIT_LOG_DEL` `Irbis64_struct.rtf (txt:106-111)`.
- **Реорганизация**: сжатие MST через файл BKP; при восстановлении логически удалённые записи становятся физически удалёнными (`XRF_FLAG = BIT_PHYS_DEL`); копирование невозможно при наличии хотя бы одной непроинвертированной записи `Irbis64_struct.rtf (txt:118-122)`.

## Словарь и инвертирование (как устроен IFP/термины/ссылки)

### Бинарное дерево словаря (N01/L01) `Irbis64_struct.rtf (txt:200-251)`
- Взаимосвязи N01-L01 — через относительные адреса (порядковый номер записи в файле).
- Структура записи одинакова для N01 и L01; длина записи (блока) постоянна и зависит от реализации: 512/1024/2048/4096 байт (в данной реализации **2048**) `Irbis64_struct.rtf (txt:200, 249-250)`. Максимальный размер файла L01/N01 = 2 ГБ * Размер записи.
- Смещение на запись IFP хранится в L01, длина 64 байта (используется только младшее слово смещения) `Irbis64_struct.rtf (txt:202)`.

Запись N01/L01 = лидер + справочник + ключи переменной длины `Irbis64_struct.rtf (txt:207)`.

**Лидер записи N01/L01** `Irbis64_struct.rtf (txt:209-215)`:
- `NUMBER` (32) — номер записи (с 1; в N01 номер первой = номеру корневой записи дерева);
- `PREV` (32) — номер предыдущей записи (нет = -1);
- `NEXT` (32) — номер следующей записи (нет = -1);
- `TERMS` (16) — число ключей в записи;
- `OFFSET_FREE` (16) — смещение на свободную позицию (от начала записи).

**Справочник** (вход на каждый ключ) `Irbis64_struct.rtf (txt:217-234)`:
- `LEN` (16) — длина ключа;
- `OFFSET_KEY` (16) — смещение на ключ от начала записи;
- `LOW` (32): в N01 — ссылка на запись N01 (LOW>0, более низкий уровень индекса) или на запись L01 (LOW<0, лист); в L01 — младшее слово 8-байтового смещения на запись IFP;
- `HIGH` (32): в N01 — всегда 0; в L01 — старшее слово смещения на запись IFP.

Длина справочника = `12*TERMS`; длина ключей = `Размер записи - OFFSET_FREE`; свободное место = `16 + 12*TERMS - длина ключей` `Irbis64_struct.rtf (txt:246-248)`. Ключи пишутся с конца записи, входы упорядочены алфавитно; сами ключи лежат вплотную без разделителей в порядке поступления `Irbis64_struct.rtf (txt:237)`.

### Файл IFP — списки ссылок терминов `Irbis64_struct.rtf (txt:252-283)`
Формат размещения ссылок выбирается при загрузке словаря из Lk1 в зависимости от общего числа ссылок термина:
- **Обыкновенный формат** — заголовок блока + упорядоченный набор ссылок.
- **Специальный формат** — включается при превышении `MIN_POSTINGS_IN_BLOCK` (в данной реализации **256**): специальный блок + набор блоков обыкновенного формата размером 4/8/16/32 Кб для числа ссылок 256-32000 / 32000-64000 / 64000-128000 / 128000 и более `Irbis64_struct.rtf (txt:254-255)`.

**Ссылка (posting)** `Irbis64_struct.rtf (txt:260-265)`:
- `PMFN` (32) — номер записи (MFN);
- `PTAG` (32) — идентификатор поля, назначенный при отборе терминов в словарь;
- `POCC` (32) — номер повторения;
- `PCNT` (32) — номер термина в поле.

**Заголовок блока IFP** `Irbis64_struct.rtf (txt:266-273)`: `LOW`/`HIGH` (смещение на следующую запись, нет = 0), `TOTP` (общее число ссылок термина — только в первой записи; в последующих — число ссылок в блоке), `SEGP` (число ссылок в блоке), `SEGC` (вместимость записи в ссылках). Признак последнего блока — `LOW=HIGH=-1`.

**Специальный блок** `Irbis64_struct.rtf (txt:276-283)`: заголовок обыкновенного формата со смещениями = -1001 + входы вида `POSTING` (1-я ссылка из обыкновенной записи) + `LOW`/`HIGH` (смещения на следующую запись). Число входов кратно 4. Общее число ссылок термина хранится только в специальном блоке.

**Модификация IFP** `Irbis64_struct.rtf (txt:285-287)`: при добавлении ссылок создаётся новая запись (размером = общему числу ссылок без спецблока, либо = числу ссылок в записи при наличии спецблока); связывается через поле NXT_, ссылки равномерно распределяются между старой и новой записью с сохранением возрастающего порядка.

## Формат RTF при выводе (кратко, если относится к ИРБИС)

Документ `Формат_Rich_Text.doc.txt` — это общее описание стандарта RTF фирмы Microsoft (формат обмена текстовыми документами, аналог SYLK для таблиц), а **не** описание внутренней структуры БД ИРБИС `Формат_Rich_Text.doc (txt:1-2)`. Прямых указаний на использование RTF в выводе/печати ИРБИС в тексте нет.

Ключевые сведения из документа (применимы к выводу/печати, если ИРБИС формирует RTF):
- Документ состоит из текста и команд управления: управляющих слов (`\lettersequence<delimiter>`, перед словом — `\`) и управляющих символов `Формат_Rich_Text.doc (txt:4-13)`.
- Группировка фигурными скобками `{ }`; экранирование `\\`, `\{`, `\}` `Формат_Rich_Text.doc (txt:15-22)`.
- Кодировки: `\ansi` (ASCII/Windows — рекомендуется для переноса между системами), `\mac`, `\pc`, `\pca` (CP850) `Формат_Rich_Text.doc (txt:76-80)`.
- Метка начала файла `{\rtf<параметр>...}` `Формат_Rich_Text.doc (txt:71-74)`.
- Таблицы шрифтов `\fonttbl` и цветов `\colortbl`; стили `\stylesheet`; вставка рисунков `\pict` `Формат_Rich_Text.doc (txt:81-138)`.
- Спецслова вывода: `\chpgn` (номер страницы), `\chdate`/`\chtime` (дата/время), `\page`, `\line`, `\par` (конец абзаца, может заменяться `\10`/`\13`), `\tab`, `\cell`/`\row` `Формат_Rich_Text.doc (txt:27-66)`.
- Форматирование документа/абзаца/символов: `\sectd`, `\pard`, `\ql/\qr/\qc/\qj`, `\b`, `\i`, `\ul`, `\f000`, `\fs000`, `\cf000` и др. `Формат_Rich_Text.doc (txt:222-235)`.
- Нераспознанные команды могут пропускаться программой чтения; конец команды отмечается разделителем (чаще — точкой с запятой) `Формат_Rich_Text.doc (txt:14, 235)`.
