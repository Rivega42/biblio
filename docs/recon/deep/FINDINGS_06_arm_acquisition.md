## Назначение и БД (CMPL/PODB/POST и др.)

Работа комплектатора включает четыре основные задачи: ЗАКАЗ, ПОСТУПЛЕНИЕ, ВЫБЫТИЕ, ПОДПИСКА — под них строится интерфейс (четыре закладки) `IrbisCom.doc (txt:268, 270, 280)`. Параметрический файл по умолчанию — irbisp.ini `IrbisCom.doc (txt:270)`.

Основные функции `IrbisCom.doc (txt:272-279)`: ввод кратких БО книг и периодики (ПИ); ввод сведений о заказе/подписке и печать форм; отслеживание выполнения заказов; регистрация поступлений; ввод в ЭК сведений о выбытии/передаче (списание); сверка ЭК с реальными изданиями — проверка фонда.

## Модель данных комплектования (заказ → поступление → регистрация экземпляров → подписка периодики → списание)

### Виды записей в БД CMPL
`IrbisCom.doc (txt:307-323)`, доп. Приложение 1 `(txt:3079-3132)`:
- КНИГА — БО + сведения о заказе; дезидерата / заказ / поступления; после 1-го экз. передаётся в ЭК `(txt:309-310)`.
- ОРГАНИЗАЦИЯ (посредник заказа) — однократно `(txt:311)`.
- СУММАРНЫЙ ЗАКАЗ (SZ) — кол-во, сумма, период + поэтапное исполнение `(txt:312)`.
- ПЕРИОДИЧЕСКОЕ ИЗДАНИЕ + подписка (пролонгация с сохранением прежних периодов) `(txt:313)`.
- АДРЕСАТ ПОДПИСКИ (филиал/отделение) `(txt:314)`.
- Запись КСУ ПОСТУПЛЕНИЙ `(txt:315)`; КСУ ВЫБЫТИЯ `(txt:316)`; КСУ ИТОГОВАЯ на дату `(txt:317)`; КСУ ПОСТУПЛЕНИЙ ЗА ПЕРИОД `(txt:318)`.
- ПЛАН НА ГОД `(txt:319)`; ИТОГОВАЯ ЗАПИСЬ ПОДПИСКИ (KS3) `(txt:320, 2786)`.
- ПОЛЬЗОВАТЕЛЬ (POLZV) — реквизиты, подписи, максимумы `(txt:321)`; КАТАЛОГ ПОДПИСКИ (КАТ) `(txt:322)`; РСУ (РЛ RSU) `(txt:1305)`.

### Две технологии ведения БД (способ регистрации поступлений) `(txt:367-389)`
1. Регистрация в CMPL `(txt:370-383)`: БО сперва в CMPL (инв.номера, шифр), затем «Перенос в ЭК» (новые/дописывание). Перед регистрацией — поиск в ЭК; дубль переносится из ЭК в CMPL. ModeEnterInCMPL=1 `(txt:387)`.
2. Регистрация сразу в ЭК `(txt:384-389)`: БО в ЭК, КСУ в CMPL, пополнение по записям каталога. ModeEnterInCMPL=0; перенос в ЭК не подаётся.
Постоянные записи CMPL (техн.1): КСУ пост./выб., итоговые КСУ, организации, POLZV `(txt:383)`.

### Поток ЗАКАЗ
Трёхуровневая структура «Организация — Заказы — Книги» `(txt:2266-2270)`. Заказ выполнен, если сумма «поступивших» (62^G) ≥ сумме «заказанных» (62^D) `(txt:414, 2283)`. Спецслучай: «Аукцион»/тендеры (ФЗ № 94) — SZ без организации, обязателен номер; форма Reestr/ЛОТ `(txt:2295, 2525-2537)`. Старая технология — поля 30-32 в БО (не рекомендуется) `(txt:415, 2344)`.

### Поток ПОСТУПЛЕНИЕ `(txt:402-410)`
Ввод/выбор КСУ → поиск дубля в ЭК → перенос дубля в CMPL (техн.1) → ввод нового БО / заимствование → перенос партии в ЭК (техн.1) → пополнение записи КСУ (статистика). Поступление по заказу — дерево «Организация-Заказы-Книги», отд. КСУ с номером заказа; ModePostForZakaz=1 `(txt:397-399, 411-417)`.
Экземпляры — поле 910; статусы 0 (индивид.), C (ЦБС, общим числом), U (безынв. учёт), R (временный — размножение) `(txt:617-623)`. Автоввод: дата, тек. КСУ, акт, заказ; обновление SZ заданием SumEkzToSZ=SumEkzToSZ.gbl `(txt:634-639)`.

### Регистрация экземпляров
- Максимальные инв.номера ведутся в POLZV; модели MODEINVENT/MODEINVENT1..5; лимит MAX910Polzv=50 `(txt:841-851, 3152-3215)`.
- Размножение: статус R, конструкция Numb/First_Inv → Numb экз. со статусом 0 `(txt:852-886)`.
- Дублетность при вводе — свёртка svertka, проверка по CMPL и до 3 БД ЭК (DBEK1/2/3) `(txt:580-595)`.

### Подписка периодики
РЛ: OJK (БО издания), AZP (адресат), КАТ (каталог подписки), KS3 (итоговая) `(txt:2645, 2649, 2640, 2786)`. Поля: «Период подписки» (33), «Индекс», «Цена комплекта», «Заказ по подразделениям» (Адресат/Получатель) `(txt:2656-2673)`. Продление: «Снятие периода подписки» (архивные поля) `(txt:2685-2693)`; задания ppj.gbl, PKJ.gbl, PKJ1.gbl, PKN.gbl `(txt:2709-2724)`. Передача в ЭК — поля 901, 938 `(txt:2796-2811)`.

### Списание (ВЫБЫТИЕ)
Факт списания/передачи в поле 910 и архивном 940 `(txt:1561, 1802)`. Понятия: «выбыл» (из подразделения) vs «исключён» (выбывших > переданных) `(txt:1562-1568)`. Виды списания → отдельные задания глобальной корректировки (см. ниже).

## «Комплектатор в Каталогизаторе» (отличия интегрированного режима)

Источник: Комплектатор в Каталогизаторе.docx.

В АРМ Каталогизатор доступна часть функций Комплектатора — кнопка на сервисной панели, видима только если текущая БД — БД каталога `(txt:42-43)`. Четыре режима: Создание/Пополнение записи КСУ, Выбытие, Проверка фонда, Выходные формы `(txt:46-50)`. Параметры берутся из СЕРВЕРНОГО параметрического файла, секция CMPL (РЛ опроса, имена заданий, форматы); нет секции — умолчания `(txt:52)`.

### Ключевое отличие (важно!)
При списании номер КСУ вписывается в поле экземпляра БЕЗ даты, словарь КСУ подаётся без даты. В базовом АРМ Комплектатор — С датой `(txt:63-64, 306)`. Поэтому записи КСУ выбытия из базового модуля НЕ следует обновлять в этом режиме (и наоборот) `(txt:306)`.

### Выбытие
РЛ CmplSpisWss=СС_SpisWss `(txt:67)`. Каждый вид — набор глоб. корректировок (начальная/основная/завершающая), имена в INI секции CMPL `(txt:68-69)`. Задания `(txt:70-233)`: книги инд./групп. SpisGblBinv; утерянные SpisGblLostInv=SpisGblUT; по файлу SpisGblFile; книги целиком SpisGblBAll; журналы целиком SpisGblJAll; отд. номер SpisGblJNum + SpisJMumAfter=KUMUL909; журнал по году-месту SpisGblJYear + KUMUL909; передача с/без КСУ (прочерк + акт), факт в 940.

### Создание/Пополнение КСУ
В CMPL по записям ЭК с введёнными КСУ `(txt:234-241)`. РЛ CMPLStatKsuWss=СС_StatKsuWss; справочник БД из ECATMNU или dbname_wn.mnu `(txt:241-242)`.
- Поступление, новая: StatKsuStepNewBef=СС_StatKsuStep1, StatKsuStepNewBas=СС_StatKsuStep2 `(txt:262-264)`.
- Поступление, пополнение: StatKsuStepUpdBef=СС_StatKsuUpdate, ...Bas=СС_StatKsuStep1, ...Aft=СС_StatKsuStep2 `(txt:271-274)`.
- Выбытие, новая: StatKs2StepBef=СС_StatKs2Step1, StatKs2Step2=СС_StatKs2Step2, StatKs2StepAft=CC_StatKs2Post `(txt:291-294)`.
- Выбытие, пополнение: StatKs2StepUpdBef=СС_StatKs2Clear, ...Bas=CC_StatKs2Step2, ...Aft=CC_StatKs2Post `(txt:302-305)`.
- Передача без КСУ: номер «-», номер акта; 180^B «Выбыло всего» определяется по «-» `(txt:307-311)`.

### Проверка фонда
