## 1. RIGHT — типовые права доступа к полным текстам

**PAR** (`RIGHT.par`): все 11 путей → `.\DATAI\RIGHT\`.

**Назначение.** Хранит **типовые наборы правил** доступа к полным текстам. Каждая запись = один шаблон прав, идентифицируемый по `v1` (идентификатор записи). Шаблон привязывается к ресурсу через поле `955^B` в БД-источнике (см. алгоритм `right_ft.pft`).

### FST (`RIGHT\RIGHT.fst`)

| Поле | Префикс | Смысл |
|------|---------|-------|
| v1 | `I=` | Идентификатор записи (шаблона прав) |
| v2, v3 | `II=` | Связка «общий период (v2) + правило доступа (v3)» |

Дополнительно `RIGHT\TEXT.fst` (общий для ПТ-БД) индексирует поле 952: `TXT=` (v952^b), `TXT1=` (первая страница `__0001.PDF`), `TXT2=` (v952^c).

### Рабочий лист ввода (`RIGHT\right.ws`)

| Поле | Назв. | Тип/подформа |
|------|-------|--------------|
| 1 | Идентификатор записи | текст |
| 2 | Общий период действия права доступа | подформа `2.wss` (даты ^D начало / ^E конец, ГГГГММДД) |
| 3 | Право доступа (повторяющееся) | подформа `3.wss` (`3_wss`) |
| 4 | Описание/Название | текст |

### Структура повторяющегося поля 3 «Право доступа» (`RIGHT\3.wss`)

| Подполе | Смысл | Справочник |
|---------|-------|------------|
| ^A | Элемент доступа (категория правила) | `3A.mnu` |
| ^B | Значение элемента доступа | формат `@3B` (`3B.pft`) |
| ^C | Значение права доступа | `3C.mnu` |
| ^F | Количественное ограничение (только число; одно повторение) | — |
| ^G | Единицы ограничения | `3G.mnu` |
| ^D | Начальная дата периода доступа (ГГГГММДД) | — |
| ^E | Конечная дата периода доступа (ГГГГММДД) | — |

**Категории элемента доступа** (`RIGHT\3A.mnu`):

| Код ^A | Категория | Источник значения ^B (`3B.pft`) |
|--------|-----------|----------------------------------|
| 01 | Идентификатор читателя | поиск в БД `RDR` по `RI=`, поле v30 |
| 02 | Категория читателя | меню `50.mnu` |
| 03 | IP-адрес клиента | — |
| 04 | Доменное имя клиента | — |
| 05 | Факультет | `fak.mnu` |
| 06 | Семестр | — |
| 07 | Специальность | `spec.mnu` |

**Значения права доступа** (`RIGHT\3C.mnu`): `0` Запрет · `1` Постраничный просмотр · `2` Скачивание/Копирование.
**Единицы ограничения** (`RIGHT\3G.mnu`): (пусто) Страницы · `1` Проценты.

### INI (`RIGHT\RIGHT.ini`)
Один поисковый элемент: `ItemName0=Идентификатор записи`, `ItemPref0=I=`. `ComplSearch=1`, `MinLKeyWord=4`.

### Алгоритм применения прав — `right_ft.pft` / `right_ft_G.pft`

Эталон: `C:\IRBIS64\Datai\Deposit\right_ft.pft` (наследуется в БД ПТ: `IBIS`, `SK`, `VKR` — там лежат `right_ft.pft` и расширенный `right_ft_G.pft`).

Алгоритм (по `Deposit\right_ft.pft`):
1. Если у записи-ресурса поле `955^B` пусто → результат `'2'` (полный доступ, скачивание).
2. Иначе подгружается запись из БД `RIGHT` по `!I=<955^B>!` форматом `@RIGHT0` (`right0.pft`): в группу `g2` грузится поле v2 (общий период), в `g3` — все повторения v3 (правила).
3. Общий период: `g4 = g2^d` (или `00000101`), `g5 = g2^e` (или `99990101`).
4. Если правил нет (`g3=''`): доступ `'2'` если текущая дата `&uf('3')` в `[g4..g5]`, иначе `'0'`.
5. Если правила есть — по каждому повторению `g3` проверяется тип `g3^a`:
   - `01` идентификатор читателя — сверка читателя через `DRDR,!RI=<ID>!,@RIGHT_RID`;
   - `02` категория — `@RIGHT_RKAT`;
   - `05` факультет — `@RIGHT_RFAK`; `06` семестр — `@RIGHT_RSEM`; `07` специальность — `@RIGHT_RSPC`;
   - `03` IP — сравнение по маске `&uf(':',g3^b,'#',<IP>)`;
   - `04` доменное имя — точное сравнение.
   - Проверяется период правила (`g3^d..g3^e`, по умолчанию даты общего периода); при совпадении выдаётся значение права `g3^C` (с ограничением `^F`/`^G`).
6. Итог по приоритету `g10`: содержит `0`→`0`, `1`→`1`, `2`→`2` (берётся максимальный разрешённый уровень).

Модельные поля, на которые опирается алгоритм: `1002` (идентификатор читателя), `1100` (IP-адрес), `1110` (доменное имя). В `right_ft_G.pft` те же данные читаются из групп `g30/g31/g32` и возвращается **гранулированный** результат по каждой категории (W51..W57) с количественными ограничениями (`g3^F`, `%` при `g3^G`).

## 2. LICH — личные данные при работе с полными текстами

> **ВНИМАНИЕ:** БД содержит персональные данные читателей. Ниже — **только структура** (имена/типы полей и префиксы индексирования). Значения не извлекались.

**PAR** (`LICH.par`): пути 1–10 → `.\datai\lich\`; путь **11** → `.\DATAI\lich\texts\` (хранилище полных текстов/закладок).

**Назначение.** Фиксирует персональную активность читателя по полным текстам: закладки, скачанные страницы, личные оценки (рейтинг), счётчики обращений. Связь: читатель (`RDR`) ↔ ресурс электронной библиотеки (БД ЭБ + шифр).

### Рабочий лист (`LICH\lich.ws`) — структура полей

| Поле | Имя | Тип |
|------|-----|-----|
| 1 | Идентификатор читателя | текст |
| 30 | Идентификатор сессии (только J-ИРБИС 2.0) | текст |
| 2 | Идентификатор полного текста | текст |
| 3 | Закладки (номера страниц) | повторяющееся |
| 4 | Кол-во скачанных страниц | число |
| 5 | Имя БД электронной библиотеки | текст |
| 6 | Шифр документа в БД ЭБ | текст |
| 7 | Личная оценка (рейтинг) полного текста | число |
| 40 | Дата и время обращения к полному тексту | повторяющееся, подформа `40.wss` (^D дата ГГГГММДД, ^1 время ЧЧММСС) |

### FST — 20 префиксов индексирования (`LICH\LICH.fst`)

| # | Префикс | Из поля / правило | Смысл |
|---|---------|-------------------|-------|
| 1 | `I=` | v12: v1/v5/v2 | связка читатель/БД ЭБ/текст |
| 2 | `RI=` | v1 | читатель |
| 3 | `RI=` | v30 | сессия как «читатель» (J-ИРБИС) |
| 4 | `TXT=` | v5/v2 | идентификатор полного текста |
| 5 | `DI=` | v5/v6 | шифр документа в БД ЭБ |
| 6 | `DV=` | v40^D | даты обращений (повторения) |
| 7 | `COL=` | v5/v2 (при наличии v40) | количество обращений |
| 8 | `OI5=`…`OI0=` | v5/v2 по порогам оценки v7 (≥5…≥0) | индекс «оценка не ниже N» |
| 9 | `OR5=`…`OR0=` | v1 по тем же порогам | читатель, поставивший оценку ≥N |
| 10 | `ZKL=` | v1 (при наличии v3) | количество закладок у читателей |
| 11 | `FULL=` | v2 (если скачано ≥ числа страниц в v955^N) | тексты, скачанные ПОЛНОСТЬЮ |

Итого 20 различных префиксов: `I=, RI=, TXT=, DI=, DV=, COL=, ZKL=, FULL=, OI0..OI5 (6), OR0..OR5 (6)`.
Плюс общий `LICH\TEXT.fst`: `TXT=`/`TXT1=`/`TXT2=` (поле 952).

### INI (`LICH\LICH.ini`) — поисковые элементы (7)
`RI=` Читатели · `TXT=` Тексты · `I=` Читатели/Тексты · `DI=` Шифры · `COL=` Количество обращений · `ZKL=` Количество закладок · `FULL=` Тексты, скачанные полностью.

### Связи
- `RI=` → читатель в БД **`RDR`** (по `RI=` / полю v30).
- `v5` (имя БД ЭБ) + `v6` (шифр) → запись в **БД электронной библиотеки** (`IBIS`/`SK`/`VKR` и т.п.).
- Web-постранично-просмотровый формат `LICH\lich_web.pft` строит ссылки на постраничный просмотр ПТ (`<DB>_READER`, команда `referings_img`), используя закладки (поле 3) и обращения.
- Табличные формы для статистики ЭБ: `eBookMax_tab.*`, `eReaderMax_tab.*` (рейтинги/обращения; меню сортировки `eBookNcolon.mnu`: по кол-ву читателей / обращений / средней оценке за период / общей).

**Назначение.** Хранит сохранённые («постоянные») запросы читателей для систем ИРИ (избирательное распространение информации). Одна запись = читатель + его набор постоянных запросов.

### Рабочий лист (`ZAPR\zapr.ws`)

## 5. MORPH — морфологический словарь

**PAR** (`MORPH.par`): все 11 путей → `.\DATAI\MORPH\`.

**Назначение.** Словарь морфологии: основной термин (лемма) и его словоформы. Используется при поиске для морфологического расширения запроса (свёртка словоформ к лемме). Признак включения морфологии — `MORPHOLOGY=1` в INI БД-источника (видно, напр., в `LOGP\LOGP.INI`).

### FST (`MORPH\MORPH.fst`)

| Поле | Префикс | Смысл |
|------|---------|-------|
| v10 | `WORD=` | основной термин (лемма) |
| v11 | `DICT=` | словарь/источник |
| v12 | `LANG=` | язык |
| v20 | `FORM=` | словоформы (повторения) |
| v10 | `K=` | все слова (лемма) — общий словарь |
| v20 | `K=` | все слова (словоформы) — общий словарь |

Ключевой словарь `K=` объединяет лемму и все её словоформы, что и обеспечивает приведение любой формы к группе словоформ при поиске.

### Рабочий лист (`MORPH\morph.ws`)

| Поле | Имя |
|------|-----|
| 10 | Основной термин |
| 20 | Словоформы (повторяющееся) |
| 11 | (источник/словарь) |
| 12 | Язык |

### INI (`MORPH\MORPH.ini`)
Три элемента: `K=` «Все слова» · `WORD=` «Основные» · `FORM=` «Формы».

### Стоп-слова (`MORPH\morph.stw`)
Список служебных слов (предлоги/артикли/союзы рус., англ., нем., франц., итал., сканд. и одиночные буквы), исключаемых из словарей: `И, В, НА, ПО, THE, OF, DER, DIE, LE, LA, …`. Эти термины не индексируются как значимые.

## 6. WORK — рабочая (scratch) БД для табличных форм

**PAR** (`WORK.par`): все 11 путей → `.\datai\work\`.

**Назначение.** Временная (рабочая) БД для построения **табличных и статистических форм** (КСУ, статистика фонда, книгообеспеченность, МБА и пр.). Собственных значимых данных почти не имеет — её ценность в большом наборе шаблонов табличных форм (`.tab`/`.tbu`/`.tbg`/`.hdr`/`.srw`/`.pft`). Базовый формат `WORK\WORK.pft` пуст (1 строка), `brief.pft` = `v1`.

### FST (`WORK\WORK.fst`)

| Поле | Префикс | Смысл |
|------|---------|-------|
| v920 | (без префикса) | вид/идентификатор |
| v1 (^c,^f) | `SPEC=` | спецификация: `<v920>_<v1^f>_<v1^c>` (плюс разложение по `_`) |

Плюс `WORK\TEXT.fst`: `TXT=/TXT1=/TXT2=` (952), `GUID=` (999).

### INI (`WORK\work.ini`)
Один элемент с **пустым** префиксом `ItemName0=Идентификатор`, `ItemPref0=` (поиск без словаря, поскольку БД рабочая).

### Набор табличных форм (примеры файлов в `WORK\`)
- **КСУ / движение фонда:** `KsuFondMove*`, `KsuYearMove*`, `KSURZN`, `KS2RZN`.
- **Статистика КО (книгообеспеченность):** `KoItogCycle*`, `KoStat*`, `KOstatCikl/Cycle`, `KoListBas/BookVO/Electro`, `LicenDisc`.
- **Инвентарные / итоговые:** `ListInvKsu`, `Res_OUTDat`, `TCHKCAT`, `TCHKRDR`, `TSPR1T/2T/OST`, `TABWPW`.
- **МБА и периоды:** `StatMBA`, `StatPeriod`.
- RTF-шаблоны табличной печати (напр. `KsuFondMove.tab` — RTF с альбомной ориентацией и колонками 2..37).

### Связи
WORK — целевая БД для модуля табличных форм АРМ; наполняется временными записями из основных БД (`IBIS`, `RDR`, `CMPL` и т.п.) на время построения отчёта.

## 7. Deposit — депозитарий конфигов/шаблонов
