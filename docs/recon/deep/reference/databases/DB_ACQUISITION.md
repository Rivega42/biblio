## A.5 Модель комплектования (из конфигов)
```
ВХОД (необяз.):  PODB (заказ книг) ──MoveZakKp.gbl──┐
                 POST (подписка)   ──MoveNewKp.gbl──┤
                                                    ▼
1. ЗАЯВКА/ЗАКАЗ: БО книги (920=ZK) + поле 62 (заявка ^9/^6/^1/^W → заказ ^A/^K/^B/^C/^D).
   Мастер заказа (CreateSZ.gbl) → запись 920=SZ (на «организация+номер заказа»; ключ SORZ=, индекс SZ=).
   Организация → запись 920=IZD (поле 81).
2. ПОСТУПЛЕНИЕ: в 62 заполняется ^G ПОЛУЧЕНО + ^U №КСУ.
   SumEkzToSZ.gbl суммирует полученные экз. по всем БО заказа (ключ BORZ=).
   Контроль выполнения: BORNZ=/nonzk=/vszk=/pszk=.
3. РЕГИСТРАЦИЯ ЭКЗЕМПЛЯРОВ: поле 910 (статус ste.mnu, инв.№ ^B=MIN+1, штрих-код, МХР, дата, цена, КСУ, канал, коллекция).
   autoin.gbl → авто-907. Контроль !910.pft: дублетность инв.№/штрих-кода.
4. КСУ ПОСТУПЛЕНИЯ (920=KSU): поле 88 (N MKSU=, акт MNA=, суммы).
   Мастер «Пополнение записи КСУ»: авто-поля 17/18/19, 44–49, 145–158. Индексы KSU=, AS=, NKSUK=, DP=.
5. СПИСАНИЕ/ВЫБЫТИЕ (920=KS2): Мастер списания → поле 888 (MKS2=, причина prsp.mnu),
   проставляет 910^V (№КСУ выбытия) и ^X (кол-во); статус 910^A=6. Поле 800 — акт передачи.
   InsteadLost.gbl — замена утерянных (поле 80, кросс-БД).
6. ИТОГИ ДВИЖЕНИЯ ФОНДА (920=KSI/KS3): поле 88/889 + RTF-бланк KSUN3SH.
7. ПЕРЕНОС В ЭК (поле 66 = флаг): процесс ToCat конвертирует БО CMPL → формат IBIS
   (STN.FST/MNOG.FST/UMARCIW.FST: 922/330→700/701/200/463; 920→J/SPEC/PVK/ASP/PAZK).
   Экземпляры (910) → в ЭК с техн.путём (901) и связью 938 (pkj.gbl). FST-строка 66 ставит VD=DEL на исходник.
```
Служебные `.gbl`: `CreateSZ` (создание SZ), `SumEkzToSZ` (подсчёт поступивших), `autoin` (авто-907/пересчёт), `InsteadLost` (замена утерянных), `Movekp` (синхр. OJK с POST), `StatKsu/Staks2/StatKsuSum` (статистика КСУ), `ClearCmpl/ClearPolzv` (очистка), `pkj/pkj1` (перенос/связь с ЭК), `NEWPDP*/GROUPPDP` (книгообеспеченность).

## A.8 Связи CMPL ↔ другие БД
