# Проводной протокол ИРБИС64 (:6666) — ЭМПИРИЧЕСКИ (Проход Б)

> **Снято с живого сервера** `127.0.0.1:6666` (ИРБИС64+ версии `2022+`, демо-логин MASTER) зондом `tools/irbis-probe/probe.py`. Это закрывает gap §7 из [PROTOCOL_REFERENCE.md](PROTOCOL_REFERENCE.md): байтовый формат, который C-API дока не описывала. Все факты — наблюдаемые в реальном обмене. Дата: Проход Б, 2026-06-20.

## 1. Модель соединения — ОДНО TCP-СОЕДИНЕНИЕ НА ЗАПРОС
Сервер **закрывает соединение после каждого ответа** (наблюдалось: повторное чтение/запись в том же сокете → `ConnectionAborted`). Прикладная **сессия живёт по `client_id`**: клиент один раз регистрируется (`A`), получает/использует `client_id`, и каждый последующий запрос — это **новое TCP-соединение** с тем же `client_id` и инкрементируемым `query_id`. Для бэкенда: пул не «держит сокет», а держит зарегистрированную сессию (client_id) и открывает соединение на каждую команду.

## 2. Кадр ЗАПРОСА
```
<bodyLength>\n           # ASCII-десятичная длина тела в байтах + LF (0x0A)
<command>\n              # код команды (1-2 символа)
<workstation>\n          # тип АРМа (1 символ): 'A','C','R','M','B','K'
<command>\n              # код команды (повтор)
<client_id>\n            # целое, генерит клиент при регистрации
<query_id>\n             # целое, инкремент с каждым запросом
<password>\n             # пароль
<username>\n             # логин
\n \n \n                 # 3 пустые строки (резерв)
<arg1>\n <arg2>\n ...    # аргументы команды
```
- Разделитель строк в запросе — **`\n` (LF, 0x0A)**.
- `bodyLength` = число байт тела (всё после первой строки-длины). Сервер читает длину, затем ровно столько байт.
- **Порядок: пароль, затем логин** (подтверждено рабочим REGISTER).

## 3. Кадр ОТВЕТА
Разделитель строк в ответе — **`\r\n` (CRLF, 0x0D 0x0A = `$0D0A` из доки)**.
```
<command>\r\n            # эхо кода команды
<client_id>\r\n          # эхо client_id
<query_id>\r\n           # эхо query_id
<size>\r\n               # размер остатка ответа (служебный)
<server_version>\r\n     # ТОЛЬКО для REGISTER ('2022+'); для прочих — пусто
\r\n \r\n \r\n \r\n \r\n  # резервные пустые строки
<return_code>\r\n        # ЧИСЛОВОЙ код возврата (0 = успех; <0 = ошибка)
<command-specific data...>
```
> Для команд, где возврат = ДАННЫЕ (см. C-API §6 PROTOCOL_REFERENCE), на месте `<return_code>` стоит само значение: `O` (maxmfn) → число записей+1; `K` (search) → 0, затем число найденных и список.

## 4. Команды (коды, подтверждены)
| Код | Команда | Аргументы (после преамбулы) | Ответ |
|---|---|---|---|
| `A` | REGISTER (логин) | username, password | код 0 + версия `2022+` + **INI-профиль АРМа (ANSI/CP1251)** |
| `B` | UNREGISTER | username | код 0 |
| `N` | NOP (keepalive) | — | код 0 |
| `O` | MAX MFN | dbname | maxmfn+1 (для IBIS наблюдалось **394**) |
| `K` | SEARCH | dbname, `"<выражение>"`, want-format(0), first, max, format | код 0, число найденных, затем строки `mfn#<результат>` |
> Полный набор кодов команд `IC_*`↔буква — добрать аналогично (`C` read, `D` update, `G`/`P` format, `+`-команды admin); метод тот же.

## 5. Числовые коды возврата (ЭМПИРИЧЕСКИ)
| Код | Смысл |
|---|---|

## 6. Кодировки (ПОДТВЕРЖДЕНО)
- **INI-профиль (ответ REGISTER), ресурсы — ANSI/CP1251** (наблюдалось: текст профиля при UTF-8-декоде даёт «мояибаку» → это CP1251). Совпадает с доками (`PROTOCOL_REFERENCE §1.2`).
- **Библиографические данные/термины — UTF-8** (search/read).

## 7. Что вернул REGISTER (важно)
Сервер на вход отдаёт клиенту **его INI-профиль АРМа** — это в точности наш `recon/deep/reference/arms/ARM_ADMIN.md` / `IRBISA.INI` (`[CONTEXT] DBN=MBA`, `[FULLTEXT]`, `[MAIN]` со всеми `*Item=1`-правами, `DataPath=.\DATAI\`, `EtalonDBN=IBIS`, `WEBIRBIS=...irbis64r_IMGplus17`). → **подтверждает**, что профиль/права АРМа сервер раздаёт при логине; наш разбор INI верен.

## 8. Подтверждение состояния демо-БД
Сервер на `localhost:6666` = инсталляция `C:\IRBIS64` (DATAI). IBIS: **maxmfn 394**, по `I=$` найдено **390** (MFN 24, 110 — логически удалены, пропущены в выдаче).

## 9. Реализация для бэкенда (IrbisClient-адаптер)
1. Открыть TCP, отправить кадр §2 (тело UTF-8, `\n`; для ANSI-полей — CP1251), прочитать **до закрытия сокета** сервером.
2. Разобрать ответ §3: эхо команды/ids → `return_code` → данные.
3. Держать `client_id` (один REGISTER), инкрементить `query_id`, на каждую команду — новое соединение.
4. ANSI-декод для INI/ресурсов, UTF-8 — для данных.
5. Беречь лимит сессий (демо — 3): UNREGISTER в `finally`, иначе `-3337` копится.

## 10. CRUD подтверждён на живом сервере (read / render / write / delete)
Снято демо-скриптами `tools/irbis-probe/demo_read.py`, `demo_write.py` через адаптер `irbis_client.py`.

### 10.1 READ ('C') — `db, mfn, 0`
Возврат `0`, затем **клиентское представление записи** (строки `\r\n`, data[11:]):
```
<mfn>#<status>                # напр. 2#  (status пуст/битовая маска)
0#<version>                   # 0#1
2147483647#{GUID}             # системное поле-GUID (тег = MaxInt)
<tag>#<value>                 # напр. 700#^AКолисниченко^GДенис Николаевич
...
```
**Разделитель подполей — `^`** (подтверждено: `^A`, `^G`, `^D`…). Закрывает вопрос о подполях.
Коды чтения: `0` ок; **`-603` запись логически удалена** (REC_DELETE); **`-140` — нет такого MFN**.

### 10.2 FORMAT ('G') — `db, format, count=1, mfn` → серверный рендер PFT
- `@brief` → готовое БО: «Колисниченко, Денис Николаевич. Android для пользователя… 2013. — 254 с.»
- `@` (optimized) → RTF (`{\tab …\par \b 004.4\par …}`).
> Подтверждает стратегию ADR-003: **рендер делает сервер своим PFT**.

### 10.3 UPDATE ('D') — `db, lock(0), actualize(1), <record>` → запись/правка/удаление
`<record>` = строки записи, **склеенные разделителем `\x1F\x1E`** (US+RS):
```
0#0 \x1F\x1E 0#0 \x1F\x1E 920#SPEC \x1F\x1E 200#^A... \x1F\x1E 907#^C...
```
- `mfn=0` → **создание** (сервер присваивает MFN и GUID, version=1); ответ возвращает `<mfn>#<status>`.
- Успех → возврат = **maxMFN** (положительный).
- **Логическое удаление** — тот же UPDATE со `status=1` в строке `mfn#status`; после — READ даёт `-603`.
- Проверено round-trip: create → read-back (поля совпали байт-в-байт) → logical delete → read `-603`. Велось в служебной БД `WORK`, каталог не затронут.

### 10.4 TERMS ('H') — словарь/автодополнение — `db, start_term, count`
Возврат `-202`/`-203`/`-204` (точного термина нет — это норма) или `0`; затем строки **`<постингов>#<термин>`**:
```
17#K=ANDROID
2#A=КОЛИСНИЧЕНКО, ДЕНИС НИКОЛАЕВИЧ
178#V=05
```
Термин включает префикс индекса (`K=`,`A=`,`V=`…). Идёт от `start_term` по алфавиту, `count` штук. → основа эндпоинта `terms` (автодополнение + число найдётся заранее). Парсер должен принимать коды `0/-202/-203/-204` как успех. `P`(prev) — обратный обход (добрать аналогично).

### 10.5 FILE/RESOURCE ('L') — чтение текстового ресурса — `<pathcode>.<db>.<file>`
Контент идёт после 10-строчной преамбулы; внутренние переводы строк — **`\x1F\x1E`**; кодировка **CP1251**. `pathcode`: `2`=каталог БД (`2.IBIS.brief.pft` → весь PFT), `3`=меню/словари БД (`3.IBIS.kv.mnu` → пары меню). Пустой ответ (~32 байта) = файл не найден. → меню-словари для выпадающих списков и серверные ресурсы. Бинарные ресурсы (картинки/PDF) — отдельная команда; для P0 встроенная картинка берётся из поля `953^B` (URL-кодирование `%FF%D8…` = JPEG → URL-decode).
