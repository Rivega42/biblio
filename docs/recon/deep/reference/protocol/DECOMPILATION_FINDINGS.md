# DECOMPILATION_FINDINGS — анализ декомпиляции движка ИРБИS64.dll

> **Источник:** Ghidra-псевдокод тел экспортируемых функций `IRBIS64.dll` (975 КБ, 32-бит Delphi), полученный headless-декомпилятором. Сырые артефакты лежат рядом: [`IRBIS64_engine_decompiled.c`](IRBIS64_engine_decompiled.c) (7 функций ядра) и [`IRBIS64_UNIFOR_decompiled.c`](IRBIS64_UNIFOR_decompiled.c) (диспетчер `UNIFOR`). Это **прозаическая интерпретация** псевдо-C — алгоритмы, а не таблицы экспортов (их даёт [DLL_API_EXTRACTED](DLL_API_EXTRACTED.md)). Подтверждает/уточняет модель поиска и формат-движка, заложенную в [WIRE_PROTOCOL](WIRE_PROTOCOL.md), [PROTOCOL_REFERENCE](PROTOCOL_REFERENCE.md) и наши реализации `irbis-web/backend/access/pft.py` + `flk.py`. Дата: 2026-06-21.

## §0. Метод и оговорки

**Инструмент.** Ghidra 12.1.2 в headless-режиме (JDK 21), два скрипта в этой же папке:
- [`DecompileNamed.java`](DecompileNamed.java) — декомпиляция именованных по таблице экспортов функций ядра.
- [`DecompileUnifor.java`](DecompileUnifor.java) — отдельный проход по `UNIFOR @ 4009fbf0` с поднятым тайм-аутом декомпилятора (600 с): в первом (общем) прогоне эта функция падала с `Exception while decompiling 4009fbf0: process: timeout` — она огромна, отсюда отдельный скрипт. Эта же запись-заглушка осталась в `IRBIS64_engine_decompiled.c:213-214`, а полное тело — в `IRBIS64_UNIFOR_decompiled.c` (5952 строки).

**Состав разобранного (7 функций ядра + UNIFOR):**

**Оговорка о достоверности.** Это псевдо-C Ghidra, а не байт-точный исходник. Имена `FUN_xxxxxxxx` — это адреса невосстановленных внутренних функций, `undefined`/`undefined4` — типы, которые декомпилятор не вывел, `local_*`/`extraout_*` — синтетические имена стека/регистров. Числовые смещения (`+0x584`, `+0x806`, `-0x3e9` …) и константы — наблюдаемые, но их **семантика — это интерпретация**. Все выводы ниже привязаны к `@ адресу` и диапазону строк сырого `.c`, чтобы читатель мог сверить.

## §1. Поиск и инвертированный файл

Четыре функции складываются в полный путь «термин → постинги» и обратный путь «термин → запись в индекс». Это прямое подтверждение модели словаря/инвертированного файла из `PROTOCOL_REFERENCE §префиксы` и команды `IC_getposting(APost; AType)`.

### 1.1 `Irbisfind` — нормализация и запуск (engine `334-349`)

Тончайшая обёртка. Делает ровно три вещи:
1. **Правый трим** термина: `FUN_4000b140` даёт длину строки, затем цикл срезает с хвоста все байты `< 0x21` (пробелы и управляющие символы), записывая на их место `\0` (engine `342-345`). То есть конечные пробелы/CR/LF в поисковом термине отбрасываются ещё до лукапа.
2. Подготовка: `FUN_400bfc0c(param_1, param_2)` — общая «pre-search» подготовка контекста (та же функция вызывается из `InsertTerm`, engine `1085`, — общий код для поиска и вставки).
3. Запуск: `FUN_400c94a0(param_1, param_2, …1)` — исполнитель поиска с флагом `1` (третий аргумент собирается через `CONCAT31(…, 1)`). Возвращает его результат.

Вывод: «термин нормализуется (трим), затем prepare + execute» — подтверждено. Тот же `FUN_400c94a0` (с флагом `0`) — это собственно лукап термина в словаре; он же вызывается в `IrbisFindPosting` (`FUN_400c9a80`, соседний адрес) и в `InsertTerm` (engine `1111`).

### 1.2 `IrbisFindPosting` — обход цепочки блоков постингов (engine `233-328`)

Это ключевая функция: показывает **физический формат инвертированного файла** (`.ifp` + узлы `.l01`/`.n01`).

1. **Нормализация термина** в локальный буфер `local_144[256]` через `FUN_4000b1a8` (engine `274`), затем лукап в словаре `FUN_400c9a80(local_8, local_144, 0)` → `local_18` (engine `275`). `local_18 == 0` означает «термин найден».
2. **Выбор активного контекста инверт.-файла.** Поле `+0x5b1` — индекс «активного» сегмента: если он в диапазоне `1..10`, база контекста берётся из массива указателей по `+0x584 + idx*4`; иначе — базовый `+0x584`, а `+0x5b1` сбрасывается в `0` (engine `277-283`). То есть `+0x584` — таблица из ≤10 хэндлов открытых инверт.-файлов (мульти-сегментный словарь), `+0x5b1` — текущий.
3. **Чтение control/header-блока.** По смещению `+0x1048` лежит дескриптор, из него считается позиция (`FUN_40028978` — индексатор массива), затем сдвоенный 32-бит указатель блока собирается из `+0x81a`/`+0x81e` функцией `FUN_40026c7c` (склейка hi/lo в 64-бит файловое смещение) → `local_2c`/`local_28` (engine `285-292`).
4. **Seek + read.** `FUN_40026be0(HANDLE, …, offset)` — это `SetFilePointer` (seek по хэндлу `*(short*)(local_44+4)`), `FUN_40026c20(HANDLE, buf, 0x14)` — `ReadFile` 0x14=20 байт заголовка блока в `+0x100c`, `FUN_40023da0(buf, 8)` — пост-обработка (вероятно `ntohl` полей: блоки хранят bigend-числа, ср. явные `htonl/ntohl` в `InsertTerm`). Так читается **заголовок блока постингов** (engine `293-295`).
5. **Счётчик и быстрый путь.** `+0x1014` — число постингов в блоке (`< 1` → пустой, возврат, engine `296-299`). Поле `+0x1010` — **ссылка на следующий блок**; магическое значение `-0x3e9` (= **-1001**) означает «весь список постингов умещается в этом узле» — тогда вызывается «in-node» считыватель `FUN_400caf48` (engine `302-304`). Это fast-path.
6. **Обычный обход цепочки блоков** (engine `305-318`): пока `next != -1` и очередной считыватель `FUN_400cac04` возвращает `< 0` (постинг ещё не найден), пересчитывается смещение следующего блока из `+0x1010`/`+0x100c` (`FUN_40026c7c`), снова seek (`FUN_40026be0`) + read 0x14 (`FUN_40026c20`) + decode (`FUN_40023da0`). Классический обход **связного списка блоков постингов** на диске.
7. **Дочитывание.** `FUN_40024bb4` сверяет с `+0x1020`; если осталось — `Irbisnxtpost` подтягивает следующую порцию (engine `321-324`).

Итог по структуре блока постингов (смещения от базы контекста `local_44`):

| Смещение | Смысл (интерпретация) |
|---|---|
| `+0x584` (`+0x584+idx*4`) | таблица хэндлов сегментов словаря (≤10), `+0x5b1` — активный индекс |
| `+0x4` | файловый HANDLE текущего инверт.-файла (16-бит) |
| `+0x1048` | дескриптор/индекс control-блока |
| `+0x100c` | буфер заголовка текущего блока (20 байт) |
| `+0x1010` | ссылка на следующий блок; `-1001` = «весь постинг в узле» (fast-path) |
| `+0x1014` | число постингов в блоке |
| `+0x1020` | курсор/остаток при дочитывании |
| `+0x81a` / `+0x81e` | hi/lo стартового файлового смещения цепочки |

Это уточняет нашу модель: словарь = **B-tree-подобные узлы** (`.l01`/`.n01`), листья которых указывают в `.ifp` на **цепочку блоков по 20-байт заголовку**; чтение постингов — последовательный seek/read по этой цепочке, с оптимизацией «короткий постинг лежит прямо в узле» (`-1001`).

### 1.3 `IrbisSearch_Range` — диапазон/скан и булева комбинация (engine `355-893`)

~540 строк. Не лукап одного термина, а **построитель множества MFN** из набора термов с булевой логикой. Ключевое:

- Работает через **vtable-объекты** (косвенные вызовы `(**(code**)(*piVar+off))()`): `+0x14` — count, `+0xc` — get-by-index, `+0x18`/`+0x24` — доступ к под-спискам, `+0x38`/`+0x44`/`+0x50` — мутаторы. Это C++/Delphi-объекты «список термов» и «список постингов», а не плоские массивы (engine `425, 436-440, 457`).
- **Режим** задаётся старшим байтом `uVar14>>0x18` (значения `0..6`, инициализируется `6`, engine `427-429`). Разные ветки `cVar15 == 0/1/2` (engine `474-507`) — это разные операции над множествами: безусловное объединение `FUN_400bd0f0`, пересечение `FUN_400bd1bc`, вычитание/разность `FUN_400bd264`, и слияние результата `FUN_400bd2f0`. `piVar4` накапливает итог, `local_40` — рабочее множество; ранний выход `FUN_4005af28(...,0)` при пустом результате пересечения (engine `488-491`).
- Термы вида `#...` (engine `460, 501`) — особый случай (маска/диапазон): `*local_ac == '#'` ветвит логику. Это диапазонные/усечённые термы словаря.
- Глубже (режимы `>2`, engine `539-561`) — многоуровневая вложенность списков (`local_44/48/50/54/58/5c`), то есть **скобочные/групповые выражения** поиска с раскрытием по позициям/повторениям полей (вложенные циклы по `FUN_40028978`, engine `686-817`).
- Для каждого терма, дающего записи, формат-ссылка строится из `+0x7da` записи (engine `587, 823`) и форматируется через `FUN_400b9dd8` / `FUN_400bc388` / `FUN_400bc1bc` — то есть скан умеет **форматировать ключи на лету** (это путь `IC_searchscan`/`postingformat`).

Вывод: `IrbisSearch_Range` — это исполнитель **полного поискового выражения** (объединение/пересечение/разность множеств постингов + диапазоны `#` + групповые/позиционные уточнения), сидящий поверх `IrbisFindPosting`. Подтверждает, что булева логика поиска ИРБИС реализована на уровне движка как операции над множествами MFN, а не текстовым переписыванием запроса.

### 1.4 `InsertTerm` — запись термина в индекс (engine `899-1798`)

~900 строк — обратная операция: добавить термин в инвертированный файл (используется при актуализации записи). Симметрична чтению:

- Тот же выбор активного сегмента по `+0x584`/`+0x5b1` (engine `1086-1091`), тот же 20-байт заголовок, тот же fast-path `-0x3e9` (engine `1273`).
- **Явные `htonl`/`ntohl`** при записи счётчиков блоков (engine `1255-1259, 1530-1534`) — прямое доказательство, что числовые поля инверт.-файла хранятся в **big-endian** (это и объясняет `FUN_40023da0`/`FUN_40024450` как decode/encode-обёртки в путях чтения).
- Управление ёмкостью узла: поля `+0x80a` (число занятых слотов), `+0x80c` (текущий счётчик), `+0x80e` (вместимость) — при переполнении узел **расщепляется** (split): половина (`local_1e0 = (count+1)>>1`) уезжает в новый блок, заголовок старого правится, новый блок дописывается в конец файла (`GetFileSize` → новое смещение, engine `1173-1178, 1395-1442`). Это классический **B-tree node split** при вставке.
- Готовится и записывается слот фиксированной длины **0x20 = 32 байта** на постинг (multiple `*0x20`, `<< 5`), узловые записи — **0x28 = 40 байт** (`*0x28`, engine `1450-1499`). Это размеры записей `.ifp`/узла.
- Ветки `-0xcb`/`-0xcc` (engine `1137-1148`) — коды «термин-сосед» (вставка перед/после существующего ключа); определяют стартовый номер постинга.

Вывод: `InsertTerm` подтверждает, что индекс — это **самобалансирующееся дерево с расщеплением узлов**, числа big-endian, постинг = 32-байт слот. Наша модель «термин входит в словарь, постинги дописываются цепочкой» уточнена до конкретных размеров и механики split.

## §2. Формат-движок (A1): `Irbis_Format` + `Irbis_InitPFT` + `UNIFOR`

### 2.1 `Irbis_Format` — конвейер форматирования (engine `7-209`)

Подтверждает (и согласуется с disasm-капстоном, коммит `41bd905`) трёхтактный конвейер:

1. **Guard диапазона MFN** (engine `69-72`): если запрошенный номер ≥ `*(param_1+0x80a)` (верхняя граница), функция возвращает код **`-0x66` = -102** (`local_218`). Это та самая ошибка «MFN вне диапазона». Дополнительно `Irbismaxmfn` даёт верхнюю границу для зажима (engine `80-81`).
2. **Инициализация контекста формата** (engine `74-150`): сброс глобальных флагов (`DAT_400e31f0/ec/f4` — это флаги пост-обработки `&unifor('!')`/RTF/спец, ниже в `UNIFOR` те же DAT правятся), заведение пунктуационных таблиц (строка `"!2; 2, 2, 2…"`, алфавит `"\nABCDEFGHI<"`), аллокация рабочего буфера `FUN_4000a75c(100)` (engine `136`), установка SEH-кадра.
3. **Парс-цикл и сборка** (engine `144-150`): `FUN_400b02c4` + `FUN_400b0830` — инициализация парсера PFT, затем `FUN_400b7470(…, *(…+0x2c)+4+mfn*0x43, …)` — основной форматтер по дескриптору формата (шаг записи в таблице форматов = `0x43` = 67 байт), `FUN_400b1388` — финализация. Результат пишется в буфер записи `+0x80e` (engine `151`).
4. **Пост-обработка по флагам** (engine `160-200`): найденный маркер `"&&&&&"` (`FUN_4000b37c`) выделяет хвост; при выставленных флагах вызываются `FUN_40088428/588/790` (три варианта пост-форматирования — соответствуют `&unifor('!')`, удалению RTF и т. п.). Ветка `local_14 == 0x39` (engine `177-200`) — обработка «формата 99 / ошибки формата» с посимвольной выгрузкой через `FUN_400b0108`.

Это прямо ложится на наш `pft.py`: guard диапазона ⇄ `ctx['mfn']`-проверка, «формат 99» ⇄ наша «Format error 99»-паритетная деградация (см. docstring `pft.py`), пост-обработка `!`/RTF ⇄ наши реализованные коды `!`/`+F`.

### 2.2 `Irbis_InitPFT` — тонкая обёртка (engine `219-227`)

Ровно `return FUN_400af858(param_1, param_2)` — проброс во внутренний инициализатор PFT-контекста (загрузка `.pft`-ресурса по имени `param_2`). Подтверждает, что инициализация и исполнение формата разнесены (наш `pft.eval` против предзагрузки шаблона).

### 2.3 `UNIFOR` — 11-параметрический диспетчер форматных функций (UNIFOR `10-…`)

Сигнатура (UNIFOR `10-12`):

```c
void UNIFOR(int *param_1,        // контекст БД / запись
            undefined *param_2,   // MFN / дескриптор записи
            undefined4 param_3,
            uint *param_4,        // выходной буфер (результат)
            byte *param_5,        // СТРОКА-КОМАНДА UNIFOR (диспетчируемая)
            undefined1 *param_6, undefined *param_7,
            uint *param_8, undefined *param_9,
            char *param_10, int param_11);
```

**Что это даёт о вызывной конвенции.** UNIFOR — не «функция с кодом-числом», а **строковый интерпретатор**: вся семантика выбирается разбором текста `param_5`. Перед диспетчеризацией строка тримится справа от CR/LF (UNIFOR `787-790` — тот же приём, что и в `Irbisfind`), затем первый символ приводится к верхнему регистру `CharUpperBuffA(param_5,1)` и идёт `switch(*param_5 - 0x21)` (UNIFOR `795-797`). `0x21 = '!'` — то есть индекс ветки = `код_первого_символа − '!'`. Видимые ветки верхнего уровня соответствуют печатным ASCII: `case 0` (`!`) ставит флаг пост-обработки `DAT_400e31f0` (тот самый флаг, который читает `Irbis_Format §2.1!`), `case 2` (`#`), `case 3` (`$`), `case 10` (`+`) — большой под-switch по второму символу (`param_5[1]`), внутри него ещё под-switch по третьему (`param_5[2]`): `+0`, `+1`, `+@`, `+A`…`+C`, и т. д. (UNIFOR `847-1002`). Цифровые группы `2`,`3` (`case 0x32`/`0x33`) — дата/строковые операции.

Следствия:
- **Однобуквенный + вложенный разбор** ⇒ наша таблица `UNIFOR` в `pft.py` (диспетч по первому символу аргумента: `C`,`3`,`A/P`,`9`,`Q`,`X`,`E`,`!`,`+F`,`6`) — структурно верна; движок делает то же самое, только покрывает весь алфавит ветвей.
- **Результат пишется в `param_4`** через `FUN_40087f50/fc0` (append-в-буфер) — UNIFOR не возвращает строку, а аппендит в выходной буфер форматтера. Это объясняет, почему в PFT `&unifor(...)` встраивается inline.
- Внутри ветки `+*` (UNIFOR `818-846`) вызываются `Irbismfn` + `IrbisReadGuid` + `CreateMutexA/WaitForSingleObject` — т. е. некоторые UNIFOR-коды лезут в **серверное состояние** (GUID записи, межпроцессные мьютексы блокировок). Эти коды у нас деградированы (см. `pft.py` «degraded to empty string»), и теперь видно почему: они требуют live-сервера, а не чистой функции от записи.

**Карта на `pft.py`.** Реализованные нами коды попадают в видимые ветки диспетчера: `C` (контроль ISBN/ISSN — переиспользуется в `flk.py` как `isbn_checksum`/`issn_checksum`, см. `flk.py:127,149,357`), `3` (дата/время — ветка `case 0x33`, UNIFOR `1319-…`), `A`/`P` (N-е вхождение поля). Коды, лезущие в GUID/мьютексы/глобальное состояние сервера, корректно оставлены деградированными.

## §3. UMARCI — примитив импорта UNIMARC (engine `1804-1864`)

Короткая функция. Готовит входную строку (`FUN_40004140` ← `param_5`, нормализация длины), приводит первый символ к верхнему регистру `CharUpperBuffA(param_5,1)`, затем `switch(*param_5 - 0x30)` (`0x30 = '0'`) — то есть диспетч по **цифре `0..5`** (engine `1836-1854`): шесть вариантов импорта (`FUN_400ae284/45c/6d0/87c`, `FUN_400adf34`, `FUN_400aeb4c`) — вероятно подформаты/уровни UNIMARC-конверсии. Результат собирается `FUN_400043cc` и аппендится в `param_4` (тот же `FUN_40087f50`, что в UNIFOR). Это низкоуровневый строковый примитив конверсии UNIMARC, вызываемый из форматов; детальной семантики шести веток псевдокод не раскрывает (нужен прогон по конкретным внутренним функциям).

## §4. Что подтвердилось / уточнилось / открытые вопросы

**Подтвердилось:**
- Словарь/инвертированный файл = дерево узлов (`.l01`/`.n01`) → цепочка блоков постингов (`.ifp`); чтение = последовательный seek/read по 20-байт заголовкам блоков (`IrbisFindPosting`). Это закрывает «как физически устроен `IC_getposting`».
- Числа инверт.-файла — **big-endian** (явные `htonl`/`ntohl` в `InsertTerm`). Наш парсер постингов должен читать поля как BE.
- Булев поиск — операции над множествами MFN на уровне движка (`IrbisSearch_Range`: объединение/пересечение/разность), а не переписывание текста запроса. Наша модель поиска по префиксам совместима.
- Формат-движок: guard MFN → `-102`, init-контекст, парс-цикл, пост-обработка по флагам (`Irbis_Format`) — ровно конвейер `pft.py`. «Формат 99» — реальная ветка движка (`local_14==0x39`), наша паритетная деградация оправдана.
- `UNIFOR` диспетчирует по **первому символу строки-команды** (`*param_5-'!'`) с вложенными под-switch — структура нашей таблицы UNIFOR верна.
- Код `C` (контроль ISBN/ISSN) — реальная ветка UNIFOR, и `flk.py` правомерно переиспользует её как предикат ФЛК.

**Уточнилось (новые конкретные числа для реализации):**
- Постинг = слот **32 байта** (`0x20`); узловая запись = **40 байт** (`0x28`); заголовок блока = **20 байт** (`0x14`); шаг таблицы форматов = **67 байт** (`0x43`).
- Магия `-1001` (`-0x3e9`) = «постинг целиком в узле» — fast-path и при чтении, и при вставке.
- ≤10 сегментов словаря, активный — поле `+0x5b1`.
- Узел расщепляется пополам (`(count+1)>>1`) при переполнении вместимости `+0x80e` — B-tree split.

**Открытые вопросы (новый recon):**
- Точная семантика шести веток `UMARCI` (`0..5`) — требует прогона внутренних `FUN_400ae*`.
- Полная карта верхнего switch `UNIFOR` (восстановлены лишь видимые ветки `!`,`#`,`$`,`+`,`2`,`3`); ветки `D/J/T/$/l`-lookup, которые мы деградировали, стоит сопоставить с конкретными `case`.
- Внутренние `FUN_400caf48` (in-node reader) vs `FUN_400cac04` (chain reader) — точный формат записи постинга (поля MFN/tag/occ/pos из `AType` команды `IC_getposting`).
- `Irbismaxmfn`/`+0x80a` как граница в `Irbis_Format` — сверить с `IC_maxmfn` на проводе.
