# Формально-логический контроль (ФЛК) полей и записи — БД IBIS

> Справочник правил валидации Каталогизатора веб-ИРБИС64. Источники: реальные конфиги БД IBIS, сконвертированные в `C:\IRBIS64\_recon\_work\cfg_txt\IBIS\` (запуск `_work\conv_db.ps1 -DB IBIS`, 763 файла). Язык ФЛК — PFT (см. `PFT_LANGUAGE.md`, разд. 13). Только из файлов. Без ПДн.
>
> **Семантика результата ФЛК** (первый символ): `0` — контроль пройден; `1` — ошибка **непреодолимая** (сохранение блокируется до устранения); `2` — ошибка **преодолимая** (можно сохранить, не исправляя). Текст со 2-го символа выдаётся как сообщение. (`PFT_LANGUAGE.md` разд.13; подтверждается синтаксисом `/'1 ...'`, `/'2 ...'`, `/'0'` во всех файлах.)
>
> **Как подключается ФЛК.** Общий ФЛК записи вычисляется при сохранении — формат `dbnflc.pft` (и его вариант `dbnflck.pft`). Пофайловый (полевой) ФЛК `!NNN.pft` подключается в рабочем листе `*.ws`: блок поля состоит из 10 строк, строка 7 содержит ссылку `@!NNN` на формат контроля поля, строка 6 — имя файла расширенного ввода `*.wss` (подтверждается `zk_brief.ws`, `MN_brief.ws`: строки `@!702`,`@!910`,`@!900`,`@!10`,`@!461` и т.п. в позиции 7 блока).

## 3. Расширенный ввод — `@!NNN` и файлы `.wss`

**`@!NNN` в рабочем листе** (строка 7 блока поля) — это ссылка на формат полевого ФЛК `!NNN.pft` (тот же файл, разд. 2). Префикс `@` — обозначение «формат из файла». Т.е. контроль поля при вводе/выходе из поля и есть `!NNN.pft`.

**Расширенный ввод подполей** задаётся файлом `*.wss` (строка 6 блока поля). Структура `.wss`: для каждого подполя — блок из ~10 строк (буква подполя, наименование, флаги, тип/число повторений, и в строке-источнике значений — либо `.mnu`-справочник, либо конструкция переноса данных через словарь `,<префикс>=,@!<формат>`). Примеры (`461m.wss`): `,T=,@!tmovzh` (перенос названия по префиксу `T=`), `,A=,@!amovf` (перенос сведений об ответственности по `A=`), подполя со справочниками `200bm.mnu`, `200em.mnu`. Подсказки оператору даются текстом в блоке подполя (напр. «Перед вторым и последующими значениями ставить " : "», «Если не заполнено, формируется автоматически…»).

Таким образом расширенный ввод (`@!tmov*`, `@!amov*`, `@!mmov*`, `@!lmov*` и т.п.) отличается от полевого ФЛК назначением: это **форматы переноса/автоформатирования данных через словарь** при вводе подполя (термин словаря → значение подполя; в PFT — форматы с переменными метками `***`, см. `PFT_LANGUAGE.md` разд.13), а не проверка корректности. Сами файлы `@!*mov*` в каталоге БД IBIS не материализованы как отдельные `.pft` (резолвятся по общему пути конфигурации) — `TODO(recon #FLC-10)`.

## 4. Сводка

- 210^d|^h, 700^a, 701^a, 702^a, 710^a (1) — структурные.

- Сведения об экземплярах: 910^a статус, инв.№/штрих-код, число экз. — `!910`/`dbnflc` (1/2).

**Справочная проверка по словарю `.mnu`:** 101/911 (`jz.mnu`), 102 (`str.mnu`), 900^b (`vd.mnu`), 920 (`920.mnu`).

**Контроль ISBN/ISSN** через `&unifor('C'…)`: 10^a (ISBN), 11^a (ISSN), 225^x (ISSN серии), 461^i (ISBN)/461^j (ISSN).

**Контроль уникальности (дублетность):**
- Шифр 903 — словарь `I=` (`!903`, `dbnflc`), тип 1.
- Описание (свёртка БО) — словарь `!<ключ>` (`dbnflc`/`dbnflck`), тип 1.
- ISBN 10^a, ISSN 11^a, ид.№ 19 — словарь `B=` (`!10`/`!11`/`!19`), тип 2/1.
- Инв.№ 910^b и штрих-код 910^h — словарь `IN=` (`!910`, `dbnflc`), тип 2; + внутризаписные дубли.
- Выпуски/экз. журнала — ключи `I=903/934/935/936`, `I=933/934/935/936` (`!936`, `dbnflc`), тип 1.

**Непреодолимые (1) vs преодолимые (2):** тип 1 — структура полей, неверные коды по справочнику, дублетные шифр/описание/выпуск, неверный индекс/ISBN-структура, роли-не-число, правила предметизации, правила выбытия `!910W`/`!940W`. Тип 2 — отсутствие необязательных, но желательных данных (ФИО, цена, статус/инв.№ экземпляра при обычной регистрации), дублеты инв.№/штрих-кода при вводе, ошибки контрольной суммы ISBN/ISSN, замечания по импорту 998.
