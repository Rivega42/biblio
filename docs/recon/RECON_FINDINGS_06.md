## 1. АРМ «Комплектатор» (приобретение/учёт поступлений)

**Виды записей** в БД комплектования (CMPL); технология: регистрация поступлений в БД комплектования
или сразу в БД каталога. Начало работы: реквизиты пользователя, справочники.

**Поступление (ядро АРМа):**
- Регистрация поступления: по заказу → создание/выбор записи **КСУ** (книга суммарного учёта) →
  поиск описания в БД каталога → выбор режима ввода/корректировки → **перенос описания в БД
  комплектования** (модуль ввода, ввод элементов БО) → доукомплектование (перенос в ЭК) →
  пополнение записи КСУ → **таблица переноса** → режим «Дублетные описания».
- Многотомники: новый многотомник / очередной том в партии — в БД комплектования и в каталоге;
  заимствование описания тома.
- Оформление утерянных книг; регистрация номера журнала; макс. инвентарный номер; **размножение
  экземпляров**; передача данных в БД электронного каталога.

**Выходные формы:** Инвентарный список · Книга суммарного учёта поступлений (КСУ) · итоговые записи
КСУ / движение фонда · поступления журналов · «похожие» описания в другой БД (DublDB).

> Веб-Комплектатор: ядро (поиск/чтение/запись) + модель КСУ + перенос между БД комплектования↔каталог
> + генерация выходных форм (`IC_print`, находка #2). Базы: `CMPL`, `POST` (подписка), каталог.

## 2. АРМ «Книгообеспеченность» (КО)

**Интерфейс (3 окна):** Окно основных данных · Окно связанных данных · Окно переноса.

**Функционально:** опирается на **БД `VUZ`** (учебные дисциплины) и её связанность с БД каталога и
студентов. Средства ввода/корректировки; выделенные фонды; филиалы; режим «Настройка»; параметры INI.

**БД VUZ — ввод/корректировка:** записи дисциплин (в т.ч. на иностранном языке), записи контингентов,
учебный план, **импорт дисциплин** из внешних файлов (режимы «Загрузка»/«Обновление»), перенос данных.

> Веб-КО: формы дисциплин/контингентов поверх `VUZ`, связи с каталогом/студентами, выходные
> табличные/стат. формы (`IC_print`/`IC_stat`). Рабочая БД табличных форм — `WORK`.

## 3. АРМ «Администратор» (две части)

### 3.1. Серверный Администратор (локально на сервере)
- **БАЗА ДАННЫХ:** Открыть · Новая · Опустошить · Удалить · Импорт · Экспорт · Копировать ·
  Снять блокировку.
- **АКТУАЛИЗАЦИЯ:** Актуализировать словарь · Создать словарь заново · Реорг. словаря ·
  Реорг. файла документов · Реорг. с исключением удалённых.
- **СЕРВИС:** Копировать/Восстановить файл документов · Списки удалённых/неактуализированных/
  заблокированных · Диагностика файла документов и словаря · ISO-файл по MST · **Выполнить пакетное
  задание**.
- **Команды пакетного задания:** `OpenDB, NewDB, ImportDB, ExportDB, CopyDB, DIAGNOSMF, SEARCH,
  GLOBAL, PRINT, STAT, STATF`.
- **ИНСТРУМЕНТЫ:** редактор ISO/MST · глобальная корректировка · просмотр log-файла сервера.
- Рекомендации по восстановлению БД при разрушении; профиль администратора.

### 3.2. Клиентский Администратор (удалённо по протоколу)
- Режимы: БАЗА ДАННЫХ · СЕРВЕР · АКТУАЛИЗАЦИЯ (создать словарь заново и др.) · СЕРВИС ·
  Выполнить пакетное задание · Опции.

> **Прямая привязка к протоколу (находка #2):** клиентский Администратор = функции `IC_adm_*`
> (`adm_newdb/dbempty/dbdelete/dbunlock/dbstartcreatedictionry/reorgdictionry/reorgmaster/
> getdeletedlist/getclientlist/getprocesslist/setclientslist/restartserver`). То есть веб-Администратор
> реализуем поверх этих команд. Серверная часть (реорг/диагностика/восстановление файлов) — самая
> чувствительная, требует прямого доступа к серверу/файлам и строжайших прав.

## 4. Каталогизатор — правила полей (структура раздела 4)

Детальные правила заполнения (`irbiscat.doc` разд.4) идут по страницам РЛ (вкладкам формы, см. #3 §4.2):
Коды (ISBN/ISSN, кодир. инф.) · Основное БО · Расширенное БО · Специфические · Экземпляры · Технология ·
Систематизация · Содержание · КО · Редкие · Краеведение; плюс специфика по видам (книги/многотомники/
аналитика/журналы/музейный предмет). Это нормативная база валидаций форм веб-Каталогизатора —
разбирается точечно по конкретному полю при реализации.
