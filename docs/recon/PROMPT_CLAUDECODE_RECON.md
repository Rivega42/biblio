# Промт-ТЗ для Claude Code — полная картография продукта ИРБИС64 из архива

> Скопируй это целиком как задание Claude Code. Работай в выданном репозитории. На вход получишь
> **архив(ы) с продуктом и его документацией** (конфиги, документы, справка). Твоя задача —
> восстановить и закартографировать продукт **максимально точно, не пропустив ни одного документа**.
>
> Это независимая разведка: НЕ ищи готовых ответов на стороне, выводи всё **из самих файлов архива**
> и фиксируй источник каждого факта (имя файла + место). Результат будут сверять с другой,
> независимо собранной картой — поэтому важны полнота, точность и прослеживаемость, а не скорость.

---

## 0. Контекст и миссия

- Продукт: **САБ ИРБИС64(+)** — система автоматизации библиотек: сервер (нативный протокол TCP)
  + настольные АРМы (Читатель, Каталогизатор, Книговыдача, Комплектатор, Книгообеспеченность,
  Администратор) + базы данных с конфигами.
- Разработчика продукта больше нет, исходников сервера нет. Поэтому **поставка = действующая
  спецификация**, а твоя картография = её восстановление.
- Цель картографии — стать основой для будущих **веб-версий АРМов**. Тебе строить веб-АРМы не нужно;
  нужно полно и точно описать, **что продукт умеет и как устроен** (данные, протокол, АРМы, форматы,
  поиск, права, файлы).

## 1. Репозиторий и окружение

1. Инициализируй git-репозиторий (локальный/приватный). Заведи `.gitignore` (см. §6: бинарь, данные,
   секреты, картинки — не коммитим).
2. Распакуй архив(ы) во временную рабочую папку ВНЕ репозитория (или в `./_work/`, добавленную в
   `.gitignore`). В репозиторий идут только твои **производные документы** карты, не исходные файлы продукта.
3. Веди работу итеративно, коммить по шагам с осмысленными сообщениями. Поддерживай документацию в
   актуальном состоянии (см. §3).

## 2. Мандат полноты — «ни одного документа»

1. **Первым делом** построй `DOCUMENT_REGISTER.md` — реестр ВСЕХ файлов архива: путь, тип,
   размер, краткое назначение, статус обработки (`todo`/`done`/`skipped+причина`).
2. Особо выдели: все `*.doc/*.docx/*.rtf`, все `*.chm`/`*.hlp` (справка), все типы конфигов
   (`*.par *.fst *.ini *.pft *.mnu *.wss *.opt *.tab *.gbl`), текстовые `*.txt`.
3. Обрабатывай реестр до конца. Файл считается `done` только когда его содержимое отражено в карте.
   `skipped` допустим лишь для бинаря/данных/картинок с явной причиной (см. §6). **Никаких «прочее»** —
   каждый документ и конфиг должен иметь статус.
4. В финале дай отчёт о полноте: сколько `done`/`skipped`, и подтверждение, что `todo`=0.

## 3. Документация, которую ведёшь (структура репозитория)

- `DOCUMENT_REGISTER.md` — реестр и статусы (§2).
- `FINDINGS_NN_<тема>.md` — нумерованные находки по областям (по мере разбора).
- `CAPABILITY_MAP.md` — **сводная мастер-карта** возможностей (обновляешь по ходу): среда/установка,
  реестр баз, карта протокола, структура каждого АРМа, поиск/префиксы, форматы/отображение, ввод,
  файлы/права, администрирование.
- `OPEN_QUESTIONS.md` — что осталось неясным или требует живого сервера/боевых баз; со статусом.
- Соглашение о метках незакрытого: `TODO(recon #N: …)` — машинно-находимые, собирай их в отчёт.
- Каждый факт в картах — со **ссылкой на источник** (`файл:строка/раздел`).

## 4. Методика — проходы

**Проход 0. Инвентаризация.** Распакуй, построй `DOCUMENT_REGISTER.md` (§2). Определи общую раскладку:
где сервер-конфиг, где папка данных с базами, где документация, где справка, где АРМ-ресурсы.

**Проход 1. ВСЯ документация (не пропускать).** Для каждого `*.doc/*.docx/*.rtf`: конвертируй в текст
(см. §6) и извлеки — оглавление, назначение, ключевые сущности (АРМы, базы, поля, форматы, режимы,
команды, ограничения). Для каждого `*.chm`: распакуй, сними оглавление (`*.hhc`), пройди по разделам;
большие справки (десятки МБ) — через оглавление + точечный разбор разделов, но **охвати все разделы
оглавления**. Зафиксируй по каждому документу краткую выжимку в соответствующий `FINDINGS`.

**Проход 2. Статические конфиги баз (папка данных).** По каждому типу:
- `*.PAR` — пути базы и где лежат полные тексты/документы.
- `*.FST` — таблица инвертирования: как поля попадают в словарь и **с какими префиксами**.
- `*.INI` (особенно секция поиска) — соответствие «вид поиска (подпись) → префикс», словари,
  авторитетные файлы, логика, усечение.
- `*.PFT` — форматы отображения: какие поля, области, ссылки, иконки; как формат **ветвится по типу
  записи**; различай экранные и веб(HTML)-варианты.
- `*.MNU` — меню/справочники: значения списков, виды поиска, реестры баз.
- `*.WSS` — рабочие листы: структура форм ввода (главные и вложенные по полям).
- Сведи **профиль каждой базы**: поля, префиксы, виды поиска, цепочки форматов, рабочие листы, связи,
  пути файлов. Корпус большой — для повторяющихся форматов фиксируй **механику** и помечай образцы.

**Проход 3. Протокол.** Если в архиве есть документация клиентского API/DLL или протокола — построй
по ней **полный каталог функций/команд** (сессия, чтение/запись записей, поля, словарь, поиск,
форматирование, пакетная обработка, администрирование) с параметрами, кодами доступа к АРМам и
кодами ошибок. Это источник истины по протоколу — не угадывай.

**Проход 4. Структура АРМов.** Из общего описания и справки по каждому АРМу собери карту экранов:
рабочие области, меню/режимы, формы и их поля, пользовательские потоки. Свяжи действия с протоколом,
где это выводимо из документации.

**Проход 5. Сводка.** Обнови `CAPABILITY_MAP.md`: как слои собираются вместе (поиск→префиксы,
словарь, меню, рабочие листы, форматы, файлы, протокол). Заполни `OPEN_QUESTIONS.md`.

## 5. Что обязательно закартографировать (чек-лист охвата — без подсказок-ответов)

Выведи из файлов сам, по каждому пункту указав источник:
- Полный **реестр баз** и назначение каждой; какие базы видит читатель, какие — каталогизатор.
- Для каждой базы: **набор видов поиска и их префиксы**; словари/авторитетные файлы; усечение/логика.
- **Структура записи**: используемые поля/подполя; поле, определяющее тип/уровень записи.
- **Связи между записями** (иерархии, ссылки на связанные документы): через какие поля реализованы.
- **Форматы отображения**: как выбирается формат по типу записи; экранные vs веб-форматы.
- **Полные тексты/файлы**: где хранятся (пути), какими полями описываются, как регулируются права.
- **Книговыдача**: модель заказа/выдачи/возврата; какие базы участвуют.
- **Каталогизатор**: типология рабочих листов и страниц форм ввода; правила полей.
- **Комплектатор, Книгообеспеченность, Администратор**: функции и режимы.
- **Серверная конфигурация**: пути, порт, реестры баз, список клиентов/прав, гейтинг по АРМам и
  коды ошибок доступа.
- **Протокол**: полный список функций/команд (если есть API-дока).

## 6. Правила обработки и безопасности

- **Кодировки:** конфиги и документы часто в Windows-1251 — конвертируй (`iconv -f CP1251 -t UTF-8`).
- **Инструменты:** `.doc/.rtf` → конвертируй через LibreOffice headless (`soffice --headless
  --convert-to txt`); `.docx` → извлечение текста; `.chm` → распаковать (`7z x`), читать `*.hhc`/HTML;
  архивы `.rar/.zip` → `7z`/`unrar`.
- **Бинарь:** `*.exe/*.dll` — НЕ декомпилируй (логику не реверсим). Если для карты UI нужны ресурсы
  форм — допустимо автоизвлечение ресурсов (DFM/строки), но это вторично и только при необходимости.
- **Данные, не возможности:** файлы данных баз (`*.mst/*.xrf/*.ifp/*.l01/*.n01/*.cnt`), картинки
  (`*.jpg`) — не анализируй, помечай `skipped: данные/изображение`.
- **Секреты/ПДн:** в INI/списках клиентов бывают логины/пароли; в базах — персональные данные.
  **Не коммить их.** В картах приводи только имена ключей/структуру, не значения. Чисти перед любым
  коммитом; `.gitignore` на исходные файлы продукта, бинарь, данные, секреты.
- **Прослеживаемость:** любой нетривиальный факт — со ссылкой на файл и место. Не выдумывай; если не
  уверен — ставь `TODO(recon #N: проверить …)`.

## 7. Результат (Definition of Done)

- `DOCUMENT_REGISTER.md` — все файлы со статусами; `todo`=0; отчёт о полноте.
- Комплект `FINDINGS_NN_*.md` по областям + сводная `CAPABILITY_MAP.md` + `OPEN_QUESTIONS.md`.
- Каждый документ архива отражён в картах (или явно `skipped` с причиной).
- Все факты — с источниками; незакрытое — помечено `TODO(recon #N …)` и собрано в отчёт.
- Репозиторий чист от секретов/бинаря/данных; история коммитов отражает ход разбора.

## 8. Стартовая команда

> Прими архив(ы) продукта ИРБИС64. Выполни полную независимую картографию по этому ТЗ, выводя всё из
> самих файлов и фиксируя источники. Сначала Проход 0: распакуй и построй `DOCUMENT_REGISTER.md` со
> статусами по КАЖДОМУ файлу. Затем Проходы 1–5: обработай **всю документацию** (ни одного документа
> не пропусти), статические конфиги всех баз, протокол (по API-доке), структуру всех АРМов; сведи в
> `CAPABILITY_MAP.md` и `OPEN_QUESTIONS.md`. Соблюдай §6 (кодировки, инструменты, бинарь не
> декомпилируем, секреты/данные не коммитим). В конце дай отчёт о полноте (`todo`=0) и список
> `TODO(recon #N …)`. Не ищи ответы на стороне — только из архива, с указанием источников.
