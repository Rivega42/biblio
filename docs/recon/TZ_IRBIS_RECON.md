# ТЗ на полное картографирование ИРБИС64+ (`irbis-recon`) — для Claude Code

> **Что это.** Задание для **локального Claude Code** на твоём ноуте: максимально полно
> «разобрать» установленную поставку **ИРБИС64+** (сервер + АРМы + демо-базы) и составить
> **карту возможностей** — какие функции, методы, команды и форматы есть и как они работают.
>
> **Зачем.** Разработчиков ИРБИС больше нет, исходников сервера нет. Поэтому **демо-поставка =
> единственная действующая спецификация продукта**, а эта карта — её восстановление. Карта —
> фундамент для перевода АРМов ИРБИС в полноценные **веб-версии** (первая боевая инсталляция —
> каталог СПб ГТБ `irbis-web` = веб-АРМ «Читатель» + часть «Книговыдачи»).
>
> **Где выполняется.** Только локально (WSL рекомендуется, либо нативно Windows / Docker). Живой
> сервер слушает `localhost:6666` и виден лишь с этого компьютера. Облачные среды сюда не достают.
>
> Версия 1.0. Дата 20.06.2026.

---

## 0. Принцип и два прохода

Картографируем **всё, что продукт умеет** (scope ресёрча — максимальный). Что из этого потом
*выставлять в веб* — отдельное продуктовое решение (см. §8), на этапе карты ничего не отсекаем.

Работа идёт в три прохода:

- **Проход А — статика данных/конфигов** (§4): чтение текстовых конфигов поставки. Даёт «как
  настроено»: поля, словари, префиксы, форматы, меню, пути. Не требует живого сервера.
- **Проход Б — динамика протокола** (§5): зондирование живого сервера `:6666`. Даёт «как сервер
  реально отвечает»: команды, формат запрос/ответ, коды ошибок, доступность АРМ-команд под логином.
- **Проход В — картография клиентского UI** (§5А): формы, кнопки, окна, настройки, потоки самого
  настольного клиента. Даёт «как с этим работает человек». Максимально автоматизирован (см. §5А).

Цель — **«полнейшая картография приложения»**: данные (А) + возможности сервера (Б) + интерфейс
клиента (В). Целевой веб-АРМ — **структурный 1:1** с настольным клиентом (те же экраны, поля,
кнопки, подписи, раскладка, потоки); где десктоп-поведение не ложится на веб/ГОСТ — помечаем.

Порядок: А (дешевле, задаёт словарь) → Б (возможности) → В (интерфейс). В на практике идёт
параллельно с Б — перехват трафика (§5А.3) связывает действия UI с командами протокола.

## 1. Источники истины (по убыванию достоверности)

1. **Клиент BookCabinet** (рабочий клиент нативного протокола) — *эталон того, что уже реализовано

   ответа. Всё остальное — расширение этого набора.
2. **Живой демо-сервер** `:6666` — единственный достоверный источник реального поведения (вендора и
   исходников сервера нет).
3. **Текстовые конфиги поставки** — `.FST/.PFT/.MNU/.FDT/.WSS/.PAR/INI` (см. §4).
4. **Мануал J-ИРБИС 2.0** (`ReadMe_jirbis2`) — описывает, как вендор использует протокол; референс,

5. **Открытые клиенты протокола** (напр. библиотеки `irbis` для разных языков) — для сверки опкодов;
   помечать как гипотезу до подтверждения на живом сервере.

> Опкоды команд в этом ТЗ даются как **рабочая гипотеза для проверки**, а не факт. Подтверждать по
> BookCabinet-клиенту и живому серверу. Не доверять памяти/чужим докам без проверки на `:6666`.

## 2. Что включаем в анализ / что НЕ трогаем

**Включаем (текст, безопасно, полезно для карты):**
- `*.FST` — таблицы инвертирования (как поля попадают в словарь → префиксы поиска).
- `*.PFT` — форматы отображения (логика вывода, области, ссылки, иконки).
- `*.MNU`, `*.DDB` — меню, списки видов поиска, привязка «вид → префикс».
- `*.FDT` — определения полей/подполей; `*.WSS` — рабочие листы (структура записи).
- `*.PAR` — пути к файлам базы (где MST/INV, где электронные документы — параметр 11).
- `*.INI` сервера и АРМов, **секция `SEARCH`** — фактические префиксы; **после вычистки кред**.

**Бинарь — извлекаем ресурсы, но не декомпилируем логику:**
- Из `*.exe`/`*.dll` АРМов допустимо **автоматически извлекать UI-ресурсы** (DFM-формы, строки,
  иконки) для картографии интерфейса (§5А.1) — это разбор *своей* установки для воспроизведения
  осиротевшего продукта, а не реверс логики. Глубокую декомпиляцию Pascal-кода (IDR/DeDe) **не
  делаем** — мутно, юридически серее и не нужно (поведение берём из §5А.2–5А.3).
- ⚠️ Юридический статус (лицензия/право на доработку осиротевшего ПО) — за тобой; я не юрист. Сами
  бинарники вендора **не распространяем** (в репозиторий не кладём, наружу не выкладываем).

**НЕ трогаем вовсе:**
- Файлы данных: `*.MST, *.XRF, *.IFP, *.L01, *.N01, *.CNT` — это **данные**, не возможности.
- Любые дампы, бэкапы, базы читателей (RDR) с ПДн — даже если демо, привычка железная.

**Гигиена (после инцидента с утечкой):**
- Перед помещением в любой git — **вычистить логины/пароли/строки подключения** из INI.
- Репозиторий **локальный или приватный**, никогда публичный; `.gitignore` на бинарь/данные/дампы.
- В чат для совместного разбора нести только вычищенный текст конфигов, DFM/скриншоты без секретов
  и обезличенную карту.

## 3. Среда запуска (локально на ноуте)

- **WSL (рекомендуется для А и Б):** Linux внутри Windows. Видит `C:\` через `/mnt/c/...` и
  `localhost:6666` без настройки. Установка: `wsl --install`, затем Claude Code в дистрибутиве.
  Подходит для чтения конфигов (А) и протокол-клиента (Б).
- **Нативно Windows (обязательно для Прохода В):** интроспекция GUI (`pywinauto`/UIA), автоскриншоты
  и перехват трафика Wireshark работают с Windows-GUI-процессами — это **только нативный Windows**,
  не WSL2. Сюда же ставим Python для извлечения DFM-ресурсов (`pefile`).
- **Docker:** к серверу через `host.docker.internal:6666`, папку установки — томом; для GUI не годится.

**Инструменты Прохода В (Windows):** Resource Hacker / `pefile` (DFM-ресурсы), `pywinauto` + Inspect.exe
(дерево контролов), `mss`/`pywinauto` (скриншоты), Wireshark или логирующий TCP-прокси (трафик :6666).

## 4. Проход А — статическая картография конфигов

Для каждого типа файла — что извлечь в карту:

| Тип | Что извлекаем в карту |
| --- | --- |
| `*.PAR` (по каждой БД) | Список баз; пути к MST/INV; **параметр 11** (папка электронных документов); связки файлов |
| `*.FST` | Какие поля/подполя индексируются, с какими методами; **карта «поле → префикс словаря»** |
| INI / секция `SEARCH` | Фактические префиксы точек доступа (подтверждение префиксов из FST); виды поиска |
| `*.MNU`/`*.DDB` | Списки видов поиска, значения для комбобоксов, привязка «вид поиска → префикс» |
| `*.FDT` | Полный перечень полей/подполей базы (метки, повторяемость, подполя) |
| `*.WSS` | Рабочие листы — какие поля в каком РЛ, порядок, обязательность (модель ввода для Каталогизатора) |
| `*.PFT` | Форматы вывода: какие поля выводятся, области, условная логика, ссылки/иконки; цепочки `@<формат>` |
| INI АРМов | Параметры режимов; **какие АРМ-функции вообще сконфигурированы** в поставке |

**Результат прохода А:** по каждой демо-базе — профиль (поля, префиксы, виды поиска, цепочки PFT,
пути файлов) + сводный список «что умеет поставка по конфигам».

## 5. Проход Б — динамическое зондирование протокола `:6666`

Зондировать по областям. Для каждой команды фиксировать по шаблону §6. Начинать с того, что уже
есть в BookCabinet-клиенте, затем расширять.

**5.1. Сессия и сервер.** Подключение/регистрация клиента, выход, keepalive/NOP, версия и
конфигурация сервера, статистика, список баз, max MFN базы.

**5.2. Поиск.** Поиск по словарю с префиксом; усечение (`$`/`~`); булева логика; полнотекст
`KT=FT!` (ранжирование); морфологическое расширение; сортировка; постинги/частоты терминов;
авторитетные файлы и тезаурус; фасеты (PLUS). Зафиксировать формат поискового выражения и ответа
(набор MFN, total).

**5.3. Чтение записей.** Чтение по MFN; пакетное чтение; чтение конкретного поля/подполя; формат
сырой MARC-записи в ответе.

**5.4. Форматирование (PFT).** Расформатирование записи именованным форматом (`@<pft>`); цепочки
форматов; разница «сырой PFT-вывод» vs готовый HTML; **резервный канал JSON-RPC web-сервис** (если
включён) — даёт поля записи и расформатирование; проверить, включён ли он в поставке.

**5.5. Запись/корректировка (АРМ Каталогизатор).** ⚠️ write-операции — пробовать **только на демо**,
на тестовой записи, осознанно. Создание записи; обновление; логическое/физическое удаление;
блокировка/разблокировка записи и базы; глобальная корректировка (пакетная). Зафиксировать, какие
права/АРМ-логин нужны (ошибка `-3338` = нет доступа к командам АРМа).

**5.6. Файлы и полные тексты.** Чтение файла/документа по протоколу; разрешение пути из 951/955 +
`ed_path`/параметр 11; ограничения: **TCP/IP-сервер НЕ пишет бинарные файлы** (загрузка через
протокол невозможна), лимит выгрузки (по умолчанию ~10 МБ); права доступа к файлу (база **Right** —
проверить, есть ли она в демо и что отдаёт по протоколу).

**5.7. Читатели (АРМ Читатель / RDR).** Структура базы читателей; аутентификация (фамилия + билет);
чтение формуляра; список выданного/заказанного. ⚠️ на демо — тестовые читатели, не ПДн.

**5.8. Книговыдача.** Постановка заказа в очередь; выдача/возврат экземпляра; бронирование;
просмотр очереди. Зафиксировать корректный протокол постановки (в J-ИРБИС был сломан — это
открытый вопрос проекта). Высокая чувствительность — write по фондам.

**5.9. Комплектование / Книгообеспеченность / табличные формы.** Наличие и команды; генерация
табличных форм (Word/RTF) как в АРМах; — для соответствующих веб-АРМов (опционально, по приоритету).

**5.10. Администрирование.** Статистика сервера; список пользователей и **их АРМ-права**; управление
базами (создание/удаление/реорганизация); очистка кэша/актуализация словаря; перезапуск. Самая
чувствительная область — отдельный модуль с жёстким доступом.

**5.11. Внешние/корпоративные.** Z39.50 (вендором **не рекомендован** для ИРБИС-источников — родной
путь TCP/IP); виртуальные базы; подключение внешних ИРБИС/WEB-ИРБИС источников.

## 5А. Проход В — автоматическая картография клиентского UI

> Цель — снять **весь интерфейс настольных АРМов** (формы, кнопки, окна, настройки, меню, потоки)
> для воспроизведения «структурно 1:1» в вебе. Исходников нет, поэтому собираем из четырёх слоёв,
> максимально автоматически. ⚠️ Интроспекция GUI и перехват трафика выполняются **нативно в Windows**
> (не в WSL — WSL2 не видит Windows-GUI-процессы). Статический разбор ресурсов — где угодно.

**Почему не «просто из кода»:** сорсов нет; в скомпилированном .exe автоматически достаётся
*раскладка форм* (DFM-ресурсы), но НЕ *поведение* (логика кликов, динамические формы) — оно в
бинарном Pascal-коде. Поэтому поведение восстанавливаем не реверсом, а наблюдением живого клиента
и перехватом трафика. Глубокую декомпиляцию Delphi (IDR/DeDe) не делаем — мутно и не нужно.

### 5А.1. Статическое извлечение ресурсов из .exe (раскладка форм) — автоматически
- АРМы ИРБИС — вероятно Delphi; формы хранятся в .exe как **DFM-ресурсы** (бинарь с сигнатурой
  `TPF0`). Извлекать: Resource Hacker, либо скрипт на `pefile` (обход RT_RCDATA, поиск `TPF0`),
  затем бинарный DFM → текстовый DFM.
- Из DFM получаем: список форм, на каждой — компоненты (поля, кнопки, гриды, вкладки), их подписи,
  типы, позиции/размеры, порядок табуляции, связи. Это «скелет» каждого экрана.
- Также снять: строки/ресурсы (`pefile`), иконки/битмапы (для иконографии веб-версии), версии.
- **Ограничение:** даёт статику форм, не поведение и не динамически создаваемые формы.

### 5А.2. Интроспекция живого клиента (реальные экраны) — автоматически
- Запустить каждый АРМ и обойти **дерево окон/контролов** через Windows UI Automation:
  `pywinauto` (`app.windows()`, `.dump_tree()`), Inspect.exe, WinSpy, Accessibility Insights.
- Для каждого окна/диалога автоматически снять: заголовок, все контролы (тип, текст, состояние,
  координаты), пункты меню и тулбаров, статус-бары. + **автоскриншот** каждого окна (`pywinauto`/`mss`).
- Ловит и то, что строится в рантайме (мастера, динамические вкладки) — чего нет в DFM.
- Прогнать по всем пунктам меню/режимам, чтобы открыть максимум окон.

### 5А.3. Перехват трафика: UI-действие → команды протокола (поведение) — полу-автоматически
- Снять, **что каждый экран/кнопка делает на проводе**: запустить **Wireshark** (фильтр
  `tcp.port==6666`) или маленький **логирующий TCP-прокси** между клиентом и сервером.
- Кликать по функциям клиента, фиксируя соответствие «действие в UI → последовательность команд
  ИРБИС» (связывает Проход В с Проходом Б §5). Для копии 1:1 это ключевой слой — даёт семантику
  каждого экрана.
- Результат — таблица «экран/кнопка → команды → эффект».

### 5А.4. Residual — ручные заметки (минимум)
- Только там, где смысл не выводится автоматически: бизнес-правила, валидации, неочевидные потоки.
  Фиксировать коротко, со ссылкой на скриншот/трафик.

### 5А.5. Шаблон описания экрана (заполнять на каждое окно АРМа)

```
АРМ / экран: <напр. Каталогизатор / Ввод записи>
Тип окна: <главное | MDI-дочернее | модальный диалог | мастер>
Назначение: <что делает экран>
Контролы: <поля (метка, тип, привязка к полю MARC), кнопки, гриды, вкладки, меню/тулбар>
Источник раскладки: DFM / UIA-дамп / скриншот  | файл: <…>

Динамика: <что появляется/меняется в рантайме>
```

«UI→протокол».

## 7. Жёсткие ограничения (уже известны из мануала — подтвердить на демо)

- **Команды гейтятся АРМ-правами логина** — `-3338` «Нет доступа к командам АРМа». Для веб-АРМов
  понадобятся служебные логины с правами соответствующего АРМа.
- **TCP/IP-сервер не пишет бинарные файлы** → загрузка вложений через протокол невозможна; нужен
  прямой файловый доступ к хранилищу. Влияет на веб-Каталогизатор (загрузка сканов/PDF).
- **Лимит выгрузки файла** по умолчанию ~10 МБ (настраивается на сервере).
- **Z39.50** — только для внешних не-ИРБИС источников; между ИРБИС — нативный TCP/IP.
- **Кодировки** — сервер отдаёт в своей кодировке; нормализация в UTF-8 на границе клиента.

## 8. Как карта питает веб-АРМы (архитектурный вывод)

- **Общее ядро строим один раз:** протокольный клиент (из BookCabinet) + реестр баз + поиск +
  форматы (серверный PFT). Все веб-АРМы — модули поверх ядра.
- **АРМ-модули** поверх ядра: Читатель (read) → Книговыдача → Каталогизатор (write) → Комплектатор →
  Администратор. Порядок — по приоритету и по риску (write/admin — позже и с усиленной защитой).
- **Развилка безопасности (зафиксировать до экспонирования наружу):** read-функции (Читатель) и
  write/admin (Каталогизатор, Книговыдача, Администратор) — разный класс риска. Полный веб-АРМ =
  write-доступ к фондам через веб. С учётом прошлого инцидента: строгая аутентификация по ролям,
  аудит, сеть в контуре, никакого внешнего доступа к write/admin-модулям.

## 9. Безопасность ресёрча

- Креды из INI и любые ПДн — **не в репозиторий, не в чат**. Вычищать перед шарингом.
- Бинарь и данные — не коммитим (`.gitignore`).
- Write-команды (§5.5, 5.8, 5.10) — **только на демо, на тестовых объектах**, осознанно; не на
  данных, которые жаль потерять.
- Репозиторий приватный/локальный; секрет-скан (gitleaks) на коммит — как в основном проекте.
