# Recon-находки #5 — слой отображения (PFT) и ввода (WSS)

> Источник: DATAI (PFT/WSS баз IBIS, IMAGE, Deposit). Дата 20.06.2026. Дополняет #4 и мастер-карту.
> Корпус PFT огромен (3903 файла) — здесь зафиксирована **механика**; конкретные форматы извлекаются
> точечно по запросу. Подтверждает ADR-003 (отображение — серверным PFT, не переписываем в коде).

---

## 1. Отображение записи — диспетчер по типу (v920)

Стандартный краткий формат `optim.pft` — это **диспетчер**: по полю `v920` (тип/уровень записи)
вызывает именованный подформат через `&uf('6XXX')`:
- `v920='J'` → `6JW` (журнал/периодика)
- `v920:'NJ'` → `6NJW` (газета/статья)
- `v920:'SPEC'` → `6MN` (многотомник/спецификация)
- `v920:'ASP'|'AUNTD'` → `6ASP` (аналитическое описание)
- `v920:'MUSP'` → `6MUSI` (ноты/музыка)
- `v920:'BOUNI'` → `6VS600`
- иначе → `6KN` (книга)

**Вывод:** один формат-точка-входа ветвится по типу записи и делегирует подформатам. Для веб-клиента
это значит: запрашиваем у сервера расформатирование нужным именованным PFT (`IC_readformat`/`IC_sformat`,
находка #2), а ветвление по типу делает сам PFT. Переписывать форматную логику в коде не нужно.

## 2. Веб-варианты форматов: суффикс `_H`

Форматы с суффиксом `_H` (`Jw_H.pft`, `Uni_H.pft`, `!!kk0shk_H.pft`…) — **HTML-представления** для
веба. То есть у вендора уже есть готовые HTML-форматы вывода → веб-клиент берёт их вывод и оформляет
поверх (статусы/ссылки/иконки), как и заложено в `TZ_CLAUDE_CODE` (рендер + санитайзинг + оверлей).

## 3. IMAGE — имидж-каталог (образы карточек)

Базы имидж-каталога рисуют **сканы каталожных карточек** через семейство форматов `!!kkN…_H.pft`
(`kk` = каталожная карточка; разные виды: shk/bo/udk/ann/odk/j/ex) + `brief_land.pft`, `dublinc.pft`.
Это образец для имидж-баз СПб ГТБ (IMAGE/IMGZENZ/IMGOPERA/IMGBALET): вывод — изображение карточки,
а не текстовое БО.

## 4. Ввод (Каталогизатор) — композиция из вложенных рабочих листов

Рабочие листы IBIS (`*.wss`) — это **вложенные пофайловые формы по элементам данных**: `200dm` (заглавие),
`210m` (выходные данные), `215m` (объём), `461m`/`463` (связи), `488m`/`488n` (связь/иерархия), `910m`/`910pm`
(экземпляры), `951m` (файлы), `961m`/`962m`, `923m` (аналитика), `701MP`/`900m` и др. Суффикс `m` = модальный
вложенный РЛ поля.

**Вывод:** форма ввода Каталогизатора = главный РЛ + набор вложенных РЛ по полям. Для веб-Каталогизатора
это прямая модель: вкладки/секции страницы (находка #3, §4.2) наполняются полями из соответствующих WSS;
справочники значений — из MNU; правила — из FST/INI.

## 5. Связь слоёв (как это собирается в веб-АРМ)

```
[SEARCH] (INI)  → подписи и префиксы форм поиска        (находка #4)
FST             → генерация словаря/префиксов            (находка #4)
MNU             → значения комбобоксов/виды/реестры баз
WSS             → структура форм ввода (Каталогизатор)   (эта находка)
PFT (+ _H)      → отображение записи, ветвление по v920  (эта находка)  ← ADR-003
PAR п.11        → папка полных текстов (951/955)          (находка #4)
протокол IC_*   → как всё это запросить у сервера         (находка #2)
```
