# Ревью миграции jirbis (J-ИРБИС 2.0 / Joomla) → Biblio

> Цель: бесшовно встать **вместо** jirbis (публичный веб-OPAC СПб ГТБ). Грунтовано на
> `docs/design/specs/SPEC_reader_jirbis.md` + `docs/reference/ReadMe_jirbis2_text.txt`
> + проверках на боевом RDR. Статус: концептуальное ревью; для табличного ревью нужен дамп Joomla.

## TL;DR (главный вывод)
**Учётки читателей, пароли, формуляр, заказы и брони — НЕ в Joomla, а в самом ИРБИС (RDR / RQST).** К ним мы **уже подключены**. Поэтому «бесшовная замена jirbis» **не требует переноса пользователей/кредов/заказов из Joomla** — нужно лишь, чтобы наш портал аутентифицировал против того же RDR и поддержал формат их интернет-пароля (это уже работает). Из Joomla переносим узкий, но ценный слой: **соц-данные (рейтинги/комментарии), постоянные запросы, CMS-контент**.

## Архитектура jirbis (что это технически)
- Стек: **PHP 5.6 / Joomla 2.5 / MariaDB** — всё EOL.
- Две подсистемы: (1) **Joomla CMS**; (2) **ИРБИС-подсистема** = компонент `com_irbis` + модули/плагины Joomla.
- **Каталог в Joomla-MySQL — это КЭШ** записей ИРБИС (регенерируемый; переносить не нужно).
- **Вход и читательские данные — поверх ИРБИС:** логин = фамилия + билет (поле 30) + интернет-пароль (`user_password_tag` = поле **130/24**, *MD5 без соли*); запрет входа должникам; формуляр; заказы; продление — всё через TCP-сервер ИРБИС, а не через Joomla.

| Интернет-пароль | **RDR** поле 130/24 | ❌ уже есть (подтвердить поле+формат) |
| Формуляр / выдачи / сроки | **RDR** (поле 40) | ❌ уже есть |
| Заказы / брони / очередь | **RQST** (30/903/40/41) | ❌ уже есть |
| Рейтинги / комментарии / Top-10 (WEB 2.0) | **Joomla MySQL** (`#__irbis_*`) | ✅ → соц-граф (#174) |

## Бесшовный вход — почему он уже почти решён
jirbis логинит против **RDR** (билет + пароль поля 130), и наш `auth_reader` делает **ровно то же** (поиск `RI=<билет>` + сверка пароля поля 130/100, plaintext или несолёный MD5). Значит **читатели входят теми же кредами без какой-либо миграции учёток**. Проверено на проде: поле 130 заполнено у **258/258** (9 симв.), вход работает.

⚠️ **МОДЕЛЬ ВХОДА НЕОДНОЗНАЧНА — требует подтверждения (по jirbis-конфигу + дампу Joomla). Прежний вывод «бесшовно подтверждено» снят.**
- **Дефолт J-ИРБИС (док стр. 761):** идентификация читателя = **фамилия + идентификатор (поле 30)**; **поле 130 (пароль) по умолчанию НЕ используется**. То есть «из коробки» вход — НЕ по паролю, а по фамилия+билет.
- **Связь RDR ↔ Joomla (док стр. 1303):** категории читателей RDR маппятся на уровни пользователей Joomla; «Уровень 2 Joomla = зарегистрированный пользователь». → **Joomla-пользователи в схеме УЧАСТВУЮТ**, не всё в чистом ИРБИС.
- **Факт на их RDR (read-only):** поля 24/100 отсутствуют; поле 130 заполнено (9-символьный plaintext, 258/258). Это значит ЛИБО они включили парольный режим (`user_password_tag`=130), ЛИБО это сгенерированное значение, которое читателю не показывают (и тогда оно НЕ его логин-пароль).
- **Вывод:** как именно читатели входят в jirbis — **фамилия+билет** или **билет+пароль(130)** — и где живут их веб-учётки, определяется их **jirbis-конфигом** (`user_password_tag`, режим авторизации, поле логина) и таблицей Joomla `#__users`. **Это и есть то, что обязательно перепроверить дампом.** Наш портал затем поддержит их реальную схему (умеем билет+пароль; можем добавить фамилию как фактор и Joomla-маппинг).

## Что реально переносим ИЗ Joomla (нужен дамп)
1. **Соц-данные** (`com_irbis` таблицы): рейтинги (5-балльные), комментарии с модерацией, Top-10. → наш кросс-тенантный соц-граф (#174).
2. **Постоянные запросы** (если хранятся в Joomla, а не в RDR) → наши сохранённые поиски + алерты на новинки.
3. **CMS-контент**: новости, события (афиша), страницы «О библиотеке» → `tenantContent`/news (закрывает и issue идентичности #248).
4. **SSO/ЕСИА-связки** (если настроены `сквозная авторизация в ЭБС`) → маппинг внешняя-идентичность ↔ билет в наш `reader_identity`.

## Что НЕ переносим
- Кэш каталога (регенерируется из ИРБИС).
- Cookie-закладки (физически не существуют в БД — привязаны к браузеру).
- EOL-стек и ручной PHP-конфиг (у нас декларативный).

## Что нужно для ТАБЛИЧНОГО ревью (доступ)
- **`mysqldump` базы Joomla jirbis** (схема + данные), либо read-доступ к MySQL. Тогда:
  - перечислю реальные таблицы `#__*` (особенно `#__irbis_*` / `com_irbis`) с их колонками;
  - посчитаю объёмы (сколько рейтингов/комментов/постоянных запросов/Joomla-юзеров/новостей);
  - построю точный маппинг → наши таблицы + ETL-скрипт переноса.
- Read-доступ к их **RDR** (или один тест-билет) — подтвердить поле/формат пароля (130 vs 24, MD5 vs plaintext).
- ⚠️ **ПДн (152-ФЗ):** дамп содержит email/хэши паролей/контент читателей. Передавать защищённым каналом, хранить в закрытом контуре, **не коммитить** в репозиторий.

## План миграции (после получения дампа)
1. Их сторона: `mysqldump` Joomla + (опц.) экспорт RDR/RQST уже не нужен (читаем по TCP).
2. Я: ревью схемы + объёмов → точный маппинг.
3. ETL: Joomla-соц/постоянные-запросы/CMS → наши таблицы; вход — уже через RDR (без переноса).
4. **Параллельный прогон** (jirbis жив) → сверка выборок → переключение DNS/прокси на наш портал.
5. Откатопригодность: jirbis оставить read-only несколько недель.

## Открытые вопросы (подтвердить с библиотекой)
- Поле и формат интернет-пароля (130/24, MD5/plaintext) + есть ли «фамилия» как второй фактор.
- Где хранятся постоянные запросы (Joomla vs RDR).
- Используются ли в jirbis рейтинги/комментарии/SSO (есть ли что переносить из соц-слоя).
- Объём Joomla-`#__users` (только админы или есть само-регистрировавшиеся веб-читатели вне RDR).
