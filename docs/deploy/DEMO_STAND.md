# Демо-стенд Biblio (сервер в РФ) — для реестра росПО

> **Зачем.** Для включения Biblio в **Единый реестр российского ПО** (Минцифры)
> эксперты должны проверить **работающий экземпляр** ПО, **сервер которого
> размещён в РФ** (issue [#226](https://github.com/Rivega42/biblio/issues/226),
> правовой контекст — [`docs/legal/REGISTRATION_GUIDE.md` §3.5](../legal/REGISTRATION_GUIDE.md)).
> Этот документ описывает, как поднять публичный демонстрационный экземпляр
> Biblio на сервере в РФ, наполнить его показательными данными **без реальных
> персональных данных (ПДн)**, держать его опрятным (периодический сброс) и
> выдавать демодоступ по заявке.

Демо-стенд — это базовый стек `deploy/` (nginx TLS → backend → PostgreSQL) с
**overlay-профилем демо** (`deploy/docker-compose.demo.yml`): own-store, **без
живого ИРБИС**, **без реальных ПДн**, помечен как ДЕМО, read-mostly, с ротацией
логов и периодическим сбросом к эталону.

---

## 1. Требования к серверу

| Требование | Значение |
|---|---|
| **Размещение** | **Сервер физически в РФ** (российский ЦОД/облако; домен `.ru`/`.рф`). Это требование реестра росПО — экземпляр для проверки должен быть на сервере в РФ. |
| ОС | Linux (рекомендуется); подойдёт любая с Docker Engine 24+. |
| Docker | Docker Engine + Docker Compose v2 (`docker compose version`). |
| Ресурсы | от 2 vCPU / 4 ГБ ОЗУ / 20 ГБ диска (демо-нагрузка невелика). |
| Сеть | публичный IP, открыты 80/443; домен вида `demo.biblio.<ваш>.ru`. |
| TLS | реальный сертификат (Let's Encrypt / внутренний CA). Самоподписанный — только для локальной проверки. |

> Размещение в РФ важно и для **152-ФЗ**: даже демо-данные (синтетические) не
> покидают РФ. См. §6.

---

## 2. Что входит в профиль демо

Overlay `deploy/docker-compose.demo.yml` накладывается поверх базового
`deploy/docker-compose.yml` и меняет ровно следующее:

- **`APP_ENV=demo`** — бэкенд знает, что это демонстрационный режим.
- **ИРБИС отключён** (`IRBIS_HOST=""`) — мост к живому ИРБИС не используется,
  данные только из **own-store** (sqlite-каталог + PostgreSQL Access-стор).
  Демо-стенд не зависит от внешнего ИРБИС и не ходит в чужие базы.
- **Ротация docker-логов** — `json-file`, `max-size=10m`, `max-file=3` на каждый
  сервис, чтобы публичный стенд не забил диск.
- **read-mostly + сброс по расписанию** — стенд наполняется сидом и периодически
  возвращается к эталонному снимку (§5); правки посетителей живут до сброса.
- **Метки** `com.biblio.role=demo`, `com.biblio.stand=reestr-rospo-demo` на всех
  сервисах — для инвентаризации и мониторинга.

Базовые **healthcheck'и** (`db`/`backend`/`proxy`) наследуются из базового
compose без изменений (§4).

## 3. Подъём демо-стенда (по шагам)

```sh
cd deploy

# 1. Окружение демо (секреты НЕ коммитятся: .env в gitignore).
cp .env.demo.example .env
#    Отредактировать .env: задать POSTGRES_PASSWORD и APP_SECRET (метки «СМЕНИТЬ НА ПРОДЕ»).
#    Сгенерировать секреты, например:  openssl rand -hex 32

# 2. TLS-сертификат.
#    Локальная проверка — самоподписанный:
sh proxy/gen-cert.sh
#    На сервере в РФ — РЕАЛЬНЫЙ сертификат: положить server.crt/server.key в
#    proxy/certs/ (Let's Encrypt/CA) и включить HSTS в proxy/nginx.conf.

# 3. Поднять демо-стек (базовый compose + overlay demo).
docker compose -f docker-compose.yml -f docker-compose.demo.yml up -d --build

# 4. Наполнить демо-данными (провижининг арендатора + каталог + читатели).
sh demo/seed_demo.sh

# 5. Зафиксировать ЭТАЛОННЫЙ снимок (для периодического сброса, см. §5).
sh demo/reset_demo.sh --snapshot
```

Открыть `https://demo.biblio.<ваш>.ru/` (при самоподписанном dev-сертификате —
принять предупреждение браузера).

Остановка / полная очистка:

```sh
docker compose -f docker-compose.yml -f docker-compose.demo.yml down       # стоп, тома сохранить
docker compose -f docker-compose.yml -f docker-compose.demo.yml down -v    # стоп + стереть данные
```

> Для удобства можно задать `COMPOSE_FILE=docker-compose.yml:docker-compose.demo.yml`
> в окружении — тогда `-f …` можно не повторять в каждой команде.

## 4. Ops-обвязка

### 4.1. Healthcheck'и

Все три сервиса сообщают здоровье (наследуется из базового compose):

- **`db`** — `pg_isready`.
- **`backend`** — GET **`/api/health`** (без авторизации, 200 даже при
  недоступном ИРБИС; на демо ИРБИС и не задействован).
- **`proxy`** — `curl -fsk https://localhost/`.

Проверка:

```sh
docker compose -f docker-compose.yml -f docker-compose.demo.yml ps
# STATUS должен показывать (healthy) у db, backend, proxy

# Прямая проверка эндпойнта здоровья:
docker compose -f docker-compose.yml -f docker-compose.demo.yml exec backend \
  python -c "import urllib.request as u; print(u.urlopen('http://127.0.0.1:8080/api/health', timeout=4).status)"
```

### 4.2. Логи и их ротация

Overlay включает ротацию docker-логов (10 МБ × 3 файла на сервис) — публичный
демо не забьёт диск. Просмотр:

```sh
docker compose -f docker-compose.yml -f docker-compose.demo.yml logs -f --tail=200 backend
```

### 4.3. Бэкап и восстановление PostgreSQL

Используются те же скрипты, что и в проде (`deploy/backup/`):

```sh
# Бэкап Access-стора (один дамп с таймстампом → том backups):
sh backup/pg_backup.sh
KEEP=7 sh backup/pg_backup.sh           # + оставить только 7 новейших

# Список доступных дампов / восстановление:
sh backup/pg_restore.sh                 # список
sh backup/pg_restore.sh <файл-дампа>    # restore (подтверждение; FORCE=1 — пропустить)

# Репетиция restore в throwaway-БД (не трогая живые данные):
FORCE=1 RESTORE_DB=demo_restore_test sh backup/pg_restore.sh <файл-дампа>
```

> Для **демо** ценность бэкапа невелика (данные синтетические и легко пересоздаются
> сидом), но скрипты те же — они заодно лежат в основе механизма сброса (§5):
> эталонный снимок — это обычный `pg_dump`.

## 5. Демо-данные, сброс и ретенция

### 5.1. Что заливает сид

`sh demo/seed_demo.sh` переиспользует **штатные** механизмы бэкенда (ничего не
дублирует):

1. **Провижининг демо-арендатора** — `python -m access.provision demo`: схема
   арендатора + системные словари + админ-аккаунт + план/модули.
2. **Демо-каталог** — `deploy/demo/seed_demo_catalog.py` кладёт ~10 показательных
   библиографических записей в own-store каталога через `CatalogStore.save()`
   (классика + методические издания; **без живого ИРБИС**).
3. **Демо-читатели** — 3 учётные записи `reader1..reader3` со **синтетическими**
   именами и email на зарезервированном домене `example.org` (RFC 2606).
   **Реальных ПДн нет.**

Демодоступ после сида:

| Роль | Логин | Пароль |
|---|---|---|
| Администратор демо | `demo` | `demo` (или `DEMO_ADMIN_PASSWORD` из `.env`) |
| Читатель | `reader1` / `reader2` / `reader3` | совпадает с логином |

### 5.2. Сброс к эталону

Демо read-mostly: посетители могут что-то менять, поэтому стенд **периодически
возвращается к эталонному снимку**.

```sh
sh demo/reset_demo.sh --snapshot   # ОДИН РАЗ после сида — зафиксировать эталон
sh demo/reset_demo.sh              # сброс к эталону (Access-стор из golden-дампа + пере-сид каталога)
sh demo/reset_demo.sh --reseed     # полный пере-сид «с нуля»
```

Механизм сброса:
- **Access-стор (PostgreSQL)** восстанавливается из golden-дампа `demo-golden.dump`
  (через `backup/pg_restore.sh`).
- **Own-store каталога (sqlite)** очищается и пере-сидится (`seed_demo_catalog.py`).

Планирование (ежедневный сброс) — cron / Task Scheduler, примеры в конце
`deploy/demo/reset_demo.sh` (блок SCHEDULING). Рекомендация: **ночной сброс раз в
сутки** (например, 04:00). RPO для демо не важен — ценных данных стенд не хранит.

### 5.3. Ретенция

- Дампы (бэкап + golden) лежат в именованном томе `backups`, не попадают в git/на
  хост. Глубину бэкапов задаёт `KEEP` (демо-дефолт `DEMO_BACKUP_KEEP=7`).
- Логи ротуются (10 МБ × 3 на сервис).
- Демо-правки посетителей не сохраняются дольше периода сброса.

## 6. Соответствие 152-ФЗ для демо-данных

- **Нет реальных ПДн.** Демо-каталог — общеизвестные/вымышленные авторы и издатели
  (это не ПДн). Демо-читатели — **синтетические** записи (вымышленные ФИО, email на
  `example.org`). В демо-стенд **не загружаются** реальные читатели и не
  подключается живой ИРБИС с реальными формулярами.
- **Размещение в РФ.** Сервер демо-стенда — в РФ; синтетические данные не покидают
  юрисдикцию.
- **Демодоступ по заявке** (§7) собирает ПДн заявителя (ФИО, e-mail, телефон,
  учреждение, должность) — это обрабатывается на **публичной странице проекта**
  (issue [#226](https://github.com/Rivega42/biblio/issues/226)) с согласием на
  обработку ПДн и минимизацией; сам демо-стенд этих ПДн не хранит.
- **Логи.** ПДн не должны попадать в операционные логи; на демо это упрощается
  отсутствием реальных ПДн. Общие требования — `deploy/README.md` → чек-лист
  прод-хардненинга и [`docs/design/PILOT_READINESS.md`](../design/PILOT_READINESS.md).

## 7. Выдача демодоступа по заявке (#226)

Связь с issue [#226](https://github.com/Rivega42/biblio/issues/226) («Публичная
страница проекта с документацией и демодоступом»):

1. **Публичная страница проекта** (адрес указывается в заявлении в реестр росПО)
   содержит документацию и **форму заявки на демо**: ФИО, e-mail, телефон,
   учреждение, должность (+ согласие на обработку ПДн по 152-ФЗ).
2. **После заявки** ответственный выдаёт доступ к этому демо-стенду на сервере в
   РФ: либо общий демодоступ (`demo` / `reader1..3`), либо персональная учётка
   через `python -m access.provision` / создание аккаунта.
3. **Для экспертов Минцифры** в заявлении указывается URL демо-стенда и
   демодоступ — это и есть «доступ к работающему экземпляру на сервере в РФ»
   из §3.5 [`REGISTRATION_GUIDE.md`](../legal/REGISTRATION_GUIDE.md).

> Критерий приёмки #226 «демодоступ выдаётся на стенд с сервером в РФ»
> закрывается этим стендом + публичной страницей. Техническая часть (поднять
> демо-контейнер) — этот документ и `deploy/docker-compose.demo.yml`.

## 8. Чек-лист готовности демо-стенда

- [ ] Сервер **в РФ**, домен `.ru`/`.рф`, открыты 80/443.
- [ ] **Реальный** TLS-сертификат (не самоподписанный); HSTS включён в `nginx.conf`.
- [ ] `.env` заполнен: сильные `POSTGRES_PASSWORD` и `APP_SECRET`; `.env` не в git.
- [ ] Демо-стек поднят overlay'ем `docker-compose.demo.yml`; `ps` → все `healthy`.
- [ ] `sh demo/seed_demo.sh` отработал; каталог и демо-читатели на месте.
- [ ] `sh demo/reset_demo.sh --snapshot` снят; cron/Task Scheduler на ежедневный сброс.
- [ ] Проверено: на стенде **нет реальных ПДн**, ИРБИС не подключён.
- [ ] URL демо-стенда и демодоступ переданы для заявления в реестр (#226).
