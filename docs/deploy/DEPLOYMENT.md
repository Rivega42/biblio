# Развёртывание Biblio в контуре библиотеки (СПб ГТБ)

> Полный практический гайд: как поднять контейнерный стек Biblio
> (`proxy` nginx + TLS → `backend` Python: API + собранный SPA → `db`
> `postgres:16`) на **сервере в контуре библиотеки**, подключить его к **живому
> ИРБИС64** библиотеки (или перенести данные в собственное хранилище) и довести до
> приёмки пилота.
>
> Этот документ — про **внутренний пилот в одной библиотеке** (СПб ГТБ / первый
> арендатор). Публичный демо-стенд для реестра росПО описан отдельно
> ([`DEMO_STAND.md`](DEMO_STAND.md)); go-live чек-лист по DoD —
> [`PILOT_READINESS.md`](../design/PILOT_READINESS.md); топология и модель env —
> [`deploy/README.md`](../../deploy/README.md).

**Кратко (TL;DR) для оператора:**

```sh
cd deploy
cp .env.example .env            # 1. заполнить секреты (см. §4) — НИКОГДА не коммитить
sh proxy/gen-cert.sh            # 2. TLS-сертификат (dev — самоподписанный; прод — §3)
docker compose up -d --build    # 3. поднять db + backend + proxy
docker compose ps               # 4. дождаться, пока все три → healthy
```

Затем открыть `https://<ваш-домен>/` (в dev — `https://localhost`).

## 1. Системные требования

### Сервер (хост в контуре библиотеки)

| Ресурс | Минимум (пилот, 1 библиотека) | Рекомендуется | Примечание |
|---|---|---|---|
| **CPU** | 2 vCPU | 4 vCPU | Сборка образа (`vite build` + `npm ci`) кратковременно нагружает CPU; рантайм — лёгкий. |
| **RAM** | 4 ГБ | 8 ГБ | PostgreSQL + Python-бэкенд + nginx. 4 ГБ хватает на пилот; сборка SPA в Docker требует ~1–2 ГБ. |
| **Диск** | 20 ГБ свободно | 40 ГБ+ SSD | Образы (~1–1.5 ГБ) + том PostgreSQL + том бэкапов. Объём растёт с каталогом и историей дампов. Закладывайте запас под `backups`. |
| **ОС** | Linux x86-64 (Ubuntu 22.04+/RHEL 8+/Debian 12) | то же | Поддерживается и Windows Server с Docker Desktop/WSL2, и macOS (dev). Для прод-контура рекомендуется Linux. |

### Программное обеспечение

| Компонент | Версия | Проверка |
|---|---|---|
| **Docker Engine** | ≥ 24.0 | `docker --version` |
| **Docker Compose** | ≥ v2.20 (плагин `docker compose`, не legacy `docker-compose`) | `docker compose version` |
| **OpenSSL** | любой современный (для `gen-cert.sh`) | `openssl version` |
| **git** | для клонирования репозитория и откатов по SHA | `git --version` |

> Стек проверен на Docker 27.4 / Compose v2.31. Команды используют форму
> `docker compose` (v2, плагин). Если на сервере только legacy `docker-compose`
> (v1) — обновите Docker; скрипты бэкапа умеют оба варианта, но документация ниже
> предполагает v2.

### Сеть и порты

| Порт | Направление | Назначение | Открывать наружу? |
|---|---|---|---|
| **443/tcp** | вход на хост | HTTPS (основной доступ к порталу + API) | **Да** (в контур / читателям) |
| **80/tcp** | вход на хост | HTTP → редирект на 443 + ACME `http-01` (Let's Encrypt) | Да (только редирект + ACME) |
| **6666/tcp** (или ваш) | хост → сервер ИРБИС | Подключение бэкенда к **живому ИРБИС64** (опционально) | Нет (внутренний трафик контура) |
| **5432/tcp** | внутри compose-сети | PostgreSQL | **Нет** — не публикуется на хост (по умолчанию) |
| **8080/tcp** | внутри compose-сети | бэкенд | **Нет** — не публикуется на хост (по умолчанию) |

- Наружу (читателям/сотрудникам) открываются **только 80 и 443**. Порты `db`
  (5432) и `backend` (8080) в `docker-compose.yml` закомментированы — сервисы
  общаются по внутренней compose-сети. **Не раскомментируйте их в проде.**
- Если бэкенд подключается к **их живому ИРБИС**, серверу нужен сетевой доступ к
  TCP-порту ИРБИС (обычно 6666) — см. §4.
- Менять порты хоста — через `HTTP_PORT` / `HTTPS_PORT` в `.env` (если 80/443
  заняты).

### Что должно быть на сервере заранее

- Установлены Docker Engine + Compose (root/sudo для установки; рантайм-команды
  можно дать пользователю в группе `docker`).
- Исходники репозитория (склонированы или скопированы в контур): нужны каталоги
  `deploy/` и `irbis-web/` (бэкенд + фронтенд собираются из них).
- Доступ к интернету **на время сборки** (`npm ci`, `pip install`, pull базовых
  образов). Если контур изолирован — соберите образы заранее на машине с доступом
  и перенесите их (`docker save`/`docker load`), либо поднимите внутреннее
  зеркало реестра.

## 2. Быстрый старт

Из корня репозитория:

```sh
cd deploy

# 1. Конфигурация (секреты). .env в .gitignore — НИКОГДА не коммитить.
cp .env.example .env
#    Откройте .env и задайте реальные значения (см. §4):
#    POSTGRES_PASSWORD, APP_SECRET (обязательно), IRBIS_* (если есть мост к ИРБИС).

# 2. TLS-сертификат.
#    Dev / быстрая проверка — самоподписанный:
sh proxy/gen-cert.sh
#    Прод — реальный сертификат вашего домена (см. §3).

# 3. Поднять стек (сборка + запуск в фоне).
docker compose up -d --build

# 4. Дождаться готовности всех трёх сервисов.
docker compose ps
#    Ожидаемо: db, backend, proxy — все три в статусе "healthy".
#    Порядок управляется healthcheck'ами: backend стартует после healthy db,
#    proxy — после healthy backend. Первый старт занимает 1–3 минуты (сборка SPA).
```

### Проверка работоспособности (smoke-test)

После того как `docker compose ps` показал все три сервиса `healthy`:

```sh
# /api/health — без авторизации, возвращает 200 даже при недоступном ИРБИС.
curl -fsk https://localhost/api/health
#    Ожидаемо: JSON-конверт {"ok": true, ...} и HTTP 200.

# Корень портала (SPA) — отдаётся бэкендом.
curl -fsk -o /dev/null -w "%{http_code}\n" https://localhost/
#    Ожидаемо: 200.

# Витрина продукта.
curl -fsk -o /dev/null -w "%{http_code}\n" https://localhost/product
#    Ожидаемо: 200 (SPA-маршрут отдаёт index.html).
```

> `-k` нужен только пока используется **самоподписанный** сертификат (curl не
> доверяет ему). С реальным сертификатом (§3) флаг `-k` убирается.

Откройте в браузере `https://<домен>/` — в dev с самоподписанным сертификатом
браузер предупредит (это ожидаемо для dev; в проде ставится реальный сертификат).

### Остановка / сброс

```sh
docker compose down          # остановить, ТОМА СОХРАНИТЬ (данные остаются)
docker compose down -v       # остановить + УДАЛИТЬ ТОМА (стереть PG + sqlite!)
```

> ⚠️ **Никогда** не запускайте `down -v` на пилоте с реальными данными — это
> безвозвратно стирает тома `pgdata`, `backups`, `backend-data`.

## 3. TLS — сертификат в контуре

Терминацию TLS выполняет `proxy` (nginx) на `:443`; HTTP `:80` редиректит на
HTTPS и обслуживает ACME `http-01` (`/.well-known/acme-challenge/`). Сертификат и
ключ монтируются в контейнер из `deploy/proxy/certs/{server.crt,server.key}`
(файлы в `.gitignore`, не коммитятся).

Есть три сценария — выберите подходящий вашему контуру.

### Вариант A. Самоподписанный сертификат (dev / закрытый контур без публичного домена)

```sh
sh proxy/gen-cert.sh                       # CN=localhost, 825 дней
CERT_CN=biblio.gtb.local sh proxy/gen-cert.sh   # под ваше внутреннее имя
```

Скрипт пишет `proxy/certs/server.crt` + `server.key` (кросс-платформенно:
Linux/macOS **и** Windows git-bash — см. заметку про MSYS path-mangling в шапке
самого скрипта). Браузеры будут предупреждать о недоверенном сертификате — это
**нормально только для dev / закрытого тестового контура**. Для пилота с
реальными читателями используйте вариант B или C.

### Вариант B. Сертификат внутреннего УЦ библиотеки (рекомендуется для контура)

Если у библиотеки есть собственный внутренний удостоверяющий центр (часто так в
госучреждениях):

1. Сгенерируйте на сервере приватный ключ и CSR (запрос на сертификат):
   ```sh
   openssl req -new -newkey rsa:2048 -nodes \
     -keyout proxy/certs/server.key \
     -out /tmp/biblio.csr \
     -subj "/C=RU/O=СПб ГТБ/CN=biblio.sptl.spb.ru"
   ```
   (На Windows git-bash inline `-subj` ломается MSYS path-mangling — задайте субъект
   через конфиг-файл, как это сделано в `gen-cert.sh`, либо выполните шаг на Linux.)
2. Передайте `/tmp/biblio.csr` в УЦ библиотеки, получите подписанный
   `server.crt` (плюс при необходимости цепочку промежуточных сертификатов —
   приклейте их в тот же `server.crt` после серверного).
3. Положите итоговый `server.crt` рядом с `server.key` в `proxy/certs/`.
4. Корневой сертификат вашего УЦ должен быть в доверенных у клиентов (обычно уже
   распространён по рабочим станциям библиотеки через GPO).
5. Перезапустите прокси: `docker compose restart proxy`.

### Вариант C. Let's Encrypt (если у домена есть выход в интернет)

Если домен библиотеки публично резолвится и сервер доступен из интернета по
`:80`:

1. nginx уже отдаёт ACME `http-01` webroot из `/var/www/certbot` (см.
   `proxy/nginx.conf`). Для боевой выдачи добавьте volume под webroot и запустите
   `certbot` (отдельным контейнером или на хосте), указав webroot-путь.
2. Полученные `fullchain.pem` / `privkey.pem` положите как
   `proxy/certs/server.crt` / `proxy/certs/server.key` (или настройте symlink /
   hook на обновление).
3. Настройте автопродление (`certbot renew` по cron) — сертификаты Let's Encrypt
   живут 90 дней.

### После установки реального сертификата (B или C)

- **Включите HSTS:** раскомментируйте в `proxy/nginx.conf` строку
  `add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;`.
  ⚠️ **Не** включайте HSTS, пока стоит самоподписанный сертификат — иначе браузеры
  «запомнят» требование строгого TLS и заблокируют доступ.
- Проверьте, что в `nginx.conf` оставлены только `ssl_protocols TLSv1.2 TLSv1.3`
  (уже задано).
- Перезапустите прокси и проверьте сертификат:
  ```sh
  docker compose restart proxy
  curl -fsv https://<домен>/api/health 2>&1 | grep -E "subject|issuer|HTTP"
  ```

### ⚠️ Честное ограничение по криптографии

Это стандартный OpenSSL-TLS, **не** сертифицированные СКЗИ / ГОСТ-TLS. Если
площадка пилота требует аттестованную криптографию (госсегмент, аттестованный
контур) — это отдельная задача (ГОСТ-TLS / СКЗИ / аттестация по требованиям
ФСТЭК/ФСБ) и **вне области этого пилота**. См.
[`PILOT_READINESS.md`](../design/PILOT_READINESS.md) → «TLS».

## 4. Конфигурация `.env` и подключение к их ИРБИС

Вся конфигурация — через переменные окружения; ничего чувствительного не зашито в
образ. Скопируйте `deploy/.env.example` → `deploy/.env` (в `.gitignore`) и
заполните. Файл `.env` прокидывается в контейнеры базовым `docker-compose.yml`.

### Обязательные секреты (без них стек либо не стартует, либо небезопасен)

| Переменная | Назначение | Как задать в проде |
|---|---|---|
| `POSTGRES_PASSWORD` | пароль суперпользователя PostgreSQL | длинный случайный, напр. `openssl rand -hex 24`. **Не** `devpassword`. Compose требует его явно. |
| `APP_SECRET` | подпись сессий/JWT приложения | `openssl rand -hex 32`. **Не** `dev-insecure-secret-change-me`. |

После заполнения убедитесь, что в `.env` **не осталось** плейсхолдеров
`CHANGE IN PROD` / `change-me`.

### Прочие параметры приложения

| Переменная | Дефолт | Назначение |
|---|---|---|
| `POSTGRES_USER` / `POSTGRES_DB` | `postgres` / `irbis_access` | роль и база Access-стора |
| `APP_ENV` | `prod` | режим приложения |
| `HTTP_PORT` / `HTTPS_PORT` | `80` / `443` | порты хоста для прокси (менять, если заняты) |

### Подключение к их живому ИРБИС64 (мост — опционально)

Бэкенд может работать в двух режимах:

- **Мост к живому ИРБИС** — OPAC-функции (поиск/каталог) идут в нативный сервер
  ИРБИС библиотеки по TCP. Задайте переменные `IRBIS_*` ниже.
- **Own-store** — данные перенесены в собственное хранилище Biblio (см. §5);
  живой ИРБИС не нужен. Оставьте `IRBIS_HOST` пустым.

Переменные моста к ИРБИС:

| Переменная | Дефолт | Назначение |
|---|---|---|
| `IRBIS_HOST` | `host.docker.internal` | хост сервера ИРБИС. **Пусто** → мост выключен (own-store). |
| `IRBIS_PORT` | `6666` | TCP-порт сервера ИРБИС |
| `IRBIS_USER` | `MASTER` | логин сервис-аккаунта ИРБИС |
| `IRBIS_PASS` | *(пусто)* | **пароль сервис-аккаунта** — задать в проде |
| `IRBIS_WORKSTATION` | `A` | код АРМ (workstation) для сессии |
| `IRBIS_DB_DEFAULT` | `IBIS` | БД по умолчанию |
| `IRBIS_PUBLIC_DBS` | `IBIS` | БД, видимые читателю (deny-by-default для остальных) |

#### Сетевая доступность сервера ИРБИС из контейнера

Бэкенд работает **внутри контейнера**, поэтому `127.0.0.1` указывает на сам
контейнер, а не на хост. Варианты, как достучаться до сервера ИРБИС:

- **ИРБИС на той же машине, что и Docker (Docker Desktop / Win/Mac):**
  `IRBIS_HOST=host.docker.internal` (это и есть дефолт).
- **ИРБИС на той же машине, Docker под Linux:** либо
  `IRBIS_HOST=host.docker.internal` с
  `extra_hosts: ["host.docker.internal:host-gateway"]` для сервиса `backend`,
  либо укажите IP хоста в LAN.
- **ИРБИС на отдельном сервере в контуре:** `IRBIS_HOST=<IP-или-DNS-сервера-ИРБИС>`,
  `IRBIS_PORT=<порт>`. Убедитесь, что сетевой маршрут и межсетевой экран
  пропускают трафик от хоста Docker к серверу ИРБИС.

Проверка доступности порта ИРБИС с хоста Docker:

```sh
# заменить host/port на ваши
nc -vz <IRBIS_HOST> 6666        # или: timeout 3 bash -c '</dev/tcp/<host>/6666'
```

Проверка, что бэкенд видит ИРБИС (после `up`): функции OPAC отвечают, а
`/api/health` возвращает 200 в обоих случаях (он намеренно не зависит от ИРБИС,
чтобы недоступность ИРБИС не «роняла» healthcheck).

> Полную модель env, понимаемую бэкендом, см. в `irbis-web/backend/config.py` и
> `access/pgstore.py`. Compose-файл прокидывает переменные за вас.

### Провижининг первого арендатора (после `up`)

Access-стор мультиарендный (control-схема + схема-на-арендатора `t_<slug>`). После
подъёма стека заведите библиотеку пилота:

```sh
docker compose exec backend python -m access.provision <slug> \
  --name "Санкт-Петербургская государственная театральная библиотека" \
  --admin admin --password '<надёжный-пароль-админа>' --plan standard

# проверить список арендаторов:
docker compose exec backend python -m access.provision --list
```

Смените пароль администратора с любого дефолтного — `changeme`/
`ADMIN_DEFAULT_PASSWORD` **не должны** дожить до пилота.

## 5. Миграция из их ИРБИС (own-store режим)

Если библиотека уходит с ИРБИС на собственное хранилище Biblio — данные
переносятся миграционным инструментом. Он **транзиентно** читает исходный
ИРБИС64 (каталог + читатели) и загружает в собственные стора Biblio (те же, из
которых отдаёт приложение). Код приложения при этом **не меняется**.

Инструмент находится в `irbis-web/backend/tools/migrate_irbis.py` и запускается
как модуль **внутри backend-контейнера** (где `tools/` — это пакет на PYTHONPATH):

```sh
docker compose exec backend python -m tools.migrate_irbis …
```

### ПДн и безопасность (важно)

Данные читателей (БД `RDR`) — **персональные данные**. Мигратор:

- **шифрует ПДн при записи** — имя/e-mail/телефон проходят через
  `access.crypto.encrypt` до попадания в стор (в хранилище лежит шифртекст, как у
  штатных данных приложения);
- **редактирует логи** — отчёт содержит только счётчики; пер-записьное
  логирование (`--verbose`) печатает только замаскированные образцы (`Бр***`),
  никогда полное имя/контакт и никогда пароль источника.

### Шаг 0. Интроспекция источника (что и сколько переносить)

Сначала снимите структурный план — какие БД есть и сколько в них записей
(двухфазно: без `--dbs` — быстрый список БД; с `--dbs` — инвентарь полей по
выбранным БД, с пометкой нештатных допполей `custom`):

```sh
# сетевой источник (живой сервер ИРБИС):
docker compose exec backend python -m tools.migrate_irbis \
  --source-host <IRBIS_HOST> --source-port 6666 \
  --user MASTER --pass '<пароль>' --introspect

# инвентарь полей по конкретным БД (фаза 2):
docker compose exec backend python -m tools.migrate_irbis \
  --source-host <IRBIS_HOST> --source-port 6666 \
  --user MASTER --pass '<пароль>' --introspect --dbs IBIS,RDR
```

Локальный режим (без сервера, чтение файлов БД `.mst`/`.xrf` напрямую): вместо
`--source-host/--user/--pass` укажите `--source-path <DataPath>` — путь к каталогу
данных ИРБИС (`Datai`), смонтированному в контейнер.

> Чтобы локальный режим увидел файлы, **примонтируйте** их в backend-контейнер,
> напр. добавив в `docker-compose.yml` для `backend`:
> `volumes: - /путь/к/Datai:/srv/irbis:ro` и указав `--source-path /srv/irbis`.
> Доступ — только на чтение (`:ro`).

### Шаг 1. Dry-run (ничего не пишет)

Прогон, который читает + маппит + считает записи, **не записывая** в стор (ни одна
строка не тронута, ни одни ПДн не сохранены):

```sh
docker compose exec backend python -m tools.migrate_irbis \
  --source-host <IRBIS_HOST> --source-port 6666 \
  --user MASTER --pass '<пароль>' \
  --target-tenant <slug> --dbs IBIS,RDR \
  --dry-run --limit 100 --verbose
```

Сверьте счётчики отчёта (`records_read`, `records_loaded`, `readers_loaded`,
`skipped`, `errors`) с ожиданием по числу записей из интроспекции.

### Шаг 2. Боевой перенос

> **Сделайте бэкап (§6) ПЕРЕД миграцией** — чтобы откатить загрузку, если что-то
> пойдёт не так.

```sh
docker compose exec backend python -m tools.migrate_irbis \
  --source-host <IRBIS_HOST> --source-port 6666 \
  --user MASTER --pass '<пароль>' \
  --target-tenant <slug> --dbs IBIS,RDR --catalog-db IBIS
```

- Перенос **идемпотентен**: повторный запуск обновляет записи на месте (каталог —
  по исходному MFN, читатели — по номеру билета), а не плодит дубли.
- Удалённые/пустые/нечитаемые записи пропускаются и считаются в `skipped`.

### Шаг 3. Проверка

- Сверьте число загруженных записей (`records_loaded` / `readers_loaded`) с
  числом записей источника (из интроспекции).
- Откройте портал и проверьте поиск/карточки на выборке записей.
- Убедитесь, что **ПДн читателей в хранилище — шифртекст** (сырой дамп не должен
  показывать имя в открытом виде).

> Замечание о масштабе: массовый импорт каталога — задача
> **пропускной способности/свежести**, не latency-SLO. Большой фонд лучше
> переносить в окно обслуживания.

## 6. Персистентность, бэкап и restore

### Тома (что и где хранится)

Стек объявляет три именованных тома (переживают пересоздание контейнеров):

| Том | Что хранит | Критичность |
|---|---|---|
| `pgdata` | данные PostgreSQL (Access-стор: аккаунты/гранты/аудит/мультиаренда) | **высшая** — это основное состояние |
| `backend-data` | sqlite-стора движков (notify/catalog/circ/acq/bp) | высокая — рантайм-данные движков |
| `backups` | дампы `pg_dump` (см. ниже) | высокая — точки восстановления |

`docker compose down` сохраняет тома; **только** `docker compose down -v` их
стирает (никогда не запускайте `-v` на пилоте).

### Бэкап (логический дамп PostgreSQL)

Скрипты используют `docker compose exec` против запущенного стека (host-pg_dump не
нужен, креды не попадают в командную строку хоста), дамп пишется в том `backups`:

```sh
sh deploy/backup/pg_backup.sh                 # один дамп с таймстампом → том backups
KEEP=14 sh deploy/backup/pg_backup.sh         # + оставить только 14 новейших
FORMAT=plain sh deploy/backup/pg_backup.sh    # вместо custom-формата — plain SQL (gzip)
```

#### Расписание (обязательно для пилота)

Базовый уровень одноузлового пилота — **ночной логический дамп**:

- **Linux/macOS cron** (02:30, хранить 14 дней):
  ```
  30 2 * * *  KEEP=14 /путь/к/repo/deploy/backup/pg_backup.sh >> /var/log/biblio-backup.log 2>&1
  ```
- **Windows Task Scheduler** (git-bash):
  ```
  schtasks /Create /SC DAILY /ST 02:30 /TN "biblio-pg-backup" ^
    /TR "\"C:\Program Files\Git\bin\sh.exe\" C:\IRBIS64\_recon\deploy\backup\pg_backup.sh"
  ```

#### Правило 3-2-1

Копируйте том `backups` **за пределы хоста** во второе место (и желательно одну
офлайн/неизменяемую копию):

```sh
docker compose -f deploy/docker-compose.yml exec db ls -lh /backups
docker run --rm -v deploy_backups:/b -v "$PWD":/out alpine cp -r /b /out/backups-copy
#   имя тома — "<проект>_backups"; уточнить через `docker volume ls`
```

### Restore (восстановление)

```sh
sh deploy/backup/pg_restore.sh                  # список доступных дампов
sh deploy/backup/pg_restore.sh <файл-дампа>     # восстановить (с подтверждением; FORCE=1 пропустить)
```

⚠️ Восстановление **перезаписывает** объекты целевой БД (custom-формат —
`pg_restore --clean --if-exists`). Скрипт спрашивает подтверждение, если не задан
`FORCE=1`.

#### Проверка restore — бэкап надёжен только после восстановления

Прогон в throwaway-БД **без касания живых данных** (репетиция):

```sh
FORCE=1 RESTORE_DB=irbis_restore_test sh deploy/backup/pg_restore.sh <файл-дампа>
docker compose -f deploy/docker-compose.yml exec db \
  psql -U postgres -d irbis_restore_test -c "\dn"   # схемы на месте?
docker compose -f deploy/docker-compose.yml exec db \
  dropdb -U postgres irbis_restore_test             # убрать
```

Делайте restore-тест **до go-live**, затем **ежемесячно** (фиксируйте прогон).

> ⚠️ **Честное ограничение:** это **только ночной логический дамп**. **Нет**
> непрерывного WAL-архивирования / PITR в одноузловом пилоте, поэтому худший
> случай потери данных (RPO) — до ~24ч. Укажите RPO в SLA пилота. PITR и реплики
> появятся в многоузловой фазе.

## 7. Постура по 152-ФЗ

> Пилот обрабатывает персональные данные читателей (ФИО, контакты, история
> выдач). Ниже — что заложено в продукт и что остаётся на стороне библиотеки как
> оператора ПДн. **Это не юридическое заключение** — финальную модель угроз и
> комплект документов оператор ПДн утверждает сам.

| Требование 152-ФЗ | Что обеспечивает Biblio | Что остаётся библиотеке |
|---|---|---|
| **Персональные данные читателей не публичны** | RDR-данные не отдаются в открытый OPAC; портал показывает только разрешённое грантами. | Проверить настройки видимости/грантов под политику библиотеки. |
| **Шифрование ПДн при хранении** | ПДн читателей шифруются at-rest (V1 seam, `access/crypto.py`); миграция пишет ФИО/контакты как шифртекст. | Хранить `APP_SECRET`/ключи в менеджере секретов; ограничить доступ к серверу. |
| **Шифрование при передаче** | TLS-терминация на nginx (TLS 1.2/1.3). | Поставить реальный сертификат (§3). |
| **Согласие на обработку ПДн** | — | Получение и хранение согласий читателей — процесс библиотеки (вне ПО V1). |
| **Аудит действий** | Access-стор ведёт аудит (схема control/аудит). | Определить срок хранения и доступ к журналам аудита. |
| **ПДн не попадают в логи/телеметрию** | `/api/health` без ПДн; миграция логирует только маскированные образцы. | Проверить, что операционные логи (`docker logs`) не содержат ПДн; настроить ротацию. |
| **Хранение данных в РФ** | Контейнеры разворачиваются на сервере библиотеки — данные физически в контуре в РФ. | Гарантировать, что сервер и любые off-box бэкапы (3-2-1) **остаются в РФ**. |

### ⚠️ Честные ограничения (проговорить с оператором ПДн)

- **Криптография не сертифицирована.** TLS и шифрование приложения — стандартный
  OpenSSL, **не** аттестованные СКЗИ / ГОСТ-крипта. Для аттестованного контура
  (госсегмент) требуется отдельная работа по СКЗИ/аттестации — **вне области
  этого пилота**.
- **Модель угроз и оргмеры** (приказ о назначении ответственного за ПДн,
  политика обработки, уведомление РКН при необходимости) — зона ответственности
  библиотеки как оператора.

См. также [`PILOT_READINESS.md`](../design/PILOT_READINESS.md) → «Известные
ограничения пилота».

## 8. Обновление и откат

### Обновление (новая версия приложения)

Приложение пересобирается из репозитория. В **окно обслуживания**:

```sh
cd deploy
git pull                          # подтянуть новую версию кода (или скопировать новые исходники)
docker compose up -d --build      # пересобрать образы + перезапустить; ТОМА СОХРАНЯЮТСЯ
docker compose ps                 # дождаться healthy
```

- Тома (`pgdata`, `backend-data`, `backups`) **сохраняются** — данные не теряются.
- **Перед** обновлением сделайте бэкап (§6) — это точка отката.
- Зафиксируйте развёрнутый git-SHA на видном месте (для воспроизводимого отката).

#### Миграции схемы БД

Провижининг/миграции схемы выполняются механизмами Access-стора
(`ensure_control_schema()` + миграции арендатора при провижининге/обновлении).
Если новая версия требует ручного шага миграции — это указывается в release notes
этой версии; в общем случае обновление образа схему арендаторов не ломает (стор
применяет свои миграции). Всегда держите свежий бэкап до обновления.

### Откат

- **Откат приложения/конфига:** чекаут предыдущего хорошего коммита (по
  зафиксированному SHA) + `docker compose up -d --build`.
- **Откат базовых образов:** `db` = `postgres:16`, `proxy` = `nginx:1.27-alpine`.
  Не делайте внезапных мажорных апгрейдов; `docker compose pull && docker compose
  up -d` — только в анонсированное окно, тома сохранять.
- **Откат данных:** восстановите последний хороший дамп в живую БД (§6;
  custom-restore делает `--clean --if-exists`). Всегда делайте свежий бэкап
  *перед* рискованным изменением — это и есть точка восстановления.
- **Down/up без потери данных:** `docker compose down` сохраняет тома; `down -v`
  их **стирает** — никогда не запускать `-v` на пилоте.

> Держите этот гайд и план отката **и вне хоста** (копия снаружи восстанавливаемой
> системы).

## 9. Чек-лист приёмки

Отметьте перед сдачей пилота в эксплуатацию.

### Инфраструктура и запуск
- [ ] Сервер соответствует требованиям §1 (CPU/RAM/диск/ОС).
- [ ] Установлены Docker ≥ 24 и Compose v2 (`docker compose version`).
- [ ] `docker compose -f deploy/docker-compose.yml config` проходит без ошибок.
- [ ] `docker compose up -d --build` поднимает стек; `docker compose ps`
      показывает `db`, `backend`, `proxy` как **healthy**.
- [ ] `curl https://<домен>/api/health` → 200; `/` и `/product` → 200.

### Секреты и конфигурация
- [ ] `deploy/.env` создан из `.env.example`, в `.gitignore`, **не закоммичен**.
- [ ] Задан сильный `POSTGRES_PASSWORD` (не `devpassword`).
- [ ] Задан сильный `APP_SECRET` (`openssl rand -hex 32`, не дефолт).
- [ ] Не осталось плейсхолдеров `CHANGE IN PROD` / `change-me` в `.env`.
- [ ] (Если мост к ИРБИС) задан `IRBIS_PASS`, сервер ИРБИС доступен из контейнера.

### TLS
- [ ] Установлен **реальный** сертификат (внутренний УЦ / Let's Encrypt), не
      самоподписанный.
- [ ] Включён HSTS в `nginx.conf` (только после реального сертификата).
- [ ] Только TLS 1.2/1.3.

### Данные
- [ ] (Own-store) выполнена миграция: dry-run сверен, боевой перенос проверен,
      ПДн в хранилище — шифртекст.
- [ ] Провижинен арендатор пилота; пароль админа сменён с дефолтного.

### Эксплуатация
- [ ] Настроено расписание ночных бэкапов (cron / Task Scheduler).
- [ ] Выполнен и зафиксирован restore-тест в throwaway-БД.
- [ ] Настроена off-box копия тома `backups` (3-2-1), **в РФ**.
- [ ] Настроена ротация Docker-логов (диск не забьётся за пилот).
- [ ] Порты `db`/`backend` не публикуются на хост; наружу открыты только 80/443.
- [ ] Назначен человек для ежедневного контроля (`ps`/`logs` + `/api/health`).
- [ ] Этот гайд и план отката сохранены **вне** хоста.

### 152-ФЗ
- [ ] ПДн читателей не публичны; проверены гранты/видимость.
- [ ] Сервер и off-box бэкапы — **в РФ**.
- [ ] Оператор ПДн уведомлён о несертифицированности крипты (V1) и взял оргмеры.

## 10. Траблшутинг

| Симптом | Вероятная причина | Что делать |
|---|---|---|
| **`backend` не становится healthy** | падает старт приложения или нет связи с `db` | `docker compose logs backend`; проверить, что `db` уже `healthy` и `ACCESS_PG_DSN` корректен (пароль = `POSTGRES_PASSWORD`). |
| **`backend`: `password authentication failed for user "postgres"`** при заданном `POSTGRES_PASSWORD` | том `pgdata` инициализирован РАНЬШЕ с другим паролем: PostgreSQL задаёт пароль только при первом init **пустого** тома; существующий том игнорирует новый `POSTGRES_PASSWORD` из `.env`. (Проверено эмпирически при контейнерном смоке.) | Сменить пароль роли в работающей БД: `docker compose exec db psql -U postgres -c "ALTER USER postgres PASSWORD '<пароль из .env>'"`. **Либо** (только если данные неважны, напр. dev/демо) пересоздать том: `docker compose down -v && docker compose up -d`. На реальном пилоте `-v` НЕ применять (сотрёт данные) — только `ALTER USER`. |
| **`db` не становится healthy / не стартует** | не задан `POSTGRES_PASSWORD` | Compose требует его явно — задайте в `.env`. Смотреть `docker compose logs db`. |
| **`proxy` не становится healthy** | нет сертификата / nginx не нашёл `server.crt`/`server.key` | Выполнить `sh proxy/gen-cert.sh` (или положить реальный сертификат), затем `docker compose restart proxy`; `docker compose logs proxy`. **На Windows** запускать скрипт через **Git Bash** (`bash proxy/gen-cert.sh`) — в PowerShell `sh` не распознаётся. |
| **Браузер ругается на сертификат** | самоподписанный сертификат | Норма для dev. Для пилота — поставить реальный сертификат (§3). |
| **`/api/health` отвечает, но OPAC-поиск пустой/ошибка** | ИРБИС недоступен из контейнера | Проверить `IRBIS_HOST/PORT` и сетевой доступ (`nc -vz <host> 6666` с хоста Docker). `127.0.0.1` внутри контейнера ≠ хост — используйте `host.docker.internal` или IP сервера ИРБИС. `/api/health` намеренно остаётся 200 даже при недоступном ИРБИС. |
| **Порт 80/443 занят** (`bind: address already in use`) | другой сервис слушает порт | Освободить порт или задать `HTTP_PORT`/`HTTPS_PORT` в `.env`, перезапустить. |
| **Первый `up` долгий / падает на сборке** | `npm ci` / `pip install` / pull образов без интернета | Дать серверу доступ в интернет на сборку, либо собрать образы заранее и перенести (`docker save`/`load`). |
| **Миграция: `network mode требует --source-host/--user/--pass`** | не передан источник | Указать `--source-host/--user/--pass` (сетевой режим) **или** `--source-path` (локальный режим). |
| **Миграция: `адаптер локального режима … ещё не готов`** при `--source-path` | не виден `tools.irbis_mst` / не примонтированы файлы | Проверить, что каталог `Datai` примонтирован в контейнер (`:ro`) и путь в `--source-path` указывает на него. |
| **`down` стёр данные** | запущен `down -v` | `-v` удаляет тома. Восстановить из бэкапа (§6). Впредь — только `docker compose down` без `-v`. |
| **Бэкап: `the 'db' service is not running`** | стек не поднят | Поднять стек (`docker compose up -d db`) перед бэкапом/restore. |

Полезные команды диагностики:

```sh
docker compose ps                          # статусы и health всех сервисов
docker compose logs -f backend             # живые логи бэкенда
docker compose logs --tail=100 proxy       # последние 100 строк прокси
docker compose exec db psql -U postgres -d irbis_access -c "\dn"   # схемы в БД
docker compose config                      # как Compose видит итоговую конфигурацию
```

## 11. Что остаётся человеку

Это ПО и документация готовят стек к развёртыванию, но **физическую
инфраструктуру в контуре библиотеки заводит человек**. Остаётся на стороне
библиотеки / интегратора:

1. **Реальный сервер в контуре** — выделить хост (§1), установить Docker + Compose,
   разместить **в РФ**, за межсетевым экраном площадки.
2. **Домен и DNS** — внутреннее или публичное имя (напр. `biblio.<домен>`),
   резолвинг в контуре.
3. **TLS-сертификат** — выпуск через внутренний УЦ библиотеки или Let's Encrypt
   (§3); распространение корневого сертификата УЦ по рабочим станциям (если
   внутренний УЦ).
4. **Сетевой доступ к их живому ИРБИС** (если нужен мост) — маршрут и правила
   межсетевого экрана от хоста Docker к серверу ИРБИС (порт 6666); либо решение
   уйти в own-store с миграцией (§5).
5. **Секреты** — сгенерировать и положить в менеджер секретов (Vault/SOPS) или, для
   пилота, в закрытый `.env` (`chmod 600`) на доверенном хосте; задокументировать,
   у кого доступ.
6. **Расписание бэкапов + off-box копия** — настроить cron/Task Scheduler и второе
   (в РФ) место хранения дампов (3-2-1).
7. **Оргмеры 152-ФЗ** — назначить ответственного за ПДн, утвердить политику
   обработки и модель угроз, организовать сбор согласий; при необходимости —
   аттестованную крипту (вне V1).
8. **Эксплуатационный контроль** — назначить человека на ежедневный мониторинг
   (ручной: `ps`/`logs` + `/api/health`), пока нет стека метрик/алертов.
