## 2. Архитектура (кратко)
- **Мультиарендный SaaS:** облачное ядро + локальные узлы (edge) на местах с офлайн-работой и синхронизацией; тонкие библиотеки могут работать только в облаке.
- **Изоляция данных по тенантам:** схема-на-тенанта в PostgreSQL (control-схема + `t_<slug>`); в общих облачных сервисах — `tenant_id` + RLS.
- **Модульность:** функции подключаются per-tenant/по тарифу (entitlements).
- **Единый шов:** REST-контракт (OpenAPI); веб-клиент (SPA/PWA) → API-шлюз → сервисы → PostgreSQL.
- **Безопасность контура:** без внешних CDN/трекеров; обновления edge-узлов подписаны; секреты только в переменных окружения.

## 3. Технические характеристики
| Параметр | Значение |
|---|---|
| Языки программирования | Python (бэкенд), JavaScript/TypeScript — React (фронтенд) |
| Серверная часть | Python-сервер (`server.py`): JSON API (`/api/*`) + отдача собранного SPA |
| Клиент | Веб-браузер; SPA на React (Vite-сборка); PWA-офлайн |
| СУБД | PostgreSQL 16 (Access-стор: учётки/гранты/аудит/мультиаренда); JSONB + полнотекстовый поиск (`tsvector`, `pg_trgm`) |
| Поиск | PostgreSQL FTS (edge) / OpenSearch (облако, кросс-тенант — опц.) |
| Хранилище объектов | S3-совместимое (MinIO) — обложки, полные тексты/PDF |
| Развёртывание | Docker + docker-compose (узел: nginx-TLS-proxy → backend → PostgreSQL); Kubernetes (облако) |
| ОС (сервер) | Linux; поддержка российских доверенных ОС (Astra Linux и др. — подтвердить ≥2 по критерию «м») |
| Инфраструктура хранения кода/сборки/ключей | на территории РФ (критерии «и», «к») |
| Интерфейс | на русском языке (критерий «л») |

## 4. Функциональные возможности (по модулям)
> Привести к фактическому составу регистрируемой версии; ниже — реализованная функциональность (движки покрыты автотестами, см. `docs/design/ROADMAP_MVP.md` §3).

**Каталогизация (ядро):**
- ведение библиографических записей (поля/подполя в JSONB), рабочий лист каталогизатора;
- формально-логический контроль ввода (ФЛК), декларативные правила;
- управление экземплярами фонда (поле 910);
- авторитетные файлы/словари (связи по ^3); глобальная корректировка (.gbl), в т.ч. cross-DB;
- форматирование вывода (PFT/UNIFOR).

**Учёт фонда и размещение (модуль):**
- экземпляры; иерархическое хранение (адрес→здание→…→стеллаж→полка→ячейка); план помещения.

**Книговыдача (модуль, критична офлайн-работа):**
- выдача, возврат, продление по читательскому билету;
- формуляр читателя (поле 40), очередь бронирования, бронеполка, штрафы;
- быстрый ввод по сканеру; сценарии RFID/ТСД (модуль);
- диспетчеризация уведомлений из книговыдачи.

**Комплектование и книгообеспеченность (модуль):**
- заказ → поступление → КСУ (книга суммарного учёта) → передача в каталог / списание;
- учёт книгообеспеченности дисциплин (ККО) для вузов.

**Читательский веб-портал (OPAC):**
- поиск (несколько режимов), фасеты, навигаторы-классификаторы, карточка записи;
- личный кабинет/формуляр, онлайн-бронирование и очередь, уведомления (инбокс);
- пользовательские полки, постоянные ссылки (permalink), мульти-layout.

**Администрирование и доступ (ядро):**
- управление пользователями, ролями и грантами (функция × база × уровень);
- провижининг арендатора (создать тенант → схема `t_<slug>` → сидинг словарей → админ-аккаунт → гранты/модули);
- аудит-журнал действий; публично видимые базы по allow-list.

**Платформа/коммерция (ядро/модули):**
- мультиарендность; тарифные планы → entitlements per tenant → лимиты использования (записи/читатели/хранилище) и их enforcement;
- синхронизация edge↔облако (outbox/журнал событий, разрешение конфликтов); уведомления; (опц.) аналитика/BI.

**Опциональные модули:** оцифровка/DAM, AI-навигатор по справке, интеграции (OAI/discovery, MARC/Z39.50 — по дорожной карте), оплата.

## 5. Информационная безопасность и соответствие
- Хэширование паролей (argon2/bcrypt); TLS на канал API (терминация на reverse-proxy); журнал доступа к персональным данным; секреты в переменных окружения (не в коде/образах); обработка ПДн читателей в логике 152-ФЗ. Подробнее — `docs/design/AUDIT_REPORT_IRBIS64_compliance.md`, `deploy/README.md` (чек-лист прод-хардненинга).
- Отсутствие принудительного обновления/управления из-за рубежа; обновления edge-узлов подписаны (критерий «ж»).
