# Чем плох ИРБИС и почему наш продукт будет лучше (обоснование)

> Проектное обоснование (статус П). Критика **обоснована нашим ресерчем** (ссылки на `docs/recon/deep/` и `docs/recon/`), а не общими словами. Принцип честности: ИРБИС — **зрелая система с очень глубокой предметной логикой**, проверенной десятилетиями; мы критикуем **технологию, UX, безопасность и эксплуатацию**, но **сохраняем предметную модель** (её и восстанавливал весь ресерч). Каждая слабость → что делаем мы → почему лучше → связь с issue.

## 0. Что у ИРБИС сильного (СОХРАНЯЕМ, не выбрасываем)
- **Глубочайшая библиотечная логика**: книговыдача, комплектование, КСУ, книгообеспеченность, МБА, авторитетный контроль, сводный каталог — выстраданная годами модель (`DB_*`, `FINDINGS_05..08`). Это главная ценность, её переносим.
- **Гибкость через конфиги**: всё настраивается без перекомпиляции (49 БД, профили, форматы).
- **Мощный поиск** и инвертирование (FST), поддержка **российских стандартов** (RUSMARC, ГОСТы, ГРНТИ/УДК/ББК), авторитетные файлы.
- Единая точка форматирования (серверный PFT).
> Вывод: мы делаем не «другой каталог», а **ту же предметную модель на здоровом фундаменте**.

---

## 1. Технологический стек и архитектура
**Плохо:** сервер только под Windows (Delphi/Pascal, win32-DLL), процессная форк-модель (`irbis_server.exe`→`server_64.exe`), **закрытый бинарный протокол `:6666`** с недокументированным проводным форматом, лимиты лицензий (демо — 3 клиента). **Вендора больше нет, исходников сервера нет.**
**Обоснование:** `FINDINGS_04` (архитектура/процессы), `PROTOCOL_REFERENCE §7` (wire-формат не описан), бриф §1 (вендор/исходники).
**Мы:** кроссплатформенно (Linux/контейнеры, Astra), открытый код, **документированный HTTP/REST API (+OpenAPI)**, горизонтальное масштабирование, без vendor lock-in.
**Почему лучше:** нет привязки к Windows и к умершему вендору; систему можно развивать, масштабировать и аудировать. → эпики [#33](https://github.com/Rivega42/biblio/issues/33)/[#34](https://github.com/Rivega42/biblio/issues/34).

## 2. Клиенты и UX
**Плохо:** 6 толстых **десктоп-АРМов** (Windows), интерфейс разрезан «по АРМам», а не по задачам; web — **J-ИРБИС на EOL-стеке** (PHP 5.6 / Joomla 2.5 / Apache 2.4.38, всё снято с поддержки), локализация библиотечной части не реализована, конфиг = ручная правка PHP-файлов, обложки тянутся из внешнего интернета, в истории версий — баги именно в читательских сценариях.
**Обоснование:** `JIRBIS2_BEHAVIOR §1` (стек EOL), `§7` (нет i18n), `§8` (дефекты, правка PHP, обложки).
**Мы:** **единое веб-приложение по учёткам/грантам** (АРМ — лишь контекст протокола), современный фронт (React/TS), декларативный конфиг (не код), i18n, кэш/прокси обложек.
**Почему лучше:** один тонкий клиент с любого устройства; сотрудник совмещает функции по грантам, а не «прыгает по 5 программам». → `ACCESS_MODEL`, эпик [#33](https://github.com/Rivega42/biblio/issues/33).

## 3. Доступность (a11y) — критично для госкаталога
**Плохо:** **версии для слабовидящих / ГОСТ Р 52872-2019 / WCAG в J-ИРБИС НЕТ** (есть только адаптив/киоск).
**Обоснование:** `JIRBIS2_BEHAVIOR §7` (явно: accessibility отсутствует).
**Мы:** доступность как **базовое требование** (ГОСТ Р 52872/WCAG 2.1 AA), семантическая вёрстка, клавиатура, скринридеры, контраст.
**Почему лучше:** соответствие закону для публичного государственного каталога; реальная инклюзивность. → issue [#14](https://github.com/Rivega42/biblio/issues/14) (D7).

## 4. Безопасность
**Плохо:** пароли клиентов **открытым текстом в `*.ini`** (был реальный инцидент утечки боевых кред/ПДн в публичный репозиторий); права читателей — **MD5 без соли** (J-ИРБИС); ссылки на полные тексты (HTTP/FTP) **отдаются напрямую** («степень защиты низкая»); часть прав `RIGHT` не поддерживается; гейт прав — грубый, по типу АРМа.
**Обоснование:** `RECON_FINDINGS_02 §1` (креды в INI), `JIRBIS2_BEHAVIOR §4-5` (MD5 без соли, прямые ссылки, RIGHT частично), бриф §8 (прошлая утечка).
**Мы:** секреты **только в env/секрет-менеджере**, секрет-сканеры (gitleaks) в CI, современная аутентификация (хэш с солью/Argon2, OIDC/SSO), **проксирование файлов с проверкой прав** (никаких прямых ссылок), гранулярная модель доступа, аудит write/admin.
**Почему лучше:** класс прошлой утечки исключён архитектурно; данные читателей и контролируемый контент реально защищены. → issue [#11](https://github.com/Rivega42/biblio/issues/11)/[#20](https://github.com/Rivega42/biblio/issues/20).

## 5. Формат данных и язык форматирования (PFT)
**Плохо:** проприетарный MARC-подобный формат с числовыми метками + **эзотерический язык PFT** (наследие CDS/ISIS, 41+ UNIFOR-кодов), которым надо владеть, чтобы сделать любой вывод; ФЛК-валидация — на том же языке; «магические» поля (920 = тип записи). Высокий порог входа, хрупкость, знания «в головах ирбисоводов».
**Обоснование:** `PFT_LANGUAGE.md` (44 конструкции + 41 UNIFOR), `FIELD_CATALOG` (3366 подполей — масштаб сложности), `FLC_VALIDATION.md`.
**Мы:** данные — в нормальной модели (реляционная/JSON), **рендер — современные декларативные шаблоны**, валидация — декларативные схемы/правила; совместимость через **импорт/экспорт MARC/ISO2709**, а не через перенос PFT-DSL в код приложения.
**Почему лучше:** правки форм/валидаций делает обычный разработчик/администратор, а не редкий эксперт; ниже порог, выше надёжность. → issue [#8](https://github.com/Rivega42/biblio/issues/8)/[#10](https://github.com/Rivega42/biblio/issues/10).
> Нюанс: для **пилота-моста** (ADR-003) мы временно используем серверный PFT существующего сервера — это осознанный компромисс, чтобы не переписывать форматы на старте; в своём продукте — заменяем.

## 6. Поиск (UX)
**Плохо:** движок мощный, но **язык запросов эзотеричный** (префиксы `A=/K=/T=`, операторы `(G)(F).`); «свободный поиск для чайников» (`DS=`) в ИРБИС64+ **не поддерживается**; ранжированный поиск **ограничен 100 записями**; **сортировка «значительно замедляет поиск»** (предупреждение самого вендора).
**Обоснование:** `PROTOCOL_REFERENCE §5` (грамматика), `JIRBIS2_BEHAVIOR §2` (DS= не поддержан, лимит 100, сортировка тормозит).
**Мы:** современный индекс (релевантность, **фасеты без жёсткого лимита**, быстрая сортировка/пагинация на стороне индекса), простой UX + экспертный режим, морфология; язык запросов — деталь реализации, не порог для пользователя.
**Почему лучше:** быстрый, предсказуемый поиск без «магии префиксов» и без деградации на сортировке. → issue [#9](https://github.com/Rivega42/biblio/issues/9).

## 7. Файлы, полные тексты, изображения
**Плохо:** файлы на диске, привязка хрупкими полями 950/951/952/955 + 11-й параметр PAR; **запись бинарей по протоколу невозможна** (нужен прямой файловый доступ); постраничный просмотр PDF «на лету» **ненадёжен** (не для всех PDF, может висеть >5 мин); права на файлы частично.
**Обоснование:** `PROTOCOL_REFERENCE §3.1` (`IC_putresourse` только текст), `JIRBIS2_BEHAVIOR §4` (PDF ненадёжно, права).
**Мы:** **объектное хранилище** (S3-совместимое), надёжная серверная нормализация/рендер PDF, кэш превью/обложек, проксирование с правами, **ячеистое хранение** экземпляров.
**Почему лучше:** масштабируемое надёжное хранилище медиа с контролем доступа. → issue [#13](https://github.com/Rivega42/biblio/issues/13)/[#39](https://github.com/Rivega42/biblio/issues/39).

## 8. Расширяемость и специфические базы библиотек
**Плохо:** каждая библиотека лепит свои базы **ручной правкой десятков конфигов** (`.fst/.pft/.wss/.mnu/.opt`) — мощно, но требует эксперта-ирбисовода, плохо версионируется, легко сломать; знания не формализованы.
**Обоснование:** сам объём ресерча — 49 БД, ~11k конфигов, 3366 подполей (`reference/databases/`, `inventory.csv`) — это и доказательство мощи, и доказательство сложности/порога.
**Мы:** **профили баз как версионируемые данные/конфиг** + UI-конструктор + валидация схем; добавление профильной базы (театральная/мед/вуз) — без правки кода.
**Почему лучше:** расширяемость без «шаманства», с историей изменений и проверками. → issue [#12](https://github.com/Rivega42/biblio/issues/12).

## 9. Хранилище данных, целостность, надёжность
**Плохо:** своя бинарная БД (`MST/XRF/IFP/L01/N01`), **ручная реорганизация словаря** (АРМ Администратор), риск рассинхронизации словарь↔мастер-файл, блокировки/версии записей вместо полноценных транзакций.
**Обоснование:** `FINDINGS_12` (структура файлов), `PROTOCOL_REFERENCE §3.8` (adm-реорганизация словаря/мастера вручную).
**Мы:** **ACID-СУБД (PostgreSQL)** — транзакции, автоиндексация, бэкап/репликация, целостность на уровне БД; поиск — отдельный индекс, синхронизируемый автоматически.
**Почему лучше:** надёжность и целостность «из коробки», без ручных реорганизаций. → issue [#21](https://github.com/Rivega42/biblio/issues/21)/[#22](https://github.com/Rivega42/biblio/issues/22).

## 10. Интеграции, стандарты, API
**Плохо:** **закрытый бинарный протокол** (wire не документирован), интеграции через Z39.50/OAI лишь частично, нет современного REST/GraphQL, web — только через устаревший шлюз.
**Обоснование:** `PROTOCOL_REFERENCE §7`, `RECON_FINDINGS_02` (Z39.50 yaz.dll), `JIRBIS2_BEHAVIOR §1`.
**Мы:** открытый **HTTP/REST(+OpenAPI)**, вебхуки, стандартные форматы (MARC/Dublin Core/SRU, опц. RDF), интеграция с ЭБС/ЭИОС/СНГ-системами, токен-SSO.
**Почему лучше:** система — часть экосистемы вуза/библиотеки, а не «вещь в себе».

## 11. Эксплуатация и сопровождение
**Плохо:** вендора нет → нет поддержки/исходников/обновлений; обновления J-ИРБИС **ломаются при локализации/правках**; нет CI/тестов/наблюдаемости; документация разрозненная (`.doc/.chm`, часть в нечитаемом виде — мы сами столкнулись: CHM не извлекается, иллюстрации теряются).
**Обоснование:** бриф §1, `JIRBIS2_BEHAVIOR §7-8`, `OPEN_QUESTIONS #R01/#R04` (CHM/иллюстрации).
**Мы:** открытый код, контейнеры, **CI/CD + тесты + наблюдаемость**, живая документация (этот репозиторий), обычная веб-эксплуатация.
**Почему лучше:** систему можно поддерживать и развивать командой, а не молиться на исчезнувшего вендора.

## Итог — чем наш продукт лучше (одной таблицей)
| Измерение | ИРБИС | Наш продукт |
|---|---|---|
| Платформа | Windows-only, вендор исчез | Linux/контейнеры, открыто |
| Клиент | 6 десктоп-АРМ + EOL-web | одно веб-приложение по грантам |
| Доступность | нет (ГОСТ/WCAG) | базовое требование |
| Безопасность | креды в INI, MD5 без соли, прямые ссылки | env-секреты, соль/SSO, прокси+права, CI-сканеры |
| Форматирование | эзотерический PFT-DSL | декларативные шаблоны/схемы |
| Поиск | мощный, но «магия» + лимиты | современный индекс, фасеты, скорость |
| Медиа | хрупко, PDF ненадёжно | объектное хранилище, надёжный рендер, ячейки |
| Расширяемость | ручная правка конфигов | профили-данные + конструктор |
| Хранилище | бинарь + ручная реорг | ACID-СУБД, авто-индекс |
| API/интеграции | закрытый бинарный протокол | HTTP/REST+OpenAPI, стандарты |
| Сопровождение | без вендора, без CI | open-source, CI/CD, тесты |

> **Главный тезис:** ИРБИС силён предметной логикой и слаб технологически/эксплуатационно. Мы **берём его предметную модель** (ради этого и весь обратно-инженерный ресерч) и **ставим её на современный, безопасный, доступный и поддерживаемый фундамент**. Это не «ещё один ИРБИС», а его суть без его технологического долга.
