# Модель доступа веб-ИРБИС — учётки, функции, базы (организующий стержень)

> Принцип (по предложению заказчика): **функции и доступ к базам привязаны к учётке**, а не к жёстким
> ролям. Учётка несёт набор грантов; сотрудник может совмещать много функций и доступов к разным базам.
> Роли = опциональные пресеты грантов. Это заменяет деление веб-версии «по АРМам» на деление «по
> учёткам/грантам». Опирается на карту возможностей `IRBIS_CAPABILITY_MAP_v2.md`. Версия 1.0, 20.06.2026.

---

## 1. Базовый принцип

**Разрешение = (функция × база × уровень).** На учётке — произвольный набор таких грантов.

- **Функция** — что можно делать (поиск, заказ, правка записи, выдача, …). Каталог — §3.
- **База** — к какой БД применяется (EK, GUAR, RDR, …) или «*» (все/группа).
- **Уровень** — нет / чтение / запись / админ.

**Роль** — именованный пресет грантов («Каталогизатор» = набор функций каталогизации на заданных
базах). Удобно назначать, но истина — на учётке: итоговые права = объединение всех грантов/пресетов.

> Пример: учётка Иванова = {выдача:RDR:запись} + {каталогизация:EK:запись} + {поиск:GUAR:чтение}.
> Без «роли» — просто три гранта.

## 2. Привязка к реальности ИРБИС (важно)

Веб-учётка — абстракция над протоколом; под ней действуют ограничения сервера:

1. **Гейт по типу АРМа.** Команды протокола доступны только при регистрации сессии нужным типом АРМа
   (`IRBIS_READER/CATALOG/COMPLECT/BOOKLAND/BOOKPROVD/ADMINISTRATOR`); иначе `-3338`. → Каждая функция
   (§3) имеет **требуемый АРМ-контекст**. Бэкенд под грант учётки открывает сессию нужного типа
   служебными кредами (из env), проверив грант.
2. **Видимость/права баз** у вендора — через `dbnam1.mnu`/`dbnam3.mnu` (реестры) и `client_m.mnu`
   (права клиентов). → Измерение «база × уровень» бэкенд проверяет до вызова и сопоставляет с реестром.
3. **Хранение учёток:**
   - *Читатели* — в БД `RDR` (вход по фамилии+билету); грант по умолчанию = {поиск/заказ/ЛК: видимые
     базы: чтение}. Гость — без персональных гgrантов (без ЛК/корзины).
   - *Сотрудники* — собственное хранилище учёток веб-приложения (логин + гранты), при необходимости
     сопоставленное с клиентами сервера (`client_m.mnu`). Не плодим веб-логины как ИРБИС-клиентов 1:1 —
     бэкенд держит служебные АРМ-креды и выступает посредником.

## 3. Каталог функций (из инвентаря АРМов → грант + требуемый АРМ-контекст)

| Домен | Функция | Требуемый АРМ-контекст |
| --- | --- | --- |
| Каталог/поиск | поиск, просмотр записи, автодополнение/словарь | READER |
| | просмотр полного текста/файла (с учётом RIGHT) | READER |
| | экспорт, печать/табличные формы | READER (print — также через IC_print) |
| Читательские | оформить заказ, корзина, ЛК/формуляр, закладки, постоянные запросы | READER |
| Книговыдача | очередь заказов (отбор/отказ/переадресация) | BOOKLAND |
| | выдача / возврат / продление / посещение | BOOKLAND |
| | регистрация/корректировка читателя, право обслуживания | BOOKLAND |
| | должники, бронеполка, штрих-коды, статистика спроса | BOOKLAND |
| Каталогизация | создать / редактировать / удалить запись | CATALOG |
| | глобальная корректировка, работа с рабочими листами | CATALOG |
| Комплектование | регистрация поступления, КСУ, перенос в каталог, выходные формы | COMPLECT |
| Книгообеспеченность | дисциплины/контингенты VUZ, учебный план, отчёты | BOOKPROVD |
| Администрирование | базы (новая/опустошить/удалить/импорт/экспорт) | ADMINISTRATOR |
| | актуализация/реорг словаря и файла документов, диагностика | ADMINISTRATOR |

> Гранулярность уровня (чтение/запись/удаление/админ) уточняет функцию: напр. «каталогизация:чтение» =
> видеть запись в РЛ без правки; «каталогизация:запись» = создавать/редактировать; «:админ» = удалять/gbl.

## 4. Как бэкенд применяет грант (поток)

```
запрос(учётка, функция, база) →
  1. найти грант (функция×база×уровень) у учётки → нет → 403 (не светим детали)
  2. определить требуемый АРМ-контекст функции (§3)
  3. открыть/взять из пула серверную сессию этого АРМ-типа (служебные креды из env)
  4. проверить видимость/уровень базы (реестр dbnam, права)
  5. выполнить команду протокола (IC_*) → вернуть результат
```

Аудит: каждое write/admin-действие журналируется (кто, что, где). Контур: write/admin — только
внутри контура; читательский слой может быть шире (публичный каталог).

## 5. Что это меняет в архитектуре

- Веб-приложение **одно**, а не «5 АРМов». Навигация/экраны собираются **по грантам учётки**: показываем
  только то, что разрешено. Инвентарь экранов/операций берём из карты АРМов (он не пропадает — он
  становится «меню функций», доступных по грантам).
- В `TZ_CLAUDE_CODE` ядро дополняется **слоем доступа**: хранилище учёток+грантов, резолвер
  «функция→АРМ-контекст», проверка «база×уровень», пул серверных сессий по АРМ-типам, аудит.
- Модули-«АРМы» из плана фаз остаются как **группы функций**, но включаются не пакетом, а по грантам.

---

## 6. Открытые вопросы по доступу

- Точная гранулярность уровней на функцию (где нужен отдельный «удалить»/«gbl») — уточнить на боевом
  процессе СПб ГТБ.
- Сопоставление сотрудничьих веб-учёток с `client_m.mnu` сервера (один служебный АРМ-логин на тип, или
  пер-сотрудник) — решить по политике безопасности контура.
- Реальный набор пресетов-ролей — по штатке СПб ГТБ (кто что совмещает).
- Видимость баз для читателя — из `dbnam3.mnu`; для сотрудника — из `dbnam1.mnu` + гранты.
