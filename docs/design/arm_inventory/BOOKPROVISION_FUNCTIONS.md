## Топ-5 🟡 (частичных — наибольший риск недопаритета)

1. **Блок 7 целиком — поиск как UX-функция (19 🟡 + 24 ❌).** ws4 описывает префиксы только как доменную модель (§1), но **не специфицирует экран поиска КО**: 43 точки `ko.ini` + связочные переходы [SEARCHKO] (контингент→книги→студенты в трёх БД одновременно — ядро АРМа). Без этого библиотекарь не сможет находить дисциплины/контингенты по факультету/кафедре/циклу. **Действие: завести SPEC экрана поиска ws4 с маппингом всех 43 префиксов и межбазовых переходов.**

2. **Блок 8 — отчёты на полевом уровне (16 🟡).** Названы группы (KoDiscFull/LicenDisc/GRTabPart1-4), но **не расписаны колонки/агрегаты/формы лицензирования** (части 1/1.1/2/3/4 + итоговые по циклам). Лицензионные таблицы — регуляторное требование вуза. **Действие: SPEC таблиц лицензирования с точными колонками из `KoPart*`/`KoItogCycleN`.**

3. **4.x — УПЛ и РП на полевом уровне (5 🟡).** Типы UPL/WPD упомянуты, но РЛ (UPL: 91-99/910; WPD: 110/205/210/215/701/702/900) **не разложены построчно**; жизненный цикл РП (версии/замена/продление `ProlongWP`) отсутствует. **Действие: дополнить ws4 §2 полевыми РЛ UPL и WPD.**

4. **6.9/6.10 — формулы KoStatEkzKmi/KoKmiEkz/693 (golden TODO #VUZ-01/03).** Формулы реконструированы как **кандидатные** (E1 §2.5, R1); сами `.pft` и воркшит `Kofor693.wss` физически в БД каталога, не извлечены. Расхождение исказит весь расчёт обеспеченности. **Действие: извлечь `.pft`/`.wss` из боевого конфига тенанта и закрыть golden-тестами (E1 §9).**

5. **3.11 — поле 90 RDR (структура подполей, TODO #VUZ-07).** Синхронизация контингента с RDR покрыта на уровне потока (LinkVuz→MoveVuzRdr), но **полная структура подполей поля 90** (доп. контингент студента) реконструирована только по заданиям VUZ. **Действие: взять справочник поля 90 из БД RDR и сверить с `MoveVuzRdr.gbl`.**
