# Архитектура собственного сервера (замена сервера ИРБИС64)

> Цель: заменить вендорский сервер ИРБИС64 нашим сервисом, **сохранив уже готовый веб-клиент** (он общается через [API_CONTRACT](../build/API_CONTRACT.md)). Переход — по частям (strangler), с миграцией данных из ИРБИС через адаптер Прохода Б.

## 1. Принцип объединения с текущей задачей
Один фронт — два взаимозаменяемых бэкенда за одним контрактом:
```
            ┌───────────── REST /api/* (API_CONTRACT) ─────────────┐
   фронт →  │  backend/  (адаптер к ИРБИС64, :6666)   ← демо «J-ИРБИС» │
            │  server-own/ (наш сервер, PostgreSQL)   ← наш продукт   │
            └──────────────────────────────────────────────────────┘
```
Демо завтра идёт на `backend/`. Свой сервер растёт параллельно как `server-own/`, обслуживая те же запросы из своей БД; базы переключаются по мере миграции и сверки.

## 2. Модель данных (наш сервер)
- **Запись** = устойчивый id + база + **поля/подполя** (сохраняем MARC-совместимую модель ИРБИС для точной миграции и round-trip): `record(id, db, mfn_legacy, version, status, fields JSONB, created_at, updated_at)`, где `fields` = `[{tag, subfields:{code:value}}...]`.
- **Материализованные колонки** для скорости/выдачи: `title, author, year, doctype, isbn, availability, has_cover` + `search_vector tsvector`.
- **Экземпляры** нормализованы отдельно (под ячеистое хранение #39): `holding(record_id, inv_no, location, cell, status, rfid)`. Это место, где наш сервер сразу лучше ИРБИС (ячейки/RFID — первый класс).
- **Словари/авторитетные файлы** — отдельные таблицы (миграция ATHR*/MNU/TRE).
- **Обложки/файлы** — объектное хранилище (S3-совместимое) или BLOB; ссылки в записи.
- Dev — sqlite (как ADR-004), prod — **PostgreSQL** (JSONB + GIN/tsvector + pg_trgm для префиксного поиска), при росте — OpenSearch для полнотекста.

## 3. Поиск
- Префиксные индексы ИРБИС (`K=`/`A=`/`T=`/`V=`…) маппятся на: материализованные колонки + `tsvector` (полнотекст) + `pg_trgm` (префикс/усечение `$`). Грамматику (И/ИЛИ/НЕ, усечение) транслируем в SQL/FTS.
- Контракт `/api/search` не меняется; меняется только реализация под капотом.

## 4. Рендеринг
- Карточку фронт уже строит из структурированных полей (не нужен серверный PFT). Для совместимости можно держать PFT-совместимый рендер-модуль, но это опционально — наш сервер отдаёт чистые данные.

## 5. Миграция из ИРБИС (через адаптер Прохода Б)
1. Перечень баз — `/api/databases` (реестр `dbnam`). 
2. Для каждой базы: читать записи (`IC_search "I=$"` → MFN-список → `read_record`), трансформировать поля/подполя → наша модель, **upsert** с сохранением `mfn_legacy` (для сверки и обратных ссылок).
3. Обложки (953) и файлы (951/955) — выгрузить в объектное хранилище.
4. Инкрементальная синхронизация по дате правки (поле 907/`updated`), пока обе системы живут параллельно.
5. **Сверка**: число записей, контрольные выборки, дифф полей адаптер↔наш сервер на одинаковых запросах. Базу переключаем на свой сервер только после зелёной сверки.
6. PoC уже в репозитории: `irbis-web/server-own/migrate.py` льёт записи IBIS из живого сервера в наш стор.

## 6. Дорожная карта (увязка с issues)
- Контракт-совместимый слой — #24. Миграция — отдельный эпик. Ячеистое хранение — #39 (в нашей модели — первоклассно). 
- Этапы: (а) свой сервер обслуживает **чтение** (search/record) одной базы из мигрированных данных → демо «наш сервер»; (б) запись/каталогизация на нашем сервере; (в) книговыдача/заказы (RQST→наша модель, ячейки); (г) полный cutover, ИРБИС в архив.

## 7. Что сделано как PoC сейчас
`irbis-web/server-own/` — самостоятельный сервер (sqlite+FTS, stdlib), реализует контракт (`auth/guest`, `health`, `databases`, `search`, `terms`, `record`, `cover`) и отдаёт тот же фронт. `migrate.py` наполняет его реальными записями IBIS через адаптер. Итог: **тот же веб-клиент работает и на ИРБИС, и на нашем сервере** — это и есть объединение демо и продукта.
