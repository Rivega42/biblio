> **Цель.** Пользователь ощущает наш портал «не наполненным». Этот документ отвечает: **больше ли в прототипе**, и **недо-спроектирован ли наш портал или просто недо-реализован**.
> Сравниваются три источника:
> - **ПРОТОТИП** — `C:\Users\admin\AppData\Local\Temp\irbis_proto\irbis-prototype.html` (1.3 МБ, self-contained; gzip+base64-бандл Claude-артефакта). Реальный исходник распакован: модули `App / Shell / HomeScreen / ResultsScreen / RecordScreen / AccountScreens / OrderModal / SeasonalFX / data.js`. Демо-данные — обезличенный театрально-архивный корпус (Чехов / МХТ), 4 базы.
> - **НАШ КОД** — `C:\IRBIS64\_recon\irbis-web\frontend\src\App.tsx` (966 строк) + `export.ts`. Это то, что реально работает поверх живого ИРБИС.
> - **НАШ СПЕК** — `docs/design/specs/SPEC_reader_jirbis.md` (42 поведения J-ИРБИS) + `SPEC_I2_catalog_search.md`.
>
> **Легенда вердиктов:**

## 0. Архитектурное наблюдение (ключ к ответу)

| | ПРОТОТИП | НАШ КОД (App.tsx) |
|---|---|---|
| Источник данных | мок-данные `data.js` (4 базы, обезличено) | **живой ИРБИС64** через `api.*` (search/record/facets/terms/cabinet/order) |
| Конфиг-движок | **да**: экраны рендерятся из `databases[]` (поля поиска / фильтры / layout / typeIcon per-база); «добавить базу = добавить конфиг» | нет: одна база IBIS, префиксы захардкожены (`PREFIXES`), фасеты с бэкенда |
| Мульти-база | **4 базы разных типов** (книги-list, эскизы-gallery, архив-list+даты, премьеры-calendar) | одна публичная БД (селектор баз есть, но единый layout) |
| Экранов-маршрутов | 6: `home / results / record / login / account / catalog-error` | 3 представления: поиск(=results), карточка, кабинет (+staff) |

**Это и есть корень ощущения «не наполненности»:** прототип — это **демо-витрина с лендингом, мульти-базовым конфиг-движком и богатым набором типов баз** на статике; наш код — это **рабочее ядро на реальном ИРБИС**, но с одним стартовым экраном (сразу список результатов, без главной/витрины) и единым форматом записи. Прототип «выглядит полнее», потому что показывает **поверхностей больше**, хотя за ними нет бэкенда.

## 8. Что есть только у нас (⚪ — наши преимущества)

1. **Экспертный поиск** — язык запросов ИРБИС (`* + ^ $`, кавычки-термины). В прототипе нет.
2. **Экспорт цитирования** RIS / BibTeX / ГОСТ Р 7.0.100 (`export.ts`) — в прототипе кнопки экспорта вовсе нет.
3. **Постоянная ссылка** `?db=&mfn=` с копированием в буфер — в прототипе «Поделиться» это пустая toast-заглушка.
4. **Вкладка сырых меток MARC** в карточке.
5. **Live-формуляр** из реального ИРБИС (поле 40, сроки/просрочки/продления, склонение «день/дня/дней»). В прототипе формуляр статичен.
6. **Корзина-drawer** с экспортом списка (RIS/BibTeX/txt) и **отправкой на почту** (mailto).
7. **Демо-блоки вовлечения в кабинете**: очередь брони (позиция), полки чтения, reading-challenge — задел под #174/PMB.
8. **Контекст «Сотрудник» (StaffArea)** — двух-контурный портал (читатель/сотрудник) — вне периметра прототипа.
9. **Работа на живом ИРБИС** (а не на моках): реальные поиск/фасеты/термины/запись/кабинет/заказ.

## 11. Честный итог: недо-спроектирован или недо-реализован?

**Главный вывод: наш портал НЕ недо-спроектирован — он недо-реализован, плюс недо-«офасаден».**

1. **Спек глубже прототипа.** Прототип демонстрирует ~поверхностный срез: он не имеет ни морфологии, ни навигаторов-деревьев, ни PDF-просмотрщика с правами, ни ЕСИА, ни продления, ни рекомендаций/FRBR, ни Z39.50/OAI/SIP2, ни экспорта цитирования, ни постоянных ссылок. Всё это **есть в нашем спеке** (`SPEC_reader_jirbis` §1–7, 42 поведения; `SPEC_I2`). То есть по **широте замысла мы прототип превосходим** — большинство «недостающего» сидит в корзине 📐 (спек есть, код нет), а не ❌.

2. **Код уже local-сильнее прототипа в ядре.** Наш App.tsx работает на **живом ИРБИС**, имеет экспертный поиск, экспорт RIS/BibTeX/ГОСТ, постоянные ссылки, MARC-вкладку, live-формуляр, корзину с email — **всего этого в прототипе нет** (13 ⚪). Это не «пустой» портал.

3. **Почему ощущается «не наполненным» — три конкретные причины (и все исправимы):**
   - **Нет «фасада» discovery.** Прототип открывается **лендингом с витриной и примерами** (G1–G3); мы стартуем сразу в списке результатов «Android». Первое впечатление — «голо».
   - **Один тип представления.** Прототип показывает **4 разных типа баз** (книги/галерея/архив/премьеры) с разными layout (gallery-сетка, date-фильтр, календарь-витрина). У нас всё — один книжный список. Визуального «объёма» меньше (G4, G5, G9).
   - **Не хватает мелкой «фурнитуры» выдачи и кабинета**, которую прототип показал и которую мы даже не записали в спек как AC: **сортировка** (G6), **мастер заказа с выбором МХ** (G10), **вкладка «Заказы» + отмена** (G12), **batch-заказ** (G11), **темы/сезонные эффекты** (G15), **режим без картинок** (G16). Это 19 ❌, и каждый — небольшой, но вместе они и создают ощущение «живости» прототипа.

4. **Количественно:** наш App.tsx реализует **22 из ~56** поведений ледгера как ✅ (+13 ⚪ сверх прототипа). Прототип показывает порядка **~40** фич, из которых **19 (G1–G19)** у нас **не покрыты ни кодом, ни явным AC спека** — это и есть настоящий пробел, который стоит занести в reader-бэклог. Остальной разрыв (**~34 в 📐**) — это **запланированная, но не написанная** функциональность, т.е. вопрос реализации, а не замысла.

**Резюме одной фразой:** *Портал недо-реализован относительно собственного (более глубокого, чем прототип) спека и недо-офасаден относительно прототипа; прототип «полнее» не по сути, а по числу показанных поверхностей (лендинг + витрина + 4 типа баз + UI-фурнитура). 19 прототип-онли фич (G1–G19) — реальный список для пополнения спека/бэклога; всё остальное у нас уже спроектировано и частично сильнее.*
