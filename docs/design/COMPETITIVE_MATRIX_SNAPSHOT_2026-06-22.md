## §0. Методология (честно)

- **«Мы» в этом срезе = ФАКТИЧЕСКАЯ реализация на 2026-06-22, а НЕ план.** Там, где в живой матрице ([`COMPETITIVE_MATRIX.md`](COMPETITIVE_MATRIX.md)) колонка «Мы» отражает **целевой план**, здесь она перемаркирована под **текущее состояние кода/спек**. Расхождения «план vs факт» сведены в §1 и §4.
- **ИРБИС64 по функциям АРМов = ЭТАЛОН.** Все ~722 функции реестра (§2) **реверс-инжинирены ИЗ ИРБИС** (его INI/PFT/GBL/FST/БД), поэтому по этим функциям у ИРБИС64 стоит **✅ по определению** — это и есть та система, чей функциональный периметр мы воспроизводим. Сравнивать ИРБИС «сам с собой» бессмысленно: он — мерило полноты.
- **Остальные конкуренты (МАРК Cloud / Руслан-Нео / МегаПро / FOLIO / Alma+Primo / OCLC WMS) сравниваются только НА УРОВНЕ ВОЗМОЖНОСТЕЙ (§1), а НЕ по каждой из 722 функций.** Поштучные оценки для них **не выдумываются** — это было бы недостоверно (ни один публичный источник не раскладывает чужой продукт на 722 пункта меню ИРБИС). Где источник по возможности есть — оценка из point-in-time ресерча; где нет — «?».
- **Источник марок «Мы сейчас» в §2** — отметки покрытия из 6 inventory-файлов (`docs/design/arm_inventory/*.md`), сопоставление: inventory **✅ (покрыто, спека+код/паритет) → ✅**, **🟡 (тонко/частично) → ◑**, **❌ (нет спеки/кода) → ❌**, **⚫ (DROP, легаси) → ❌⚫**. Марки скорректированы под то, что **реально приземлилось в этой сессии** (движки и дески комплектования/книгообеспеченности, циркуляция+каталогизатор-деск, холды/полки/отзывы/рекомендации, провижининг/биллинг, комплаенс V1/V3/V4/V5/V9/V10) — где сессия подняла функцию выше inventory-уровня, это отмечено в примечании.
- **Точка отсчёта дашборда готовности (для §4):** портал **92%** · Книговыдача **30%** · Администратор **35%** · Каталогизатор **20%** · Комплектование **22%** · Книгообеспеченность **22%** · Утилиты **15%**. Пофункциональный счёт §2 — отдельная (более строгая, построчная) метрика; он НЕ обязан совпадать с процентами дашборда (дашборд взвешивает по важности/усилию, §2 считает строки).

### Что перемаркировано в §1 относительно плана (оригинала)

Где оригинал ставил «Мы ✅/◑» как **цель**, здесь честно понижено под факт кода на 2026-06-22:

- **Edge-узлы + офлайн-синхронизация: ✅(план) → ◑(факт).** Инвариант офлайн-выдачи спроектирован (event-sourced circulation, детерминизм офлайн — LENDING RF-11), но sync-каскад/конфликт-резолюция-UI/OTA-флот ещё на скелете (DESIGN_COVERAGE C2, I4 ~50%). Кода полного edge-узла нет → ◑, не ✅.
- **Мобильное приложение: ◑(план) → ❌(факт).** Нативного приложения (билет/self-checkout/push) в коде нет; только серверные события-крючки. Честно ❌.
- **Семантический/AI-поиск + RAG: ✅(план) → ◑(факт).** Архитектура RAG/guardrails описана (DESIGN_COVERAGE C3), но вектор-схема/анти-галлюцинация-сьют/автокаталогизация не реализованы. ◑.
- **Маркетплейс модулей per-tenant: ✅(план) → ❌(факт).** Дескриптор/sandbox/подпись плагинов (DESIGN_COVERAGE C5) — спека-намерение, кода маркетплейса нет. ❌.
- **Ячеистое хранение + постаматы 24/7: ✅(план) → ◑(факт).** Самый проработанный из «синих океанов» на уровне спек (LENDING RF-06…RF-14, lockers §1–7 AC1–13: pickup-код, slot-allocation, device-gateway, автосдача, офлайн, хаос-тест), есть own-server PoC иерархии хранения (I3 §B/§C ✅), но pickup-lifecycle и device-адаптеры — спека/PoC, не прод-код. ◑.
- **Цифровая библиотека / DAM / IIIF: ✅(план) → ◑(факт).** DAM-пайплайн (извлечение ПТ/OCR/ALTO, кэш) — `[TBD]` (UTILITIES G1 — главный недострой), хранилище/деривативы названы. ◑.
- **Linked Data / BIBFRAME / entity-URI: ◑(план) → ❌(факт).** В коде нет; эпик PME. ❌.
- **OpenAPI-контракт + webhooks + расширения: ✅(план) → ◑(факт).** Контракты движков частично формализованы (SPEC_api_contracts), но единого OpenAPI+webhooks ещё нет (DESIGN_COVERAGE A7 🟡). ◑.
- **Доступность WCAG 2.2 / ГОСТ: ✅(план) → ◑(факт).** Сейчас «hollow» по собственному аудиту (DESIGN_COVERAGE C4): нет axe/Lighthouse-гейта, VPAT, DAISY. Заявлять ✅ нечестно → ◑.
- **Импортозамещение (реестр + Astra): ✅(план) → ◑(факт).** Стек целевым образом совместим (PostgreSQL/Linux), но записи в реестре отеч.ПО и аттестации на Astra ещё нет (это процесс, не код). ◑.

Не понижены (факт совпал с планом): multi-tenant SaaS ✅, веб-клиент читателя ✅ (портал 92%), реляц.СУБД+FTS ✅. Прочие ◑ (discovery/комплектование/консорциумы/стандарты/гос-интеграции/аналитика/соц-слой) совпали с планом-◑.
