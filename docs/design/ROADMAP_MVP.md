# Biblio — дорожная карта до продаваемого MVP

**Статус документа:** план перед следующим наступом. Дата: 2026-06-21.
**Версия продукта на момент плана:** общая готовность ~38% (см. дашборд готовности). Понимание ИРБИС ~100%, дизайн ~95%, фундамент/движки сильные, АРМ-UI и платформа — основной незакрытый объём.

> Метод оценки: «инженеро-недели» (solo-эквивалент). Реальный wall-clock короче за счёт параллельных агентов (на наступе #222 шесть параллельных потоков закрыли ~2 недели работы за один проход). Недели здесь — для приоритизации и критического пути, не обязательство по календарю.

---

## 1. Тезис MVP

**Одна небольшая/средняя библиотека может зарегистрироваться, получить арендатора, каталогизировать фонд, выдавать книги, а читатели — искать и бронировать онлайн. Развёрнуто и тарифицируемо.** Это минимальная жизнеспособная замена ядра ежедневного цикла ИРБИС для одного клиента на нашем хранилище (PostgreSQL), без живого ИРБИС-сервера.

MVP — НЕ полная замена всех 6 АРМов и 700+ функций. Это продаваемый срез: каталог + книговыдача + читательский портал + развёртывание + базовый биллинг + комплаенс-минимум.

## 2. Scope: что в MVP, что в v2+

| Область | MVP | v2+ |
|---|---|---|
| Читательский портал (OPAC, бронь, кабинет, полки, инбокс) | ✅ есть (88%) | отзывы/рейтинги, RAG-рекомендации, постраничный PDF |
| АРМ Каталогизатор (UI) | ✅ рабочий лист + ФЛК + экземпляры + .gbl-базовый | полная глубина 170 функций, шаблоны вывода |
| АРМ Книговыдача (UI) | ✅ выдача/возврат/продление/штрафы/бронь | МБА/ИРИ, полная статистика |
| АРМ Комплектатор (UI) | 🟡 минимум: заказ→поступление→КСУ→ToCat | полная глубина 158 функций, бюджеты/поставщики |
| АРМ Книгообеспеченность | ❌ v2 (движок готов, UI позже) | полный ВУЗ-цикл |
| АРМ Администратор | 🟡 минимум: пользователи/гранты/базы | полный 101-функциональный АРМ |
| Провижининг арендатора | ✅ создать тенант→схема→словари→админ | self-service онбординг, маркетплейс модулей |
| Развёртывание | ✅ docker-compose + TLS + бэкапы | k8s/Helm, мульти-регион, авто-скейл |
| Биллинг | 🟡 тарифы + entitlements + лимиты | платёжный шлюз, инвойсы, метеринг по использованию |
| Комплаенс | 🟡 хэш паролей, TLS, журнал ПДн, секреты в env | сертифицированные СКЗИ (V6/V7), полное шифрование at-rest |
| Миграция из ИРБИС | ❌ v1.5 (импорт .mst/RDR) | полный мигратор всех БД |

**Явно вне MVP:** сертифицированные СКЗИ/CryptoPro (V6/V7 — это закупка лицензии + оргпроцесс, не код), low-code конструктор модулей, DAM, федерация/сводный каталог, мобильные приложения, полная глубина всех функций.

## 3. Текущий baseline (что уже готово к MVP)

- **Читательский портал** — поиск (3 режима), фасеты, запись, кабинет/формуляр, бронь+очередь, инбокс, полки, permalink, мульти-layout. Проверено e2e.
- **Движки (с тестами, CI sqlite 1118 / pg 1181):** каталог (save/ФЛК/экземпляры), книговыдача (выдача/возврат/бронь/штрафы, 910^A-флип), уведомления (диспетч из книговыдачи), .gbl (cross-DB), авторитеты (^3), PFT/UNIFOR, комплектование (ToCat/КСУ), книгообеспеченность (Кко). Все с HTTP-роутами под guard.
- **Фундамент:** мультитенант schema-per-tenant, JWT+entitlements, public-only базы, self-heal, аудит-лог, сидинг словарей.
- **Не хватает для MVP:** staff-UI поверх движков, провижининг+деплой, биллинг-скелет, комплаенс-минимум.

## 4. Фазы (порядок: staff-UI → платформа → комплаенс)

### Фаза 1 — Staff-UI: рабочие места сотрудника (~3 нед)
Превращает готовые движки в видимый продукт. Самый наглядный результат.

1. **АРМ Каталогизатор (UI)** — рабочий лист (worklist-эндпойнт есть) → редактор полей/подполей, live-ФЛК, сохранение записи, управление экземплярами (910), поиск-в-редактирование, базовый .gbl над выборкой.
2. **АРМ Книговыдача (UI)** — выдача/возврат по билету, формуляр читателя (поле 40), продление, штрафы, очередь брони (бэкенд готов), быстрый ввод по сканеру.
3. **АРМ Комплектатор (минимум)** — список заказов, поступление→КСУ→ToCat (роуты готовы), реестр КСУ.
4. **Staff-оболочка** — единый desktop с грант-навигацией (grant desktop PoC есть), переключение АРМов по правам.

- **Зависимости:** движки + роуты (✅ готовы). Дизайн-система Biblio (✅).
- **Риск:** глубина каталогизации (ФЛК-покрытие, полнота worklist) — самое сложное; начать с PAZK (книги), расширять.
- **DoD:** каталогизатор создаёт/правит/сохраняет запись с экземплярами и проходит ФЛК; книговыдача делает полный цикл выдача→возврат по билету с формуляром.

### Фаза 2 — Платформа: провижининг + деплой + биллинг (~2.5 нед)
Делает продукт разворачиваемым и тарифицируемым — главный блокер до коммерции.

1. **Контейнеризация** — Dockerfile (backend, frontend-build), docker-compose (backend + PostgreSQL + reverse-proxy с TLS). Одна команда `up`. (см. `specs/platform/SPEC_iac_fleet.md`)
2. **Провижининг арендатора** — поток: создать тенант → создать схему t_<slug> → засеять словари (seed_vocabularies есть) → создать админ-аккаунт → выдать гранты/модули. CLI + минимальный admin-UI.
3. **Биллинг-скелет** — тарифные планы → entitlements per tenant (механизм есть, #101) → лимиты использования (записи/читатели/хранилище) → enforcement. Без платёжного шлюза (ручная активация плана). (см. `specs/platform/SPEC_billing.md`)
4. **Ops-минимум** — `/api/health` (есть) + метрики, структурные логи, бэкап PostgreSQL, миграции схемы.

- **Зависимости:** мультитенант-изоляция (✅), entitlements (✅).
- **Риск:** надёжность провижининга/миграций схемы; сделать own-store (PostgreSQL) первичным контуром (сейчас демо на dealer-ИРБИС с лимитом 3 клиента — это не продукт).
- **DoD:** новый клиент провижинится одной командой; разворачивается через docker-compose с TLS; данные персистентны + бэкап; план назначается и enforce'ится.

### Фаза 3 — Комплаенс-минимум: критические V1–V5 (~1.5 нед)
Делает Biblio проходимым по 152-ФЗ там, где сам ИРБИС — нет (см. `AUDIT_REPORT_IRBIS64_compliance.md`, `specs/gov/SPEC_compliance_152fz.md`, `specs/platform/SPEC_pki_keys.md`).

1. **V3 — пароли** читателей и staff: хэш + соль (argon2/bcrypt). Секреты в env (✅ частично).
2. **V4 — TLS** на API-канал (терминация на reverse-proxy из Фазы 2).
3. **V5 — журнал доступа к ПДн** (РСБ): audit_log есть — расширить на чтение данных читателя (кто/когда смотрел RDR-поля), ретенция, защита от правки.
4. **V1 — шифрование ПДн at-rest**: чувствительные колонки (паспорт/контакты) через pgcrypto или прозрачное шифрование диска; для MVP — документированный частичный объём + план.
5. **V9/V10 (док-минимум)** — слой согласий + право на удаление (минимум), акт классификации ИСПДн/УЗ (документ).

- **Зависимости:** Фаза 2 (TLS-прокси, схема).
- **Риск/вне MVP:** V6/V7 (сертифицированные СКЗИ) — НЕ код, это лицензия CryptoPro + оргрежим; явно вне MVP, отметить в карте рисков клиента.
- **DoD:** пароли хэшируются; канал под TLS; доступ к ПДн журналируется; критические колонки шифруются или есть документированный план; акт УЗ оформлен.

### Фаза 4 — Сведение + e2e + хардненинг (~1 нед)
Сквозной smoke-e2e ежедневного цикла (провижининг → каталог → выдача → читатель бронирует), нагрузочный sanity, фикс хвостов, документация развёртывания.

## 5. Таймлайн и критический путь

| Фаза | Инж.-недели | Зависит от |
|---|---|---|
| 1. Staff-UI | 3 | движки/роуты (готовы) |
| 2. Платформа | 2.5 | изоляция/entitlements (готовы) |
| 3. Комплаенс | 1.5 | Фаза 2 (TLS/схема) |
| 4. Сведение/e2e | 1 | Фазы 1–3 |
| **Итого** | **~8 инж.-недель** | |

**Критический путь:** Фаза 2 → Фаза 3 → Фаза 4 (комплаенс и сведение упираются в платформу). Фаза 1 (staff-UI) параллелится с Фазой 2 (разные слои: frontend-staff vs infra/backend). При параллельных агентах wall-clock ~4–5 недель.

## 6. Риски и митигации

| Риск | Влияние | Митигация |
|---|---|---|
| Глубина каталогизации (ФЛК/worklist) | Фаза 1 затягивается | начать с PAZK-книг, расширять по типам; ФЛК уже декларативна |
| Провижининг/миграции схемы хрупкие | Фаза 2 блокер | идемпотентные DDL (паттерн есть), тесты провижининга |
| Демо на dealer-ИРБИС (лимит 3) ≠ продукт | путаница контуров | сделать own-store PostgreSQL первичным, ИРБИС — только источник миграции |
| Шифрование ПДн at-rest (V1) тяжёлое | Фаза 3 | pgcrypto на критичные колонки + документированный план на полное |
| Сертифицированные СКЗИ (V6/V7) | вне MVP | явно в v2/закупка; в карте рисков клиента |
| Миграция данных существующей библиотеки | блокер первой продажи | импорт .mst/RDR — v1.5, спланировать сразу после MVP |

## 7. Definition of Done — MVP (приёмочный чек-лист)

- [ ] Новый клиент провижинится (тенант + схема + словари + админ) одной командой/потоком.
- [ ] Каталогизатор: создать/править/сохранить библиозапись с экземплярами, live-ФЛК.
- [ ] Книговыдача: выдача/возврат/продление по билету, формуляр, штрафы, бронь.
- [ ] Читатель: поиск, бронь, кабинет, уведомления — по каталогу своего тенанта.
- [ ] Развёртывание: docker-compose с TLS; данные персистентны; бэкап настроен.
- [ ] Безопасность: пароли хэшированы; доступ к ПДн журналируется; секреты в env; критичные ПДн шифруются (или документированный план).
- [ ] Биллинг: план назначается, entitlements/лимиты enforce'ятся.
- [ ] CI зелёный; smoke-e2e ежедневного цикла проходит.
