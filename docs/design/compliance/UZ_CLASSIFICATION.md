# Акт классификации ИСПДн и определения уровня защищённости — продукт Biblio

> **Закрывает аудит-находку V10** ([AUDIT_REPORT_IRBIS64_compliance §3](../AUDIT_REPORT_IRBIS64_compliance.md): «ИСПДн не классифицирована — нет акта определения УЗ, модели угроз, перечня ПДн»). Нормативная основа спроектирована в [SPEC_compliance_152fz §1](../specs/gov/SPEC_compliance_152fz.md); настоящий документ — **комплаенс-артефакт уровня акта**, материализующий §1 для НАШЕГО продукта.
>
> ⚠ **Объект классификации — продукт Biblio** (мультиарендный SaaS + edge, веб-бэкенд `irbis-web/backend`, собственный сервер `server-own`), **а НЕ легаси-установка САБ ИРБИС64+**. Аудит V1–V11 проверял исходную ИРБИС-установку (`C:\IRBIS64`); здесь классифицируется то, что хранит и обрабатывает **наша** реализация. Это принципиальное различие: наш инвентарь ПДн **меньше** легаси (мы не тянем паспорт/контакты по умолчанию) и устроен иначе (PostgreSQL/SQLite, не ISIS-файлы).
>
> ⚠ **В документе — только структура, категории полей, метки и правила. Никаких реальных ПДн, паролей, ключей, идентификаторов субъектов.** Все значения ниже — типы и категории, не данные.
>
> **Граница ответственности.** Юридически акт классификации утверждает **оператор** (библиотека-тенант) с привлечением DPO / лицензиата ФСТЭК. Продукт (обработчик по поручению, ст. 6 152-ФЗ) поставляет **техническую доказательную базу и расчёт-шаблон**: онбординг гос-тенанта вычисляет УЗ анкетой (категория ПДн × угрозы × объём) и формирует **черновик** этого акта per-tenant ([SPEC_compliance_152fz §6.3 AC-F1](../specs/gov/SPEC_compliance_152fz.md)). Настоящий файл — **референс-шаблон** этого черновика.

## 1. Назначение и нормативная база

**Назначение.** Определить для информационной системы персональных данных (ИСПДн) продукта Biblio:
1. перечень обрабатываемых ПДн и их категории (152-ФЗ);
2. категории субъектов и их отношение к оператору;
3. тип актуальных угроз;
4. объём обрабатываемых ПДн;
5. **уровень защищённости (УЗ)** по матрице ПП-1119;
6. краткую модель угроз и состав мер ФСТЭК-21;
7. распределение ответственности оператор/обработчик.

**Проверяемые нормы:**
- 152-ФЗ «О персональных данных» (ст. 5 минимизация, ст. 10 спец. категории, ст. 19 меры защиты);
- Постановление Правительства РФ № 1119 от 01.11.2012 («Об утверждении требований к защите ПДн при их обработке в ИСПДн») — уровни защищённости;
- Приказ ФСТЭК России № 21 от 18.02.2013 — состав и содержание мер защиты ПДн;
- БДУ ФСТЭК (банк данных угроз) — источник модели угроз.

## 2. Перечень ПДн — что НАШ продукт реально хранит

Ключевое отличие от легаси-ИРБИС: продукт **не реплицирует** всю картотеку RDR (58 полей, включая паспорт и фото). Источник идентификации читателя — **внешний ИРБИС-сервер RDR** (read-only, поиск по `RI=<билет>`); продукт **резолвит** оттуда минимум и хранит у себя только то, что нужно его собственным функциям (отзывы, брони, полки, история, уведомления, циркуляция).

### 2.1 ПДн, которые продукт ХРАНИТ (в своих БД — PostgreSQL/SQLite)

| Группа | Конкретные данные продукта | Где хранится (наш стор) | Категория по 152-ФЗ |
|---|---|---|---|
| **Идентификатор читателя** | номер читательского билета (ticket, `RI`) — ключ, по которому привязаны все наши записи | reader-scoped стор (social/holds/shelves/notifications), `subject_id`/`ticket` | иные ПДн (идентификатор) |
| **Резолвленное отображаемое имя** | display name (RDR 10/11), кэшируется для меток отзывов/формуляра | social-стор (метка карточки отзыва), circ-формуляр | иные ПДн |
| **Поведение / контент читателя** | отзывы + рейтинги (`/api/review`), история чтения (`/api/history`, **opt-in**), сохранённые поиски (`/api/savedsearch`), полки/списки (`/api/shelves`) | social-стор (reviews/history/savedsearch/shelves) | иные ПДн (поведенческие) |
| **Брони / заказы** | холды (`/api/hold`, `/api/holds`) — связка «билет ↔ запись (db,mfn) ↔ очередь» | holds-стор | иные ПДн (поведенческие) |
| **Циркуляция** | выдачи/возвраты/долги/штрафы (`/api/circ/*`) — формуляр field-40-модели, loan/hold/fine | circulation-engine-стор | иные ПДн (поведенческие) |
| **Уведомления** | очередь нотификаций читателю + статус прочтения (`/api/notifications`) | notifications-стор | иные ПДн |
| **Учётка персонала** | login + **pbkdf2-sha256 солёный хэш** пароля + ФИО сотрудника + роли/гранты | access-стор `staff_account` | иные ПДн (сотрудник оператора) |
| **Журнал аудита / доступа** | actor + function + db + mfn + result + detail (write/admin-аудит I1; журнал доступа к ПДн — landing this phase) | access-стор `audit`, `pdn_access_log` | by-reference (сам журнал — ПДн-by-ref) |

> **Пароль сотрудника** хранится строго как `pbkdf2$<iterations>$<salt>$<dk>` (200 000 итераций SHA-256, 16-байтовая соль, сверка `hmac.compare_digest`) — `irbis-web/backend/access/store.py` (`hash_password`/`verify_password`). Plaintext/несолёный MD5 (антипаттерн легаси V3) в продукте **не воспроизводится**.

### 2.2 ПДн, которые продукт НЕ хранит (без явного reader-модуля)

| Не храним | Почему |
|---|---|
| **Паспортные данные** (RDR 14) | продукт их не запрашивает и не реплицирует; остаются в источнике-ИРБИС RDR под управлением оператора. По умолчанию вне периметра наших стораджей. |
| **Контакты** (адрес 13, телефоны 17/18, e-mail 32) | не реплицируются по умолчанию; e-mail/тел. появляются у нас **только** если оператор включает reader-модуль уведомлений по каналам (тогда — по согласию `notifications`, §2 CONSENT_POLICY). |
| **Фото** (RDR 950) | не реплицируется; продукт не выполняет биометрическую идентификацию по фото по умолчанию. |
| **Демография** (год рождения 21, пол 23, гражданство 25, образование 20) | не реплицируется; статистика 6-НК строится агрегатами на стороне источника. |
| **Спец. категория — данные о здоровье** (RDR 41–49, библиотека для незрячих) | **по умолчанию вне периметра продукта**; включается только отдельным пресетом тенанта с явным письменным согласием (поднимает УЗ — §4). |

> **Граница забвения** (важно для CONSENT_POLICY §3): право на удаление в продукте касается **наших** стораджей (отзывы/брони/полки/история/циркуляция/уведомления + резолвленное имя/билет), **а не** исходной записи RDR в ИРБИС-сервере оператора — её жизненным циклом управляет оператор средствами ИРБИС (`DelDate.gbl` и т.п.).

### 2.3 Производные хранилища ПДн (в периметре ИСПДн)

- **Журнал доступа к ПДн** (`pdn_access_log`, [SPEC_compliance_152fz §3](../specs/gov/SPEC_compliance_152fz.md)) — by-reference (категории полей + `subject_id`, не значения), но субъект идентифицируем → в периметре.
- **Резервные копии** БД тенанта и **edge-реплики** (БД-на-тенанта на узле) — те же ПДн, тот же режим защиты ([SPEC_pki_keys §1.4/§6](../specs/platform/SPEC_pki_keys.md)).
- **Аудит I1** (`audit`) — write/admin-действия с `actor`/`subject` by-ref.

## 3. Категории субъектов и параметры расчёта (ПП-1119)

| Параметр | Значение для продукта Biblio |
|---|---|
| **Категория ПДн** | **иные** (имя, идентификатор, поведение, контакты-по-согласию). При включении пресета «библиотека для незрячих» — **специальная** (данные о здоровье, ст. 10). Биометрию по умолчанию **не обрабатываем** (фото не реплицируется/не используется для идентификации). |
| **Отношение субъектов к оператору** | читатели — **НЕ сотрудники** оператора (третьи лица). Сотрудники-операторы (staff-учётки) — отдельная малая группа субъектов-сотрудников. |
| **Тип актуальных угроз** | базово **тип 3** (угрозы, не связанные с НДВ в системном и прикладном ПО) — продукт без декларированных НДВ, supply-chain-гейты I6 §B. Для ведомственного/защищённого контура оператор может определить **тип 2** (актуальны НДВ в прикладном ПО). |
| **Объём субъектов** | per-tenant: школа/малая — `≤ 100 000`; крупная публичная сеть / вуз — `> 100 000`. Параметр анкеты онбординга. |

## 4. Определение уровня защищённости (УЗ)

Матрица ПП-1119 «категория ПДн × тип угроз × объём субъектов» применяется **per-tenant** (УЗ — атрибут тенанта `tenant_pdn_profile.uz` / `tenant_key_policy.uz_level`).

| № | Профиль тенанта | Категория ПДн | Тип угроз | Объём субъектов | **УЗ** |
|---|---|---|---|---|---|
| П1 | Школа / малая публичная библиотека | иные | 3 | ≤ 100 000 | **УЗ-4** |
| П2 | Крупная публичная сеть / вуз | иные | 3 | > 100 000 | **УЗ-3** |
| П3 | Библиотека для незрячих (пресет 41–49, здоровье) | специальные | 3 | любой | **УЗ-3** |
| П4 | Библиотека для незрячих, крупная сеть | специальные | 3 | > 100 000 | **УЗ-2** |
| П5 | Ведомственная / в защищённом контуре | спец./иные | 2 | любой | **УЗ-2** |

### 4.1 Итоговая определённость (для типовой установки)

**Базовая инженерная цель продукта — технически удовлетворять УЗ-3** (наиболее частый профиль для библиотек: профили П2/П3), оставаясь конфигурируемым до **УЗ-2** в защищённом контуре (on-prem/edge на Astra Linux, сертифицированные СКЗИ — профили П4/П5). Для малого школьного профиля П1 фактический норматив — УЗ-4, который продукт перекрывает мерами УЗ-3 «сверху».

> **Вывод акта:** для типового тенанта Biblio — **УЗ-3**. При обслуживании незрячих (включён пресет полей здоровья) — не ниже УЗ-3, до УЗ-2 для крупной/ведомственной установки.

### 4.2 Чем это отличается от легаси-вывода аудита

Аудит ([AUDIT §3](../AUDIT_REPORT_IRBIS64_compliance.md)) дал ту же **базовую оценку УЗ-3** для легаси-RDR, но констатировал: *«Установка не реализует ни одной меры ФСТЭК-21, обязательной уже для УЗ-4»*. Продукт Biblio эту дыру закрывает: реестр ПДн-полей + минимизация (мы не храним паспорт/фото), pbkdf2-хэш (V3), TLS в deploy (V4), секреты в env/Vault (V2), public-only-guard (V11), журнал доступа к ПДн + согласия + забвение (V1/V5/V9, landing this phase) — трассировка в [SPEC_compliance_152fz §6.1](../specs/gov/SPEC_compliance_152fz.md) и в статус-матрице того же файла.

## 5. Модель угроз (краткая сводка)

Полная модель угроз — комплаенс-артефакт per-тип-развёртывания (cloud-SaaS / edge-on-prem / гибрид) на базе **БДУ ФСТЭК**; ниже — каркас (из [SPEC_compliance_152fz §1.3](../specs/gov/SPEC_compliance_152fz.md)), адаптированный к фактическому инвентарю продукта (§2).

| Класс угрозы | Источник | Объект | Базовая контрмера продукта |
|---|---|---|---|
| Несанкц. доступ к ПДн (внешний) | нарушитель из сети | reader-стор / circ / журнал | TLS-терминация (nginx, `deploy/proxy/nginx.conf`), zero-trust/mTLS, RBAC-гранты (I1), журнал доступа |
| Несанкц. доступ (внутренний — избыточный грант сотрудника) | оператор с лишним грантом | ПДн читателя | минимально-необходимые гранты `(функция×база×уровень)`, public-DB-guard, маскирование, журнал доступа к ПДн (`sensitive`) |
| Утечка через кросс-тенант | дефект изоляции | чужой тенант | схема-на-тенанта + RLS + per-tenant KEK (крипто-изоляция), негатив-тесты |
| Утечка через резерв.копии/edge | бэкап/реплика | ПДн at-rest | per-tenant envelope-шифрование ([SPEC_pki_keys](../specs/platform/SPEC_pki_keys.md)), крипто-стирание |
| Утечка через логи/уведомления | трассы, e-mail/SMS | ПДн открытым видом | маскирование ПДн в логах; шаблоны уведомлений без избыточных ПДн; пароль — никогда в логах |
| Раскрытие пароля сотрудника | дамп БД учёток | `staff_account.pass_hash` | pbkdf2-sha256 200k+соль (V3); plaintext/MD5 не воспроизводятся |
| Передача в третьи системы | СМЭВ/ЕСИА/НЭБ | ПДн субъекта | согласие на передачу (CONSENT_POLICY §2), журналирование передачи, ЭП ГОСТ через СКЗИ |
| Неправомерное хранение | истёкшая ретенция | история/архив | авто-чистка по сроку (паритет `DelDate.gbl`), opt-in история, ретенция журнала |
| Доступ читателя к служебной/ПДн-базе | guest/reader к RDR/LICH | служебные БД | **public-DB-guard** `_public_db_guard` (V11) — non-staff confined к public allow-list |

## 6. Состав мер защиты под УЗ-3 (трассировка ФСТЭК-21)

Полный чек-лист готовности — [SPEC_compliance_152fz §6.1](../specs/gov/SPEC_compliance_152fz.md). Краткая трассировка обязательных групп мер:

| Группа мер ФСТЭК-21 | Что обеспечивает продукт | Состояние |
|---|---|---|
| **ИАФ** — идентификация/аутентификация | staff-учётки + pbkdf2-хэш; JWT+гранты I1; ЕСИА (gov) | реализовано (V3) |
| **УПД** — управление доступом | гранты `(функция×база×уровень)`, public-DB-guard, минимально-необходимые права | реализовано (V11) |
| **РСБ** — регистрация событий безопасности | журнал доступа к ПДн `pdn_access_log` (append-only) + аудит I1 | landing this phase (V5) |
| **ЗИС** — защита среды/каналов | TLS-терминация (deploy/proxy/nginx.conf), mTLS, без внешних CDN | реализовано (V4) / проектируется |
| **СКЗИ** — криптозащита | envelope at-rest per-tenant; для УЗ-2 — **сертифицированные ФСТЭК СКЗИ** | проектируется (V1) / вне MVP (V6/V7) |
| **Минимизация / шифрование at-rest** | реестр ПДн-полей; мы не храним паспорт/фото; per-tenant ключ | реализовано (минимизация) / landing (шифрование) |

## 7. Распределение ответственности (оператор / обработчик)

| Сторона | Роль | Ответственность |
|---|---|---|
| **Оператор** (библиотека-тенант) | оператор ПДн, ст. 6 152-ФЗ | утверждает акт классификации; определяет/утверждает модель угроз и тип угроз; ведёт политику обработки ПДн; организует аттестацию ИСПДн силами лицензиата ФСТЭК; управляет жизненным циклом исходной записи RDR в своём ИРБИС-сервере |
| **Продукт Biblio** (вендор) | обработчик по поручению | поставляет технические меры ФСТЭК-21 и доказательную базу (журналы, отчёты, конфигурацию); вычисляет УЗ-черновик per-tenant анкетой онбординга; минимизирует/шифрует/маскирует/журналирует ПДн; исполняет право на забвение в **своих** стораджах; **не аннулирует** обязанность оператора утвердить документы |

> **Аттестацию ИСПДн проводит лицензиат ФСТЭК на объекте оператора.** Cloud-SaaS-вариант может опираться на аттестованный ЦОД; защищённый контур (on-prem/edge на Astra Linux) аттестуется на объекте библиотеки. Продукт — поставщик мер и артефактов, не аттестующий орган.

## 8. Заключение акта

ИСПДн продукта Biblio обрабатывает **иные** категории ПДн читателей (идентификатор-билет, резолвленное имя, поведенческие данные — отзывы/брони/полки/история/циркуляция/уведомления) и учётные данные сотрудников (login + pbkdf2-хэш), не реплицируя из источника паспорт/контакты/фото/демографию/здоровье без явного reader-модуля и согласия. Субъекты-читатели — третьи лица; базовый тип угроз — 3. **Определённый уровень защищённости для типового тенанта — УЗ-3** (профили П2/П3); при включении пресета данных о здоровье или в ведомственном/защищённом контуре — до **УЗ-2** (профили П4/П5). Состав мер ФСТЭК-21 под УЗ-3 трассирован в §6 и [SPEC_compliance_152fz §6.1](../specs/gov/SPEC_compliance_152fz.md); права/обязанности распределены оператор/обработчик (§7). Акт утверждается оператором; продукт поставляет техническую доказательную базу и расчёт-черновик per-tenant.
