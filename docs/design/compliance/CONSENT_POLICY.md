# Политика согласий и приватности — продукт Biblio

> **Слой согласий + право на забвение для продукта Biblio.** Спроектировано в [SPEC_compliance_152fz §2/§4](../specs/gov/SPEC_compliance_152fz.md); настоящий документ — **политика-артефакт** (что покрывает согласие, версионирование, объём права на удаление, ретенция), материализующая ту спеку как комплаенс-документ оператора. Парный документ — [UZ_CLASSIFICATION](UZ_CLASSIFICATION.md) (классификация ИСПДн, перечень ПДн).
>
> ⚠ **Объект — продукт Biblio**, не легаси-ИРБИС. Право на удаление касается **наших** хранилищ (отзывы/брони/полки/история/циркуляция/уведомления + резолвленное имя/билет), **а не** исходной записи RDR в ИРБИС-сервере оператора (см. §3.2).
>
> ⚠ **В документе — только структура и правила. Никаких реальных образцов согласий с именами, ПДн, ID субъектов.**
>
> **Эндпоинты слоя** (landing this MVP-фаза, в openapi): `POST /api/reader/consent` (дать), `GET /api/reader/consent` (текущие), `POST /api/reader/erase` (запросить удаление), `GET /api/admin/pdn-access` (журнал доступа к ПДн — для DPO/проверок). Гранулярная механика схем (`consent`, `privacy_policy`, `erasure_request`, `pdn_access_log`) — в [SPEC_compliance_152fz §2.5/§3.5/§4.6](../specs/gov/SPEC_compliance_152fz.md).

## 1. Принципы

- **Правовое основание обработки.** Базовая обработка ПДн читателя (обслуживание: идентификация по билету, выдача, брони) ведётся на основании договора обслуживания / устава библиотеки и **согласия на базовую обработку** — предусловия создания учётки в продукте.
- **Согласие — отдельная сущность-данные, не флаг в профиле.** Хранится **версионированно и неизменяемо** (append-only): запись фиксирует, на какую **версию политики** субъект согласился, когда, каким способом, на какие **цели**. Отзыв — новая строка `withdrawn`, не удаление прежней (доказуемость).
- **Гранулярность по целям** (`purpose`), не одно «согласие на всё».
- **Минимизация по умолчанию.** Любая цель сверх базовой по умолчанию **не дана** (чекбоксы сняты); поля повышенной чувствительности и спец. категория собираются только при явном правовом основании и согласии.
- **Этика ALA + 152-ФЗ.** Долговременная персональная история чтения формируется **только при явном opt-in** субъекта; по умолчанию продукт хранит лишь оперативно-необходимое.

## 2. Что покрывает согласие (цели / `purpose`)

| Цель (`purpose`) | Что покрывает | По умолчанию | Отзываемость |
|---|---|---|---|
| `base_processing` | базовая обработка для обслуживания (идентификация по билету, брони, циркуляция, резолв отображаемого имени) | **обязательна** (предусловие учётки) | не отзывается без прекращения обслуживания |
| `reading_history` | **opt-in** долговременная персональная история чтения (`/api/history`, «для вас»-рекомендации) | снято | да → история очищается/обезличивается |
| `notifications` | каналы уведомлений (e-mail/SMS/push) — появление контактных ПДн у продукта | снято | да → канал отключается |
| `public_social_profile` | публичный соц-профиль: рейтинги/отзывы видны другим (`/api/review`, `/api/reviews`) | снято | да → отзывы скрываются/удаляются субъектом |
| `transfer_esia` | обработка атрибутов, полученных из ЕСИА | снято (только при входе через ЕСИА) | да |
| `transfer_smev` / `transfer_eios` | передача ПДн в СМЭВ / ЭИОС вуза | снято | да |
| `photo_950` | хранение/показ фото читателя (если оператор включает соответствующий модуль) | снято | да → фото удаляется |
| `marketing` | маркетинговые рассылки | снято | да |

> **Спец. категория (данные о здоровье, библиотека для незрячих).** Включается оператором **только при явном письменном согласии** субъекта (отдельный пресет тенанта). Поднимает категорию субъектов и УЗ ([UZ_CLASSIFICATION §4](UZ_CLASSIFICATION.md)). Не покрывается общим `base_processing`.

**Несовершеннолетние.** Для школьного сегмента запись согласия несёт отметку `requires_guardian` и фиксирует факт согласия законного представителя (без хранения избыточных ПДн представителя сверх необходимого).

## 3. Право на удаление / забвение

### 3.1 Принцип: анонимизация ≠ удаление

Удаление субъекта **не каскадит** в стирание операционных/статистических записей (иначе ломается ссылочная целостность и гос-отчётность). Поэтому **purge = смесь удаления и анонимизации** ([SPEC_compliance_152fz §4.1/§4.2](../specs/gov/SPEC_compliance_152fz.md)):

| Данные продукта | Действие при забвении | Обоснование |
|---|---|---|
| Резолвленное имя + привязка билета в наших сторах | **удалить** | прямые идентификаторы, более не нужны |
| Учётка/пароль читателя (если есть), токены | **удалить** | аутентификация субъекта |
| Отзывы/полки/сохранённые поиски/история чтения | **удалить/обезличить** | контент и поведенческий профиль субъекта |
| Завершённые выдачи / архив циркуляции | **анонимизировать** `subject_id`→`anon:<токен>` | сохранить статистику фонда/посещаемости без субъекта |
| Активные брони/выдачи (книга на руках, долг) | **блокировать удаление** до закрытия → `on_hold` | нельзя «забыть» читателя с книгой на руках |
| Согласия (`consent`) | **сохранить** (без прямых ПДн) + пометить субъекта удалённым | доказательство правомерности прошлой обработки (ст. 6) |
| Журнал доступа к ПДн (`pdn_access_log`) | **псевдонимизировать** `subject_id` | tamper-evidence доказуемость, но не идентифицируемость |
| Резервные копии / edge-реплики | стирание по циклу ретенции бэкапов + крипто-стирание (уничтожение ключа) + пометка «к забвению» | нельзя мгновенно стереть из неизменяемого бэкапа |

### 3.2 Объём права — НАШИ данные, не источник RDR

Право на удаление в продукте исполняется над **хранилищами продукта Biblio** (§3.1). Оно **НЕ распространяется** на исходную карточку читателя в **ИРБИС-сервере RDR** оператора — её жизненным циклом (заведение, правка, удаление по `DelDate.gbl`/`del40d.gbl`) управляет **оператор** средствами своей ИРБИС-установки. Продукт лишь резолвит из RDR минимум (имя по билету) и хранит это резолвленное значение у себя; удаляя своё хранилище, продукт удаляет резолвленную копию, но не трогает источник.

> Это явно сообщается субъекту в уведомлении о завершении забвения: что удалено в продукте, что анонимизировано, и что исходная запись в библиотечной системе оператора управляется отдельно.

### 3.3 Workflow и сроки

Workflow `erasure_request` (submitted → verifying → on_hold? → approved → completed/rejected) — [SPEC_compliance_152fz §4.3/§4.6](../specs/gov/SPEC_compliance_152fz.md). Запрос инициирует субъект (`POST /api/reader/erase`, ЛК «Запросить удаление») или сотрудник по бумажному заявлению. При незакрытых обязательствах (активная выдача/долг) → `on_hold` с причиной. Завершение фиксирует `purge_manifest` (что удалено/анонимизировано) и уведомляет субъекта.

- **Срок реакции.** 152-ФЗ задаёт предельные сроки; продукт ведёт таймер от `submitted` и эскалирует при приближении к дедлайну. Точные дни — параметр тенанта (`tenant_pdn_profile`), **не зашиты в политику**.

### 3.4 Экспорт данных субъекта (право на доступ, ст. 14)

Машиночитаемая выгрузка всех ПДн субъекта (профиль, согласия, история-если-opt-in, брони/выдачи) — `GET /api/privacy/subject/{id}/export` / ЛК «Скачать мои данные». Требует аутентификации субъекта или гранта DPO; сама выгрузка журналируется (`action=export`).

## 4. Версионирование политики и согласий

- **Политика обработки ПДн тенанта** — версионируемый артефакт (`privacy_policy`: `version` монотонный, `effective_from`, локализованный текст). Изменение текста → **новая версия**; прежняя сохраняется (доказательство «на что согласился субъект тогда»).
- **Запись согласия ссылается на версию** (`policy_version`). При новой версии с флагом `requires_reconsent`, затрагивающей права субъекта, продукт требует **переподтверждение** при следующем входе.
- Изменение текста политики и переключение действующей версии — операция под грантом, пишется в **общий аудит I1** (кто/когда/diff).

## 5. Ретенция

| Данные | Ретенция | Примечание |
|---|---|---|
| `consent` (записи согласий) | хранятся бессрочно (доказуемость прошлой обработки), без прямых ПДн после забвения субъекта | append-only |
| История чтения (opt-in) | по политике тенанта; при отзыве согласия — очистка/обезличивание на заданном горизонте | `tenant_pdn_profile.retention_*` |
| Завершённые выдачи / архив циркуляции | плановая авто-чистка по сроку (паритет `DelDate.gbl`) → авто-анонимизация | конфиг per-tenant |
| Журнал доступа к ПДн (`pdn_access_log`) | длительная (покрыть период проверок Роскомнадзора/ФСТЭК); партиционирование по периоду; архив — в шифрованное холодное хранилище | конфиг per-tenant, точные сроки не в документе |
| Аудит I1 (`audit`) | длительная, append-only | доказательная база |
| Резервные копии | по циклу ретенции бэкапов; забвение применяется при естественной ротации + крипто-стирание | — |

> **Принцип:** точные сроки (ретенция, реакция на запрос) различаются по тенантам и меняются в НПА → они **конфигурируемы** (`tenant_pdn_profile`), не зашиты в код/политику; таймеры с эскалацией.

## 6. Краткое соответствие нормам

| Норма | Как закрывается |
|---|---|
| 152-ФЗ ст. 6/9 (согласие, правовое основание) | слой `consent`, версионирование, гранулярность по целям (§2, §4) |
| 152-ФЗ ст. 5 ч.5 (минимизация) | по умолчанию минимум; opt-in история; не реплицируем паспорт/фото ([UZ §2.2](UZ_CLASSIFICATION.md)) |
| 152-ФЗ ст. 10 (спец. категории) | здоровье — только явное письменное согласие, отдельный пресет (§2) |
| 152-ФЗ ст. 14 (право на доступ) | экспорт данных субъекта (§3.4) |
| 152-ФЗ ст. 21 (право на удаление/уточнение) | erasure-workflow с анонимизацией ссылок (§3) |
