Иерархия: **Инициатива** = milestone (бизнес-цель/поток) · **Эпик** = issue · **Таск** = саб-issue · **Сабтаск** = чекбокс. **Сквозные потоки** (нити через все инициативы): **Миграция (strangler)** — [бенчмарк](MIGRATION_BENCHMARK.md) + dual-run/сверка/cutover; **Наблюдаемость** (OTel + trace/tenant-id) — с первого сервиса; **Безопасность/152-ФЗ** — в каждом таске; **Контракт-first** — OpenAPI как шов.

**Принцип «готовности» (Definition of Done) на каждый таск:** код + автотест + контракт-тест (если затрагивает API) + тест изоляции тенанта (если затрагивает данные) + наблюдаемость (trace/лог) + обновление контракта/доков. Без этого таск не «done».

## I1 — Фундамент платформы · Фаза 1 (старт реализации)
**Цель:** превратить PoC в мультиарендный фундамент на PostgreSQL с чистым швом и наблюдаемостью. Снимает упрёк «не постгрес».

### Эпик #112 — Мультиарендность и лицензирование
- **Таск #100 — Модель тенанта + изоляция (схема-на-тенанта / RLS)**
  - [ ] Спецификация изоляции (схема-на-тенанта; edge — БД-на-тенанта; `tenant_id`+RLS в общих сервисах)
  - [ ] Версионируемые миграции схемы + оркестрация апгрейда по тенантам
  - [ ] Tenant-aware пул соединений (PgBouncer), лимиты ресурсов (анти-noisy-neighbor)
  - [ ] **Негативные тесты изоляции** (ни один запрос не пересекает тенант) — в CI
  - [ ] Per-tenant ключи шифрования (envelope encryption)
- **Таск #101 — Сервис Identity / Tenancy / Licensing**
  - [ ] Сущности tenant / account / grant / role / audit (перенос Access-store)
  - [ ] JWT с tenant-claims; сессии guest/reader/staff
  - [ ] **Entitlements-API** (включённые модули у тенанта) — отдельно от биллинга
  - [ ] Аудит действий (запись/админ)
- **Таск #102 — Онбординг тенанта**
  - [ ] Создание библиотеки-тенанта: тип (публичная/школьная/вузовская), бренд/тема, пресет модулей
  - [ ] Начальная конфигурация баз/полей

### Эпик #99 — Прод-инфраструктура и PostgreSQL
- **Таск — Access-store sqlite → PostgreSQL** (#92): схема `schema_postgres.sql`, DAO на psycopg3, миграция данных, тесты
- **Таск — Прод-упаковка ядра**: Docker/Compose, конфиг через env/секреты (без секретов в образе), healthcheck/readiness
- **Таск — Каталог-ядро на PostgreSQL как сервис** (вынести из server-own PoC): записи JSONB+tsvector, holdings, location-дерево

### Эпик #128 — Платформенные практики
- **Таск #116 — OpenAPI contract-first** *(агент в работе)*: `openapi.yaml` как источник истины, валидация, версионирование, contract-tests (Pact)
- **Таск #117 — Наблюдаемость**: OpenTelemetry (трейсы+метрики+логи), сквозные **trace-id + tenant-id** во всех сервисах с первого дня
- **Таск #118 — CI/CD**: trunk-based, feature-flags, IaC (Terraform/GitOps), ephemeral preview на PR, авто contract/migration/изоляция-тесты в пайплайне

## Сквозной поток: Миграция из ИРБИС (strangler)
Нить через I1→I2→I4. Per-library: адаптер (Проход Б) → зеркало → **dual-run + сверка** → переключение чтения → записи → вывод ИРБИС. Производительность перелива — [MIGRATION_BENCHMARK](MIGRATION_BENCHMARK.md) *(бенчмарк-агент в работе)*. Миграция = часть онбординга тенанта (коммерчески — услуга, таск #111).

## Рекомендуемый порядок старта
1. **I1** целиком (фундамент SaaS+Postgres+контракт+наблюдаемость) — параллельно идут OpenAPI (#116) и бенчмарк миграции.
2. **I2** (релевантный поиск + навигаторы + ЛК) — быстрый видимый прогресс читателя.
3. **I3 + I4** (книговыдача/ячейки + edge/офлайн) — ключевые differentiators.
4. **I5 + I7** (сотрудник-паритет + коммерциализация), **I6** сопровождает всё.
