# Схема экранов сотрудника — по функциям и грантам (веб-ИРБИС)

> Веб-интерфейс сотрудника организован **по функциям, доступным по грантам учётки** (функция × база ×
> уровень, см. `ACCESS_MODEL`). Понятие «АРМ» здесь НЕ используется как раздел интерфейса — это лишь
> технический контекст протокола, который бэкенд подставляет сам (деталь реализации, см. колонку
> «контекст протокола»). Источник инвентаря функций — находки #3/#6 и мастер-карта §6. Версия 1.0, 20.06.2026.

---

## 1. Принцип навигации

**Одно приложение, без «переключения АРМов».** Левое меню показывает только те **группы функций**, на
которые у учётки есть хоть один грант. Внутри группы каждый экран/действие гейтится конкретным грантом
(функция × база × уровень). Сотрудник с грантами «выдача» + «каталогизация» видит обе группы вперемешку,
без понятия «я в Книговыдаче / я в Каталогизаторе».

«Контекст протокола» в таблицах ниже — это какой служебной сессией бэкенд дёрнет сервер (`IC_reg`
нужным типом). Пользователь его не видит; для интерфейса это неважно.

## 2. Группа функций «Обслуживание читателей» (гранты `circ.*`)

| Маршрут | Экран/функция | Компоненты | API (`P0`/`FUNCTION_PROTOCOL_MAP`) | Грант | Контекст протокола |
| --- | --- | --- | --- | --- | --- |
| `/circ/queue` | Очередь заказов | список заказов (RQST), отбор/отказ/переадресация | `GET /search` RQST, `update` RQST | `circ.queue:RQST:write` | BOOKLAND |
| `/circ/issue` | Выдача | поиск читателя (билет/ШК), выдача экземпляра (910) | `read/update` RDR+каталог | `circ.issue:RDR:write` | BOOKLAND |
| `/circ/return` | Возврат / продление | по ШК экземпляра/читателю | `update` RDR | `circ.return:RDR:write` | BOOKLAND |
| `/circ/reader/:id` | Карточка читателя | формуляр, выданное, право обслуживания, корректировка, посещение | `read/update` RDR | `circ.reader:RDR:write` | BOOKLAND |
| `/circ/debtors` | Должники | список, уведомления | `search/stat` | `circ.debtors:RDR:read` | BOOKLAND |

## 3. Группа функций «Каталогизация» (гранты `record.*`, `cat.gbl`)

| Маршрут | Экран/функция | Компоненты | API | Грант | Контекст |
| --- | --- | --- | --- | --- | --- |
| `/cat/search` | Поиск для правки | результаты с действием «редактировать» | `GET /search` | `record.read:db:read` | CATALOG |
| `/cat/record/new` | Создать запись | выбор рабочего листа → вкладки полей (Коды/Основное БО/Расш. БО/Специфические/Экземпляры/Систематизация/Содержание/КО/Редкие/Краеведение) | `recdummy`→`field*`→`update` | `record.write:db:write` | CATALOG |
| `/cat/record/:db/:mfn/edit` | Редактировать | те же вкладки РЛ; блокировка на время правки | `read(lock)`→`field*`→`update` | `record.write:db:write` | CATALOG |
| `/cat/record/:db/:mfn` (действие) | Удалить | логич./физич. удаление | `recdel`→`update` | `record.delete:db:admin` | CATALOG |
| `/cat/global` | Глобальная корректировка | задание gbl по отобранным | `gbl` | `cat.gbl:db:admin` | CATALOG |

> Загрузка сканов/PDF к записи — **не через протокол** (`putresourse` только текст) → файловый доступ к хранилищу.

## 6. Группа функций «Администрирование» (гранты `admin.*`)

| Маршрут | Экран/функция | Компоненты | API | Грант | Контекст |
| --- | --- | --- | --- | --- | --- |
| `/admin/databases` | Базы | новая/опустошить/удалить/импорт/экспорт | `adm_newdb/dbempty/dbdelete` | `admin.db:*:admin` | ADMINISTRATOR |
| `/admin/maintenance` | Обслуживание | словарь/реорг/диагностика/снятие блокировок | `adm_dbstart*/dbunlock` | `admin.db:*:admin` | ADMINISTRATOR |
| `/admin/accounts` | Учётки и гранты | **наша модель доступа**: учётки сотрудников, гранты, роли-пресеты | PostgreSQL (наша БД) | `admin.users:*:admin` | — (приложение) |
| `/admin/clients` | Клиенты/процессы | список клиентов/процессов сервера | `adm_getclientlist/getprocesslist/setclientslist` | `admin.clients:*:admin` | ADMINISTRATOR |

> Управление учётками/грантами (`/admin/accounts`) — это **наш** слой доступа в PostgreSQL, не команда
> ИРБИС. Остальное администрирование — поверх `IC_adm_*`. Самый строгий контур и аудит.

---

## 7. Общие правила

- **Сборка меню — по грантам.** Нет грантов группы → её нет в навигации. Внутри группы — гейтинг по
  конкретному гранту (например, видеть запись, но не редактировать).
- **Аудит** всех действий записи/администрирования (кто, функция, база, MFN, результат).
- **Доступность** ГОСТ Р 52872-2019 для всех форм (особенно вкладки РЛ каталогизации и таблицы).
- **Совмещение функций** — норма: один сотрудник = объединение грантов, экраны разных групп соседствуют.
