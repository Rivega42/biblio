# SPEC · ws2 — АРМ Комплектатор (полевой уровень)

> Детализация задачи **ws2** ([#184](https://github.com/Rivega42/biblio/issues/184), эпик #98) до уровня полей/БД/цикла — веб-замена десктоп-Комплектатора (самое низкое покрытие, 5%). Грунтовано на recon: [DB_ACQUISITION](../../recon/deep/reference/databases/DB_ACQUISITION.md) (CMPL/PODB/POST, цикл, КСУ, бланки), [FLC_VALIDATION](../../recon/deep/reference/format/FLC_VALIDATION.md). Родитель — [SPEC_I5](SPEC_I5_staff_arm_parity.md); движок — **ER #182**.

## 1. Доменная модель (3 БД → наши сущности)
| ИРБИС | Назначение | У нас |
|---|---|---|
| **CMPL** | комплектование (центральная): заказ/поступление/КСУ/выбытие | `acquisition` (orders, receipts, КСУ, write-offs) |
| **PODB** | заказ книг (входной каталог) | источник заказов → CMPL |
| **POST** | подписка периодики (входной каталог) | подписка → CMPL (связь ER #182/er2) |
**Тип записи — поле 920** (`ZK` заказ книги · `SZ` суммарный заказ · `OJK` период.+подписка · `KSU` поступление · `KS2` выбытие · `KSI/KS3` итоги · `KAT` каталог подписки · `IZD` организации). **27 точек доступа** комплектования (CMPL.INI [SEARCH]).

## 2. Заявка → заказ (поле 62)
**Подход:** БО книги (920=ZK) + **поле 62**: заявка (`^9` автор заявки `,AZ=` · `^-` кафедра · `^6` дата · `^1` требуемое кол-во · `^W` статус `StatusZ.mnu`) → заказ (`^A` организация `,OR=` · `^K` код · `^B` номер `,SZ=` · `^C` дата · `^D` ЗАКАЗАНО · `^U` №КСУ · `^G` ПОЛУЧЕНО). Мастер **CreateSZ** → запись 920=SZ (суммарные данные заказа на «организация+номер», ключ `SORZ=`); организация → запись 920=IZD (поле 81). Контроль `!62.pft` (цены `^n/p/s/r`), `!30.pft` («вводить в 62»).
**AC:** AC1 создание заявки/заказа (62) с организацией из справочника (IZD); AC2 Мастер формирует суммарный заказ SZ; AC3 статус заявки из `StatusZ.mnu`; AC4 одобрение/отклонение (поле 162); AC5 заявка на учебную лит-ру ВУЗ (поле 694, связь книгообеспеченность ws4).

## 3. Поступление + регистрация экземпляров (910)
**Подход:** в 62 заполняется `^G` ПОЛУЧЕНО + `^U` №КСУ; Мастер **SumEkzToSZ** суммирует полученные экз. (ключ `BORZ=`); контроль выполнения (`BORNZ=/nonzk=/vszk=/pszk=`). **Экземпляры — поле 910** (`910k.wss`, 18 подполей): `^A` статус (`ste.mnu`) · `^B` инв.№ (`,MIN=`, словарь подсказывает N+1) · `^H` штрих-код · `^D` МХР (`mhr.mnu`) · `^C` дата · `^E` цена (NNNN.NN) · `^U` №КСУ · `^Q` коллекция · `^F` канал (`KP.mnu`) · `^V`/`^X` выбытие. autoin→907; контроль `!910.pft` (дублетность инв.№/штрих-кода).
**AC:** AC1 фиксация поступления (62^G/^U) и пересчёт SZ; AC2 регистрация экземпляров (910) с инв.№ (MIN+1), статусом, ценой, КСУ; AC3 дублетный инв.№/штрих-код ловится.

## 4. КСУ поступления (920=KSU, поле 88) — ядро «наполненности»
**Подход:** **Книга суммарного учёта, часть 1 (ПОСТУПЛЕНИЕ)** — поле **88** (`88.wss`, 16): `^A` год+№ записи (`,MKSU=`) · `^Y` №акта · `^B` дата · `^C` №сопр.док · `^D` источник (`,OR=`) · `^9` №заказа · `^E` число наименований · `^F` число экз. · `^G` на сумму · `^H` платно/бесплатно (`netda.mnu`) · `^N` НДС · `^L` источник финанс. (`Istfin.mnu`) · `^M` оплачено (`Nd.mnu`) · `^J/^I` счёт (ИРБИС-Парус). Мастер **«Пополнение записи КСУ»** авто-заполняет распределение партии: 17/18/19 (числа наименований), 44–49 (тип/язык/УДК/ББК/направления), 145–158 (печатные/непечатные/отеч.-иностр./фонды/канал/ВУЗ). Индексы `KSU=/AS=/NKSUK=/DP=`. Контроль `!88.pft` (дублетность №, структура суммы). **Печать — RTF-бланк `KSUN1SH`.**
**AC:** AC1 запись КСУ поступления (88) с источником/суммой/финансированием; AC2 авто-распределение партии (наименований/экз./разделы); AC3 печать бланка КСУ ч.1; AC4 контроль дублетности № записи и структуры сумм.

## 5. Списание / выбытие (920=KS2, поля 888/800)
**Подход:** **КСУ часть 2 (ВЫБЫТИЕ)** — Мастер списания → поле **888** (`888.wss`: `^A` №записи `,MKS2=` · `^B` дата · `^P` причина `prsp.mnu` · `^D` МХР · `^F` цена), проставляет в экземплярах `910^V` (№КСУ выбытия) + `^X` (кол-во), статус `910^A=6`. Поле **800** — акт передачи (`^A` №акта · `^E` вид `vp.mnu` · `^M` новое МХР · `^Z` число). `InsteadLost.gbl` — замена утерянных (поле 80, кросс-БД с ЭК). Контроль `!888.pft`/`!910W.pft`. **Печать — RTF-бланк `KSUN2SH`.**
**AC:** AC1 списание с причиной (`prsp.mnu`), проводкой в 910 (^V/^X, статус 6) и актом (800); AC2 число выбывающих ≤ имеющихся; AC3 печать бланка КСУ ч.2; AC4 замена утерянных.

## 6. Итоги движения фонда (920=KSI/KS3, поля 88/889)
**Подход:** итоговые записи движения фонда; **КСУ часть 3 (ИТОГИ)** по разделам/видам/носителям — **RTF-бланк `KSUN3SH`**; статистика (`StatKsu/StatKsuSum.gbl`). Подписка-итог — поле 889.
**AC:** AC1 формирование итогов движения фонда (поступило/выбыло по разделам); AC2 печать бланка КСУ ч.3; AC3 статистика КСУ за период.

## 7. Подписка периодики (POST → 920=OJK)
**Подход:** запись подписки: 86 индекс (`@!86`) · 110 (тип/вид сериального/периодичность) · 33 период (`33.mnu`: год/полугодия/кварталы/нестандарт) · цена/комплекты (34, шаблон `34.wss`) · 39 заказ по подразделениям. Перенос в CMPL как OJK (`MoveNewKp.gbl`); связь по индексу 86 (`TP=` есть в CMPL / `TA=` нет). ФЛК (`DBNFLC` POST): обязательны 110^b/110^x/86/цена/заглавие + дубль-индекса.
**AC:** AC1 оформление подписки (индекс/период/цена/периодичность); AC2 предупреждение о дубле индекса в CMPL; AC3 заказ по подразделениям; AC4 перенос подписки в CMPL (OJK).

## 8. Заказ книг (PODB) и перенос в ЭК
**Подход (PODB):** входной каталог заказа (Заглавие/Автор/Издательство/КС), дедупликация против **CMPL и IBIS** (`!KDEX/!KDK.pft`), перенос в CMPL как ZK (`MoveZakKp.gbl`: организации→IZD, заказы→SZ). **Перенос в ЭК (поле 66 = флаг):** ToCat конвертирует БО CMPL → формат IBIS (`STN.FST/MNOG.FST/UMARCIW.FST`: 922/330→700/701/200/463; 920→PAZK/SPEC/PVK/J/ASP); экземпляры (910)→ЭК с техн.путём (901) и связью 938 (`pkj.gbl`); FST-строка 66 ставит `VD=DEL` на исходник.
**AC:** AC1 заказ из PODB с дедупликацией (CMPL/IBIS); AC2 перенос поступившего БО в ЭК (формат IBIS) с экземплярами и техн.путём; AC3 исходник помечается удалённым.

## 9. Финансы и комплексный поиск
**Финансы:** цены/суммы (62, 88^G/N/P), источник финанс. (`Istfin.mnu`), каналы (`KP.mnu`: ГПНТБ/ВИНИТИ/Покупка/Книгообмен/Дары), валюта (`val.mnu`), интеграция со счётом (ИРБИС-Парус). **Комплексный поиск (SEARCHCMP, 27 точек):** недовыполненные заказы (`NZT=`), дезидераты (`DES=`), выполненные (`BVZ=`), КСУ поступления/выбытия (`KSU=/KS2=`), по дате/акту/инв.№. Плюс **demand-driven** дозаказ по отказам/спросу (PMD) и под книгообеспеченность (ws4).
**AC:** AC1 комплексный поиск по статусам заказа/КСУ; AC2 финансовые проводки/суммы; AC3 сигнал дозаказа (низкая обеспеченность / отказы).

## Критический путь ws2
1. **заявка→заказ (62) + SZ + IZD** → 2. **поступление + экземпляры (910)** → 3. **КСУ поступления (88) + распределение + бланк ч.1** → 4. **списание/выбытие (888/800) + бланк ч.2** → 5. **итоги движения (KSI) + бланк ч.3** → 6. **подписка (POST/OJK)** → 7. **перенос в ЭК (66/ToCat)** → 8. финансы + комплексный поиск.
**Выходной критерий ws2:** комплектатор ведёт полный цикл в вебе — заявка/заказ, поступление с регистрацией экземпляров, **КСУ (3 части с печатными бланками)**, списание/выбытие с актами, подписка периодики, перенос в ЭК, финансы и комплексный поиск — без десктоп-АРМа.
