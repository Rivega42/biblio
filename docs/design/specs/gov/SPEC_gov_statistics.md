> Закрытие гос-рыночного пробела **B1** из [DESIGN_COVERAGE §B](../../DESIGN_COVERAGE.md) (эпик [#188](https://github.com/Rivega42/biblio/issues/188)): «Гос-статистика — 6-НК, КСУ-выгрузки, дневники, COUNTER/SUSHI. 🟡/❌ → I5/I2». Цель — единый **движок регламентированной отчётности**: декларативные определения отчётов → агрегаты на PostgreSQL → экспорт в гос-форматах (XML/XLSX Росстата, RTF/PDF-бланки КСУ, COUNTER R5).

Источники-агрегаты — из спек АРМов: [SPEC_ws2 §4–6](../SPEC_ws2_acquisition.md) (КСУ ч.1–3: поля 88/888/800/889, индексы `KSU=/KS2=/AS=/DP=`), [SPEC_ws3 §*](../SPEC_ws3_circulation.md) (выдача RDR.40, статусы экземпляра), [SPEC_ws5 §7](../SPEC_ws5_admin.md) (отчёты КСУ/6-НК/COUNTER в мониторинге).
>
> **Переиспользуемые движки (не дублируем):** агрегация КСУ ложится на **A3 `.gbl`** ([SPEC_engine_gbl](../engines/SPEC_engine_gbl.md) — те же `StatKsu.gbl` как golden-тест §9), а рендеринг бланков/таблиц/статформ — на **A1 PFT/печать** ([SPEC_engine_pft](../engines/SPEC_engine_pft.md) §3 — табличный движок `.tbg/.tab`, статформы `.stf`, эндпоинты `POST /api/print/table`, `POST /api/print/stat`). >
> **DoD-гейт (как в I1/I5/A1–A3):** код + автотест + контракт-тест (openapi) + **тест изоляции тенанта** (агрегаты/выгрузки per-tenant) + наблюдаемость (trace/`tenant_id`/`report_run_id`, время сборки отчёта в мс) + a11y HTML-показа отчётов (WCAG 2.2 AA) + обновление openapi/доков. Размеры S ≤2 дн · M 3–5 дн · L 1–2 нед.

## 0. Границы движка и принципы

- **Что делает:** хранит **декларативные определения отчётов** (что считать, как группировать, на какой бланк/в какой формат выгружать) → собирает **агрегаты** запросом/материализованным представлением PostgreSQL по выборке записей тенанта → прогоняет результат через рендереры (A1) и **экспортёры гос-форматов** → отдаёт файл/поток + сохраняет прогон (`report_run`) для аудита и воспроизводимости.
- **Что НЕ делает (делегирует):**
  - вычисление PFT-выражений ячеек/заголовков, RTF/PDF/XLSX-вёрстку таблиц и бланков → **A1 PFT/печать** (§3 там; здесь — только определение отчёта и набор агрегатов);
  - сложную пакетную перекладку данных в записи КСУ-итогов (поля 44–49/145–158, 889) → **A3 `.gbl`** (мастер «Пополнение записи КСУ» в [SPEC_ws2 §4](../SPEC_ws2_acquisition.md));
  - авторизацию/гранты/энтайтлмент/аудит-хранилище → **I1 #101**;
  - доставку выгрузок во внешние системы (СМЭВ/НЭБ/казначейство) — это смежная дыра **B3** ([DESIGN_COVERAGE §B3](../../DESIGN_COVERAGE.md)); здесь движок **формирует файл по схеме**, маршрут/транспорт — за B3 (стык §7).
- **Принцип «отчёт = данные + представление + формат»:** одно определение отчёта может рендериться в несколько целевых форматов (КСУ ч.1 → RTF-бланк `KSUN1SH` **и** XLSX **и** HTML-показ) — как один IR в A1 даёт HTML+PDF+RTF.
- **Принцип воспроизводимости:** каждый прогон фиксирует **снимок параметров** (период, БД, фильтры) + **версию определения отчёта** + результат → повторный рендер старого прогона байт-в-байт воспроизводим (важно для гос-сдачи: отправленная форма должна быть восстановима).
- **Принцип паритета методики:** агрегаты считаются по тем же полям/индексам, что и формы ИРБИС (`StatKsu.gbl`, `TABSTATJ`, `Form1-5.stf`, `TSPROST`/`StatLand`); методические расхождения с ИРБИС фиксируются явно в определении отчёта (а не «зашиты» в код).

## 1. 6-НК — федеральная форма стат-наблюдения

> 6-НК «Сведения об общедоступной (публичной) библиотеке» (Росстат/Минкультуры) — годовая. Recon её **не содержит** (форма — внешний нормативный документ, не файл дистрибутива; `TODO #GOV-01` — точная редакция формы/шаблона года уточняется при реализации). Поэтому здесь определяем **структуру-агрегатор и маппинг на наши данные**, а конкретные строки/коды граф выносятся в версионируемое определение отчёта (`report_def`), сидируемое per-год.

### 1.1 Структура формы (разделы)

6-НК — табличная форма из нумерованных разделов; для маппинга существенны:

| Раздел | Содержание | Наш агрегат-источник |
|---|---|---|
| **Раздел 2. Формирование и использование библиотечного фонда** | объём фонда на конец года, поступило/выбыло за год — по видам документов и носителям (печатные, электронные сетевые/локальные, АВ, прочие), выдано документов | КСУ ч.1/ч.2/ч.3 (поля 88/888/889, [SPEC_ws2 §4–6](../SPEC_ws2_acquisition.md)) + экземпляры 910 (наличие/статусы) + книговыдача RDR.40 ([SPEC_ws3](../SPEC_ws3_circulation.md)) |
| **Раздел 3. Электронные (сетевые) ресурсы** | объём собственных БД, доступ к инсталлированным/сетевым удалённым лицензионным БД, число выданных э-документов | счётчик записей ЭК; реестр лицензий ЭР (ER #182); COUNTER (§4) |
| **Раздел 4. Число пользователей и посещений** | зарегистрированные пользователи (всего/до 14/15–30/инвалиды), посещения (физические/обращения к веб-сайту) | реестр читателей RDR + дневник посещений (§3) + веб-аналитика OPAC |
| **Раздел 5. Информационное обслуживание** | справки и консультации, число обращений удалённых пользователей | дневник справок/обращений (§3), COUNTER (§4) |
| **Раздел 7. Поступление и использование финансовых средств** | (за пределами библиографического ядра — берётся из финмодуля/Парус, [SPEC_ws2 §9](../SPEC_ws2_acquisition.md)) | финансовые проводки (вне ядра — `TODO #GOV-02`) |

> Точная нумерация граф/строк действующей формы — в `report_def` версии года (см. §5.2); ниже — методический маппинг, инвариантный к перенумерации.

### 1.2 Источники-агрегаты и маппинг на наши данные

| Показатель 6-НК | Источник (поле/событие) | Формула агрегата |
|---|---|---|
| **Состоит на конец года** (по видам/носителям) | экземпляры 910 со статусом «в фонде» (не выбывшие: `910^A` ≠ списан/утерян; [SPEC_ws2 §3/§5](../SPEC_ws2_acquisition.md)) + вид документа 900^B/раздел 60 | `COUNT(*)`/`SUM(910^…)` GROUP BY вид/носитель на дату `≤ 31.12` |
| **Поступило за год** | КСУ ч.1 — поле 88 (`^E` число наименований, `^F` число экз., `^G` сумма), индекс `KSU=`; распределение партии (поля 145–158: печатные/непечатные/отеч.-иностр., канал, ВУЗ — [SPEC_ws2 §4](../SPEC_ws2_acquisition.md)) | `SUM(88^F)` за период GROUP BY вид/носитель/канал |
| **Выбыло за год** | КСУ ч.2 — поле 888 (`^P` причина `prsp.mnu`, число), проводка в 910 (`^V`/`^X`, статус 6); акты 800 | `SUM(888^…)` за период GROUP BY причина/вид |
| **Выдано документов за год** | книговыдача — выдачи (RDR.40, маркер выдачи; [SPEC_ws3](../SPEC_ws3_circulation.md)), дневник выдачи (§3) | `COUNT(выдача)` за период GROUP BY вид/категория читателя/форма (зал/абонемент/ЭДД) |
| **Зарегистрированные пользователи** | реестр читателей RDR (активные за год: с операцией в периоде), категория/возраст (поля RDR) | `COUNT(DISTINCT reader)` GROUP BY возрастная группа/категория |
| **Посещения** | дневник посещений (§3, события входа/обращения) + веб-обращения OPAC | `SUM(посещения)` физ. + `SUM(сессии/визиты)` веб |
| **Справки/консультации** | дневник справок (§3, событие «справка») | `COUNT(справка)` GROUP BY тип |
| **Обращения к ЭР** | COUNTER (§4, Total_Item_Requests) + обращения к собственным БД | `SUM(COUNTER metric)` + лог OPAC |
| **Объём собственных БД / э-каталога** | счётчик записей ЭК (MFN тенанта), оцифрованные объекты (DG) | `COUNT(record)` по БД |

**Где данных нет в ядре:** финансы (раздел 7), часть посещений (турникеты/массовые мероприятия) — помечаются в `report_def` как **ручной ввод/импорт** (поля формы заполняются оператором или подтягиваются из смежных систем); движок поддерживает «гибридный» отчёт (часть граф — авто-агрегат, часть — ручной override с журналированием).

### 1.3 Формат выгрузки

- **XML по шаблону Росстата** (электронная сдача через ПО «Подготовка отчётов» / web-сбор): формируем XML по XSD-шаблону формы соответствующего года. Шаблон/XSD — внешний (`TODO #GOV-01`); движок хранит **схему-маппинг** «показатель → graph/row XPath» в `report_def`, генерирует XML, валидирует против XSD перед отдачей.
- **XLSX по бумажному бланку** — машиночитаемая таблица для внутренней сверки/печати (рендер A1 через `POST /api/print/table` с табличной формой-аналогом).
- **HTML-показ** — экранная сводка с раскрытием «как посчитано» (drill-down до выборки MFN/событий) — для проверки методистом перед сдачей.

**AC (6-НК):** AC1 агрегаты разделов 2–5 собираются из КСУ/910/RDR/дневников/COUNTER за выбранный год; AC2 граф формы маппится на агрегаты через версионируемое `report_def` (перенумерация года не ломает движок); AC3 XML валидируется против XSD-шаблона и отвергается с диагностикой при несоответствии; AC4 гибридные графы (ручной ввод/override) журналируются; AC5 прогон воспроизводим (снимок параметров + версия `report_def`).

## 2. КСУ-отчёты и выгрузки (части 1–3 + итоги движения фонда)

> КСУ (Книга суммарного учёта) — полевой цикл описан в [SPEC_ws2 §4–6](../SPEC_ws2_acquisition.md); здесь — **отчётный/выгрузочный** слой над ним: бланки, агрегаты периода, выгрузка в бухгалтерию/казначейство.

### 2.1 Агрегаты по частям

| Часть КСУ | Запись/поля (ws2) | Бланк (recon) | Агрегаты периода |
|---|---|---|---|
| **Ч.1 Поступление** | 920=KSU, поле **88** (источник `^D`, наименований `^E`, экз. `^F`, сумма `^G`, финанс. `^L`/`^N`/`^M`); распределение 17/18/19, 44–49, 145–158 | RTF-бланк **`KSUN1SH`** ([SPEC_ws2 §4](../SPEC_ws2_acquisition.md)) | итог поступлений за период по номерам записей КСУ; разбивка по разделам/видам/каналам/источникам финанс. |
| **Ч.2 Выбытие** | 920=KS2, поля **888** (причина `^P`, число), **800** (акт, вид выбытия `vp.mnu`) | RTF-бланк **`KSUN2SH`** ([SPEC_ws2 §5](../SPEC_ws2_acquisition.md)) | итог выбытий по причинам/актам/разделам |
| **Ч.3 Итоги движения** | 920=KSI/KS3, поля 88/**889** (подписка-итог) | RTF-бланк **`KSUN3SH`** ([SPEC_ws2 §6](../SPEC_ws2_acquisition.md)) | состояло + поступило − выбыло = состоит, по разделам/видам/носителям; баланс на начало/конец |

- **Движок агрегации** — `StatKsu.gbl`/`Staks2.gbl`/`StatKsuSum.gbl` ([GLOBAL_CORRECTION §3.2](../../../recon/deep/reference/format/GLOBAL_CORRECTION.md)): накопление по полям 44/744, счётчики `&uf('++C…')`, переменные между записями, `PUTLOG`. В вебе — **на выбор**: (а) прогон эталонного `.gbl` через движок **A3** (golden-паритет методики, [SPEC_engine_gbl §9](../engines/SPEC_engine_gbl.md) — `StatKsu.gbl` уже в его тест-матрице), либо (б) **SQL-агрегат** по тем же полям (быстрее, но методику обязаны сверить с `.gbl` golden-тестом). Решение: §5.4 — SQL-агрегат как первичный путь, `.gbl`-прогон как эталон-сверка.
- **Индексы-источники** (CMPL): `KSU=` (поступление), `KS2=` (выбытие), `AS=` (№ акта), `DP=` (дата/подразделение), `NKSUK=` — из [SPEC_ws2 §4](../SPEC_ws2_acquisition.md).

### 2.2 Выгрузка в казначейство/бухгалтерию (вузы)

- **Назначение:** библиотека вуза передаёт данные о поступлении/выбытии (стоимость, инв.№, основные средства) в бухгалтерию (ИРБИС-Парус, [SPEC_ws2 §9](../SPEC_ws2_acquisition.md): поля 88 `^J/^I` счёт, цена 910^E) и/или в казначейскую систему.
- **Форматы выгрузки:** XLSX-ведомость (инв.№, БО, цена, дата, № акта — аналог инвентарной книги `TABINW`, [PFT_MODULES §3.3](../../../recon/deep/reference/format/PFT_MODULES.md)); CSV/XML-обмен с Парус (маппинг поле 88^J/^I → счёт); акт выбытия (RTF/PDF — A1, бланк `KSUN2SH` + акт `!WKW`). **Структуру обмена Парус** recon не фиксирует (`TODO #GOV-03`) — маппинг выносится в `export_profile` тенанта.
- **Per-tenant кастомизация:** реквизиты учреждения, план счетов, формат инв.№ — через `print_profile`/`export_profile` (A1 §4–5).

**AC (КСУ):** AC1 агрегаты ч.1/ч.2/ч.3 за период собираются из полей 88/888/800/889 и совпадают с прогоном эталонного `StatKsu.gbl` (golden-сверка методики); AC2 бланки `KSUN1SH/KSUN2SH/KSUN3SH` рендерятся в RTF/PDF (A1) с реквизитами тенанта; AC3 итоги движения сводят баланс (состояло+поступило−выбыло=состоит) по разделам/видам/носителям без расхождения; AC4 выгрузка в бухгалтерию (XLSX/Парус-обмен) формируется по `export_profile`; AC5 прогон отчёта КСУ воспроизводим и аудируется.

## 3. Дневники (журналы операций)

> «Дневник работы библиотеки» — учёт ежедневных операций: посещения, выдача, поиск/обращения, справки. В recon табличные формы по книговыдаче подтверждены: `TSPROST` (спрашиваемость), `StatLand` (выдачи по месяцам), `TReadHistory`/`TextHistory` (история чтения), `ProvRDR_U` (контроль статуса U) ([PFT_MODULES §3.5](../../../recon/deep/reference/format/PFT_MODULES.md), порт `.STF`/`.tab`). Дневник — **витрина над потоком событий**, а не отдельная БД.

### 3.1 Сбор событий

- **Источник — поток доменных событий** (единая событийная шина платформы, ср. edge/синк [SPEC_I4](../SPEC_I4_edge_offline.md) и аудит I1 #101). Релевантные типы:
  - `circulation.issue` / `circulation.return` / `circulation.renew` — выдача/возврат/продление (RDR.40, статусы экземпляра; [SPEC_ws3](../SPEC_ws3_circulation.md));
  - `reader.visit` — посещение (физический вход / открытие сессии в зале);
  - `opac.search` / `opac.session` — поисковые обращения, веб-визиты (OPAC, [SPEC_I2](../SPEC_I2_catalog_search.md));
  - `reference.query` — справка/консультация (тип: тематическая/адресная/фактографическая/уточняющая);
  - `eresource.access` — обращение к ЭР (см. COUNTER §4).
- **Хранение:** партиционированная по времени таблица событий схемы тенанта `t_<slug>` (`stat_event(tenant_id, ts, kind, actor_ref?, object_ref?, payload jsonb)`), индекс по `(kind, ts)`; ретенция/архивация — настройка тенанта. Дневник **читает** этот поток, не порождает его.

### 3.2 Табличные формы и периодичность

- **Формы-витрины** (рендер A1 — `POST /api/print/table`, табличный движок `.tbg`):
  - **дневной/месячный/годовой свод** посещений/выдач/справок (аналог `StatLand` «выдачи по месяцам»);
  - **спрашиваемость** (`TSPROST`) — обращаемость фонда (выдачи/экземпляр);
  - **история чтения** (`TReadHistory`/`TextHistory`) — по читателю/изданию.
- **Периодичность:** день/неделя/месяц/квартал/год; отчёты планируются (§5.3) либо строятся по запросу за произвольный период.
- **Привязка к 6-НК:** дневник посещений → раздел 4; справки → раздел 5; выдачи → раздел 2 (агрегаты §1.2 читают тот же `stat_event`).

**AC (дневники):** AC1 события выдачи/возврата/посещения/поиска/справки регистрируются в `stat_event` тенанта; AC2 свод за день/месяц/год строится как табличная форма (A1) с корректными итогами; AC3 спрашиваемость/обращаемость считается (выдачи на экземпляр); AC4 дневник изолирован per-tenant; AC5 ретенция/архивация событий настраивается.

## 4. COUNTER / SUSHI (статистика э-ресурсов)

> Международный стандарт учёта использования э-ресурсов: **COUNTER R5** (отчёты использования) + **SUSHI** (REST/SOAP-API автоматического сбора). В recon-дистрибутиве ИРБИС отсутствует (`TODO #GOV-04` — это новая, не-паритетная возможность; в DESIGN_COVERAGE отмечена как часть B1 ❌). Нужны две роли: библиотека как **потребитель** (harvest у поставщиков) и как **поставщик** (отдаём свою статистику ЭБ/собственных БД).

### 4.1 Потребитель (harvest у поставщиков через SUSHI)

- **SUSHI-клиент**: периодический сбор COUNTER R5-отчётов (`TR` Title Master Report, `DR` Database Report, `PR` Platform Report, `IR` Item Report) у поставщиков (eLibrary, ЭБС, издатели) по их SUSHI-эндпоинту (`/reports`, `/reports/{id}`, `/status`, `/members`) с `customer_id`/`requestor_id`/`api_key`.
- **Хранение:** нормализованные метрики (`counter_usage(tenant_id, platform, report_type, title_ref?, db_ref?, metric_type, period_month, count)`) — `Total_Item_Requests`, `Unique_Item_Requests`, `Searches_Regular` и т.п.
- **Применение:** разделы 3/5 формы 6-НК (обращения к лицензионным ЭР), отчёт окупаемости ЭР (cost-per-use, связь с реестром лицензий ER #182).

### 4.2 Поставщик (отдаём COUNTER/SUSHI по своим ЭР)

- **COUNTER-отчёты по собственной ЭБ/БД:** на основе `eresource.access`-событий (§3.1) формируем `DR`/`PR`/`IR` в формате COUNTER R5 (JSON по COUNTER_SUSHI API spec + табличный XLSX/TSV).
- **SUSHI-сервер:** REST-эндпоинты `GET /sushi/r5/reports`, `GET /sushi/r5/reports/{id}` (per-tenant, аутентификация `requestor_id`+`api_key`) — отдаём свою статистику внешним агрегаторам/учредителю.

**AC (COUNTER/SUSHI):** AC1 SUSHI-клиент собирает COUNTER R5 (`TR/DR/PR/IR`) у настроенного поставщика и нормализует метрики per-tenant; AC2 метрики питают разделы 3/5 формы 6-НК и cost-per-use; AC3 собственные COUNTER-отчёты формируются из `eresource.access` в формате R5 (JSON+TSV); AC4 SUSHI-сервер отдаёт отчёты по `requestor_id`/`api_key` с изоляцией тенанта; AC5 расхождения харвеста (нет данных/ошибка поставщика) журналируются, прогон не падает.

## 5. Движок отчётности (ядро B1)

### 5.1 Архитектура

```
 report_def (декларативное определение, per-tenant, версионируемое)
        │  · источники-агрегаты (SQL / matview / .gbl-эталон)
        │  · группировки/оси/фильтры (период, БД, вид, носитель)
        │  · маппинг показатель → graph/row целевой формы
        │  · целевые форматы (xml-rosstat | rtf-blank | xlsx | html | counter-r5)
        ▼
 [Планировщик/триггер] ──▶ [Сборщик агрегатов] ──┬─ SQL-агрегат по выборке MFN/событий
        (cron | по запросу)                       ├─ materialized view (тяжёлые/частые)
        │                                          └─ async job (большие периоды)
        ▼                                          (изоляция: search_path=t_<slug>)
 [report_run] (снимок параметров + версия def + результат-агрегат)
        ▼
 ┌───────────────┬──────────────┬──────────────┬──────────────┐
 ▼               ▼              ▼              ▼              ▼
[A1 табл./бланк][XML-Rosstat] [XLSX]      [COUNTER R5]   [HTML-показ
 RTF/PDF (КСУ)] (+XSD-валид.)             (JSON/TSV)      + drill-down]
```

- **Сборщик агрегатов** не интерпретирует PFT и не верстает — он считает числа (SQL/matview/job) и складывает в `report_run`. **Рендеринг** (вёрстка таблицы/бланка) — всегда A1; **экспорт гос-схемы** (XML/COUNTER) — собственные экспортёры B1.

### 5.2 Декларативное определение отчёта (`report_def`)

Схема `t_<slug>` + наследуемые системные дефолты в `control` (как `format_template` в A1 §5):

| Поле | Назначение |
|---|---|
| `name`, `kind` (`6nk`/`ksu1`/`ksu2`/`ksu3`/`diary`/`counter`/`custom`) | идентификатор и класс |
| `version`, `is_active`, `inherited_from` (system/tenant) | версионирование (immutable-версии + указатель активной — паритет с A1) |
| `sources` (jsonb) | агрегаты: список `{metric, expr(sql|matview|gbl-ref), group_by[], filter}` |
| `axes` (jsonb) | группировки/оси сводной таблицы (аналог `.stf` строк 2–6 + раскодировка `.mnu`, [PFT_MODULES §4.2](../../../recon/deep/reference/format/PFT_MODULES.md)) |
| `params` (jsonb) | параметры запуска (период `от/до`, БД, подразделение — аналог `%n` `.stf`/`.gbl`) |
| `mapping` (jsonb) | показатель → graph/row/XPath целевой формы (для XML-Росстат); пустое — для свободных таблиц |
| `targets` (`xml-rosstat`/`rtf-blank`/`xlsx`/`html`/`counter-r5`) + ссылки на A1-формы/бланки | целевые форматы |
| `manual_fields` (jsonb) | графы ручного ввода/override (гибридные формы, §1.2) |

> `report_def` — это «`.stf`/`.tbg` нового поколения»: то, что в ИРБИС разнесено по `.stf` (статзадание), `.tab/.tbg` (вёрстка), `.srw` (сортировка) и `.gbl` (накопление), у нас сведено в одно версионируемое определение; вёрстка делегирована A1, накопление — SQL/`.gbl`-эталон.

### 5.3 Планировщик и async-исполнение

- **Планировщик:** cron-расписания per-tenant (ежедневный дневник, месячный КСУ-свод, годовая 6-НК, периодический SUSHI-harvest). Реализация — на платформенном шедулере; задание = `{report_def, params-шаблон, расписание, target}`.
- **Async-jobs:** тяжёлые/большие периоды считаются в очереди (per-tenant воркеры, изоляция — как в A3 §5.3): статус через `GET /api/reports/runs/{id}`, прогресс. Лёгкие (малый период) — синхронно.
- **Materialized views:** частые/тяжёлые агрегаты (фонд на конец года, обращаемость) материализуются и обновляются по расписанию/инкрементально; источник — события `stat_event` и поля КСУ/910.

### 5.4 Агрегаты на PostgreSQL — методический паритет

- **Первичный путь — SQL/matview** по тем же полям/индексам, что формы ИРБИС (быстро, наблюдаемо).
- **Сверка методики — `.gbl`-эталон:** для КСУ-агрегатов SQL-результат **обязан совпадать** с прогоном эталонного `StatKsu.gbl`/`StatKsuSum.gbl` через A3 на тех же данных (golden-тест §8). При расхождении — расхождение методики фиксируется в `report_def` (а не «правится» молча).
- **Per-tenant кастом-формы:** тенант может определить свой `report_def` (`kind:custom`) поверх тех же агрегатов (low-code D1, [SPEC_ws5 §4](../SPEC_ws5_admin.md)) — без релиза; изоляция обязательна.

### 5.5 Экспортёры гос-форматов

| Экспортёр | Вход | Выход | Валидация |
|---|---|---|---|
| `xml-rosstat` | агрегаты + `mapping` | XML по XSD-шаблону года | XSD-валидация перед отдачей; отказ с диагностикой |
| `rtf-blank`/`pdf` | агрегаты | КСУ-бланки `KSUN1-3SH`, акты (через A1) | визуальный snapshot-тест бланка |
| `xlsx` | агрегаты/таблица | ведомость/свод (через A1 `print/table` или прямой XLSX-writer) | схема колонок |
| `counter-r5` | `counter_usage`/события | COUNTER R5 JSON + TSV | соответствие COUNTER_SUSHI spec |

## 7. Стык с гос-выгрузками (B3) и финмодулем

- **B3 (НЭБ/СМЭВ/СКБР/ЛИБНЕТ/казначейство, [DESIGN_COVERAGE §B3](../../DESIGN_COVERAGE.md)):** B1 **формирует файл по схеме** (6-НК XML, КСУ-ведомость, Парус-обмен); **транспорт/маршрут/подпись** (отправка в СМЭВ, ЭП) — за B3. Контракт стыка: B1 кладёт готовую выгрузку в `report_run` с типом схемы; B3 забирает и доставляет.
- **Финансы (раздел 7 формы 6-НК, выгрузка в бухгалтерию):** числовое ядро — за финмодулем/Парус ([SPEC_ws2 §9](../SPEC_ws2_acquisition.md), ER #182); B1 подтягивает финансовые агрегаты как внешний `source` (или принимает ручным вводом, §1.2).
- **DG/ЭБ (объём оцифровки, обращения к ПТ):** агрегаты собственных БД и обращений к полным текстам — из DG/DAM ([SPEC_ws5 §5](../SPEC_ws5_admin.md)).

## 9. Критический путь B1

1. **Поток событий `stat_event` + дневники** (§3) — фундамент агрегатов выдачи/посещений/справок; питает 6-НК раздел 2/4/5. →
2. **Сборщик агрегатов + `report_def` + `report_run`** (§5.1–5.2) на JSONB-ядре и SQL/matview, с изоляцией тенанта. →
3. **КСУ-агрегаты + бланки** (§2): SQL-агрегат + golden-сверка с `StatKsu.gbl` (A3) + рендер `KSUN1-3SH` (A1) — это ИРБИС-паритет, без него вузы не перейдут. →
4. **6-НК**: маппинг агрегатов → `report_def` + XML-Росстат-экспортёр + XSD-валидация + гибридные/ручные графы (§1). →
5. **Планировщик + async-jobs + materialized views** (§5.3) — годовые/месячные своды без блокировки. →
6. **COUNTER/SUSHI** (§4): сначала harvest-потребитель (для разделов 3/5 формы 6-НК), затем SUSHI-сервер. →
7. **Выгрузки в бухгалтерию/казначейство** (§2.2) + стык с B3 (§7) + per-tenant кастом-формы (§5.4).

**Выходной критерий B1:** методист тенанта формирует за период **6-НК** (агрегаты из КСУ/910/RDR/дневников/COUNTER → XML по XSD Росстата + сводка с drill-down), **КСУ ч.1–3** (бланки `KSUN1-3SH` + выгрузка в бухгалтерию, методика сверена с `StatKsu.gbl`), **дневники** (свод посещений/выдач/справок/спрашиваемости) и **COUNTER/SUSHI** (harvest + отдача) — по расписанию или запросу, воспроизводимо и с изоляцией тенанта, без десктоп-ИРБИС.

## 10. Риски и открытые вопросы

**Риски:**
- **R1 (методический паритет КСУ):** SQL-агрегат легко разойтись с методикой `StatKsu.gbl` (накопление переменных между записями, фильтры по 44/744). Митигация — golden-сверка с прогоном `.gbl` через A3 на одних данных (§8), расхождения фиксируются в `report_def`, не правятся молча.
- **R2 (внешние схемы вне recon):** 6-НК XSD, формат Парус, COUNTER R5/SUSHI — нормативные документы, не файлы дистрибутива. Митигация — схемы-маппинги в версионируемых `report_def`/`export_profile` (правятся под редакцию года без релиза), XSD/спек-валидация на выходе.
- **R3 (зависимость от A1/A3):** без A1 не рендерятся бланки/таблицы, без A3 нет golden-эталона КСУ. Митигация — контракт §2/§5 как требование; до готовности — SQL-агрегат + прямой XLSX/HTML как минимальный путь.
- **R4 (производительность годовых сводов):** агрегаты по всему фонду/году × раскодировки. Митигация — materialized views, async-очередь, инкрементальное обновление по `stat_event`.
- **R5 (полнота источников):** часть граф 6-НК (финансы, турникеты, мероприятия) вне библиографического ядра. Митигация — гибридные формы (авто-агрегат + ручной ввод/override с журналированием), явные `manual_fields` в `report_def`.
- **R6 (приватность статистики):** дневники/COUNTER по читателям — ПДн; SUSHI-сервер отдаёт наружу. Митигация — агрегация без PII, изоляция тенанта по ключу, согласование со слоем 152-ФЗ (**B4**, [DESIGN_COVERAGE §B4](../../DESIGN_COVERAGE.md)).

**Открытые вопросы (`TODO #GOV-NN` → к реализации):**
- `#GOV-01` — точная редакция/XSD формы **6-НК** действующего года (структура граф/строк, web-сбор Росстата) — внешний документ, уточнить при реализации; `report_def` сидируется per-год.
- `#GOV-02` — раздел 7 (финансы) 6-НК: источник чисел (финмодуль/Парус vs ручной ввод) и стык с ER #182.
- `#GOV-03` — формат обмена с **ИРБИС-Парус**/казначейством (структура выгрузки, маппинг счёта 88^J/^I) — recon не фиксирует.
- `#GOV-04` — COUNTER R5/SUSHI — новая (не-паритетная) возможность: приоритет поставщиков для harvest, объём собственного SUSHI-сервера.
- `#GOV-05` — методические нюансы КСУ-распределения (поля 44–49/145–158, [SPEC_ws2 §4](../SPEC_ws2_acquisition.md) `TODO #CMPL-05`) и сетка RTF-бланков КСУ (`#CMPL-07`) — сверить на боевых конфигах тенанта.
