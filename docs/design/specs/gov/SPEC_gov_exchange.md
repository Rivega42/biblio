**Назначение.** Единый, конфигурируемый per-tenant **exchange-pipeline** для выгрузки/обмена записями и сведениями с государственными системами РФ: **НЭБ** (выгрузка библиозаписей + объектов), **СМЭВ 3** (межведомственные сведения, SOAP/ЭП), **СКБР/ЛИБНЕТ** (заимствование записей + кооперативная выгрузка своих в сводный каталог). Один конвейер `extract → map → validate → sign → send → ack → report`, набор адаптеров-коннекторов per-источник и low-code маппинг-конфиг per-tenant.

## 0. Принципы и границы

- **Один конвейер, разные адаптеры.** Все направления обмена проходят общий pipeline (разд. 4); специфика источника изолирована в адаптере (профиль формата + маршрут + правила приёмки). Добавление источника = новый адаптер + конфиг, не переписывание ядра.
- **Декларативный маппинг.** Маппинг наших полей в целевую схему — **конфигурация** (FST/GBL-эквивалент, как `LIBNETFST`/`RUSMARCGBL` в [ARM_CATALOGER](../../../recon/deep/reference/arms/ARM_CATALOGER.md)), а не код. Канон-профиль (из recon) + per-tenant дельты ([SPEC_ws5](../SPEC_ws5_admin.md), low-code D1).
- **Тенант-изоляция.** Выгружаются записи каталога **тенанта**; сигла/идентификатор участника, креды источников, расписания — из конфигурации тенанта. Креды (`LIBNETUSER`/`LIBNETPASSWORD`, ЭП, токены СМЭВ) — в secrets-сторадж (envelope-шифрование, ADR-006 [§D2](../../DESIGN_COVERAGE.md)), **никогда** в маппинг-конфиге. (Recon-факт: в `irbisc.ini` креды лежали открытым текстом — **антипаттерн, не воспроизводим**.)
- **Идемпотентность и журнал.** Каждая отправка имеет `exchangeJobId` + `Idempotency-Key`; повтор не дублирует. Полный журнал отправок/приёмок (разд. 6) — источник истины «что, когда, кому отправлено и с каким результатом».
- **ЭП где требует регламент.** Подпись (ГОСТ Р 34.10-2012, отделённая/CAdES) — обязательна для СМЭВ-конвертов; для НЭБ/СКБР — по требованию площадки (флаг профиля). Реализация подписи — общий crypto-сервис (D2/I6), pipeline только вызывает `sign(payload, certRef)`.
- **Границы.** Эта спека **не** реализует MARC-конвертер, транспортные протоколы, ЭП-крипто, статотчётность и ЕСИА-вход — она их **оркеструет**. Точные wire-форматы НЭБ/СМЭВ (XSD конкретного вида сведений) — **внешние артефакты площадки**, грузятся как профиль-схема (разд. 9, открытые вопросы).

## 1. НЭБ — Национальная электронная библиотека (выгрузка записей + объектов)

> НЭБ агрегирует библиографические записи и **цифровые объекты** (полные тексты/образы) библиотек-участников. Точная схема приёмки задаётся регламентом НЭБ конкретной версии — здесь проектируется **адаптер**, который грузит схему как профиль и валидирует против неё.

### 1.1 Формат выгрузки

- **Метаданные:** профиль на базе **MARCXML (RUSMARC-семантика)** как канон-носитель (НЭБ исторически принимает RUSMARC/UNIMARC; ИРБИС уже умеет `rmarce_flc`/`umarcw.pft` экспорт — [CAPABILITY_MAP §11](../../../recon/deep/CAPABILITY_MAP.md), [SPEC_ws1 §5](../SPEC_ws1_cataloger.md)). Альтернативный носитель — **собственный XML НЭБ** или **OAI-PMH-фид** (см. ниже). Носитель — параметр профиля `neb.carrier ∈ {marcxml-rusmarc, oai-pmh, neb-xml}`.
- **Цифровой объект:** файл ПТ/образа (PDF/изображение) из хранилища тенанта (слот 11 PAR / поля 951/950/955 — [CAPABILITY_MAP §11](../../../recon/deep/CAPABILITY_MAP.md)) + манифест (контрольная сумма, MIME, права доступа/лицензия). Связь «запись↔объект» — через идентификатор записи (разд. 1.4) и поле URL объекта.
- **Транспорт:** предпочтительно **OAI-PMH-провайдер** (НЭБ как «харвестер» забирает у нас инкрементально) — переиспользуется OAI-сервер из `SPEC_protocols.md`; либо **batch-пакет** (ZIP: `records.xml` + объекты + `manifest.xml`) по согласованному каналу (SFTP/API площадки). Носитель объектов больших размеров — по ссылке (URL в записи), не инлайн.

### 1.2 Маппинг наших полей → НЭБ

Декларативный профиль `neb.map` (канон + tenant-дельты). Принцип — как `RUSMARCFST`+`RUSMARCGBL` ([ARM_CATALOGER](../../../recon/deep/reference/arms/ARM_CATALOGER.md)): FST-выборка поля → трансформ → целевой MARC-тег/подполе.

| Наше поле (ИРБИС) | Целевой элемент (RUSMARC/MARCXML) | Примечание |
|---|---|---|
| 200^a заглавие | 200 $a | обязательное (ФЛК-export) |
| 700/701 авторы | 700/701 | роль `^4` → relator |
| 10^a ISBN | 010 $a | контроль контр.суммы перед выгрузкой |
| 101 язык, 102 страна | 101/102 | словарные коды (`jz/str.mnu` тенанта) |
| 675/621 УДК/ББК | 675/686 | классификация |
| 951/950/955 ПТ/образ/URL | 856 $u (или нативный блок объекта НЭБ) | ссылка на цифровой объект |
| 903 шифр | local id / 090 | внутренний → маппится в наш идентификатор записи (разд. 1.4) |
| 902^s сигла держателя | holder/sigla НЭБ | для НЭБ обычно = сигла участника тенанта |

> Полный канон-маппинг машинно импортируется из export-FST/GBL ([SPEC_ws1 §5](../SPEC_ws1_cataloger.md), recon TODO GOV-01 — точная таблица RUSMARC↔НЭБ). Неизвестные расхождения профиля помечаются `blocked-on:neb-schema` и грузятся при получении актуальной XSD/регламента (разд. 9).

### 1.3 Валидация перед выгрузкой

Фаза `export` движка **A2** ([SPEC_engine_flk §2.1](../engines/SPEC_engine_flk.md)) + **схемная** валидация против XSD НЭБ:
1. **Бизнес-ФЛК (A2):** обязательность 200^a, корректность ISBN/ISSN (`unifor('C',…)`), словарные коды (101/102) — те же правила, что на сохранении, но в режиме `export` (severity 1 ⇒ запись **не выгружается**, попадает в карантин-отчёт).
2. **Схемная (XSD):** сгенерированный MARCXML валидируется против схемы НЭБ профиля; нарушение схемы ⇒ запись в отчёт-отбраковку с указанием XPath.
3. **Полнота объекта:** если запись заявляет ПТ/образ (951/950) — проверка доступности файла + лицензии (право на передачу в НЭБ — связь 152-ФЗ/авторское, флаг записи).

### 1.4 Идентификаторы

- **Стабильный исходящий ID записи** = `{tenantSigla}:{db}:{mfn}` или шифр 903 (политика профиля). Один и тот же ID при повторной выгрузке ⇒ НЭБ трактует как обновление, не дубль (инкрементальная синхронизация).
- НЭБ возвращает свой **приёмный ID** в ack (разд. 1.6); храним соответствие `localId ↔ nebId` в журнале (для последующих update/delete и для показа «есть в НЭБ» в карточке).

### 1.5 Периодичность

- **Batch (регламентная):** по расписанию (разд. 6) — полная или инкрементальная выгрузка (по метке изменения `W=`/дате актуализации).
- **Online/инкрементальная:** при OAI-PMH-носителе НЭБ харвестит по `from/until`; мы лишь поддерживаем фид (новые/изменённые с курсора). Триггерная выгрузка одной записи («отправить в НЭБ») — из карточки каталогизатора, опционально.

### 1.6 Отчёт о приёмке (ack)

После отправки пакета/харвеста НЭБ возвращает квитанцию: `{accepted, rejected[], nebIds{localId→nebId}, errors[]}`. Pipeline:
- пишет результат в журнал (разд. 6), помечает записи `nebId`/`rejected+reason`;
- формирует **отчёт оператору** (через A6): «выгружено N, принято M, отбраковано K (список с причинами)»;
- отбракованные ⇒ в очередь повторной выгрузки после исправления (ретрай-политика, разд. 6.3).

## 2. СКБР / ЛИБНЕТ — Сводный каталог библиотек России (заимствование + кооперативная выгрузка)

> Два направления: **(а) заимствование** (импорт чужих записей в наш каталог — копикаталогизация) и **(б) кооперативная каталогизация** (выгрузка наших записей в сводный каталог). Recon даёт реальную модель обоих: ЛИБНЕТ-импорт по HTTP ([ARM_CATALOGER](../../../recon/deep/reference/arms/ARM_CATALOGER.md)) и union-модель SK с `impsvk*` ([DB_CATALOG_VARIANTS §2](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)).

### 2.1 Заимствование (импорт из СКБР/ЛИБНЕТ) — адаптер `borrow`

Реальная recon-механика ([ARM_CATALOGER §[Main]](../../../recon/deep/reference/arms/ARM_CATALOGER.md)): ЛИБНЕТ-импорт — **HTTP-поиск** на портале + конверсия по FST/GBL:

| Параметр (recon `irbisc.ini`) | Роль в адаптере |
|---|---|
| `LIBNETIP` | URL поискового шлюза СКБР/ЛИБНЕТ |
| `LIBNETSEARCHPAGE` | страница/эндпоинт поиска |
| `LIBNETFORMAT` | формат отдаваемой записи (RUSMARC) |
| `LIBNETFST` | FST-маппинг входной записи → наши поля |
| `LIBNETGBL` | GBL пост-обработки импортированной записи |
| `LIBNETUSER`/`LIBNETPASSWORD` | креды (→ secrets-сторадж, не конфиг) |

**Поток заимствования** ([SPEC_ws1 §4](../SPEC_ws1_cataloger.md)): оператор ищет в источнике (СКБР по HTTP-шлюзу **или** Z39.50/SRU через `SPEC_protocols.md`; целевой URL/база — профиль) → выбирает запись → адаптер конвертит RUSMARC→наши поля по `LIBNETFST` + `LIBNETGBL` → запись попадает в РЛ → **ФЛК + 998-контроль дублей** ([SPEC_ws1 §4 AC2](../SPEC_ws1_cataloger.md)) → правка → сохранение. Метка происхождения импорта **998** ставится автоматически (для последующего ревью).

> Z39.50/SRU-вариант источника (`ZIMPORTSEARCHPAGE`/`ZIMPORTFORMAT` в recon) — тот же адаптер, транспорт делегируется в `SPEC_protocols.md`; различается только профиль (URL, набор атрибутов поиска, формат записи).

### 2.2 Кооперативная выгрузка (наши записи → СКБР) — адаптер `union-push`

Union-модель **точно описана в recon** ([DB_CATALOG_VARIANTS §2](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)) — воспроизводим её семантику в адаптере:

- **Модель держателя (поле 902):** `^a` наименование, `^b` адрес, `^d` телеком, **`^s` сигла** (идентификатор участника). Наша сигла берётся из настройки тенанта (`SKSG`-эквивалент) или из `902^s`.
- **Дедупликация при приёме сводным каталогом** (`impsvk*`, трёхъярусный ключ — [§2.2 recon](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)): (1) по **шифру** `I=`(v903)→`^ZI=`; (2) по **ISBN** `B=`(v10^a)→`^ZB=`; (3) по **свёртке БО** (композит библиоданных, своя на тип 920). Результат сверки → `907^Z` (непусто = дубль найден). **Эта логика — на стороне СКБР**, но мы её зеркалим локально, чтобы (а) предсказать, что запись будет распознана как дубль/новая, (б) корректно сформировать выгрузку с тегом источника.
- **Тег источника:** при слиянии в сводный каждое добавленное поле приоритетной сиглой получает подполе `^_<сигла>`; поставщик пишется в `907^B` ([§2.2 recon](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)). Адаптер `union-push` формирует выгрузку с нашей сиглой в 902^s и (опц.) предварительной разметкой источника.
- **Авторитетный контроль:** выгружаемые записи проходят авторитет-резолв `^3` ([SPEC_service_authority](../engines/SPEC_service_authority.md), A4) — заголовки приводятся к нормированной форме перед отправкой в сводный (иначе СКБР отбракует/задвоит). Это предусловие выгрузки (ФЛК-export правило `!700`/`!964`).

### 2.3 Дедуп и авторитетный контроль (сводная роль)

- **Локальный предиктивный дедуп** перед `union-push`: считаем те же три ключа (`I=`/`B=`/свёртка БО), что и `impsvk`, против **своего** инвертированного индекса (через A1 `dupKey`, как в [SPEC_engine_flk §1.2](../engines/SPEC_engine_flk.md)) — отсекаем заведомо дублирующие отправки.
- **Сверка авторитетов:** перед выгрузкой — резолв персоналий/предметных по A4; неразрешённые `^3` ⇒ severity 2 (выгрузить можно, но помечается «без авторитетной привязки»).

## 3. СМЭВ 3 — межведомственное электронное взаимодействие

> СМЭВ — транспорт обмена **сведениями** между госсистемами (не библиозаписями). Для библиотеки применимые сценарии узкие; проектируем адаптер `smev`, оставляя конкретные виды сведений конфигурируемыми (регламент вида сведений — внешний артефакт).

### 3.1 Применимые сценарии (если регламент площадки требует)

- **Выгрузка статистики/реестров** в ведомственную систему (Минкультуры/региональный орган) — например, агрегаты фонда/обслуживания (пересечение с **B1** статистикой; данные берутся из статдвижка, СМЭВ — лишь конверт-транспорт).
- **Сверка/запрос сведений** (реже): получение справочных данных из реестров. По умолчанию адаптер проектируется как **исходящий** (мы — поставщик сведений); входящие запросы — расширение.

> ⚠ Применимость СМЭВ для конкретного тенанта определяется его регламентом/соглашением. Адаптер — **опциональный** (энтайтлмент-модуль `gov-smev`); по умолчанию выключен.

### 3.2 Конверт и ЭП

- **SOAP/XML-конверт** по спецификации **СМЭВ 3** (`SMEV3` envelope: `SenderProvidedRequestData`/`MessagePrimaryContent`, маршрутные заголовки, `Ack`/`StatusResponse`). Бизнес-контент (`MessagePrimaryContent`) — XML по XSD конкретного вида сведений (грузится как профиль-схема).
- **Электронная подпись:** ЭП-ОВ (подпись органа власти, ГОСТ Р 34.10-2012) на контенте + транспортная подпись конверта (XMLDSig/CAdES) — **обязательна**. Реализация — общий crypto-сервис (D2/I6); адаптер вызывает `sign(content, orgCertRef)` и встраивает `<ds:Signature>`.
- **Асинхронная модель СМЭВ:** отправка `SendRequest` → получение `Ack` → поллинг `GetResponse`/`GetStatus` до готовности. Pipeline моделирует это `exchangeJobId` + статус-поллинг (как long-job в [SPEC_api_contracts §6.3](../engines/SPEC_api_contracts.md)).

### 3.3 Регламент

- Вид сведений (VS), мнемоника, namespace, частота, SLA ответа — **профиль СМЭВ тенанта**.
- Журнал СМЭВ (разд. 6) хранит `messageId`/`originalMessageId` (корреляция запрос↔ответ), статусы СМЭВ, подписанный конверт (для аудита/разбора).

## 4. Архитектура обмена — единый exchange-pipeline

### 4.1 Форма конвейера

```
                        ┌─────────────── Маппинг-конфиг тенанта (low-code) ───────────────┐
                        │     профиль источника: схема + map(FST/GBL) + маршрут + расписание + cert │
                        └───────────────────────────────────────────────────────────────┘
                                                   │ (читается каждой стадией)
 [selection]        [extract]        [map]            [validate]        [sign]        [send]         [ack]          [report]
  что выгружаем  →  тянем записи  →  наши поля →    A2 ФЛК(export) +  →  ЭП (опц/  →  адаптер-      →  квитанция  →  журнал +
  (query/курсор/    каталога        целевая схема   XSD целевой схемы    обяз. для   коннектор по    источника:     отчёт
   дельта по W=)    тенанта (A1)    (НЭБ/RUSMARC/    (карантин брака)     СМЭВ)       транспорту      accepted/      оператору
                                    СМЭВ-XML)                                         (OAI/SOAP/      rejected[]     (A6) +
                                                                                       HTTP/SFTP)                     ретраи
```

Общие стадии — ядро; **специфика источника инкапсулирована в адаптере** (реализует интерфейс `Connector`, разд. 4.3). Стадии `extract`/`map` переиспользуют A1 (PFT/FST) и MARC-движок; `validate` — A2; `sign` — crypto D2; `send`/`ack` — транспорт из `SPEC_protocols.md`.

### 4.2 Адаптеры-коннекторы (per-источник)

| Адаптер | Направление | Носитель/транспорт | Схема/маппинг | Дедуп/авторитет |
|---|---|---|---|---|
| `neb` | push (выгрузка) | OAI-PMH \| batch-ZIP \| API | MARCXML-RUSMARC \| neb-xml | — (НЭБ агрегирует) |
| `union-push` (СКБР) | push | API/HTTP-шлюз \| Z39.50 | RUSMARC + 902^s сигла | `impsvk`-ключ (предиктивно) + A4 |
| `borrow` (СКБР/ЛИБНЕТ) | pull (заимствование) | HTTP-шлюз \| Z39.50/SRU | RUSMARC→наши (LIBNETFST/GBL) | 998-контроль (A2) |
| `smev` | push (\| pull) | SOAP/СМЭВ3 + ЭП | СМЭВ-VS-XSD | — |

Все push-адаптеры реализуют единый интерфейс; различаются только профилем (схема, маршрут, дедуп-политика, требование ЭП).

### 4.3 Интерфейс коннектора (внутренний модульный контракт)

```
Connector {
  prepare(selection, profile) -> Batch         // extract+map: записи → целевые элементы (через A1/MARC)
  validate(batch, profile)    -> {ok[], rejected[]}   // A2 export + XSD
  sign?(batch, certRef)       -> SignedBatch    // опц.; обяз. для smev
  send(batch, route)          -> SendResult{jobId, transportRef}
  poll(jobId)                 -> Ack{accepted, rejected[], remoteIds, status}
}
```

Pipeline-оркестратор детерминированно прогоняет стадии, пишет журнал на каждой границе, применяет ретраи (разд. 6.3). Коннектор — чистая функция профиля; состояние (курсоры, маппинг localId↔remoteId) — в журнале.

### 4.4 Маппинг-конфиг (low-code, per-tenant)

- **Канон-профиль** (из recon: export-FST/GBL, `LIBNET*`, 902-модель) — read-only, версионируется в коде; **tenant-дельты** (сигла, креды-ref, расписание, вкл/выкл правил, кастом-маппинг полей) накладываются слоем (как override в [SPEC_engine_flk §3](../engines/SPEC_engine_flk.md)).
- Редактор — в Администраторе ([SPEC_ws5](../SPEC_ws5_admin.md), D1 low-code): таблица соответствий «наше поле → целевой элемент», тест-прогон на образце записи (предпросмотр сгенерированного MARCXML/конверта), привязка сертификата ЭП, настройка расписания.
- Версионирование профиля (`exchangeProfileVersion`) — каждая отправка помечается версией маппинга (воспроизводимость: «чем именно выгрузили»).

### 4.5 Мониторинг

- Дашборд обмена (ws5): по каждому источнику — последняя успешная выгрузка, очередь, % приёмки, отбраковки, ошибки транспорта/ЭП.
- Метрики/трейсы (наблюдаемость I1): `exchange_jobs_total{source,status}`, `exchange_records{source,accepted|rejected}`, латентность стадий, ошибки подписи. Алерты на провал регламентной выгрузки (через A6).

## 5. Маппинг и валидация (детализация)

### 5.1 Движок маппинга

- Реализация маппинга = MARC-конвертер ([SPEC_ws1 §5](../SPEC_ws1_cataloger.md)) + FST/GBL-эквивалент через A1 ([SPEC_engine_pft](../engines/SPEC_engine_pft.md)): `extract` = FST-выборка полей, `map` = трансформ (GBL-эквивалент) в целевые теги. **Не пишем новый конвертер** — конфигурируем существующий профилем источника.
- Профиль хранит: `carrier` (носитель), `fieldMap` (наше↔целевое), `pre`/`post`-GBL (нормализация), `schemaRef` (XSD), `route`, `signPolicy`, `dedupKey`.

### 5.2 Обратный маппинг (заимствование)

`borrow`-адаптер использует **входной** профиль (`LIBNETFST`+`LIBNETGBL`-эквивалент): RUSMARC→наши поля, затем 998-метка + ФЛК импорта ([SPEC_engine_flk §4.1](../engines/SPEC_engine_flk.md), фаза `import`). Круговой тест (импорт→экспорт без потерь) — общий с #123 ([SPEC_I2 §C AC1](../SPEC_I2_catalog_search.md)).

### 5.3 Фаза `export` в ФЛК (A2)

Новая фаза движка A2 (дополнение к save/fieldExit/import/delete — [SPEC_engine_flk §2.1](../engines/SPEC_engine_flk.md)): прогон ruleset в режиме «готовность к выгрузке» (обязательность целевых полей источника, нормированность авторитетов, валидность объектных ссылок). severity 1 ⇒ запись в карантин (не выгружается); severity 2 ⇒ выгружается с пометкой в отчёте. Плюс **схемная** валидация (XSD целевой схемы) — отдельный валидатор, не ФЛК.

## 6. Регламентные выгрузки — расписание, журнал, ретраи, ЭП

### 6.1 Расписание

- Per-источник cron-расписание в профиле (напр., НЭБ — еженедельная инкрементальная; СКБР — по событию/ночная; СМЭВ — по регламенту VS). Исполнитель — планировщик (A6/инфраструктура задач).
- Режимы: **full** (вся БД) / **incremental** (по курсору `W=`/дате актуализации) / **on-demand** (ручной запуск оператором / по событию сохранения записи).

### 6.2 Журнал отправок/приёмок

Таблица `exchange_log` (per-tenant, RLS) — источник истины:

| Поле | Назначение |
|---|---|
| `exchangeJobId` | идентификатор задания (идемпотентность) |
| `source` | `neb`/`union-push`/`borrow`/`smev` |
| `profileVersion` | версия маппинга |
| `mode` | full/incremental/on-demand |
| `recordsSent` / `accepted` / `rejected` | счётчики приёмки |
| `idMap` | `localId ↔ remoteId` (nebId/СКБР-id/messageId) |
| `transportRef` | ref квитанции/SOAP messageId/OAI-resumptionToken |
| `signatureRef` | ссылка на подписанный конверт (для СМЭВ — обяз.) |
| `status` | queued/running/sent/acked/partial/failed |
| `rejections[]` | список отбракованных + причина (XPath/код) |
| `actor` / `startedAt` / `finishedAt` | аудит |

Журнал доступен в ws5 (история обмена) и питает отчёт оператору. Связь с журналом доступа/аудита (I6 B4) для записей, содержащих ПДн.

### 6.3 Ретраи

- Транспортные сбои (сеть/таймаут) — экспоненциальный ретрай с лимитом; идемпотентность по `exchangeJobId` исключает дубль на стороне площадки.
- Бизнес-отбраковки (схема/ФЛК) — **не** ретраятся автоматически: запись → очередь «требует исправления», повторная выгрузка после правки (ручной/следующий регламентный цикл).
- Частичная приёмка (`partial`) — принятые фиксируются, отбракованные остаются в очереди.

### 6.4 ЭП в регламенте

- Профиль источника несёт `signPolicy ∈ {none, optional, required}`. `required` (СМЭВ всегда; НЭБ/СКБР — по площадке) ⇒ pipeline не отправит неподписанный пакет (стадия `sign` обязательна, иначе job→`failed` с причиной).
- Сертификат/ключ — по `certRef` из secrets-стораджа; ротация/отзыв — D2/PKI (I6).

## 8. Критерии приёмки (AC)

- **AC1 (НЭБ — выгрузка проходит валидацию).** Корректная запись с ПТ выгружается в профиле НЭБ; проходит ФЛК-export + XSD; квитанция возвращает `nebId`, соответствие `localId↔nebId` в журнале. Запись с битым ISBN / без 200^a — в карантин-отчёт, **не** выгружается. *([SPEC_I2 §C #124 AC2](../SPEC_I2_catalog_search.md))*
- **AC2 (НЭБ — инкрементальность).** Повторная выгрузка той же записи с тем же исходящим ID трактуется как обновление (не дубль); инкрементальный режим выгружает только изменённые с курсора.
- **AC3 (СКБР — кооперативная выгрузка).** Наша запись выгружается в сводный с сиглой в 902^s; локальный предиктивный дедуп (`I=`/`B=`/свёртка БО) совпадает с `impsvk`-ключом; неразрешённый авторитет `^3` даёт severity 2 с пометкой.
- **AC4 (СКБР/ЛИБНЕТ — заимствование).** Поиск в источнике (HTTP-шлюз или Z39.50 через `SPEC_protocols.md`) → импорт записи в РЛ по `LIBNETFST/GBL`-маппингу → 998-метка → ФЛК предупреждает о дубле. Круговой импорт→экспорт без потери полей. *([SPEC_ws1 §4 AC1/AC2](../SPEC_ws1_cataloger.md))*
- **AC5 (СМЭВ — конверт + ЭП).** Исходящее сведение упаковано в СМЭВ3-конверт, подписано ЭП-ОВ (ГОСТ Р 34.10-2012); без валидной подписи отправка не выполняется (`signPolicy:required` ⇒ job `failed`). Корреляция запрос↔ответ по `messageId` в журнале.
- **AC6 (журнал/приёмка).** Каждое задание пишет в `exchange_log` счётчики/idMap/статус/подпись; отчёт `accepted/rejected[]` доступен оператору; отбракованные ставятся в очередь повтора.
- **AC7 (ретраи/идемпотентность).** Транспортный сбой ретраится с тем же `exchangeJobId` без дубля на площадке; бизнес-отбраковка не ретраится автоматически; частичная приёмка фиксирует принятые.
- **AC8 (маппинг-конфиг per-tenant).** Профиль источника редактируется в ws5 (low-code), версионируется, тест-прогон показывает сгенерированный MARCXML/конверт; смена маппинга помечает последующие задания новой `profileVersion`.
- **AC9 (тенант-изоляция/секреты).** Выгружаются только записи тенанта; сигла/креды/сертификат — из конфигурации/secrets тенанта; cross-tenant — негатив-тест; креды никогда не в маппинг-конфиге и не в логах.
- **AC10 (энтайтлмент).** При выключенном `gov-exchange`/`gov-smev` эндпоинты дают 404/403 даже при наличии гранта ([SPEC_api_contracts §0](../engines/SPEC_api_contracts.md)).

## 10. Критический путь

1. **Интерфейс `Connector` + pipeline-оркестратор (4.1–4.3)** + журнал `exchange_log` (6.2) — ядро.
2. **Маппинг-движок поверх MARC/A1 (5.1)** + профиль-схема (`ExchangeProfile`) + фаза ФЛК `export` (5.3).
3. **Адаптер `borrow` (СКБР/ЛИБНЕТ, 2.1)** — самый «грунтованный» recon (есть `LIBNET*`), быстрый паритет копикаталогизации; переиспользует транспорт из `SPEC_protocols.md`.
4. **Адаптер `neb` (1.x)** — выгрузка записей+объектов (OAI-провайдер/batch), ack-отчёт.
5. **Адаптер `union-push` (СКБР, 2.2–2.3)** — кооперативная выгрузка с 902-моделью и предиктивным дедупом.
6. **Адаптер `smev` (3.x)** — SOAP3+ЭП (зависит от crypto D2 и регламента VS); опциональный.
7. **Расписание/ретраи (6.1/6.3) + дашборд/алерты (4.5)** + low-code редактор профиля (ws5).

**Зависимости:** `SPEC_protocols.md` (B2 — транспорт Z39.50/SRU/OAI/SOAP) — **блокер** для borrow/neb/smev отправки; MARC-движок ([SPEC_ws1 §5](../SPEC_ws1_cataloger.md)); A2 ФЛК (фаза export); A4 авторитеты (нормализация перед union-push); D2/PKI (ЭП для СМЭВ); A6 (отчёты/расписание); B1 статистика (источник данных для СМЭВ-статсведений).

**Выходной критерий B3:** выгрузка в НЭБ проходит валидацию и возвращает приёмку; заимствование из СКБР/ЛИБНЕТ импортирует записи в РЛ с 998-контролем; кооперативная выгрузка в сводный использует 902-сиглу и `impsvk`-совместимый дедуп; СМЭВ-конверты подписываются ЭП; всё проходит единый pipeline с журналом, ретраями и per-tenant low-code маппингом.

## 11. Риски и открытые вопросы

- **R1 (точные wire-форматы НЭБ/СМЭВ неизвестны из recon).** ИРБИС-конфиги дают ЛИБНЕТ/Z39.50/RUSMARC, но **не** актуальную XSD приёмки НЭБ и **не** регламент конкретного вида сведений СМЭВ — это внешние артефакты площадки, меняющиеся по версиям. **Митигация:** схема целевого формата грузится как **профиль-артефакт** (`schemaRef`), маппинг — декларативный; неизвестные элементы помечаются `blocked-on:neb-schema`/`blocked-on:smev-vs` и не теряются. Точная таблица RUSMARC↔НЭБ — `TODO(recon GOV-01)`.
- **R2 (зависимость от `SPEC_protocols.md`).** Транспорт (Z39.50/SRU/OAI/SOAP-клиент, серверный OAI-провайдер) живёт в соседней спеке; рассинхрон интерфейса ломает send/borrow. **Митигация:** фиксированный интерфейс `Connector.send/poll` поверх транспорта; контракт-тест на стыке.
- **R3 (ЭП/крипто ГОСТ).** СМЭВ требует ГОСТ-подписи (CryptoPro-стек), что выходит за обычный TLS. **Митигация:** изолировать в crypto-сервис D2; адаптер вызывает абстрактный `sign()`; без D2 — `smev` остаётся `blocked-on:D2`.
- **R4 (дедуп СКБР — логика на стороне площадки).** Реальный `impsvk`-дедуп исполняет сводный каталог; наше зеркало предиктивно и может расходиться с его версией ключа свёртки БО. **Митигация:** зеркалим документированный recon-ключ ([§2.2](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)), помечаем «предиктивно»; финальный вердикт — из ack площадки.
- **R5 (применимость СМЭВ варьируется по тенанту).** Не у всех библиотек есть соглашение/вид сведений СМЭВ. **Митигация:** `smev` — опциональный энтайтлмент-модуль, выключен по умолчанию.
- **R6 (права на передачу объектов в НЭБ).** Выгрузка ПТ/образов затрагивает авторское право/152-ФЗ. **Митигация:** флаг лицензии/права передачи на записи; ФЛК-export блокирует объект без права (связь B4/I6).

## Сводка

**Форма exchange-pipeline:** единый конвейер `selection → extract → map → validate(A2 export + XSD) → sign(ЭП, опц/обяз для СМЭВ) → send → ack → report`, с инкапсуляцией специфики источника в **адаптерах-коннекторах** (`neb` push, `union-push`/СКБР push, `borrow`/ЛИБНЕТ pull, `smev` SOAP) поверх общего транспорта (`SPEC_protocols.md`), MARC-движка и A1/A2; маппинг — декларативный low-code профиль per-tenant (FST/GBL-эквивалент канона из recon `LIBNET*`/902-модели), версионируемый; журнал `exchange_log` + ретраи + отчёт о приёмке. Грунтовано на реальной recon-механике: ЛИБНЕТ HTTP-импорт ([ARM_CATALOGER](../../../recon/deep/reference/arms/ARM_CATALOGER.md)) и union-модель SK с `impsvk`-дедупом и сиглой 902^s ([DB_CATALOG_VARIANTS §2](../../../recon/deep/reference/databases/DB_CATALOG_VARIANTS.md)).
