## Эпик #112 — Мультиарендность и лицензирование
**Цель:** общий фундамент SaaS — изоляция тенантов, идентификация, гранты, лицензии/модули. Снимает «не постгрес» и открывает все остальные инициативы.

### Таск #100 — Модель тенанта + изоляция (схема-на-тенанта / RLS) · L
**Цель:** надёжная изоляция данных библиотек.
**Подход:**
- **Схема-на-тенанта** в PostgreSQL: общий контрольный слой `control` (таблица `tenant`, `tenant_module`, `account` глобально или per-tenant — см. #101), и по схеме `t_<slug>` на каждого тенанта (catalog/holdings/loan…). Edge — БД-на-тенанта (та же DDL).
- **Резолвинг тенанта:** из host/subdomain ИЛИ JWT-claim `tenant_id`; middleware шлюза кладёт tenant в контекст запроса; `SET search_path = t_<slug>` на соединении.
- **Пул:** PgBouncer; tenant-aware выдача соединений (search_path на чек-аут), без утечки между тенантами.
- **Миграции схемы:** версионируемые, прогоняются по каждой схеме; оркестратор «apply to all tenants», идемпотентно.
- **RLS** для любых общих таблиц (где tenant_id-колонка вместо отдельной схемы).
**Acceptance criteria:**
- AC1: запрос, аутентифицированный как тенант A, **никогда** не возвращает строки тенанта B (негативный тест на пересечение).
- AC2: провижининг новой схемы тенанта идемпотентен (повторный прогон безопасен), все миграции применены.
- AC3: соединение из пула, переиспользованное между тенантами, не «протекает» (search_path/контекст сбрасывается).
- AC4: **автоматический тест изоляции** в CI; сборка падает при любой возможности кросс-тенантного доступа.
**Зависимости:** PostgreSQL (#99). **DoD:** + тест изоляции в CI обязателен.

### Таск #101 — Сервис Identity / Tenancy / Licensing · L
**Цель:** единый источник «кто/в каком тенанте/что можно».
**Подход:**
- Сущности: `tenant`, `account`, `role`, `grant` (функция×база×уровень), `tenant_module` (entitlement), `audit`. Перенос текущего Access-store (#92) как основы.
- **JWT** с claims `sub, tenant_id, grants|roles, kind(guest/reader/staff)`; срок/refresh; подпись секретом из env.
- **Entitlements-API**: `is_enabled(tenant, module)` + список включённых модулей — **отдельно от биллинга** (биллинг в I7 потребляет).
- Аудит write/admin действий (actor, tenant, функция, объект, результат).
**Acceptance criteria:**
- AC1: токен несёт `tenant_id`+гранты; шлюз авторизует по грантам (точная база > `*`, ранги read/write/admin), отказ → 403 с аудитом.
- AC2: entitlements отражают `tenant_module`; выключенный модуль недоступен (404/403) даже при наличии гранта.
- AC3: контракт эндпоинтов auth/tenancy зафиксирован в openapi.yaml (contract-test зелёный).
**Зависимости:** #100, #92. **DoD:** + контракт-тест.

### Таск #102 — Онбординг тенанта · M
**Цель:** «новая библиотека за минуты».
**Подход:** мастер: создать `tenant` → провижинить схему → прогнать миграции → seed (admin-аккаунт, дефолт-роли, **пресет модулей по типу** публичная/школьная/вузовская/ведомственная, бренд/тема по умолчанию из дизайн-системы) → готово к входу.
**Acceptance criteria:**
- AC1: из пустого состояния создаётся рабочий тенант; admin может войти; включён корректный пресет модулей по типу.
- AC2: бренд/тема тенанта применяются (связка с дизайн-системой #154); откат/удаление тенанта чистит схему.
**Зависимости:** #100, #101, дизайн-система (#149/#151). **DoD:** e2e сценарий онбординга.

## Эпик #99 — Прод-инфраструктура и PostgreSQL
**Цель:** перевести ядро на PostgreSQL и упаковать для прода.

### Таск #92 — Access-store sqlite → PostgreSQL · M
**Подход:** `PgAccessStore` (psycopg3) по `schema_postgres.sql`; зеркалит API текущего `AccessStore`; переключение `ACCESS_BACKEND=postgres`; перенос seed; те же 17 unit-тестов гоняются на PG (testcontainers/локальный PG).
**AC:** AC1 — весь набор тестов Access зелёный на PG; AC2 — паритет поведения с sqlite (authz/audit идентичны); AC3 — миграция dev-данных без потерь. **DoD:** тесты на обоих бэкендах в CI.

### Таск — Каталог-ядро как сервис на PostgreSQL · L
**Подход:** вынести из `server-own/` PoC в сервис: записи JSONB поля/подполя + материализованные колонки + `tsvector`; holdings; location-дерево. **Bulk-загрузка по выводам бенчмарка**: параллельные/конвейерные чтения из ИРБИС, батч-вставки, **отложенная FTS-индексация** (создавать GIN после bulk). 
**AC:** AC1 — тот же API-контракт (search/record/storage), фронт работает без изменений; AC2 — миграция IBIS укладывается в проекцию (≈50 зап/с оптимизированно); AC3 — FTS/поиск идентичны PoC. **DoD:** + контракт-тест, + e2e фронта.

### Таск — Прод-упаковка · M
**Подход:** многоступенчатый Dockerfile (node-сборка фронта + python-рантайм), `docker-compose.prod.yml` (app + PostgreSQL + PgBouncer + объектное хранилище MinIO), конфиг через env/секреты (нет секретов в образе), `/healthz`+`/readyz`.
**AC:** AC1 — `docker compose up` поднимает стек, readiness зелёный; AC2 — секреты только через env; AC3 — бэкап/restore PG задокументированы. **DoD:** smoke-деплой.

### Таск #117 — Наблюдаемость (OTel + trace/tenant-id) · M
**Подход:** OpenTelemetry SDK (Python) во всех сервисах; middleware шлюза генерирует/пробрасывает `trace_id`, кладёт `tenant_id` в span/лог; структурные логи (JSON) с `trace_id`+`tenant_id`; метрики (RED) ключевых потоков; экспорт OTLP (коллектор позже).
**AC:** AC1 — каждый запрос имеет сквозной `trace_id` и `tenant_id` в логах/спанах через все хопы; AC2 — дашборд латентности/ошибок поиска и выдачи; AC3 — корреляция «лог↔трейс» работает. **DoD:** включено в шаблон сервиса с первого дня.

### Таск #118 — CI/CD (trunk-based, тесты, preview) · M
**Подход:** GitHub Actions: lint (ruff/eslint) → unit → **openapi-lint + Pact contract-test** → **migration-test** (прогон миграций на чистой PG) → **tenant-isolation-test** → build (подписанный образ — связка с #119) → (опц.) ephemeral preview на PR. Feature-flags (library) для раскатки. Trunk-based, защита main.
**AC:** AC1 — PR не мёржится без зелёных contract/migration/isolation тестов; AC2 — preview-URL на PR (или обоснованный пропуск); AC3 — флаги управляют включением фич без релиза. **DoD:** пайплайн зелёный на репозитории.

## Порядок внутри I1 (критический путь)
1. **#99/#92** PostgreSQL + Access на PG (фундамент данных) →
2. **#100** изоляция тенантов (схема-на-тенанта, пул, миграции) →
3. **#101** Identity/Tenancy/Licensing (JWT+гранты+entitlements) →
4. **#117** наблюдаемость и **#118** CI/CD — параллельно с 1–3 (закладываем с первого сервиса) →
5. **Каталог-ядро на PG** (bulk по бенчмарку) + **#102** онбординг (нужна дизайн-система #149/#151).

**Выходной критерий I1:** мультиарендный стек на PostgreSQL, новый тенант онбордится за минуты с изоляцией под тестом, единый контракт + наблюдаемость + CI-гейты; миграция IBIS укладывается в проекцию бенчмарка.
