# SPEC · I5 — Паритет АРМов сотрудника (наполненность замены ИРБИС)

> Цель: **функционально заменить настольные АРМы ИРБИС** веб-рабочими местами. Разбор по-АРМно (что делает АРМ → что нужно у нас → критерии приёмки), грунтованный на нашем recon: [ARM_CATALOGER](../../recon/deep/reference/arms/ARM_CATALOGER.md), [ARM_ACQUISITION](../../recon/deep/reference/arms/ARM_ACQUISITION.md), [ARM_LENDING](../../recon/deep/reference/arms/ARM_LENDING.md), [ARM_BOOKPROVISION](../../recon/deep/reference/arms/ARM_BOOKPROVISION.md), [ARM_ADMIN](../../recon/deep/reference/arms/ARM_ADMIN.md) + [DB-референсы](../../recon/deep/reference/databases/). Текущее покрытие — [COVERAGE_MATRIX](../COVERAGE_MATRIX.md). Эпик **#98** (→ I5) + связи с #95 (книговыдача-движок), **ER #182** (комплектование/КО-движок), #123 (стандарты), I1 (админ=tenancy/гранты).

**Принцип «наполненности»:** АРМ заменён, когда сотрудник выполняет **все ежедневные сценарии** этого АРМа в вебе — не только «создать запись», но валидацию, заимствование, инвентаризацию, печать, пакетные операции, отчёты. DoD-гейт как в I1/I2 (код+тест+контракт+a11y+аудит).

---
## АРМ «Каталогизатор» → задача **ws1** (покрытие 20% → паритет) · L
Назначение: создание/редактирование библиографических записей (БД IBIS/ЭК). Из [ARM_CATALOGER]: ввод по рабочим листам, рубрикатор ГРНТИ, инвентаризация, ИРБИС-Навигатор, полные тексты, словарный поиск, пользовательские режимы (DLL).
**Что нужно у нас:**
- **Рабочие листы ввода** (.ws/.wss → `DynamicField`, частично есть) — все типы РЛ по полю 920 (PAZK/SPEC/NJ/...), подполя, повторяющиеся группы.
- **ФЛК-валидация** (наш [FLC_VALIDATION]: `dbnflc.pft` + 35 `!NNN.pft`): обязательность, дублетность инв.№/шифра, контроль ISBN/ISSN, справочные проверки — **на сохранении**.
- **Рубрикатор/классификаторы**: ГРНТИ + УДК/ББК (дерево, выбор, автоввод).
- **Авторитетный контроль** (ATHR*: подполе `^3`, autoin генерация связанных) — ввод через авторитетные файлы.
- **Заимствование/копикаталогизация** по **Z39.50/СКБР/WorldCat** (связь #123, **CN #170/#169**).
- **Импорт/экспорт** ISO2709/MARC21/RUSMARC/MARCXML (#123).
- **Инвентаризация** (экземпляры — поле 910), штрихкоды/RFID-метки.
- **Полные тексты** (951/953/955 — связь **DG #159/#180**).
- **Глобальная корректировка** (наш [GLOBAL_CORRECTION]: 22 оператора, `.gbl`) — пакетные правки **с превью и откатом**.
- **Печать**: каталожные карточки, списки (PFT/табличные формы из [PFT_MODULES]).
- **ИИ-автокаталогизация** (PME #158) — черновик метаданных/классификации.
**AC:** AC1 создание/правка записи с прохождением ФЛК (ошибки показаны по полям); AC2 заимствование записи по Z39.50; AC3 импорт→экспорт MARC без потерь; AC4 глоб.корректировка по выборке с превью/откатом; AC5 инвентаризация экземпляра (910) + привязка к ячейке (#95); AC6 печать каталожной карточки.

## АРМ «Комплектатор» → задача **ws2** + движок **ER #182** (покрытие 5% → паритет) · L
Назначение: комплектование фонда. Из [ARM_ACQUISITION] + [DB_ACQUISITION] (CMPL/PODB/POST): ввод+КСУ, поступление/подписка по издательским каталогам, сценарии КСУ и списания, **комплексный поиск по БД комплектования (SEARCHCMP)**.
**Что нужно у нас:**
- **Цикл**: заявка → заказ → поступление → **КСУ (книга суммарного учёта)** → списание → перенос в ЭК.
- **Подписка** по издательским каталогам (POST), периодика (связь **ER #182/er2**).
- **КСУ-бланки/отчёты** (RTF-формы из recon), акты списания.
- **Комплексный поиск** по статусам заказа/поступления (SEARCHCMP — ключевая фишка АРМа).
- **Бюджет/финансы** заказа; распределение по экземплярам/фондам.
- **Demand-driven** дозаказ по отказам/спросу (PMD).
**AC:** AC1 полный цикл заказ→поступление→КСУ→списание с проводками; AC2 КСУ-отчёт формируется; AC3 комплексный поиск по статусам; AC4 перенос поступившего в ЭК (связь с записью).

## АРМ «Книговыдача» → задача **ws3** + движок **#95** (покрытие 10% → паритет) · L
Назначение: обслуживание читателей. Из [ARM_LENDING] + [DB_CIRCULATION] (RDR/RQST): работа с читателем (`[Reader]`), заказы (`[Request]`).
**Что нужно у нас:**
- **Регистрация/редактирование читателя** (RDR) + читательский билет (+ ЕСИА, I2 #124).
- **Формуляр** (книги на руках, история, штрафы).
- **Заказ/бронь** (RQST), очередь, **бронеполка**.
- **Выдача / возврат / продление**, смена статуса экземпляра (ste.mnu/reservstatus.mnu из recon).
- **Штрафы/оплаты** (связь PAY).
- **Наш +**: выдача/возврат через **ячейки/постамат 24/7** (#95, #145), self-checkout/RFID, holds-aware локеры с авто-checkout.
**AC:** AC1 регистрация читателя; AC2 выдача меняет статус «на руках», возврат — обратно; AC3 формуляр актуален; AC4 очередь брони; AC5 выдача через ячейку/постамат с кодом (наш differentiator); AC6 офлайн-выдача на edge (I4).

## АРМ «Книгообеспеченность» → задача **ws4** + **ER #182/er3** (покрытие 3% → паритет) · M
Назначение: ККО для вузов. Из [ARM_BOOKPROVISION] + [DB_VUZ]: ввод, **отчёты/таблицы (ядро КО)**, **поиск/связывание по книгообеспеченности (SEARCHKO, ~2150 строк)**, поле 693.
**Что нужно у нас:**
- **Справочники**: специальности/дисциплины/контингент.
- **Связывание литература ↔ дисциплина** (693, SEARCHKO).
- **Расчёт коэффициента книгообеспеченности (ККО)**.
- **Отчёты/таблицы** КО (декларативные формы).
- Связь с **курсовыми резервами** (ER er3) и комплектованием (дозаказ под обеспеченность).
**AC:** AC1 связывание дисциплина↔литература; AC2 расчёт ККО по дисциплине/специальности; AC3 отчёт КО формируется; AC4 сигнал на дозаказ при низком ККО.

## АРМ «Администратор» → задача **ws5** + **I1** (покрытие 25% → паритет) · M
Назначение: администрирование системы. Из [ARM_ADMIN]: глобальные параметры, обслуживание БД и полных текстов.
**Что нужно у нас:**
- **Управление БД/тенантами**: создание, реорганизация/актуализация, импорт/экспорт, бэкап/restore (связь I1 онбординг #102, прод-инфра #99).
- **Пользователи/роли/гранты** (наш Access — I1 #101) + аудит.
- **Справочники/словари** (MNU/TRE — наш [CONTROL_VOCABULARIES], 940 пар) — редактор справочников.
- **Параметры баз** (PAR/INI-аналог) — конфиг баз/полей/РЛ per-tenant (low-code, PMF).
- **Обслуживание полных текстов**, переиндексация (без простоя).
- **Мониторинг/логи** (OTel-дашборды, I1 #117), статистика (КСУ/6-НК).
**AC:** AC1 админ создаёт/обслуживает БД тенанта; AC2 управляет пользователями/грантами; AC3 правит справочники; AC4 параметры баз/РЛ настраиваются без кода; AC5 видит логи/мониторинг/статистику.

## Сводка: что даёт «наполненность»
| АРМ | Задача | Движок/связи | Текущее → цель |
|---|---|---|---|
| Каталогизатор | ws1 | #123 стандарты, CN заимствование, DG тексты, PME ИИ-каталог | 20% → паритет+ |
| Комплектатор | ws2 | ER #182 | 5% → паритет |
| Книговыдача | ws3 | #95/#145, I4 офлайн | 10% → паритет+ (ячейки/постамат) |
| Книгообеспеченность | ws4 | ER er3, VUZ | 3% → паритет |
| Администратор | ws5 | I1 (#101/#102/#117), PMF low-code | 25% → паритет |

## Критический путь I5
1. **ws1 Каталогизатор** (ядро ежедневной работы) + **ws5 Администратор** (на базе I1) →
2. **ws3 Книговыдача** (поверх движка #95, differentiator ячейки/постамат) →
3. **ws2 Комплектатор** (поверх ER #182) →
4. **ws4 Книгообеспеченность** (вузовский сегмент).

**Выходной критерий I5:** сотрудник выполняет полный рабочий день без десктоп-АРМа ИРБИС — каталогизация с ФЛК/заимствованием/печатью, комплектование с КСУ, книговыдача (вкл. ячейки/постамат, офлайн), книгообеспеченность, администрирование — в вебе, по грантам, на дизайн-системе.
