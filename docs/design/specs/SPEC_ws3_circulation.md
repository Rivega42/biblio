# SPEC · ws3 — АРМ Книговыдача (полевой уровень)

> Детализация задачи **ws3** ([#185](https://github.com/Rivega42/biblio/issues/185), эпик #98) до уровня полей/БД/цикла — веб-замена десктоп-Книговыдачи + наш differentiator (ячейки/постамат/офлайн). Грунтовано на recon: [DB_CIRCULATION](../../recon/deep/reference/databases/DB_CIRCULATION.md) (RDR/RQST/RDR_ARH/RDRV, цикл из конфигов), [CONTROL_VOCABULARIES](../../recon/deep/reference/format/CONTROL_VOCABULARIES.md). Родитель — [SPEC_I5](SPEC_I5_staff_arm_parity.md); движок — [#95](https://github.com/Rivega42/biblio/issues/95); офлайн — [SYNC_DESIGN](../SYNC_DESIGN.md). ⚠ **ПДн читателей (152-ФЗ): минимизация, opt-in история, журнал доступа.**

## 1. Доменная модель (из 4 БД ИРБИС → наши сущности)
| ИРБИС | Назначение | У нас |
|---|---|---|
| **RDR** | картотека читателей (поле 40 = на руках) | `reader` + `loan` (активные) |
| **RQST** | заказы/брони/отказы (транзит) | `request`/`hold` (события) |
| **RDR_ARH** | архив завершённых выдач | `loan` (history, opt-in) |
| **RDRV** | временные читатели (+поле 31 регистратор) | `reader(kind=temp)` |
**Принцип:** оперативные операции — **event-sourced, edge-master** (работают офлайн, ADR-005); архив выдач — по согласию читателя (приватность ALA/152-ФЗ).

## 2. Регистрация читателя (RDR, 58 полей)
**Подход:** РЛ по 920/`VDR.mnu` (`RDR` читатель · `RDRU` студент-уч.план · `ST` студент-дисциплины · `VUZ`). Идентификатор — **поле 30** (`RI=` = № билета/штрих-код). Контролы по типу редактора (0 текст·1 меню·2 словарь·5 .wss·6 дерево·7 да/нет).
**Ключевые поля:** 10/11/12 ФИО · 13 адрес · 21 год рожд (`@!21`) · 23 пол (`pol.mnu`,`@!23`) · 24 № билета (`@!24`) · 29 право пользования (`29.mnu`) · 30 идентиф (`@!30`) · 50 категория (`50.mnu`: В01–В05/Д01–Д03/STD/GUEST) · 56/57 разрешённые/запрещённые места выдачи (`kv.mnu`) · 90 учебный профиль студента (факультет/курс/группа/спец, деревья) · 950 фото · 32 e-mail/17/18 телефоны.
**Наш +:** вход/привязка **ЕСИА** (I2 #124), единый билет; самозапись.
**AC:** AC1 регистрация/правка читателя по РЛ типа (920), идентификатор `RI=` уникален; AC2 категория/право/места выдачи из словарей; AC3 студенческий профиль (90) с деревьями фак/спец; AC4 печать **читательского билета** (`CHB.pft`-аналог) + фото; AC5 ПДн под минимизацией/журналом доступа.

## 3. Заказ и бронирование (RQST)
**Подход:** запись заказа: 30 читатель · 903 шифр · 1 БД ЭК · 40 дата создания · 201/31 brief. **Тип по полю 41** (`I=1` выполнен · `2` бронь(43) · `3` отказ(44) · `0` невыполнен). Подбор **свободных экземпляров** (`freekz` — статусы 910 в ЭК, учёт мест 56/57). Печать **требования** (`ORDER.pft`, RTF) / талон; отметка печати → 911.
**Бронь:** 43 дата; **статус брони 910^A по `reservstatus.mnu`** (`0` отправлен МХР→МВ · `1` получен на МВ/для выдачи · `2` от читателя · `3` для возврата на МХР · `4` МВ→МХР); уведомление (`reservmail.pft`). Прозрачная **очередь** («вы N-й»).
**AC:** AC1 заказ создаётся, подбираются свободные экз. (по статусам ste.mnu); AC2 печать требования/талона; AC3 бронь со сменой статуса (reservstatus) + уведомление; AC4 очередь брони видна читателю (ЛК, I2).

## 4. Выдача / возврат / продление
**Подход (ключевой формат `RQSTRDR.pft`):**
- **ВЫДАЧА:** оператор выбирает свободный экз (910^A=`0/U/C` в ЭК) → RQST.910 (`^B`инв/`^H`ш-к/`^D`МХР), 41=дата → **строится RDR.40** (`^A`=903, `^B`=910^B, `^H`=910^H, `^K`=910^D, `^D`=выдача, `^E`=план.возврат, `^F`=`******` маркер «на руках», `^G`=БД, `^C`=brief); экз в ЭК → **910^A=`1` (выдан)**.
- **ВОЗВРАТ:** в RDR.40^F пишется факт.дата (вместо `******`); запись переносится в **RDR_ARH** (архив, opt-in); экз → свободный статус; заказ с 41 чистится.
- **ПРОДЛЕНИЕ:** 40^L=дата, обновляется 40^E; лимит `MaxReturnDays`; флаги `PROLONGABLE/STRONGPROLONG`.
- **Лимиты:** `MaxBooks`/`MaxDolgBooks` (книг на руках / при долге), `MaxReturnDays`; задолженность = `40^F:'******'` (FST `DOLG=`).
**AC:** AC1 выдача меняет 910^A→`1` и создаёт loan (40); AC2 возврат закрывает loan (40^F) и архивирует по согласию; AC3 продление с контролем лимита; AC4 блок выдачи при превышении `MaxBooks`/долге; AC5 **офлайн-выдача/возврат на edge** (event-sourced, синк при сети, ADR-005).

## 5. Отказ / утеря / штрафы
**Подход:** **отказ** RQST.44 (`^A` причина `otkaz.mnu`: 01 Занято · 02 Нет на месте · 04 В переплёте · 05 Списано · 06 Лакуна · 07 Дефектный · 08 Ветхий · 09 Перенаправление · * Прочие; `^B` дата). **Утеря:** RDR.40^U=`1` (поиск `HU=`), долговые формы с ценой (910^E из ЭК). **Штрафы/оплаты:** интеграция с БД **PAY** (ключ читатель `RI=`); fine-free-опции/автопродление (PRODUCT_PRACTICES).
**AC:** AC1 отказ с причиной из словаря; AC2 утеря помечает экз + формирует долг с ценой; AC3 штраф/оплата проводится (PAY), видна в формуляре/ЛК.

## 6. Статусы экземпляра (две кодировки — внимание!)
- **В ЭК (910^A, `ste.mnu`):** `0` свободен · `1` **выдан** · `2` ожидается · `3` в переплёте · `4` утерян · `5` врем. не выдаётся · `6` списан · `7` докомплектование · `R/U/C/E` группы/сеть. Свободны для выдачи: `0/U/C`.
- **В RQST (910^A, `reservstatus.mnu` 0–4):** статусы движения брони. ⚠ **Одно подполе — разные кодировки в разных БД** (recon TODO #CIRC-06) — в нашей модели развести явными перечислениями (item_status vs hold_status).
**AC:** AC1 переходы статуса экземпляра соответствуют операциям; AC2 item_status и hold_status — раздельные enum, без путаницы.

## 7. Наш differentiator (сверх паритета)
**Ячейки/постамат 24/7 (#95/#145):** бронь → размещение в **ячейку/постамат** → **код выдачи** → читатель забирает 24/7 → **авто-checkout** (loan создаётся в момент изъятия). **Self-checkout/RFID**, holds-aware локеры (outdoor IP54). **Книгоприём/автосдача** + сортировка по ячейкам. **Рекомендации в момент выдачи** на киоске (NoveList-стиль, cross-sell).
**AC:** AC1 выдача брони через ячейку/постамат по коду с авто-checkout; AC2 self-checkout по RFID/штрих-коду проводит loan; AC3 возврат через станцию автосдачи меняет статус; AC4 всё работает **офлайн** на edge.

## 8. Печать и формы
Читательский билет (`CHB`), **формуляр/расписка** (`rdrw_html_dolg*` — на руках/выдано сегодня/задолженные + подпись), требование (`ORDER`), талон-контроль (`CNTRL`), списки должников (с ценой/книгообеспеч.), свободные экземпляры (`freekz`). На сервере — рендер форматов.
**AC:** AC1 печать билета/формуляра/требования/талона/списка должников.

## Критический путь ws3
1. **Регистрация читателя** (RDR, +ЕСИА) → 2. **заказ/подбор свободных экз.** (RQST/freekz) → 3. **выдача/возврат/продление** (RQSTRDR-логика, статусы) → 4. **бронь/очередь/уведомления** → 5. **ячейки/постамат + self-checkout + офлайн** (differentiator) → 6. **отказ/утеря/штрафы (PAY)** → 7. печать форм.
**Выходной критерий ws3:** библиотекарь (и читатель-самообслуживание) ведёт полный цикл книговыдачи в вебе — регистрация, заказ, выдача/возврат/продление, бронь/очередь, отказ/утеря, штрафы — с корректными статусами и лимитами, **плюс выдача 24/7 через ячейки/постамат и работа офлайн** — без десктоп-АРМа.
