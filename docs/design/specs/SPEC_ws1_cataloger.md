# SPEC · ws1 — АРМ Каталогизатор (полевой уровень)

> Детализация задачи **ws1** ([#183](https://github.com/Rivega42/biblio/issues/183), эпик #98) до **уровня полей / рабочих листов / ФЛК** — чтобы веб-каталогизация заменяла десктоп-Каталогизатор «наполненно». Грунтовано на recon: [FLC_VALIDATION](../../recon/deep/reference/format/FLC_VALIDATION.md) (~70 правил), [FIELD_CATALOG](../../recon/deep/reference/format/FIELD_CATALOG.md) (3366 пар поле×подполе), [DB_IBIS](../../recon/deep/reference/databases/DB_IBIS.md), [CONTROL_VOCABULARIES](../../recon/deep/reference/format/CONTROL_VOCABULARIES.md), [GLOBAL_CORRECTION](../../recon/deep/reference/format/GLOBAL_CORRECTION.md), [PFT_LANGUAGE §13](../../recon/deep/reference/format/PFT_LANGUAGE.md). Родитель — [SPEC_I5](SPEC_I5_staff_arm_parity.md).

## 1. Рабочие листы ввода (worklist engine)
**Подход:** РЛ = тип записи по полю **920** (`PAZK`/`PVK` книги, `SPEC` сводное, `J` журнал, `NJ*` выпуск, `ASP`/`AUNTD` аналитика…). РЛ из `.ws` (страницы+поля) + расширенный ввод подполей из `.wss`. На фронте — `DynamicField` (есть): тип поля → контрол по [FIELD_CATALOG] (`text/menu/dict/tree/authority/yes-no/date`), подполя, **повторяющиеся группы**, подсказки оператору.
**Источник РЛ:** в проде — из конфигурации тенанта (аналог `.ws/.wss`), редактируемой в Администраторе (ws5, low-code). Сейчас — `GET /api/worklist/{db}` (профиль РЛ).
**AC:** AC1 РЛ выбирается по 920, рендерит все страницы/поля/подполя; AC2 повторяющиеся поля/подполя добавляются/удаляются; AC3 контрол подполя соответствует типу (меню/словарь/дерево/авторитет/дата) из FIELD_CATALOG.

## 2. ФЛК — движок валидации (ядро «наполненности»)
**Семантика результата** (из FLC_VALIDATION): `0` ок · `1` **непреодолимая** (сохранение блокируется) · `2` **преодолимая** (предупреждение, сохранить можно); текст сообщения — со 2-го символа.
**Подход:** портировать ~70 правил ФЛК (PFT) как **декларативный ruleset/валидатор** (не PFT-интерпретатор), исполняемый сервером **на сохранении** (общий `dbnflc` по записи) и **на выходе из поля** (полевой `!NNN`). Ключ ветвления — 920. Словарные проверки — лукап по справочникам; дублеты — по инвертированному индексу каталога **тенанта**.
**2.1 Общий ФЛК записи (`dbnflc`, 18 проверок):** ФИО исполнителя (907), код РЛ↔сводное (NJ), регистрируемые комплекты (кумуляция J), **дублетный шифр 903** (словарь `I=`), **дублетное описание** (свёртка БО по 210/200/700/710/215…; для книг +ISBN 10^a/ид.№19/215^1), отсутствие заглавия 200^a, **дублетный инв.№ 910^b / штрих-код 910^h** (`IN=`), внутризаписные дубли, статус/инв.№/число экз-ра, дубли выпусков/экз. журнала (`I=903/934/935/936`), корректность импорта 998, запись-связка 955. Вариант `dbnflck` — для записей-аналогов (699^U).
**2.2 Полевой ФЛК (`!NNN`, ~52 правила):**
- **Идентификаторы:** ISBN 10^a (дублет `B=` + контроль `&unifor('C'…)`), ISSN 11^a/225^x/461^j, ид.№ 19, цена 10^d (структура «NNNN.NN»), шифр 903 (дублет `I=` + запрет правки если выдана).
- **Справочники (.mnu):** язык 101/911 (`jz.mnu`), страна 102 (`str.mnu`), вид док-та 900^b (`vd.mnu`), код РЛ 920 (`920.mnu`).
- **Структура (обязательность подполей):** 110^b/^x, 210^d|^h, 700^a/701^a/702^a/710^a, 901^b, 907.
- **Предметизация/роли:** 600/601 (правила ^w/^z/^(/персоналия), 606^A, 607^A, 699^U0 уникальность, роли авторов (330/454/470/481/488/922/925/963/964 — подполе роли = число).
- **Индексы:** 964 (структура индекса).
- **Экземпляры:** 910 ввод (`!910` — полная регистрация), 910 выбытие (`!910W` — КСУ выбытия ^v/место ^m/акт ^w/число ≤ имеющихся), 940 списание подразделения (`!940W`), 934/936 (журнал).
**AC:** AC1 запись с ошибкой типа 1 **не сохраняется**, сообщение привязано к полю; AC2 тип 2 — предупреждение, сохранение разрешено с подтверждением; AC3 дублет шифра/инв.№/ISBN/описания ловится против каталога тенанта (с ссылкой на конфликтную запись); AC4 контроль контрольной суммы ISBN/ISSN; AC5 справочные коды (язык/страна/вид/РЛ) валидируются по словарю; AC6 ветвление правил по 920 корректно.
**DoD:** покрытие правил тестами (каждое правило — позитив+негатив); правила версионируются per-tenant (Администратор может включать/настраивать).

## 3. Авторитетный контроль и словари
**Подход:** поля-персоналии/предметы (700/701/710/600/601/606/607) — ввод через **авторитетные файлы** (ATHR*: подполе `^3` ссылка, autoin генерация связанных), словари MNU/TRE ([CONTROL_VOCABULARIES] — 940 пар) как источники меню/деревьев. Классификаторы **ГРНТИ/УДК/ББК** — дерево (выбор + автоввод индекса).
**AC:** AC1 выбор персоналии из авторитетного файла подставляет `^3`+autoin; AC2 справочные подполя берут значения из словаря тенанта; AC3 индекс из классификатора проходит `!964`.

## 4. Заимствование / копикаталогизация
**Подход:** поиск во внешнем источнике (**Z39.50/SRU**, **СКБР/ЛИБНЕТ**, WorldCat — связь #123, CN #169/#170) → импорт записи в РЛ с мэппингом полей → ФЛК → правка → сохранение.
**AC:** AC1 поиск и импорт записи по Z39.50/СКБР в текущий РЛ; AC2 импортированная запись проходит ФЛК и предупреждает о дубле (998-контроль).

## 5. Импорт / экспорт
**Подход:** ISO2709 / MARC21 / RUSMARC / MARCXML, пакетный импорт с мэппингом (`import_flc.mnu`) и ФЛК импорта; экспорт (вкл. `rmarce_flc` для RUSMARC). 
**AC:** AC1 круговой импорт→экспорт без потери полей/подполей; AC2 пакетный импорт прогоняет ФЛК и отчитывается об ошибках построчно.

## 6. Инвентаризация (экземпляры 910) и списание
**Подход:** регистрация экземпляров (статус ^a, инв.№ ^b/штрих-код ^h, число ^1, цена ^e) с правилами `!910`; **выбытие/списание** (`!910W`/`!940W`: КСУ выбытия, место, акт, число ≤ имеющихся) + печать **акта списания** (RTF `!WKW`). Привязка экземпляра к **ячейке/локации** (наш differentiator, #95). Запрет правки шифра выданного экземпляра.
**AC:** AC1 регистрация экз. с проверками `!910`; AC2 списание с обязательными КСУ/место/акт и контролем числа; AC3 акт списания печатается; AC4 экземпляр привязывается к ячейке.

## 7. Глобальная корректировка и удаление
**Подход:** пакетные правки по выборке — язык `.gbl` ([GLOBAL_CORRECTION]: 22 оператора ADD/REP/CHA/DEL/DELR/UNDOR/IF/REPEAT…) **с превью и откатом** (журнал изменений, связь с аудитом); ФЛК при удалении (`delflc`).
**AC:** AC1 глоб.корректировка по выборке: превью diff → применение → откат; AC2 удаление проходит `delflc` (блокировки на связанные/выданные).

## 8. Печать и формы
**Подход:** каталожные карточки, списки, табличные формы ([PFT_MODULES]: `.tab/.tbg`), RTF-акты. На сервере — рендер форматов (PFT-совместимый слой или реимплементация ключевых форматов).
**AC:** AC1 печать каталожной карточки/списка по выбранному формату.

## 9. Данные и сервисы
Запись — JSONB поля/подполя (ядро I1). Сервисы: `worklist` (профиль РЛ per-tenant), `validate` (ФЛК-движок, разд.2), `authority` (поиск/подстановка), `import/export` (MARC), `globalcorrect` (.gbl), `print` (форматы). Контракт — в openapi.

## Критический путь ws1
1. **worklist-движок + DynamicField** (есть база) → 2. **ФЛК-движок** (dbnflc + !NNN, разд.2 — ядро) → 3. **авторитеты/словари/классификаторы** → 4. **инвентаризация 910 + ячейки** → 5. **заимствование + импорт/экспорт MARC** → 6. **глоб.корректировка + печать**.
**Выходной критерий ws1:** каталогизатор создаёт/правит запись любого типа (920) по РЛ с **полной ФЛК-валидацией** (≥70 правил, блокирующие/преодолимые), заимствует по Z39.50/СКБР, импортирует/экспортирует MARC, ведёт экземпляры с привязкой к ячейкам, делает пакетную глоб.корректировку с откатом и печать — без десктоп-АРМа.
