# SPEC · ws5 — Администрирование (полевой уровень)

> Детализация задачи **ws5** ([#187](https://github.com/Rivega42/biblio/issues/187), эпик #98) — веб-замена десктоп-«Администратора» (покрытие 25%). Грунтовано на recon: [ARM_ADMIN](../../recon/deep/reference/arms/ARM_ADMIN.md) (IRBISA.INI: `[Main]` ~55 прав-Item, `[FullText]`, `[CONTEXT]`), [CONTROL_VOCABULARIES](../../recon/deep/reference/format/CONTROL_VOCABULARIES.md). Родитель — [SPEC_I5](SPEC_I5_staff_arm_parity.md). **Большая часть ложится на I1** (мультиарендность/Identity/гранты/прод-инфра) — ws5 = веб-модуль поверх него.

> Особенность: Администратор ИРБИС работает **напрямую с физической структурой БД** (актуализация/реорганизация/импорт-экспорт), а не через поисковый интерфейс. У нас физический уровень — это PostgreSQL-операции (миграции/реиндексация/бэкап) + конфиг тенанта.

## 1. Управление БД / тенантами
**ИРБИС-функции** (`[Main]` *Item): создать/открыть/очистить/удалить/скопировать БД; **актуализация/реорганизация** словаря и мастер-файла (`ActualItem`/`ReorgIf`/`ReorgMf`); диагностика (`DiagnosMfItem`); списки удалённых/неактуализированных/заблокированных; разблокировки; импорт/экспорт.
**У нас:** жизненный цикл **тенанта/схемы** (провижининг/удаление — I1 #100/#102), **реиндексация без простоя** (аналог актуализации словаря — alias-swap FTS/GIN), **бэкап/restore** (прод-инфра #99), диагностика целостности, импорт/экспорт данных тенанта (анти-lock-in).
**AC:** AC1 админ создаёт/обслуживает БД/схему тенанта; AC2 реиндексация запускается без простоя; AC3 бэкап/restore документированы и выполнимы; AC4 экспорт данных тенанта.

## 2. Пользователи, роли, гранты, аудит
**ИРБИС:** серверные права/клиенты (`ServerClientList`/`ServerProcessList`/`ServerRestart`).
**У нас (= I1 #101 Identity/Tenancy):** управление **учётками/ролями/грантами** (функция×база×уровень), пресеты-роли, **аудит** действий, активные сессии/клиенты, мониторинг.
**AC:** AC1 CRUD учёток/ролей/грантов; AC2 пресеты ролей; AC3 журнал аудита (фильтр по actor/функции/тенанту); AC4 список активных сессий.

## 3. Справочники и словари (MNU/TRE)
**ИРБИС:** меню импорта/экспорта (`IMPORTMNU`/`EXPORTMNU`), наборы `.mnu`/`.tre`.
**У нас:** **редактор справочников** ([CONTROL_VOCABULARIES] — 31 словарь, 940 пар: статусы экз./брони, причины отказа/выбытия, виды документов, языки, страны…), деревья-классификаторы (УДК/ББК/ГРНТИ/специальности), импорт/экспорт, версии, per-tenant переопределения.
**AC:** AC1 правка значений справочника тенанта; AC2 импорт/экспорт; AC3 деревья-классификаторы редактируются; AC4 системные словари как сиды + override.

## 4. Параметры баз / полей / рабочих листов (low-code конфиг тенанта)
**ИРБИС:** аналог PAR/INI (структура БД, форматы, FST-префиксы, рабочие листы) правится в конфигах.
**У нас:** **low-code конфигуратор тенанта** (PMF): поля/подполя, **рабочие листы** (профили РЛ по 920 → DynamicField), **ФЛК-правила** (вкл/настройка), поисковые префиксы, форматы показа/печати, бренд/тема (связь с редактором тем #154). Без правки кода.
**AC:** AC1 настройка полей/РЛ/ФЛК тенанта через UI; AC2 поисковые префиксы и форматы; AC3 бренд/тема; AC4 изменения версионируются и применяются без релиза.

## 5. Обслуживание полных текстов / DAM
**ИРБИС** (`[FullText]`): актуализация ПТ, извлечение/кэш страниц PDF, проверка извлекаемости, построитель кэша.
**У нас (= DG):** реиндексация полнотекста (OCR/ALTO), деривативы/кэш страниц, IIIF-тайлы, проверка целостности (fixity), очередь обработки.
**AC:** AC1 запуск переиндексации ПТ; AC2 статус/очередь обработки сканов; AC3 проверка извлекаемости/целостности.

## 6. Импорт/экспорт и форматы обмена
**ИРБИС:** MARC/XML-настройки (`PRMARCFORMAT=MARC`, XML-теги), меню import/export/copy.
**У нас:** конфигурация MARC21/RUSMARC/ISO2709/MARCXML обмена (#123), пакетный импорт/экспорт, мэппинг, ФЛК импорта.
**AC:** AC1 настройка форматов обмена; AC2 пакетный импорт/экспорт с отчётом.

## 7. Мониторинг, логи, статистика
**ИРБИС:** серверные списки клиентов/процессов, логи.
**У нас:** **наблюдаемость** (I1 #117 — OTel-дашборды: латентность/ошибки поиска/выдачи, trace+tenant-id), логи, **статистика/отчётность** (КСУ/6-НК/COUNTER), кросс-тенант бенчмарк (анонимно).
**AC:** AC1 дашборды состояния/нагрузки; AC2 журналы; AC3 отчёты КСУ/6-НК; AC4 health/readiness.

## Связь с I1 (не дублируем)
| Область ws5 | Реализуется в |
|---|---|
| Тенанты/схемы/бэкап | I1 #100/#102/#99 |
| Учётки/роли/гранты/аудит | I1 #101 |
| Наблюдаемость/мониторинг | I1 #117 |
| Low-code конфиг тенанта | PMF (маркетплейс/модули) |
| Полнотекст/DAM | DG #161 |
| Справочники/форматы обмена | #123 + reference-данные |
ws5 = **веб-UI администрирования** поверх этих движков; собственного «тяжёлого» бэкенда сверх I1 почти не требует.

## Критический путь ws5
1. **Учётки/роли/гранты/аудит** (на I1 #101) → 2. **управление тенантами/схемами** (провижининг/реиндекс/бэкап, I1 #100/#99) → 3. **редактор справочников** (MNU/TRE) → 4. **low-code конфиг** (поля/РЛ/ФЛК/форматы) → 5. **полнотекст/DAM-обслуживание** → 6. **мониторинг/логи/статистика**.
**Выходной критерий ws5:** администратор управляет тенантом в вебе — учётки/гранты/аудит, БД/схемы (реиндекс/бэкап), справочники, параметры полей/РЛ/ФЛК (low-code), полнотекст, мониторинг и отчётность — без десктоп-модуля. **Завершает ARM-паритет (ws1–ws5).**

> Открытые вопросы: точные серверные операции актуализации/реорганизации проецируются на PostgreSQL-эквиваленты (VACUUM/REINDEX/миграции) — детализировать при реализации I1; права-Item ИРБИС → наши гранты функция×база×уровень.
