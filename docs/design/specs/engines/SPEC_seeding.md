## 0. Постановка: почему это блокер

Три потока read-пути ИРБИС опираются на посеянные справочники (из [CAPABILITY_MAP](../../../recon/deep/CAPABILITY_MAP.md) §3 «Сборка слоёв»):

1. **ФЛК на вводе** (`@`-форматы рабочих листов, [SPEC_I5 §ws1](../SPEC_I5_staff_arm_parity.md)): подполя с контролем `@!amovu` валидируют значение **по словарю** (`vd.mnu`, `jz.mnu`, `str.mnu`, `ste.mnu`…). Пустой словарь → проверка либо всегда падает, либо всегда проходит — оба варианта ломают целостность.
2. **Выбор места хранения / места выдачи (МХ/МВ)**: экземпляр (поле 910) и заказ (RQST) ссылаются на справочник мест (`mhr.mnu` хранилища, `kv.mnu` места обслуживания). Пусто → каталогизатор не выберет фонд, книговыдача не выберет точку выдачи → **выдача/возврат не работают**.
3. **Выпадающие списки и расшифровка кодов на показе**: рендер записи раскрывает `код → расшифровку` (`v900^c` → `hd.mnu` и т.д.). Пусто → пользователь видит сырые коды.

Вывод: сидинг — **предусловие** для ws1 (ФЛК), ws3 (МХ/МВ), I2 (показ). Онбординг #102 не «завершён», пока тенант не получил рабочий набор справочников.

## 1. Модель хранения (PostgreSQL, per-tenant)

Справочники и деревья хранятся в **схеме тенанта** `t_<slug>` (модель «схема-на-тенанта» из [SPEC_I1 #100](../SPEC_I1_foundation.md)), плюс **общесистемный каталог сидов** в `control`. Edge — та же DDL в БД-на-тенанта.

### 1.1 Таблицы тенанта (`t_<slug>`)

```sql
-- Справочник (MNU): метаданные одного словаря
CREATE TABLE vocabulary (
  id            bigserial PRIMARY KEY,
  name          text NOT NULL,            -- логич. имя ИРБИС: 'vd.mnu','ste.mnu','kv.mnu'…
  title         text NOT NULL,            -- человекочитаемо: «Виды документа»
  kind          text NOT NULL,            -- 'system' | 'institution'
  field_hint    text,                     -- к какому полю/подполю привязан: '900^c','910^A','101'
  seed_version  int,                      -- версия системного сида, из которого посеян (NULL для пустых институтских)
  is_overridden boolean NOT NULL DEFAULT false, -- тенант менял системные значения
  updated_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (name)
);

-- Значения справочника: пары код → расшифровка, упорядочены
CREATE TABLE vocabulary_value (
  id            bigserial PRIMARY KEY,
  vocabulary_id bigint NOT NULL REFERENCES vocabulary(id) ON DELETE CASCADE,
  code          text NOT NULL,            -- 'KN','01','1' (регистрозависим — ИРБИС-коды)
  label         text NOT NULL,            -- «Книги в целом»
  sort_order    int  NOT NULL DEFAULT 0,
  is_active     boolean NOT NULL DEFAULT true,  -- мягкое выключение (а не удаление сид-значения)
  origin        text NOT NULL DEFAULT 'seed',   -- 'seed' | 'custom' (для merge при апгрейде)
  UNIQUE (vocabulary_id, code)
);

-- Узел дерева-классификатора (TRE): spec.tre, CIKOD.tre, ГРНТИ, УДК/ББК
CREATE TABLE classification_node (
  id            bigserial PRIMARY KEY,
  tree          text NOT NULL,            -- 'spec.tre','cikod.tre','grnti','udc','bbk'
  parent_id     bigint REFERENCES classification_node(id) ON DELETE CASCADE,
  code          text NOT NULL,            -- индекс/код узла
  label         text NOT NULL,
  depth         int  NOT NULL DEFAULT 0,  -- из числа ведущих точек ИРБИС-.tre
  sort_order    int  NOT NULL DEFAULT 0,
  path          ltree,                    -- материализованный путь для поддеревьев (расширение ltree)
  origin        text NOT NULL DEFAULT 'seed',
  UNIQUE (tree, code)
);
CREATE INDEX ON classification_node USING gist (path);
CREATE INDEX ON classification_node (tree, parent_id);
```

> **Формат-источник** (recon): `.mnu` ИРБИС — пары строк `код` / `расшифровка` подряд, финал-маркер `*****` ([CONTROL_VOCABULARIES](../../../recon/deep/reference/format/CONTROL_VOCABULARIES.md) TODO #VOC-03 — пустая финальная пара фильтруется). `.tre` — текст с **ведущими точками** = уровень вложенности (`depth`), значение узла после точек.

### 1.2 Каталог сидов в `control` (общий, неизменяемый набор «из коробки»)

```sql
-- control.seed_vocabulary / control.seed_vocabulary_value / control.seed_tree_node
-- — мастер-копия СИСТЕМНЫХ словарей и деревьев с версией; источник bootstrap'а.
CREATE TABLE control.seed_catalog (
  name          text PRIMARY KEY,         -- 'vd.mnu'…
  title         text NOT NULL,
  kind          text NOT NULL,            -- 'system' | 'institution'
  field_hint    text,
  version       int  NOT NULL,            -- глобальная версия сида (растёт при обновлении эталона)
  preset        text[] NOT NULL DEFAULT '{all}'  -- к каким пресетам тенанта применять: public/school/vuz/dept
);
```

`control.seed_catalog` + значения — единственный источник правды для «что и какими значениями посеять». Версионируется в коде/миграциях, поставляется со сборкой (см. §3.3).

### 1.3 Почему per-tenant, а не общий справочник

- Институт-специфичные словари (места хранения, факультеты, категории читателей) **различаются у каждой библиотеки** — общая таблица с `tenant_id` всё равно изолируется RLS, но «схема-на-тенанта» даёт тривиальную изоляцию, простой бэкап/экспорт одной библиотеки и совпадает с уже принятой моделью I1 #100.
- Тенант **переопределяет** даже системные значения (выключить вид документа, переименовать статус) — `origin`/`is_overridden`/`is_active` фиксируют кастом, чтобы апгрейд эталона его не затёр (§3.3).

### 2.3 ИНСТИТУТ-СПЕЦИФИЧНЫЕ — авто-импорт с исходного ИРБИС при онбординге (fallback: пусто + ручной ввод)

`kind='institution'`. **Основной путь (разворачивание поверх / миграция с существующего ИРБИС):** мастер онбординга **автоматически вытягивает эти словари и деревья с того ИРБИС-сервера, с которого/на котором разворачивается тенант** — через миграционный адаптер «Проход Б»: чтение `.mnu`/`.tre` по протоколу (`EXPORTMNU` / FILE-команда `L`) либо напрямую из каталога данных сервера (`C:\IRBIS\DATA`). Подключение и учётные данные исходного сервера берутся из **env/конфига онбординга, не из репозитория**. Значения пишутся в `vocabulary_value` с `origin='imported'`; библиотека лишь подтверждает/правит. Так `kv.mnu`/`mhr.mnu`/`spec.tre` сразу рабочие — **блокер закрывается без ручного ввода**. **Fallback (greenfield, исходного ИРБИС нет):** создаётся **пустая запись `vocabulary`** (метаданные + `seed_version=NULL`), значения вносит библиотека в редакторе ws5. Пайплайн импорта — §4.2.

| Словарь / дерево | Что | Откуда брать значения |
|---|---|---|
| `mhr.mnu` | **места хранения (МХ)** | данные библиотеки; для пилота — сервер СПб ГТБ |
| `kv.mnu` | **места обслуживания / выдачи (МВ)** | данные библиотеки (⚠ пуст в текущем конфиге — корень блокера) |

| `kafch.mnu`/`kafv.mnu` | кафедры | вузовская библиотека |
| `spec.mnu` / **`spec.tre`** | специальности (плоский / дерево) | вузовская библиотека (⚠ `spec.tre` пуст) |
| `50.mnu` | категории читателей | данные библиотеки |

## 3. Bootstrap на провижининге (расширение `provision_tenant`)

### 3.1 Шаг в мастере онбординга

Расширяем `provision_tenant` ([SPEC_I1 #102](../SPEC_I1_foundation.md)) новым **идемпотентным** шагом `seed_vocabularies(tenant, preset)` после миграций схемы, до выдачи доступа:

```
create tenant → provision schema t_<slug> → migrate →
  seed access (admin, роли)                       # access/seed.py (есть)
  seed_vocabularies(tenant, preset):              # НОВОЕ — этот движок
    for v in control.seed_catalog where preset ∈ v.preset:
       upsert vocabulary(name,title,kind,field_hint, seed_version=v.version)
       if v.kind == 'system':  copy seed values (origin='seed')
       if v.kind == 'institution': leave empty (seed_version=NULL)
    seed_trees(preset)                             # ГРНТИ/УДК/ББК/cikod
→ tenant ready (ФЛК и выбор МХ/МВ работают для системной части)
```

**Пресеты** (по типу библиотеки из [SPEC_I1 #102](../SPEC_I1_foundation.md): публичная/школьная/вузовская/ведомственная) фильтруют `seed_catalog.preset`: вузовскому тенанту создаются (пустыми) `fak/kaf/spec`, публичному — нет.

### 3.2 Идемпотентность и изоляция

- `upsert` по `UNIQUE(name)` / `UNIQUE(vocabulary_id, code)` — повторный прогон безопасен (AC #100).
- Все вставки — в `search_path = t_<slug>`; тест изоляции: сид тенанта A не виден тенанту B (DoD-гейт I1).

### 3.3 Версионирование системных сидов (апгрейд без перезатирания кастома)

Проблема: эталон обновился (новый вид документа в `vd.mnu`), у тенанта уже есть правки. Алгоритм **`reseed(tenant)`** (трёхсторонний merge по `code`):

| Случай (code) | В сиде vN+1 | У тенанта | Действие |
|---|---|---|---|
| Новый код | есть | нет | **добавить** (`origin='seed'`) |
| Изменён label, тенант не трогал | изм. | `origin='seed'`, не менялся | **обновить label** |
| Изменён label, тенант переопределил | изм. | `origin='custom'` или `is_overridden` | **не трогать**, лог-конфликт в отчёт |
| Удалён из эталона | нет | есть `origin='seed'` | **`is_active=false`** (мягко, не DELETE) |
| Кастом тенанта | нет | `origin='custom'` | **сохранить** |

`vocabulary.seed_version` поднимается до vN+1; отчёт reseed (добавлено/обновлено/конфликты) — в ws5. Запускается админом тенанта или массово оркестратором («apply seed to all tenants», как миграции I1 #100). Институтские словари (`seed_version=NULL`) reseed **не трогает**.

### 3.4 Объёмные деревья (УДК/ББК/ГРНТИ)

Потоковая загрузка батчами (как bulk-загрузка каталога в [SPEC_I1](../SPEC_I1_foundation.md): отложенная индексация — GIST по `path` строится после bulk). Большие деревья seedятся **лениво/в фоне** post-provision, чтобы онбординг укладывался «в минуты»: тенант доступен сразу с MNU + малыми деревьями, УДК/ББК догружаются джобой.

## 4. Редактор, импорт/экспорт (связь с ws5)

### 4.1 Редактор (ws5 §3 — CRUD MNU/TRE)

[SPEC_ws5_admin §3](../SPEC_ws5_admin.md) уже определяет UI «редактор справочников». Этот движок — его бэкенд:
- **MNU**: CRUD значений (`vocabulary_value`), drag-сортировка (`sort_order`), активация/деактивация, отметка `origin='custom'` при ручном добавлении.
- **TRE**: древовидный редактор (`classification_node`) — добавить/переместить узел (пересчёт `path`/`depth`), свернуть/развернуть.
- **Override-индикатор**: системные значения помечены «из сида»; правка ставит `is_overridden=true` и защищает от reseed (§3.3).

### 4.2 Импорт из ИРБИС `.mnu`/`.tre` (миграция / онбординг)

Критично для институт-специфичных словарей (§2.3) — библиотека приносит свои `mhr/kv/fak/spec.tre` со старого сервера:
- **Парсер `.mnu`**: пары строк `код`\n`label`, CP1251→UTF-8, обрыв по `*****` (recon TODO #VOC-03). → `vocabulary_value` (`origin='custom'`).
- **Парсер `.tre`**: ведущие точки → `depth`; восстановление `parent_id`/`path` по стеку уровней.
- **Источник**: загрузка файла в ws5 **или** вытяжка по протоколу с живого сервера ИРБИС (`IMPORTMNU`/`EXPORTMNU`, [SPEC_ws5 §3](../SPEC_ws5_admin.md)) при миграции.
- **Отчёт импорта**: сколько добавлено/обновлено/пропущено (пустые/дубли кодов).

### 4.3 Экспорт (бэкап, анти-lock-in)

- В **ИРБИС-`.mnu`/`.tre`** (обратная совместимость со старым сервером) и в **JSON/CSV** (бэкап/перенос между тенантами).
- Экспорт включён в «экспорт данных тенанта» ([SPEC_ws5 §1](../SPEC_ws5_admin.md), AC4).

Гранты: чтение — staff любого АРМа; запись — `admin` грант на тенант (ws5). Все под `tenant_id` из JWT.

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/api/vocab` | список справочников тенанта (name/title/kind/seed_version/счётчик) |
| `GET` | `/api/vocab/{name}` | значения словаря (код/label/sort/active) — для выпадающих списков и ФЛК |
| `PUT` | `/api/vocab/{name}` | заменить/обновить значения (CRUD, ставит `origin='custom'`) |
| `POST` | `/api/vocab/{name}/import` | импорт `.mnu`/CSV/JSON → отчёт |
| `GET` | `/api/vocab/{name}/export?fmt=mnu\|csv\|json` | экспорт |
| `GET` | `/api/classification/{tree}` | дерево (узлы, lazy по `parent`/поддереву) |
| `PUT` | `/api/classification/{tree}` | правка узлов дерева |
| `POST` | `/api/onboarding/seed` | **bootstrap-эндпоинт**: посев системных сидов + деревьев для тенанта (idempotent, пресет); тело: `{preset, reseed?:bool}` |
| `GET` | `/api/vocab/{name}/validate?code=X` | проверка кода по словарю (используется ФЛК-движком A2 на сохранении) |

**Контракт ФЛК-моста (A2):** ФЛК-движок ([DESIGN_COVERAGE A2](../../DESIGN_COVERAGE.md)) на сохранении записи зовёт `validate(vocab=name, code=value)` → `{valid, label}`; пустой/неактивный код словаря → ошибка ФЛК по полю (показ в ws1). Это формализует «справочную проверку» из [SPEC_I5 §ws1](../SPEC_I5_staff_arm_parity.md).

## 6. Критерии приёмки (AC)

- **AC1 (главный, онбординг):** из пустого состояния **новый тенант онбордится за минуты** (мастер #102) — после провижининга **системные словари посеяны, ФЛК-проверки по словарю работают, выбор МХ/МВ доступен** (после внесения институтских значений). e2e: создать тенант → ввести запись с проверкой `vd.mnu` → выдать экземпляр с выбором МХ.
- **AC2 (split):** системные словари (~31, ~880 значений из CSV) посеяны дефолтами; институт-специфичные (`mhr/kv/fak/spec/...`) созданы **пустыми** с подсказкой; пресет тенанта определяет набор.
- **AC3 (идемпотентность/изоляция):** повторный `seed`/`provision` безопасен; сид тенанта A не виден тенанту B (тест изоляции в CI — DoD-гейт).
- **AC4 (версионирование):** `reseed` добавляет новые эталонные значения, **не затирая** кастом/override тенанта; отчёт о конфликтах; массовый прогон по всем тенантам идемпотентен.
- **AC5 (редактор):** CRUD MNU/TRE в ws5; override помечается и защищается от reseed; сортировка/активация работают.
- **AC6 (импорт/экспорт):** импорт ИРБИС `.mnu`/`.tre` (CP1251, `*****`-обрыв, точки-уровни) → значения тенанта с отчётом; экспорт в `.mnu`/CSV/JSON.
- **AC7 (API/контракт):** все эндпоинты §5 в `openapi.yaml`, contract-test зелёный; `validate` отдаёт `{valid,label}` для ФЛК.

## 7. Критический путь

1. **Каталог сидов** `control.seed_catalog` + значения системных словарей из `CONTROL_VOCABULARIES.csv` (вендор-данные, версия 1).
2. **DDL тенанта** (`vocabulary`/`vocabulary_value`/`classification_node`) + миграция (прогон по всем схемам).
3. **`seed_vocabularies()` в `provision_tenant`** (#102) — системные + пустые институтские по пресету → разблокирует онбординг.
4. **API** `GET/PUT /api/vocab`, `/api/classification`, `/api/onboarding/seed` + контракт.
5. **Мост ФЛК** `validate` (связь A2) и **выбор МХ/МВ** в ws1/ws3 (потребители).
6. **Редактор ws5** (CRUD) + **импорт/экспорт** `.mnu`/`.tre` (миграция институтских данных).
7. **`reseed`/версионирование** + объёмные деревья (УДК/ББК) фоном.

**Выходной критерий:** новый тенант онбордится за минуты с рабочими ФЛК-проверками по словарю и выбором места хранения/выдачи; библиотека довносит институт-специфичные справочники импортом или в редакторе; обновление эталонных сидов не затирает кастом.

## 8. Риски

- **R1 — институтские данные не готовы к онбордингу.** `kv/mhr/spec.tre` пусты у нового тенанта → ws3/ws1 частично нерабочи до их заполнения. *Митигация:* мастер онбординга **блокирует** перевод тенанта в «прод» до минимального набора (≥1 МХ, ≥1 МВ); чек-лист готовности; импорт со старого сервера в один клик.
- **R2 — затирание кастома при апгрейде эталона.** *Митигация:* `origin`/`is_overridden` + трёхсторонний merge (§3.3), мягкое `is_active=false` вместо DELETE, отчёт конфликтов.
- **R3 — объём деревьев (УДК/ББК — тысячи узлов).** *Митигация:* фоновая потоковая загрузка post-provision, lazy-выдача поддеревьев, индекс `ltree` после bulk.
- **R4 — кодировка/артефакты импорта** (CP1251, `*****`, пустые пары — recon #VOC-03). *Митигация:* парсер с явной перекодировкой и фильтром, отчёт пропусков.
- **R5 — расхождение «системное vs институтское»** для гибридов (`kp.mnu`). *Митигация:* явная колонка `preset`/`kind` в `seed_catalog`, ревизия спорных словарей по колонке DB в CSV.
- **R6 — дрейф ФЛК-контракта** (A2 ожидает иной формат ответа `validate`). *Митигация:* совместная фиксация контракта `validate` в openapi с движком ФЛК (#188 ↔ A2).
