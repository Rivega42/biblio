# SPEC · Движок ФЛК — декларативный валидатор записи (формально-логический контроль)

> Сквозной движок **A2** из [DESIGN_COVERAGE §A](../../DESIGN_COVERAGE.md) (эпик [#188](https://github.com/Rivega42/biblio/issues/188)). Закрывает дыру: «~70 правил ФЛК перечислены, нет схемы правил/валидатора/per-tenant-тюнинга». Грунтовано на recon: [FLC_VALIDATION](../../../recon/deep/reference/format/FLC_VALIDATION.md) (авторитетный каталог: `dbnflc.pft` 18 проверок записи + ~52 полевых `!NNN.pft` + обязательность + словарные лукапы + семантика результата 0/1/2), [PFT_LANGUAGE §9.1/§13](../../../recon/deep/reference/format/PFT_LANGUAGE.md) (UNIFOR `C` контроль ISBN/ISSN; первый символ результата ФЛК), [CAPABILITY_MAP §7](../../../recon/deep/CAPABILITY_MAP.md). Заменяет тонкую ссылку в [SPEC_ws1 §2](../SPEC_ws1_cataloger.md). Родитель — [SPEC_I5](../SPEC_I5_staff_arm_parity.md). Движок выражений — **переиспользуется из A1** ([SPEC_engine_pft](SPEC_engine_pft.md)), не дублируется.

**Назначение.** Декларативный, конфигурируемый per-tenant движок, который проверяет черновик записи (на сохранении и на выходе из поля) против набора правил, портированных из PFT-форматов ФЛК ИРБИС, и возвращает список нарушений с severity (0/1/2) и сообщением, привязанным к полю/подполю. **Это не PFT-интерпретатор**: правила хранятся декларативно (JSON/YAML); сложные предикаты (контрольная сумма ISBN, словарный лукап, дублет по индексу) делегируются в общий слой выражений A1 через именованные функции.

---

## 0. Принципы и границы

- **Декларативность.** Каждое правило — запись в схеме (разд. 1), а не код. Добавление/смягчение правила = правка конфигурации тенанта, не релиз.
- **Семантика результата ИРБИС сохранена точно** ([PFT_LANGUAGE §13](../../../recon/deep/reference/format/PFT_LANGUAGE.md)): первый символ результата ФЛК → `severity`: `0` контроль пройден · `1` **непреодолимая** ошибка (сохранение блокируется до устранения) · `2` **преодолимая** ошибка (можно сохранить, не исправляя, с подтверждением). Текст со 2-го символа → `message`.
- **Переиспользование A1.** Движок ФЛК не считает сам контрольную сумму, не парсит словарь, не строит ключ свёртки БО — он вызывает функции A1 (`isbnIssnCheck`, `dictLookup`, `dupKey`, `unifor`). A1 — единственный слой, знающий про `&unifor('C'…)`, `I=`/`B=`/`IN=` индексы, свёртку `&unifor('B')`.
- **Тенант-изоляция.** Дублетные проверки бьются по инвертированному индексу каталога **тенанта**; словари (`jz/str/vd/920.mnu`) — из конфигурации тенанта (зависимость A5-сидинг, [DESIGN_COVERAGE §A5](../../DESIGN_COVERAGE.md)).
- **Границы.** Движок ФЛК **только валидирует**. Он не переносит данные (форматы `@!*mov*` — это автоформатирование ввода, не ФЛК, см. [FLC_VALIDATION §3](../../../recon/deep/reference/format/FLC_VALIDATION.md), отдельный механизм worklist), не печатает (`!WKW` — RTF-формат, не контроль), не правит связки (955 — побочный эффект, см. разд. 4.3).

## 1. Схема правил (ядро деливерабла)

Правило — декларативная запись. Хранится как JSON (канон) / YAML (редактирование). Ключевая идея: **`condition` — это выражение в DSL, делегируемое в A1**, а `severity`/`message`/`scope`/`branch` — метаданные движка ФЛК.

### 1.1 JSON-схема правила

```jsonc
{
  "id": "string",              // стабильный ID, напр. "fld.10.isbn.checksum"
  "source": "string",          // происхождение из recon: "!10.PFT" | "dbnflc.pft#4" | "delflc.pft"
  "scope": "record|field|subfield",
  "field": "010",              // метка поля (для field/subfield); null для record
  "subfield": "a",             // буква подполя (для subfield); null иначе
  "branch": {                  // ветвление по типу записи (поле 920); опускается = применять всегда
    "field": "920",
    "match":  ["J", "NJ*"],    // список значений/масок (суффикс * = префиксное совпадение, как v920:'NJ')
    "negate": false            // true = правило активно, КОГДА 920 НЕ в match
  },
  "when": "<expr>",            // предусловие применимости (напр. p(v10^a) — поле непустое); опускается = всегда
  "condition": "<expr>",       // ИСТИНА ⇒ нарушение. Выражение DSL A1.
  "severity": 1,               // 0|1|2 — семантика ИРБИС. 1 непреодолимая, 2 преодолимая.
  "message": "Ошибка в ISBN: {{v10^a}}",   // шаблон; {{...}} — подстановка из записи через A1.format()
  "path": "010/a",             // куда привязать ошибку в UI (поле/подполе/инстанс)
  "phase": "save|fieldExit|import|delete", // когда исполнять (см. разд. 2.1)
  "enabled": true,             // глобальный дефолт (можно переопределить per-tenant)
  "tenant_override": {         // НЕ хранится в каноне; накладывается слоем тенанта (разд. 3)
    "enabled": null,           // null = наследовать
    "severity": null,          // null = наследовать; тенант может СМЯГЧИТЬ 1→2, не ужесточить (политика)
    "message": null            // null = наследовать
  },
  "doc": "string",             // человекочитаемое пояснение правила (для редактора в ws5)
  "version": 1                 // версия определения правила (для миграций/аудита)
}
```

**Семантика `condition`.** Выражение возвращает булево; `true` означает **нарушение** (срабатывание правила). Это удобнее, чем кодировать «первый символ 0/1/2» внутри выражения: severity вынесен в метаданные. Движок A1 уже отдаёт «контроль пройден» через `&unifor('C'…)='0'`; в нашей схеме `condition` для ISBN — это `unifor('C', v('010','a')) != '0'`.

### 1.2 DSL выражений (тонкий слой над A1)

DSL — минимальный, читаемый, детерминированный. Парсится в AST, исполняется поверх A1-контекста (запись = дерево поле→подполе→инстанс). Полный список встроенных функций — контракт с A1.

| Конструкция | Значение | Маппинг на PFT/ИРБИС |
|---|---|---|
| `v('NNN','x')` / `v10^a` | значение подполя (первый инстанс) | `v10^a` |
| `vall('NNN','x')` | список всех инстансов подполя | `v910^b` (повторяющееся) |
| `p(expr)` | поле/подполе присутствует и непусто | `p(vNNN)` |
| `a(expr)` | поле/подполе отсутствует | `a(vNNN)` |
| `unifor(code, arg)` | вызов UNIFOR через A1 | `&unifor('C'v10^a)`, `&unifor('B')` свёртка |
| `isbnIssnCheck(s)` | контроль контр. суммы (= `unifor('C',s)=='0'`) | `&unifor('C'…)='0'` ([PFT §9.1](../../../recon/deep/reference/format/PFT_LANGUAGE.md)) |
| `dictLookup(file, key)` | значение из `.mnu`-справочника тенанта; пусто = нет в справочнике | `&unifor('K<файл>.mnu|<знач>')` |
| `dictHas(file, key)` | булево «код есть в справочнике» | словарная проверка 101/102/900/920 |
| `dupKey(index, key, excludeMfn)` | есть ли в инвертированном индексе тенанта запись с таким ключом ≠ текущий MFN | словари `I=`/`B=`/`IN=` |
| `count(expr)` / `distinctCount(expr)` | счётчики (внутризаписные дубли) | глоб. перем. `+7` в `dbnflc` |
| `matches(s, /regex/)` | регэксп (структура цены, индекса, двойной пробел) | разбор `*N.1`, `"NNNN.NN"` |
| `branch920(masks)` | предикат ветвления по 920 (поддержка масок) | `if v920='J'`, `v920:'NJ'` |
| логика | `&&`,`\|\|`,`!`,`==`,`!=`,`>`,`<`,`in`,`startsWith`,`contains` | операторы PFT |

> **Контракт с A1.** `unifor/dictLookup/dictHas/dupKey/format` реализованы в A1 ([SPEC_engine_pft](SPEC_engine_pft.md)); движок ФЛК их только вызывает. Если A1 ещё не поставил функцию (напр. полная свёртка БО `&unifor('B')`), правило помечается `enabled:false` с тегом `blocked-on:A1` и не теряется (см. риски).

**(1) ISBN — контрольная сумма** (`!10.PFT`, [FLC §2.1](../../../recon/deep/reference/format/FLC_VALIDATION.md)):
```jsonc

  "when":"p(v10^a)", "condition":"unifor('C', v10^a) != '0'",
```

```jsonc
  "branch":{"field":"920","match":["NJ*"],"negate":true},
```

## 2. Движок исполнения (evaluation engine)

### 2.1 Фазы и порядок

Правила группируются по `phase`; в каждой фазе исполняются в детерминированном порядке.

1. **`fieldExit`** — на выходе из поля в UI (живая обратная связь). Исполняются только правила с `scope` field/subfield этого поля. Полевой ФЛК `!NNN`.
2. **`save`** — на сохранении записи. Полный прогон: общий `dbnflc` (record-scope) + все полевые. Порядок внутри фазы:
   - (a) **920 первым** — `fld.920.worklist.dict`. 920 — селектор ветвления; если он невалиден/пуст, остальное ветвление неопределённо → правило 920 даёт severity 1 и помечает контекст.
   - (b) **полевые `!NNN`** в порядке возрастания метки поля (детерминизм; правила одного поля — по `id`).
   - (c) **record-scope `dbnflc`** (дублеты, обязательность, экземпляры, импорт 998, связка 955).
3. **`import`** — на пакетном/копикаталог-импорте (998-контроль, разд. 4.1).
4. **`delete`** — на удалении (`delflc`, разд. 4.2).

### 2.2 Ветвление по 920

Перед прогоном движок вычисляет `recordType = v920` один раз. Для каждого правила: если есть `branch`, применяется предикат `branch920` — правило исполняется только если `recordType` (НЕ)совпадает с масками. Маски: точное значение (`J`), префикс (`NJ*` ≡ `v920:'NJ'`). Так одно PFT-условие `if v920='J' … else …` раскладывается на пару правил с противоположными `branch.negate`, и severity остаётся в метаданных, а не зашит в выражение.

### 2.3 Применимость и вычисление

Для каждого правила фазы (после фильтра по `enabled` и `branch`):
1. Если есть `when` — вычислить; ложь ⇒ правило пропущено (пустое поле проходит, как `if p(vNNN) then … else '0'` в PFT).
2. Вычислить `condition` через A1. Исключение в A1 (нерешённая функция, недоступный индекс) ⇒ правило → результат `engineError` (severity 2, спец-сообщение «правило N не вычислено»), не роняет весь прогон.
3. Если `condition == true` ⇒ сформировать `Violation { ruleId, severity (с учётом tenant_override), message (A1.format(template, record)), path, dupRef? }`.

### 2.4 Агрегация результата

- Собрать все `Violation` фазы в список.
- **`overallSeverity = worst(severity)`** по списку: `1` если есть хоть одна непреодолимая; иначе `2` если есть преодолимые; иначе `0`.
- **Решение о сохранении:** `overallSeverity == 1` ⇒ сохранение **блокируется**; `== 2` ⇒ сохранение разрешено **после подтверждения** оператором (разд. 5.2); `== 0` ⇒ сохранение без вопросов.
- Список сортируется: сначала severity 1, затем 2; внутри — по `path` (чтобы UI показал ошибки сверху вниз по форме).

### 2.5 Производительность

- Полный `save`-прогон ≤ ~70 правил; целевая латентность p95 ≤ 150 мс при тёплом индексе тенанта. Дублетные `dupKey` — основная стоимость (обращение к индексу); A1 батчит лукапы (один запрос на все ключи записи).
- `fieldExit` — единичное правило, ≤ 30 мс (контр. сумма/регэксп — локально; словарь — кэш тенанта).
- Словари и скомпилированный ruleset кэшируются per-tenant (инвалидация по версии конфигурации, разд. 3.3).

## 3. Per-tenant тюнинг

### 3.1 Слои конфигурации

`effectiveRule = canonRule ⊕ tenantOverride` (наложение). Канон (разд. 1.3, импорт из recon) — read-only, версионируется в коде. Слой тенанта хранит **только дельты**:
- `enabled` — включить/выключить правило.
- `severity` — **смягчение** `1→2` (политика по умолчанию: тенант может ослабить, но не ужесточить `2→1` без флага `allowHardening`, чтобы не сломать сценарии сотрудников; настраивается).
- `message` — кастом-текст (мультиязычность, локальная терминология).
- `params` — параметризованные пороги, где применимо (напр. список служебных статусов инв.№ для правила 4).

### 3.2 Редактор (ws5 / low-code)

Управляется в Администраторе ([SPEC_ws5](../SPEC_ws5_admin.md), связь D1 low-code [DESIGN_COVERAGE §D1](../../DESIGN_COVERAGE.md)): таблица правил (id, поле, severity, вкл/выкл, источник recon), переключатели, поле сообщения, предпросмотр («прогнать на тестовой записи»). Изменение severity 1→2 требует роли с грантом и пишется в аудит.

### 3.3 Версионирование

- Набор правил тенанта = версионируемый артефакт (`flkRulesetVersion`, монотонный). Каждое сохранение конфигурации → новая версия + запись в аудит (кто/когда/diff).
- Результат `/api/validate` помечается `rulesetVersion`, использованной при прогоне (воспроизводимость, объяснение «почему тогда прошло»).
- Откат: восстановление предыдущей версии целиком (как `.gbl`-откат, единая модель журналирования с ws5).

## 4. Специальные фазы

### 4.1 Валидация импорта (998-контроль)

Фаза `import` ([FLC §1#17](../../../recon/deep/reference/format/FLC_VALIDATION.md): «Корректность импорта 998»). Правило `rec.998.import.check` (severity 2): при наличии `v998` (метка происхождения импорта) и не-`J`/спецслучай 907 — предупреждение «Проверьте корректность импорта заглавия/выходных данных/источника по метке 998». При пакетном импорте движок прогоняет **полный `save`-ruleset построчно** и отдаёт отчёт: на каждую запись — список нарушений; записи с severity 1 не импортируются (или импортируются в карантин-черновик, политика тенанта), severity 2 — импортируются с флагом «требует ревью». Связь с `import_flc.mnu` (маппинг, recon TODO FLC-05).

### 4.2 Валидация удаления (delflc)

Фаза `delete` (`delflc.pft`, recon TODO FLC-04, [FLC §5](../../../recon/deep/reference/format/FLC_VALIDATION.md)). Блок на удаление записи, у которой есть **связанные/выданные** экземпляры или входящие ссылки:
- экземпляр 910 в статусе выдачи/брони (по `ste.mnu`/`reservstatus.mnu` тенанта) ⇒ severity 1, блок;
- запись является базовой для аналога (699) или мишенью связки (955) ⇒ severity 1, блок «есть зависимые записи».
Точные статусы/правила — `TODO(recon FLC-04)`; движок исполняет их декларативно, как только статусы зафиксированы на боевом конфиге тенанта.

### 4.3 Побочный эффект связки 955

`dbnflc.pft#18` вызывает `&uf('6!955')` — правку записи-связки. Это **не валидация**, а мутация. Движок ФЛК его **не выполняет**; он лишь может вернуть `Violation` severity 2 «связка 955 рассинхронизирована» (сравнение с эталоном `ref(mfn,v955)`), а саму правку делает сервис записи после успешного сохранения (recon TODO FLC-08). Граница ответственности зафиксирована, чтобы движок оставался чистой функцией record→violations.

## 5. API

### 5.1 `POST /api/validate`

Валидация черновика без сохранения (вызывается на `save`-намерении и в редакторе правил для предпросмотра).

**Запрос:**
```jsonc
{
  "db": "IBIS",
  "phase": "save",                 // save | fieldExit | import | delete
  "field": "010",                  // только для phase=fieldExit
  "record": { /* JSONB поля/подполя, ядро I1 */ },
  "currentMfn": 12345,             // для исключения себя из дублетных проверок; null для новой записи
  "rulesetVersion": null           // null = текущая версия тенанта
}
```

**Ответ:**
```jsonc
{
  "overallSeverity": 1,            // 0 | 1 | 2
  "canSave": false,                // false при overallSeverity==1
  "rulesetVersion": 42,
  "violations": [
    { "ruleId":"fld.10.isbn.checksum", "severity":1,
      "message":"Ошибка в ISBN: 5-7654-0001-X", "path":"010/a", "field":"010", "subfield":"a" },
    { "ruleId":"rec.910b.dup.inv", "severity":2,
      "message":"Дублетный инв.номер: 1024365 (см.N 8891)", "path":"910/b", "dupRef":8891 }
  ]
}
```

Сервис сохранения записи вызывает `/api/validate` (phase=save) синхронно перед записью: `canSave==false` ⇒ 409-подобный отказ с `violations`; `canSave==true` и есть severity 2 ⇒ сохранение проходит только если запрос несёт `acknowledgedWarnings: [ruleId…]` (подтверждение оператора, 5.2).

### 5.2 Как UI показывает преодолимое (тип 2)

- severity 1 ⇒ ошибки подсвечиваются у соответствующих полей (`path`), кнопка «Сохранить» заблокирована, сводка сверху.
- severity 2 ⇒ поля помечены предупреждением; при нажатии «Сохранить» открывается диалог-подтверждение со списком преодолимых замечаний и чекбоксом «сохранить, не исправляя». Подтверждение шлёт `acknowledgedWarnings` с ID этих правил → повторный `save` проходит. Это точная веб-проекция семантики ИРБИС «2 — можно не исправлять».

### 5.3 Вспомогательные

- `GET /api/flk/rules?db=IBIS` — действующий ruleset тенанта (для редактора ws5 и документации).
- `PUT /api/flk/rules` — сохранить дельты тенанта (новая `rulesetVersion`, аудит).
- `POST /api/flk/preview` — прогон правила/набора на тестовой записи (предпросмотр в редакторе).

Контракты — в общем OpenAPI ([SPEC_I1 §контракты](../SPEC_I1_foundation.md), связь A7 [DESIGN_COVERAGE §A7](../../DESIGN_COVERAGE.md)).

## 6. Критерии приёмки (AC)

- **AC1.** Запись с нарушением **severity 1** не сохраняется; `/api/validate` отдаёт `canSave:false`, каждое нарушение привязано к `path` (поле/подполе). *(тип 1 → блок)*
- **AC2.** Нарушение **severity 2** не блокирует: сохранение проходит после `acknowledgedWarnings`; без подтверждения — отказ с предложением подтвердить. *(тип 2 → преодолимо)*
- **AC3.** Дублет шифра 903 / инв.№ 910^b / ISBN 10^a / описания ловится против **инвертированного индекса каталога тенанта**, в сообщении — ссылка на конфликтную запись (`dupRef`); своя запись (по `currentMfn`) не считается дублем.
- **AC4.** Контроль контрольной суммы ISBN/ISSN (`unifor('C',…)`) даёт severity 1 на заведомо битом ISBN и 0 на корректном (тест-векторы в матрице).
- **AC5.** Справочные коды язык 101 / страна 102 / вид 900^b / РЛ 920 валидируются по словарю тенанта; отсутствующий в `.mnu` код → нарушение нужной severity.
- **AC6.** Ветвление по 920 корректно: правило с `branch.match:["J"]` не срабатывает на `PAZK` и наоборот; маска `NJ*` ловит `NJ31` (как `v920:'NJ'`).
- **AC7.** Per-tenant override: выключенное правило не исполняется; смягчённое `1→2` меняет `canSave`; кастом-сообщение возвращается; изменение версионируется и пишется в аудит.
- **AC8.** Импорт (998): пакетный прогон отчитывается построчно; записи с severity 1 не попадают в чистовик; severity 2 импортируются с флагом ревью.
- **AC9.** Удаление (delflc): запись с выданным/связанным экземпляром не удаляется (severity 1).
- **AC10.** Воспроизводимость: ответ помечен `rulesetVersion`; прогон той же записи на той же версии даёт идентичный результат (детерминизм, отсортированный список).
- **AC11.** Отказоустойчивость: нерешённая в A1 функция в одном правиле не роняет весь прогон (правило → `engineError` severity 2, остальные исполняются).

## 7. Тест-матрица

> **Главный тест: прогон реальных правил из [FLC_VALIDATION](../../../recon/deep/reference/format/FLC_VALIDATION.md).** Каждое из ~70 правил — позитив (нарушение ловится) + негатив (валидная запись проходит). Машинный импорт каталога в фикстуры.

| Группа | Правило (источник) | Позитив | Негатив | Ожидание |
|---|---|---|---|---|
| Контр. сумма | ISBN `!10` / ISSN `!11`/`!11S`/`!461` | битый ISBN `5-7654-0001-0` | валидный `5-7654-0001-X` | sev1 / 0 |
| Обязательность | 200^a `dbnflc#6` | книга (920=PAZK) без 200^a | с 200^a | sev1 / 0 |
| Обязательность (ветвл.) | 200^a для аналитики (920=A*) | аналитика без 200^a | — | sev2 (смягчено ветвлением) |
| Словарь | 101 `jz.mnu`, 102 `str.mnu`, 900^b `vd.mnu`, 920 `920.mnu` | код вне справочника | код из справочника | sev1/2 / 0 |
| Дублет (индекс) | 903 `I=`, 910^b/^h `IN=`, 10^a `B=` | значение есть у другого MFN | уникально / у себя (`currentMfn`) | sev / 0 |

| Внутризаписные дубли | инв.№/штрих-код `dbnflc#9/10` | два одинаковых 910^b в записи | все уникальны | sev2 / 0 |
| Структура | 210^d\|^h, 700^a, 901^b, цена 10^d, индекс 964 | нарушенная структура | корректная | sev / 0 |

## 8. Критический путь

1. **Схема правил 1.1 + DSL 1.2** (контракт с A1) → 2. **импортёр recon→ruleset** (машинный перенос 70 правил из FLC_VALIDATION в JSON; разд. 1.3 — образец) → 3. **движок исполнения 2.x** (фазы, ветвление 920, агрегация worst-severity) → 4. **`POST /api/validate` + UI преодолимого (тип 2) 5.x** → 5. **per-tenant тюнинг + версионирование 3.x** (редактор в ws5) → 6. **фазы import/delete 4.x**.

**Зависимости:** A1 ([SPEC_engine_pft](SPEC_engine_pft.md)) — функции `unifor/dictLookup/dupKey/format`; A5-сидинг ([DESIGN_COVERAGE §A5](../../DESIGN_COVERAGE.md)) — словари `jz/str/vd/920.mnu` и инвертированные индексы тенанта; I1 — модель записи JSONB и OpenAPI.

**Выходной критерий A2:** все ~70 правил из FLC_VALIDATION исполняются декларативно, `/api/validate` блокирует severity 1 и проводит severity 2 с подтверждением, дублеты бьются по индексу тенанта, правила настраиваются/версионируются per-tenant — каталогизатор ([SPEC_ws1](../SPEC_ws1_cataloger.md)) валидирует запись «как ИРБИС», без PFT-интерпретатора и без десктоп-АРМа.

## 9. Риски и открытые вопросы

- **R1 (recon-пробелы блокируют точность).** Значения справочников `jz/str/vd.mnu` не извлечены (TODO FLC-07); `delflc`/`rmarce_flc` не разобраны (FLC-04); INI-параметры ветвления (`KUMJEX`,`FIO`,…) не зафиксированы (FLC-09). **Митигация:** соответствующие правила импортируются с `enabled:false`/`blocked-on:recon` и активируются по факту извлечения с боевого конфига тенанта — ни одно правило не теряется.
- **R2 (зависимость от A1).** Сложные предикаты (полная свёртка БО `&unifor('B')`, словарные индексы) живут в A1. Рассинхрон контрактов ломает правила. **Митигация:** фиксированный список функций-контракта (1.2), `engineError`-изоляция (AC11), общий тест-вектор A1↔A2.
- **R3 (производительность дублетов).** `dupKey` на каждый ключ — обращение к индексу; на больших каталогах p95 может расти. **Митигация:** батч-лукап, кэш, асинхронная де-дубликация для severity 2.
- **R4 (политика ужесточения).** Разрешать ли тенанту `2→1`? По умолчанию нет (ломает привычные сценарии). Нужна явная политика и грант.
- **R5 (соответствие 1:1 с PFT).** Раскладка одного ветвящегося PFT-условия на пару правил (200^a) увеличивает число правил; нужен линк `source` и тест эквивалентности against исходного `dbnflc.pft`.
