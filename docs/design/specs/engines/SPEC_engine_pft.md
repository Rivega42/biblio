# SPEC · Движок PFT-форматирования и печати (engine)

> Закрытие сквозного пробела **A1** ([DESIGN_COVERAGE §A](../../DESIGN_COVERAGE.md), эпик **#188**) — единый движок показа/печати, без которого АРМы Каталогизатор/Комплектатор/Книговыдача **не печатают**, а карточка записи/выдача — не форматируются. Грунтовано на recon: [PFT_LANGUAGE](../../../recon/deep/reference/format/PFT_LANGUAGE.md) (~44 конструкции + 41 UNIFOR-код), [PFT_MODULES](../../../recon/deep/reference/format/PFT_MODULES.md) (выходные/табличные формы, `.tab/.tbg/.stf`, ~1396 тел форматов, 68 модулей-подпрограмм), [CAPABILITY_MAP §6/§11](../../../recon/deep/CAPABILITY_MAP.md). Заменяет «тонкие» ссылки на печать в [SPEC_ws1 §8](../SPEC_ws1_cataloger.md) и [SPEC_I2 §B](../SPEC_I2_catalog_search.md). Родители — [SPEC_I5](../SPEC_I5_staff_arm_parity.md) (АРМ-паритет) и [SPEC_I2](../SPEC_I2_catalog_search.md) (карточка/discovery).
>
> Платформа: **PostgreSQL + Python-бэкенд**, без переиспользования `irbis64.dll`/`genpft64.exe`. Движок — отдельный сервис `format` (cross-cutting, потребляется ws1–ws5 и читателем).

**DoD-гейт (как в I1/I2):** код + автотест + контракт-тест (openapi) + **тест изоляции тенанта** (форматы/бланки per-tenant) + наблюдаемость (trace/`tenant_id`, время рендера в мс — паритет `genpft64`, [PFT_LANGUAGE §1](../../../recon/deep/reference/format/PFT_LANGUAGE.md)) + a11y (HTML-показ WCAG 2.2 AA) + обновление доков. Размеры S/M/L.

## 1. Архитектура: что мы реально строим

### 1.1 Решение — **гибрид: PFT-интерпретатор (AST) → IR-документ → backend-рендереры**

Рассмотрены три варианта; решение обосновано ниже.

| Вариант | Суть | Вердикт |
|---|---|---|
| **(A) Реимплементировать PFT как интерпретатор** | Лексер/парсер PFT → AST → исполнитель над записью | Дорого целиком (41 UNIFOR + IF + группы + `ref`/`l` с обращением к инвертированному файлу), но **только так round-trip с реальными `.pft`** |
| **(B) Транслировать PFT → Handlebars/Jinja-шаблон** | Однократный транспил PFT→шаблон, дальше рендер шаблонизатором | Невозможно полно: PFT — **императивный** язык над рабочей областью (последовательное исполнение команд, [PFT_LANGUAGE §1](../../../recon/deep/reference/format/PFT_LANGUAGE.md)), с побочными эффектами (глоб. переменные `+1/+7`, счётчики `S`, постредактура `!`/`+F`), рекурсией (`ref`) и обращением к индексу (`l()`, `ref(l(...))` в `brief.pft`). Jinja-таргет потерял бы семантику автопунктуации/режимов `MP/MH/MD` |
| **(C) Гибрид (выбран)** | **Интерпретатор PFT-AST** исполняет формат над записью → строит **промежуточное представление (IR)** = дерево узлов «текст+разметка» (абзацы, run-ы с курсив/жирн/подчёрк, ячейки таблиц, разрывы) → **backend-рендереры** (HTML / PDF / RTF / DOCX). UNIFOR — **плагинный реестр кодов** | ✅ |

**Обоснование гибрида.** PFT нельзя «оттранслировать раз и навсегда» (вариант B) из-за императивной модели рабочей области и побочных эффектов. Но и «голый интерпретатор, эмитящий сразу HTML/RTF» (наивный A) сцепил бы логику формата с разметкой и заставил бы **дублировать** каждый формат под RTF и HTML (как в ИРБИС: пары `KN`/`Kn_H`, два OPT-файла `PFTW.OPT`/`Pftw_H.opt` — [PFT_MODULES §2.1](../../../recon/deep/reference/format/PFT_MODULES.md)). Введение **IR между интерпретатором и рендерером** даёт: один проход исполнения формата → много целевых форматов (HTML для показа, PDF для печати, RTF/DOCX для бланков), и развязывает команды размещения (`/ # % Xn Cn`, [PFT_LANGUAGE §4](../../../recon/deep/reference/format/PFT_LANGUAGE.md)), которые по-разному применимы в RTF и HTML (в HTML `/ # % Cn Xn` не работают — [PFT_LANGUAGE §4](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).

### 1.2 Компоненты сервиса `format`

```
 PFT-исходник (.pft тела/модули per-tenant) ──▶ [Loader+Cache]
                                                    │ резолв пути поиска (Deposit→БД, аналог PFTPTH)
                                                    ▼
 запись (JSONB поля/подполя, ядро I1) ──▶ [Lexer] ─▶ [Parser→AST] ─▶ [Interpreter]
                                                                         │  · рабочая область (вывод)
                                                                         │  · глоб. переменные сеанса (+1/+7)
                                                                         │  · UNIFOR-реестр (плагины)
                                                                         │  · OPT-диспетчер по полю 920
                                                                         ▼
                                                                    [IR-документ]
                                          ┌──────────────┬──────────┴─────────┬──────────────┐
                                          ▼              ▼                    ▼              ▼
                                    [HTML-рендер]  [PDF-рендер]        [RTF/DOCX-рендер] [Plain/MARC*]
                                     (показ, a11y) (WeasyPrint)        (бланки/акты)
```

- **Loader/Cache:** загрузка тел форматов из конфигурации тенанта (аналог пути поиска `Deposit\` → папка БД, [PFT_MODULES §0](../../../recon/deep/reference/format/PFT_MODULES.md)); кэш скомпилированного AST по `(tenant, format_name, version)`; инвалидация при правке формата.
- **OPT-диспетчер:** выбор главного формата по **полю 920** (`.OPT`-таблица: метка/длина/пары значение→формат/`+++++` умолчание, [PFT_LANGUAGE §12](../../../recon/deep/reference/format/PFT_LANGUAGE.md), [PFT_MODULES §2.1](../../../recon/deep/reference/format/PFT_MODULES.md)).
- **UNIFOR-реестр:** каждый код (`3`,`A`,`S`,`6`,`+I`,…) — отдельный обработчик; неизвестный код → `*** Format error 99` (паритет ошибок, [PFT_LANGUAGE §15](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).

### 1.3 Фазовый план реализации

| Фаза | Содержание | Реализуемо «наполненно» |
|---|---|---|
| **Phase 1 — поля/подстановка** | Лексер/парсер базовых команд: `v<tag>`, `v<tag>^<sub>`, `v…^*`, фрагмент `v…*off.len`, `mfn(d)`; литералы `"…"`/`\|…\|`/`'…'` (условный/повторяющийся/безусловный, [PFT_LANGUAGE §5](../../../recon/deep/reference/format/PFT_LANGUAGE.md)); фиктивный вывод `d`/`n`; режимы `MP/MH/MD`+`U/L` с заменой разделителей подполей ([PFT_LANGUAGE §3](../../../recon/deep/reference/format/PFT_LANGUAGE.md)); размещение `/ # % Xn Cn`; HTML- и RTF-рендер IR. **Цель: brief-уровень показа** (`v910`, `v200^a`) | Покрывает короткое БО / списки |
| **Phase 2 — управляющие конструкции** | `IF…THEN…ELSE…FI` (вложенность, своё `FI`); логические выражения (`= <> < <= > >=`, `:` «содержит»; `NOT/AND/OR`); предикаты `p()`/`a()`; **повторяющиеся группы `(…)`** (поэкземплярная обработка, ограничения вложенности, `+`-литералы); числовые/строковые выражения (`val`,`rsum`,`rmin/rmax/ravr`, `f()`, `s()`); `ref()` (рекурсия, обращение к др. записи); `l()` (MFN термина в инверт. файле) → **brief.pft и KN.PFT исполняются целиком** ([PFT_MODULES §2.2/§2.4](../../../recon/deep/reference/format/PFT_MODULES.md)) | Полный показ моноиздания, ветвление по 920 |
| **Phase 3 — UNIFOR-библиотека + таблицы/печать** | Приоритетные UNIFOR-коды (§2), `&uf('6…')` модули-подпрограммы + шаблон-форматы `%N`, глоб. переменные `+1/+7`, постредактура `!`/`+F`; табличный движок (`.tab/.tbg`, §3); PDF/RTF-бланки (карточки KKK*, формуляры, КСУ, акты, наклейки/ШК) | Печать-паритет АРМов |

## 2. Покрытие UNIFOR (из 41 кода — приоритизация)

Полный каталог — [PFT_LANGUAGE §9.1](../../../recon/deep/reference/format/PFT_LANGUAGE.md). Стратегия исполнения: **бóльшую часть кодов реализуем в Python** (строковая/датная логика чистая), отдельные — как **SQL-функции** (когда нужен доступ к индексу/каталогу тенанта или агрегация).

### 2.1 Критичные для ws1–ws5 / reader (Phase 3, must-have)

| Код | Назначение | Где встречается (recon) | Реализация |
|---|---|---|---|
| `6` | Вложенный/шаблон-формат `&uf('6<имя>#<п>)` — **вся модульная архитектура показа** (68 модулей `osvex/ozboz/otitkn/o200*`…) | `KN.PFT` строки 4–6, `Pft.tre` | Python (рекурсивный вызов интерпретатора + подстановка `%N`) |
| `!` | Постредактура: очистка двойных разделителей | `KN.PFT:23`, `brief.pft:74` | Python (пост-проход по IR) |
| `+F` | Постредактура: очистка RTF-конструкций | `brief.pft:74` | Python (для HTML-ветки) |
| `A` / `P` | Заданное повторение поля `Av<tag>^<d>*off.len#occ` | `&unifor('Av920#1'):'SPEC'` (`brief.pft`) | Python |
| `3` | Дата/время (`3`,`30..39`,`3A/3B/3C`, названия месяцев) — даты в актах/КСУ/формулярах | бланки, заголовки отчётов | Python |
| `S` | Универсальный счётчик (`SN`, X=римские) — нумерация строк инв. книги/таблиц | `TABINW.PFT` (`&unifor('S1A')`), `.tbg` | Python (per-render state) |
| `+7` / `+1` | Глоб. переменные/списки (накопление, суммы, сортировка) — **итоги таблиц** | `.tab [Sum]`, `Form*.stf`, `TABSTATJ` | Python (рабочая область сеанса) |
| `+I` | Гиперссылка `+I/содерж/назв` (типы 0/1/2), **только RTF/web** | связи в карточке (I2) | Python (эмит ссылки в IR/HTML) |
| `+5` / `+4` | Элемент справочника `.mnu` по индексу / метка-значение поля повтор. группы | раскодировка статусов/мест | Python + лукап `.mnu` тенанта |
| `+6` | Статус записи (логически удалена/нет) | служебные плашки | Python |
| `D` / `7` | Расформатирование записи(ей) из **другой БД** REF «1→1»/«1→много» (`&unifor('DBOOK,/K=.../,v200')`) | связь Журнал↔Номер, источник аналитики | **SQL-функция** (поиск по термину в каталоге тенанта) + Python-рендер |
| `J` | Кол-во ссылок термина в инверт. файле | подсчёты | **SQL-функция** (индекс) |

> `l()`/`ref(l(...))` — формально функции ([PFT_LANGUAGE §8.1/§8.2](../../../recon/deep/reference/format/PFT_LANGUAGE.md)), но по нагрузке аналогичны `J`/`D`: требуют **обращения к инвертированному индексу тенанта** → реализуются связкой SQL-лукап + интерпретатор. Это критичный для `brief.pft` путь (подтягивание заглавия журнала по 933, [PFT_MODULES §2.4](../../../recon/deep/reference/format/PFT_MODULES.md)).

### 2.2 Полезные, не блокирующие (Phase 3, second wave)

`9` (снять кавычки), `E`/`F`/`G` (N первых слов / часть строки до символа), `T` (транслитерация кириллицы — для латинских ключей/URI), `Q`/`X` (нижний регистр / снять `<>`), `C` (контроль ISSN/ISBN — **переиспользуется ФЛК-движком A2**), `+3E/+3D` (URL-encode), `+9` (тех. выходы: имя/путь/расш. файла, ANSI-символ), `L` (окончание термина), `R` (random), `$` (MD5), `+0`/`+8` (вся запись / пользовательские DLL — `+8` маппится на плагин-капабилити, не на нативную DLL).

### 2.3 Откладываем / не реализуем как PFT

- `Z`, `M`, `+7`-мутации, `+1G/+1I` — **только в глобальной корректировке** ([PFT_LANGUAGE §9.1](../../../recon/deep/reference/format/PFT_LANGUAGE.md)); относятся к движку `.gbl` (**A3**, отдельный спек) — в `format` не нужны для показа/печати.
- `U`/`V`/`W` (кумуляция/декумуляция номеров журналов) — узкий периодический кейс; **defer** до периодики (связь ws3/комплектование).
- `+8` нативные DLL (`test_dll`) — **не портируем**; пользовательская логика — через плагин-API маркетплейса (C5), не через PFT.
- `B` (библиографическая свёртка), `++B` (длительность сеанса) — **defer**, узкие.

> **Принцип:** UNIFOR-реестр расширяем инкрементально; каждый код — отдельный обработчик с юнит-тестом (позитив+негатив) и эталоном из recon. Неизвестный/неподдержанный код в проде → `*** Format error 99` (диагностируемо), а не молчаливый сбой.

## 3. Конвейер печати

### 3.1 Что печатаем (из recon) → целевой формат

| Документ | Источник в ИРБИС | Целевой backend |
|---|---|---|
| **Показ карточки записи** (HTML) | `Kn_H/Jw_H/Asp_H…` (`Pftw_H.opt`) | **HTML** (a11y, в т.ч. `<IMG SRC="IRBIS:…">` ресурсы, штрих-код) |
| **Каталожные карточки** (комплект/основная/добавочные/ссылочные/оборот) | модули `!kkk*` (инд.) и `!!kk*` (пакетн.) через `!!kk*.tab` ([PFT_MODULES §1.4/§1.5](../../../recon/deep/reference/format/PFT_MODULES.md)) | **PDF** (точная вёрстка карточки 75×125 мм) |
| **Формуляры** | `!kkkfw`, `!!KKF`/`!!kk8f` | **PDF** |
| **Требования / книжные формуляры** | формы по книговыдаче (ws3) | **PDF** |
| **КСУ-бланки** (KSUN1-3SH), **инвентарная книга** | `TABINW`/`TABIUW`, `Form*.stf` ([PFT_MODULES §3/§4](../../../recon/deep/reference/format/PFT_MODULES.md)) | **PDF** (таблица) + **XLSX**-экспорт (опц.) |
| **Акты** (выбытия/списания) | `TABF3W/TABF3WI`, RTF-акт `!WKW` ([SPEC_ws1 §6](../SPEC_ws1_cataloger.md)) | **PDF** + **RTF/DOCX** (редактируемый бланк) |
| **Наклейки / штрих-коды** | `<IMG SRC="IRBIS:!!ШТРИХ_КОД!!">` ([PFT_LANGUAGE §14](../../../recon/deep/reference/format/PFT_LANGUAGE.md)) | **PDF** (раскладка на лист этикеток) |
| **Указатели / списки / таблицы** | `TABBUW/TABSUW/BRIEF_TAB`, `.tbg` | **PDF** + **XLSX**/**HTML** |

### 3.2 Выбор PDF-backend — **WeasyPrint** (recommend)

| Backend | За | Против | Вердикт |
|---|---|---|---|
| **WeasyPrint** | Чистый Python (в процессе сервиса, без headless-браузера/внешнего бинаря), отличная CSS Paged Media (`@page`, размеры/поля/колонтитулы — критично для карточек/бланков фикс. размера), детерминированный вывод → стабильный round-trip-тест | Нет JS (нам не нужен — IR уже готов) | ✅ **выбран** |
| wkhtmltopdf | Зрелый | **Заброшен** (deprecated), устаревший WebKit, слабый Paged Media | ✗ |
| Puppeteer/headless Chrome | Идеальный CSS/JS | Тяжёлый рантайм (Node+Chromium), недетерминизм, ресурсоёмкость на пакетной печати тысяч карточек | ✗ (резерв для сложного web-показа, если понадобится) |

**Печатный путь:** IR → **печатный HTML+CSS Paged Media** → WeasyPrint → PDF. Один IR → и экранный HTML, и печатный (разные CSS-профили). **RTF/DOCX-бланки** (акты/требования, где нужен редактируемый файл) — отдельный RTF-рендерер IR (RTF-команды из [PFT_LANGUAGE §14](../../../recon/deep/reference/format/PFT_LANGUAGE.md): `\par \li \ri \fs \b \i \ul \qc \page` маппятся напрямую из IR-узлов) либо `python-docx`.

### 3.3 Табличный движок (`.tab` / `.tbg`)

- **`.tbg` (декларативный, новый тип ИРБИС64)** — **первичный путь** для веб ([PFT_MODULES §3.4](../../../recon/deep/reference/format/PFT_MODULES.md)): парсим секцию `[TAB]` (ориентация, поля, `CELL_TAB_COL_COUNT`, по колонкам `CELL_TABn_1` ширина / `_2` заголовок / `_3` формат-выход PFT / `_4_5` сноски) → модель таблицы → рендер HTML-`<table>`/PDF/XLSX. Тело ячейки `_3` исполняется **тем же PFT-интерпретатором** на каждой записи.
- **`.tab` (старый ручной RTF)** — секции `[Tab]/[Header]/[FormatCode]/[Sum]/[Numeration]` ([PFT_MODULES §3.3](../../../recon/deep/reference/format/PFT_MODULES.md)); `.pft`-тело = одна строка таблицы (ячейки `\cellxNNNN…\cell`). Парсим как fallback/совместимость; `\cellxNNNN` (правая граница в твипах) → ширины колонок.
- **Итоги** (`[Sum]`/`.tbu`) — на **PostgreSQL-агрегатах**: вместо PFT-накопления `&uf('+7W…')`+`rsum` ([PFT_MODULES §4.2](../../../recon/deep/reference/format/PFT_MODULES.md)) считаем `SUM/COUNT/AVG` SQL-запросом по выборке MFN; PFT-`rsum` остаётся для построчных полей-повторений.
- **Сортировка** (`.srw`: `[KeyOptions]`, ключ строится PFT-форматом, [PFT_MODULES §3.6](../../../recon/deep/reference/format/PFT_MODULES.md)) → транслируется в `ORDER BY` по материализованному ключу/`f(val(...))`-числовой ключ.
- **Протокол-аналог:** старый `IC_print(Adbn,Atab,Ahead,…;Amfnlist;…)` ([PFT_MODULES §3.1](../../../recon/deep/reference/format/PFT_MODULES.md)) → наш `POST /api/print/table` (см. §4); статистика `IC_stat` ([PFT_MODULES §4.1](../../../recon/deep/reference/format/PFT_MODULES.md)) → `POST /api/print/stat`.

## 4. API-контракт

Все эндпоинты — tenant-scoped (JWT `tenant_id`, гранты), фиксируются в `openapi.yaml` (contract-test, как I1 #116).

| Метод | Назначение |
|---|---|
| `POST /api/record/{id}/format` | Форматировать **одну** запись. Тело: `{ "pft": "<inline>"  \| "profile": "Kn_H", "mode": "html\|pdf\|rtf\|docx\|plain", "params": {…} }`. Если ни `pft`, ни `profile` — авто-выбор по OPT (поле 920). Ответ: `text/html` \| `application/pdf` \| `application/rtf` (+ `X-Format-Render-Ms`, паритет genpft64). |
| `POST /api/format/preview` | Превью **inline-PFT** на записи (для редактора форматов / low-code D1) — синхронный рендер + ошибки `*** Format error N` со ссылкой на код ([PFT_LANGUAGE §15](../../../recon/deep/reference/format/PFT_LANGUAGE.md)). |
| `POST /api/print/cards` | Пакетная печать карточек: `{ "mfn_list" \| "query", "card_set": "!!KK_K\|!!KK_O\|…", "mode":"pdf" }` → PDF-комплект. |
| `POST /api/print/table` | Табличная форма (аналог `IC_print`): `{ "table": "TABINW\|@<имя>", "head":[…≤3], "mfn_list"\|"query", "sort":"@srw", "mode":"pdf\|xlsx\|html" }`. |
| `POST /api/print/stat` | Статформа (аналог `IC_stat`): `{ "stat_form":"Form1", "params":{…}, "range":{…}, "mode":"pdf\|xlsx" }`. |
| `POST /api/print/labels` | Наклейки/штрих-/QR-коды: `{ "mfn_list", "barcode_field":"910^h", "layout":"<этикетки>", "mode":"pdf" }`. |
| `GET /api/formats` | Доступные форматы показа/печати **per-tenant** (имя, тип записи 920, mode-ы, описание из `PFTW.MNU`-аналога). |
| `GET /api/print/forms` | Доступные табличные/печатные формы тенанта (аналог `Tabw.mnu`/`stf.mnu`). |

**Per-tenant кастомизация бланков:** шрифты, реквизиты (название/адрес/ИНН библиотеки, подписи, печать-логотип), размеры карточек/этикеток, ориентация — отдельный ресурс `print_profile` (см. §5), инжектится в CSS/RTF при рендере. Изоляция тенанта обязательна (тест).

## 5. Модель данных (хранение форматов/бланков per-tenant)

Схема `t_<slug>` (как ядро I1), плюс наследуемые системные дефолты в `control` (аналог `Deposit\` — общий набор, переопределяемый тенантом).

| Сущность | Поля (суть) | Аналог в ИРБИС |
|---|---|---|
| `format_template` | `name`, `kind`(display/module/flc/table-cell), `record_type_920`(nullable), `mode_targets`(html/pdf/rtf), `body_pft`(текст), `version`, `is_active`, `inherited_from`(system/tenant) | `.pft`-тела (`Deposit\`+БД), `Pft.tre` модули |
| `format_opt` | `selector_field`(920), `value_len`, `rules`(jsonb: значение→format_name, маска `+`), `default_format`, `target`(rtf/html) | `PFTW.OPT` / `Pftw_H.opt` ([PFT_LANGUAGE §12](../../../recon/deep/reference/format/PFT_LANGUAGE.md)) |
| `format_menu` | `name`→`title`, `group`, `mode`, видимость по гранту | `PFTW.MNU`, `Tabw.mnu`, `stf.mnu` |
| `table_form` | `name`, `kind`(tbg/tab), `definition`(jsonb колонок) или `tab_body`, `sort_key`(srw), `sum_spec` | `.tbg/.tab/.srw/.tbu` |
| `stat_form` | `name`, `group_params`(jsonb), `decode_mnu`, `header_pft`, `axes` | `.stf` ([PFT_MODULES §4.2](../../../recon/deep/reference/format/PFT_MODULES.md)) |

| `format_module_link` | связь `format_template`↔вызываемые `&uf('6…')` модули (граф зависимостей) | дерево `Pft.tre` |

**Версионирование:** `format_template`/`table_form`/`stat_form` версионируются (immutable-версии + указатель `is_active`); правка в редакторе (low-code D1) создаёт новую версию, старые рендеры воспроизводимы (важно для round-trip-теста и аудита). Связь с каталогом [PFT_MODULES](../../../recon/deep/reference/format/PFT_MODULES.md) — через `format_module_link` (граф 68 модулей + главные форматы), сидинг системных дефолтов из recon-зеркала (связь с **A5 сидинг**).

## 6. Критерии приёмки (AC по фазам)

**Phase 1 (поля/подстановка):**
- AC1: `v910`, `v200^a`, `v210^c.3` (фрагмент), `v200^*` (первое подполе), `mfn(6)` дают корректный вывод против записи-эталона.
- AC2: литералы — условный `"…"` (1 раз при наличии), повторяющийся `|…|` (на каждый экземпляр), безусловный `'…'` (всегда); префикс `|x|+` / суффикс `+|x|` ([PFT_LANGUAGE §5](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).
- AC3: фиктивный вывод `d461`/`n700,v700` — литерал по наличию/отсутствию поля ([PFT_LANGUAGE §6](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).
- AC4: режимы `mhl`/`mpl`/`mpu` — замена разделителей подполей (`^a`→`"; "`, `^b…`→`", "`) и регистр ([PFT_LANGUAGE §3](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).
- AC5: один IR → корректный **HTML** и **RTF**; `/ # %` применяются в RTF, игнорируются в HTML ([PFT_LANGUAGE §4](../../../recon/deep/reference/format/PFT_LANGUAGE.md)).

**Phase 2 (управляющие конструкции):**
- AC6: `IF v920='J' THEN … ELSE … FI` и `IF v920:'NJ'` ветвят по типу записи; вложенные IF с собственными `FI`.
- AC7: повторяющаяся группа `(… v910 …)` обрабатывает каждый экземпляр; `+`-литералы между экземплярами; ошибка вложенности → `*** Format error 2`.
- AC8: `val/rsum/rmin/rmax/ravr`, `f(val(v999),0,0)`, `s(…):'…'`, `p()/a()` работают (эталон — счётчик выдач в `KN.PFT:23`).
- AC9: `ref(l(|I=|v933), …)` подтягивает заглавие журнала по номеру (`brief.pft`) — обращение к индексу тенанта.
- AC10: **`brief.pft` (74 стр.) и `KN.PFT` (25 стр.) исполняются целиком** и совпадают с эталонным выводом (round-trip).

**Phase 3 (UNIFOR + печать):**
- AC11: `&uf('6osvex')`, `&uf('6author#700')` (шаблон с `%N`) — модульный показ; граф `Pft.tre` собирается.
- AC12: приоритетные UNIFOR (§2.1) проходят юнит-эталоны; `&unifor('!')`/`&unifor('+F')` чистят разделители/RTF.
- AC13: каталожная карточка (`!!KK_O`) и формуляр печатаются в **PDF** с корректной вёрсткой (размер карточки, реквизиты тенанта).
- AC14: инвентарная книга (`TABINW`) и КСУ-бланк рендерятся как таблица (PDF+XLSX); итоги `[Sum]` считаются SQL-агрегатом; сортировка по `.srw`.
- AC15: штрих-код (`910^h`) выводится в HTML-показе и на наклейке (PDF-раскладка этикеток).
- AC16: ошибка формата возвращает `*** Format error N` с верным кодом ([PFT_LANGUAGE §15](../../../recon/deep/reference/format/PFT_LANGUAGE.md)), а не 500.

**Сквозные (все фазы):**
- AC17: формат/бланк тенанта A недоступен тенанту B (тест изоляции); `print_profile` применяется per-tenant.
- AC18: время рендера экспонируется (`X-Format-Render-Ms`) — паритет `genpft64` (мс).

## 7. Критический путь

1. **Lexer/Parser/AST + Phase 1** (поля, литералы, режимы, размещение) + **IR** + **HTML-рендер** → короткое БО (`brief`-уровень) в показе. →
2. **RTF-рендер IR** (бланки) + **WeasyPrint PDF-рендер IR** (печать). →
3. **Phase 2**: `IF`/выражения/группы/`ref`/`l` → `KN.PFT`/`brief.pft` целиком; **OPT-диспетчер** по 920. →
4. **Phase 3 UNIFOR**: код `6` (модули `Pft.tre`) → приоритетные коды (§2.1) → постредактура. →
5. **Табличный движок** (`.tbg` приоритетно, `.tab` fallback) + **SQL-итоги** + `.srw`-сортировка → инв. книга / КСУ / указатели. →
6. **Печать-формы**: карточки `!!kk*`/формуляры/акты (PDF+RTF) + наклейки/ШК + статформы `.stf`. →
7. **API + модель данных + per-tenant бланки** (`print_profile`) + редактор форматов (связь low-code **D1**).

**Выходной критерий движка:** любая запись тенанта форматируется в HTML (показ, a11y) и печатается в PDF/RTF по выбранному формату (авто по 920 или явному `profile`/inline-PFT); **реальные `.pft` из recon** (`KN.PFT`, `brief.pft`, модули `Pft.tre`, формы `TABINW`/`!!KK_*`/`Form*`) исполняются с round-trip-совпадением; АРМы ws1/ws2/ws3 печатают карточки/формуляры/КСУ/акты/наклейки без десктоп-ИРБИС.

## 9. Риски

| Риск | Митигировать |
|---|---|
| **Семантика UNIFOR/`ref`/`l` не извлечена полностью из recon** (полный справочник — внешняя вики UNIFOR, [PFT_LANGUAGE §9](../../../recon/deep/reference/format/PFT_LANGUAGE.md)) | Реализуем по приоритету (§2), эталоны round-trip ловят расхождения; неподдержанный код → диагностируемый `Format error 99`, не молчаливый сбой |
| **Тела модулей лежат в `Deposit\`, а не в папке БД** (PFTPTH/путь поиска не подтверждён — `TODO recon #MOD-01`) | Loader реализует наследуемый путь поиска (system→tenant), сидинг из recon-зеркала Deposit (связь A5) |
| **Глоб. переменные / счётчики / постредактура — побочные эффекты** (не чистая функция) | Рабочая область — явный per-render контекст; интерпретатор однопоточный на рендер; повторяемость под тестом |
| **Точная вёрстка карточек/бланков (мм, твипы→CSS)** | CSS Paged Media + калибровочные эталоны размеров; `\cellxNNNN` (твипы) → мм по фикс. коэффициенту |
| **Производительность пакетной печати** (тысячи карточек) | Кэш AST, батч-рендер, WeasyPrint в пуле воркеров; тяжёлый Puppeteer не на горячем пути |
| **QR-коды не описаны в Приложении 4** (только `!!ШТРИХ_КОД!!`, `TODO recon #FMT-01`) | ШК — паритет сразу; QR — расширение `print_profile`/UNIFOR-кодом, уточнить по другим разделам докум. |
| **Расхождение HTML-показа и PDF-печати** из общего IR | Один IR, два CSS-профиля; визуальные snapshot-тесты обоих |
