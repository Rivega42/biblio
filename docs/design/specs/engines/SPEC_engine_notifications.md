# SPEC · Движок мультиканальных уведомлений (notification engine)

> Закрытие сквозного пробела **A6** ([DESIGN_COVERAGE §A](../../DESIGN_COVERAGE.md), эпик **#188**) — единый событийно-управляемый движок доставки уведомлений по каналам (email/SMS/push/in-app) с предпочтениями читателя, шаблонами на тенант, очередью, ретраями и идемпотентностью. Без него у читателя нет обратной связи о жизненном цикле заказа/выдачи: «бронь готова», «срок подходит/просрочка», «штраф», «продление подтверждено», «новые поступления по постоянному запросу» — всё это в ИРБИС держится на россыпи INI-флагов `ReservMail`/`OTKAZMAIL`/`RESERVSTATUSMAIL` и формате `reservmail.pft`, без очереди, ретраев, SMS/push и предпочтений.
>
> **Грунтовано на recon (читать как ground truth):**
> [DB_CIRCULATION §3/§5/§6](../../../recon/deep/reference/databases/DB_CIRCULATION.md) (цикл брони, статусы `reservstatus.mnu` 0–4, формат `reservmail.pft`, задолженность `40^F:'******'`, причины отказа `otkaz.mnu`) ·
> [ARM_LENDING `[Main]`](../../../recon/deep/reference/arms/ARM_LENDING.md) (INI-флаги `ReservMail=1`/`ReservMailPft=ReservMail`/`ReservMailAsk=1`/`RESERVSTATUSMAIL=1`/`OTKAZMAIL=1`; SMTP-параметры `MailHost/MailPort/MailFrom/MailFromAdress/MailSSL/MailOption`, `MailUser/MailPassword`; контакты читателя `32` e-mail / `17`/`18` телефоны) ·
> [CAPABILITY_MAP](../../../recon/deep/CAPABILITY_MAP.md) (карта функций книговыдачи/ИРИ).
>
> **Опорные спеки:** родитель — I3 (книговыдача) и I2 (читатель/discovery). Потребители-источники событий: [SPEC_ws3_circulation §3/§7](../SPEC_ws3_circulation.md) (бронь/очередь/уведомления, ячейки/постамат, авто-checkout) и [SPEC_reader_jirbis §4/§5/§6](../SPEC_reader_jirbis.md) (авто-письма о смене статуса заказа, постоянные запросы/ИРИ как saved-search alerts, ЛК). Транспорт событий — событийная шина [SPEC_I7_commercialization §B](../SPEC_I7_commercialization.md) (#107, outbox/CDC). Рендер шаблонов — **переиспользуется из A1** [SPEC_engine_pft.md](SPEC_engine_pft.md) (подстановка полей записи/читателя, `reservmail.pft` → наш шаблонизатор), **не дублируется**. Словари статусов/причин (`reservstatus.mnu`/`otkaz.mnu`/`ste.mnu`) — из [SPEC_seeding.md](SPEC_seeding.md). Согласия/тихие часы/журнал ПДн координируются с I6 (152-ФЗ).
>
> Платформа: **PostgreSQL + Python-бэкенд**, событийная шина (NATS/Kafka/RabbitMQ — выбор I7 #107). Движок — отдельный сервис `notify` (cross-cutting, потребляется ws3 и читателем; источники событий — любой сервис).

**DoD-гейт (как в I1/I2/A1):** код + автотест + контракт-тест (openapi) + **тест изоляции тенанта** (шаблоны/предпочтения/журнал per-tenant) + наблюдаемость (trace/`tenant_id`, метрики доставки: enqueued/sent/failed/bounced, latency) + a11y (in-app/email-HTML WCAG 2.2 AA) + **152-ФЗ-гейт** (opt-out соблюдён, ПДн минимизированы, тихие часы) + обновление доков. Размеры S/M/L.

## 0. Принципы и границы

- **Событие → решение → доставка — три развязанных слоя.** Источник события не знает про каналы; движок не знает про бизнес-логику книговыдачи. Контракт между ними — типизированное доменное событие (разд. 2). Это паритет ИРБИС-флагов (`ReservMail`/`RESERVSTATUSMAIL`/`OTKAZMAIL`), но вынесенный из АРМа в общий сервис.
- **Канал — абстракция.** Email/SMS/push/in-app реализуют единый интерфейс `Channel` (разд. 1); добавление провайдера РФ или нового канала = плагин, не релиз ядра. Fallback-цепочка декларативна (разд. 1.3).
- **Предпочтения читателя — first-class** ([SPEC_reader_jirbis §5](../SPEC_reader_jirbis.md)): какой канал на какое событие, opt-out по категории события (152-ФЗ). В ИРБИС этого нет — письма шлются всем, у кого заполнено поле 32; у нас — явное согласие и управление в ЛК.
- **Идемпотентность сквозная.** Одно доменное событие → ровно одна доставка на канал (дедуп по ключу события). Повтор события (ретрай шины, двойной CDC) не порождает второе письмо. Это критично: событийная шина гарантирует *at-least-once* ([SPEC_I7 §B](../SPEC_I7_commercialization.md)), движок обязан дать *effectively-once* на адресата.
- **Рендер — не наш.** Подстановка полей записи/читателя в шаблон (`v200^a`, `v40^e` срок, ФИО `v10/11/12`) делается **движком A1** ([SPEC_engine_pft.md](SPEC_engine_pft.md)) — `reservmail.pft` портируется как шаблон-тело, исполняется тем же интерпретатором. Движок уведомлений знает про каналы/очередь/предпочтения, **не про PFT**.
- **Минимизация ПДн в сообщении** (разд. 6): SMS не несёт ФИО/полное БО без необходимости; внешним провайдерам (SMS-агрегатор) уходит минимум; журнал хранит факт+метаданные, не тело с ПДн дольше политики.
- **Границы.** Движок **только доставляет уведомления**. Он не меняет статус брони/заказа (это ws3), не печатает формуляры/требования (это A1/печать), не считает штрафы (PAY, [SPEC_ws3 §5](../SPEC_ws3_circulation.md)) — он реагирует на их события. Постоянные запросы/ИРИ движок **рассылает**, но саму переиндексацию/поиск новинок делает поиск I2 (разд. 2.2, событие `new_arrivals_match`).

## 1. Каналы: абстракция, провайдеры, fallback

### 1.1 Интерфейс канала

Каждый канал реализует единый контракт; воркер доставки не знает деталей транспорта.

```python
class Channel(Protocol):
    name: str                      # 'email' | 'sms' | 'push' | 'inapp'
    def address_of(reader, prefs) -> str | None     # поле 32 e-mail / 17,18 телефон / push-token / reader_id
    def render(message: RenderedMessage) -> Payload  # из IR A1 → MIME / текст SMS / web-push JSON / in-app JSON
    def send(payload, address) -> DeliveryResult     # {status: accepted|rejected, provider_id, error?}
    def supports_receipts: bool                      # есть ли webhook о доставке/bounce (разд. 4.4)
```

| Канал | Транспорт | Адрес из recon | Провайдер | Примечания |
|---|---|---|---|---|
| **email (SMTP)** | SMTP/ESP | поле **32** (e-mail читателя, [DB_CIRCULATION §3](../../../recon/deep/reference/databases/DB_CIRCULATION.md)) | SMTP (параметры `MailHost/MailPort/MailFrom/MailSSL`, [ARM_LENDING `[Main]`](../../../recon/deep/reference/arms/ARM_LENDING.md)) или ESP с webhook-приёмом | HTML+plain (multipart), a11y-разметка; DKIM/SPF per-tenant домен |
| **SMS** | HTTP-API агрегатора | поля **17/18** (телефоны) | **абстракция «провайдер РФ»** (разд. 1.2) — SMS Aero / SMSC / МТС Exolve / Devino и т.п. | строгая минимизация ПДн; стоимость → дедуп/тихие часы важнее |
| **push** | Web Push (VAPID) + native (FCM/APNs) | push-subscription из PWA/мобильного ([SPEC_reader_jirbis §5](../SPEC_reader_jirbis.md), #179) | браузер/ОС | подписка хранится per-reader-per-device; отзыв при 410 Gone |
| **in-app** | внутренняя БД + SSE/WebSocket | `reader_id` (ЛК) | сами | «колокольчик» в ЛК; всегда доступен (не требует контакта/согласия на внешнюю связь), дефолтный безопасный канал |

> **Контакты — из recon.** RDR-поля: `32` e-mail (повт.), `17` тел. домашний, `18` тел. служебный (повт.) ([DB_CIRCULATION §3](../../../recon/deep/reference/databases/DB_CIRCULATION.md)). Отсутствие контакта на канал = канал недоступен → fallback (разд. 1.3). Нормализация телефона (РФ-формат) — на стороне SMS-канала.

### 1.2 Абстракция «SMS-провайдер РФ»

SMS-провайдеры РФ не унифицированы (разные API, баланс, отчёты о доставке, требования к отправителю/имени). Поэтому — **адаптер на провайдера** за единым интерфейсом `SmsProvider`:

```python
class SmsProvider(Protocol):
    def send(to_e164: str, text: str, sender_name: str | None) -> ProviderAck   # {provider_msg_id, queued_balance?}
    def parse_receipt(webhook_body) -> DeliveryReceipt | None                    # delivered | undelivered | expired
    def balance() -> Money | None                                               # для алертов «кончились SMS»
```

- Конфиг провайдера — per-tenant ресурс `channel_config` (разд. 5): API-ключ (в KMS/секретах, связь D2 PKI I6), имя отправителя, лимиты.
- Один провайдер активен на тенант + резервный (failover при `rejected`/недоступности API — это **внутриканальный** fallback, отличается от межканального разд. 1.3).
- Тест: контракт-тест провайдера на mock-API; интеграционный — в sandbox-режиме провайдера.

### 1.3 Fallback-цепочка

Доставка по событию пытается каналы по **упорядоченной цепочке**, пока один не примет (или пока не исчерпан критический набор). Цепочка вычисляется как пересечение трёх множеств:

```
chain = order( preferences(reader, event) ∩ event.allowed_channels ∩ channels_with_address(reader) )
```

- **Источник порядка:** предпочтения читателя (разд. 1.4); если читатель не настраивал — дефолт события (разд. 2, столбец «каналы по умолчанию»).
- **Семантика fallback по событию** (декларативно в каталоге, разд. 2):
  - `first-success` — слать по первому доступному каналу (типично: «бронь готова» — email, если нет → SMS, если нет → in-app). Успех на одном останавливает цепочку.
  - `all` — слать по всем выбранным каналам (редко; напр. критическая просрочка с блокировкой — и email, и SMS).
  - `escalate` — слать по основному; если канал вернул недоставку (bounce/undelivered) в течение T — эскалировать на следующий (напр. срок→просрочка).
- **in-app — терминальный fallback**: всегда в цепочке последним, т.к. не требует внешнего контакта и согласия на внешнюю связь; читатель увидит в ЛК даже при пустых контактах/отозванном email-согласии. Системные уведомления (разд. 2.2) могут идти только in-app.
- Цепочка и попытки фиксируются в журнале (разд. 5, `delivery_attempt`) для разбора «почему не дошло».

### 1.4 Предпочтения читателя

Матрица **событие × канал** + глобальные настройки, в ЛК ([SPEC_reader_jirbis §5](../SPEC_reader_jirbis.md)) и через API (разд. 5.2):

- На каждую **категорию события** (разд. 2): включён/выключен, упорядоченный список каналов.
- **opt-out (152-ФЗ):** транзакционные события жизненного цикла заказа (бронь готова, срок/просрочка, штраф, продление) — отключаемы по каналу, но in-app остаётся (читатель не может «не знать», что книга его ждёт, но может не получать SMS). **Маркетинговые/рекомендательные** (новинки по постоянному запросу, showcases) — **строгий opt-in**: по умолчанию выключены, включаются явным согласием.
- Язык сообщений (разд. 3.4), тихие часы (разд. 6.3) — тоже в предпочтениях.
- Дефолты при регистрации — из пресета тенанта (консервативные: только транзакционные, канал email/in-app), читатель донастраивает.

## 3. Шаблоны (templates)

### 3.1 Модель шаблона на тенант

Шаблон — версионируемый ресурс per-tenant (разд. 5), портируется из ИРБИС-форматов (`reservmail.pft` → `tpl.hold_ready`). Тело — **PFT-совместимое** (исполняется A1) **или** упрощённый плейсхолдер-синтаксис, транслируемый в A1-вызов.

```jsonc
{
  "id": "tpl.hold_ready",
  "event_type": "hold_ready",
  "source": "reservmail.pft",            // происхождение из recon
  "lang": "ru",                          // мультиязычность: один шаблон на (event, lang, channel?)
  "channel": null,                       // null = общий IR, рендерится под каждый канал; иначе канал-специфичный
  "subject": "Книга по вашей брони готова к выдаче",   // для email/push; null для SMS/in-app
  "body_pft": "Уважаем&uf('...')!\n«&unifor('6tpl_title')» ...",  // тело: A1 исполняет подстановку полей
  "sms_body": "Б-ка: бронь '{{title_short}}' готова. Срок {{due}}. Код {{pickup}}",  // SMS-минимум (без ФИО)
  "version": 3,
  "is_active": true
}
```

### 3.2 Переиспользование рендера A1 (не дублируем)

- Тело шаблона исполняется **движком A1** ([SPEC_engine_pft.md](SPEC_engine_pft.md)): подстановка полей записи каталога (заглавие `v200^a`, шифр `v903`) и читателя (ФИО `v10/11/12`, срок `v40^e`, контакты) — те же `v<tag>^<sub>`, `&unifor`, `IF…FI`, что в PFT. `reservmail.pft` ([DB_CIRCULATION §6](../../../recon/deep/reference/databases/DB_CIRCULATION.md)) портируется как тело шаблона **без переписывания логики подстановки** — движок уведомлений вызывает `A1.format(template_body, context)`.
- **Контекст рендера** — собирается движком уведомлений из `event.payload` (он знает, какие записи/поля приложить к событию) и передаётся в A1 как «запись» (JSONB поля/подполя ядра I1). A1 не знает про события — он форматирует данные.
- A1 отдаёт **IR-документ** (дерево «текст+разметка», [SPEC_engine_pft.md §1.1](SPEC_engine_pft.md)); каждый канал рендерит IR в свой payload: email — HTML+plain (a11y), SMS — plain c усечением, push/in-app — короткий заголовок+тело. Один IR → несколько каналов (как один IR → HTML/PDF/RTF в A1).
- Расшифровка кодов (`reservstatus`→текст, `otkaz`→причина) — через A1 `&unifor('+5'…)`/словарь тенанта ([SPEC_seeding.md](SPEC_seeding.md)), не дублируется в движке.

### 3.3 SMS-минимум

SMS не несёт полный рендер (стоимость + ПДн): отдельное поле `sms_body` с короткими плейсхолдерами, без ФИО и полного БО (разд. 6.1). Если `sms_body` пуст — деградация: усечённый plain из IR до лимита одного сегмента + ссылка в ЛК.

### 3.4 Мультиязычность

- Шаблон ключуется `(event_type, lang, channel?)`. Язык выбирается: предпочтение читателя → язык интерфейса ЛК → дефолт тенанта (рус). i18n как база ([SPEC_reader_jirbis §7](../SPEC_reader_jirbis.md)).
- Отсутствует перевод на язык читателя → fallback на язык тенанта по умолчанию (с логом «нет перевода шаблона»).

### 3.5 Агрегация в дайджест

Подписочные (`new_arrivals_match`, `iri_digest`) — **не по одному письму на запись**: события буферизуются в окне (per-tenant период, напр. сутки) и схлопываются в один дайджест `tpl.new_arrivals` со списком новинок (рендер каждого БО через A1, обёртка-список). Дедуп: одна запись не попадает в два дайджеста (разд. 4.3).

## 4. Конвейер доставки (delivery pipeline)

### 4.1 Поток

```
 источник (ws3/reader/поиск/система)
        │ публикует доменное событие
        ▼
 [событийная шина I7 #107]  ──(outbox/CDC, at-least-once)──▶  [notify: Ingest]
                                                                  │  · валидация event-схемы
                                                                  │  · вычислить dedup_key
                                                                  │  · idempotency-чек (разд. 4.3)
                                                                  ▼
                                                            [Resolver]
                                              · загрузить prefs читателя + контакты (32/17/18/push)
                                              · вычислить fallback-цепочку (разд. 1.3)
                                              · проверить opt-out / тихие часы / согласия (разд. 6)
                                              · выбрать шаблон (event,lang,channel)
                                                                  │
                                       (отложить до конца тихих часов / агрегировать в дайджест)
                                                                  ▼
                                            [Notification queue (PostgreSQL/брокер)]
                                              запись `notification` + `delivery_attempt` per канал
                                                                  ▼
                                              [Worker pool]  ──▶ A1.format (рендер IR)
                                                                  ──▶ Channel.send (email/SMS/push/inapp)
                                                                  ──▶ результат → журнал, метрики
                                                                  ▼
                                       success → stop (first-success) | retry/backoff (transient)
                                       reject  → next channel (fallback) | dead-letter (исчерпано)
                                                                  ▲
                                            [Webhook receiver] ◀── провайдер (delivered/bounce/undelivered)
                                                                  обновляет статус, триггерит escalate
```

### 4.2 Очередь и воркер

- **Очередь** — таблица `notification` + `delivery_attempt` в PostgreSQL (надёжность, транзакционность с журналом, `SELECT … FOR UPDATE SKIP LOCKED` как пул задач) **или** брокер шины I7; выбор — конфигурацией. Для старта — PostgreSQL-очередь (одна система хранения, проще идемпотентность и аудит), вынос в брокер при росте нагрузки.
- **Воркер** — пул, забирает задачи батчами, рендерит (A1), шлёт через канал, пишет результат. Однопоточная обработка одной задачи; параллелизм между задачами. Воркеры stateless, горизонтально масштабируются.
- **Планировщик (отложенные)** — задачи с `not_before` (тихие часы, дайджесты, due_soon заранее посчитанные) — воркер берёт по `not_before <= now()`.

### 4.3 Идемпотентность и дедуп

- **`dedup_key`** на событие: детерминирован из (`tenant_id`, `event_type`, бизнес-ключ). Примеры: `hold_ready:{tenant}:{rqst_mfn}:{reservstatus=1}`; `overdue:{tenant}:{reader}:{shifr}:{ekz}:{due_date}`; `due_soon:{...}:{N_days}`. Уникальный constraint на `notification(dedup_key)` → второе событие с тем же ключом не создаёт вторую доставку (паритет «одно письмо на одну готовую бронь», даже при двойной публикации шины).
- **Уровень канала:** `delivery_attempt(notification_id, channel)` уникален → ретрай не шлёт второе письмо тем же каналом.
- **Дайджест-дедуп:** запись/новинка помечается «включена в дайджест vX» → не повторяется в следующем (разд. 3.5).
- **Эффект:** шина даёт *at-least-once* ([SPEC_I7 §B AC1](../SPEC_I7_commercialization.md)), движок — *effectively-once на адресата*.

### 4.4 Ретраи, backoff, bounce/недоставка

- **Transient-ошибки** (SMTP 4xx, таймаут провайдера, 429): ретрай с экспоненциальным backoff + jitter (напр. 1м/5м/30м/2ч/6ч), предел попыток; исчерпано → `dead-letter` + `staff_alert`.
- **Permanent-ошибки** (SMTP 5xx hard bounce, неверный номер, push 410 Gone): **не ретраить** на этом канале → fallback на следующий канал цепочки; пометить контакт/подписку проблемным (push-410 → удалить subscription; email hard-bounce → флаг `email_invalid`, уведомить читателя/staff пере­проверить контакт).
- **Bounce/receipt-обработка (webhook, разд. 5.3):** провайдер шлёт статус доставки → обновляем `delivery_attempt.status` (delivered/undelivered/expired) → для `escalate`-событий недоставка триггерит следующий канал; для `first-success` уже доставленное по другому каналу — игнор.
- **Идемпотентность ретрая:** ретрай — та же задача (тот же `attempt_id`), не новая.

### 4.5 Планировщик: крон-скан vs по-индексации vs broadcast

- **Крон-скан** (`due_soon`/`overdue`/`account_lifecycle`): периодическая джоба сканирует активные выдачи (RDR.40: `40^E` срок, `40^F='******'` на руках, [DB_CIRCULATION §3](../../../recon/deep/reference/databases/DB_CIRCULATION.md)) и эмитит события для попадающих в окно. Идемпотентно за счёт `dedup_key` с датой (повторный скан в тот же день не дублирует). N дней до срока и эскалация просрочки — параметры тенанта.
- **По-индексации** (`new_arrivals_match`): поиск I2 при индексации новинки матчит сохранённые запросы → событие (разд. 2.2). Не крон — реактивно на пополнение каталога.
- **Событийно** (транзакционные `hold_ready`/`refused`/`renewal`/`fine`): источник публикует немедленно при изменении состояния (через outbox/CDC, чтобы событие было атомарно с бизнес-транзакцией, без потери).
- **Broadcast** (`system_broadcast`): админ ws5 шлёт одно сообщение когорте (фильтр по категории/группе читателей) → фан-аут в per-reader-задачи (с учётом prefs/тихих часов каждого).

## 6. Приватность и комплаенс (152-ФЗ)

> ⚠ RDR содержит ПДн ([DB_CIRCULATION](../../../recon/deep/reference/databases/DB_CIRCULATION.md)). Координация с I6 (B4 152-ФЗ/ФСТЭК, [DESIGN_COVERAGE §B4](../../DESIGN_COVERAGE.md)).

### 6.1 Минимизация ПДн в сообщениях

- **SMS** — отдельное короткое тело `sms_body` (разд. 3.3) без ФИО/полного БО: «Б-ка: ваша бронь готова, срок до DD.MM, подробности в ЛК». Полные данные — за авторизацией в ЛК, не в открытом канале.
- **Email** может нести БО/ФИО (адресная, защищённая доставка), но не несёт лишнего (паспорт `14`, адрес `13`, учебный профиль `90` — никогда).
- **Код выдачи** (`hold_pickup_code`) — чувствителен: предпочтительно SMS/push на верифицированный контакт; срок жизни кода ограничен; в журнале — хэш, не сам код.
- **Внешним провайдерам** (SMS-агрегатор, ESP) уходит минимум: номер/адрес + готовый текст; провайдер не получает структуру записи. Выбор провайдера РФ (локализация ПДн, разд. 1.2).

### 6.2 Согласия (opt-in/opt-out)

- Маркетинг/подписки (`new_arrivals`/`showcase`/`recommendation`) — **строгий opt-in**, по умолчанию выкл.; включение — явное согласие, фиксируется в `consent_log` (кто/когда/категория).
- Транзакционные — отключаемы по внешним каналам (SMS/email), in-app неотключаем (читатель должен иметь возможность узнать о готовой брони/просрочке).
- Каждое сообщение внешнего канала несёт способ отписки (email — ссылка unsubscribe → `PUT /api/me/notification-prefs`; SMS — инструкция/стоп-слово в рамках провайдера). Право на отзыв согласия — мгновенное (отзыв сразу подавляет будущие отправки категории).

### 6.3 Тихие часы

- Per-tenant дефолт + per-reader override: окно (напр. 22:00–08:00 по таймзоне читателя/библиотеки), в которое **внешние** каналы (SMS/push) откладываются (`not_before` = конец окна, разд. 4.2). Email/in-app — по политике (email обычно разрешён, не звонит). Критические (`overdue` с блокировкой, `account_security`) — могут игнорировать тихие часы по флагу события.

### 6.4 Журнал и хранение

- `consent_log` — кто/когда дал/отозвал согласие (доказуемость 152-ФЗ).
- Журнал отправок (`notification`/`delivery_attempt`) — факт, канал, статус, метаданные; **тело с ПДн** не хранится дольше операционной необходимости (политика ретеншена per-tenant); адреса — `address_hash`. Право на забвение (I6 B4): удаление читателя каскадит/анонимизирует его уведомления и предпочтения.
- Доступ к журналу — по гранту staff, под журналом доступа (152-ФЗ, как в ws3/reader).

## 7. Критерии приёмки (AC)

- **AC1 (бронь готова).** Переход брони в `reservstatus=1` → событие `hold_ready` → доставка по fallback-цепочке предпочтений; шаблон — порт `reservmail.pft` через A1; адрес из поля 32 (email)/17-18 (SMS). Повторная публикация события (двойной CDC) **не шлёт второе письмо** (дедуп по `dedup_key`).
- **AC2 (срок/просрочка).** Крон-скан по RDR.40 (`40^E` срок, `40^F='******'` на руках) эмитит `due_soon` за N дней и `overdue` после срока; повторный скан в тот же день не дублирует; `overdue` эскалирует на следующий канал при недоставке (`escalate`).
- **AC3 (отказ/штраф/продление).** RQST.44 (отказ, причина из `otkaz.mnu`) → `request_refused`; проводка PAY → `fine_charged/paid`; `40^L` → `renewal_confirmed` — каждое с корректным шаблоном и каналами.
- **AC4 (постоянные запросы/ИРИ).** Индексация новинки, матчащей сохранённый запрос читателя (RDR `140`/`IRI=`), → `new_arrivals_match`; события агрегируются в **один дайджест** за окно; рассылка только при **opt-in**; одна новинка не попадает в два дайджеста.
- **AC5 (каналы и fallback).** При отсутствии e-mail (поле 32 пусто) доставка уходит на SMS; при отсутствии всех внешних контактов — in-app (терминальный fallback) всегда срабатывает. `first-success` останавливается на первом принявшем; `all` шлёт по всем.
- **AC6 (предпочтения/opt-out).** `GET/PUT /api/me/notification-prefs` управляет каналами на событие; выключенный внешний канал не получает сообщение, in-app для транзакционных остаётся; маркетинг по умолчанию выключен (opt-in). Изменение пишется в `consent_log` + версионируется.
- **AC7 (идемпотентность/ретраи).** Transient-ошибка ретраится с backoff; hard-bounce/410 не ретраится, идёт fallback и помечает контакт; ретрай не порождает дубль (тот же `attempt_id`); исчерпание → dead-letter + `staff_alert`.
- **AC8 (webhook провайдера).** `POST /api/webhooks/{provider}` с валидной подписью обновляет статус доставки (delivered/bounce/undelivered); повтор webhook не двоит статус; bounce триггерит fallback/флаг контакта.
- **AC9 (приватность/152-ФЗ).** SMS не содержит ФИО/полного БО (только `sms_body`); внешнему провайдеру уходит минимум; тихие часы откладывают SMS/push; отзыв согласия мгновенно подавляет категорию; журнал хранит `address_hash`, не открытый адрес; удаление читателя анонимизирует его уведомления.
- **AC10 (мультиязычность).** Шаблон выбирается по языку читателя; при отсутствии перевода — fallback на язык тенанта с логом.
- **AC11 (изоляция тенанта).** Шаблоны/предпочтения/журнал/конфиг каналов тенанта A недоступны тенанту B (тест в CI — DoD-гейт); провайдер/секреты per-tenant.
- **AC12 (наблюдаемость).** Метрики enqueued/sent/delivered/failed/bounced + latency по каналу/событию экспонируются; trace несёт `tenant_id`; `staff_alert` при низком балансе SMS/массовых провалах.

## 8. Критический путь

1. **Контракт события + `dedup_key` + Ingest** (приём из шины и `POST /api/notify`) + **idempotency-чек** → ничего не двоится. →
2. **Resolver** (prefs + контакты 32/17/18 + fallback-цепочка) + **in-app канал** (терминальный, без внешних зависимостей) + **email-канал (SMTP)** → базовая доставка `hold_ready` (порт `reservmail.pft` через **A1**). →
3. **Очередь + воркер + ретраи/backoff + dead-letter** + журнал (`notification`/`delivery_attempt`). →
4. **Предпочтения читателя** (`GET/PUT /api/me/notification-prefs`) + opt-out/согласия + **тихие часы** (152-ФЗ-гейт). →
5. **SMS-канал** (абстракция провайдера РФ) + **push** (web/native) + **webhook-приём** (bounce/receipt → fallback/escalate). →
6. **Планировщик**: крон-скан `due_soon`/`overdue` (RDR.40) + событийные `refused`/`renewal`/`fine`. →
7. **Подписочные**: `new_arrivals_match` по индексации (поиск I2) + **дайджест-агрегация** + мультиязычность + шаблоны-редактор (ws5) + `system_broadcast`.

**Выходной критерий движка A6:** любое доменное событие книговыдачи/discovery (бронь готова, срок/просрочка, штраф, продление, отказ, новинки по постоянному запросу, системные) доставляется читателю по выбранным каналам (email/SMS/push/in-app) с fallback-цепочкой, идемпотентно (одно событие → одна доставка на адресата), с ретраями/обработкой bounce, по предпочтениям и согласиям читателя (152-ФЗ: opt-out, тихие часы, минимизация ПДн), с шаблонами на тенант (порт `reservmail.pft` через рендер **A1**, мультиязычно), журналом отправок и вебхуками провайдеров — заменяя россыпь INI-флагов `ReservMail`/`RESERVSTATUSMAIL`/`OTKAZMAIL` единым сервисом; ws3 ([SPEC_ws3_circulation.md](../SPEC_ws3_circulation.md)) и читатель ([SPEC_reader_jirbis.md](../SPEC_reader_jirbis.md)) получают авто-уведомления без десктоп-ИРБИС.

## 10. Риски

| Риск | Митигация |
|---|---|
| **Точная семантика `reservmail.pft`/триггеров не извлечена из recon** (тело формата не разобрано построчно; флаги `ReservMailAsk`/`RESERVSTATUSMAIL` — поведение подтверждено по INI, не по логике) | Шаблон портируется через **A1** (исполняет реальное тело как есть); триггеры завязаны на наблюдаемые переходы `reservstatus`/RQST.44/RDR.40, не на внутренний код АРМа; уточнить на боевом конфиге тенанта (`TODO recon #NOTIF-01`) |
| **Событийная шина (#107) ещё проектируется** ([SPEC_I7 §B](../SPEC_I7_commercialization.md)) | Старт на PostgreSQL-очереди + `POST /api/notify` (синхронный путь) + outbox-таблица у источников; миграция на брокер за интерфейсом Ingest без переписывания движка |
| **SMS-провайдеры РФ не унифицированы** (API/доставка/баланс/имя отправителя) | Адаптер на провайдера (`SmsProvider`, разд. 1.2), контракт-тест на mock, sandbox-режим, резервный провайдер; деградация на email/in-app при недоступности |
| **at-least-once шины → дубли** | Сквозной `dedup_key` (разд. 4.3) + UNIQUE-constraints; *effectively-once на адресата* |
| **Крон-скан просрочек на больших фондах — нагрузка** | Инкрементальный скан по индексу срока (`40^E`/`DOLG=`), батчи, идемпотентность по дате; вынос в фоновую джобу |
| **Стоимость/спам SMS** | Тихие часы, дедуп, дайджесты для подписочных, минимизация, алерт баланса; SMS только для важных событий по умолчанию |
| **152-ФЗ: согласия/локализация ПДн/право на забвение** | Координация с I6 (B4); `consent_log`, провайдер РФ, анонимизация при удалении, `address_hash` в журнале |
| **Контакты в RDR неактуальны/пусты** (email-bounce, неверный номер) | Hard-bounce помечает контакт + `staff_alert` «проверить контакт»; fallback на in-app; подсказка читателю обновить контакты в ЛК |

## Сводка

Движок A6 — **событийный мультиканальный диспетчер уведомлений**: доменное событие (из ws3/reader/поиска/системы через шину I7 #107 или `POST /api/notify`) проходит Ingest (дедуп по `dedup_key`) → Resolver (предпочтения читателя + контакты RDR 32/17/18/push + fallback-цепочка + opt-out/тихие часы/согласия 152-ФЗ + выбор шаблона) → очередь (PostgreSQL/брокер) → воркер (рендер шаблона **движком A1**, порт `reservmail.pft`; отправка через канал) → журнал/метрики/webhook-приём (bounce/receipt → ретрай/escalate). Каналы — абстракция (email-SMTP / SMS-провайдер-РФ / web-push+native / in-app-терминальный) с декларативным fallback (`first-success`/`all`/`escalate`). Каталог событий: бронь готова (`reservmail`), срок/просрочка (крон-скан RDR.40 `DOLG=`), штраф/оплата, продление, отказ (`otkaz.mnu`), новинки по постоянному запросу (`new_arrivals_match` по индексации, дайджест, opt-in), системные. Идемпотентность *at-least-once → effectively-once*, ретраи с backoff, мультиязычные шаблоны на тенант, изоляция тенанта, минимизация ПДн (152-ФЗ) — заменяет INI-флаги `ReservMail`/`RESERVSTATUSMAIL`/`OTKAZMAIL` единым сервисом.
