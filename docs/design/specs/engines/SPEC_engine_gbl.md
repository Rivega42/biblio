# SPEC · Движок A3 — Исполнитель глобальной (пакетной) корректировки `.gbl`

> Закрытие архитектурного пробела **A3** ([DESIGN_COVERAGE](../../DESIGN_COVERAGE.md) §A: «синтаксис описан, нет парсера/превью/отката/переменных»), эпик **#188** «Доглубление дизайна до реализуемости». Цель — портировать серверную функцию ИРБИС `IC_gbl` в веб-движок поверх PostgreSQL-ядра, с **превью/откатом/переменными**, как сквозной сервис для АРМов Каталогизатор/Комплектатор/Книговыдача/Администратор.
>
> **Грунтовано на recon:** [GLOBAL_CORRECTION](../../../recon/deep/reference/format/GLOBAL_CORRECTION.md) (авторитетный — 22 оператора, ФОРМАТ 1/2, режим `F`, переменные/регистры `&uf('+7…')`, счётчики `&uf('++C…')`, отбор записей, сигнатура `IC_gbl`), [PFT_LANGUAGE](../../../recon/deep/reference/format/PFT_LANGUAGE.md) (UNIFOR/`&uf`), [PROTOCOL_REFERENCE §3.7](../../../recon/deep/reference/protocol/PROTOCOL_REFERENCE.md). Родитель — [SPEC_I5](../SPEC_I5_staff_arm_parity.md); тонкая ссылка — [SPEC_ws1 §7](../SPEC_ws1_cataloger.md). Сквозные зависимости: **A1** (движок PFT/печати) и **I1 #101** (аудит/Identity).
>
> **DoD-гейт (как в I1/I5):** код + автотест + контракт-тест (openapi) + тест изоляции тенанта + наблюдаемость (trace/`tenant_id`/`job_id`) + обновление openapi/доков. Размеры S ≤2 дн · M 3–5 дн · L 1–2 нед.

## 0. Границы движка и принципы

- **Что делает:** принимает задание `.gbl` (тело или `@имя`) + выборку записей → применяет операторы последовательно к каждой записи → выдаёт **diff (превью)** либо **применяет** с записью обратимого журнала.
- **Что НЕ делает (делегирует):**
  - вычисление ФОРМАТА 1/2 и любых `&uf(...)`/PFT-выражений → **движок A1 (PFT)**, переиспользуем, не дублируем (см. §2);
  - ФЛК при удалении/сохранении (`delflc`/`dbnflc`) → **движок A2 (ФЛК)** (вызывается на границе применения, §4.4);
  - авторизацию/гранты/аудит-хранилище → **I1 #101** (§6, §5.4).
- **Принцип паритета:** семантика операторов и порядок исполнения — **бит-в-бит** по [GLOBAL_CORRECTION], включая «каждый оператор видит запись в состоянии после предыдущих» (`txt:5920`) и режим `F` (строка ФОРМАТА 1 ↔ повторение поля).
- **Принцип обратимости:** **любое** применение записывает old→new по каждой записи; откат восстанавливает состояние без повторного прогона задания (журнал — источник истины, §4.5).
- **Модель записи:** JSONB поля/подполя ядра I1 (как в [SPEC_ws1 §9]); адресация `<метка>[^<подполе>]` + индекс повторения отображается в путь JSONB.

## 1. Парсер и AST задания `.gbl`

### 1.1 Лексика и структура файла
Источник: [GLOBAL_CORRECTION §2]. Файл — текст, кодировка CP1251 (на входе перекодируем в UTF-8 при импорте/загрузке `@имя` из конфигурации тенанта). Построчная грамматика (строки значимы — оператор занимает фиксированное число строк):

```
gbl          := param_header  operator_group+
param_header := INT(0..9)  param_pair{N}
param_pair   := value_or_mnu_or_wss  caption          // value | имя .MNU/.WSS | пусто; 2-я строка — подсказка
operator_group := op_name  op_line*                   // число op_line зависит от оператора
```

- **Строка 1** — число параметров `%1…%9` (0 = без параметров). Параметры подставляются в тело **до** парсинга операторов (`txt:5918`, `txt:6173-6187`); значение берётся из формы запуска / `.MNU` / `.WSS` (в вебе — из payload API, §5).
- **Заполнитель** незанятых строк — `XXXXXXXXXXXXXXXXXXX` **или** пустая строка → нормализуется в `∅` (`txt:5934`; открытый `TODO #GBL-05` — поведение пустая vs. заполненная фиксируем как эквивалент при разборе).
- **Комментарии** `//` (до 4 строк текста после), `/*…` — отбрасываются на этапе лексики, но сохраняются в AST как `comment` для воспроизведения протокола.

### 1.2 Операторы → число строк (таблица разбора)
Парсер — табличный: имя оператора задаёт «арность» в строках. Все 22 имени из recon:

| Оператор | Строк | Доп. строки (роль) | Класс |
|---|---|---|---|
| `ADD` `REP` `CHA` `CHAC` `DEL` | 5 | метка[^подп] · повторение · ФОРМАТ 1 · ФОРМАТ 2 | поле/подполе |
| `DELR` `UNDEL` `ALL` `EMPTY` `END` `FI` | 1 | — | запись/поток |
| `IF` `UNTIL` `UNDOR` `PUTLOG` `PUTFLD`† `NEWREC`‡ | 2 | формат (условие / N-шагов / строка протокола / условие) | поток/протокол |
| `NEWMFN` | 2 | формат → имя БД (`'*'`=текущая) | др. БД |
| `CORREC` | 4–5 | формат→БД · формат→поле 1001 · формат→термины (`$`=усеч.) · [число записей] | др. записи |
| `REPEAT` | 1 | (тело до `UNTIL`) | цикл |
| `DEFFLD`† `GETFLD`† | 5 / 2 | метка (+ заполнители) / метка | модельные поля |
| `//` | 1–4 | текст | комментарий |

† `PUTFLD`, `DEFFLD`, `GETFLD` — устар./недокумент. (recon `TODO #GBL-02/03`, `RELEASE_OVERALL txt:3405`): парсим для совместимости старых заданий; `PUTFLD`≡`PUTLOG`. ‡ `NEWREC` — недокумент., семантика ≈ `NEWMFN`+условие (recon `TODO #GBL-01`); парсим, помечаем `experimental`.

### 1.3 AST (узлы)
```
Program{ params:[ParamDef], body:[Op] }
ParamDef{ idx, source: literal|mnu|wss|prompt, caption }
Op = FieldOp{ kind:ADD|REP|CHA|CHAC|DEL, tag, subfield?, occ:Occ, fmt1:PftRef, fmt2:PftRef }
   | RecOp{ kind:DELR|UNDEL|ALL|EMPTY }
   | NewmfnOp{ dbFmt:PftRef, body:[Op] }          // до END
   | CorrecOp{ dbFmt, keyFmt(→1001), termsFmt, limit?, body:[Op] }
   | NewrecOp{ condFmt:PftRef, body:[Op] }         // experimental
   | IfBlock{ condFmt:PftRef, body:[Op] }          // IF…FI
   | RepeatBlock{ body:[Op], untilFmt:PftRef }     // REPEAT…UNTIL
   | UndorOp{ stepsFmt:PftRef }                    // N | '*' | ∅
   | PutlogOp{ lineFmt:PftRef }
   | DefFld{ tag } | GetFld{ tag }                 // experimental
Occ = ALL('*') | NTH(int) | LAST('L') | LAST_MINUS('L-'int) | BY_FORMAT('F')
```
`PftRef` — **скомпилированная ссылка** на формат, исполняемая движком A1 (не строка). Блоки `IF/REPEAT/NEWMFN/CORREC/NEWREC` парсятся в дерево (вложенность — массово в `autoin.gbl`), `FI/UNTIL/END` — закрывающие маркеры, в AST не узлы.

### 1.4 Валидация задания (статическая, до запуска)
- незакрытые `IF`/`REPEAT`/`NEWMFN`/`CORREC` → ошибка парсинга с номером строки;
- неизвестное имя оператора → ошибка (не «молча пропустить»);
- ссылки на `%n` сверх объявленного числа параметров → ошибка;
- предупреждения: `experimental`-операторы, `PUTFLD` (рекомендовать `PUTLOG`).
- **`POST /api/batch-correct/validate`** возвращает AST-резюме + список ошибок/предупреждений **без** выборки записей (быстрый «компилятор» для редактора задания, low-code D1).

## 2. Вычисление форматов через движок A1 (PFT) — переиспользование

ФОРМАТ 1/2 и все `&uf(...)` — **обычные PFT-форматы, расформатируемые на текущей записи** (`txt:5916-5917`). Движок `.gbl` **не интерпретирует PFT сам** — он вызывает A1 с контекстом исполнения.

**Контракт вызова A1 (расширение интерфейса A1 «evaluate(format, record, ctx)»):**
```
pft.eval(pftRef, record, ctx) -> string            // обычный одностр. результат
pft.evalLines(pftRef, record, ctx) -> string[]     // многостр. (групповой (...)/ ) — для режима F и ADD по полю
```
`ctx` (контекст пакетной корректировки — то, чего нет в обычном показе):
- **`%1…%9`** — параметры задания;
- **переменные/регистры `&uf('+7…')`** — модель хранения §3 (A1 читает/пишет через колбэк `ctx.vars`);
- **глобальные счётчики `&uf('++C<idx>#<val>')`** (БД COUNT, `txt:7087-7097`) — через `ctx.counters` (персистентный, per-tenant, §3.3);
- **доступ к другим БД на лету** — `&uf('J<db>,…')`, `&uf('D<db>,…')`, `&uf('7,?…?,fmt')` (`SumEkzToSZ.gbl`, `Move691.gbl`) — A1 резолвит БД **в рамках тенанта** (§6), поиск идёт по PostgreSQL-индексу (§3 ниже), не по инвертированному файлу;
- **спец-выходы только глобальной корректировки:** `&unifor('Z')` (размножение экземпляров, `txt:4810`), `&uf('4…')` (форматирование предыдущей копии записи — для `UNDOR`, `txt:6016`) — A1 получает доступ к **истории копий** записи из журнала §4.5 через `ctx.history`.

**Требование к A1 (фиксируем как зависимость):** A1 обязан поддержать инъекцию `ctx.vars/counters/history` и БД-кросс-доступ под тенантом. Если A1 этого не умеет — это блокер A3 (см. открытые вопросы).

## 3. Отбор записей и состояние исполнения

### 3.1 Отбор фрагмента БД → список MFN
Три режима (`txt:6783-6786`, сигнатура `IC_gbl`: `Asexp`/`Amin`/`Amax`/`Aseq`/`Amfnlist`):
1. **Диапазон MFN** `mfnRange:{from,to}` → `Amin/Amax`.
2. **Поисковый запрос** `query` (синтаксис ИРБИС или прямой/последовательный `Asexp`+`Aseq`) → выполняется **поисковым сервисом I2 на PostgreSQL** (FTS/материализованные индексы каталога тенанта), возвращает список MFN. Это ключевое отличие от ИРБИС: **нет инвертированного файла**, отбор = SQL-запрос ядра.
3. **Комбинация** query ∩ mfnRange (последовательный поиск по результату прямого).
4. Явный **список MFN** `mfnList` (диапазон `0/min/max` · список `N/mfn…` · отрицательный `-N/…` = все кроме — `txt:936-952`).

Результат отбора фиксируется как **снимок списка MFN на момент старта job** (записи, добавленные/удалённые конкурентно во время выполнения, в выборку не попадают — детерминизм превью↔применения).

### 3.2 Переменные и регистры (модель хранения)
Регистры `&uf('+7…')` ИРБИС — глобальные строковые/числовые ячейки, живущие **в пределах одного прогона задания** (буферы цикла, накопители). Модель:
- **`ExecVars`** — in-memory словарь `name → string` на время job; операции из recon (`+7W` запись, `+7R`/`Gnnn` чтение, `+7U` добавление, `+7S` сортировка/арифметика, `+7G` исключение неоригинальных — `autoin.gbl`, `+97` нормализация регистра).
- **Область видимости:** по умолчанию — **на всю выборку** (накопители вроде «список MFN» в `SumEkzToSZ.gbl` живут между записями). Сброс — явный (`+7W` с новым значением) либо по флагу задания. Документируем: переменные **не** обнуляются между записями, если задание само не обнуляет (паритет с ИРБИС).
- В превью-режиме переменные вычисляются так же, как в применении (детерминизм), но без побочных эффектов на БД.

### 3.3 Глобальные счётчики `&uf('++C…')`
Персистентная БД COUNT (`txt:7087-7097`) → таблица `gbl_counter(tenant_id, idx, value)` в схеме тенанта. Чтение/инкремент атомарны (`SELECT … FOR UPDATE`/upsert). **В превью-режиме счётчики НЕ инкрементируются** (read-only проекция текущего значения) — иначе dry-run имел бы побочный эффект. Это явное расхождение с буквальной семантикой ИРБИС в пользу безопасного превью; фиксируется в AC.

### 3.4 Конвейер исполнения (на одну запись)
```
for mfn in selection_snapshot:
  rec   = load(mfn)                 # JSONB, под блокировкой на запись (как edit-lock ws1)
  before = deepcopy(rec)
  for op in program.body:
     apply(op, rec, ctx)           # последовательно; rec мутирует, op видит результат предыдущих
  diff = makeDiff(before, rec)     # пофайл./пополевой
  if preview: collect(diff)
  else:        delflc?/dbnflc? -> persist(rec) + journal(before, rec)
```
`apply` по классам операторов:
- **FieldOp**: разворачивает `Occ` в множество повторений; для `BY_FORMAT('F')` — `pft.evalLines(fmt1)` → i-я строка управляет i-м повторением (лишние повторения/строки игнор., `txt:5947-5955`); `REP` с пустой строкой → удаление поля; `CHA`/`CHAC` — замена подстроки (CHAC с учётом регистра); `ADD` без подполя → каждая строка ФОРМАТА 1 = новое повторение.
- **IfBlock**: `pft.eval(cond)=='1'` → выполнить тело, иначе пропустить.
- **RepeatBlock**: выполнять тело, пока `pft.eval(until)=='1'`; **обязателен лимит итераций** (защита от зацикливания, конфиг — напр. 10⁵) → при превышении job → `failed` с диагностикой.
- **NEWMFN/NEWREC/CORREC**: переключают «текущую запись» на новую/чужую; ФОРМАТ 1 во вложенных `ADD` считается на **исходной** записи исходной БД (`txt:5988-5991`); группа закрыта `END` → возврат. Создание/правка чужих записей тоже журналируется (откат должен снести созданные записи — §4.5).
- **UNDOR**: `pft.eval(steps)` → `N` (на N копий назад) / `'*'` (к исходной) / `∅` (ничего); реализуется поверх журнала копий §4.5 (а НЕ повторным прогоном).
- **DELR/UNDEL/EMPTY/ALL/DEFFLD/GETFLD/PUTLOG** — по семантике recon §1.3.

### 3.5 Идемпотентность и границы транзакции
- **Применение всего job — одна логическая транзакция** на тенанта (батч-режим): либо все записи закоммичены, либо откат (для больших выборок — чанками по N записей с чекпойнтами журнала, чтобы не держать гигантскую транзакцию; при сбое чанка — job `failed`, уже закоммиченные чанки откатываемы через журнал по `job_id`).
- Повторный запуск **того же** job недопустим (job одноразов); повторная подача того же задания — новый job (счётчики/переменные могут дать иной результат — это ожидаемо, не идемпотентно по построению, как и в ИРБИС).

## 4. Превью, применение и откат (ключевой AC)

### 4.1 Превью (dry-run → diff)
`preview:true` → полный прогон конвейера §3.4 **без** `persist`/`journal`/инкремента счётчиков, с теми же вычислениями форматов и переменных. Результат — структурированный diff:
```
DiffReport{ jobPreviewId, totals:{records, changed, created, deleted, errors},
            records:[ RecordDiff{ mfn, db, status:changed|created|deleted|unchanged|error,
                                  fields:[ FieldChange{ tag, occ, subfield?, before, after, op } ],
                                  putlog:[string], error? } ],
            warnings:[…], truncated:bool }
```
- Diff — **пополевой/поподпольный** (что именно меняется в каждом повторении), пригодный к рендеру в UI как «было → стало».
- Для больших выборок — `truncated` + пагинация превью (детальный diff по первым K записей + сводка по остальным).

### 4.2 Применение
`preview:false` → конвейер с `persist+journal`. На каждой записи — необязательный прогон ФЛК (см. 4.4). Возвращает `{ jobId, status, totals }` (202 + поллинг статуса для долгих, §5.3).

### 4.3 Журнал изменений (обратимый) — связь с аудитом I1 #101
Каждое применение пишет в таблицу схемы тенанта:
```
gbl_change_log(
  id, tenant_id, job_id, mfn, db, seq,
  op_index,                 -- какой оператор задания (для трассировки)
  before_record  jsonb,     -- полная запись ДО (источник отката и истории копий для &uf('4…')/UNDOR)
  after_record   jsonb,     -- полная запись ПОСЛЕ
  change_kind    text,      -- update|create|delete|undelete
  created_at, actor_id )
```
- `before_record` хранит **полную** запись → откат = восстановить `before` (не «обратная операция»), это надёжнее для произвольных `.gbl`.
- **История копий** записи (для `UNDOR`/`&uf('4…')`) собирается из `before_record` по `(mfn)` в порядке `seq` — движок A1 получает её через `ctx.history`.
- **Связь с аудитом #101:** на каждый job — одна запись в общий `audit` (actor, tenant, функция `cat.gbl`, объект=выборка, результат, `job_id`); `gbl_change_log` — детальный под-журнал, ссылается на `audit.id`. Аудит — неизменяемый; change-log — основа отката.

### 4.4 ФЛК на границе применения
- Перед `persist` обычной правки — опц. `dbnflc` (A2); перед `DELR`/`recdel` — `delflc` (A2, [SPEC_ws1 §7], `delflc/rmarce_flc` recon `TODO FLC-04`). Режим: `flc: off|warn|block` в payload. `block` + ошибка типа 1 → запись пропускается, помечается `error` в отчёте, job продолжается (не падает целиком). Паритет с `Aifupdate`/`FlcAble` (`TODO #GBL-04`).

### 4.5 Откат (`POST /api/batch-correct/{jobId}/rollback`)
- Восстанавливает `before_record` для всех строк job в обратном порядке `seq`; `create`→удалить созданную запись; `delete`→восстановить; `undelete`→снять восстановление.
- **Откат сам журналируется** (новый `job_id` типа `rollback-of:<jobId>`) → обратим повторно, аудируем.
- **Ограничения отката:** если после job запись правил другой пользователь/job (конфликт по версии записи) → откат этой записи **отклоняется** с диагностикой (не затирать чужую правку); отчёт об откате перечисляет применённые/отклонённые/конфликтные.
- **`UNDOR`-семантика внутри задания** (откат к копии **во время** прогона, `txt:6006-6031`) ≠ `rollback` API (откат **после** job): первое — операция языка над текущей записью через `ctx.history`; второе — административная отмена всего job. Оба опираются на один журнал копий.

```
  selection: { query?: "...", mfnRange?: {from,to}, mfnList?: [...], sequential?: bool },
  preview: true|false,
  flc: "off"|"warn"|"block"
}
→ 200 (preview:true)  : DiffReport
→ 202 (preview:false) : { jobId, status:"queued"|"running", totalsEstimate }
→ 400 parse/validate error (номер строки)
→ 403 нет гранта/модуль выключен (entitlement) — с аудитом
```
- `gbl.source:"@имя"` → задание грузится из конфигурации тенанта (аналог каталога БД, [GLOBAL_CORRECTION §2]; список — `GLOB.MNU`, `MERGEGBLMNU` и т.п., `txt:6796`); `inline` → `body`.
- Многострочное тело передаётся как массив/строка с `\n` (внутренний аналог упаковки `$0D0A` протокола, §3.7).

### 5.2 Вспомогательные
```
POST /api/batch-correct/validate          { gbl } → AST-резюме + errors/warnings  (без выборки)
GET  /api/batch-correct/jobs/{jobId}       → { status, totals, progress, startedAt, finishedAt, putlog[], error? }
GET  /api/batch-correct/jobs/{jobId}/diff  → DiffReport (для завершённого превью/применения)
POST /api/batch-correct/jobs/{jobId}/rollback → 202 { rollbackJobId } | 409 (нечего/уже откачено)
GET  /api/batch-correct/jobs              → список job тенанта (история, фильтр по db/actor/date)
```

### 5.3 Долгая задача (очередь)
- `preview:false` (и тяжёлые превью) — **асинхронны**: job в очередь (per-tenant воркеры), статус через `GET …/jobs/{jobId}`, прогресс `processed/total`. Лёгкое превью (малая выборка, порог конфигурируем) — синхронно (200).
- Очередь изолирована по тенанту (нет монополизации воркеров одним тенантом); конкурентность по одной БД сериализуется (избегаем гонок на одних записях).
- Наблюдаемость: `trace_id`+`tenant_id`+`job_id` в логах/спанах (DoD #117).

### 5.4 Права и тенант
- Грант `cat.gbl:<db>:admin`; модуль каталогизации должен быть включён (entitlement #101) — иначе 403/404.
- Кросс-БД операторы (`NEWMFN '*'→другая БД`, `CORREC`, `&uf('D<db>…')`) проверяются на грант к **целевой** БД того же тенанта; выход за тенант запрещён жёстко (тест изоляции).
- `GLOBTOTALMAIN` (правка без предварительной отметки, `txt:3591`) ↔ разрешение запускать по `query`/всему диапазону — отдельный флаг гранта/настройки тенанта.

## 6. Изоляция тенанта (инвариант)

- Выборка, загрузка записей, кросс-БД доступ (`J/D/7,…`), счётчики, журнал — **только в схеме `t_<slug>`** текущего тенанта ([SPEC_I1 #100]); `search_path` фиксирован на соединении.
- A1, вызываемый из A3, наследует тенант-контекст; `&uf('I…')` (INI-параметры сервера — логин/этап оператора, `autoin.gbl`/`Move691.gbl`) резолвятся из настроек тенанта/сессии, не из глобального INI.
- **Негативный тест:** задание тенанта A, ссылающееся на БД/MFN тенанта B, не видит и не правит данные B (CI-гейт изоляции, обязателен по DoD I1).

## 7. Acceptance criteria (тестируемые)

- **AC1 (парсер):** все 22 оператора из [GLOBAL_CORRECTION] парсятся в AST; 5-строчные операторы разбираются по фиксированной арности; заполнитель `XXXX…`/пустая строка → `∅`; незакрытый блок/неизвестный оператор/ссылка `%n` сверх объявленного → ошибка с номером строки.
- **AC2 (форматы через A1):** ФОРМАТ 1/2 и `&uf(...)` вычисляются движком A1 (не дублируются); режим `F` сопоставляет i-ю строку результата с i-м повторением поля; переменные `&uf('+7…')` и счётчики `&uf('++C…')` читаются/пишутся через контекст.
- **AC3 (отбор):** выборка формируется по `mfnRange` | `query` | их комбинации | `mfnList` через поиск I2 на PostgreSQL; снимок MFN фиксируется на старте job (детерминизм).
- **AC4 (последовательность):** операторы применяются по порядку, каждый видит запись после предыдущих; `IF…FI`, `REPEAT…UNTIL` (с лимитом итераций), `NEWMFN/CORREC/END` работают по семантике recon.
- **AC5 (превью):** `preview:true` даёт пополевой diff (было→стало по каждому повторению) **без** изменения БД и **без** инкремента персистентных счётчиков; превью и последующее применение дают совпадающий diff на неизменной выборке.
- **AC6 (применение+журнал):** применение пишет `before/after` по каждой записи, связывает с `audit` #101 (`job_id`); долгий job асинхронен, статус/прогресс доступны.
- **AC7 (откат):** `rollback` восстанавливает все записи job (вкл. созданные/удалённые); сам журналируется и обратим; запись, изменённая после job другим актором, при откате **отклоняется** с диагностикой, остальные откатываются.
- **AC8 (UNDOR):** оператор `UNDOR` откатывает текущую запись к N-й/исходной копии через журнал копий (`ctx.history`), совместно с `&uf('4…')` в `REPEAT…UNTIL` (паттерн поиска копии по условию, `txt:6017-6031`).
- **AC9 (ФЛК):** при `flc:block` запись с непреодолимой ошибкой (тип 1) пропускается и помечается `error`, job не падает; `delflc` отрабатывает на `DELR`.
- **AC10 (права/тенант):** запуск требует `cat.gbl:db:admin`+включённый модуль; кросс-БД операторы остаются внутри тенанта; негативный тест изоляции зелёный в CI.

**DoD:** каждый оператор и каждый AC покрыты тестами (позитив+негатив); эталонные `.gbl` из recon прогоняются как golden-тесты (§9); контракт API в openapi + Pact-тест; наблюдаемость с `job_id`.

## 8. Критический путь A3

1. **Парсер/AST + статическая валидация** (§1, `/validate`) — независимо от данных, разблокирует редактор задания (D1).
2. **Интеграция с A1** (`ctx.vars/counters/history`, кросс-БД под тенантом, `evalLines`) — **жёсткая зависимость**; без A1 движок не считает форматы.
3. **Отбор записей через I2** (§3.1) + конвейер исполнения FieldOp/IF/REPEAT (§3.4) на JSONB-ядре.
4. **Превью (diff)** (§4.1) — раньше применения, как безопасный режим по умолчанию.
5. **Журнал + применение + откат + UNDOR** (§4.3–4.5) — ключевой AC обратимости; связь с аудитом #101.
6. **Очередь/долгие задачи + права/изоляция** (§5.3–5.4, §6) + операторы др. БД (`NEWMFN/CORREC`).
7. **ФЛК-интеграция** (A2) + остаточные `experimental`-операторы (`NEWREC/DEFFLD/GETFLD`).

**Выходной критерий A3:** оператор запускает любое эталонное задание `.gbl` из дистрибутива по выборке (диапазон/запрос) в режиме превью → видит пофайловый/пополевой diff → применяет → при необходимости **откатывает** через журнал, с полной изоляцией тенанта, аудитом и асинхронным статусом — функциональный паритет с `IC_gbl` без десктоп-Администратора.

## 10. Риски и открытые вопросы

**Риски:**
- **R1 (зависимость от A1):** без поддержки в A1 инъекции `ctx` (переменные/счётчики/история/кросс-БД под тенантом) движок нефункционален. Митигация — зафиксировать контракт §2 как требование к A1 в одном PR; до готовности A1 — стаб-эвалюатор для парсера/превью-каркаса.
- **R2 (семантический паритет):** тонкости (`F`-режим, пустая строка=удаление, накопление переменных между записями, `&unifor('Z')`) легко реализовать «почти правильно». Митигация — golden-тесты §9 + сверка с протоколом ИРБИС на тех же входах.
- **R3 (превью≠применение):** счётчики/`&uf` с побочными эффектами и конкурентные правки могут расходить превью и применение. Митигация — снимок выборки на старте, счётчики read-only в превью (зафиксировано в AC5), версия записи при откате.
- **R4 (производительность):** большие выборки × тяжёлые форматы (кросс-поиски в цикле, как `SumEkzToSZ`). Митигация — очередь/чанки, кэш кросс-БД лукапов в пределах job, лимит итераций цикла.
- **R5 (откат после стороннего изменения):** конфликт версий → откат частично невозможен. Принято: отклонять конфликтные записи, не затирать чужое (AC7).

**Открытые вопросы (recon TODO → к реализации):**
- `#GBL-01` `NEWREC` — семантика 2-й строки (флаг создания vs. имя БД); `#GBL-02/03` `DEFFLD/GETFLD` — формальный синтаксис/назначение; `#GBL-05` заполнитель пустая vs. `XXXX…`; `#GBL-04` `FlcAble/AutoinAble` через `IC_gbl` (как передаётся ФЛК/автоввод) — уточнить на боевых конфигах/сервере при реализации.
- Точная область видимости/время жизни переменных `&uf('+7…')` (между записями vs. per-record) — подтвердить на эталонах (`StatKsu`, `SumEkzToSZ`).
