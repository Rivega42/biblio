Сквозной движок: «поиск авторитета + автоподстановка `^3` + autoin генерация связанных записей + классификационные деревья (ГРНТИ/УДК/ББК)». Без него ws1 §3 ([SPEC_ws1_cataloger](../SPEC_ws1_cataloger.md)) даёт авторитетный контроль «по названию», но не по реализуемости.

> [FIELD_DICTIONARY](../../../recon/deep/reference/format/FIELD_DICTIONARY.md) (поля 700/701/710/600/601/606/607 с `^3`; индексы 621/675/686/964/965) ·
> [FLC_VALIDATION §3](../../../recon/deep/reference/format/FLC_VALIDATION.md) (расширенный ввод `@!*mov*`, тип ввода 2 «редактор заголовка по словарю»; `!964` структура индекса) ·
> [CONTROL_VOCABULARIES](../../../recon/deep/CONTROL_VOCABULARIES.md) и [reference/format/CONTROL_VOCABULARIES](../../../recon/deep/reference/format/CONTROL_VOCABULARIES.md) (`.tre`-деревья, `vkol.mnu`, `kod.mnu`).

## 0. Область ответственности и границы

**Сервис `authority` отвечает за:**
1. Хранение и версионирование 6 авторитетных БД (ATHRA/ATHRC/ATHRS/ATHRG/ATHRU/ATHRB) на PostgreSQL.
2. Lookup API (поиск по словарю/префиксу + полная запись) для редактора Каталогизатора (ws1) и OPAC/читателя (I2).
3. **Substitution-pipeline**: при выборе авторитета — автозаполнение подполей каталожной записи + протягивание ссылки `^3`.
4. **autoin**: автоматическое до-создание связанных авторитетных записей (510/710 без `^3`) — порт `autoin.gbl`.
5. Maintenance: дедупликация (алкод + ML-кандидаты), слияние, контроль ссылочной целостности при правке/удалении.
6. Классификационные деревья ГРНТИ/УДК/ББК + АПУ (ATHRU/ATHRB) как иерархия в PostgreSQL: выбор + автоввод индекса (621/675/686/964/965), валидация `!964`.

**НЕ отвечает (границы):**
- Рендер карточек/печать → [SPEC_engine_pft](SPEC_engine_pft.md) (A1).
- Запуск `.gbl`-заданий как пользовательских пакетов → [SPEC_engine_gbl](SPEC_engine_gbl.md) (A3). `autoin` здесь — **встроенный системный** хук, не пользовательское задание.
- Полевой ФЛК каталожной записи (`!700`, `!606`, `!964` как блокировка сохранения) → ФЛК-движок (A2, ws1 §2). Authority-сервис **поставляет данные и валидирует связь `^3`**, ФЛК-движок их **вызывает**.
- Сидинг базовых классификаторов/авторитетов на тенант → [SPEC](../../DESIGN_COVERAGE.md) A5 (онбординг). Здесь — только структура и import API.

## 1. Модель данных авторитета (PostgreSQL)

### 1.1 Шесть баз → единая полиморфная схема

Все 6 БД ИРБИС построены на **единой 4-блочной RUSMARC-Authorities модели** (DB_AUTHORITY §0): 210 принятый заголовок · 410 «см.» (вариант) · 510/500/515/520/550 «см. также» (связанные принятые) · 710 параллельные формы. Различие — только в наборе подполей 210 и составе ссылок. Поэтому храним **одну полиморфную таблицу `authority_record`** с дискриминатором `kind`, а подполя — в JSONB (паритет с JSONB-моделью записи ядра I1).

| `kind` | БД ИРБИС | Объект заголовка | Самостоятельность | Подполей в 210 |
|---|---|---|---|---|
| `personal` | ATHRA | Индивидуальный автор (имя лица) | да | 13 |
| `corporate` | ATHRC | Коллектив / мероприятие / издательство | да (тип в `^9`) | 20 |
| `subject` | ATHRS | Предметная рубрика (+ имя/орг/гео как предмет) | да | 19 (+210a≈31/210c≈35/210g=19) |
| `geographic` | ATHRG | Географическое наименование | да | 14 |
| `udc_index` | ATHRU | Рубрика АПУ к УДК (1 код) | да | 3 |
| `bbk_index` | ATHRB | Рубрика АПУ к ББК (до 5 кодов) | да | 7 |

> Решение: **одна таблица, JSONB-подполя** (не таблица-на-БД и не EAV). Обоснование: набор полей стабилен в пределах `kind`, но различается между `kind`; поиск идёт по нормализованным проекциям (`authority_term`, §1.3), а не по сырому JSONB; правки структуры РЛ тенанта не требуют DDL. Это симметрично модели библиозаписи ядра I1.

### 1.2 Таблица `authority_record`

```
authority_record
  id            uuid  PK
  tenant_id     uuid  NOT NULL            -- RLS-изоляция (I1)
  kind          enum  NOT NULL            -- personal|corporate|subject|geographic|udc_index|bbk_index
  worklist      text  NOT NULL            -- поле 920: ATHRA|REF|EXP|... (тип записи внутри БД)
  status        enum  NOT NULL            -- accepted | needs_review (210^!) | deprecated | merged
  heading_210   jsonb NOT NULL            -- принятый заголовок: {a,b,g,...} + спец-подполя ^9/^7/^8
  see_410       jsonb[]                   -- вариантные формы «см.»: каждый {подполя, ^5, ^0}
  see_also      jsonb[]                   -- связанные «см.также»: 510/500/515/520/550/710 c {kind, ^5, ^3, target_id}
  notes         jsonb                     -- 300/305/310/320/330 и блок 8 (801/810/.../836)
  classmarks    jsonb                     -- 675 УДК / 689 ББК (для subject); коды АПУ (^b..^f)
  ext_ids       jsonb                     -- ORCID(10), SPIN(923), КРН(921/922), РИНЦ(924), и т.п.
  shifr_903     text                      -- собственный шифр авторитетной записи (autoin: 210^a.1 + R9)
  alcode        text  NOT NULL            -- алкод дублетности (нормализованный ключ имени/наименования)
  uri           text  NOT NULL UNIQUE     -- стабильный URI записи (см. §1.4)
  merged_into   uuid  NULL REFERENCES authority_record(id)  -- при слиянии
  source        jsonb                     -- происхождение (импорт RUSMARC / ручной / autoin / СКБР)
  version       int   NOT NULL            -- оптимистическая блокировка
  created_at / updated_at / created_by / updated_by
  UNIQUE (tenant_id, kind, alcode) WHERE status='accepted'   -- дублетный контроль на уровне БД
```

Связи «см. также»/«параллельные» **внутри одной БД** (510/710 с `^3`) хранятся и денормализованно (`see_also[].^3` = `shifr_903` цели) и **нормализованно** в графе ссылок (§1.5) для контроля целостности.

### 1.3 Проекция терминов (поисковый индекс — порт FST)

FST ИРБИС (DB_AUTHORITY §2) генерирует словарные термины из полей по префиксам (`A=`, `M=`, `S=`, `G=`, `K=`, `H=`, `U=`…). Воспроизводим таблицей-проекцией `authority_term`, заполняемой при сохранении (триггер/доменный сервис):

```
authority_term
  id, tenant_id, record_id (FK), kind
  prefix     text   -- A|M|S|G|K|H|U|D|N|... (паритет FST-префиксов)
  term_raw   text   -- отображаемая форма
  term_norm  text   -- нормализованная (lower, без диакритики, схлопнутые пробелы) — для поиска по префиксу
  term_kw    text   -- словарная (метод 8/250): ключевые слова из ^a/^g/... — для морфо-поиска
  is_variant bool   -- из 410 (вариантная форма) vs из 210 (принятая)
  blocked    bool   -- ^5 ∈ {f0,a0,b0,e0,z0} → не индексировать на показ (FST 410 исключает)
  INDEX (tenant_id, kind, prefix, term_norm text_pattern_ops)   -- префиксный/usечение поиск
  INDEX gin(term_kw gin_trgm_ops)                               -- словарный (контейнерный) поиск
```

Соответствие FST → `prefix` (из DB_AUTHORITY §2.x, итоговые префиксы):
- ATHRA: `A` (210/410 имя), `K` (словарь), `H` (510/710), `ID/I`, `MR/FAK/POS/ORG` (место работы), `UCHS/OTR/DPS/UCHZ`, `AN/SPIN/ORCID`.
- ATHRC: `M` (общий), `Z` (коллектив/мероприятие, `^9:0/1`), `I` (издательство, `^9:2`), `K`, `D/N` (дата/номер меропр.), `G` (география), `T` (тип).
- ATHRS: `S` (рубрика), `A/M/G` (имя/орг/гео как предмет — мультиплекс по 920), `P` (подрубрики), `HS` (хронолог.), `K`.
- ATHRG: `G` (наименование), `L` (лидер-форма без инверсии), `K`, `PG/HG/SP`.
- ATHRU/ATHRB: `M` (заголовок АПУ), `U` (код рубрики), `K`.

> Ключевой нюанс (DB_AUTHORITY §2.2): один заголовок ATHRC индексируется под `M` (общий), `Z` (коллектив) **и** `I` (издательство) — разнесение по `^9`. Проекция порождает несколько строк `authority_term` на одну запись.

### 1.4 Стабильные URI (BIBFRAME/LOD-перспектива)

Решение: **opaque-URI на UUID**, а не на шифр (шифр может меняться при слиянии). Шаблон:
```
https://{tenant-host}/authority/{kind}/{id}          -- HTML/JSON-LD по Accept
https://{tenant-host}/authority/{kind}/{id}.jsonld    -- machine
```
- URI **persistent**: при слиянии запись получает `status=merged`, `merged_into=<target>`, а её URI отдаёт `301`/`sameAs` на целевой URI (паритет «непринятая форма ведёт к принятой»).
- В перспективе LOD: `authority_record` → `madsrdf:Authority` / `bf:Agent|bf:Topic|bf:Place`; 510/710 → `mads:hasReciprocalAuthority`; внешние ID (`ORCID`, VIAF, СКБР) → `owl:sameAs`. Эти отображения держим в маппинг-таблице `uri_external_map` (record_id → scheme → external_uri), наполняемой при импорте и дедупликации. Реализация JSON-LD сериализации — за рамками MVP, но схема URI фиксируется **сейчас**, чтобы не ломать ссылки позже.

### 1.5 Граф связей (целостность)

```
authority_link
  tenant_id, src_id (FK), dst_id (FK NULL — может быть «висячей» по ^3 до резолва),
  link_type enum (see_also_510 | parallel_710 | subject_500/515/520/550 | broader_g | narrower_h),
  kod_5     text,         -- ^5 (kod.mnu): a/b/e/f/z (+*0 blocked)
  shifr_3   text,         -- ^3 как было записано (для резолва/диагностики «висячих»)
  UNIQUE (tenant_id, src_id, dst_id, link_type)
```
Этот граф — основа контроля при правке/удалении (§4) и навигации «см. также» (DB_AUTHORITY §5.1: 510/710 требуют самостоятельных записей).

## 2. Lookup API

Базовый путь `/api/authority`. Аутентификация/тенант — через платформенный middleware (I1). `{db}` ∈ `{athra, athrc, athrs, athrg, athru, athrb}` (маппинг на `kind`).

### 2.1 Поиск по словарю/префиксу

```
GET /api/authority/{db}/search?q=толс&prefix=A&mode=prefix&worklist=ATHRA&limit=20&cursor=...
```
- `mode`: `prefix` (по `term_norm`, усечение справа — паритет словарного поиска `A=ТОЛС…`), `keyword` (морфо/триграмма по `term_kw` — паритет метода 8 `K=`), `exact`.
- `prefix`: один из §1.3 (по умолчанию — главный для БД: ATHRA→`A`, ATHRC→`M`, ATHRS→`S`, ATHRG→`G`, ATHRU/B→`U`/`M`).
- Ответ — отранжированный список кандидатов с **признаком формы** (принятая/вариантная) и подсказкой для подстановки:

```json
{ "items": [
  { "id":"…","uri":"…/authority/personal/…","shifr_903":"ТОЛСТ123",
    "kind":"personal","worklist":"ATHRA","is_variant":false,
    "label":"Толстой, Л. Н. (Лев Николаевич), 1828–1910",
    "see_from":["Толстой, Лев Николаевич"],          // 410 формы, ведущие сюда
    "fill":{ "a":"Толстой","b":"Л. Н.","g":"Лев Николаевич","f":"1828–1910","9":"1" } },
  ...], "next_cursor":"…" }
```
- `is_variant=true` (нашли по 410) → UI показывает «см.: <принятая форма>» и подставляет **принятую** запись (никогда вариантную — паритет «410 не самостоятельна»).
- `mode=advanced` соответствует `[SEARCH] ItemAdv = ATHRA,A=,@sadv` (ARM_CATALOGER): расширенный отбор через авторитетную БД с применением формата свёртки заголовка.

### 2.2 Полная запись

```
GET /api/authority/{db}/{id}            -> полная JSON-запись (210/410/510/710/notes/classmarks/ext_ids)
GET /api/authority/{db}/{id}.jsonld     -> JSON-LD (LOD)
GET /api/authority/{db}/{id}/related    -> развёрнутый граф «см.также»/параллельные (для навигации OPAC)
```

### 2.3 Деревья классификации (см. §6)

```
GET /api/classification/{scheme}/tree?node={code}&depth=1      -- ленивое раскрытие узла
GET /api/classification/{scheme}/search?q=...                  -- поиск рубрики по тексту/коду
GET /api/classification/{scheme}/{code}/path                   -- путь от корня (хлебные крошки)
```
`scheme` ∈ `{grnti, udc, bbk}`. Навигаторы паритетны `[SEARCH] WnLink` (ARM_CATALOGER): RSUDC, RSBBK, ГРНТИ, athr*, tez, MeSH.

### 2.4 AC — Lookup

- **AC-L1** Поиск по префиксу `A=`/`M=`/`S=`/`G=`/`U=` возвращает кандидатов усечением (паритет словаря FST); пустой словарь тенанта → пустой результат, а не ошибка.
- **AC-L2** Поиск по варианту (410) возвращает запись с `is_variant=true` и принятой формой в `see_from`; подстановка использует **принятую** запись.
- **AC-L3** `GET /{db}/{id}` отдаёт полную запись со всеми блоками; `related` разворачивает 510/710 в принятые формы.
- **AC-L4** Ответ содержит готовый блок `fill` (карта «авторитет→подполя каталога», §3.2) — клиент не реконструирует подполя.
- **AC-L5** Все ответы изолированы по тенанту (RLS); cross-tenant утечка невозможна (негатив-тест).

## 3. Substitution + autoin (ядро сервиса)

### 3.1 Pipeline подстановки (выбор авторитета в РЛ Каталогизатора)

Каталогизатор в поле 700/701/710/600/601/606/607/710/675/686/964 нажимает «выбрать из авторитетного файла». Pipeline (синхронный, на стороне сервиса):

```
[1] resolve     Каталог-поле → (kind, db, target-prefix)        # по MODATHR-маппингу, §3.3
[2] search      lookup §2.1 (prefix), оператор выбирает кандидата
[3] normalize   если выбран вариант (410) → подменить на принятую запись (210)
[4] map         applyFillMap(kind, catalog_field): authority.heading_210 → подполя каталог-поля   # §3.2
[5] link        записать ^3 = authority.shifr_903 (номер авторитетной записи) в каталог-поле
[6] enrich      доп. подполя по правилам поля (роль ^9 не трогаем; графика ^7/язык ^8 — по политике тенанта)
[7] return      патч каталог-поля {подполя + ^3} → клиент применяет в DynamicField
                # autoin (§3.4) НЕ здесь — он срабатывает при сохранении каталог-записи
```

**Где `@!*mov*` (FLC_VALIDATION §3, DB_AUTHORITY §6/§7).** В ИРБИС подполе типа ввода `2` («редактор заголовка по словарю») задаётся конструкцией `,<префикс>=,@!<формат>` (напр. 210^a ATHRA = `,A=,@!amov`; каталог 700^a в IBIS = `,A=,@!amovf`). `@!amov/@!cmov/@!pmov/@!gmov/@!bmov/@!dmov` — это **форматы свёртки/разворота заголовка** (инверсия ФИО «Фамилия, И. О.» ⇄ разбор на ^a/^b/^g; сборка наименования организации с подразделением; формирование строки рубрики из подрубрик). Это **не динамические формы** и **не ФЛК** — это детерминированные **преобразователи заголовок⇄подполя**.

> Решение: реализуем `@!*mov*` как **именованные нормализаторы заголовков** (встроенные функции сервиса), а не как PFT-интерпретатор и не как пользовательские шаблоны. Реестр нормализаторов:
>
> | Нормализатор | БД | Назначение | Направление |
> |---|---|---|---|
> | `amov` / `amovf` / `amov5` | ATHRA | свёртка/разворот имени лица (инверсия ФИО, `^<` число слов) | заголовок ⇄ ^a/^b/^g/^f |
> | `cmov` / `cmov5` | ATHRC | наименование НО + подразделение + города/даты | ⇄ ^a/^b/^c/^e/^9 |
> | `pmov` / `pmov5` / `smov` | ATHRS | сборка предметной рубрики из тематич./геогр./хронолог. подрубрик | ⇄ ^a/^b/^c/^h/^9 |
> | `gmov` / `dmov` / `bmov` | ATHRG | географ. наименование (инвертируемая часть `^G`, термины `^H/^I/^J`) | ⇄ ^A/^V/^W/^B |
>
> Каждый нормализатор — пара `parse(headingString) → subfields` и `format(subfields) → headingString`, плюс `fold(subfields) → alcode` (для дублетного контроля §4). Семантика свёртки ФИО — `TODO(recon #ATHR-05)`: объёмный код инверсии; на этапе реализации сверить с боевыми `!AMOV*.PFT`.

### 3.2 Таблица маппинга «авторитет → поле каталога» (fill-map)

Карта подстановки для каждой связки (каталог-поле IBIS ↔ kind ATHRx). Подполя каталога — из [FIELD_DICTIONARY] (700/701/710/600/601/606/607); подполя авторитета — из DB_AUTHORITY §4. `^3` всегда = `shifr_903`.

| Каталог-поле (IBIS) | kind / БД | Префикс отбора | Нормализатор | Подполя, переносимые из 210 авторитета → каталог | Ссылка |
|---|---|---|---|---|---|
| **700/701** 1-й/др. индивид. автор | personal / ATHRA | `A=` | `amovf` | `^a`Фамилия, `^b`Инициалы, `^g`Расширение, `^f`Даты(→`^1`/нет), `^9`Признак | `^3` |
| **702** вторичная ответственность | personal / ATHRA | `A=` | `amovf` | `^a/^b/^g` (роль `^4` оператора не трогаем) | `^3` |
| **600** персоналия (о ком) | personal / ATHRA | `A=` | `amovf` | `^a/^b/^g/^f` + аспект `^9` | `^3` |

### 3.3 MODATHR-маппинг (привязка каталог→БД авторитета)

ARM_CATALOGER `[Main]`: ключи `MODATHRA/C/S/B/U/G` привязывают каталог к авторитетным БД. Воспроизводим конфигом тенанта `authority_binding`:
```
MODATHRA → ATHRA (personal)    : поля 700/701/702/600/961…
MODATHRC → ATHRC (corporate)   : поля 710/711/601…
MODATHRS → ATHRS (subject)     : поля 606/605…
MODATHRG → ATHRG (geographic)  : поле 607…
MODATHRU → ATHRU (АПУ УДК)     : поле 675
MODATHRB → ATHRB (АПУ ББК)     : поле 621
```
Если у тенанта `MODATHR*` не задан — соответствующее поле вводится свободно, без авторитетного контроля (graceful degradation; паритет с демо-установкой без авторитетов).

### 3.4 autoin — автогенерация связанных записей (порт `autoin.gbl`)

Срабатывает **при сохранении авторитетной записи** (не каталожной), как системный хук (DB_AUTHORITY §5.3). Алгоритм (порт `ATHRA/autoin.gbl`):

```
on save(authority_record R) where R.see_also has 510/710 without ^3 and R.notes.905^7 allows:
  for each link L in R.see_also (510/710) without resolved ^3:
     key = normalizer.fold(L.subfields)                      # алкод
     hit = find accepted record by (tenant, kind, alcode=key)
     if hit:  L.^3 = hit.shifr_903; L.dst_id = hit.id        # привязка к существующей
     else:                                                    # NEWMFN — создать новую авторитетную запись
        N = new authority_record:
            heading_210 = carry(L.subfields: a,b,g,1,c,d,8,9,f,<,3,:)
            shifr_903   = L.^3 := key-derived (210^a.1 + R9)
            notes.801   = (страна/орг/дата из org-config + now)
            worklist(920) = R.kind default (ATHRA/...)
            status = needs_review                              # требует ревизии каталогизатором
        L.^3 = N.shifr_903; L.dst_id = N.id
  also: ensure 907 (каталогизатор/дата), normalize ^< (число слов ФИО)
  rebuild authority_link graph; reproject authority_term
```
Особый случай ATHRS: поле **510 формируется автоматически** из 500/520/550/515 (DB_AUTHORITY §3.3 — `510k.wss`); реализуем как детерминированную деривацию при сохранении subject-записи.

> Это материализует правило `.WS` «связанные формы требуют самостоятельных записей». Созданные autoin записи помечаются `needs_review` (паритет `^!`) и попадают в очередь курирования (§4.4).

### 3.5 AC — Substitution + autoin

- **AC-S1** Выбор автора в поле 700 подставляет `^a/^b/^g` (+`^f`/`^9` по fill-map) и пишет `^3` = `shifr_903` авторитета; роль `^9` каталога не перетёрта.
- **AC-S2** Выбор коллектива в 710 подставляет `^a/^b/^c/^s` + `^3` (нормализатор `cmov`).
- **AC-S3** Выбор предметной рубрики в 606 собирает заголовок+подрубрики (`pmov`) и пишет `^3`.
- **AC-S4** Выбор вариантной формы (410) подставляет **принятую** запись (не вариант) и её `^3`.
- **AC-S5** Сохранение авторитетной записи с 510/710 без `^3` и `905^7`=да → autoin привязывает к существующей или создаёт `needs_review`-запись с `903`/`801`/`907`/`920`.
- **AC-S6** Сохранение subject-записи деривирует 510 из 500/520/550/515.
- **AC-S7** Нормализатор `amov` обратим: `parse(format(x)) == x` для корректных заголовков (round-trip тест).

## 4. Maintenance — дедупликация, слияние, целостность

### 4.1 Дедупликация (детерминированная + ML)

**Слой 1 — детерминированный (алкод).** Паритет FST-алкода (`&unifor('B'…)`, метод 7/8): `normalizer.fold(heading) → alcode`. `UNIQUE(tenant, kind, alcode) WHERE status=accepted` (§1.2) ловит точные дубли на уровне БД при сохранении (блокирующе — паритет «контроль дублетности по имени»). Это основной и **обязательный** механизм.

**Слой 2 — ML-кандидаты (некритичный путь, фоновый).** Алкод ловит орфо-варианты слабо (разные транслитерации, инициалы vs полное имя, опечатки). Фоновый матчер строит **кандидатов на слияние** (не автослияние): блокинг по `term_kw` (триграммы) + скоринг (Jaro-Winkler по фамилии, совпадение дат `^f`, пересечение внешних ID ORCID/SPIN). Результат → таблица `merge_candidate(src, dst, score, features)` → очередь курирования (§4.4). **Порог автодействия = 0** (ML только предлагает; решение — за человеком), кроме совпадения по сильному ID (ORCID/SPIN идентичны → авто-предложение high-confidence).

> Решение: дедуп = **детерминированный gate (блокирует точные дубли) + ML-suggester (предлагает нечёткие)**. Авто-слияния по ML нет — авторитеты слишком чувствительны к ложным слияниям (риск R3).

### 4.2 Слияние (merge)

```
POST /api/authority/{db}/{id}/merge  { target_id, keep:"target"|"source", field_resolution:{...} }
```
1. Выбрать `master` (целевая) и `loser`.
2. Перепривязать все каталожные `^3`, указывающие на `loser.shifr_903` → `master.shifr_903` (фоновая глоб.корректировка по каталогу тенанта — вызов A3 движка в режиме system-job, с журналом).
3. Перепривязать `authority_link` (510/710 от/к loser → master).
4. `loser.status=merged`, `loser.merged_into=master.id`; URI `loser` → 301 на `master` (§1.4).
5. Объединить 410-формы (loser.210 и loser.410 становятся 410-вариантами master, если ещё не там).
6. Журнал слияния (аудит I6) с возможностью **отката** (хранить снимок до слияния).

### 4.3 Контроль ссылочной целостности при правке/удалении

- **Правка принятого заголовка (210):** каскадно обновляет отображаемую форму везде, где есть `^3`-ссылка (каталог показывает форму **по `^3` из авторитета**, а не копию — см. §4.5); пере-проецирует `authority_term`; пере-вычисляет `alcode` (если изменился — повторный дублет-контроль).
- **Удаление:** запрещено, если на запись есть входящие `^3` из каталога **или** входящие 510/710 от других авторитетов (паритет `delflc`-подхода). Варианты: (а) `deprecated` вместо delete; (б) принудительное слияние; (в) каскадный отвяз только при пустых ссылках. Возврат — список блокирующих ссылок (как `!903` «книга выдана — правка запрещена»).
- **«Висячие» `^3`** (импорт RUSMARC, где 510/710 имеют `^3` без резолва — FST метка `W=NNN`, DB_AUTHORITY §2): фоновый резолвер `authority_link.dst_id IS NULL` сопоставляет по алкоду; нерезолвенные → очередь курирования.

### 4.4 Очередь курирования

Записи `status=needs_review` (autoin-созданные, `^!`, импортированные, ML-кандидаты слияния, висячие `^3`) → рабочая очередь каталогизатора в ws1/ws5: подтвердить/исправить/слить/отклонить. Это паритет ИРБИС-практики «записи на редактирование» (FST `R=`).

### 4.5 Принцип «связь, а не копия»

Каталожная запись хранит `^3` (ссылку), а отображаемые подполя (`^a/^b/...`) — как **денормализованный кэш** на момент подстановки (паритет ИРБИС: подполя физически лежат в библиозаписи). При правке авторитета:
- **Показ** (OPAC/карточка) — может резолвить актуальную форму по `^3` (опционально, политика тенанта).
- **Хранимые подполя** обновляются пакетно (фоновая глоб.корректировка по `W=`-метке — точный паритет с тем, как ИРБИС метит записи на актуализацию после правки авторитета).

### 4.6 AC — Maintenance

- **AC-M1** Сохранение точного дубля (совпадение алкода) блокируется с ссылкой на существующую запись (паритет тип-1 ФЛК).
- **AC-M2** ML-матчер строит кандидатов слияния без автослияния; совпадение ORCID → high-confidence предложение.
- **AC-M3** Слияние перепривязывает все каталожные `^3` и межавторитетные 510/710 на master; loser→merged; откат возможен.
- **AC-M4** Удаление записи с входящими `^3`/510 блокируется со списком ссылок; разрешён `deprecate`.
- **AC-M5** Правка принятого заголовка ставит зависимые каталожные записи в очередь актуализации (`W=`).

### 6.2 Хранилище дерева — выбор

Решение: **closure-table** для деревьев классификации (а не `ltree`, не adjacency-only).

```
classification_node
  id, tenant_id, scheme (grnti|udc|bbk|other),
  code        text,         -- индекс рубрики ("20.23", "004.7", "Ч73")
  label       text,         -- наименование рубрики
  parent_id   uuid NULL,    -- adjacency (для редактирования/перемещения)
  sort_key    text,
  apu_ref     uuid NULL,    -- ссылка на ATHRU/ATHRB запись (АПУ) для УДК/ББК
  UNIQUE(tenant_id, scheme, code)

classification_closure
  ancestor_id, descendant_id, depth          -- все пары предок→потомок (вкл. self, depth=0)
  PK(ancestor_id, descendant_id)
```

**Обоснование выбора (closure-table vs ltree vs adjacency):**

| Критерий | adjacency-list | `ltree` | **closure-table (выбран)** |
|---|---|---|---|
| Поддеревья/предки одним запросом | ✗ (рекурсия) | ✓ | ✓ |
| Перемещение узла (реклассификация) | дёшево | **дорого** (переписать пути всего поддерева) | средне (пересчёт closure поддерева) |
| Коды нечисловые/неоднородные (ББК `Ч73`, УДК `004.7:002`) | ок | требует чистки в `label`-путях | **ок** (code — обычная колонка) |
| Мультитенантность / разные версии схем | ок | ок | ок |
| Зависимость от расширения PG | нет | нужен `ltree` ext | нет (чистый SQL) |

ГРНТИ/УДК/ББК — большие, относительно **стабильные**, но периодически **реклассифицируемые** деревья с нечисловыми кодами; closure-table даёт быстрые «поддерево/предки/путь» без рекурсии и без боли `ltree` при переносе узлов и нечистых кодах. (Если профилирование покажет, что реклассификация массовая, а чтение редкое — пересмотреть; зафиксировано как open-question Q2.)

### 6.3 Выбор + автоввод индекса

- Дерево раскрывается лениво (`/tree?node=&depth=1`), поиск рубрики по коду/тексту (`/search`).
- Выбор листа/узла → автоввод **кода** в каталог-поле (964/675/621/686) + опционально текст рубрики в показ.
- Для УДК/ББК узел может ссылаться на АПУ-запись (ATHRU/ATHRB, `apu_ref`) — выбор по алфавитно-предметному указателю ведёт к индексу (паритет навигаторов RSUDC/RSBBK и `RSUDCINPUT/RSBBKINPUT`).
- ГРНТИ: иерархия по `[Rubricator]` (группа `G=`/рубрика `R=`, 3 уровня по 3 знака, паттерн `NN.NN.NN`).

### 6.4 Валидация структуры индекса (`!964`)

Порт `!964.PFT` (FLC_VALIDATION §2.5): индекс 964 разбирается по позициям `*N.1` — **чередование числовой части и разделителя `.`**; нарушение шаблона → ошибка тип-1. Реализуем как структурный валидатор `validateIndexStructure(scheme, code)`:
- ГРНТИ: `^\d{2}(\.\d{2}){0,2}$` (паритет `RubLevelLength=3`/`MaxRubLevel=3`, паттерн `.00`).
- УДК/ББК: проверка по грамматике схемы + (если строго) существование узла в дереве тенанта.
Вызывается ФЛК-движком (A2) на выходе из поля; здесь — сервис-функция-предоставитель.

### 6.5 AC — Классификация

- **AC-C1** Дерево ГРНТИ/УДК/ББК раскрывается лениво по узлу; путь от корня (хлебные крошки) возвращается.
- **AC-C2** Выбор рубрики автоматически вводит код в 964/675/621 (+текст в показ).
- **AC-C3** УДК/ББК: навигация по АПУ (ATHRU/ATHRB) приводит к индексу.
- **AC-C4** Индекс 964 проходит структурную валидацию `!964` (позитив+негатив: правильный/сломанный шаблон).
- **AC-C5** Реклассификация (перемещение узла) корректно пересчитывает closure и пути потомков.

## 9. Критический путь

1. **Модель данных** (`authority_record` JSONB + `authority_term` проекция + `authority_link`) → база поиска.
2. **Lookup API** (§2) + нормализаторы `amov/cmov/pmov/gmov` (§3.1) → выбор работает.
3. **Substitution-pipeline + fill-map + MODATHR-binding** (§3.1–3.3) → подстановка `^a/^b/^3` в ws1.
4. **autoin** (§3.4) + дублетный gate (§4.1 слой 1) → авторитетная целостность при сохранении.
5. **Деревья ГРНТИ/УДК/ББК** (closure-table) + `!964`-валидатор (§6) → классификационный ввод.
6. **Maintenance** (слияние, целостность правки/удаления, очередь курирования §4) → эксплуатация.
7. **ML-дедуп + импорт/экспорт RUSMARC/LOD** (§4.1 слой 2, §7) → курирование и обмен.

**Выходной критерий A4:** каталогизатор в ws1 вводит автора/коллектив/предмет/гео/УДК/ББК/ГРНТИ **через авторитетный файл/дерево** с автоподстановкой подполей и `^3`; сохранение авторитета с 510/710 без `^3` автоматически создаёт/привязывает связанные записи (autoin); точные дубли блокируются, нечёткие — предлагаются к слиянию; правка/удаление авторитета контролирует ссылочную целостность каталога; индексы валидируются `!964`.

## 12. Открытые вопросы

- **Q1.** Нормализаторы `@!*mov*` реализовать как набор встроенных функций (выбрано) или как ограниченный DSL-движок свёрток, конфигурируемый per-tenant? Зависит от того, насколько различаются правила инверсии ФИО между библиотеками (`TODO recon #ATHR-05`).
- **Q2.** Хранилище деревьев: closure-table (выбрано для нечисловых кодов + умеренной реклассификации) vs `ltree` (быстрее по чтению, дороже по перемещению, требует чистых путей). Решить профилированием на реальном объёме ГРНТИ/УДК/ББК тенанта.
- **Q3.** Глубина «связь vs копия»: показывать в OPAC всегда актуальную форму по `^3` (резолв на лету) или довериться денормализованному кэшу с фоновой актуализацией по `W=`? Влияет на стоимость показа и на поведение при массовой правке авторитетов.
