Движок — сервис `sync` на edge (агент) + `sync-hub` в облаке (приёмник). **Оценить ElectricSQL / PowerSync как ускорители транспорта/дельт** ([ADR-005 «Рассмотренные альтернативы»](../../SYNC_DESIGN.md)) — за интерфейсами разд. 4, без переписывания доменной модели владения/конфликтов.

Размеры S/M/L.

## 0. Принципы и границы

- **Не глобальный LWW по wall-clock.** Ось решения ADR-005: **событийный журнал + явное владение по доменам + доменные политики конфликтов**. Wall-clock LWW отвергнут ([ADR-005 «Альтернативы»](../../SYNC_DESIGN.md)) — теряет корректность книговыдачи и данные при перекосе часов. Глобальный порядок — **причинный (HLC)**, а не «по настенным часам».
- **Владение — first-class и асимметрично** (разд. 3.0). Направление и политика конфликта определяются **доменом**, не движком: каталог/авторитеты/словари/лицензии = **облако-master** (edge — read-реплика + очередь предложений); книговыдача/экземпляры/перемещения по ячейкам = **edge-master, event-sourced** (факты, не перезапись состояния → почти бесконфликтно); читатели (RDR) = двусторонне с LWW-по-HLC + журнал конфликтов.
- **Захват изменений — transactional outbox, не приложение-уровень.** Событие пишется в `outbox` **в одной транзакции** с изменением данных (разд. 1.2). Это исключает потерю/раздвоение без 2PC/распределённых транзакций. CDC (logical replication) — допустимая альтернатива за тем же интерфейсом Ingest.
- **Идемпотентность сквозная — *at-least-once → effectively-once*.** Транспорт даёт at-least-once (ретраи, докачка, двойная доставка); приёмник применяет событие **ровно один раз** по `event_id` (разд. 1.4). Повтор/переупорядочивание безопасны.
- **Курсор переживает рестарт.** Прогресс пира = персистентный `cursor` (последний подтверждённый `seq`/`hlc`), хранится в БД, а не в памяти (разд. 4.2). Падение/перезапуск узла → продолжение с того же места, не с нуля.
- **Конфликт — наблюдаемая сущность с UI, не лог в файле.** Любой неавтоматически разрешённый конфликт материализуется как запись `conflict` (разд. 3.3) + событие → **UI разбора** (обе версии, выбрать/слить, аудит). Нет «тихих потерь».
- **Сходимость доказуема.** Для каждого домена — критерий сходимости (eventual consistency) + хаос-тест, проверяющий, что узлы сходятся к одному состоянию независимо от порядка/задержек/дублей (разд. 9.4). Это гейт C2.
- **Границы.** Движок **переносит и согласует события**. Он **не** содержит бизнес-логику книговыдачи (это ws3 — он лишь применяет события), **не** валидирует запись (это ФЛК A2), **не** рендерит уведомления (A6), **не** управляет раскаткой версий узла (fleet/OTA — I4-E/#120; движок лишь *использует* skew-политику и офлайн-лицензию, разд. 7). Union-каталог/межтенантный resource sharing (I4-D, CN #181) — **отдельный слой над** этим движком, не его часть.

## 1. Схема события и журнал (event schema)

### 1.1 Каноническое событие

Событие — иммутабельный факт изменения одной сущности, единица синхронизации. Сериализация — JSON (канон) / protobuf (транспорт, батч). Расширяет минимум ADR-005 §2 (`event_id, tenant_id, entity, entity_id, op, payload, base_version, origin_node, hlc`) причинными зависимостями и доменом.

```jsonc
{
  "event_id": "uuid-v7",          // глобально уникальный; ключ идемпотентности (разд. 1.4). v7 = время-сортируемый
  "hlc": "01890a2b-c3d4-...:0042:nodeA",  // Hybrid Logical Clock (разд. 2): wall_ms:counter:node_id
  "seq": 184213,                  // монотонный per-origin локальный номер в журнале (курсор/дельта since, разд. 4)
  "tenant_id": "spb-gtb",         // арендатор; синхронизируется ТОЛЬКО свой тенант (152-ФЗ, изоляция)
  "domain": "circulation",        // catalog|authority|dict|circulation|holding|reader|config|license
  "entity": "loan",               // тип сущности внутри домена
  "entity_id": "loan:RDR:1234:EX:0007",  // стабильный бизнес-ключ сущности
  "op": "create|update|delete|event",    // event = event-sourced факт (выдача/возврат), не CRUD
  "payload": { /* дельта полей или тело факта; JSONB поля/подполя ядра I1 */ },
  "base_version": 7,              // версия сущности, на которой основано изменение (детекция конфликта, разд. 3.1)
  "origin_node": "edge:spb-gtb:node-01", // узел-источник (edge id или 'cloud')
  "causal_deps": ["uuid-..", "uuid-.."], // event_id событий, ДО которых это не применимо (порядок, разд. 4.3)
  "schema_ver": 2,               // версия схемы события (эволюция/совместимость, разд. 1.6)
  "module": "circulation"         // включённый модуль-источник; приёмник дропает события выключенного модуля
}
```

> **Связь с recon-моделью ([ARCHITECTURE §5](../../ARCHITECTURE.md)):** каждая синхронизируемая сущность в БД несёт `tenant_id, version, deleted(tombstone), origin(node)` + (новое) `hlc_last`. Событие — производное от изменения этих полей через outbox. `payload` для каталога/RDR — JSONB поля/подполя (`v200^a`, `v40^e` и т.п. — те же теги, что в I1/A1), для книговыдачи — тело факта (см. 1.5).

### 1.2 Outbox-таблица (захват, edge и облако)

В одной транзакции с UPDATE/INSERT/DELETE доменной таблицы пишется строка в `outbox`. Релейер (relay) читает `outbox` и отдаёт в транспорт; запись помечается отправленной после ACK пира.

| Поле `outbox` | Назначение |
|---|---|
| `seq` (bigserial, PK) | монотонный локальный порядок публикации (курсор-дельта `since seq`, разд. 4.1) |
| `event_id` (uuid, UNIQUE) | идемпотентность; стабилен при ретрае relay |
| `hlc` | присвоен при коммите доменной транзакции (разд. 2.3) |
| `domain, entity, entity_id, op` | маршрутизация по политике владения (разд. 3.0) |
| `payload` (jsonb) | тело/дельта |
| `base_version, causal_deps` | детекция конфликта / порядок |
| `published_at` | NULL = не отправлено; ставится после ACK пира (докачка, разд. 4.4) |
| `tenant_id, module` | фильтр частичной синхронизации |

- **Без 2PC.** Доменное изменение и событие атомарны (одна локальная транзакция). Сбой после коммита, но до отправки → relay подберёт по `published_at IS NULL` (надёжность, докачка).
- **CDC-альтернатива.** Вместо явного `outbox` допустим CDC по logical replication (Postgres `wal2json`/Debezium) за тем же интерфейсом `OutboxSource` — выбор конфигурацией (оценка ElectricSQL/PowerSync здесь, разд. 4.6). Старт — явный outbox (проще аудит, контроль `base_version`/`causal_deps`).
- **Компакция.** Отправленные строки старше ретеншена компактятся (разд. 1.5); снапшот-граница (разд. 4.5) защищает от потери при компакции до bootstrap нового пира.

### 1.3 Журнал применённого (applied log, приёмник)

На приёмнике (облако для edge-событий; edge для облачных) — журнал применённых событий для идемпотентности и аудита.

| Поле `applied_log` | Назначение |
|---|---|
| `event_id` (UNIQUE) | дедуп: повторная доставка → no-op (разд. 1.4) |
| `applied_hlc`, `received_at` | момент применения / приёма |
| `origin_node, peer` | от кого получено |
| `result` | `applied \| deduped \| conflict \| deferred \| rejected` |
| `conflict_id` | ссылка на `conflict`, если разрешение отложено в UI (разд. 3.3) |

> **Аудит «бесплатно».** Event-sourcing книговыдачи + `applied_log` дают полный журнал «кто/что/когда/откуда» без отдельной подсистемы аудита ([ADR-005 «Последствия»](../../SYNC_DESIGN.md)). Координация с журналом доступа 152-ФЗ (I6 B4).

### 1.4 Идемпотентность по `event_id`

- Приёмник держит множество обработанных `event_id` (UNIQUE-constraint на `applied_log.event_id`). Применение — `INSERT … ON CONFLICT DO NOTHING` + применение эффекта **в той же транзакции** → повтор не двоит эффект.
- Бизнес-операции несут собственный idempotency-ключ, производный от `event_id` (напр. «выдать экземпляр» идемпотентна по `event_id`, повторная доставка не создаёт второй loan).
- Транспортный ACK по `seq` (разд. 4) — отдельно от прикладной идемпотентности по `event_id`: ACK двигает курсор, `event_id` защищает эффект.

### 1.5 Tombstones и компакция

- **Удаление = событие `op:delete` + tombstone** (`deleted=true, deleted_hlc`), а не физическое удаление строки сразу. Tombstone живёт ≥ ретеншена (гарантия, что все пиры узнали об удалении до физической чистки), затем компактится.
- **Event-sourced домены (книговыдача) не «удаляют» — компенсируют.** Возврат не удаляет факт выдачи; он добавляет факт-возврат. Состояние = свёртка. Это снимает класс конфликтов «удалил/изменил».
- **Компакция журнала:** для CRUD-доменов хранится последнее состояние + хвост событий после снапшот-границы; для event-sourced — снапшот свёртки на дату + события после неё (как в обычном event-store). Компакция не должна обгонять самый отстающий активный пир (разд. 4.5).

### 1.6 Эволюция схемы

- `schema_ver` на событии; приёмник умеет читать `>= min_supported`; неизвестные поля сохраняются (forward-compat, не теряются при пере-эмиссии). Изменение payload-структуры домена — миграция за версией, апгрейд узлов через fleet/OTA (I4-E) с skew-окном (разд. 7).

## 2. Гибридные логические часы (HLC)

> ADR-005 §3: **не доверяем wall-clock на edge.** HLC даёт причинный порядок + человекочитаемую метку, устойчив к перекосу часов и переупорядочиванию.

### 2.1 Представление

`HLC = (pt, c, node)` где `pt` — физическое время (ms, wall-clock-приближение), `c` — логический счётчик (uint), `node` — id узла (тай-брейк). Сериализация: `pt_hex:c:node` (лексикографически сортируема ≈ причинно).

### 2.2 Алгоритм (Kulkarni HLC)

```
now() = physical wall-clock ms (может прыгать/отставать)

on local event:
  pt' = max(pt_last, now())
  c'  = (pt' == pt_last) ? c_last + 1 : 0
  (pt_last, c_last) = (pt', c')
  return (pt', c', node)

on receive event with remote HLC (rpt, rc):
  pt' = max(pt_last, rpt, now())
  if pt' == pt_last == rpt:  c' = max(c_last, rc) + 1
  elif pt' == pt_last:       c' = c_last + 1
  elif pt' == rpt:           c' = rc + 1
  else:                      c' = 0
  (pt_last, c_last) = (pt', c')
```

### 2.3 Монотонность и присвоение

- **Локальная монотонность гарантирована:** `c` инкрементируется, когда `now()` не продвинулся (или откатился) → HLC никогда не убывает на узле, даже если NTP откатил часы назад.
- HLC присваивается **в момент коммита доменной транзакции** (тот же, что пишет outbox-строку), чтобы порядок событий узла = порядок их HLC.
- `node` в составе — финальный тай-брейк: две операции с равными `(pt,c)` упорядочены детерминированно по `node` (важно для воспроизводимости LWW и хаос-тестов).

### 2.4 Перекос часов (clock skew)

- HLC ограничивает дрейф `pt` от реального времени логическим счётчиком, но **счётчик не бесконечен**: при большом перекосе (узел опередил реальное время на дни) — `pt` «застревает» в будущем. Защита: **порог skew** (напр. ±5 мин относительно облака при онлайне) → при превышении узел логирует `clock_skew_alert` (staff/телеметрия I4-E), но **не блокирует офлайн-работу** (книговыдача должна идти). Облако при приёме фиксирует `recv_pt - event_pt` для наблюдаемости.
- **HLC не зависит от точности NTP для корректности** — только для читаемости метки. Корректность порядка (причинность) держится на `c` и `causal_deps`, не на абсолютном времени. Это и есть устойчивость к перекосу.

### 2.5 Сравнение и слияние

- **Сравнение:** `(pt, c, node)` лексикографически. `a < b` ⇒ `a` причинно-до-или-параллельно `b`.
- **Happens-before vs параллельность:** HLC даёт *тотальный* порядок, но **не различает причинность и параллельность** сам по себе. Для детекции *настоящего* конфликта (параллельные правки) используется `base_version` + `causal_deps` (разд. 3.1), а HLC — для **разрешения** (LWW-тай-брейк) и упорядочивания применения. Это ключ: HLC решает «кто победил», `base_version` решает «есть ли конфликт».

## 3. Разрешение конфликтов + UI

### 3.0 Политика по доменам (таблица владения)

Расширение [ADR-005 §1/§5](../../SYNC_DESIGN.md) до исполнимой матрицы. **Политика выбирается по `domain`, не глобально.**

| Домен | Master | Направление | Детекция | Политика разрешения | Конфликт-UI |
|---|---|---|---|---|---|
| `catalog` (записи) | **облако** | edge→облако = **предложения**; облако→edge = реплика | `base_version` правки edge ≠ облако | **облако побеждает**; правка edge → **очередь ревью** каталогизатора | да: очередь предложений (3.4) |
| `authority` (авторитетные файлы) | **облако** | как catalog | как catalog | облако-master; правка edge → ревью | да: очередь ревью |
| `dict` (словари `.mnu`/`.tre`, A5-сидинг) | **облако** | облако→edge (read) | edge не правит | облако — истина; edge read-only | нет (edge не источник) |
| `circulation` (выдача/возврат/бронь/перемещения) | **edge** | edge→облако (агрегат) | **event-sourced — конфликта нет** (факты складываются) | свёртка событий; межузловой спор по экземпляру → **авторитет локации** (3.2) | редко: split-brain экземпляра (3.2) |
| `holding` (экземпляры, ячейки) | **edge** (локация) | edge→облако | инвариант «ячейка занята одним экземпляром» — локальный | edge-владелец локации авторитетен; межузловой — авторитет локации | редко |
| `reader` (RDR) | двусторонне (edge на месте ↔ облако агрегат) | edge↔облако | `base_version` параллельной правки поля | **LWW по HLC** на уровне поля + журнал конфликтов | да: журнал конфликтов (3.4) |
| `config` (тариф/модули) | **облако** | облако→edge (read) | edge не правит | облако — истина | нет |
| `license` | **облако** | облако→edge (read, подписано) | edge не правит | облако — истина; офлайн-grace (разд. 7) | нет |

> **Итог модели:** каталог един и консистентен (облако-master), оперативная книговыдача корректна офлайн (edge-master, события), редкие поля-конфликты (reader) — LWW-HLC + видимый журнал. **Почему это безопасно:** перезаписываемое состояние принадлежит ровно одному master на домен; то, что пишется конкурентно с двух сторон (книговыдача), — event-sourced и коммутативно.

### 3.1 Детекция конфликта (три уровня)

- **Record-level (`base_version`):** событие несёт `base_version` сущности. Если на приёмнике текущая версия ≠ `base_version` и обе стороны меняли — параллельная правка ⇒ конфликт-кандидат. Без расхождения версий — fast-forward, конфликта нет.
- **Field-level:** для CRUD-доменов (reader) дельта несёт изменённые поля; конфликт только если **пересекаются множества изменённых полей** обеих сторон. Непересекающиеся поля сливаются автоматически (3.3, merge). Это резко снижает ложные конфликты vs record-level.
- **Business-rule-level:** доменный инвариант, который нельзя разрешить версией/HLC. Примеры: «экземпляр выдан на узле A и одновременно на узле B» (split-brain, 3.2); «бронь подтверждена дважды на разных узлах». Детектируется доменным предикатом при свёртке → конфликт бизнес-уровня (всегда в UI/авторитет локации, не LWW).
- **Event-sourced домены конфликта record-level не имеют:** факты не несут `base_version` состояния, они аппендятся; «конфликт» возможен только бизнес-уровня (3.2).

### 3.2 Split-brain экземпляра (книговыдача, бизнес-конфликт)

Редкий, но критический: один физический экземпляр «выдан» на двух офлайн-узлах (или на узле и в облаке) за время рассинхрона.

- **Гарантия инварианта локально:** на одном узле «ячейка/экземпляр занят одним loan» — транзакционный constraint. Межузловой спор возможен только если экземпляр физически перемещался без синхронизации.
- **Разрешение — авторитет локации (`location authority`):** экземпляр принадлежит локации (узлу), где он *физически* находится; событие выдачи от узла-владельца локации авторитетно, конкурирующее с другого узла → помечается `superseded` + создаётся `conflict` бизнес-уровня для staff-разбора (а не молчаливо теряется). Компенсирующее событие (отмена ошибочной выдачи) аппендится — история сохранна.
- Это не LWW по времени: побеждает **владелец локации**, не «последний по часам» (две выдачи могут быть «одновременны» по HLC).

### 3.3 Модель данных `conflict`

```jsonc
{
  "id": "uuid",
  "tenant_id": "spb-gtb",
  "domain": "reader|catalog|circulation",
  "entity": "RDR", "entity_id": "RDR:1234",
  "type": "field|record|business_rule",
  "field_path": "v32^a",                 // для field-level; null иначе
  "local_version":  { "hlc": "...", "origin_node": "cloud",        "value": {...}, "base_version": 7 },
  "remote_version": { "hlc": "...", "origin_node": "edge:node-01",  "value": {...}, "base_version": 7 },
  "auto_resolution": "lww_hlc|location_authority|none",  // что движок сделал бы/сделал
  "auto_winner": "remote",               // null если требует ручного разбора
  "status": "open|auto_resolved|resolved_manual|superseded",
  "resolution": { "choice": "local|remote|merged", "merged_value": {...}, "actor": "user:cat-07", "at": "hlc" },
  "created_hlc": "...", "resolved_hlc": "...",
  "audit": [ /* кто открыл/разрешил/переоткрыл — журнал */ ]
}
```

- **Авто-разрешённые** (LWW-HLC по полю, непересекающиеся merge) — `status:auto_resolved`, попадают в журнал, но **не требуют** действия; видны в UI для постфактум-аудита/отката.
- **Ручные** (`catalog` предложения, `business_rule`, опц. поля reader с флагом «всегда ревью») — `status:open` → UI разбора.

### 3.4 UI разбора конфликтов

Две поверхности, обе переиспользуют существующие виджеты:

1. **Очередь ревью каталога** (catalog/authority предложения) — для каталогизатора (ws1). Список предложений правок edge, отклонённых в пользу облака. Карточка: **обе версии side-by-side** (наша запись-облако vs предложение-edge), diff по полям/подполям (переиспользует diff-виджет редактора записи ws1), действия: **принять предложение** (применить к облаку-master → новое событие облако→edge), **отклонить** (предложение архивируется), **слить** (поле-за-полем). Аудит: кто/когда/что выбрал.
2. **Журнал конфликтов** (reader field-level, business-rule, split-brain) — для staff. Список `conflict` с фильтром по домену/статусу/узлу. Карточка: обе версии + метаданные (HLC, `origin_node`, `base_version`), действие: **выбрать local/remote/слить**, для split-brain — **подтвердить авторитет локации / откатить ошибочную выдачу** (генерит компенсирующее событие). Авто-разрешённые показаны помеченными, с возможностью **переоткрыть** (override авто-LWW).

- **Принцип UI:** показать **обе версии**, дать **выбрать или слить**, **записать аудит**. Никаких «тихих потерь» — даже авто-LWW виден и обратим.
- **API UI** — разд. 5.3.

### 3.5 Почему сходится (кратко; полный proof — 9.4)

- **Event-sourced (circulation):** факты коммутативны и идемпотентны (по `event_id`); свёртка детерминирована порядком HLC → все узлы при одинаковом множестве событий дают одно состояние. Конфликт бизнес-уровня разрешается детерминированным правилом (авторитет локации), не зависящим от порядка доставки.
- **Single-master (catalog/dict/config/license):** один писатель состояния (облако); edge-правки — предложения, не применяются к master без ревью → нет дивергенции master.
- **LWW-по-HLC (reader field-level):** HLC даёт тотальный детерминированный порядок (с тай-брейком по `node`) → при одинаковом множестве правок все узлы выбирают одного победителя на поле → сходятся.

## 4. Sync-протокол (транспорт, дельты, курсор, порядок)

### 4.1 Дельта «since cursor»

- Пир запрашивает изменения `since (peer_cursor)`; релейер отдаёт события `outbox.seq > cursor.seq` (свои, по политике направления домена), батчами, отсортированными по `seq`. Курсор пира = последний подтверждённый `seq` источника.
- **Двунаправленность:** edge↔облако — два независимых потока: `edge→cloud` (книговыдача/предложения/reader) и `cloud→edge` (каталог-реплика/словари/config/license/применённые reader-правки). Каждый со своим курсором.
- **Частичность (152-ФЗ + модули):** поток фильтруется `tenant_id = свой` И `module ∈ включённые`. Узел не получает/не отдаёт чужих тенантов и выключенных модулей.

### 4.2 Персистентность курсора (рестарт)

| Поле `sync_cursor` | Назначение |
|---|---|
| `peer` (PK часть) | пир (`cloud` / `edge:node-01`) |
| `direction` | `inbound \| outbound` |
| `acked_seq` | последний `seq` источника, **подтверждённо применённый** пиром |
| `acked_hlc` | соответствующий HLC (для skew-наблюдаемости) |
| `updated_at` | для детекции застрявших потоков (lag-метрика) |

- Курсор — **в БД**, продвигается **только по ACK** приёмника (применил + записал в `applied_log`). Падение узла/сети между отправкой и ACK → переотправка с `acked_seq` (idempotent по `event_id`, разд. 1.4). Гарантия: **никаких потерь при рестарте**, максимум — повтор уже применённого (no-op).
- ACK — батчевый (последний `seq` успешно применённого батча); частичный сбой батча → ACK до последнего успешного, остаток переотправляется.

### 4.3 Порядок и out-of-order

- **Внутри одного origin** события строго упорядочены по `seq` → применяются по порядку (FIFO на поток origin).
- **Между origin** порядок частичный. `causal_deps` несёт `event_id`, которые должны быть применены раньше. Приёмник:
  - если все `causal_deps` уже в `applied_log` → применить;
  - иначе → **отложить** (`result:deferred`, буфер reorder) до прихода зависимостей; периодический ретрай буфера.
- **Защита от бесконечного откладывания:** TTL буфера + alert; недостающая зависимость дольше TTL → запрос таргетной докачки у пира (разд. 4.4) либо эскалация (рассинхрон → bootstrap, 4.5).
- **Event-sourced коммутативность** снимает большинство требований к порядку: факты выдачи/возврата можно применять в любом порядке, свёртка детерминируется HLC при чтении, а не порядком приёма. `causal_deps` нужен в основном для CRUD-доменов (напр. «создание записи» до «правки записи»).

### 4.4 Backpressure, ретраи, докачка

- **Backpressure:** приёмник управляет темпом — отвечает `max_batch`/`pause` при перегрузке (queue-depth метрика); отправитель снижает batch-size/частоту. Поток не валит приёмник.
- **Batch-размер** — адаптивный (старт N=500 событий или ≤M КБ сжато, ↓ при ошибках/backpressure, ↑ при стабильности). Сжатие батча (gzip/zstd) — обязательно (узкий/дорогой канал edge).
- **Ретраи:** экспоненциальный backoff + jitter на сетевые/5xx; предел → переход в `degraded` (узел работает офлайн, копит outbox), периодический реконнект.
- **Докачка (resumable):** прерванная передача докачивается с `acked_seq` (не сначала). Большие bootstrap-снапшоты (4.5) — chunked + resumable (range/offset), устойчивы к обрыву.

### 4.5 Bootstrap (снапшот) и компакция-граница

- Новый/давно офлайн узел (курсор устарел за компакцию): **полный снапшот** состояния тенанта (по политике направления) → далее дельты `since snapshot_hlc`. Снапшот — consistent-cut (на `hlc` границе), chunked/resumable/сжатый.
- **Компакция не обгоняет пиров:** `min(acked_seq по активным пирам)` — нижняя граница компакции outbox. Узел, офлайн дольше ретеншена снапшот-границы, форсит bootstrap (а не дельту) — это корректно, не потеря.

### 4.6 Транспорт и ускорители

- **Транспорт:** sync-эндпоинт (REST/gRPC) поверх mTLS (разд. 7) **или** событийная шина I7 (#107) — за интерфейсом `Transport`. Старт — HTTP/gRPC long-poll/stream (проще на edge без брокера), миграция на шину без переписывания движка.
- **Ускорители ([ADR-005](../../SYNC_DESIGN.md)):** ElectricSQL / PowerSync (Postgres-ориентированные) — **оценить как замену слоёв 4.1–4.5** (дельты/курсор/снапшот/докачка), сохранив нашу модель владения (разд. 3.0) и event-sourcing книговыдачи поверх. Не изобретать транспорт/дельты с нуля, но и не отдавать им доменную политику конфликтов. CRDT — точечно (счётчики/множества), не на весь домен (отвергнуто в ADR-005).

## 6. Офлайн-книговыдача (edge-master, без потерь/дублей)

> Деталь C (офлайн-книговыдача) [SPEC_I4 §C](../SPEC_I4_edge_offline.md) на стороне sync; бизнес-цикл — [SPEC_ws3_circulation](../SPEC_ws3_circulation.md).

- **Запись без облака:** выдача/возврат/бронь/перемещение по ячейкам пишутся локально на edge как **факты-события** (`domain:circulation, op:event`) в доменную таблицу + outbox, в одной транзакции. Узел полностью функционален офлайн неограниченно (AC I4-A2). Локальный инвариант «экземпляр выдан одному» — транзакционный constraint.
- **Слияние при сети (без потерь):** при реконнекте relay отдаёт накопленный outbox `since acked_seq`; облако агрегирует факты; идемпотентность по `event_id` → **дубля нет** даже при повторной доставке/докачке прерванной передачи. Порядок не критичен (факты коммутативны, 4.3).
- **Без потерь:** транзакционный outbox гарантирует, что ни одна выдача не «потерялась» между записью и отправкой (`published_at IS NULL` подбирается). Курсор продвигается только по ACK → сбой не теряет события.
- **Split-brain** (один экземпляр на двух узлах) — разрешается авторитетом локации (3.2), компенсирующим событием, видим в UI — без молчаливой потери/дубля.
- **AC-связь:** I4-B AC3 (выдача складывается без конфликта — event-sourced), I4-C AC2 (дозагрузка без потерь/дублей).

## 7. Grace-лицензия офлайн

> Деталь E (fleet/OTA, офлайн-валидация лицензии) [SPEC_I4 §E](../SPEC_I4_edge_offline.md); PKI/подпись — I6 D2; облако-master лицензии (3.0).

- **Подписанная лицензия в кэше:** облако выдаёт узлу **подписанный** (per-tenant ключ, I6 D2 PKI) lease лицензии/entitlement (`tenant_id, modules[], not_after, grace_days, sig`). Узел хранит в `license_state`, **валидирует подпись офлайн** (публичный ключ облака закреплён) — не нужен онлайн для проверки.
- **Grace-окно:** при потере связи узел работает по последнему валидному lease до `not_after`; после — **grace-период** `grace_days` (узел продолжает обслуживать книговыдачу — критичную функцию — но баннер «связь с лицензированием потеряна, осталось N дней», `sync.license.grace`).
- **Поведение по истечении grace:** мягкая деградация — **книговыдача (edge-master, оффлайн-критичная) не отключается жёстко** (нельзя «запереть» читателей библиотеки из-за лицензии), но **немодульные/коммерческие функции** (новые модули, аналитика, синхронизация витрин) ограничиваются; крупный баннер + `sync.license.expired` + телеметрия оператору. Полное восстановление — при первом успешном sync (облако обновляет lease). Точная лестница деградации — параметр тарифа (координация I7 D5 биллинг).
- **Skew-политика версий:** lease несёт допустимое окно версий edge↔облако (I4-E); при выходе за окно — форс OTA-апгрейда (fleet, #120) либо отказ синхронизации несовместимых схем (1.6).
- **AC-связь:** I4-E AC2 (узел работает на просроченной связи в grace).

## 8. Критерии приёмки (AC)

- **AC1 (двусторонняя синхронизация + восстановление).** Изменения edge↔облако переносятся в обе стороны; обрыв канала на произвольной фазе → после реконнекта оба пира сходятся к консистентному состоянию, докачка с `acked_seq`, без потерь/дублей (I4-B AC1).
- **AC2 (каталог — облако-master, очередь ревью).** Правка каталога/авторитета на edge → **предложение** (не применяется к облаку молча); при расхождении облако побеждает, предложение в очереди ревью каталогизатора; принятие/отклонение/слияние видны (I4-B AC2).
- **AC3 (книговыдача — event-sourced без конфликта).** Конкурентные выдачи/возвраты на разных узлах складываются как факты, свёртка детерминирована HLC, дубля нет (идемпотентность `event_id`); split-brain экземпляра → авторитет локации + `conflict` бизнес-уровня в UI, не молчаливая потеря (I4-B AC3).
- **AC4 (поля-конфликты reader → журнал + разбор).** Параллельная правка пересекающихся полей RDR → авто-LWW по HLC, запись в журнал конфликтов; UI показывает **обе версии**, позволяет выбрать/слить/переоткрыть, пишет аудит; непересекающиеся поля сливаются автоматически (I4-B AC4).
- **AC5 (HLC / перекос часов).** При откате/прыжке wall-clock на edge HLC монотонен (порядок не ломается); порог skew → alert, но офлайн-работа не блокируется; порядок применения детерминирован при любом перекосе.
- **AC6 (out-of-order / causal).** Событие, пришедшее раньше своих `causal_deps`, откладывается и применяется после зависимостей; TTL-буфер + таргетная докачка; event-sourced факты применяются в любом порядке с одинаковой свёрткой.
- **AC7 (курсор / рестарт).** Падение узла между отправкой и ACK → после рестарта продолжение с `acked_seq`, повтор уже применённого = no-op (по `event_id`), потерь нет.
- **AC8 (backpressure / batch / докачка).** Перегрузка приёмника → backpressure снижает batch/темп без падения; прерванная передача докачивается с места, не сначала; сжатие батча работает.
- **AC9 (bootstrap / компакция).** Давно офлайн узел (курсор за компакцией) форсит consistent-cut снапшот → далее дельты; компакция не обгоняет самый отстающий активный пир.
- **AC10 (офлайн-книговыдача).** Полный цикл выдачи офлайн (I4-C AC1); при сети — слияние без потерь/дублей даже при повторной доставке/докачке (I4-C AC2).
- **AC11 (grace-лицензия).** Узел валидирует подписанную лицензию офлайн; работает в grace по истечении связи; по истечении grace — мягкая деградация (книговыдача не отключается), баннер + телеметрия; восстановление при sync (I4-E AC2).
- **AC12 (изоляция тенанта).** Узел/курсор/конфликты тенанта A недоступны B; поток фильтруется по `tenant_id` + включённым модулям (152-ФЗ + DoD-гейт).
- **AC13 (сходимость — хаос).** Хаос-сценарии (разд. 9) сходятся к одному состоянию во всех узлах; convergence proof выполняется в CI как гейт (разд. 9.4).
- **AC14 (наблюдаемость).** Метрики lag (seq/время), queue-depth, conflict-rate, clock-skew, degraded-peers экспонируются; trace несёт `tenant_id`/`origin_node`; alert при застрявшем потоке/росте конфликтов/skew.

## 9. Хаос-тесты и доказательство сходимости

> Обязательный гейт C2/I4 ([ADR-005 §8](../../SYNC_DESIGN.md), [SPEC_I4 КП](../SPEC_I4_edge_offline.md)). «Автоматические хаос-тесты конвергенции и инвариантов».

### 9.1 Сценарии (chaos)

| Сценарий | Инъекция | Проверяет |
|---|---|---|
| **Обрыв канала** | kill соединения на произвольной фазе pull/push/ack | докачка с `acked_seq`, нет потерь/дублей (AC1, AC7) |
| **Перекос часов** | сдвиг wall-clock узла ±дни, откат NTP | HLC-монотонность, детерминизм порядка (AC5) |
| **Дубли** | повторная доставка батча / двойной CDC | идемпотентность `event_id`, no-op (AC3, AC10) |
| **Переупорядочивание** | доставка событий вне `seq`/без `causal_deps` | reorder-буфер, отложенное применение, та же свёртка (AC6) |
| **Долгий офлайн** (недели) | узел офлайн дольше компакции | bootstrap-снапшот вместо дельты, консистентность (AC9) |
| **Конкурентная правка** | параллельная правка одного поля reader на двух узлах | LWW-HLC детерминирован, журнал+UI (AC4) |
| **Split-brain экземпляра** | выдача одного EX на двух офлайн-узлах | авторитет локации, компенсация, `conflict` (AC3) |
| **Конкурентная правка каталога** | правка записи на edge + на облаке | облако-master, предложение в ревью (AC2) |
| **Прерванный bootstrap** | обрыв снапшота на середине | resumable докачка снапшота (AC8, AC9) |
| **Backpressure-шторм** | всплеск событий > пропускной | backpressure, без падения приёмника (AC8) |

### 9.2 Свойства-инварианты (property-based)

Проверяются на **случайных** перестановках/задержках/дублях (property-based testing, напр. `hypothesis`):

- **I-конвергенция:** для любого множества событий и любого порядка/задержек доставки все узлы достигают одинакового состояния (eventual consistency).
- **I-идемпотентность:** применение события N раз ≡ один раз.
- **I-отсутствие-потерь:** каждое закоммиченное событие источника в итоге применено на всех уполномоченных пирах (или явный `conflict`).
- **I-инвариант-книговыдачи:** ни один экземпляр не «выдан дважды» в финальном консистентном состоянии (split-brain → разрешён, не оставлен).
- **I-владение:** состояние master-домена меняется только его master’ом (caталог-облако не перезаписан edge’ем напрямую).

### 9.3 Симулятор сети (deterministic)

- Детерминированный симулятор: N узлов + хаб, управляемые задержки/потери/дубли/переупорядочивание/перекос-часов с фиксированным сидом → **воспроизводимые** падения. Тай-брейк HLC по `node` делает прогон детерминированным при одном сиде.

### 9.4 Convergence proof (CI-гейт)

Формальный критерий, проверяемый автоматически в CI:

1. **Event-sourced домены** (circulation/holding) — состояние = детерминированная свёртка коммутативного, идемпотентного множества фактов, упорядоченного тотальным HLC; конфликт бизнес-уровня разрешается порядко-независимым правилом (авторитет локации) ⇒ **строгая сходимость** независимо от порядка/дублей.
2. **Single-master домены** (catalog/dict/config/license) — ровно один писатель состояния ⇒ нет дивергенции master по построению; реплики — функция от master.
3. **LWW-HLC домены** (reader field-level) — HLC + тай-брейк `node` = детерминированная функция выбора на множестве правок поля ⇒ все узлы при одинаковом множестве выбирают одно ⇒ сходимость.
4. **Гейт:** property-тест (9.2 I-конвергенция) + симулятор (9.3) на наборе сценариев (9.1) должны давать идентичное финальное состояние всех узлов; расхождение = провал сборки.

## 10. Критический путь

1. **Схема события + outbox (транзакционный захват) + HLC** (#125) → события захватываются без потерь, упорядочены причинно. →
2. **applied_log + идемпотентность по `event_id` + tombstones** → повтор/удаление безопасны. →
3. **Sync-протокол: pull/push/ack + персистентный курсор + дельта since-cursor** (#126) → двусторонний перенос с восстановлением после обрыва (AC1, AC7). →
4. **Доменное владение: catalog cloud-master (предложения) / circulation edge-master (event-sourced)** + детекция конфликта (`base_version`/field/business-rule) → AC2, AC3. →
5. **Offline-книговыдача end-to-end** (запись офлайн → слияние при сети без потерь/дублей) → AC10. →
6. **Конфликт-модель `conflict` + UI разбора** (очередь ревью каталога ws1 + журнал конфликтов staff: обе версии/выбрать/слить/аудит) (#127) → AC4. →
7. **Backpressure/batch/докачка/bootstrap-снапшот + компакция-граница** → AC8, AC9. →
8. **Grace-лицензия офлайн** (подписанный lease, валидация офлайн, grace, деградация) + skew-политика версий (I4-E) → AC11. →
9. **Хаос-тесты + convergence proof** (симулятор сети, property-тесты, CI-гейт) → AC13. Параллельно с 1–8, ужесточается на каждом шаге.

**Выходной критерий движка C2:** библиотека-узел работает офлайн неограниченно (книговыдача edge-master, event-sourced) и при появлении сети **двусторонне синхронизируется** с облаком — события захватываются транзакционным outbox, упорядочены HLC (устойчиво к перекосу часов), применяются идемпотентно по `event_id` (без потерь/дублей при обрыве/докачке/переупорядочивании), каталог остаётся облако-master (правки edge → очередь ревью), конфликты полей разрешаются LWW-по-HLC с **видимым журналом и UI разбора** (обе версии/выбрать/слить/аудит), лицензия валидируется офлайн с grace-окном, а **хаос-тесты доказывают сходимость** всех узлов к одному состоянию — закрывая wireframe-уровень C2 ([DESIGN_COVERAGE §C2](../../DESIGN_COVERAGE.md)) до реализуемости и реализуя самый рисковый differentiator архитектуры ([ARCHITECTURE §13](../../ARCHITECTURE.md)).

## 12. Риски

| Риск | Митигация |
|---|---|
| **Сложность распределённой синхронизации/конфликтов** (главный риск архитектуры, [ARCHITECTURE §13](../../ARCHITECTURE.md)) | Асимметричное владение по доменам (3.0) сводит конфликты к минимуму: event-sourcing книговыдачи (нет конфликта), single-master каталога (нет дивергенции), LWW-HLC только для редких полей reader; convergence proof + хаос-гейт |
| **Перекос часов на edge ломает порядок** | HLC (корректность не зависит от точности NTP, 2.4); порог skew + alert; порядок держится на `c`+`causal_deps`, не на wall-clock |
| **at-least-once транспорт → дубли** | Сквозная идемпотентность по `event_id` (`INSERT ON CONFLICT`), курсор по ACK; *effectively-once* |
| **Split-brain книговыдачи** (один экземпляр на двух узлах) | Локальный инвариант + межузловой авторитет локации (3.2) + компенсирующее событие + видимый `conflict`; редкость (требует физ. перемещения без sync) |
| **Тихая потеря данных при конфликте** | Все конфликты материализуются (`conflict`) и видимы в UI; авто-LWW обратим (переоткрыть); нет молчаливых перезаписей |
| **Компакция обгоняет отстающий узел → потеря** | Компакция ≤ `min(acked_seq)` активных пиров; bootstrap-снапшот для слишком отставших (4.5) |
| **Узкий/дорогой/нестабильный канал edge** | Сжатие батча, адаптивный batch, backpressure, resumable-докачка, дельты вместо полных синков |
| **Событийная шина (#107) ещё проектируется** | Транспорт за интерфейсом `Transport` (REST/gRPC старт; шина позже), как и outbox/CDC за `OutboxSource` — миграция без переписывания движка |
| **Готовые движки (ElectricSQL/PowerSync) могут навязать свою конфликт-модель** | Оценить только как ускоритель слоёв 4.1–4.5; доменное владение/event-sourcing/конфликт-UI — наши, поверх; не отдавать им политику (3.0) |
| **152-ФЗ: ПДн RDR в синхронизации** | Только свой тенант + включённые модули; шифрование in-transit (mTLS)/at-rest; координация I6 B4 (журнал доступа, право на забвение → tombstone каскадит) |
| **Grace-лицензия: «запереть» библиотеку нельзя, но и не торговать бесконтрольно** | Книговыдача (критичная) в grace не отключается; ограничиваются коммерческие/немодульные функции; лестница деградации — параметр тарифа (I7 D5) |

## 13. Открытые вопросы (top-3 + прочие)

- **OQ-1 (granularity конфликта reader / CRDT-точечно).** Field-level LWW-HLC выбран для RDR. Открыто: какие именно поля RDR требуют **always-review** (а не авто-LWW) — напр. финансовые (штрафы/баланс) могут требовать суммирующего CRDT-счётчика, а не LWW (потеря денег недопустима). Решить per-field-политику reader/holding на пилоте (СПб ГТБ); ADR-005 допускает «точечный CRDT для счётчиков/множеств».
- **OQ-2 (ElectricSQL/PowerSync vs самописный транспорт).** Дать/нет ускорителю слои 4.1–4.5 — критическое решение объёма реализации #126. Требует PoC-оценки: ложатся ли они на схему-на-тенанта + event-sourcing книговыдачи + асимметричное владение, или навязывают свою (LWW-CRDT) модель, несовместимую с авторитетом локации. Гейт перед задачей #126.
- **OQ-3 (лестница деградации grace + skew-окно версий).** Точная политика «что отключается по истечении grace» (книговыдача — нет; что — да) и допустимое окно версий схемы edge↔облако (когда форсить OTA vs отказать в синхронизации) — координация с I7 D5 (биллинг/enforcement) и I4-E (#120 fleet/OTA). Сейчас задан принцип (книговыдача не отключается), не точные пороги.
- Прочее: формат `causal_deps` (полный список `event_id` vs векторная компактификация) при росте журнала; порог компакции/ретеншена tombstone (баланс «офлайн недели» vs размер журнала); граница между этим движком и union-каталогом/resource-sharing (I4-D, CN #181) — межтенантный обмен поверх sync; точный outbox vs CDC (logical replication) на проде.

## Сводка

**Event-schema:** иммутабельное событие `{event_id(uuid-v7), hlc, seq, tenant_id, domain, entity, entity_id, op, payload, base_version, origin_node, causal_deps, schema_ver, module}`, захватываемое **transactional outbox** (в одной транзакции с изменением данных — без 2PC, без потерь), журналируемое на приёмнике (`applied_log`) с идемпотентностью по `event_id` (*at-least-once → effectively-once*), удаления через tombstones; время — **HLC** (Kulkarni, монотонный при откате NTP, тай-брейк по `node`), дающий тотальный причинный порядок при перекосе часов.

**Conflict-resolution:** **асимметрично по доменам** (не глобальный wall-clock-LWW) — каталог/авторитеты/словари/config/license = **облако-master** (правки edge → **очередь ревью**), книговыдача/экземпляры/ячейки = **edge-master, event-sourced** (факты коммутативны → почти без конфликта; split-brain экземпляра → **авторитет локации** + компенсация), reader-поля = **LWW-по-HLC** на уровне поля (непересекающиеся поля авто-сливаются) + **видимый журнал конфликтов**; всякий неавтоматический конфликт → модель `conflict` + **UI разбора** (обе версии side-by-side, выбрать/слить/переоткрыть, аудит). Транспорт — pull/push/ack с персистентным курсором, дельтами since-cursor, reorder по `causal_deps`, backpressure, resumable-докачкой, bootstrap-снапшотом; офлайн-книговыдача без потерь/дублей; офлайн-grace лицензии (подписанный lease, мягкая деградация); **хаос-тесты + convergence proof** как CI-гейт.

**Top-3 открытых вопроса:**
1. **Гранулярность конфликта reader/holding** — какие поля требуют CRDT-счётчик (финансы/штрафы — потеря денег недопустима) вместо LWW-HLC, какие — always-review; решить per-field на пилоте.
2. **ElectricSQL/PowerSync vs самописный транспорт** (#126) — ложатся ли ускорители на схему-на-тенанта + event-sourcing + асимметричное владение, или навязывают свою LWW-CRDT-модель; PoC-гейт перед реализацией.
3. **Лестница деградации grace-лицензии + skew-окно версий** — точные пороги «что отключается по истечении grace» (книговыдача — нет) и когда форсить OTA vs отказывать в синхронизации несовместимых схем; координация I7 D5 + I4-E (#120).
