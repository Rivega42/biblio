# SPEC · Accessibility-first — программа доступности (WCAG 2.2 AA + ГОСТ Р 52872-2019)

> Differentiator **C4** из [DESIGN_COVERAGE §C](../../DESIGN_COVERAGE.md) (эпик [#188](https://github.com/Rivega42/biblio/issues/188)). Закрывает дыру: «Accessibility-first (сейчас ❌ hollow) — реальная a11y-задача: рубрика WCAG 2.2 AA, axe/Lighthouse CI-гейт, VPAT, ГОСТ Р 52872, DAISY/аудио». Превращает **необоснованное утверждение** [SPEC_reader §7 AC3](../SPEC_reader_jirbis.md) («WCAG 2.2 AA/ГОСТ-аудит проходит») в проверяемую программу с задачей, рубрикой, CI-гейтом, протоколом и владельцем. Опора: [TZ_CLAUDEDESIGN_design-system §1](../../TZ_CLAUDEDESIGN_design-system.md) (доступность с нуля, contrast-guards, `prefers-reduced-*`), [BEST_PRACTICES](../../BEST_PRACTICES.md), [SPEC_I1 §#118 CI](../SPEC_I1_foundation.md). Родитель — **I2** (читательский веб) + **I6** (комплаенс/гос-рынок). Гос-привязка: [SPEC_compliance_152fz](../gov/SPEC_compliance_152fz.md) (аттестация гос-сегмента). DoD-гейт как в I1.
>
> ⚠ **Защищённый контур:** приложение **самохостит шрифты, без внешних CDN/трекеров** ([TZ §1](../../TZ_CLAUDEDESIGN_design-system.md)). Это влияет на дизайн CI-гейта (axe/Lighthouse исполняются локально на собранном артефакте, без обращения к сети), на screen-reader-метки (не полагаемся на CDN-иконшрифты) и на аудио/DAISY (ассеты тоже самохост).

## 0. Принципы и границы

- **Доступность — функция, а не довесок.** Каждый компонент дизайн-системы поставляется с a11y-контрактом (роли, фокус, клавиатура, метки, reduced-motion) и не считается «готовым» без него. Связь: [TZ §1](../../TZ_CLAUDEDESIGN_design-system.md) «Доступность с нуля».
- **Цель измерима и тестируема.** «Доступно» = (а) автоматический гейт зелёный (0 critical axe + Lighthouse a11y ≥ порога) **И** (б) ручной AT-аудит ключевых маршрутов пройден **И** (в) опубликован VPAT/декларация. Без всех трёх утверждение C4 остаётся hollow.
- **Два стандарта, один свод требований.** WCAG 2.2 AA — международная база (разд. 1); ГОСТ Р 52872-2019 — РФ-надстройка для гос-закупок (разд. 1.4). Мы реализуем **объединение** требований и документируем расхождения, а не выбираем одно.
- **Гейт ≠ соответствие.** Автоматика ловит ~30–40% нарушений ([axe-core docs]; consensus у Deque/W3C). Поэтому CI-гейт (разд. 3) — **необходимое, но не достаточное** условие; ручной протокол (разд. 4) обязателен. Спека явно не выдаёт «axe зелёный» за «WCAG AA».
- **Брендирование не ломает доступность.** Мультиарендная кастомизация ([TZ §3](../../TZ_CLAUDEDESIGN_design-system.md)) перекрашивает accent/поверхности, но **contrast-guards** гарантируют AA после любого seed-цвета; статус — цвет **+ иконка + текст** (не только цвет). A11y-паттерны и статус-семантика — вне зоны кастомизации тенанта.
- **Границы.** Эта спека определяет **программу доступности** (стандарт-таргет, компонентные требования, CI-гейт, протокол, VPAT, reading-disabilities). Она **не** определяет токены/темы (это [TZ_CLAUDEDESIGN](../../TZ_CLAUDEDESIGN_design-system.md)), **не** редактирует `.github/workflows/ci.yml` (описывает джоб для оркестратора, разд. 3.4), **не** определяет AI-навигатор/Cmd+K (#162 — лишь как доступный компонент). Стиль B (Apple Glass, маркетинг-сайт) подчиняется тем же критериям через фолбэки (`prefers-reduced-transparency/-motion`), но его hero-анимации — отдельный риск (разд. 9 R3).

## 1. Стандарт-таргет

### 1.1 WCAG 2.2 уровень AA — обязательная база

Целевой уровень — **WCAG 2.2 AA** (включает все A + AA; 2.2 добавляет 9 новых критериев поверх 2.1). Ниже — **ключевые success criteria, релевантные нашим экранам** (читательский OPAC, ЛК, инструменты сотрудника/АРМ, админка, киоск/постамат), сгруппированные по теме задачи. Полный чек-лист всех 56 AA-критериев — в рубрике (разд. 2 + Приложение A эталонного ruleset).

| Тема | SC (WCAG 2.2) | Уровень | Где у нас критично |
|---|---|---|---|
| **Контраст текста** | 1.4.3 Contrast (Minimum) ≥ 4.5:1 (крупный ≥ 3:1) | AA | весь UI; **contrast-guards** для tenant seed ([TZ §3](../../TZ_CLAUDEDESIGN_design-system.md)) |
| **Контраст не-текста** | 1.4.11 Non-text Contrast ≥ 3:1 | AA | границы инпутов, иконки статуса (выдано/бронь/в ячейке), фокус-кольцо, чарты BI |
| **Изменение размера** | 1.4.4 Resize Text 200%, 1.4.10 Reflow 320px | AA | каталогизация (плотные формы), DataTable, карточка записи |

| **Предсказуемость** | 3.2.1 On Focus, 3.2.2 On Input, 3.2.3/3.2.4 Consistent Nav/Identification, **3.2.6 Consistent Help** ⟵ 2.2 | A/AA | поиск-по-вводу не перебрасывает фокус; **Help-портал #142** в одном месте |

### 1.2 Чего НЕ берём (явно)

- **AAA целиком** — не таргет (нереалистично для плотных АРМ-таблиц). Но **выборочно** тянемся к AAA там, где дёшево и ценно для нашей аудитории: 1.4.6 Contrast (Enhanced) 7:1 — как опция в теме «высокий контраст» (разд. 6); 2.3.3 Animation from Interactions — фактически выполняем через `prefers-reduced-motion`; 2.4.8 Location (хлебные крошки).
- **Уровень A в одиночку** — недостаточен для гос-закупок; берём AA целиком.

### 1.3 Привязка таргета к нашим поверхностям

| Поверхность | Стиль ([TZ §4](../../TZ_CLAUDEDESIGN_design-system.md)) | Профиль доступности |
|---|---|---|
| Читательский веб-OPAC, ЛК | A (Claude-like) | WCAG 2.2 AA + ГОСТ полностью (публичный — макс. требования) |
| Инструменты сотрудника / АРМ, каталогизация | A, плотность compact | AA; плотность не отменяет 2.5.8 Target Size (24px-исключение для табличных контролов задокументировать) |
| Админка / редактор тем | A | AA + contrast-guard как часть продукта |
| Киоск самообслуживания / постамат | A, large-touch | AA + усиленный Target Size (≥ 44px рекомендация Apple/WCAG AAA 2.5.5), голосовые/аудио-подсказки |
| Маркетинг-сайт | B (Apple Glass) | AA **через фолбэки** reduced-transparency/-motion; контраст текста поверх стекла гарантирован затемняющим слоем |

### 1.4 ГОСТ Р 52872-2019 — РФ-надстройка

**ГОСТ Р 52872-2019 «Интернет-ресурсы и другая информация, представленная в электронно-цифровой форме. Приложения для стационарных и мобильных устройств, иные пользовательские интерфейсы. Требования доступности для людей с инвалидностью и других лиц с ограничениями жизнедеятельности»** — действующий национальный стандарт РФ, **гармонизированный с WCAG 2.0/2.1** (базируется на тех же четырёх принципах: воспринимаемость, управляемость, понятность, надёжность). Для гос-закупок и аттестации (вузы, гос-библиотеки — наш целевой рынок) это **формальное требование закупки**, тогда как WCAG — отраслевой стандарт де-факто.

**Соответствие/различия (что фиксируем в матрице, разд. 1.5):**

- **Базис.** ГОСТ Р 52872-2019 = РФ-адаптация POUR-принципов; в части success criteria сильно пересекается с WCAG 2.1 AA. Реализовав WCAG 2.2 AA, мы покрываем подавляющее большинство пунктов ГОСТ.
- **Дельты ГОСТ → отдельные требования:**
  - **Версия для слабовидящих (БВВ).** ГОСТ исторически связан с требованием «версии сайта для слабовидящих» (увеличенный шрифт, ограниченная палитра ч/б/контраст, отключаемые изображения). Мы реализуем это **темой «высокий контраст» + крупный шрифт** ([TZ §6](../../TZ_CLAUDEDESIGN_design-system.md), разд. 6 ниже) как режим основного UI, **а не как урезанный отдельный сайт** (современная трактовка: один доступный продукт ≫ деградированная «версия для слепых»). Это решение и его обоснование документируем для аудитора закупки (разд. 9 R5).
  - **Терминология/формулировки** ГОСТ оперируют категориями инвалидности (по зрению, слуху, опорно-двигательные, когнитивные) — маппим на наши тест-персоны (разд. 4.2).
  - **Декларация о соответствии** в РФ-форме (не VPAT, а заявление о соответствии ГОСТ) — шаблон в разд. 5.
- **Расхождения, требующие явного решения:** ГОСТ ссылается на WCAG 2.0/2.1; критерии, **добавленные** в 2.2 (разд. 1.1), формально вне ГОСТ — мы их выполняем сверх (плюс к закупке), но в матрице помечаем «WCAG-only». Точная редакция применимого ГОСТ и перечень его пунктов уточняются у юриста закупки (**TODO-A11Y-01**, разд. 9).

### 1.5 Матрица соответствия стандартам (артефакт)

Машиночитаемая таблица `a11y/standards-matrix.json` — единый источник «что чему соответствует». Поддерживается a11y-экспертом (разд. 7), ревизуется при аудите.

```jsonc
      "components": ["Button", "Input", "StatusBadge", "BI-Charts"],
      "status": "planned"                  // planned | implemented | verified
    }
    // … все 56 AA-критериев + 5 новых 2.2
  ],
  "gost_deltas": [
    { "topic": "версия для слабовидящих", "decision": "тема высокий-контраст+крупный шрифт", "ref": "разд.6" }

```

**AC-якорь:** существование и заполненность этой матрицы — измеримое условие, что C4 не hollow (каждый критерий имеет владельца-проверку, не «галочку»).

## 2. Доступность на уровне компонентов (DS-контракт)

Связь: дизайн-система DS (#150/#153/#154, [TZ §6](../../TZ_CLAUDEDESIGN_design-system.md)). **Каждый компонент** инвентаря DS поставляется с a11y-контрактом и **a11y-аннотацией в галерее** ([TZ §9](../../TZ_CLAUDEDESIGN_design-system.md) — «a11y-аннотации: роли/клавиатура/контраст»). Ниже — общий чек-лист на компонент + специфика по группам.

### 2.1 Чек-лист на компонент (применять к каждому в галерее)

```
[ ] СЕМАНТИКА    нативный элемент где можно (<button>/<a>/<input>/<nav>/<table>);
                 ARIA только когда нативного нет; верный role/aria-* (4.1.2)
[ ] ИМЯ          доступное имя у каждого интерактива (label/aria-label/aria-labelledby);
                 имя содержит видимый текст (2.5.3 Label in Name)
[ ] ФОКУС        фокусируем с клавиатуры; видимое фокус-кольцо ≥3:1 (2.4.7/1.4.11);
                 не перекрыт sticky (2.4.11); логичный порядок (2.4.3)
[ ] КЛАВИАТУРА   полная операбельность без мыши (2.1.1); нет ловушки (2.1.2);
                 ESC закрывает оверлеи; стрелки/Home/End в композитных виджетах (APG)
[ ] СОСТОЯНИЯ    disabled/selected/expanded/checked/invalid отражены в ARIA, не только цветом
[ ] СТАТУС       динамика объявляется screen-reader (aria-live/role=status/alert) (4.1.3)
[ ] КОНТРАСТ     текст ≥4.5:1, не-текст/границы/иконки ≥3:1 (1.4.3/1.4.11) во ВСЕХ темах
[ ] TARGET       цель указателя ≥24×24px (2.5.8); ≥44px в kiosk/large-touch
[ ] MOTION       анимации уважают prefers-reduced-motion; нет авто-движения >5с без паузы (2.2.2)
[ ] ОШИБКИ       (для форм) ошибка у поля, текстом+иконкой, с подсказкой (3.3.1/3.3.3),
                 aria-describedby связывает поле с сообщением, aria-invalid
[ ] RTL/i18n     не ломается на длинных строках/англ.; lang на иноязычных фрагментах
[ ] ZOOM         работает при 200% и reflow 320px без потери функций (1.4.4/1.4.10)
```

### 2.2 Требования по группам компонентов

Маппинг на инвентарь [TZ §6](../../TZ_CLAUDEDESIGN_design-system.md). Опора на **WAI-ARIA Authoring Practices (APG)** паттерны — не изобретать клавиатурные модели.

| Группа DS | Компоненты | Ключевые a11y-требования |
|---|---|---|
| **Примитивы** | Button, IconButton, Input, Select/Combobox, Checkbox/Radio/Switch, Chip, Tooltip, Spinner/Progress | нативные теги; IconButton — обязательный `aria-label`; Combobox — APG combobox (`aria-expanded`/`aria-activedescendant`/стрелки); Switch — `role=switch aria-checked`; Tooltip — 1.4.13 (наводимо/фокусируемо/закрываемо, не исчезает при наведении на сам тултип); Progress — `role=progressbar`/`aria-valuenow` |
| **Формы** | DynamicField (text/menu/dict/tree/authority/date/bool + подполя + **ФЛК**), FilterChip, DatePicker, AuthorityPicker, TreeSelect, ValidationMessage, FieldArray | каждое поле — `<label for>`; обязательность `aria-required`; **ФЛК-ошибка** ([SPEC_engine_flk §5.2](../engines/SPEC_engine_flk.md)): severity 1 → `aria-invalid` + `role=alert` + фокус на первое битое поле; severity 2 (преодолимое) → `role=status` предупреждение, диалог-подтверждение доступен с клавиатуры; FieldArray — кнопки add/remove с осмысленным именем (не «×»), объявление об изменении числа инстансов |
| **Навигация** | AppShell, TopNav, Breadcrumbs, Tabs, Pagination, **CommandPalette (Cmd+K #162)**, TreeNav (ГРНТИ/УДК/ББК), Menu | **Bypass blocks** (skip-to-content); landmark-роли (`header/nav/main/aside`); Tabs — APG tabs (стрелки, `aria-selected`); TreeNav — APG tree (стрелки вверх/вниз/вправо-раскрыть/влево-свернуть, `aria-expanded`/`level`/`setsize`); **CommandPalette — APG combobox+listbox**, ловушка фокуса внутри, ESC-выход, объявление числа результатов; смена роута SPA → обновить `<title>` + перенести фокус на h1 + объявить (2.4.2/2.4.3) |
| **Данные** | ResultCard, RecordCard, **DataTable/Grid**, StatusBadge, HoldingsList, CoverImage, **IIIF/PDF-viewer**, PftBlock, FacetRail | DataTable — нативная `<table>` с `<th scope>`/`<caption>`; сортировка `aria-sort`; выбор строк объявляется; **StatusBadge — цвет + иконка + текст** (выдано/бронь/в ячейке/возврат), не только цвет (дальтонизм, [TZ §3](../../TZ_CLAUDEDESIGN_design-system.md)); CoverImage — осмысленный `alt` (или `alt=""` если декоративна и заглавие рядом); **PDF/IIIF-viewer — доступная альтернатива** (текстовый слой/OCR из DG #180 для screen-reader, постраничная навигация с клавиатуры, подсветка терминов не только цветом); PftBlock (серверный формат) — семантический HTML на выходе, не таблицы-для-верстки |
| **Обратная связь** | Toast, Modal/Dialog, Drawer, EmptyState, Banner/Alert, ConfirmDialog, OfflineIndicator, SyncStatus | Modal — APG dialog (`role=dialog aria-modal`, фокус-ловушка, фокус на открытии → внутрь, на закрытии → к триггеру, ESC); Toast/Alert — `role=status` (инфо) / `role=alert` (ошибка), не красть фокус; **OfflineIndicator/SyncStatus — `aria-live=polite`** (edge-режим объявляется незрячему: «офлайн / синхронизировано») |
| **Читательские** | SearchBar, фасеты+чипы, Полки/Списки, Reviews/Rating, ReadingChallenge, QueueStatus, Cabinet/Formular, CitationExport | **поиск-по-вводу** ([SPEC_reader §2](../SPEC_reader_jirbis.md)): результаты в `aria-live=polite` с числом найденных, индикатор прогресса `role=status`, **фокус не перебрасывается** на ввод (3.2.2); Rating — доступный radiogroup (не только звёзды-мышью); QueueStatus (очередь брони) — позиция объявляется; фасеты — чекбоксы/disclosure с `aria-expanded`, ленивые counts объявляются при раскрытии |
| **Сотрудник/АРМ** | CatalogingWorksheet, GrantDesktop, CirculationDesk, **CellMap + RoomPlan (SVG)**, Postamat/LockerUI, BI-Charts, BatchCorrection, InventoryScan | **SVG CellMap/RoomPlan** — `role=img`/`role=application` с текстовой альтернативой + **табличная/списочная альтернатива** того же содержимого (план зала недоступен незрячему как картинка → дублировать списком ячеек с статусом+клавиатурной навигацией); drag-n-drop ячеек — клавиатурная альтернатива (2.5.1/2.5.7); **BI-Charts** — `<title>`/`<desc>` + дата-таблица под графиком (1.1.1), не полагаться на цвет рядов (паттерны/метки); Postamat/LockerUI — large-touch, аудио-подтверждение, высокий контраст (часть аудитории — слабовидящие у терминала) |
| **Админ/тенант** | **ThemeEditor (contrast-guard)**, ModuleMarketplace, User/GrantManagement, AuditLog | **ThemeEditor — contrast-guard как функция доступности**: при выборе tenant seed-цвета live-проверка контраста (axe-core/APCA-движок в браузере), блок/предупреждение при < AA ([TZ §3](../../TZ_CLAUDEDESIGN_design-system.md)); это превентивный a11y-гейт **в продукте**, не только в CI |
| **Маркетинг (Glass)** | Hero, FeatureCard, PricingTable, CTASection | контраст текста поверх стекла гарантирован затемняющим слоем; `prefers-reduced-transparency` → непрозрачные поверхности; `prefers-reduced-motion` → стоп-параллакс/блики; декоративные mesh-фоны `aria-hidden` |

### 2.3 Интеграция в галерею DS

Каждый компонент в Storybook-подобной галерее ([TZ §9](../../TZ_CLAUDEDESIGN_design-system.md)) получает:
- вкладку **«Доступность»**: роли/ARIA, клавиатурная карта (таблица клавиш), результат axe прямо в стори (через `@storybook/addon-a11y` или эквивалент), отметки контраста для каждой темы/режима.
- статус критериев из матрицы (разд. 1.5): какие SC покрывает компонент.

## 3. Автоматический CI-гейт

> **Связь [SPEC_I1 §#118](../SPEC_I1_foundation.md):** добавляем a11y-джоб в существующий `.github/workflows/ci.yml` (рядом с `backend-tests`/`contract-lint`/`frontend-build`). **Эта спека НЕ редактирует workflow** — описывает джоб, пороги и команды для оркестратора. Контур защищён: всё исполняется на собранном фронте локально, без сети/CDN.

### 3.1 Инструменты и роли

| Инструмент | Что ловит | Где запускается | Блокирующий |
|---|---|---|---|
| **axe-core** (через `@axe-core/playwright` или `jest-axe`) | программно-детектируемые нарушения (контраст, имена, роли, ARIA-валидность) на отрендеренных страницах/компонентах | unit (на компонентах галереи) + e2e (на маршрутах) | **да** (0 critical/serious) |
| **Lighthouse CI** (`lhci autorun`, категория accessibility) | агрегированный a11y-скор + набор аудитов (часть пересекается с axe) | e2e на ключевых маршрутах поверх собранного фронта | **да** (порог скора) |
| **pa11y-ci** / **playwright-axe** | прогон axe/HTML-CodeSniffer по списку маршрутов (`.pa11yci`), без полноценного Playwright-сценария — быстрый sweep | e2e sweep маршрутов | **да** (0 errors) |
| **eslint-plugin-jsx-a11y** (если React-слой) / линтер шаблонов | статически — забытый alt, label, неверный ARIA — **до** рендера | в шаге `lint` (#118) | да (как часть lint) |

> Перекрытие axe ↔ Lighthouse намеренное (Lighthouse a11y во многом использует axe-core внутри): Lighthouse даёт **скор-тренд** и удобный отчёт для не-инженеров; axe-через-Playwright даёт **точные ассерты на критичность** и работу с авторизованными/динамическими состояниями (поиск-по-вводу, открытый Modal, ошибки ФЛК), которые Lighthouse «холодным» прогоном не увидит. Берём оба, не дублируя смысл: Playwright-axe — основной блокирующий ассерт, Lighthouse — скор-бюджет.

### 3.2 Бюджет и пороги (явные)

- **axe (Playwright/pa11y): 0 нарушений `critical` и `serious`** на всех маршрутах из списка 3.3. `moderate`/`minor` — логируются, заводятся в трекер, **не блокируют** (иначе гейт станет флакающим и его отключат — анти-паттерн). По мере закрытия — ужесточение до 0 moderate (политика разд. 7).
- **Lighthouse accessibility ≥ 95** на каждом ключевом маршруте (старт-порог; цель 100). Регрессия скора > 0 относительно baseline на изменённом маршруте — предупреждение; падение ниже 95 — блок.
- **Контраст** — частный, но самый частый класс: contrast-guard в ThemeEditor (разд. 2.2) гарантирует AA на tenant-палитрах; CI дополнительно прогоняет axe `color-contrast` на дефолт-теме **и** тёмной теме **и** теме высокого контраста.
- **Бюджет «без новых долгов»:** PR, добавляющий новый маршрут/компонент без покрытия в списке 3.3 / без a11y-стори, **не проходит ревью** (чек-лист PR, разд. 7) — это процессный, а не машинный гейт.

### 3.3 Покрываемые маршруты (ключевой путь, не «всё подряд»)

Прогоняем гейт на маршрутах критического пути продукта (а не на 100% экранов — баланс стоимости/ценности). Список — версионируемый артефакт `a11y/routes.json`:

**Читатель (I2):** `/` (главная/поиск) · результаты поиска (с фасетами, в т.ч. состояние «поиск-по-вводу») · карточка записи (профили показа, закладки) · PDF/IIIF-viewer · вход (билет/ЕСИА) · ЛК/формуляр · заказ книги (1 клик + очередь) · регистрация.
**Сотрудник:** GrantDesktop · CatalogingWorksheet (с открытой ФЛК-ошибкой sev1 и sev2) · CirculationDesk · CellMap/RoomPlan (+ списочная альтернатива) · BI-дашборд.
**Админ:** ThemeEditor (контраст-гард) · GrantManagement.
**Системные:** 404/403 · офлайн-состояние (edge) · пустые/ошибки.
**Состояния (не только «холодные» страницы):** открытый Modal/Drawer, активный Toast/Alert, форма с ошибками, тёмная тема, тема высокого контраста, `prefers-reduced-motion` on.

> Каждый маршрут тестируется **минимум в двух темах** (light + high-contrast) и с включённым reduced-motion. Авторизованные маршруты — через программный логин-фикстуру (не хардкод секретов).

### 3.4 Дизайн CI-джоба (для оркестратора — НЕ применяем здесь)

Описание джоба `a11y` для добавления в `.github/workflows/ci.yml` (оркестратор коммитит; см. фактическую структуру workflow — джобы `backend-tests`/`contract-lint`/`frontend-build`, Node 20, фронт под `irbis-web/frontend`, шрифты самохост в `public/design/fonts/`):

```yaml
# ДЛЯ ОРКЕСТРАТОРА — НЕ редактируется этой спекой. Эскиз джоба.
  a11y:
    name: a11y (axe + lighthouse)
    runs-on: ubuntu-latest
    needs: [frontend-build]            # переиспользуем артефакт сборки Vite
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: npm,
                cache-dependency-path: irbis-web/frontend/package-lock.json }
      - run: npm --prefix irbis-web/frontend ci
      - run: npm --prefix irbis-web/frontend run build       # самохост-шрифты, без сети
      - name: Serve build + Playwright-axe on routes
        run: npm --prefix irbis-web/frontend run test:a11y    # playwright-axe по a11y/routes.json
        # падает при любом critical/serious
      - name: Lighthouse CI (accessibility budget)
        run: npx --yes @lhci/cli autorun                      # порог из lighthouserc.json (a11y>=0.95)
      - name: pa11y sweep (fallback/быстрый)
        run: npx --yes pa11y-ci                               # .pa11yci, 0 errors
```

Конфиги-артефакты, которые поставляет эта программа (вне workflow): `a11y/routes.json`, `lighthouserc.json` (assertion `categories:accessibility >= 0.95`), `.pa11yci`, тест-сценарий `test:a11y` (Playwright + `@axe-core/playwright`, прогон по маршрутам/темам/состояниям), фикстура программного логина.

**Гейт-семантика:** `a11y`-джоб обязателен для мёржа в main (как isolation/contract-тесты в #118 DoD). Падение critical/serious axe **или** Lighthouse < порога **или** pa11y errors ⇒ PR красный.

## 4. Ручное тестирование с ассистивными технологиями (AT)

> Автоматика ≈ 30–40% нарушений. Остальное — **ручной протокол** с реальными AT + клавиатурой. Без него «WCAG AA» недоказуем.

### 4.1 Матрица AT × браузер (целевая)

| AT | ОС | Браузер | Приоритет (наша аудитория) |
|---|---|---|---|
| **NVDA** | Windows | Firefox / Chrome | **П1** (доминирует в РФ, бесплатна — гос-сегмент) |
| **JAWS** | Windows | Chrome | П2 (корпоратив/вузы) |
| **VoiceOver** | macOS / iOS | Safari | П2 (Mac/iPhone-читатели, мобильное приложение #179) |
| **Экранная клавиатура / только Tab** | все | все | **П1** (полная операбельность без мыши) |
| **Голосовой ввод** (Voice Control / Dragon) | — | — | П3 (по мере наличия ресурса) |
| **TalkBack** | Android | Chrome | П2 (мобильное приложение #179) |
| **Экранная лупа / 200–400% zoom** | все | все | **П1** (слабовидящие — ядро ГОСТ) |

### 4.2 Тест-персоны (маппинг на ГОСТ-категории)

- **П-Зрение-1 (незрячий, screen-reader):** только NVDA/JAWS + клавиатура. Проверяет: озвучивание, порядок чтения, метки, объявления статусов, SVG-альтернативы (CellMap → список), PDF/OCR-слой.
- **П-Зрение-2 (слабовидящий):** 200–400% zoom, тема высокого контраста, крупный шрифт. Проверяет: reflow, контраст, target size, отсутствие потери контента при увеличении.
- **П-Моторика (только клавиатура/switch):** Tab/Shift-Tab/стрелки/Esc. Проверяет: фокус-порядок, ловушки, target size, отсутствие drag-only операций (CellMap).
- **П-Слух (глухой):** видео/аудио-контент с субтитрами/транскриптом (озвучка постамата, talking-books — текстовая альтернатива).
- **П-Когнитивный:** простота форм, отсутствие тайм-аутов без продления (2.2.1), консистентная помощь (3.2.6), отсутствие когнитивного теста на входе (3.3.8).

### 4.3 Протокол ручного аудита (чек-лист прогона)

Прогон на каждом маршруте критического пути (3.3), на каждый релиз-кандидат + при изменении компонента:

```
КЛАВИАТУРА (без мыши)
[ ] весь функционал достижим Tab/Shift-Tab/стрелки/Enter/Space/Esc
[ ] порядок фокуса логичен и соответствует визуальному
[ ] фокус всегда видим и не перекрыт sticky-элементами (2.4.11)
[ ] нет ловушек; из оверлеев выход по Esc; фокус возвращается к триггеру
[ ] composite-виджеты (Tabs/Tree/Combobox/Grid) — клавиши по APG
[ ] skip-link работает первым Tab

SCREEN-READER (NVDA + JAWS, минимум один проход каждым)
[ ] каждый интерактив имеет осмысленное имя и роль
[ ] заголовки h1–h6 образуют корректную структуру (навигация по заголовкам)
[ ] landmark-навигация (main/nav/search) работает
[ ] динамика объявляется (результаты поиска, тосты, ошибки ФЛК, offline/sync)
[ ] формы: метка + обязательность + ошибка + подсказка читаются
[ ] таблицы: заголовки строк/столбцов ассоциированы; сортировка объявляется
[ ] изображения: осмысленный alt / декоративные скрыты; SVG-план имеет альтернативу
[ ] язык страницы и иноязычных фрагментов корректен

ЗРЕНИЕ / ЦВЕТ
[ ] контраст текста/иконок/границ проходит во всех темах (вкл. tenant seed)
[ ] информация не передаётся ТОЛЬКО цветом (статусы = цвет+иконка+текст)
[ ] zoom 200% и reflow 320px — без потери функций и горизонт-скролла
[ ] тема высокий-контраст и крупный шрифт работают на всех экранах

ДВИЖЕНИЕ / ТАЙМИНГ
[ ] prefers-reduced-motion отключает параллакс/блики/автокарусели
[ ] нет мигания >3 раз/с (2.3.1)
[ ] тайм-ауты сессии/токенов продлеваемы или предупреждают заранее (2.2.1)
```

Результат прогона — запись в журнале аудита (разд. 7): дата, маршрут, AT, найденные дефекты → тикеты с severity.

## 5. VPAT / Декларация о доступности

### 5.1 VPAT (международный) + РФ-декларация

Два артефакта, один источник истины (матрица 1.5):
- **VPAT** (Voluntary Product Accessibility Template, формат ITI; для совместимости с WCAG-нотацией и интер-рынка) — по каждому SC статус `Supports / Partially Supports / Does Not Support / Not Applicable` + примечание. Генерируется из `standards-matrix.json`.
- **Декларация соответствия ГОСТ Р 52872-2019** (РФ-форма для закупки) — заявление о соответствии национальному стандарту, перечень категорий пользователей, ссылка на отчёт аудита. Шаблон — в `a11y/conformance/`.

### 5.2 Страница «Доступность» (в продукте)

Публичная страница ЛК/OPAC (требование ГОСТ де-факто и хорошая практика):
- заявление о приверженности доступности и целевой уровень (WCAG 2.2 AA / ГОСТ Р 52872-2019);
- известные ограничения (что пока «Partially Supports») и план;
- **контакт обратной связи по доступности** (email/форма) + обязательство ответить в срок (напр. 5 рабочих дней) — процесс поддержки;
- ссылки на VPAT/декларацию;
- инструкции включения темы высокого контраста / крупного шрифта / аудио-режима.

### 5.3 Процесс поддержки

Обращение по доступности (через страницу 5.2) → тикет с лейблом `a11y` и приоритетом → триаж a11y-экспертом (разд. 7) → fix/обходной путь → ответ заявителю. Барьеры, блокирующие пользователя AT, — приоритет P1 (как функциональный баг). Метрика: время ответа, время устранения.

## 6. Reading disabilities (расстройства чтения) — содержательный слой

> Это превращает «accessibility» из «UI доступен screen-reader» в **доступность самого контента библиотеки** — что особенно ценно для библиотечного домена и гос-миссии (обслуживание инвалидов по зрению — прямая задача ГОСТ).

### 6.1 Темы высокого контраста и крупного шрифта

- **Тема «высокий контраст»** — отдельный набор семантических токенов ([TZ §6](../../TZ_CLAUDEDESIGN_design-system.md), связь с темизацией) с контрастом ≥ 7:1 (тянемся к WCAG AAA 1.4.6), ограниченная палитра, усиленные границы/фокус. Переключается пользователем (сохраняется в профиле) и уважает системный `prefers-contrast: more` / `forced-colors` (Windows High Contrast).
- **Крупный шрифт / интервалы** — пользовательская шкала размера (без поломки layout, тест 1.4.4/1.4.12), отдельно от zoom. Это и есть современная реализация «версии для слабовидящих» ГОСТ (разд. 1.4), **встроенная в основной UI**.
- **forced-colors / Windows High Contrast** — не ломать UI в режиме принудительных цветов ОС (иконки видимы, границы сохранены).

### 6.2 DAISY / talking books / аудио-версии

- **DAISY** (Digital Accessible Information System) — международный стандарт «говорящих книг» для лиц с нарушением чтения (синхронизированный текст+аудио, навигация по структуре). Для библиотек, обслуживающих незрячих, — профильный формат фонда.
- **Требования слоя:** (а) **каталогизация и поиск** DAISY/аудио-ресурсов как типа э-текста (связь с MARC, разд. 6.3); (б) **доступный плеер** для DAISY/MP3/аудио в нашем file-viewer ([SPEC_reader §4](../SPEC_reader_jirbis.md) — MP3/MP4 уже в скоупе): постраничная/структурная навигация с клавиатуры, скорость, закладки, синхро-подсветка текста; (в) ассеты — **самохост** (защищённый контур), стрим через бэкенд-прокси (модель прав 951/955/RIGHT, [SPEC_reader §4](../SPEC_reader_jirbis.md)); (г) для текстов без аудио — путь к озвучке (TTS как опция / связь с DG-конвейером #180 для OCR-текстового слоя сканов → доступно screen-reader).
- **Право доступа** к talking-books часто ограничено категорией читателя (инвалид по зрению) — ложится на нашу модель прав на файл (951^D категории, RIGHT), [SPEC_reader §4](../SPEC_reader_jirbis.md). Не изобретаем — переиспользуем.

### 6.3 MARC 380 — типы электронного текста

- **Поле MARC 380 «Form of Work»** (и связанные 33X — `336` content / `337` media / `338` carrier; для РУСМАРК — аналоги) фиксирует **форму/тип ресурса**, в т.ч. электронные/аудио/DAISY-формы. Каталогизатор ([SPEC_ws1](../SPEC_ws1_cataloger.md)) проставляет тип; читатель ([SPEC_reader §2](../SPEC_reader_jirbis.md)) **фильтрует/фасетирует по типу э-текста** («только аудио», «только DAISY», «крупный шрифт»).
- **Требование:** фасет «доступные форматы» в поиске (на базе 380/33X) — чтобы читатель с нарушением чтения находил пригодный для себя ресурс; маппинг кодов 380/33X на пользовательские категории фасета — артефакт (`a11y/etext-types-map.json`), уточняется по боевому профилю тенанта (**TODO-A11Y-02**).

## 7. Владелец, процесс, регулярный аудит

- **Владелец:** **a11y-эксперт** (роль; может быть совмещённой) — держит матрицу 1.5, рубрику, VPAT/декларацию; триажит тикеты `a11y`; ведёт журнал аудита; точка приёмки a11y-AC.
- **В процессе разработки (shift-left):**
  - PR-чек-лист обязателен: новый/изменённый компонент → a11y-стори в галерее + покрытие в `routes.json` при новом маршруте; ESLint-a11y и axe-unit зелёные локально.
  - DoD таска (как I1): для UI-изменения — a11y-джоб зелёный **+** (для нового сложного виджета) ручной клавиатурный проход.
- **Регулярный аудит (cadence):**
  - **Каждый релиз-кандидат** — ручной протокол (разд. 4.3) на изменённых маршрутах.
  - **Ежеквартально** — полный AT-аудит ключевого пути (NVDA+JAWS+VoiceOver+клавиатура+zoom) → отчёт + обновление VPAT/декларации + переоценка статусов в матрице.
  - **Ежегодно / перед гос-закупкой** — внешний независимый аудит (привлечённый эксперт/организация по доступности) для декларации ГОСТ.
  - **При смене дизайн-системы (#154)** — ре-аудит затронутых компонентов.
- **Журнал аудита** (`a11y/audit-log/`): дата, скоуп, инструмент/AT, дефекты, статусы → тренд закрытия. Связь с общим аудитом продукта ([SPEC_I6 §E](../SPEC_I6_security_compliance.md)).

## 8. Критерии приёмки (AC)

- **AC1 (таргет определён и отслеживаем).** Существует заполненная матрица `standards-matrix.json` (разд. 1.5), покрывающая все 56 WCAG 2.2 AA-критериев + 5 новых 2.2 + маппинг на пункты ГОСТ Р 52872-2019; каждый критерий имеет surface, способ проверки (auto/manual) и статус. *(C4 перестаёт быть hollow — у утверждения есть структура.)*
- **AC2 (компонентный контракт).** Каждый компонент инвентаря DS ([TZ §6](../../TZ_CLAUDEDESIGN_design-system.md)) имеет a11y-аннотацию в галерее (роли, клавиатурная карта, контраст по темам) и проходит чек-лист 2.1; axe-unit на компоненте зелёный (0 critical/serious).
- **AC3 (CI-гейт реален и блокирует).** В `ci.yml` присутствует джоб `a11y` (разд. 3.4); PR с новым `critical`/`serious` axe-нарушением, или Lighthouse a11y < 95, или pa11y errors на любом маршруте из `routes.json` — **не мёржится**. Гейт исполняется офлайн на самохост-сборке (без CDN).
- **AC4 (контраст под брендингом).** Tenant seed-цвет, не проходящий AA, блокируется/корректируется в ThemeEditor (contrast-guard, разд. 2.2); axe `color-contrast` зелёный на default + dark + high-contrast темах. Статусы передаются цвет+иконка+текст (тест дальтонизма).
- **AC5 (клавиатура полная).** Каждый маршрут критического пути полностью операбелен с клавиатуры (ручной протокол 4.3): нет ловушек, фокус видим и не перекрыт (2.4.11), composite-виджеты по APG, skip-link работает. CellMap/RoomPlan имеет клавиатурную **и** screen-reader альтернативу (списком).
- **AC6 (screen-reader).** Ключевые маршруты пройдены NVDA **и** JAWS (минимум): имена/роли/структура заголовков/landmark/динамические объявления (поиск-по-вводу, ФЛК-ошибки, offline/sync) корректны; результаты прогона в журнале аудита.
- **AC7 (формы и ошибки).** ФЛК-ошибки ([SPEC_engine_flk §5.2](../engines/SPEC_engine_flk.md)) доступны: severity 1 → фокус+`role=alert`+`aria-invalid`+текст; severity 2 → доступный диалог-подтверждение; каждое поле имеет метку, обязательность и связанное сообщение об ошибке (3.3.1/3.3.2/3.3.3).
- **AC8 (тайминги/движение).** `prefers-reduced-motion` отключает параллакс/блики/автокарусели (стиль B и Top-10); сессия/токены ЭИОС предупреждают/продлеваемы (2.2.1); нет мигания >3/с.
- **AC9 (VPAT/декларация/контакт).** Опубликованы VPAT (из матрицы) + декларация ГОСТ + страница «Доступность» с рабочим контактом обратной связи и заявленным SLA ответа; процесс триажа `a11y`-тикетов действует.
- **AC10 (reading disabilities).** Тема высокого контраста (≥7:1) и крупный шрифт работают на всех экранах и уважают `prefers-contrast`/`forced-colors`; DAISY/аудио воспроизводятся в доступном плеере через прокси; поиск фасетирует по типу э-текста (MARC 380/33X).
- **AC11 (процесс/аудит).** Назначен владелец-a11y; действует cadence аудита (релиз / квартал / год-внешний); журнал аудита ведётся; PR-чек-лист a11y обязателен в ревью.

## 9. Тест-матрица

> Главный принцип: **каждый AA-критерий имеет хотя бы одну проверку** (auto или manual), а каждый ключевой маршрут прогоняется и автоматикой, и хотя бы одной AT. «Зелёная автоматика» НЕ засчитывается за AA без ручного прохода.

| Группа | Что проверяем | Метод | Позитив | Негатив | Ожидание |
|---|---|---|---|---|---|
| Контраст | текст 4.5:1 / не-текст 3:1, все темы + tenant seed | axe `color-contrast` + ThemeEditor guard | пара ниже порога | пара ≥ порога | блок/флаг / pass |
| Имя-роль-значение | кастом-виджеты (Combobox/Tree/Tabs/Modal) | axe + ручной SR | виджет без `aria-*` | по APG | нарушение / читается |
| Клавиатура | полная операбельность, нет ловушки | ручной 4.3 + Playwright keyboard | Modal без фокус-ловушки/Esc | по APG dialog | дефект / pass |
| Фокус не перекрыт (2.4.11) | sticky-хедер vs фокус | ручной + visual | фокус под хедером | смещение скролла | дефект / pass |
| Target size (2.5.8) | ≥24px (≥44 kiosk) | axe target-size + ручной | чип 16px в плотной таблице | ≥24 / исключение задокум. | нарушение / pass |
| Динамика/aria-live | поиск-по-вводу, тосты, ФЛК, offline/sync | ручной SR + axe | результаты не объявлены | `aria-live=polite` объявляет | тишина / озвучено |
| Формы/ошибки | метка+обязат.+ошибка+подсказка | axe + ручной SR | инпут без label, ошибка только цветом | `<label>`+`role=alert`+текст | нарушение / pass |
| ФЛК sev1/sev2 | блок vs подтверждение, доступно | ручной (CatalogingWorksheet) | ошибка без фокуса/без alert | фокус+alert / диалог | дефект / pass |
| SVG-альтернатива | CellMap/RoomPlan, BI-charts | ручной SR + axe | план только картинкой | список+таблица-данных | недоступно / доступно |
| Reflow/zoom | 200% + 320px reflow | Playwright viewport + ручной | горизонт-скролл/потеря | без потери функций | fail / pass |
| Reduced-motion | параллакс/блики/Top-10 | Playwright `prefers-reduced-motion` | анимация не гаснет | статично | fail / pass |
| Тайминги | сессия/токен ЭИОС | ручной | резкий logout без предупр. | продление/предупреждение | fail / pass |
| Высокий контраст / forced-colors | тема + Windows HC | ручной + `forced-colors` эмул. | контент исчезает в HC | видим, границы целы | fail / pass |
| DAISY/аудио | плеер, навигация, прокси | ручной SR + клавиатура | плеер не с клавиатуры | навигация+скорость+закладки | недоступно / доступно |
| Lighthouse-скор | a11y-категория на маршрутах | lhci autorun | скор < 95 | ≥ 95 | блок / pass |

## 10. Критический путь

1. **Стандарт-таргет + матрица 1.5** (`standards-matrix.json`: WCAG 2.2 AA + 2.2-новые + маппинг ГОСТ) — превращает hollow-утверждение в перечень проверок.
2. **Компонентный a11y-контракт** (чек-лист 2.1 + APG-паттерны + a11y-аннотации в галерее DS #154) — встраиваем в дизайн-систему, пока компоненты пишутся (дешевле, чем ретрофит).
3. **CI-гейт** (`a11y`-джоб, `routes.json`/`lighthouserc.json`/`.pa11yci` + Playwright-axe-сценарий; оркестратор добавляет в `ci.yml` — связь #118) — ловит регрессии автоматически с первого дня.
4. **Ручной AT-протокол** (4.3) + тест-персоны (4.2) — закрывает 60–70%, что автоматика не видит; запускаем на ключевом пути.
5. **Reading disabilities** (тема высокого контраста/крупный шрифт + DAISY/аудио-плеер + фасет 380/33X) — содержательная доступность контента.
6. **VPAT/декларация + страница «Доступность» + процесс** (контакт, триаж, cadence-аудит, владелец) — формализация для гос-закупки и поддержка пользователей.

**Зависимости:** дизайн-система ([TZ_CLAUDEDESIGN](../../TZ_CLAUDEDESIGN_design-system.md), #154 — токены/темы/contrast-guard/forced-colors/reduced-motion); CI ([SPEC_I1 §#118](../SPEC_I1_foundation.md) — оркестратор добавляет джоб); ФЛК ([SPEC_engine_flk](../engines/SPEC_engine_flk.md) — доступные ошибки); модель прав на файл ([SPEC_reader §4](../SPEC_reader_jirbis.md) — права на DAISY/аудио); DG/OCR (#180 — текстовый слой PDF для screen-reader); i18n (#i18n — `lang`/локали); комплаенс ([SPEC_compliance_152fz](../gov/SPEC_compliance_152fz.md) — аттестация гос-сегмента, где ГОСТ-доступность — часть требований). Защищённый контур (самохост-шрифты/ассеты, без CDN) — сквозное ограничение.

**Выходной критерий C4:** утверждение «WCAG 2.2 AA / ГОСТ-аудит проходит» ([SPEC_reader §7 AC3](../SPEC_reader_jirbis.md)) **обосновано**: есть матрица соответствия, компонентный контракт в DS, блокирующий CI-гейт (0 critical axe + Lighthouse ≥ 95 на ключевых маршрутах), пройденный ручной AT-аудит (NVDA/JAWS/VoiceOver/клавиатура/zoom), опубликованные VPAT + декларация ГОСТ + страница «Доступность» с контактом, доступный контент (высокий контраст/DAISY/аудио, фасет по типу э-текста), назначенный владелец и регулярный аудит. Accessibility-first становится **проверяемым differentiator**, а не маркетинговым тегом.

## 11. Риски и открытые вопросы

- **R1 (автоматика создаёт ложное чувство соответствия).** «axe зелёный» ≠ WCAG AA (покрытие ~30–40%). **Митигация:** ручной протокол (разд. 4) — обязательная часть DoD/cadence; AC явно требует AT-прохода, не только CI.
- **R2 (плотные АРМ-таблицы vs Target Size 2.5.8 / density).** Режим compact ([TZ §1](../../TZ_CLAUDEDESIGN_design-system.md)) конфликтует с 24px-целью. **Митигация:** документированное исключение «inline в плотной таблице» (WCAG допускает) + достаточный spacing вокруг цели; large-touch профиль для kiosk/постамата с ≥44px.
- **R3 (стиль B — Apple Glass — врождённо рискован для a11y).** Прозрачность/блюр/параллакс/контраст-поверх-стекла. **Митигация:** обязательные фолбэки `prefers-reduced-transparency/-motion`, затемняющий слой под текстом, маркетинг-сайт в скоупе CI-гейта тоже; но hero-анимации — отдельный ручной чек.
- **R4 (зависимость от DS #154).** Если компоненты DS поставятся без a11y-контракта, ретрофит дорог. **Митигация:** shift-left — a11y-аннотация и axe-unit как Definition of Done компонента, не отдельная фаза.
- **R5 (ГОСТ: точная редакция и трактовка «версии для слабовидящих»).** Применимый пункт ГОСТ Р 52872-2019, перечень его требований и допустимость «один доступный продукт вместо отдельной версии для слепых» — юридический вопрос закупки. **TODO-A11Y-01:** подтвердить редакцию ГОСТ и приемлемость нашей трактовки у юриста/эксперта закупки; зафиксировать в `standards-matrix.json.gost_deltas`.
- **R6 (DAISY/MARC-маппинг неполон).** Точные коды 380/33X (или РУСМАРК-аналоги) и их маппинг на пользовательские фасеты доступности — по боевому профилю тенанта. **TODO-A11Y-02:** извлечь типы э-текста из конфигурации тенанта, заполнить `etext-types-map.json`.
- **R7 (флакающий гейт → его отключат).** Слишком строгий стартовый порог (0 moderate) или нестабильные e2e сделают `a11y`-джоб «красным всегда» → команда его обойдёт. **Митигация:** старт с 0 critical/serious + Lighthouse ≥ 95; ужесточение по графику (разд. 3.2/7); стабильные фикстуры логина, ожидание готовности динамики перед axe-проходом.
- **R8 (ресурс на AT-аудит).** NVDA/JAWS/VoiceOver-прогоны требуют экспертизы и времени. **Открытый вопрос:** in-house a11y-эксперт vs внешний подрядчик для квартального/ежегодного аудита; бюджет внешнего аудита перед гос-закупкой.

## Сводка

Программа делает differentiator **C4 «Accessibility-first»** реальным: **таргет** — WCAG 2.2 AA (с явным перечнем релевантных success criteria, включая 5 новых в 2.2: Focus Not Obscured / Target Size / Consistent Help / Redundant Entry / Accessible Authentication) **+ ГОСТ Р 52872-2019** (РФ-надстройка для гос-закупки, гармонизирована с WCAG, дельты задокументированы), сведённые в машиночитаемую **матрицу соответствия** (разд. 1.5). **Компонентный контракт** (чек-лист на компонент + APG-паттерны + a11y-аннотации) встраивается в дизайн-систему #154. **CI-гейт** — джоб `a11y` (Playwright-axe + Lighthouse CI + pa11y) поверх самохост-сборки, бюджет **0 critical/serious axe + Lighthouse a11y ≥ 95** на ключевых маршрутах (оркестратор добавляет джоб в `ci.yml`, связь #118; эта спека workflow не редактирует). **Ручной AT-протокол** (NVDA/JAWS/VoiceOver/клавиатура/zoom, тест-персоны по ГОСТ-категориям) закрывает то, что автоматика не видит. **VPAT + декларация ГОСТ + страница «Доступность»** с контактом и процессом поддержки. **Reading disabilities:** тема высокого контраста/крупный шрифт (встроенная «версия для слабовидящих»), DAISY/аудио в доступном плеере через прокси, фасет по типу э-текста (MARC 380/33X). Владелец — a11y-эксперт; cadence-аудит (релиз/квартал/год-внешний) + журнал.
