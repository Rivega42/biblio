## 0. Модель угроз (что именно защищаем)

> Маркетплейс = **исполнение чужого кода рядом с данными библиотек-тенантов**. Без этого раздела остальные — решения без задачи.

**Активы:** ПДн читателей (152-ФЗ, [SPEC_I6 §E](../SPEC_I6_security_compliance.md)), каталог/фонд тенанта, секреты тенанта (SMTP/SMS/платёжные ключи), доступность ядра, целостность цепочки поставок, репутация платформы (один злой плагин = инцидент у всех).

**Доверительные границы:**

| Граница | Кто по разные стороны | Контроль |
|---|---|---|
| Ядро ↔ плагин | наш код vs чужой код | sandbox (разд. 2) + capability-broker (разд. 3) |
| Плагин ↔ данные тенанта | код плагина vs БД/файлы | **нет прямого доступа к БД** — только Host-API под грантом+capability ([BEST_PRACTICES §11](../BEST_PRACTICES.md)) |
| Тенант A ↔ тенант B | данные двух библиотек | инвариант изоляции I1 #100 распространён на плагины (разд. 2.4) |
| Издатель ↔ платформа | автор модуля vs реестр | подпись пакета + верификация + ревью (разд. 4) |
| Плагин ↔ внешний мир | webhook/мини-апп vs интернет | egress-политика + подпись исходящих (разд. 6) |

**Каталог угроз (STRIDE-срез, верхнеуровнево):**

- **T1. Кросс-тенантная утечка** — плагин тенанта A читает данные тенанта B. → разд. 2.4, тест изоляции (AC-C5-2).
- **T2. Эскалация прав** — плагин вызывает то, на что нет capability/гранта. → разд. 3, capability-broker.
- **T3. Прямой доступ к БД/ФС/сети** в обход API. → разд. 2 (sandbox без сокетов/ФС/драйверов БД).
- **T4. DoS/noisy-neighbor** — плагин съедает CPU/память/IO, кладёт ядро или соседей. → разд. 2.3 (ресурс-лимиты, watchdog), T-isolation crash-тест.
- **T5. Supply-chain** — подменённый/тамперенный/тайпсквоттинг-пакет, уязвимая зависимость, утечка секрета в коде. → разд. 4 (подпись/верификация/SBOM/скан).
- **T6. Скомпрометированный ключ издателя** — валидно подписанный злой апдейт. → разд. 4.4 (ротация + **revocation list** + kill-switch).
- **T7. Злой/уязвимый webhook-эндпоинт** — SSRF, утечка через исходящий вызов, replay. → разд. 6 (egress-allowlist, подпись HMAC, идемпотентность приёмника).
- **T8. UI-инъекция через мини-апп** — XSS/clickjacking/кража токена staff. → разд. 2.2 (iframe-sandbox, postMessage-контракт, CSP), разд. 3.4.
- **T9. Отравление данных** — плагин с правом записи портит каталог/выдачу массово. → разд. 5.4 (rate-cap записи, обратимость, аудит), мост в ФЛК (A2)/глоб.правку (A3).
- **T10. Тихая деградация приватности** — плагин выкачивает историю чтения/ПДн в свой бэкенд. → capability `pii:read` отдельная, журнал доступа к ПДн (152-ФЗ), согласие тенанта (разд. 3.5).

## 1. Дескриптор модуля (манифест, реестр, валидация)

### 1.1 Подход

Каждый модуль несёт **декларативный манифест** `module.yaml` (FOLIO ModuleDescriptor-стиль, [SPEC_I7 §A](../SPEC_I7_commercialization.md)) — единственный источник «что модуль умеет, что просит, куда встраивается». Манифест — часть **подписанного пакета** (`.kpz`-стиль, разд. 4): подпись покрывает манифест + артефакты, поэтому декларация неподделываема. Реестр (Module-registry, [ARCHITECTURE §3](../ARCHITECTURE.md) — ядро) хранит версии, статусы, SBOM, результаты сканов, ревизии ревью.

### 1.2 Схема манифеста

```jsonc
ModuleManifest {
  // --- идентичность ---
  id: string,                      // обратный домен: "ru.example.shelf-map" (уникален в реестре, защита от тайпсквота — разд. 1.4)
  version: string,                 // semver строго (X.Y.Z[-pre]); реестр отвергает не-semver
  name: string, description: string, vendor: { name, url, email, inn? },
  license: string,                 // SPDX id ("MIT","Proprietary"…)
  minPlatform: string,             // мин. semver платформы (skew-политика, разд. 5.3)
  maxPlatform?: string,            // верхняя граница совместимости (опц.)

  // --- что просит (capability-requested) ---
  capabilities: CapabilityRequest[],   // см. разд. 3.2 — закрытый словарь, не свободный текст
  grantsRequired: GrantSpec[],         // гранты I1 #101 в нотации функция:база:уровень, которые проксирует Host-API
  dataScopes: ("catalog"|"holdings"|"loan"|"reader"|"pii"|"analytics"|"files")[],

  // --- куда встраивается (UI-slots) ---
  uiSlots: UiSlotBinding[],         // см. разд. 3.4 — только из закрытого реестра слотов
  miniApps?: MiniAppDecl[],         // встроенные мини-приложения (iframe, разд. 2.2)

  // --- события/webhooks ---
  subscribes?: EventSubscription[], // на какие доменные события реагирует (разд. 6.1)
  webhooks?: WebhookDecl[],         // исходящие вызовы (egress, разд. 6.2)
  emits?: string[],                 // какие события сам публикует (для цепочек; проверяется на конфликт)

  // --- рантайм ---
  runtime: "wasm"|"isolate"|"container"|"ui-only",   // разд. 2.1 (recommend: wasm)
  entrypoints: { backend?: string, ui?: string },    // путь к артефакту в пакете
  resources: ResourceLimits,        // CPU/mem/wall/io — разд. 2.3 (по умолчанию строгие)
  deps?: Dependency[],              // межмодульные зависимости (id+semver-range) — разд. 1.3

  // --- поставка ---
  sbom: "sbom.spdx.json"|"sbom.cdx.json",   // путь к SBOM внутри пакета (обязателен — разд. 4.3)
  integrity: { algo: "sha256", artifacts: {path: digest}[] }   // дайджесты артефактов (подпись покрывает)
}
```

### 1.3 Зависимости и совместимость

- **Межмодульные зависимости** `deps[]` — `{id, range}` (semver-range). Реестр строит граф, отвергает циклы и неудовлетворимые диапазоны при публикации; при установке — резолвит против включённых у тенанта модулей.
- **Платформенный интерфейс** версионируется semver (`platformApi: "2.x"`). Несовместимость `minPlatform/maxPlatform` → установка/апгрейд блокируется с понятной ошибкой (skew-политика — разд. 5.3, параллель edge↔облако [SPEC_I6 §C](../SPEC_I6_security_compliance.md)).
- **UI-слоты и события** версионируются вместе с платформенным API; удаление слота/события — major-bump платформы с deprecation-окном.

### 1.4 Валидация манифеста (gate публикации)

Реестр прогоняет манифест через цепочку проверок (любая красная → пакет **не публикуется**):

1. **Схема** — валиден против JSON-Schema `ModuleManifest` (строгий, `additionalProperties:false`).
2. **Semver** — `version` строго semver; апгрейд — строго выше предыдущей.
3. **Закрытые словари** — каждая `capability`/`uiSlot`/`dataScope` — из реестра разрешённых (разд. 3.2/3.4); неизвестное → отказ (нельзя выдумать право).
4. **Минимальность** — статический анализ: запрошенные capability ⊇ реально вызываемых Host-API (warn при over-request; издатель видит «просишь pii:read, но не зовёшь PII-API»).
5. **Идентичность/тайпсквот** — `id` уникален, vendor-домен подтверждён (DNS-TXT/email-челлендж); похожие на популярные id (расстояние Левенштейна) → ручное ревью.
6. **Integrity** — дайджесты `integrity.artifacts` совпадают с файлами пакета; подпись покрывает манифест+артефакты (разд. 4.2).
7. **SBOM + скан** — SBOM присутствует и валиден; Trivy-скан без критических CVE; gitleaks по артефактам без секретов (разд. 4.3).

> **Принцип «deny by default».** Всё, что не объявлено в манифесте и не подтверждено тенантом при установке, — недоступно в рантайме. Манифест — **контракт-первый** для плагина: brokered Host-API в рантайме сверяется с манифестом, а не с «как написал автор».

## 2. Sandbox / изоляция (runtime-модель)

### 2.1 Рекомендация по runtime

> Решение, которое в I7 названо («sandbox»), но не выбрано. Выбираем осознанно по оси «изоляция × производительность × DX».

| Модель | Изоляция | Старт/оверхед | DX/языки | Вердикт |
|---|---|---|---|---|
| **WASM (Wasmtime/WasmEdge), компонентная модель + WASI-preview2** | Очень высокая: память-песочница, capability-based по дизайну (нет ambient authority — нет ФС/сокетов, если не дали), детерминируемые лимиты | Низкий (мс), плотная мультиарендность | Rust/AssemblyScript/TinyGo/(Python-on-wasm) | **РЕКОМЕНДАЦИЯ для backend-плагинов.** Совпадает с capability-моделью «из коробки» |
| **Изолированный процесс / V8-isolate / gVisor-контейнер** | Высокая (namespace/seccomp/cgroups) | Средний | любой язык | Fallback для модулей, не лёгших в WASM (тяжёлые либы); строже ops |
| **Container (OCI) на edge** | Высокая, но тяжёлая | Высокий | любой | Только для «толстых» модулей на edge/on-prem; OTA-подпись [SPEC_I6 §C](../SPEC_I6_security_compliance.md) |
| **iframe + postMessage (UI)** | UI-изоляция браузера (origin, CSP, sandbox-атрибуты) | Низкий | JS/любой фронт | **РЕКОМЕНДАЦИЯ для UI/мини-приложений** (разд. 2.2) |

**Итог:** **WASM-first для backend** (capability-native, плотная мультиарендность, переносимо edge↔облако), **iframe-sandbox для UI**, **isolate/container — обоснованный fallback** под ops-нагрузкой. `runtime` в манифесте фиксирует выбор; платформа может запретить `container` для cloud-SaaS-тенантов (только edge/on-prem).

### 2.2 UI-изоляция (мини-приложения, Alma Cloud Apps-стиль)

- Мини-апп грузится в **`<iframe sandbox="allow-scripts allow-forms">`** (без `allow-same-origin` → отдельный origin, нет доступа к cookie/localStorage/DOM хоста), на **отдельном поддомене** (`apps.<tenant>.<host>`), под жёстким **CSP** (без inline-eval, `frame-ancestors` = только наш хост → анти-clickjacking, `connect-src` = только Host-bridge + объявленный egress).
- Связь с хостом — **только через `postMessage`** по типизированному контракту (`MiniAppBridge`): запрос capability-вызова → хост проверяет origin+capability+грант → отвечает. Токен staff **никогда** не передаётся в iframe; мини-апп получает **краткоживущий scoped-токен** (только capability мини-аппа, TTL минуты, audience = этот мини-апп).
- Запрет навигации верхнего окна, `window.open` на произвольный URL, доступа к буферу/гео без явной capability.

### 2.3 Ресурс-лимиты (анти-noisy-neighbor, T4)

`ResourceLimits` в манифесте, с **жёсткими потолками платформы** (манифест может просить меньше, не больше):

```jsonc
ResourceLimits {
  cpuMillis: number,        // бюджет CPU на вызов (напр. 200 ms) — watchdog прерывает (WASM epoch interruption / process kill)
  memMiB: number,           // лимит памяти инстанса (напр. 64) — OOM в песочнице, не в ядре
  wallMs: number,           // макс. wall-time вызова (напр. 5000) — таймаут
  egressCalls: number,      // макс. исходящих webhook/HTTP на вызов
  hostApiQps: number,       // потолок частоты Host-API (rate-limit на брокере)
  storageKiB?: number       // квота key-value стора плагина (если дан capability storage)
}
```

- **Изоляция сбоя (T4):** превышение → инстанс убит, **вызов** падает с `module_error`, **ядро и соседние тенанты не затронуты**. Плагин с повторными падениями → авто-`circuit-breaker` (разд. 5) и `staff_alert`.
- **Честность планировщика:** пул воркеров-инстансов per-tenant с квотой; один тенант/плагин не выедает глобальный пул (связь с I6 §A «лимиты ресурсов»).
- **Наблюдаемость:** каждый вызов плагина — span с `tenant_id`+`module_id`+`module_version` ([SPEC_I1 #117](../SPEC_I1_foundation.md) OTel), метрики CPU/mem/latency/error-rate per module.

### 2.4 «Плагин X не видит данные тенанта Y» (инвариант изоляции)

Распространяем инвариант I1 #100 на плагинный слой:

- Инстанс плагина **привязан к контексту тенанта** на момент инвокации; `ctx.tenant = {tenant_id, search_path=t_<slug>}` инъектируется хостом, **плагин не может его переопределить** (read-only в sandbox).
- Любой Host-API-вызов исполняется под `tenant_id` контекста; RLS/`search_path` — тот же инвариант. Плагин **не имеет драйвера БД, сокета, ФС** — физически не может адресовать чужую схему.
- **KV-стор плагина** (если выдан capability) — namespaced `(tenant_id, module_id)`; кросс-ключи невозможны.
- **Негатив-тест изоляции плагина** — обязательный CI-гейт (разд. 7, AC-C5-2): инстанс, запущенный под тенантом A, при любой попытке Host-API/стора получает только данные A; попытка адресовать B → отказ, утечки нет.

## 3. Capability / permission-модель

### 3.1 Двойной (тройной) гейт

Доступ плагина к действию = **И** трёх условий:

1. **Capability объявлена** в манифесте и **подтверждена тенантом** при установке (согласие, разд. 3.5).
2. **Грант** ([SPEC_I1 #101](../SPEC_I1_foundation.md)) у субъекта, **от имени которого** идёт действие, покрывает `функция×база×уровень` (capability — это не «обход RBAC», а ограничение **поверх** грантов).
3. **Энтайтлмент** — модуль включён у тенанта (`is_enabled(tenant, module)`, [SPEC_api_contracts §0](../engines/SPEC_api_contracts.md)); выключен → плагин не исполняется.

> Capability **сужает**, грант **разрешает**. Плагин не может получить больше, чем у субъекта вызова; capability дополнительно режет даже то, на что грант есть. Это closure: `effective = grants(subject) ∩ capabilities(module) ∩ entitlement(tenant)`.

**Субъект вызова (от чьего имени):**
- **interactive** — действие в UI-слоте/мини-аппе от имени staff/reader: capability ∩ гранты **этого пользователя** (scoped-токен, разд. 2.2).
- **service** — фоновая реакция на событие/webhook: плагин действует от **сервисной субъект-роли модуля у тенанта** (`module-service-account`), с явным, тенант-подтверждённым набором грантов (минимальный; по умолчанию read).

### 3.2 Словарь capability (закрытый)

Capability — **не свободный текст**, а закрытый, версионируемый словарь. Каждая мапится на конкретные Host-API и на гранты I1. Примеры (срез):

| Capability | Что даёт (Host-API) | Базовый грант I1 | dataScope |
|---|---|---|---|
| `catalog:read` | поиск/чтение записей (через A7-контракт) | `record.read:{db}:read` | catalog |
| `catalog:write` | создание/правка записи (через ФЛК A2) | `record.write:{db}:write` | catalog |
| `holdings:read` / `holdings:write` | экземпляры/локации | `holdings.*:{db}:*` | holdings |
| `loan:read` / `loan:op` | чтение/операции выдачи (событийно) | `loan.*:{db}:*` | loan |
| `reader:read` | карточка читателя **без ПДн** (id/статусы) | `reader.read:{db}:read` | reader |
| `pii:read` | **ПДн читателя** (отдельно! 152-ФЗ журнал доступа) | `reader.pii:{db}:read` | pii |
| `events:subscribe` | подписка на доменные события | — | (по scope события) |
| `webhook:egress` | исходящие HTTP на объявленный allowlist | — | — |
| `ui:slot` | рендер в UI-слот | — | — |
| `notify:send` | поставить уведомление (через A6) | `notify.send:*:write` | — |
| `storage:kv` | namespaced KV-стор плагина | — | — |
| `files:read` | view-only-прокси к файлам | `files.read:{db}:read` | files |

- **Нет capability `db:raw` / `sql` / `socket` / `fs`.** Прямого доступа к БД/сети/ФС в словаре **не существует** — это архитектурный инвариант ([BEST_PRACTICES §11](../BEST_PRACTICES.md)).
- **`pii:read` — всегда отдельная, «красная» capability**: требует явного согласия тенанта-админа с предупреждением 152-ФЗ, каждый вызов → запись в **журнал доступа к ПДн** ([SPEC_I6 §E](../SPEC_I6_security_compliance.md)).

### 3.3 Capability-broker (точка принуждения)

Между плагином и платформой — **broker** (часть Plugin-host): единственный путь плагина «наружу». На каждый вызов broker:
1. сверяет capability против манифеста (декларация) и против согласия тенанта;
2. сверяет грант субъекта (точная база > `*`, ранги read/write/admin — [SPEC_I1 #101](../SPEC_I1_foundation.md));
3. навешивает `tenant_id`+rate-limit+квоту, проксирует в Host-API (REST A7 или in-process);
4. пишет аудит-строку (разд. 5.5).
Отказ → `403 capability_denied` / `403 forbidden` / `404 module_disabled` (различимо, как в [SPEC_api_contracts §0](../engines/SPEC_api_contracts.md), Q-вопрос про 403/404 наследуется).

### 3.4 UI-слоты и права на данные пользователя

- **Слоты — закрытый реестр** (напр. `catalog.record.detail.tab`, `reader.card.action`, `staff.dashboard.widget`, `opac.search.facet`). Манифест биндит слот → артефакт; неизвестный слот → отказ валидации (разд. 1.4).
- Слот рендерится в **изолированном UI-контексте** (iframe/web-component с shadow-DOM), получает **только** данные, разрешённые capability слота (напр. `catalog.record.detail.tab` видит запись, на чтение которой есть грант, но **не** ПДн читателя без `pii:read`).
- Действия из слота идут через scoped-токен (разд. 2.2) → broker → двойной гейт. UI не может «дотянуться» до данных в обход broker.

### 3.5 Согласие тенанта при установке (consent)

- Установка модуля показывает админу-тенанта **экран согласия**: список capability человекочитаемо («читает каталог», «**доступ к персональным данным читателей**» — выделено), UI-слоты, egress-домены, требуемые гранты сервис-аккаунта.
- Согласие — **гранулярно**: админ может отказать в опциональных capability (`optional:true` в манифесте) → модуль ставится в урезанном режиме; в обязательных — отказ = модуль не ставится.
- Согласие фиксируется (кто/когда/какая версия манифеста/какой набор) — **аудит** (разд. 5.5); апгрейд, **расширяющий** capability, требует **повторного согласия** (разд. 5.2).

## 4. Подпись / supply-chain

> Опора: [SPEC_I6 §B](../SPEC_I6_security_compliance.md) (SBOM, cosign, Trivy, gitleaks, Vault/SOPS, ротация) + [BEST_PRACTICES §6](../BEST_PRACTICES.md). Маркетплейс **наследует** уже принятую supply-chain-дисциплину, добавляя специфику пакета-плагина (KPZ-стиль).

### 4.1 Формат пакета

Подписанный архив `.kpz`-стиль ([SPEC_I7 §A](../SPEC_I7_commercialization.md): «Koha KPZ»): `module.yaml` (манифест) + артефакты (wasm/ui-bundle) + `sbom.{spdx,cdx}.json` + `signature` (detached). Дайджесты артефактов — в манифесте (`integrity`), подпись покрывает **манифест+артефакты** (а через манифест — и дайджесты), исключая подмену любого файла.

### 4.2 Подпись и верификация

- **Подпись издателя:** cosign-стиль ([SPEC_I6 §B](../SPEC_I6_security_compliance.md)) — ключевая или **keyless (Fulcio/OIDC + Rekor transparency log)**, рекомендуется keyless+Rekor для аудируемости «кто/когда подписал».
- **Верификация при установке (и на edge перед активацией):** проверяется (1) валидность подписи, (2) издатель в доверенных (или прошёл ревью маркетплейса), (3) ключ/сертификат **не отозван** (разд. 4.4), (4) дайджесты совпадают, (5) `Rekor`-инклюжен (для keyless). Любой пункт красный → установка/активация блокируется.
- **Контр-подпись маркетплейса:** после ревью реестр **контр-подписывает** пакет (платформенный ключ) → тенант доверяет одной цепочке (платформа), а не каждому издателю. На edge верифицируется платформенная контр-подпись (как OTA, [SPEC_I6 §C](../SPEC_I6_security_compliance.md)).

### 4.3 SBOM и сканирование

- **SBOM на модуль** (SPDX/CycloneDX) — обязателен (gate публикации, разд. 1.4); хранится в реестре, доступен тенанту (прозрачность зависимостей).
- **Скан** при публикации **и периодически** (новые CVE на старые версии): Trivy/Grype по SBOM → критический CVE на установленной версии → авто-`advisory` тенантам + опц. принудительный disable (политика тенанта). gitleaks/trufflehog по артефактам — секрет в пакете → отказ публикации.
- **Pinned deps** ([SPEC_I6 §B](../SPEC_I6_security_compliance.md)): SBOM фиксирует версии; реестр отвергает «плавающие» диапазоны в production-пакете.

### 4.4 Ротация ключей + revocation list (T6 — критично)

> Это то, чего в I7 нет совсем, а без него подпись бесполезна при компрометации.

- **Ротация:** ключи издателя и платформенный контр-подписной ключ ротируются по расписанию (envelope/PKI, связь [SPEC_I6 §B](../SPEC_I6_security_compliance.md) + дыра [D2 ADR-006](../../DESIGN_COVERAGE.md)). Старые подписи остаются валидными до явного отзыва (transparency log хранит историю).
- **Revocation list (CRL/OCSP-стиль):** реестр публикует **список отозванных** ключей/сертификатов/**конкретных версий модулей** (`{publisher_key | module_id@version | digest}` + причина + дата). Верификатор (облако **и edge**) сверяется с CRL; edge кеширует CRL с TTL и обновляет при синке (офлайн-grace — как лицензия [SPEC_I6 §C](../SPEC_I6_security_compliance.md)).
- **Kill-switch (T6):** отзыв `module_id@version` → реестр рассылает revocation-событие по шине ([SPEC_I7 §B](../SPEC_I7_commercialization.md) #107) → Plugin-host **деактивирует** инстансы у всех тенантов (с уведомлением админов), даже офлайн-edge — при ближайшем синке. Для «горящих» — принудительный disable без согласия тенанта (политика безопасности платформы), с аудитом и нотификацией.
- **Скомпрометированный ключ:** отзыв ключа → все его подписи невалидны → затронутые модули требуют переподписи новым ключом + повторного ревью.

## 5. Lifecycle модуля (install/enable/disable/upgrade/uninstall) + аудит

### 5.1 Состояния и переходы

```
[в реестре] --install--> [installed/disabled] --enable--> [enabled]
   ^                            |  ^                          |
   |                       uninstall |                    disable
   |                            v  |                          v
[uninstalled]<--purge-- [installed/disabled] <----------------+
                    upgrade: [enabled vN] --(verify+consent?)--> [enabled vN+1]  (rollback -> vN)
```

- **install** — добавить модуль тенанту (entitlement `tenant_module`, [SPEC_I1 #101](../SPEC_I1_foundation.md)), после верификации (разд. 4.2) и согласия (разд. 3.5). Ставится **disabled**.
- **enable** — активировать (загрузить инстанс/зарегистрировать слоты/подписки/webhooks). **disable** — деактивировать, не удаляя данных плагина (быстрый kill-switch).
- **upgrade** — на новую semver: верификация+CRL; если новая версия **расширяет** capability → требует **повторного согласия** (разд. 3.5); патч/минор в рамках того же capability-набора — автоматизируемо (политика тенанта).
- **uninstall** — снять entitlement, остановить инстансы, отвязать слоты/webhooks; **purge** — удалить KV-стор плагина (право на забвение / чистый выход).

### 5.2 Безопасность апгрейда/отката

- **Canary/поэтапно** (как edge-OTA, [SPEC_I6 §C](../SPEC_I6_security_compliance.md)): новая версия включается на части (тенант-опт-ин / процент сессий), метрики ошибок/латентности под наблюдением → авто-`rollback` при росте error-rate.
- **Откат** — на предыдущую закреплённую версию (реестр хранит N последних), детерминирован; данные плагина мигрируются обратимо или версионируются (плагин обязан декларировать совместимость стора между версиями).
- **Расширение прав при апгрейде = consent-gate** (не молчаливая эскалация, T2).

### 5.3 Semver-совместимость и skew

- Платформенный API — semver; модуль объявляет `minPlatform/maxPlatform`. Апгрейд платформы, ломающий major, → модуль помечается incompatible до апдейта (не падает молча). Edge↔облако skew — та же политика, что в [SPEC_I6 §C](../SPEC_I6_security_compliance.md).

### 5.4 Защита от «отравления данных» (T9)

- Capability `*:write` плагина дополнительно ограничены **rate-cap** (макс. записей/мин) и проходят те же движки: запись каталога → **ФЛК (A2)** ([SPEC_engine_flk](../engines/SPEC_engine_flk.md)), массовая правка — через **глоб.корректировку (A3)** с превью/откатом ([SPEC_engine_gbl](../engines/SPEC_engine_gbl.md)). Плагин не имеет «прямого» массового write в обход обратимости/аудита.

### 5.5 Аудит (install/uninstall/disable/enable/upgrade/consent/access)

- **Каждый** lifecycle-переход и **каждое** согласие → запись в аудит I1 ([SPEC_I1 #101](../SPEC_I1_foundation.md): actor, tenant, функция, объект, результат) + специфика: `module_id`, `version`, `capabilities_granted`, `publisher_key_id`, `signature_verified`, `consent_by`.
- **Доступ плагина к данным** — аудируется на broker (разд. 3.3): `pii:read`-вызовы → **журнал доступа к ПДн** (152-ФЗ, [SPEC_I6 §E](../SPEC_I6_security_compliance.md)); прочие — sample/aggregate в наблюдаемости (объём не убивает хранилище), с полной записью write-операций.
- Аудит **неизменяем** (append-only), коррелируется по `trace_id`+`tenant_id`+`module_id` ([SPEC_I1 #117](../SPEC_I1_foundation.md)).

## 6. Webhooks / мини-приложения

> Транспорт — **событийная шина I7 #107** ([SPEC_I7 §B](../SPEC_I7_commercialization.md)); дисциплина доставки — **переиспользуем конвейер уведомлений** ([SPEC_engine_notifications §4](../engines/SPEC_engine_notifications.md): outbox/CDC at-least-once, dedup, backoff+jitter, dead-letter), не изобретаем заново.

### 6.1 Реестр подписок (событие → модуль)

- `subscribes[]` манифеста (после согласия) регистрируется в **реестре подписок** `(tenant_id, event_type, module_id, filter?)`. При публикации доменного события (loan/catalog/reader/system) шина матчит подписки тенанта → доставляет подписанным модулям.
- Подписка ограничена capability `events:subscribe` + scope события (плагин без `loan:read` не подпишется на loan-события с полезной нагрузкой; получит обеднённое событие или отказ).
- Изоляция: события тенанта A доставляются **только** модулям тенанта A (инвариант разд. 2.4).

### 6.2 Доставка / ретраи / подпись исходящих

- **Доставка at-least-once** через outbox: событие → попытка доставки в webhook-эндпоинт плагина (или in-process для WASM-инстанса). Транзиентная ошибка → **backoff+jitter** (1м/5м/30м/2ч/6ч), предел → **dead-letter** + `staff_alert` ([SPEC_engine_notifications §4.4](../engines/SPEC_engine_notifications.md)).
- **Подпись исходящих:** каждый доставляемый payload подписан платформой (**HMAC-SHA256** общим per-(tenant,module)-секретом, заголовок `X-Signature` + `X-Timestamp` + `X-Delivery-Id`) — приёмник верифицирует подлинность и **отвергает replay** по timestamp+delivery-id (идемпотентность приёмника, как webhook-receiver в [SPEC_engine_notifications §5.3](../engines/SPEC_engine_notifications.md)).
- **Идемпотентность:** `X-Delivery-Id` (UUID) — повторная доставка того же события не двоит эффект на стороне плагина (плагин обязан дедупить; контракт это требует).

### 6.3 Egress-контроль (анти-SSRF, T7)

- `webhooks[]` объявляют **allowlist доменов**; capability `webhook:egress` + согласие тенанта. Broker разрешает исходящие **только** на объявленные домены; запрет приватных диапазонов (RFC1918/link-local/metadata-ендпоинтов 169.254.169.254) — анти-SSRF.
- Лимит исходящих на вызов (`egressCalls`, разд. 2.3); таймаут; mTLS/TLS-pinning опц. для чувствительных интеграций.

### 6.4 Мини-приложения

- Изоляция — разд. 2.2 (iframe-sandbox, отдельный origin, CSP, scoped-токен, postMessage-контракт). Мини-апп = «UI-only» или «UI + backend-плагин» (тогда backend под WASM-sandbox разд. 2.1).
- Мини-апп не получает staff-токен; все его действия — через broker под scoped-capability (Alma Cloud Apps-модель: «приложение действует от пользователя, но в рамках выданного scope»).

## 7. Критерии приёмки (AC)

- **AC-C5-1 (дескриптор+валидация).** Пакет с манифестом, объявляющим неизвестную capability / не-semver-версию / отсутствующий SBOM / несовпадающий дайджест, **не публикуется** (gate разд. 1.4 красный). Валидный — публикуется с SBOM в реестре.
- **AC-C5-2 (тест изоляции плагинов — обязательный CI-гейт).** Инстанс плагина, запущенный под тенантом A, при любой попытке Host-API / KV-стора / события получает **только** данные A; адресация тенанта B → отказ, нулевая утечка. Сборка падает при любой возможности кросс-тенантного доступа из плагина (распространение [SPEC_I1 #100 AC4](../SPEC_I1_foundation.md) на плагинный слой).
- **AC-C5-3 (изоляция сбоя, T4).** Плагин, падающий/зацикливающийся/исчерпывающий лимит CPU/mem, убивается watchdog'ом; **ядро и соседние тенанты остаются работоспособны** (crash-тест: kill инстанса ≠ падение системы); инцидент попадает в наблюдаемость + `staff_alert`.
- **AC-C5-4 (capability/двойной гейт).** Плагин без объявленной/подтверждённой capability получает `403 capability_denied`; с capability, но без гранта субъекта → `403 forbidden`; при выключенном модуле → `404/403 module_disabled`. `pii:read` без явного согласия тенанта недоступна.
- **AC-C5-5 (подпись/верификация).** Тамперенный пакет (изменён артефакт/манифест после подписи) **не устанавливается**; неподписанный/неверифицируемый — тоже; на edge активация только после проверки платформенной контр-подписи.
- **AC-C5-6 (ротация+отзыв, T6).** Отзыв `module_id@version` (или ключа издателя) → revocation-событие по шине → Plugin-host деактивирует инстансы у всех тенантов (включая офлайн-edge при ближайшем синке); попытка установки отозванной версии блокируется по CRL.
- **AC-C5-7 (lifecycle+аудит).** install/enable/disable/upgrade/uninstall и каждое согласие → неизменяемая аудит-запись (actor/tenant/module/version/capabilities/signature_verified); апгрейд, **расширяющий** capability, требует повторного согласия (не молчит).
- **AC-C5-8 (webhooks: доставка/ретраи/подпись).** Доставка at-least-once с backoff+jitter и dead-letter; payload подписан HMAC (`X-Signature`+`X-Timestamp`+`X-Delivery-Id`); повтор delivery-id не двоит эффект; replay по timestamp отвергается.
- **AC-C5-9 (egress/SSRF, T7).** Исходящий вызов плагина на не-объявленный домен или приватный диапазон/metadata-IP блокируется broker'ом.
- **AC-C5-10 (UI-изоляция, T8).** Мини-апп грузится в sandbox-iframe на отдельном origin под CSP, не имеет доступа к staff-токену/cookie хоста; действия идут через scoped-токен + broker; XSS из мини-аппа не выходит за iframe.

| GET | `/api/marketplace/revocations` | CRL: отозванные ключи/версии (для edge-кеша) | `marketplace.read:*:read` | — | 401, 403 |

- Мутации (`install/enable/disable/upgrade/rollback/uninstall/consent/publish`) принимают `Idempotency-Key`; долгие (`install`/`upgrade` с верификацией+скан) — `202 {jobId}` + поллинг (паттерн A7 §6.3).
- Отзыв/CRL также распространяется **событием по шине** ([SPEC_I7 §B](../SPEC_I7_commercialization.md)) — REST `revocations` для холодного старта/edge-кеша, событие — для live-деактивации (разд. 4.4).

## 11. Открытые вопросы

- **Q1 (runtime по умолчанию для backend).** WASM-first — целевой, но миграция тяжёлых либ нетривиальна. Допустить ли `isolate`/`container` как **равноправный** старт (быстрее заполнить маркетплейс) ценой более тяжёлого ops, или WASM-only с самого начала ради единой модели изоляции? Влияет на DX, плотность мультиарендности и сроки.
- **Q2 (отзыв без согласия тенанта).** Принудительный disable отозванного/уязвимого модуля у тенанта **без** его согласия — обязателен для безопасности платформы (T6), но конфликтует с автономией тенанта и SLA («вы выключили то, чем я пользуюсь»). Где граница: всегда принудительно по critical-CVE/compromise, иначе — advisory? Нужна **политика отзыва** (security-решение I6) + договорная формулировка.
- **Q3 (модель доверия издателей).** Открытый маркетплейс (любой подписавшийся, ревью пост-фактум) против курируемого (только проверенные издатели, ревью до публикации)? Первый растит экосистему-USP быстрее, второй безопаснее для гос-сегмента (152-ФЗ/ФСТЭК, [SPEC_I6 §E](../SPEC_I6_security_compliance.md)). Возможен гибрид (verified-бейдж + sandbox-tier для непроверенных), но это удваивает ops capability-/scan-контура.

## Сводка

**Sandbox + capability + signing-модель (закрытие C5):**
- **Sandbox:** **WASM-first** для backend-плагинов (capability-native, нет ambient authority — нет ФС/сокетов/драйвера БД), **iframe-sandbox** (отдельный origin, CSP, scoped-токен, postMessage) для UI/мини-приложений, `isolate`/`container` — обоснованный fallback; жёсткие ресурс-лимиты (CPU/mem/wall/egress) с watchdog'ом → **сбой плагина ≠ падение системы**; инвариант «плагин X не видит тенанта Y» (контекст тенанта инъектируется хостом, read-only, нет драйвера БД) — **обязательный CI-тест изоляции плагинов**.
- **Capability:** закрытый версионируемый словарь прав, **тройной гейт** `capabilities(module) ∩ grants(subject) ∩ entitlement(tenant)` через **capability-broker** (единственный путь плагина наружу, прямого доступа к БД/сети/ФС в словаре нет); `pii:read` — отдельная «красная» capability с журналом 152-ФЗ; гранулярное **согласие тенанта** при установке, повторное — при расширении прав.
- **Signing:** подписанный `.kpz`-пакет (cosign/keyless+Rekor) + **контр-подпись платформы**, верификация при установке и на edge, **SBOM на модуль** + Trivy/gitleaks-гейт, и — главное, чего нет в I7 — **ротация ключей + revocation list (CRL) + kill-switch по шине** (деактивация отозванного модуля у всех тенантов, включая офлайн-edge при синке).
- Lifecycle install/enable/disable/upgrade/rollback/uninstall на entitlements I1 #101, безопасный canary-апгрейд+откат, **неизменяемый аудит** каждого перехода и согласия; webhooks/мини-апп — на событийной шине I7 #107 с дисциплиной доставки из движка уведомлений (at-least-once, backoff+jitter, dead-letter, HMAC-подпись+анти-replay, egress-allowlist против SSRF).
