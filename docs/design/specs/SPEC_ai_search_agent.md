# Спека: AI поиск-агент Biblio (читательский портал)

Issue: **#293** · статус: MVP-дизайн · контекст: реверс легаси-ассистента jirbis
(см. `docs/cutover/JIRBIS_CUTOVER_PLAN.md` §7.3) → нативный агент в нашем стеке.
Связано: [[prod-reality-spb-gtb]], own-store соц-слой (`access/social.py`),
бронь визита (#296), режимы/авторизация (#275–#282).

## 0. Откуда растёт (реверс jirbis, для справки)

Ассистент jirbis **инструментальный через MCP**, не «болталка». Цепочка:
`includes/ji_ai.php` (ядро чата) → `includes/answer.php` (диспетчер, ветка
`case 'mcp':` стр.556 — мост к инструментам) → `mcp/grnti.php` (ЕДИНСТВЕННЫЙ
MCP-инструмент: рубрикатор ГРНТИ поверх HTTP, заголовок `Mcp-Session-Id`,
JSON-RPC `initialize`/`tools`) → `llm_router` (LLM-прокси). Фидбэк-таблица
`jos_ai_chat_feedback`: `user_message` / `assistant_content` / `tool_calls_json`
(MEDIUMTEXT) / `deepthink` / `has_file` / `rating` (на боевом 0 строк — данных
нет, проектируем с нуля).

**Что берём из паттерна:** агент, который САМ дёргает каталог как инструмент;
тонкий диспетчер инструментов; LLM за роутером; журнал диалога с `tool_calls`.
**Что меняем:** наш набор инструментов нативный (на реальные эндпоинты Biblio,
не PHP/HTTP-MCP); тон неформальный; поведение проактивное; жёсткая
тенант-изоляция и запрет ПДн во внешний LLM.

## 1. Цель и поведение

Разговорный помощник в читательском портале, который **сам ищет в каталоге и
предлагает найденное**, а не ждёт точной формулировки. Неформальный тон,
короткие реплики, всегда показывает РЕАЛЬНЫЕ карточки записей (а не пересказ).
Не выдумывает библиографию — любой вывод про фонд опирается на ответ инструмента
(grounding). Если не нашёл — честно говорит и предлагает уточнить/расширить
запрос или фасет.

Примеры сценариев:
- «посоветуй что-нибудь про Станиславского» → агент сам зовёт `catalog_search`,
  показывает 3–5 карточек, предлагает сузить по фасету/году.
- «а есть электронные?» → переформулирует поиск (фильтр доступа/полнотекст),
  не теряя контекст диалога.
- «хочу вот эту» → предлагает `hold`/полку (если читатель авторизован).

## 2. Архитектура

```
читатель ─chat─► AiAgent (access/ai_agent.py)
                   │  1. собрать контекст (история own-store + системный промпт)
                   │  2. вызвать LLM через router → может вернуть tool_calls
                   ▼
                 Tool dispatch (реестр инструментов → существующие методы Api)
                   │  3. выполнить инструмент в КОНТЕКСТЕ сессии/тенанта (authz, audit)
                   │  4. вернуть результат в LLM → финальная реплика + карточки
                   ▼
                 own-store: chat_session / chat_message (+ tool_calls, rating)
```

- **Роут:** `POST /api/ai/chat` в `core.py` (рядом с `route()`-диспетчером,
  как `/api/search`, `/api/recommendations/foryou`). Гость/читатель-сессия;
  гранты как у остального портального контура.
- **Модуль:** `access/ai_agent.py` — оркестратор: системный промпт, цикл
  «LLM → tool_calls → выполнить → LLM», лимит шагов (anti-loop), сбор истории.
- **LLM-роутер:** `access/llm_router.py` — тонкий прокси к провайдеру (analog
  jirbis `llm_router`); провайдер/ключ из конфигурации тенанта; здесь же
  редакция ПДн ПЕРЕД отправкой (см. §5). Сменяемый провайдер (один интерфейс
  `complete(messages, tools) -> {text, tool_calls}`).
- **Инструменты — НЕ новый транспорт:** реестр в `ai_agent.py` маппит имя тула
  на УЖЕ существующий метод `Api.*` (тот же код, что обслуживает HTTP-роут), с
  той же сессией/тенантом → никакой второй копии бизнес-логики (антипаттерн
  «клон api.svc», ср. [[device-native-not-overlay]]).

## 3. Контракт инструментов (на фактический API Biblio)

Каждый инструмент = имя + JSON-схема входа/выхода + реальный метод/эндпоинт.
Форма (имя/описание/inputSchema) совместима с tool-use большинства LLM и по духу
повторяет MCP-тул jirbis (`grnti.php`), но исполняется внутри процесса.

| Инструмент | Вход (эскиз) | Выход | Эндпоинт/метод Biblio | Фаза |
|---|---|---|---|---|
| `catalog_search` | `{query, db?, prefix?, page?, limit?}` | список карточек `{db,mfn,brief,year,access}` | `/api/search` (нужен `&prefix=`) | 1 |
| `record_detail` | `{db, mfn}` | полная карточка + экземпляры/доступ | `/api/record` (+ `/api/linked/{db}/{mfn}`) | 1 |
| `suggest` | `{prefix, db?}` | автодополнения | `/api/suggest` | 1 |
| `term_browse` | `{db, prefix, from?}` | словарь терминов (просмотр) | `/api/terms` | 1 |
| `rubricator` | `{db?, code?, query?}` | узлы рубрикатора/словаря (ГРНТИ-подобно) | `/api/vocab` — **прямой аналог `grnti.php`** | 1–2 |
| `fulltext` | `{db, mfn}` | наличие/ссылка полного текста | `/api/fulltext/{db}/{mfn}` | 2 |
| `recommend` | `{seed_mfn?}` | «вам может подойти» | `/api/recommendations/foryou` | 2 |
| `place_hold` | `{db, mfn}` (только авториз. читатель) | статус брони | `/api/hold` | 2 |
| `shelf_add` | `{db, mfn}` (авториз.) | статус полки | `/api/shelves/item` | 2 |
| `book_visit` | `{service_id, time}` (авториз.) | бронь визита | **план #296** (эндпоинта нет) | 3 |

Правила: запись (`place_hold`/`shelf_add`/`book_visit`) — только из
читательской сессии и только по явному подтверждению пользователя в диалоге
(не «сам забронировал»). Read-инструменты доступны и гостю в рамках публичных баз
(`_public_db_guard`).

## 4. Проактивность

«Сам ищет и предлагает» — но без спама:
- **Триггер на намерение:** если реплика читателя выглядит как информационная
  потребность (тема/автор/вопрос про фонд), агент СРАЗУ вызывает `catalog_search`
  и показывает результат, не переспрашивая «что именно ищете».
- **Уточнение фасетом, а не словами:** после выдачи предлагает 1–2 конкретных
  сужения (год/тип/доступность/рубрика из `rubricator`), кнопками, а не абзацем.
- **Контекст диалога:** последующие реплики («а электронные?», «новее») —
  переформулировка предыдущего запроса, не новый с нуля.
- **Анти-спам:** не инициирует поиск на болтовне/благодарностях; ≤1 проактивный
  поиск на реплику; не повторяет ту же выдачу.
- **Грейсфул-fallback:** пусто → расширить запрос/снять фасет/предложить
  смежную рубрику; честно «в фонде не нашёл».

## 5. Приватность и безопасность (жёстко)

- **НИКАКИХ ПДн читателя во внешний LLM.** В `llm_router` — редактор: ФИО/билет/
  email/телефон/историю выдач не отправляем; читатель в промпте обезличен
  (роль/тенант, не личность). Инструменты, дающие ПДн (`circ/reader`, формуляр),
  в MVP агенту НЕ даём.
- **Тенант-изоляция:** все вызовы инструментов идут под сессией/тенантом
  читателя; кросс-тенантного доступа нет (как в остальном контуре).
- **Контент — это данные, не команды:** текст записей/аннотаций, попавший в
  контекст, не исполняется как инструкция (защита от prompt-injection из
  каталога); системный промпт это фиксирует.
- **Аудит:** каждый `tool_call` (имя, аргументы, тенант, сессия) пишется в
  аудит (как прочие операции `access/`), плюс журнал диалога (§6).
- **Rate-limit:** на сессию/IP — и на реплики, и на шаги tool-loop; жёсткий
  лимит шагов на один ответ (anti-runaway).
- **Грант/режим:** агент включается флагом тенанта (как режимы #275), по
  умолчанию OFF; провайдер LLM и ключ — из конфигурации тенанта, не хардкод.

## 6. Хранение диалога/фидбэка (own-store, аналог `jos_ai_chat_feedback`)

Своя таблица в `access/store.py` (НЕ Joomla-схема), напр. `chat_message`:
`tenant, session_id, role(user|assistant|tool), content, tool_calls_json,
created_at, rating?, has_file?`. Поля повторяют полезное из `jos_ai_chat_feedback`
(`tool_calls_json`, `rating`; `deepthink` — опц. флаг «глубокого» режима на
будущее). История диалога читается отсюда для контекста; рейтинг реплики (палец
вверх/вниз) — петля качества. ПДн в content не храним (см. §5).

## 7. MVP-фазы

- **Фаза 1 (минимум):** роут `/api/ai/chat` + `access/ai_agent.py` +
  `access/llm_router.py`; инструменты `catalog_search` + `record_detail`
  (+`suggest`/`term_browse` как дешёвое расширение); проактивный поиск §4;
  журнал `chat_message`; редактор ПДн §5; флаг тенанта OFF по умолчанию; тесты
  (диспетчер инструментов, редактор ПДн, anti-loop, tenant-guard) в стиле
  `tests/test_*`. Без записи (только чтение каталога).
- **Фаза 2:** `rubricator`(`/api/vocab`) + `fulltext` + `recommend`; запись по
  подтверждению — `place_hold` + `shelf_add`; рейтинг реплик.
- **Фаза 3:** `book_visit` (после #296); RAG/полнотекст-grounding
  (`/api/fulltext`, морфология own-store), «глубокий» режим (`deepthink`).
