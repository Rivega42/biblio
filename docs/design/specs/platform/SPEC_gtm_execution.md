## 0. Назначение, границы, принципы

**Назначение.** Дать команде и партнёрам **исполнимый playbook**: по какой цене и кому продавать, как войти в реестр отеч.ПО и участвовать в 44/223-ФЗ, в каком порядке выходить по сегментам/регионам, как строить партнёрскую сеть и как продавать/исполнять миграцию из ИРБИС/МАРК как платную услугу. Цель GTM ([COMPETITIVE_LANDSCAPE §«Сводные выводы»](../../COMPETITIVE_LANDSCAPE.md)): не дать продукту «застрять в пилоте» ([DESIGN_COVERAGE §«Приоритет» п.4](../../DESIGN_COVERAGE.md)).

**Границы (что НЕ здесь):**
- **Механика биллинга** (сбор метрик, выставление счёта, НДС-проводка, платёжный провайдер, enforcement entitlements) → **D5** (#108). Здесь — прайс-точки и **бизнес-определения** единиц метеринга как вход для D5.
- **Механика комплаенса** (классификация ИСПДн, журнал доступа, согласия, аттестация) → **B4** ([SPEC_compliance_152fz](../gov/SPEC_compliance_152fz.md)). Здесь — **закупочная упаковка** готовности.
- **Техника миграции** (парсер ISIS/MARC, маппинг полей, ФЛК) → I-спеки импорта + A5-сидинг. Здесь — миграция **как услуга**: фазы, runbook, ценообразование, SLA.
- **Реестр модулей / entitlements** (как тариф включает функции технически) → [SPEC_I7 §A/§C](../SPEC_I7_commercialization.md). Здесь — **что** входит в тариф и **за сколько**.
- **Маркетинговый контент/SEO/события** (Vega LX-подобная платформа вовлечения) → PMB (#144). Здесь — каналы и сегментная последовательность, не контент-план.

**Принципы исполнения:**
1. **Цена — по ценности и по способности платить, не по себестоимости.** Школа платит мало (или 0), сеть-энтерпрайз — много; оси ([GO_TO_MARKET §4](../../GO_TO_MARKET.md)): тип библиотеки × размер × площадки × места × модули × хранилище. Прайс-точки (разд. 1) — это **пресеты** этих осей под 4 тарифа.
2. **Импортозамещение — триггер, реестр — пропуск.** Без записи в реестре отеч.ПО гос-сегмент закрыт по 44/223-ФЗ (разд. 3). Это **gating-задача**, не «маркетинг».
3. **Пилот-витрина → кейс → закупка.** Сначала референс (СПб ГТБ), потом масштабирование по похожим сегментам с готовым кейсом (разд. 4).
4. **Миграция — не препятствие, а продукт.** «Страх перехода с ИРБИС» снимается услугой «под ключ» (strangler: читаем-параллельно → пишем-параллельно → переключаем), которая ещё и приносит выручку (разд. 6).
5. **Партнёры масштабируют то, что мы стандартизовали.** Реселлер/интегратор продаёт и внедряет по нашему runbook; мы держим продукт, сертификацию и L3-поддержку (разд. 5).
6. **Каждая цифра — гипотеза с критерием проверки.** Прайс/конверсия/TCO валидируются на пилоте и первых сделках, не утверждаются как факт.

### 1.5 TCO-аргумент (3 года) — как продаём «дешевле»

Позиционирование «дешевле и быстрее внедрить» ([GO_TO_MARKET §2](../../GO_TO_MARKET.md)) доказывается **TCO-калькулятором [допущение, валидировать]**: 3-летний совокупный расход = подписка ×3 + разовая миграция/онбординг (разд. 6) + обучение, **против** desktop-ИРБИС (лицензии АРМов + годовое сопровождение + ИТ-инфраструктура/серверы/администрирование desktop-парка + риск EOL веб-ИРБИС). Наши экономии: нет desktop-АРМов, нет локальных серверов (cloud SaaS), импортозамещение без миграции на иностранное. **Калькулятор — артефакт пресейла; числа подставляются из профиля клиента, не зашиты.**

### 1.6 Калибровка и проверка прайса

- **Способ калибровки [допущение]:** профили 5–8 реальных библиотек по сегментам (фонд, читатели, площадки, места) → пропустить через сетку → проверить, что чек попадает в «приемлемо для сегмента». Пилот СПб ГТБ — первый профиль (спец/ведомственная).
- **Критерий проверки прайса:** win-rate первых 5 коммерческих сделок ≥ **[цель-допущение]** при марже, покрывающей CAC + L2/L3-поддержку; отсутствие массового «too expensive» в отказах.
- **Эластичность free→paid:** доля апселла из бесплатного (разд. 2.3) ≥ целевого порога без подрыва экономики (разд. 2.4).

## 2. Free-tier политика (школы)

### 2.1 Решение: да, бесплатный тариф для школ — есть

Бесплатный тариф **вводится** ([GO_TO_MARKET §10](../../GO_TO_MARKET.md): «бесплатный тариф для школ (привлечение)» — открытый вопрос → **решаем: да**). Обоснование: (1) школьный сегмент — «очень много, малый бюджет, часто без ИТ» ([GO_TO_MARKET §1](../../GO_TO_MARKET.md)); платный вход их отсекает; (2) прямой конкурент freemium-облака (Либра-онлайн — «freemium-облако», [COMPETITIVE_LANDSCAPE §1](../../COMPETITIVE_LANDSCAPE.md)) — без free-tier мы проигрываем вход; (3) школы — воронка в платные сегменты (методобъединения, переход в публичные/вузовские закупки, кейсы).

### 2.2 Лимиты бесплатного (разд. 1.3, столбец «Бесплатный»)

Пресет = Базовый-функционал (каталог + поиск + читательский веб), облако-only (нет edge/on-prem), 1 площадка, лимиты: фонд ~10 тыс. записей, ~1 тыс. читателей, 2 staff-seat, 1 ГБ, мягкий лимит запросов — все **[вилка]**, калибруются под «типичную школьную библиотеку» **[допущение]**. Поддержка — только self-service/комьюнити (без SLA, разд. 5/[GO_TO_MARKET §7](../../GO_TO_MARKET.md)).

### 2.3 Критерии апселла (free → paid)

Сигналы для перевода в Базовый/Стандарт:
- **Превышение лимитов** (фонд/читатели/места/хранилище) — мягкий тротлинг + предложение тарифа (enforcement — D5 entitlements).
- **Спрос на функции старших тарифов:** книговыдача/уведомления/ячейки → Стандарт; книгообеспеченность/учебники (важно школам, [GO_TO_MARKET §1](../../GO_TO_MARKET.md)) → как модуль/Профи-функция.
- **Несколько площадок / переход на ЦБС-уровень** → Стандарт/Сеть.
- **Закупка по 44-ФЗ** (школа получила бюджет) → платный тариф с реестром (разд. 3).
- **Метрика апселла:** конверсия free→paid за 12 мес. ≥ **[цель-допущение]**; отслеживается в D5 (lifecycle trial/free→paid, [SPEC_I7 §C](../SPEC_I7_commercialization.md)).

### 2.4 Устойчивость (чтобы free не съел экономику)

- **Маржинальная стоимость free-тенанта мала:** cloud-only, общий мультиарендный стек ([ARCHITECTURE](../../ARCHITECTURE.md)), без выделенного edge/поддержки → предельные расходы ≈ хранилище+вычисления в лимитах.
- **Жёсткие лимиты + мягкий тротлинг:** превышение не роняет сервис, а триггерит апселл (разд. 2.3), не бесконтрольный рост затрат.
- **Анти-абьюз [допущение]:** верификация принадлежности к школе (домен/ОГРН/реестр образовательных организаций) — чтобы коммерческие не сидели на free; одна бесплатная инсталляция на организацию.
- **Self-service-онбординг** без участия людей (иначе CAC съест экономику): автосидинг базовых словарей (A5, [SPEC_seeding](../engines/SPEC_seeding.md)), гайды, без платного внедрения.
- **Критерий устойчивости:** суммарные предельные расходы free-парка < бюджета привлечения, который мы готовы тратить на сегмент; отношение «стоимость free-парка / выручка апселла» под контролем (ревизия ежеквартально).

## 4. Региональная последовательность выхода

> Принцип: **пилот-витрина → кейс → масштабирование по похожему сегменту/региону** ([GO_TO_MARKET §8](../../GO_TO_MARKET.md)). Старт — СПб ([GO_TO_MARKET §1](../../GO_TO_MARKET.md)).

### 4.1 Последовательность (волны)

| Волна | Сегмент / регион | Что даёт | Условие старта следующей |
|---|---|---|---|
| **Волна 0 — пилот-витрина** | **СПб, театральная библиотека (ГТБ)** — спец/ведомственная | референс-кейс, обкатка миграции из ИРБИС, отзыв, KPI (разд. 4.2) | KPI пилота достигнуты (data-fidelity, adoption, NPS) |
| **Волна 1 — СПб публичные** | городские/районные публичные СПб (~200, [GO_TO_MARKET §1](../../GO_TO_MARKET.md)), ЦБС | плотный гео-кластер, эффект «соседи перешли», кейс-тиражирование | 1–2 публичные референса + реестр (разд. 3) |
| **Волна 2 — СПб школы + вузы** | школьный сегмент (free-tier воронка, разд. 2) + 1–2 вуза | объём (школы) + Профи-референс (вуз) | школьный self-serve работает; вузовский паритет (Z2/Z4) |
| **Волна 3 — регионы РФ** | публичные/ЦБС соседних регионов → методобъединения, партнёры (разд. 5) | масштаб через партнёров, не прямые продажи | партнёрская сеть запущена (разд. 5); кейсы волн 1–2 |
| **Волна 4 — ведомственные/защищённый контур** | спец/ведомственные (по типу ГТБ), on-prem/edge | премиальная Сеть-зона, аттестация (B4) | ФСТЭК-дорожная карта пройдена (B4 §6) |

> Логика: сначала **спец-пилот** (узкий, контролируемый, наш конёк — гибкая модель данных + миграция), затем **плотный гео-кластер СПб** (сетевой эффект, низкий CAC через близость), затем **школы-объём** и **вуз-глубина**, и только потом **регионы через партнёров** (масштаб) и **ведомственные** (требуют аттестации). Не распыляться географически до отлаженного кейса/партнёрской механики.

### 4.2 KPI пилота (gate каждой волны)

| KPI | Определение | Цель **[допущение, калибровать на ГТБ]** | Источник данных |
|---|---|---|---|
| **Data-fidelity %** | доля записей/полей, корректно перенесённых при миграции (сверка зеркала, разд. 6.4) | ≥ 99 % библиозаписей; ≥ 99.x % ключевых полей; 0 потерь read-критичных | отчёт сверки миграции |
| **Adoption** | доля staff/функций, реально переведённых на новую систему (не параллельный desktop) | ключевые роли (каталогизатор/книговыдача) работают в продукте к концу пилота | usage-метрики D5/I1-аудит |
| **NPS / удовлетворённость** | опрос staff + (где применимо) читателей | положительный NPS; нет блокеров «вернёмся на ИРБИС» | опрос пилота |
| **TCO** | факт. совокупная стоимость пилота vs прогноз (разд. 1.5) | в пределах прогноза; аргумент «дешевле» подтверждён | финмодель пилота |
| **Time-to-value** | срок от старта до «работаем в проде» | в плане фаз (разд. 6.2) | runbook миграции |

> KPI пилота — **gate**: волна N+1 не стартует масштабно, пока пилот не дал кейс с этими цифрами. Это защита от «масштабирования недопечённого».

## 5. Партнёрская / интеграторская модель

> Масштаб волны 3+ (регионы) — **через партнёров**, не прямые продажи ([GO_TO_MARKET §8](../../GO_TO_MARKET.md): «партнёры-интеграторы»; [GO_TO_MARKET §10](../../GO_TO_MARKET.md): «модель партнёрства/реселлеров» — открытый вопрос → **определяем здесь**). Аналог — 1С-франчайзи ([COMPETITIVE_LANDSCAPE §1](../../COMPETITIVE_LANDSCAPE.md): «сила 1С — экосистема и франчайзи»): сетевой эффект внедренцев.

### 5.1 Уровни партнёрства

| Уровень | Роль | Что делает | Что получает |
|---|---|---|---|
| **Реселлер (Affiliate/Referral)** | приводит сделку | лидген, первичный контакт | реферальная комиссия (разд. 5.3) |
| **Сертифицированный интегратор** | продаёт + внедряет | внедрение/миграция по нашему runbook (разд. 6), L1/L2-поддержка | revenue-share + маржа на услугах |
| **Премьер-партнёр** | регион/сегмент-чемпион | объём, обучение под-партнёров, кастом-модули | повышенный share, приоритет, ко-маркетинг |

### 5.2 Критерии сертификации интегратора

- **Обученные специалисты:** прошли наш курс (внедрение/миграция/админ), сдали аттестацию (связь обучение разд. 6.5).
- **Опыт пилота:** ≥ 1 успешное внедрение по runbook под нашим кураторством.
- **Соответствие комплаенс-требованиям РФ** (если работают с гос-сегментом/ПДн): свои обязательства по 152-ФЗ.
- **Поддержка-готовность:** способны держать L1/L2 (мы — L3, разд. 5.4).
- **Ресертификация** при мажорных версиях продукта.

### 5.3 Комиссия / revenue-share / deal-registration

| Механизм | Модель **[вилка]** | Защита |
|---|---|---|
| **Реферальная комиссия** | **5–10 %** от первого года подписки **[вилка]** | разовая, за приведённого клиента |
| **Revenue-share интегратора** | **15–30 %** от подписки **[вилка]** (зависит от уровня и того, кто держит поддержку) | рекуррентно, пока партнёр обслуживает |
| **Маржа на услугах внедрения** | партнёр оставляет себе бóльшую часть платы за миграцию/обучение (разд. 6) | стимул внедрять качественно |
| **Deal-registration** | партнёр регистрирует сделку → эксклюзив на срок (напр. 60–90 дней **[вилка]**) | защита от конфликта каналов (два партнёра / партнёр vs прямые продажи) |

> **Deal-registration** обязателен, чтобы не было «войны за лида»: первый зарегистрировавший квалифицированную сделку получает защиту и повышенный share. Реестр сделок — в нашей CRM/портале партнёра.

### 5.4 Поддержка партнёров

- **Партнёрский портал:** runbook миграции (разд. 6), обучающие материалы, демо-стенды, маркетинг-кит, реестр сделок.
- **Разделение поддержки:** партнёр — **L1/L2** (первая линия, типовые вопросы, внедрение); мы — **L3** (продуктовые баги, edge/sync, сложная миграция) + сам продукт/обновления/реестр.
- **Ко-маркетинг** для премьер-партнёров; совместные кейсы (волна 3 кормится кейсами волн 1–2).

## 6. Миграция-как-услуга (пакет внедрения)

> «Миграция из ИРБИС под ключ» — ключевой снятый страх и **источник выручки** ([GO_TO_MARKET §2/§6](../../GO_TO_MARKET.md)). Техника адаптера «Проход Б» (импорт ISIS/MARC) — в I-спеках импорта; здесь — **услуга**: фазы strangler, runbook, success-критерии, P&L, обучение/SLA.

### 6.1 Подход: strangler (read → write → cutover)

Не «big-bang переключение», а **постепенное удушение legacy** ([GO_TO_MARKET §6](../../GO_TO_MARKET.md): «зеркало → сверка → переключение»; для сетей — поэтапно по площадкам):

| Фаза strangler | Что | Цель |
|---|---|---|
| **Read (зеркало)** | импорт ISIS/MARC из ИРБИС/МАРК через адаптер → читаемая копия в продукте, параллельно работающему ИРБИС | проверить полноту/корректность переноса без риска для боевой системы |
| **Write (параллельная запись)** | операции (каталогизация/выдача) пишутся в продукт; синхронизация/сверка с ИРБИС до уверенности | обкатать staff на проде в новом UI без отказа от страховки |
| **Cutover (переключение)** | ИРБИС выводится из эксплуатации; продукт — единственный источник истины | завершить миграцию; начать полную подписку |

> Для **сетей/ЦБС** — фазы применяются **поэтапно по площадкам** ([GO_TO_MARKET §6](../../GO_TO_MARKET.md)): пилотная площадка проходит read→write→cutover, затем тиражирование на остальные с готовым runbook.

### 6.2 Runbook миграции (этапы услуги)

| # | Шаг | Артефакт | Связь |
|---|---|---|---|
| M0 | **Аудит источника** — версия ИРБИС/МАРК, БД, РЛ, словари, объём, кастом-поля | отчёт аудита | — |
| M1 | **Экспорт + импорт (read)** — ISIS/MARC → продукт (адаптер «Проход Б») | зеркало-копия | I-импорт |
| M2 | **Сидинг тенанта** — `kv.mnu`/`jz.mnu`/`spec.tre`/локации/авторитеты (иначе ФЛК/МХ не работают — **онбординг-блокер**) | сиды на тенант | **A5** ([SPEC_seeding](../engines/SPEC_seeding.md)) |
| M3 | **Сверка (data-fidelity)** — автоматическая сверка зеркала с источником, отчёт расхождений | отчёт сверки (KPI разд. 6.4) | — |
| M4 | **Настройка** — РЛ/ФЛК/темы/гранты/места выдачи под клиента (low-code, D1) | конфиг тенанта | D1 ([DESIGN_COVERAGE §D1](../../DESIGN_COVERAGE.md)) |
| M5 | **Обучение staff** (разд. 6.5) | обученные роли | — |
| M6 | **Параллельная работа (write)** — staff на проде, страховка ИРБИС | usage-метрики | D5/I1 |
| M7 | **Приёмочные испытания** — success-критерии (разд. 6.4) | акт приёмки | контракт Z7 |
| M8 | **Cutover** — вывод ИРБИС, полная подписка | прод | — |
| M9 | **Гарантия/гиперкеа** — повышенная поддержка N недель после cutover | — | разд. 5.4 |

### 6.3 P&L / ценообразование услуги

> Все суммы — **[вилка]**, разовый платёж (capex клиента), отдельно от подписки ([GO_TO_MARKET §4](../../GO_TO_MARKET.md): «Плюс разовое: онбординг/миграция, обучение, кастом»).

| Пакет миграции | Объём | Цена разово **[вилка]** | Состав |
|---|---|---|---|
| **Lite** (малые/школы) | малый фонд, 1 площадка, типовые РЛ | **0–50 тыс. ₽** **[вилка]** (для free/Базовый — частично self-service/бесплатно) | M0–M3 + базовый сидинг, минимум обучения |
| **Standard** (публичные) | средний фонд, ≤3 площадки | **100–400 тыс. ₽** **[вилка]** | M0–M8, обучение, настройка |
| **Network** (ЦБС/сети) | десятки площадок, поэтапно | **от 800 тыс. ₽** **[вилка]**, per-площадка | поэтапный strangler, проектное управление, кастом |
| **Custom/ведомственные** | нестандартные базы/контур | по оценке | M0–M9 + кастом-модули + аттестация (B4) |

- **P&L-логика [допущение]:** услуга должна быть **минимум безубыточной** (покрывать трудозатраты внедрения), а в идеале — маржинальной; её ценность также в **снятии барьера перехода** (без неё подписка не стартует). При партнёрской модели (разд. 5) бóльшую часть платы за внедрение оставляет партнёр — мы зарабатываем на подписке + revenue-share.
- **Себестоимость** определяется человеко-днями (аудит/сверка/настройка/обучение); автоматизация (адаптер, автосверка, автосидинг A5) **снижает** её → даёт пространство для Lite-пакета и self-service школ.

### 6.4 Success-критерии миграции (приёмка)

| Критерий | Порог **[допущение, калибровать]** | Проверка |
|---|---|---|
| **Data-fidelity (записи)** | ≥ 99 % библиозаписей перенесены без потери | отчёт сверки M3 |
| **Data-fidelity (ключевые поля)** | ≥ 99.x % критичных полей (шифр, БО, экземпляры, читатели, выдачи) | сверка по полям |
| **Целостность связей** | RQST↔IBIS↔RDR, экземпляры, бронь — без «осиротевших» ссылок | проверка ссылочной целостности |
| **Функциональный паритет** | ключевые операции (каталог/выдача/поиск/печать) воспроизводятся | приёмочные сценарии M7 |
| **0 read-критичных потерь** | ничего критичного для работы не утрачено | ручная + авто-сверка |

### 6.5 Обучение и SLA внедрения

- **Обучение:** курсы по ролям (каталогизатор/комплектатор/книговыдача/админ — ws1–ws5), форматы: онлайн/очно/материалы/комьюнити ([GO_TO_MARKET §7](../../GO_TO_MARKET.md)). Для партнёров — программа сертификации (разд. 5.2).
- **SLA внедрения:** сроки фаз (read/write/cutover) фиксируются в контракте (Z5/Z7); гиперкеа после cutover (M9). Операционный SLA продукта — разд. 5.3 + D4-SRE.

## 8. Критерии приёмки (AC)

- **AC1 (прайс).** Зафиксированы прайс-точки 4 тарифов + бесплатного (разд. 1.1) + модули à la carte (1.2) + лимиты на тариф (1.3) + единицы метеринга и цена добора (1.4) как вход для D5; валюта ₽, НДС выделяется; каждая непроверенная цифра помечена **[вилка]/[допущение]**; есть критерий калибровки/проверки (1.6).
- **AC2 (free-tier).** Решение «да, бесплатный для школ» с лимитами (2.2), критериями апселла (2.3) и анализом устойчивости (2.4: предельные расходы малы, анти-абьюз, self-service); метрика конверсии free→paid определена.
- **AC3 (реестр).** Чек-лист R1–R6 входа в реестр отеч.ПО (3.1) + этапы/сроки подачи (3.2) с пометкой бюрократического риска; связь готовности с B4 (Astra/ФСТЭК) и I-стеком (OSS).
- **AC4 (44/223-ФЗ).** Чек-лист Z1–Z8 готовности к закупке (3.3) + контракт-шаблон (3.4, 8 разделов) + SLA-требования госзакупок (3.5); явная связь Z2 с паритетом must-have и Z3 с B4.
- **AC5 (региональная последовательность).** Волны 0–4 (4.1) с условиями старта и логикой (СПб-кластер → объём/глубина → партнёры → ведомственные) + KPI пилота как gate (4.2: data-fidelity, adoption, NPS, TCO, time-to-value).
- **AC6 (партнёры).** Уровни (5.1) + критерии сертификации (5.2) + комиссия/revenue-share/deal-registration **[вилка]** (5.3) + модель поддержки L1/L2 партнёр / L3 мы (5.4).
- **AC7 (миграция-как-услуга).** Фазы strangler read→write→cutover (6.1) + runbook M0–M9 (6.2) + P&L/прайс пакетов **[вилка]** (6.3) + success-критерии приёмки (6.4) + обучение/SLA (6.5); связь с A5-сидингом (M2 онбординг-блокер) и адаптером «Проход Б».
- **AC8 (вехи/синхрон).** G1–G7 (разд. 7) привязаны к продуктовым зависимостям и gate; выходной критерий D7 согласован с I7.
- **AC9 (честность данных).** Нигде нет выдуманных «фактов» о рынке/ценах конкурентов; все рыночные числа — **[вилка]/[допущение]** с обоснованием и способом валидации; дисклеймер в шапке.

## 10. Риски

| Риск | Митигация |
|---|---|
| **Прайс не угадан** (слишком дорого → теряем сделки / слишком дёшево → не окупаем) | вилки + калибровка на 5–8 профилях (1.6); win-rate первых сделок как сигнал; быстрая корректировка (подписка гибче коробки) |
| **Бюрократический лаг реестра отеч.ПО** (экспертиза месяцы) — критический путь к гос-выручке | начать подачу **рано** (параллельно продукту, П1–П2 заранее); до реестра — работать с негос-сегментом (вузы-внебюджет, спец, школы-free) и пилотами |
| **ТЗ закупок написаны «под конкурента»** (ИРБИС/МАРК) | матрица паритета (Z2, [COMPETITIVE_LANDSCAPE §«Паритет»](../../COMPETITIVE_LANDSCAPE.md)); ранняя работа с заказчиком на стадии формирования ТЗ; импортозамещение как обоснование |
| **Free-tier подрывает экономику** (абьюз/неконвертируемый рост) | жёсткие лимиты + мягкий тротлинг, анти-абьюз-верификация, self-service (минимум CAC), ежеквартальная ревизия (2.4) |
| **Миграция дороже/рискованнее ожидаемого** (грязные данные ИРБИС, кастом-поля) | аудит M0 до фиксации цены; автосверка M3 (data-fidelity gate); strangler — без big-bang; гиперкеа M9; цена услуги покрывает труд (6.3) |
| **Конкуренты сильнее по «облачности»** (МАРК Cloud — масштаб/гос-референс) | отрыв не на «облачности», а на edge/постаматах/маркетплейсе/AI/accessibility ([COMPETITIVE_LANDSCAPE §«Сводные выводы»](../../COMPETITIVE_LANDSCAPE.md)); пилот-кейс как ответ на их референс |
| **Канальный конфликт** (партнёр vs прямые продажи / два партнёра) | deal-registration (5.3); чёткие правила эксклюзива; партнёрский портал как единый реестр |
| **D5-биллинг не готов к G5** | старт с ручного/полу-ручного биллинга для первых сделок (малый объём), enforcement через entitlements I7; полный metering D5 — к масштабированию |
| **Числа P&L/конверсии — допущения** | помечены **[вилка]/[допущение]**; валидируются на пилоте/первых сделках; не коммуницируются как факты вовне |

## 11. Открытые вопросы (для валидации)

> Закрывают открытые вопросы [GO_TO_MARKET §10](../../GO_TO_MARKET.md) и [SPEC_I7 §E](../SPEC_I7_commercialization.md); требуют рыночных/юр/пилотных данных, не выдумываются здесь.

- **Калибровка прайса:** реальные прайс-точки по сегментам — после 5–8 профилей + пилота ГТБ + первых сделок (разд. 1.6). Конкурентные цены РФ закрыты → не сравниваем прайс-листы, только ценность.
- **НДС-режим:** ставка НДС и применимость ИТ-льгот к подписке/услугам — уточнить с финансами/юристами (разд. 1.4).
- **Порог иностранной доли и точные критерии реестра** (актуальная редакция правил Минцифры/ПП-1236) — юр.сверка перед подачей (R1).
- **Сроки экспертизы реестра** — фактический лаг неизвестен заранее (внешний риск, разд. 10); заложить буфер.
- **Free-tier пороги** (фонд/читатели/места/хранилище) — калибровать под «типичную школьную библиотеку» по реальным данным (разд. 2.2); критерий принадлежности к школе (анти-абьюз).
- **Конверсия free→paid и win-rate** — целевые значения подтверждаются метриками, не задаются как факт (разд. 2.3/1.6).
- **P&L миграции** — себестоимость человеко-дней vs цена пакетов — уточнить после первых внедрений (разд. 6.3); насколько автоматизация (адаптер/автосверка/A5) снизит труд.
- **Revenue-share/комиссии партнёров** — финальные проценты после переговоров с первыми интеграторами (разд. 5.3).
- **SMS/платёжные провайдеры РФ под закупку** — выбор под локализацию ПДн/реестр (связь [SPEC_engine_notifications §1.2](../engines/SPEC_engine_notifications.md), D5).
- **Готовность D5/B4 к вехам G1/G3/G4/G5** — синхронизировать сроки с сиблинг-спеками (разд. 7).

**D7 — GTM-исполнение** превращает стратегию ([GO_TO_MARKET](../../GO_TO_MARKET.md)) в playbook. **Прайс:** 4 тарифа (Базовый 30–70к / Стандарт 120–250к / Профи 350–800к / Сеть от 1.5 млн ₽/год — все **[вилка]**) + бесплатный (школы, 0 ₽) + модули à la carte (RFID/постаматы/оплата/консорциум/ИИ/DAM); лимиты и единицы метеринга (edge-узлы/seats/фонд/читатели/хранилище/запросы) как вход для **D5-биллинга**; валюта ₽, НДС выделяется. **Free-tier:** да для школ, жёсткие лимиты + мягкий тротлинг + анти-абьюз + self-service → устойчиво; апселл по превышению/функциям/закупке. **Реестр+госзакупки:** чек-лист R1–R6 входа в реестр отеч.ПО (российское лицо/OSS/Astra/доки/демо/РФ-локализация) + Z1–Z8 готовности к 44/223-ФЗ + контракт-шаблон (8 разделов) + SLA; готовность Astra/ФСТЭК — из **B4**, это закупочная упаковка. **Региональная последовательность:** СПб ГТБ-пилот → СПб публичные → школы/вузы → регионы через партнёров → ведомственные; KPI пилота (data-fidelity ≥99%, adoption, NPS, TCO, time-to-value) — gate каждой волны. **Партнёрская модель:** реселлер/сертифицированный интегратор/премьер; сертификация (обучение+опыт+комплаенс); комиссия 5–10% / revenue-share 15–30% / deal-registration **[вилка]**; L1/L2 партнёр, L3 мы. **Миграция-как-услуга:** strangler read→write→cutover (поэтапно по площадкам для сетей), runbook M0–M9 (с A5-сидингом как онбординг-блокером и адаптером «Проход Б»), пакеты Lite/Standard/Network/Custom **[вилка]**, success-критерии (data-fidelity ≥99%, целостность связей, 0 read-критичных потерь), обучение/SLA. Вехи G1–G7 синхронизированы с продуктом; выходной критерий — продукт вышел из пилота в коммерческое масштабирование. ⚠ Все рыночные числа — **[вилка]/[допущение]** с обоснованием и способом валидации; конкурентные цены РФ закрыты, позиционирование — по осям ценности, не по прайс-листам.
