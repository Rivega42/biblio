**Назначение.** Сквозной **слой управления ключами и PKI** для мультиарендного SaaS + edge/офлайн: (1) иерархия ключей master → per-tenant → per-category с envelope-шифрованием ПДн/секретов at-rest; (2) ротация без простоя; (3) отзыв/kill-switch при компрометации; (4) PKI для электронной подписи (ГОСТ Р 34.10-2012 / СКЗИ — СМЭВ, подпись модулей, OTA); (5) хранилище секретов (Vault/SOPS, не env); (6) ключи на edge с офлайн-доступом и крипто-стиранием. Слой даёт остальным доменам **четыре абстрактных операции** — `encrypt/decrypt`, `sign/verify`, `getSecret`, `rotate/revoke` — за которыми скрыта вся механика и сменный крипто-провайдер.

**Это слой/сервис, а не библиотека.** Реализуется как домен `kms` (key-broker + rotation-orchestrator + revocation) поверх внешнего KMS/Vault, со сменным **crypto-provider** (штатный AEAD ↔ сертифицированное ФСТЭК СКЗИ). Не дублирует тенант-изоляцию I1, supply-chain I6 §B, edge-OTA I6 §C — встраивается в них как криптографическая основа.

## 0. Решение (ADR-core) и границы

### 0.1 Контекст

Без явной модели ключей четыре уже спроектированных требования висят в воздухе: per-tenant шифрование ПДн (152-ФЗ B4), ЭП ГОСТ для СМЭВ (B3), подпись/отзыв модулей (C5), безопасное хранение кредов гос-интеграций (recon показал **антипаттерн**: `LIBNETUSER`/`LIBNETPASSWORD`, `MailUser`/`MailPassword`, токены — **открытым текстом в `irbisc.ini`**). Ретрофитить ключевую иерархию задним числом дорого ([BEST_PRACTICES §1/§6](../../BEST_PRACTICES.md) — «решить сейчас»), поэтому — ADR до Фазы 3.

### 0.2 Решение (кратко)

1. **Трёхуровневая иерархия:** root/master в внешнем **KMS/Vault Transit** → **per-tenant KEK** (key-encryption-key) → **per-category DEK** (data-encryption-key, envelope). Данные шифруются DEK; DEK хранится в обёрнутом виде (wrapped под KEK); KEK не покидает KMS (wrap/unwrap — операцией KMS, не выгрузкой).
2. **Envelope-шифрование** ПДн/секретов at-rest; per-tenant KEK даёт **крипто-изоляцию** (компрометация ключа одного тенанта не раскрывает другой) и **крипто-стирание** (уничтожение KEK ⇒ данные недоступны — дополнение к purge 152-ФЗ).
3. **Ротация** — плановая (ориентир 180 дней для KEK, конфигурируемо) **без простоя**: новая версия ключа, lazy/фоновое перешифрование, grace-окно со старой версией для дешифрования.
4. **Отзыв/kill-switch** — при компрометации тенанта/ключа/издателя: немедленная инвалидация, CRL, событие по шине, крипто-стирание как крайняя мера.
5. **PKI для ЭП** — отдельная ветвь: X.509-сертификаты ГОСТ Р 34.10-2012 через **сертифицированное СКЗИ** (CryptoPro-стек), CAdES/XMLDSig, хранение/ротация орг-сертификатов в secrets, проверка цепочки + CRL/OCSP.
6. **Секреты** — Vault (рантайм, динамические/статические) + SOPS (конфиг-as-code в git, зашифрован); **никогда env-переменные в проде**; доступ под грантом + аудит каждой ключевой операции.
7. **Сменный crypto-provider** — доменная логика зовёт абстрактные операции; под капотом — штатный AEAD (AES-256-GCM / ChaCha20-Poly1305) **или** сертифицированный ГОСТ-СКЗИ (Магма/Кузнечик, Стрибог) для УЗ-2/защищённого контура. Замена провайдера **не меняет доменную логику** (152-ФЗ §5.2 AC-F4).

### 0.3 Границы (что НЕ здесь)

- **Сам RBAC/гранты/JWT/общий аудит** → I1 #101. Здесь — какие гранты охраняют ключевые операции (разд. 5.3) и куда ложатся записи аудита крипто-операций.
- **Тенант-изоляция БД (RLS/`search_path`)** → I1 #100. Здесь — как KEK очерчивает **криптографическую** границу тенанта поверх логической.
- **Что именно шифровать / маскировать / журнал доступа к ПДн** → [SPEC_compliance_152fz](../gov/SPEC_compliance_152fz.md) (разд. 5). Здесь — **механика** ключа, которым шифруют.
- **Wire-форматы СМЭВ-конверта, профили НЭБ, маппинг** → [SPEC_gov_exchange](../gov/SPEC_gov_exchange.md). Здесь — `sign()`/`verify()` и хранение/ротация орг-сертификата, который тот pipeline зовёт.
- **Sandbox/capability/дескриптор модуля** → [SPEC_module_security](../diff/SPEC_module_security.md). Здесь — ротация/CRL/kill-switch для подписи модулей (общий механизм отзыва, разд. 4/7).
- **Supply-chain подпись образов (cosign), SBOM, Trivy** → [SPEC_I6 §B](../SPEC_I6_security_compliance.md). Здесь — управление **ключами** этой подписи (ротация cosign-ключа, keyless/Fulcio) как частный случай PKI (разд. 7.4).
- **mTLS между сервисами** (механика zero-trust) → [SPEC_I6 §A](../SPEC_I6_security_compliance.md). Здесь — выпуск/ротация **сервисных сертификатов** mTLS из внутреннего CA (разд. 7.3).

## 1. Иерархия ключей

### 1.1 Уровни

Трёхуровневая модель (envelope encryption, [BEST_PRACTICES §1](../../BEST_PRACTICES.md)). Стрелка = «шифрует/оборачивает нижестоящий».

```
 ┌──────────────────────────────────────────────────────────────────────────┐
 │ L0  ROOT / MASTER  — в внешнем KMS/Vault Transit (HSM-backed, где есть)    │
 │     • не экспортируется НИКОГДА; wrap/unwrap — операция KMS, не выгрузка    │
 │     • единый корень доверия инсталляции; ротируется редко (годы)           │
 └───────────────┬──────────────────────────────────────────────────────────┘
                 │ wrap
 ┌───────────────▼──────────────────────────────────────────────────────────┐
 │ L1  TENANT KEK  (key-encryption-key, ОДИН на тенанта)                      │
 │     • крипто-граница тенанта: компрометация KEK(A) ≠ раскрытие данных B    │
 │     • живёт в KMS под master; приложению доступен ТОЛЬКО как ref+операция  │
 │     • ротация ~180д (разд. 2); версионируется (kek_v1, kek_v2…)            │
 └───────────────┬──────────────────────────────────────────────────────────┘
                 │ wrap (envelope)
 ┌───────────────▼──────────────────────────────────────────────────────────┐
 │ L2  DATA DEK  (data-encryption-key, per-category — где нужно)              │
 │     • реально шифрует данные (AEAD); хранится РЯДОМ с данными в wrapped-виде│
 │     • категории: pii | secrets | files | backups | search_index | edge    │
 │     • дешёвая ротация (перевыпуск DEK без трогания KEK); кэш в памяти TTL   │
 └──────────────────────────────────────────────────────────────────────────┘
```

- **L0 master** — один на инсталляцию (cloud-регион / on-prem-контур). HSM-backed в KMS, где доступно; для защищённого контура — сертифицированный аппаратный модуль/СКЗИ. Никогда не покидает KMS.
- **L1 tenant KEK** — **ровно один активный на тенанта** (плюс старые версии в grace). Это и есть «per-tenant envelope-ключ», на который ссылаются 152-ФЗ §5.2 и I6 §A AC2. Операции wrap/unwrap DEK идут **в KMS** под `kekRef` — приложение KEK в открытом виде не видит.
- **L2 DEK** — **per-data-category** (не per-row по умолчанию — баланс производительности; per-row/per-field DEK только для особо чувствительного, разд. 1.3). DEK генерируется приложением (или KMS data-key API), сразу оборачивается KEK, и **хранится в обёрнутом виде рядом с данными** (`wrapped_dek` в строке-конверте/метаданных партиции). Расшифровка: unwrap DEK через KMS (под KEK) → AEAD-decrypt данных. DEK в открытом виде живёт только в памяти, под TTL-кэшем.

### 1.2 Деривация и per-category разбиение

- **Per-tenant — отдельный ключ, НЕ деривация из `tenant_id`.** Грунтовка задаёт вопрос «HKDF из tenant_id?» — **решение: нет** для KEK. `tenant_id` не секрет (он в JWT, в путях, в логах); деривация ключа из несекретного публичного значения не даёт крипто-стойкости и ломает крипто-стирание (нельзя «уничтожить» детерминированно выводимый ключ). KEK — **независимо сгенерированный** случайный ключ в KMS, `tenant_id` лишь **адресует** его (`kekRef = kms://tenant/{tenant_id}/kek`).
- **HKDF — уместен на L2 для sub-ключей одной категории:** из одного DEK категории можно вывести контекстные под-ключи через **HKDF с info=`{category}:{purpose}:{tenant_id}`** (доменное разделение, RFC 5869). Здесь `tenant_id` идёт в `info` как **связывание контекста** (защита от подмены контекста), а не как источник энтропии — энтропия из DEK. Это законное применение HKDF; деривация всего KEK из `tenant_id` — нет.
- **Per-category DEK** — отдельный DEK на категорию данных, потому что у категорий разная ротация/ретенция/чувствительность:

| Категория DEK | Что шифрует | Связь |
|---|---|---|
| `pii` | ПДн-поля RDR (10–14,17,18,32,950…), история чтения | [152-ФЗ §1.1/§5.2](../gov/SPEC_compliance_152fz.md) — реестр ПДн-полей |
| `secrets` | креды интеграций (`LIBNETPASSWORD`, `MailPassword`, токены СМЭВ), приватные ключи ЭП | [gov_exchange §0](../gov/SPEC_gov_exchange.md) — recon-антипаттерн открытого текста |
| `files` | ПТ/образы/обложки в хранилище объектов | [reader §3](../SPEC_reader_jirbis.md) (прокси-файлы) |
| `backups` | резервные копии БД тенанта | [152-ФЗ §4.2](../gov/SPEC_compliance_152fz.md) (забвение из бэкапа = крипто-стирание) |
| `search_index` | чувствительные поля в OpenSearch (если включён) | [BEST_PRACTICES §5](../../BEST_PRACTICES.md) |
| `edge` | БД-на-тенанта + реплики на edge-узле | разд. 6, [SPEC_I4](../SPEC_I4_edge_offline.md) |

### 1.3 Гранулярность (per-category vs per-row)

- **По умолчанию — per-category DEK** (производительность: один unwrap на категорию, кэш в памяти). Изоляция между категориями: компрометация `files`-DEK не раскрывает `pii`.
- **Per-row/per-field DEK** — для **особо чувствительного** (паспорт поле 14, поля здоровья 41–49 — спец. категория 152-ФЗ): отдельный DEK на запись/поле, обёрнутый категорийным KEK-путём, что даёт точечное крипто-стирание субъекта (уничтожил row-DEK → именно его паспорт недоступен) без перешифрования всей категории. Решает баланс «забвение в бэкапе» точечнее (разд. 6.4).
- **Не шифруем индивидуальным ключом всё** — это убивает поиск/джойны; шифруется **at-rest на уровне хранилища/колонок-ПДн**, не каждое значение в индексе (поиск по ПДн — через защищённый бэкенд, не через расшифровку миллионов строк).

### 1.4 Что шифруется (at-rest, B4)

ПДн-хранилища ([152-ФЗ §5.2](../gov/SPEC_compliance_152fz.md)) — обязательно per-tenant: ПДн-поля RDR, история чтения, бэкапы, edge-реплики, секреты, приватные ключи ЭП. Не-ПДн каталог/фонд — шифрование тома/БД (storage-level), per-tenant DEK не обязателен (нет крипто-изоляционного требования), но at-rest-шифрование тома — да (ФСТЭК ЗИС).

## 2. Ротация без простоя

### 2.1 Принцип «версия ключа, не замена»

Ротация **не перешифровывает синхронно весь датасет** (это простой и риск). Вместо этого:

- Каждый ключ **версионируется** (`kek_v1 → kek_v2`); и DEK, и зашифрованные блобы несут **`keyVersion`** в метаданных конверта.
- Новая версия становится **активной для записи** немедленно; старые версии остаются валидными **для чтения** (grace), пока существуют данные, зашифрованные ими.
- **Перешифрование — ленивое + фоновое:** при следующей записи строка перешифровывается новой версией (lazy re-wrap); фоновый job-«перешифровщик» (rate-limited, с курсором) проходит холодные данные. Простоя нет: чтение всегда находит нужную версию ключа.

### 2.2 Расписание и grace

| Ключ | Период ротации (ориентир, конфигурируемо) | Grace (старая версия валидна для чтения) |
|---|---|---|
| L0 master | редко (1–3 года) / по инциденту | до полного re-wrap всех KEK |
| L1 tenant KEK | **~180 дней** (по умолчанию) | до полного перешифрования DEK тенанта, ≥1 цикл |
| L2 DEK (category) | ~90 дней / по объёму | до lazy+фон re-encrypt категории |
| ЭП орг-сертификат | по сроку сертификата (обычно 12–15 мес) | до истечения старого + overlap-окно (разд. 4.3) |
| mTLS сервис-сертификат | короткий (часы–дни, авто) | overlap при rolling-restart |
| cosign/подпись модулей | по расписанию + по инциденту | старые подписи валидны до явного отзыва (Rekor хранит историю, [module_security §4.4](../diff/SPEC_module_security.md)) |

- **Период — `tenant_key_policy.rotation_*`, конфигурируемый per-tenant** (точные сроки не зашиты в код/документ; гос-тенант может требовать жёстче). 180д — дефолт-ориентир, не норматив.
- **Внеплановая ротация** — по событию (компрометация, увольнение администратора с доступом к ключам, требование комплаенса) — тот же механизм версии, но с **форсированным** ускоренным перешифрованием и (если компрометация) отзывом старой версии (разд. 3).

### 2.3 Оркестратор ротации

```
rotate(kekRef):
  [1] kms.createNewVersion(kekRef)        → kek_v(n+1) активна для wrap новых DEK
  [2] mark policy: writeVersion = n+1, readVersions = {…, n, n+1}
  [3] enqueue re-encrypt job (cursor, rate-limit, per-category)
      • lazy: каждая запись данных перешифровывает свой DEK под kek_v(n+1)
      • batch: фоновый проход холодных партиций (backpressure, окно вне пиков)
  [4] when 0 blobs reference kek_vn → schedule destroy(kek_vn) после grace-TTL
  [5] audit каждую фазу (kekRef, from→to version, progress, trace_id)  — разд. 5.3
```

- **Идемпотентность/возобновляемость:** job хранит курсор; падение/рестарт продолжает с курсора, повторный проход не портит уже-перешифрованное (проверка `keyVersion` блоба).
- **Наблюдаемость:** метрики `key_rotation_progress{tenant,kek_version}`, `blobs_pending_reencrypt`, алерт на «застрявшую» ротацию (старая версия не уничтожается X дней → ручной разбор).
- **Edge-нюанс:** перешифрование edge-реплик идёт на узле по его расписанию синка; пока узел офлайн — старая версия в его grace-наборе валидна (разд. 6).

## 3. Отзыв / kill-switch

### 3.1 Триггеры отзыва

Компрометация тенанта/ключа/издателя; утечка приватного ключа ЭП; уход администратора с привилегиями KMS; требование регулятора; CVE в крипто-провайдере; отзыв модуля ([module_security §4.4](../diff/SPEC_module_security.md)).

### 3.2 Уровни реакции (по серьёзности)

```
 нарастание серьёзности →
 [rotate]      плановая/внеплановая ротация (разд. 2) — старое валидно в grace
 [disable]     версия ключа помечена revoked-for-write (новые данные — только новой версией),
               но чтение старой ещё разрешено до форсированного перешифрования
 [revoke]      версия/ключ ОТОЗВАН: ни write, ни read; данные под ним требуют немедленного
               перешифрования или становятся недоступны; запись в CRL (разд. 7.5)
 [crypto-erase] УНИЧТОЖЕНИЕ KEK тенанта → все DEK под ним необорачиваемы → данные
               НЕВОССТАНОВИМЫ (крайняя мера: компрометация без возможности контейнмента,
               либо «забвение» целого тенанта/субъекта — 152-ФЗ §5.2 «крипто-стирание»)
```

### 3.3 Kill-switch тенанта (компрометация)

- **Мгновенная блокировка операций тенанта:** при сигнале компрометации — `tenant_key_state = suspended`: KMS отказывает в unwrap для этого `kekRef` (кроме форензики под особым грантом) → приложение не может расшифровать ПДн тенанта → **данные «заморожены»**, утечка остановлена даже если БД-дамп уже у атакующего (без KEK дамп бесполезен — это и есть ценность per-tenant envelope + крипто-изоляции).
- **Распространение:** событие `key.revoked`/`tenant.key_suspended` по шине ([SPEC_I7 §B](../SPEC_I7_commercialization.md) #107) → все сервисы (и edge-узлы при синке) сбрасывают кэш DEK тенанта, прекращают расшифровку. Офлайн-edge применяет при ближайшем синке; для «горящих» — короткий grace на CRL (как [module_security §4.4/R5](../diff/SPEC_module_security.md)).
- **Уведомление:** админ тенанта + платформенный security — через A6 ([SPEC_engine_notifications](../engines/SPEC_engine_notifications.md)); инцидент в наблюдаемость (I6 §D).

### 3.4 Крипто-стирание как контейнмент и забвение

- **Контейнмент:** если компрометация неустранима, уничтожение KEK тенанта (`kms.destroy(kekRef)`) делает все DEK под ним необорачиваемыми → данные тенанта недоступны навсегда (в т.ч. в уже-снятых бэкапах, если они шифровались тем же деревом). Это **последний рубеж**, не рутинная операция: требует двойного подтверждения (разд. 5.4) и аудита.
- **Забвение (152-ФЗ):** уничтожение **row/field-DEK** субъекта (разд. 1.3) делает именно его паспорт/спец-данные недоступными — дополнение к purge ([152-ФЗ §4.2](../gov/SPEC_compliance_152fz.md): «ПДн в неизменяемых бэкапах нельзя стереть мгновенно → крипто-стирание»). Решает R2 152-ФЗ.

## 4. PKI для электронной подписи (ГОСТ / СМЭВ / гос)

> Отдельная **ветвь PKI** от симметричного envelope-дерева (разд. 1): здесь — асимметрия, X.509, удостоверяющие центры, CRL/OCSP. Потребители: СМЭВ ([gov_exchange §3.2](../gov/SPEC_gov_exchange.md) — `blocked-on:D2`), подпись модулей ([module_security §4](../diff/SPEC_module_security.md)), in-future НЭБ/СКБР по требованию площадки.

### 4.1 Алгоритмы и СКЗИ

- **Подпись для гос-обмена:** **ГОСТ Р 34.10-2012** (256/512-бит, кривые `id-tc26-gost-3410-12-256/512`), хеш **ГОСТ Р 34.11-2012 (Стрибог)**. Шифрование (где нужно гост) — Магма/Кузнечик (ГОСТ Р 34.12-2015).
- **СКЗИ:** криптооперации ГОСТ выполняются **сертифицированным ФСТЭК/ФСБ СКЗИ** (CryptoPro CSP / VipNet или аппаратный токен), а **не** штатным OpenSSL-GOST в проде гос-сегмента ([152-ФЗ §5.2/§6.1 п.7](../gov/SPEC_compliance_152fz.md)). Crypto-provider абстрагирует вызов: `sign(payload, certRef, {profile: gost2012})` → провайдер маршрутизирует в СКЗИ.
- **Штатный (не-гос) контур:** для внутренних подписей (модули, mTLS, образы) — Ed25519/ECDSA-P256 + SHA-256; ГОСТ там не требуется.

### 4.2 Форматы подписи

| Контекст | Формат | Грунтовка |
|---|---|---|
| СМЭВ-конверт (SOAP бизнес-контент) | **XMLDSig** (enveloped) + **CAdES-BES/-T** на `MessagePrimaryContent`, ЭП-ОВ органа власти | [gov_exchange §3.2](../gov/SPEC_gov_exchange.md) |
| Транспортная подпись конверта СМЭВ | XMLDSig по спецификации СМЭВ 3 | там же |
| Подпись модуля маркетплейса | detached signature (`.kpz`) cosign/keyless+Rekor — **не ГОСТ** (внутренняя) | [module_security §4.2](../diff/SPEC_module_security.md) |
| НЭБ/СКБР (если площадка требует) | CAdES detached, по флагу `signPolicy` профиля | [gov_exchange §6.4](../gov/SPEC_gov_exchange.md) |
| Документы оператора (опц., будущее) | CAdES/PAdES | — |

### 4.3 Хранение и ротация орг-сертификатов

- **Приватный ключ ЭП-ОВ — НИКОГДА в БД/конфиге/коде.** Хранится: (а) на **аппаратном токене/HSM** СКЗИ (предпочтительно для гос); (б) в **Vault** (категория DEK `secrets`, обёрнут tenant-KEK) — для cloud-варианта без HSM. Доступ — только через `sign()` по `certRef`; материал ключа приложению не выдаётся.
- **`certRef`, не значение** — конфиги/профили обмена несут **ссылку** на сертификат в secrets ([gov_exchange §0/§7 `ExchangeProfile.certRef`](../gov/SPEC_gov_exchange.md)), никогда сам ключ. Это прямой ответ на recon-антипаттерн (креды открытым текстом).
- **Ротация сертификата — overlap-окно:** новый сертификат выпускается/устанавливается **до** истечения старого; оба валидны в окне перекрытия; новые подписи — новым, проверка старых подписей — по сертификату на момент подписания (CAdES-T с меткой времени фиксирует «подписано, пока сертификат был валиден»). Истёкший/отозванный сертификат → `signPolicy:required` блокирует отправку ([gov_exchange §6.4](../gov/SPEC_gov_exchange.md)).
- **Метка времени (TSP):** CAdES-T штампует подпись доверенной меткой времени (служба TSA), чтобы подпись оставалась проверяемой после истечения/ротации сертификата.

### 4.4 Проверка цепочки и статуса

- **Verify:** валидация цепочки до доверенного корня (УЦ, аккредитованный Минцифры для гос); проверка **не-отозванности** по **CRL** (периодический список) и/или **OCSP** (онлайн-запрос); проверка срока; проверка соответствия алгоритма политике.
- **Доверенные корни** — управляемый trust-store (УЦ Минцифры/ФНС для гос; платформенный CA для внутренних). Обновление trust-store — операция под грантом + аудит.

### 4.5 Внутренние УЦ (CA)

- **Платформенный CA** для внутренних сертификатов (mTLS-сервисы разд. 7.3, контр-подпись модулей разд. 7.4). Корневой CA — offline/HSM; промежуточный issuing-CA — в Vault PKI-engine, выпускает короткоживущие листовые сертификаты автоматически.

## 5. Хранилище секретов и доступ

### 5.1 Vault (рантайм) + SOPS (конфиг-as-code)

- **Vault** — рантайм-секреты: статические (креды интеграций, токены), **динамические** (короткоживущие БД-креды, выдаваемые по запросу с TTL), Transit-engine (KMS-операции wrap/unwrap/sign), PKI-engine (выпуск mTLS/внутренних сертификатов). **Никогда не env-переменные в проде** ([BEST_PRACTICES §6](../../BEST_PRACTICES.md), [SPEC_I6 §B](../SPEC_I6_security_compliance.md)).
- **SOPS** — секреты, которые обязаны жить в git (конфиг-as-code, IaC — связь D3): файл в репозитории **зашифрован** SOPS под KMS/age-ключом; в открытом виде секрета в git нет. Расшифровка — только в CI/деплое с доступом к KMS-ключу.
- **Антипаттерн recon — не воспроизводим:** `irbisc.ini` хранил `LIBNETPASSWORD`/`MailPassword`/токены открытым текстом ([gov_exchange §0](../gov/SPEC_gov_exchange.md)). Все такие значения → Vault, в конфиге — `*Ref`.

### 5.2 Модель доступа к секретам

- **Принцип наименьших привилегий:** сервис получает доступ только к своим секретам (Vault-политика на путь `secret/{tenant}/{service}/*`). Аутентификация сервиса в Vault — по рабочей нагрузке (Kubernetes ServiceAccount / AppRole / SPIFFE), **не** статический root-token.
- **TTL + auto-renew:** выданный секрет/lease истекает; долгоживущих статических кредов избегаем (динамические БД-креды предпочтительнее).
- **Тенант-изоляция секретов:** путь секрета несёт `tenant_id`; доступ под тем же инвариантом, что данные (I1 #100); cross-tenant secret access — негатив-тест.

### 5.3 Аудит ключевых операций

- **Каждая** криптооперация уровня управления — `wrap/unwrap` (на уровне ротации/админ, не каждый decrypt-горячего-пути), `sign` орг-сертификатом, `rotate`, `revoke`, `destroy`, выдача/чтение секрета, изменение политики ключа, обновление trust-store — пишется в **аудит** ([SPEC_I1 #101](../SPEC_I1_foundation.md): actor, tenant, операция, объект, результат) + специфика: `keyRef`, `keyVersion`, `algo`, `provider`, `certRef`, `trace_id` (#117).
- **Высокочастотный горячий путь** (per-row decrypt ПДн при показе карточки) **не** дублируется в ключевой аудит — он уже покрыт **журналом доступа к ПДн** ([152-ФЗ §3](../gov/SPEC_compliance_152fz.md)); ключевой аудит ловит **управление ключами**, не каждое применение.
- **Аудит крипто-операций — append-only, неизменяем**, ретенция длительная (доказательная база для ФСТЭК/проверок); сам доступ к нему — высокий грант (security-администратор/DPO).

### 5.4 Защита привилегированных операций (four-eyes)

- **`destroy(KEK)` / крипто-стирание тенанта / отзыв ключа / обновление корневого trust-store** — требуют **двойного подтверждения** (two-person rule / Vault `mfa` + второй approver): один администратор инициирует, второй подтверждает; обе личности и обоснование — в аудит. Это барьер против единичной злонамеренной/ошибочной потери данных тенанта.

## 6. Ключи на edge-узле (офлайн, защита)

> Edge = БД-на-тенанта на узле ([ARCHITECTURE §5](../../ARCHITECTURE.md), [SPEC_I4](../SPEC_I4_edge_offline.md)); ПДн живут и там → тот же режим защиты, но узел **периодически офлайн** и физически вне периметра ЦОД (библиотека, постамат). Ключи на edge — отдельная задача.

### 6.1 Что хранится на узле

- **edge-DEK** категории `edge` (шифрует локальную БД-реплику + файлы на узле), обёрнутый **edge-KEK узла**.
- **edge-KEK** — производный/выданный из tenant-KEK, **привязан к узлу** (node-bound): хранится в TPM/secure-enclave узла, где есть (Astra Linux + аппаратный корень доверия); иначе — в зашифрованном keystore узла, ключ от которого приходит при онлайн-активации.

### 6.2 Офлайн-доступ (узел работает без облака)

- Узел должен **расшифровывать локальные ПДн офлайн** (выдача книг при обрыве связи — основной сценарий [SPEC_I4](../SPEC_I4_edge_offline.md)). Значит edge-DEK/edge-KEK должны быть доступны узлу **локально** в офлайне — это снижает защиту против физической кражи узла. Контрмеры:
  - **node-bound ключ в TPM/enclave** — ключ не извлекаем с диска; кража диска без TPM-узла бесполезна.
  - **офлайн-grace с TTL:** edge-KEK валиден офлайн ограниченное время; по истечении без синка — узел требует переподключения для продления (как офлайн-лицензия [SPEC_I6 §C](../SPEC_I6_security_compliance.md), [BEST_PRACTICES §7](../../BEST_PRACTICES.md)). Баланс: достаточно длинный для реальных обрывов, достаточно короткий, чтобы украденный узел «протух».
  - **минимизация ПДн на узле:** на edge — только оперативно необходимое (активные читатели/выдачи), не весь архив (снижает ущерб от компрометации узла; связь [152-ФЗ §5.1](../gov/SPEC_compliance_152fz.md) минимизация).

### 6.3 Ротация и отзыв на edge

- **Ротация tenant-KEK** распространяется на узлы при синке (новый wrapped edge-DEK приходит, перешифрование локальной реплики — фоном на узле, разд. 2.3).
- **Отзыв/kill-switch узла** (узел украден/скомпрометирован): событие `node.key_revoked` → при ближайшем синке узел стирает локальные ключи и реплику; **офлайн-узел протухает по grace-TTL** сам (не может продлить edge-KEK без облака) — украденный офлайн-узел теряет доступ к ПДн по истечении grace, даже не связавшись с облаком.

### 6.4 Крипто-стирание и забвение на edge

- Запрос на забвение ([152-ФЗ §4.2 edge-нюанс](../gov/SPEC_compliance_152fz.md)) — событие в sync-журнале; на узле применяется уничтожением соответствующего row/field-DEK субъекта (разд. 1.3) → его ПДн на узле недоступны, даже если физический блок ещё на диске до перезаписи. Офлайн-узел — «отложенное забвение» с дедлайном по восстановлении связи.

## 7. API / модель данных / интеграция

### 7.1 Абстрактный crypto-сервис (внутренний контракт)

Домены зовут **только эти операции**; механика (envelope, версии, провайдер) скрыта.

```jsonc
CryptoService {
  // симметрия (envelope) — данные at-rest
  encrypt(tenantId, category, plaintext, aad?) -> { ciphertext, keyVersion, wrappedDek? }
  decrypt(tenantId, category, ciphertext, keyVersion, aad?) -> plaintext
  // секреты
  getSecret(tenantId, ref) -> SecretValue          // из Vault; материал не логируется
  putSecret(tenantId, ref, value) -> void          // запись под политикой+аудит
  // асимметрия (PKI / ЭП)
  sign(tenantId, certRef, payload, { profile: "gost2012"|"cades"|"xmldsig"|"cosign" }) -> Signature
  verify(payload, signature, { trustStore, checkRevocation: "crl"|"ocsp" }) -> VerifyResult
  // управление (под высоким грантом + four-eyes для destroy)
  rotate(tenantId, keyRef) -> RotationJob
  revoke(keyRef | certRef | moduleVersion, reason) -> void   // → CRL + событие шины
  cryptoErase(tenantId, scope: "tenant"|"subject", subjectId?) -> EraseManifest  // four-eyes
}
```

### 7.2 Модель данных (метаданные ключей — НЕ материал)

```
tenant_key_policy
  tenant_id        uuid PK
  kek_ref          text          -- kms://tenant/{id}/kek (адрес, НЕ ключ)
  write_version    int           -- активная версия для шифрования
  read_versions    int[]         -- версии, валидные для дешифрования (grace)
  rotation_period  interval      -- per-tenant (дефолт ~180д, конфиг)
  state            enum          -- active | rotating | suspended | revoked
  uz_level         int           -- влияет на провайдера (УЗ-2 ⇒ сертиф. СКЗИ), связь 152-ФЗ §1.2
  provider         enum          -- aead_default | gost_skzi
  updated_at       timestamptz

data_key_wrap                    -- учёт DEK (обёрнутых); материал DEK НЕ здесь в открытом виде
  id, tenant_id, category enum, key_version int,
  wrapped_dek bytea,             -- DEK, обёрнутый KEK (хранится у данных/партиции)
  created_at, retired_at NULL

cert_registry                    -- орг-сертификаты ЭП (метаданные + ref, НЕ приватный ключ)
  id, tenant_id, cert_ref text,  -- ссылка в Vault/HSM
  subject text, issuer text, algo enum,         -- gost2012_256/512 | ecdsa_p256 | ed25519
  not_before, not_after, status enum,           -- active | rotating | revoked | expired
  purpose enum                                  -- smev_org | module_sign | mtls | tls

revocation_list                  -- CRL (ключи/сертификаты/версии модулей)
  id, tenant_id NULL,            -- NULL = платформенный (модули), иначе тенантный
  revoked_ref text,              -- keyRef@version | certRef | module_id@version | digest
  reason enum, revoked_at timestamptz, propagated_at timestamptz NULL

key_audit  (append-only, immutable)  -- разд. 5.3; actor, op, keyRef, keyVersion, algo, provider, trace_id, result
```

### 7.3 mTLS-сертификаты сервисов

- Выпускаются Vault PKI-engine (issuing-CA разд. 4.5), **короткоживущие** (часы), auto-renew при rolling-restart; identity сервиса — SPIFFE/SVID. Ротация — встроена (короткий TTL = частая авто-ротация, overlap при рестарте). Механика mTLS (zero-trust) — [SPEC_I6 §A](../SPEC_I6_security_compliance.md); здесь — управление **ключами/сертификатами** этого mTLS.

### 7.4 Ключи подписи образов/модулей (supply-chain)

- **cosign-ключ образов** ([SPEC_I6 §B](../SPEC_I6_security_compliance.md)) — keyless (Fulcio/OIDC + Rekor) предпочтительно (нет долгоживущего приватного ключа, аудит в transparency log); либо ключевая пара в Vault с ротацией. Ротация/отзыв — этот слой (разд. 2/3).
- **Контр-подпись модулей платформой** ([module_security §4.2](../diff/SPEC_module_security.md)) — платформенным ключом из issuing-CA; ротация платформенного ключа = массовая (старые контр-подписи валидны до отзыва, Rekor хранит).

### 7.5 CRL и распространение отзыва

- `GET /api/kms/revocations` — список отозванных (для edge-кеша холодного старта), как [module_security §8 `/revocations`](../diff/SPEC_module_security.md). Live-отзыв — **событием по шине** ([SPEC_I7 §B](../SPEC_I7_commercialization.md) #107); REST — для холодного старта/edge-кеша с TTL+grace.

### 7.6 REST-шов (к переносу в openapi.yaml оркестратором)

> Наследует конверт `{ok,data}`/`{ok,error}`, `bearerAuth`, тенант-из-JWT, `Idempotency-Key` ([SPEC_api_contracts §0](../engines/SPEC_api_contracts.md)). `tag: kms`. Операции с ключами — **высокий грант** `security.keys:*:admin`; деструктивные — `+ four-eyes`. Живой `openapi.yaml` этой задачей **не редактируется** — фрагмент переносит оркестратор.

| POST | `/api/kms/tenant/{id}/rotate` | Запустить ротацию tenant-KEK (→ `202 {jobId}`) | `security.keys:*:admin` |

| POST | `/api/kms/tenant/{id}/crypto-erase` | Крипто-стирание (tenant\|subject) — **four-eyes** | `security.keys:*:admin` ×2 |

## 8. Критерии приёмки (AC)

- **AC-K1 (иерархия/envelope).** ПДн-поля/секреты/бэкапы/edge-реплики зашифрованы per-tenant: дамп БД/бэкапа **без доступа к KMS** не раскрывает ПДн (тест: расшифровка дампа невозможна без KEK). DEK хранятся только в обёрнутом виде; KEK не покидает KMS.
- **AC-K2 (крипто-изоляция).** Компрометация/доступ к KEK тенанта A не даёт расшифровать данные тенанта B (раздельные независимые KEK, не деривация из `tenant_id`); негатив-тест.
- **AC-K3 (ротация без простоя).** Ротация tenant-KEK не вызывает простоя: во время перешифрования и чтение, и запись работают; старые версии валидны для чтения в grace; по завершении старая версия уничтожается; job возобновляем после рестарта (курсор).
- **AC-K4 (отзыв/kill-switch).** Сигнал компрометации тенанта → unwrap для его KEK блокируется немедленно (данные «заморожены»), событие распространяется на сервисы и edge (офлайн — при синке/по grace); попытка decrypt после отзыва → отказ.
- **AC-K5 (крипто-стирание = забвение).** Уничтожение row/field-DEK субъекта делает его ПДн (в т.ч. в бэкапе) нерасшифровываемыми — реализует «забвение из неизменяемого бэкапа» ([152-ФЗ R2](../gov/SPEC_compliance_152fz.md)); `destroy(KEK)` тенанта требует four-eyes.
- **AC-K6 (ЭП ГОСТ).** `sign(payload, certRef, {profile:gost2012})` даёт подпись ГОСТ Р 34.10-2012 через сертифицированный СКЗИ; СМЭВ-конверт подписан (CAdES/XMLDSig); без валидного сертификата `signPolicy:required` блокирует отправку ([gov_exchange AC5](../gov/SPEC_gov_exchange.md)). Снимает `blocked-on:D2` с `smev`-адаптера.
- **AC-K7 (хранение/ротация сертификата).** Приватный ключ ЭП — только в HSM/Vault, в конфиге — `certRef` (не значение); ротация сертификата — overlap-окно без разрыва подписи; CAdES-T фиксирует метку времени.
- **AC-K8 (секреты не в env/конфиге).** Креды интеграций/SMTP/токены — в Vault (рантайм) / SOPS (git), `*Ref` в конфиге; grep по конфигам/образам/логам не находит секретов открытым текстом (паритет-исправление recon-антипаттерна); gitleaks-гейт в CI ([SPEC_I6 §B](../SPEC_I6_security_compliance.md)).
- **AC-K9 (аудит ключевых операций).** rotate/revoke/sign-орг/destroy/secret-access/trust-store-change → неизменяемая запись `key_audit` (actor, keyRef, version, algo, provider, trace_id); деструктивные — с двумя подтверждающими (four-eyes).
- **AC-K10 (edge-ключи/офлайн).** Узел расшифровывает локальные ПДн офлайн (node-bound ключ TPM/enclave); украденный офлайн-узел теряет доступ по grace-TTL без синка; ротация/отзыв KEK применяется на узле при синке.
- **AC-K11 (сменный провайдер).** Замена крипто-провайдера (штатный AEAD ↔ сертифицированный ГОСТ-СКЗИ) **не меняет доменную логику** — переключается конфигом `tenant_key_policy.provider` по `uz_level` ([152-ФЗ AC-F4](../gov/SPEC_compliance_152fz.md)).
- **AC-K12 (verify + revocation).** `verify` проверяет цепочку до доверенного корня + не-отозванность (CRL/OCSP) + срок; отозванный/истёкший сертификат/ключ модуля отвергается ([module_security AC-C5-6](../diff/SPEC_module_security.md)).

| Edge офлайн | расшифровка без облака | в пределах grace | по истечении grace без синка | работает / протух |
| Edge кража | украденный офлайн-узел | — | доступ к ПДн после grace | недоступно (node-bound + grace) |

## 10. Критический путь

1. **Crypto-сервис (абстракция) + KMS/Vault Transit + per-tenant KEK + per-category DEK (envelope)** (разд. 1, 7.1) — фундамент; на нём строятся 152-ФЗ §5.2 (ПДн at-rest) и секреты.
2. **Vault/SOPS секреты + `*Ref`-модель + миграция recon-кредов** (разд. 5) — закрывает антипаттерн открытого текста; разблокирует gov-интеграции (креды) и A6 (SMTP).
3. **Ротация без простоя (версии + lazy/фон перешифрование) + аудит ключевых операций** (разд. 2, 5.3) — без неё ключи «застывают».
4. **Отзыв/kill-switch + CRL + событие по шине + крипто-стирание (four-eyes)** (разд. 3, 7.5) — контейнмент + забвение из бэкапа (152-ФЗ R2).
5. **PKI ЭП: ГОСТ-СКЗИ-провайдер + sign/verify + хранение/ротация орг-сертификата + CAdES/XMLDSig** (разд. 4) — **снимает `blocked-on:D2`** со СМЭВ ([gov_exchange](../gov/SPEC_gov_exchange.md)); + ротация/CRL подписи модулей ([module_security §4.4](../diff/SPEC_module_security.md)).
6. **Edge-ключи: node-bound TPM/enclave + офлайн-grace + распространение ротации/отзыва на узлы** (разд. 6) — закрывает ПДн на edge.
7. **mTLS/cosign-ключи (issuing-CA, короткий TTL, keyless)** (разд. 7.3/7.4) — supply-chain/zero-trust как частный случай.

**Зависимости:** I1 #100 (тенант-граница для KEK), I1 #101 (гранты/аудит ключевых операций), I1 #117 (trace_id корреляции), I4 (edge-распространение/офлайн-grace), I6 §A (изоляция/mTLS-механика), I6 §B (cosign/SBOM — управление их ключами), I6 §C (edge-OTA grace-паттерн), A6 (уведомление о компрометации/ротации), шина I7 #107 (распространение отзыва).

**Выходной критерий D2:** платформа шифрует ПДн/секреты/бэкапы/edge at-rest **per-tenant envelope-ключом** (крипто-изоляция + крипто-стирание); **ротирует без простоя** (версии + ленивое/фоновое перешифрование, grace); **отзывает/kill-switch'ит** ключ при компрометации (заморозка unwrap, CRL, событие, four-eyes destroy); даёт **PKI для ЭП ГОСТ Р 34.10-2012 через сертифицированное СКЗИ** (CAdES/XMLDSig — снимает `blocked-on:D2` со СМЭВ, обслуживает подпись модулей/OTA); хранит **секреты в Vault/SOPS, не env** (исправляя recon-антипаттерн), под аудитом каждой ключевой операции; защищает **ключи на edge** (node-bound, офлайн-grace, протухание украденного узла); и абстрагирует **сменный крипто-провайдер** (штатный AEAD ↔ ГОСТ-СКЗИ по УЗ) — три комплаенс-блокера (B3 СМЭВ, B4 ПДн, C5 модули) получают единую криптографическую основу.

## 11. Риски и открытые вопросы

| # | Риск | Митигация |
|---|---|---|
| R1 | **Производительность горячего пути** — unwrap DEK на каждый показ карточки убивает латентность | DEK-кэш в памяти (TTL, инвалидация при ротации/отзыве); unwrap KEK — редко, decrypt AEAD — дёшево; per-category, не per-row по умолчанию (разд. 1.3) |
| R2 | **Потеря master/KEK = потеря всех данных** (крипто-стирание не отличимо от аварии) | KMS HA + резервирование master (escrow под строгим контролем); four-eyes на destroy; бэкап политики ключей (метаданных), не самого ключа; DR-учения восстановления KMS |
| R3 | **СКЗИ-ограничения** — сертифицированный ГОСТ-стек (CryptoPro) накладывает форматы ключей/токены/лицензии, плохо ложится на cloud-автоматизацию | абстракция провайдера (разд. 0.2/4.1); пилот УЗ-2 на аппаратном токене; для cloud-не-гос — штатный AEAD |
| R4 | **Edge: офлайн-доступ vs физическая кража** — ключ на узле снижает защиту от кражи | node-bound TPM/enclave + офлайн-grace-TTL (протухание) + минимизация ПДн на узле (разд. 6.2) |
| R5 | **Ротация «застревает»** — старая версия не уничтожается, т.к. остались блобы под ней (холодные партиции/застрявший edge) | алерт на застрявшую ротацию; форс-перешифрование; для edge — grace до синка с дедлайном |
| R6 | **Зависимость СМЭВ/модулей от D2** — если PKI задержится, B3/C5 стоят | ветвь PKI (разд. 4) изолирована от envelope (разд. 1) — можно поставлять envelope раньше PKI; `sign()`-абстракция позволяет stub до СКЗИ |
| R7 | **Управление trust-store/корнями УЦ** — устаревший/скомпрометированный корень ломает verify | trust-store под грантом+аудитом+four-eyes на корень; OCSP для свежести; CAdES-T метка времени переживает ротацию |

**Открытые вопросы:**

- **Q1 (внешний KMS: какой и где).** Vault Transit (self-hosted, под нашим контролем, ложится в защищённый контур/Astra) **vs** managed-KMS (облачный, проще ops, но внешняя зависимость и вопрос для гос-сегмента/импортозамещения). Для гос-контура почти наверняка self-hosted Vault + аппаратный СКЗИ; для cloud-SaaS-не-гос — managed может быть дешевле. Влияет на разд. 1/5 и на ФСТЭК-готовность. **Нужно решение по умолчанию + матрица «контур → KMS».**
- **Q2 (гранулярность DEK для забвения).** Per-category (дёшево, но забвение субъекта из бэкапа = перешифрование всей категории или ждать ротации) **vs** per-subject/row-DEK для ПДн (точечное крипто-стирание субъекта, но N ключей на тенанта — нагрузка на учёт DEK и кэш). Где порог: row-DEK только для спец.категории (паспорт/здоровье), category-DEK для остального? **Связь [152-ФЗ §4.2 R2](../gov/SPEC_compliance_152fz.md).**
- **Q3 (истина ключевого состояния для edge-офлайн).** Если узел офлайн, а в облаке ключ **отозван/ротирован** — узел этого не знает до синка. Как балансировать: длинный офлайн-grace (узел работает в обрывах, но дольше живёт со скомпрометированным/старым ключом) vs короткий (безопаснее, но узел чаще «протухает» в реальных обрывах). Нужна **дифференцированная grace-политика** (короткий grace для revoke-компрометации, длинный для плановой ротации) — параллель открытому вопросу Q2 [152-ФЗ](../gov/SPEC_compliance_152fz.md) и R5 [module_security](../diff/SPEC_module_security.md). **Связь [SPEC_I4](../SPEC_I4_edge_offline.md).**

## Сводка

**Иерархия ключей + envelope:** трёхуровневая — **L0 master** (в внешнем KMS/Vault Transit, HSM-backed, не экспортируется) → **L1 per-tenant KEK** (один на тенанта, **независимый случайный ключ, НЕ деривация из `tenant_id`** — даёт крипто-изоляцию и крипто-стирание; ротация ~180д) → **L2 per-category DEK** (envelope, реально шифрует AEAD, хранится обёрнутым рядом с данными; категории pii/secrets/files/backups/edge; per-row DEK только для спец.категории паспорт/здоровье). ПДн/секреты/бэкапы/edge at-rest зашифрованы per-tenant (B4). HKDF — законен на L2 для контекстных под-ключей (`info`=category:purpose:tenant), не для вывода KEK.

**Ротация/отзыв:** ротация **без простоя** — версионирование ключа + lazy (при записи) и фоновое (rate-limited, курсор) перешифрование, grace-набор валидных-для-чтения версий, уничтожение старой по завершении. Отзыв — лестница `rotate → disable → revoke → crypto-erase`: kill-switch тенанта = **заморозка unwrap** (дамп без KEK бесполезен) + CRL + событие по шине ([I7 #107](../SPEC_I7_commercialization.md)) на сервисы и edge (офлайн — по grace); крипто-стирание (уничтожение KEK тенанта или row/field-DEK субъекта) = забвение из неизменяемого бэкапа, под **four-eyes**. PKI-ветвь отдельно: **ГОСТ Р 34.10-2012 через сертифицированное СКЗИ**, CAdES/XMLDSig, орг-сертификат в HSM/Vault по `certRef` (не значение), ротация overlap-окном + CAdES-T метка — **снимает `blocked-on:D2` со СМЭВ**. Секреты — **Vault/SOPS, не env** (исправляет recon-антипаттерн открытого текста), аудит каждой ключевой операции (append-only). Сменный крипто-провайдер (AEAD ↔ ГОСТ-СКЗИ по УЗ) не меняет доменную логику.
