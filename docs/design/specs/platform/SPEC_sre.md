## 0. Принципы и границы

- **SLO ориентированы на пользователя, не на ресурс.** Цель — «поиск отвечает быстро и правильно», «книга выдаётся надёжно», а не «CPU < 80%». CPU/память — это USE-сигналы для диагностики (разд. 2.4), пороги алертов — по SLI пользовательских потоков (разд. 1). Это прямое требование BEST_PRACTICES §8 («SLO на ключевые потоки: поиск, выдача»).
- **Error budget — валюта надёжности.** Доступность 99.9% = бюджет 43.2 мин/мес недоступности. Пока бюджет не исчерпан — приоритет фич; исчерпан — приоритет надёжности (заморозка рискованных релизов, разд. 1.4). Это делает «надёжность» измеримой и управляемой, а не лозунгом.
- **Наблюдаемость уже заложена (#117) — мы её доращиваем.** I1 #117 дал OTel-SDK, сквозной `trace_id`+`tenant_id`, JSON-логи, RED-метрики. Этот спек добавляет поверх: **SLI-определения** (что считать «хорошим» запросом), **SLO-дашборды** (RED/USE), **алерты по burn-rate**, контроль **кардинальности/стоимости** телеметрии. Не дублируем инструментацию — задаём, что с ней делать.
- **Деградация вместо отказа — архитектурный инвариант** (I6/§D AC3). Недоступность одного домена не роняет продукт: при недоступной книговыдаче **поиск/каталог продолжают работать** (read-only discovery); при недоступном внешнем SMS-провайдере уведомления уходят in-app (см. notify-движок); при недоступной БД-реплике чтение уходит на primary с деградацией латентности. Разд. 4.
- **DR — проверенный процесс, не «есть бэкапы».** Бэкап без регулярного restore-учения = непроверенное предположение. Мы задаём **график учений** (разд. 3.4) и измеряем фактические RPO/RTO на учении против целевых. «Бэкап считается рабочим только после успешного restore-учения».
- **Нет алерта без runbook.** Каждое алерт-правило ссылается на плейбук с шагами диагностики и устранения (разд. 5). Алерт, на который не знаешь что делать, — шум. Это DoD-гейт.
- **Мультиарендность — в каждом сигнале.** SLI/метрики/трейсы несут `tenant_id` (из #117); SLO можно резать по тенанту (выявить «шумного соседа» / деградацию конкретной библиотеки); DR — per-tenant restore (восстановить одну схему тенанта, не весь кластер, разд. 3.5). Нарушение изоляции тенанта — **инцидент Sev-1** с собственным runbook (разд. 5.2).
- **Границы.** SRE задаёт **цели и процессы наблюдаемости/надёжности/восстановления**. Он **не** реализует инфраструктуру (репликация, гео-хранилище, мульти-регион — D3 IaC), **не** пишет бизнес-логику деградации (её реализуют сервисы по контракту health/CB здесь), **не** заменяет 152-ФЗ-журнал доступа (это gov/152-ФЗ; SRE-логи — операционные, ПДн-минимизированы, разд. 2.5). Edge-fleet-телеметрия/OTA-откат — дисциплина D3/I4, SRE даёт для неё SLO узла (разд. 1.3) и runbook «edge офлайн >24ч» (разд. 5.2).

## 1. SLO / SLI — числовые цели

### 1.1 Каталог пользовательских потоков (SLI определения)

SLI = доля «хороших» событий / всего событий в окне. «Хорошее» определено явно (success-критерий), измеряется из RED-метрик #117 (по `route`, `tenant_id`).

| Поток (journey) | SLI-тип | Определение «хорошего» события | Окно измерения | Источник метрики |
|---|---|---|---|---|
| **Поиск / discovery** (OPAC, поиск каталога) | латентность + доступность | HTTP 2xx/3xx **и** server-time ≤ порог латентности (разд. 1.2) | rolling 28 дней + 1ч/5м для burn | RED по `route=/api/search`, гистограмма латентности |
| **Книговыдача** (выдача/возврат/продление) | доступность + латентность | мутация завершилась 2xx (зафиксирована в БД) **и** ≤ порог | 28 дней | RED по `route=/api/loan/*`, исход транзакции |
| **Просмотр записи / БО** (record fetch) | латентность | 2xx и ≤ порог | 28 дней | RED по `route=/api/record/*` |
| **Аутентификация** (вход читателя/staff, ЕСИА) | доступность | 2xx (или валидный 401 на неверные креды — это «хороший» отказ, не ошибка) | 28 дней | RED по `route=/api/auth/*` |
| **Доступность платформы** (composite) | доступность | синтетический probe ключевых эндпоинтов из вне = 2xx | 28 дней | blackbox-probe (разд. 2.6) |
| **Синхронизация edge↔облако** | свежесть (freshness) + успех | sync-цикл завершился без конфликт-эскалации **и** лаг репликации ≤ порог | 28 дней | sync-метрики (I4), `replication_lag` |
| **Доставка уведомлений** | успех доставки | событие → ≥1 канал accepted в SLA-окне (см. notify) | 28 дней | notify-метрики enqueued/sent/failed |

> **Корректность как SLI.** Для поиска отдельно от латентности фиксируем **error-rate индексации** (записи, не попавшие в индекс) и **stale-index-lag** (отставание индекса от каталога) — «быстрый, но пустой поиск» = плохой SLI, даже если 2xx. Порог: lag индекса ≤ 60 с (p95) после коммита записи.

### 1.2 Целевые пороги латентности (server-side, без сети клиента)

| Поток | p50 | p95 | p99 | Примечание |
|---|---|---|---|---|
| **Поиск** (PG FTS / OpenSearch) | ≤ 120 мс | **≤ 300 мс** | ≤ 800 мс | холодный фасет/тяжёлый запрос — отдельный бакет, см. ниже |
| **Просмотр записи** | ≤ 50 мс | ≤ 150 мс | ≤ 400 мс | из кэша/БД |
| **Книговыдача (мутация)** | ≤ 150 мс | ≤ 500 мс | ≤ 1200 мс | включает запись в БД + событие в outbox |
| **Автокомплит/подсказки** | ≤ 40 мс | ≤ 100 мс | ≤ 250 мс | интерактивный, жёстче |
| **Аутентификация (локальная)** | ≤ 100 мс | ≤ 300 мс | ≤ 700 мс | ЕСИА-redirect — внешняя зависимость, исключена из SLI (разд. 1.5) |
| **Внешний протокол (Z39.50/SRU ответ)** | ≤ 200 мс | ≤ 600 мс | ≤ 1500 мс | гос-клиенты, [gov/SPEC_protocols.md](../gov/SPEC_protocols.md) |

> **Тяжёлые запросы.** Запросы с >N фасетов / широким диапазоном / экспортом — отдельный класс (`query_class=heavy`) с собственным бюджетом (p95 ≤ 2 с) и rate-limit; не загрязняют SLI интерактивного поиска. Экспорт/отчёты/реиндексация — **асинхронные** (статус-эндпоинт), вне латентного SLO (разд. 1.5).

### 1.3 Целевые SLO доступности и свежести

| Цель | Значение | Бюджет ошибок (на 28 дн) | Обоснование |
|---|---|---|---|
| **Доступность платформы (composite, облако)** | **99.9%** | 40.3 мин/28дн (≈43.2 мин/30дн) | базовый прод-таргет для SaaS; пилот СПб ГТБ |
| **Доступность поиска/discovery** | **99.95%** | 20.2 мин/28дн | публичный OPAC — лицо продукта, читатели снаружи |
| **Доступность книговыдачи (staff-критично в часы работы)** | **99.9%** в рабочие часы | — | вне рабочих часов — относ. SLO 99.5% (окно обслуживания) |
| **Аутентификация** | 99.9% | 40.3 мин | вход — ворота ко всему |
| **Успех мутаций книговыдачи** | ≥ 99.9% (доля 2xx от валидных) | — | потеря выдачи недопустима; 5xx считаются против бюджета |
| **Свежесть индекса поиска** | p95 lag ≤ 60 с | — | «коммит записи виден в поиске» |
| **Edge-узел: доступность локального обслуживания** | **99.5%** локально (автономно) | — | edge обслуживает офлайн; sync — отдельный SLO |
| **Edge-синк: свежесть** | p95 sync-лаг ≤ 15 мин (онлайн) | — | при разрыве — деградация, не нарушение (узел работает) |
| **Долговечность данных (durability)** | ≥ 99.999999% (8×9) | — | задаётся гео-репликацией бэкапов (D3); потеря зафиксированной записи недопустима |

> **Tiering целей по типу тенанта/окружения.** Прод-облако (multi-AZ) — 99.9–99.95% как выше. **Single-node on-prem / edge** — релаксированный SLO (99.5%) с явной коммуникацией в SLA: одна нода = нет HA, бюджет тратится на окна обслуживания. **Гос-сегмент (ФСТЭК-контур)** — те же числа, но восстановление только внутри защищённого контура (разд. 3.6). SLA-tier фиксируется в entitlements тенанта (I1 #101).

### 1.4 Error budget и burn-rate-политика

- **Error budget** = (1 − SLO) × окно. Для 99.9% / 28 дн = **0.1% = 40.3 мин** «плохого времени» (или эквивалент в плохих запросах для request-based SLI).
- **Multi-window multi-burn-rate алерты** (Google SRE-практика) — два окна на каждый порог, чтобы ловить и быстрые катастрофы, и медленную деградацию без флаппинга:

| Severity | Burn-rate | Окна (long / short) | Доля бюджета за окно | Что значит | Действие |
|---|---|---|---|---|---|
| **Page (Sev-2)** | **14.4×** | 1 ч / 5 мин | 2% бюджета за 1ч | бюджет 28дн сгорит за ~2 дня | будит on-call немедленно |
| **Page (Sev-2)** | **6×** | 6 ч / 30 мин | 5% за 6ч | сгорит за ~4–5 дней | будит on-call |
| **Ticket (Sev-3)** | **3×** | 24 ч / 2 ч | 10% за сутки | медленная деградация | тикет в рабочее время |
| **Ticket (Sev-3)** | **1×** | 72 ч / 6 ч | — | трендовая | тикет, разбор на review |

  Алерт срабатывает, только когда **оба** окна (long и short) превышают порог — short-окно гарантирует, что проблема ещё активна (анти-флаппинг).
- **Политика расходования бюджета (budget policy):**
  - Бюджет **> 50%** остатка → нормальная работа, релизы по обычному процессу.
  - Бюджет **10–50%** → повышенная осторожность: рискованные изменения только с canary + быстрым rollback; усиленный мониторинг после деплоя.
  - Бюджет **исчерпан (≤ 0)** → **feature-freeze критического потока**: запрещены изменения, не повышающие надёжность; вся команда переключается на восстановление бюджета; post-mortem обязателен. Гейт релиза (DoD) блокирует деплой в затронутый поток.
  - **Сброс** — по скользящему окну (28 дн); бюджет восстанавливается естественно по мере выхода плохих минут из окна.
- **Силовое исключение (error-budget exemption):** запланированное окно обслуживания, заранее объявленное тенантам, **не списывается** с бюджета (помечается как maintenance, исключается из SLI-знаменателя) — но только если объявлено за N часов и в согласованном окне.

### 1.5 Что исключено из SLI (явно)

- Внешние зависимости вне нашего контроля: ЕСИА-redirect, внешний SMS-агрегатор, СМЭВ-ответы — измеряются отдельно (dependency-SLI), но не входят в наш availability-SLO (иначе чужой даунтайм жжёт наш бюджет). Их недоступность → деградация (разд. 4), своя алерт-политика.
- Асинхронные джобы (реиндексация, экспорт, bulk-миграция IBIS, дайджесты) — у них **throughput/freshness-SLO**, не латентный.
- Клиентская сеть/устройство — server-time SLI, не end-to-end (RUM собираем отдельно для UX, но не для SLO-гейта).
- Запланированное обслуживание в объявленном окне (разд. 1.4).

## 2. Наблюдаемость (observability)

### 2.1 Три сигнала на базе OTel (#117)

Опора — уже внедрённый OTel-SDK (#117): трейсы+метрики+логи, сквозной `trace_id`, `tenant_id` в span/лог, JSON-логи, OTLP-экспорт. Доращиваем:

| Сигнал | Что добавляем поверх #117 | Бэкенд (за OTLP, выбор D3) |
|---|---|---|
| **Метрики** | RED (Rate/Errors/Duration) по каждому `route` + USE (Utilization/Saturation/Errors) по ресурсам + бизнес-метрики (выдачи/час, индекс-лаг, sync-лаг, доставки) | Prometheus-совместимый TSDB |
| **Трейсы** | sampling-политика (разд. 2.7), span-атрибуты `tenant_id`/`route`/`query_class`/`db.statement`(нормализ.), exemplar-связь метрика↔трейс | Tempo/Jaeger-класс |
| **Логи** | структурные JSON, обязательные поля (разд. 2.5), корреляция `trace_id`↔лог, уровни, ПДн-минимизация | Loki-класс |

**Обязательная корреляция:** из любой точки можно перейти метрика → exemplar-трейс → лог того же `trace_id`. Это требование #117 AC3 («корреляция лог↔трейс работает»), здесь делается обязательным для всех сервисов.

### 2.2 Обязательные атрибуты на каждом сигнале

`trace_id`, `span_id`, `tenant_id`, `service`, `route`/`operation`, `http.status`, `query_class` (для поиска), `result` (ok/error/degraded), `deployment.version` (для корреляции деградации с релизом), `region`/`node_id` (edge). Запрет: ПДн в атрибутах/лейблах (разд. 2.5, 2.8).

### 2.3 Дашборды RED (по сервису/потоку)

Стандартный набор на каждый SLO-поток (as-code, версионируется):

- **Rate** — RPS по `route`, разрез по `tenant_id` (топ-N), по версии деплоя.
- **Errors** — error-rate (5xx + бизнес-ошибки), разрез по причине, по тенанту.
- **Duration** — p50/p95/p99 латентность (гистограмма), heatmap, отдельный бакет `query_class=heavy`.
- **SLO-панель** — текущий SLI vs target, остаток error budget (gauge), burn-rate за 1ч/6ч/24ч, прогноз исчерпания.

### 2.4 Дашборды USE (по ресурсу)

Для диагностики (не для алертов-SLO): **Utilization** (CPU/mem/диск/коннекты), **Saturation** (длина очередей, PgBouncer wait, очередь воркеров notify/sync, replication lag), **Errors** (OOM, рестарты подов, ошибки диска). Ключевые: **PostgreSQL** (соединения/locks/longest query/replication lag/bloat/WAL), **PgBouncer** (waiting clients/pool saturation), **очереди** (notify/sync/индексация — глубина, возраст головы), **объектное хранилище** (latency/errors).

### 2.5 Логи: контракт и ПДн-минимизация

- **Формат:** JSON, обязательные поля `ts`, `level`, `service`, `trace_id`, `tenant_id`, `route`, `msg`, `event` (машиночитаемый код).
- **Уровни:** ERROR (нарушение SLO/исключение) / WARN (деградация/ретрай) / INFO (бизнес-событие) / DEBUG (off в проде, включается per-tenant/per-trace по флагу).
- **ПДн-минимизация (152-ФЗ, [gov/SPEC_compliance_152fz.md](../gov/SPEC_compliance_152fz.md)):** операционные логи **не содержат** ПДн читателя (ФИО/паспорт/адрес/контакты) — только `reader_id`/`tenant_id` и хэши; `db.statement` нормализуется (плейсхолдеры вместо значений). Журнал **доступа к ПДн** — отдельная сущность gov/152-ФЗ, не SRE-лог.
- **Ретеншен:** операционные логи — 30 дней горячие / 90 дней холодные (архив), затем удаление; метрики — 13 мес (год+ для годовых трендов/сравнения); трейсы — 7–15 дней (сэмплированные). Конкретика — per-tenant политика, дефолты здесь.

### 2.6 Health / readiness / synthetic probes

- `/healthz` (**liveness**) — процесс жив (не проверяет зависимости); провал → рестарт пода. Лёгкий, без БД.
- `/readyz` (**readiness**) — готов принимать трафик: проверяет критические зависимости (PG-коннект, миграции применены, кэш) с таймаутом; провал → исключение из балансировки (не рестарт). Уже задан в I1 прод-упаковке (`/healthz`+`/readyz`).
- **Startup-probe** — для медленного старта (прогрев индекса/кэша), чтобы liveness не убивал стартующий под.
- **Blackbox synthetic probes** — внешний прозвон ключевых journey (поиск, вход, fetch записи) каждые 30–60 с из нескольких точек → это источник composite-availability-SLI (разд. 1.3) и независимое подтверждение «снаружи видно».

### 2.7 Sampling трейсов

- **Голова (head-based):** базовый rate (напр. 10% обычного трафика) + **100% для ошибок и медленных** (tail-based по латентности/статусу, если коллектор поддерживает) → дешёво, но не теряем интересное.
- **Принудительный трейс:** по флагу запроса / per-tenant debug → 100% для разбора инцидента конкретного тенанта.
- **Exemplars:** метрики несут ссылки на репрезентативные трейсы (быстрый переход «пик латентности → конкретный трейс»).

### 2.8 Кардинальность и стоимость

- **Бюджет кардинальности:** лейблы метрик — **только ограниченный набор** (route, status, tenant_id, query_class, version, region). **Запрещены** в лейблах: `reader_id`, MFN, поисковый запрос, любой неограниченный ID, ПДн — это и cardinality-взрыв, и утечка ПДн. `tenant_id` ограничен числом тенантов (контролируемо); при росте до тысяч тенантов — агрегация/топ-N + отдельный высоко-кардинальный store по требованию.
- **Контроль стоимости:** алерт на рост числа активных серий (cardinality budget); sampling трейсов; ретеншен по разд. 2.5; лог-сэмплинг для шумных INFO. Цель — наблюдаемость не должна стоить дороже самого сервиса; ежемесячный обзор top-cardinality-метрик.
- **Edge:** ограниченный канал — узел шлёт **агрегаты** (не полный трейс-поток) + батч при онлайне; локальный буфер при разрыве; приоритет — алерты/SLI узла, детальные трейсы — по запросу.

## 3. Disaster Recovery (DR) — RPO/RTO, бэкапы, учения

### 3.1 Целевые RPO/RTO по классам инцидента

**RPO** (recovery point objective) — допустимая потеря данных; **RTO** (recovery time objective) — допустимое время восстановления.

| Класс инцидента | Целевой RPO | Целевой RTO | Механизм восстановления | Реализует |
|---|---|---|---|---|
| **Сбой пода/инстанса сервиса** (stateless) | 0 | ≤ 1 мин | авто-рестарт/перепланирование, health-probe | D3 (оркестратор) |
| **Сбой primary PostgreSQL (в регионе)** | **≤ 1 мин** | **≤ 5 мин** | автоматический failover на горячую реплику (streaming replication) | D3 (репликация) |
| **Потеря/повреждение БД (логическая)** | **≤ 5 мин** | **≤ 1 ч** | PITR из WAL-архива (point-in-time recovery) | бэкап-политика 3.2 |
| **Потеря всего региона** (DR-сценарий) | **≤ 5 мин** | **≤ 4 ч** | promote реплики в другом регионе / restore из гео-бэкапа | D3 (мульти-регион) |
| **Потеря объектного хранилища** (сканы/файлы) | ≤ 15 мин | ≤ 2 ч | гео-реплицированный бакет / restore | D3 |
| **Порча данных одного тенанта** (баг/ошибка оператора) | ≤ 5 мин | ≤ 1 ч | **per-tenant restore** схемы из бэкапа (разд. 3.5) | бэкап-политика |
| **Edge-узел умер** (диск/железо) | ≤ последний sync | ≤ время замены + re-sync | re-provision узла (D3 IaC) + re-sync из облака-master (каталог) + локальный бэкап оперативных данных | D3/I4 |
| **Ransomware / компрометация** | ≤ последний immutable-бэкап | ≤ 8 ч | restore из **immutable/offline** копии (разд. 3.2) | бэкап-политика + I6 |

> Это **целевые** значения для прод-облака (multi-AZ). On-prem/single-node — релаксированные (RTO часами, RPO до суток при ночном бэкапе), явно в SLA. Фактические RPO/RTO **измеряются на учениях** (разд. 3.4) и сверяются с целевыми.

### 3.2 Бэкап-политика (частота / retention / гео)

| Что | Метод | Частота | Retention | Гео |
|---|---|---|---|---|
| **PostgreSQL (база)** | базовый снапшот (`pg_basebackup`/снапшот тома) | ежедневно (ночное окно) | 35 дней | основной + **второй регион** |
| **PostgreSQL (WAL)** | непрерывный архив WAL → объектное хранилище | непрерывно (поток) | 14 дней (для PITR) | основной + второй регион |
| **Логический дамп per-tenant** | `pg_dump` схемы тенанта | еженедельно + по требованию | 90 дней | гео |
| **Объектное хранилище** (сканы DG/CDL, файлы) | репликация бакета + версионирование | непрерывно/часовая | 35 дней версий | гео |
| **Конфиги/секреты-references** (не сами секреты) | GitOps-репозиторий + Vault-снапшот | при изменении + ежедн. | история git + 35 дн | гео |
| **Immutable/offline копия** (анти-ransomware) | WORM/object-lock на ежедн. снапшот | ежедневно | 14 дней object-lock | гео, неудаляемая |

- **Шифрование бэкапов at-rest** (per-tenant envelope, связь I6 D2/PKI [ADR-006]); ключи — отдельно от данных, восстановимы.
- **Стратегия 3-2-1:** ≥3 копии, ≥2 носителя/региона, ≥1 offline/immutable.
- **Целостность:** контрольные суммы + автоматическая верификация восстановимости (test-restore в изолированный namespace, разд. 3.4) — бэкап «зелёный» только после успешного test-restore, не после успешной записи.

### 3.3 Каталог восстановления (DR runbook)

Документированный, версионируемый **DR-runbook** (разд. 5) на каждый класс 3.1: предусловия, кто инициирует (роль), пошаговое восстановление, точки верификации, ожидаемые RPO/RTO, контакты/эскалация. Хранится **вне восстанавливаемой системы** (отдельный репозиторий + офлайн-копия — иначе при полном отказе runbook недоступен).

### 3.4 Регулярные restore-учения (rehearsal) — график

> **Прямое закрытие I6/§D AC2** («DR: восстановление из бэкапа проверено учением») и BEST_PRACTICES §8 («регулярные restore-учения»). Учение = реальный restore в изолированное окружение + замер фактических RPO/RTO + проверка целостности.

| Учение | Периодичность | Что проверяет | Критерий успеха |
|---|---|---|---|
| **Авто test-restore** | **ежедневно** (CI/джоба) | вчерашний бэкап восстанавливается в эфемерный namespace, схема валидна, ключевые инварианты-запросы проходят | restore завершён, проверочные запросы зелёные; провал → алерт «бэкап не восстановим» |
| **Per-tenant restore** | **ежемесячно** | восстановление одной схемы тенанта в изоляцию, сверка контрольных сумм | RTO ≤ 1 ч, данные целы |
| **PITR-учение** | **ежеквартально** | восстановление на точку времени T (до «ошибочной» операции) | достигнут RPO ≤ 5 мин, корректная точка |
| **Региональный failover (game day)** | **ежеквартально** | promote реплики/restore в другом регионе, переключение трафика | RTO ≤ 4 ч, RPO ≤ 5 мин, продукт работает в DR-регионе |
| **Полный DR + chaos** | **раз в полгода** | сценарий «регион потерян» end-to-end + хаос (разд. 4.5) + проверка runbook’ов «вживую» | команда восстановила по runbook без эскалации к разработчику-автору; зафиксированы gaps |
| **Edge-recovery** | **ежеквартально** | «узел умер» → re-provision + re-sync, потеря только оперативных данных | узел вернулся в строй, каталог re-synced, конфликты разобраны |

- **Game day обязателен по runbook’у** (не «по памяти эксперта») — учение проверяет и сам runbook (актуальность шагов). Каждое учение → запись фактических RPO/RTO в трекинг + **action items** на расхождения.
- **Авто-учение блокирует «зелёный» статус бэкапа** (DoD): если ежедневный test-restore упал — бэкап считается нерабочим, Sev-2.

### 3.5 Per-tenant восстановление

Мультиарендность даёт гранулярность: схема-на-тенанта позволяет восстановить **одного** тенанта (порча данных/ошибка оператора/запрос «откатите вчерашнюю глобальную корректировку `.gbl`») **без** влияния на других. Runbook 3.3 включает per-tenant-вариант: restore схемы тенанта в staging → верификация → swap/merge. RTO per-tenant ≤ 1 ч. Изоляция при restore: восстановленные данные не «протекают» в другие схемы (тот же тест изоляции, что в I1 #100).

### 3.6 Гос-сегмент (ФСТЭК-контур)

Для аттестованного контура (I6/§E, [gov/SPEC_compliance_152fz.md](../gov/SPEC_compliance_152fz.md)): бэкапы и restore — **только внутри защищённого периметра** (без внешних облаков/CDN — уже соблюдено в дизайн-системе); гео-копия — во второй аттестованный ЦОД заказчика; ключи/секреты — в его контуре. RPO/RTO те же, но реализация on-prem (D3). DR-учения документируются для аттестации (доказательная база).

## 4. Resilience — деградация, health, circuit breakers, rate-limit

### 4.1 Graceful degradation — матрица (I6/§D AC3)

«Деградация без полного отказа»: при недоступности зависимости продукт теряет функцию, не падает целиком.

| Недоступна зависимость | Деградация (что продолжает работать) | Что отключается/откладывается | Сигнал пользователю |
|---|---|---|---|
| **Книговыдача** (сервис/БД loan) | **Поиск/каталог/discovery работают** (read-only OPAC) | выдача/возврат/продление, личный кабинет операций | баннер «операции временно недоступны» |
| **Реплика чтения PG** | чтение уходит на primary (выше латентность) | — | прозрачно, метрика деградации |
| **Кэш (Redis)** | работа напрямую с БД (медленнее) | — | прозрачно |
| **Поиск (OpenSearch)** | fallback на **PG FTS** (если включён) или «поиск временно ограничен» | продвинутые фасеты/релевантность | мягкое сообщение |
| **Внешний SMS-провайдер** | уведомления уходят **in-app/email** (терминальный канал, см. notify) | SMS-канал | — (fallback-цепочка) |
| **ЕСИА** | локальный вход (если разрешён тенантом) работает | вход через госуслуги | «вход через ЕСИА недоступен, используйте логин» |
| **Объектное хранилище** (сканы) | метаданные/каталог работают | просмотр скан-образов/полнотекста | плейсхолдер «файл временно недоступен» |
| **Событийная шина** (#107) | синхронный путь (`POST /api/notify`, прямые вызовы) + outbox-буфер | асинхронная обработка отложена (догоняет) | прозрачно (eventual) |
| **Edge: разрыв с облаком** | **узел обслуживает локально автономно** (I4) | свежесть каталога/sync | индикатор «офлайн-режим» |

> **Принцип load-shedding:** при перегрузке — сбрасываем **низкоприоритетный** трафик (тяжёлые экспорты, фоновые джобы, аналитика), защищая критический путь (поиск/выдача/вход). Приоритизация по `query_class`/маршруту.

### 4.2 Health/readiness как контракт деградации

`/readyz` — управляет деградацией: сервис, потерявший некритичную зависимость, остаётся **ready** (деградирует), но рапортует деградацию в метрику `degraded=1`; потеря **критичной** зависимости → `not ready` → выводится из балансировки (а не отдаёт ошибки). Разграничение critical/non-critical зависимостей — в манифесте сервиса (разд. 0).

### 4.3 Circuit breakers

На каждый вызов внешней/межсервисной зависимости — **автомат (circuit breaker)** (паттерн Closed→Open→Half-Open):

- **Порог открытия:** доля ошибок > 50% на окне (напр. 20 запросов / 10 с) **или** p99-латентность > порог × K → размыкание (fail-fast, не копим зависшие коннекты).
- **Open:** мгновенный отказ/fallback (разд. 4.1) на `cooldown` (напр. 30 с), не долбим упавшую зависимость (анти-cascade).
- **Half-Open:** пробные запросы; успех → Closed, провал → снова Open.
- **Сопутствующее:** таймауты на **каждый** внешний вызов (нет бесконечного ожидания), bounded retries с экспоненциальным backoff + **jitter** (анти-thundering-herd), bulkhead (изоляция пулов — медленная зависимость не съедает все воркеры/коннекты). Каскад sync (разд. 5.2) гасится именно CB+bulkhead.

### 4.4 Rate limiting и quotas

- **Per-tenant rate-limit** (анти-noisy-neighbor, связь I6/A ресурс-лимиты): лимит RPS/burst на тенанта; превышение → 429 + `Retry-After`. Защищает SLO одних тенантов от всплеска у других.
- **Per-endpoint:** тяжёлые (`heavy`/экспорт/реиндекс) — строже; интерактивные — мягче.
- **Глобальный admission control:** при общей перегрузке (saturation USE) — load-shedding по приоритету (разд. 4.1).
- **Анти-abuse:** лимит на вход/перебор паролей (auth), на внешние протоколы (Z39.50/SRU — гос-клиенты, [gov/SPEC_protocols.md](../gov/SPEC_protocols.md)).
- Лимиты — конфиг per-tenant (entitlements/SLA-tier I1 #101); 429 не считается против error-budget (это защита, не отказ).

### 4.5 Chaos engineering

- **Hypothesis-driven** эксперименты в staging/game-day: убить под, добавить латентность/потери, разорвать edge↔облако, насытить коннекты PG, «потерять» реплику. Цель — **проверить, что деградация/CB/failover работают как задумано до реального инцидента**.
- Edge-хаос (BEST_PRACTICES §2: обрывы, переупорядочивание, дубликаты, расхождение часов, долгий офлайн) — координируется с I4/SYNC_DESIGN хаос-тестами; SRE добавляет инфра-хаос (узел/сеть/диск).
- Прод-хаос — только при зрелости (после стабильных game-day), в окне, с kill-switch.

## 5. Runbooks и on-call

### 5.1 Структура runbook (шаблон)

Каждый плейбук — версионируемый markdown в репозитории (рядом с алерт-правилами; **нет алерта без runbook** — DoD-гейт), единый шаблон:

1. **Заголовок/severity/owner** — что за алерт, какой SLO нарушен, кто владелец.
2. **Симптомы/триггер** — какое правило сработало, что видно на дашборде.
3. **Влияние (impact)** — кого/что задевает (тенанты, потоки), оценка серьёзности.
4. **Диагностика** — конкретные запросы/дашборды/трейсы (ссылки), как локализовать (какой сервис/тенант/версия; «начни с SLO-панели → exemplar-трейс → лог»).
5. **Меры (mitigation)** — пошаговое устранение, в т.ч. быстрые: rollback релиза (#118 feature-flag/деплой), масштабирование, переключение на fallback, рестарт, ручной failover.
6. **Верификация** — как убедиться, что починили (SLI вернулся, burn-rate упал).
7. **Эскалация** — кому/когда (разд. 5.3).
8. **Post-mortem-ссылка** — для Sev-1/Sev-2 обязателен blameless post-mortem.

### 5.2 Библиотека инцидент-плейбуков (минимальный набор)

| Runbook | Триггер | Ключевые шаги (суть) | Severity |
|---|---|---|---|
| **Всплеск латентности поиска** | burn-rate латентного SLO поиска (14.4×/6×) | дашборд Duration → `query_class`/тенант/версия → проверить: новый релиз (rollback), heavy-query-флуд (rate-limit/load-shed), деградация PG/индекса (план запроса, locks), saturation PgBouncer | Sev-2 |
| **Рост 5xx / падение доступности** | availability burn-rate / blackbox-probe red | определить сервис (RED) → readyz-статус → зависимость упала? (CB open?) → rollback/scale/failover → деградация включилась? | Sev-2/1 |
| **Edge-узел офлайн >24ч** | отсутствие телеметрии/heartbeat узла > 24 ч | подтвердить (сеть vs железо vs процесс) → узел обслуживает локально? (да = не Sev-1) → удалённая диагностика (D3/I4) → при «мёртв» → edge-recovery (3.1) re-provision+re-sync → разбор конфликтов синка | Sev-3 (если локально жив) / Sev-2 |
| **Sync-каскад** (лавина конфликтов/лаг растёт) | sync-лаг SLO нарушен / рост конфликт-эскалаций / очередь sync растёт | CB на sync открыть (затормозить, анти-cascade) → bulkhead защитил остальное? → причина: расхождение часов/дубли/большой backlog после долгого офлайн → дренировать backlog батчами, разобрать конфликты (I4 UI) → не дать «retry-storm» (backoff+jitter) | Sev-2 |
| **Нарушение изоляции тенанта** | тест изоляции/аномалия: запрос тенанта A коснулся данных B | **немедленно Sev-1** → зафиксировать (трейс/лог) → отрезать затронутый путь (feature-flag/rate-limit/изоляция инстанса) → оценить экспозицию ПДн (152-ФЗ-нотификация?) → откат причины (search_path/RLS, I1 #100) → post-mortem + уведомление compliance | **Sev-1** |
| **Исчерпан error budget** | budget ≤ 0 критического потока | объявить feature-freeze (1.4) → переключить команду на надёжность → разбор главных потребителей бюджета → план восстановления | Sev-3 (процессный) |
| **Бэкап не восстановим** | ежедневный test-restore упал (3.4) | проверить: целостность бэкапа vs процесс restore → взять предыдущий зелёный бэкап → найти причину (поломка пайплайна/повреждение) → не закрывать, пока test-restore не зелёный | Sev-2 |
| **PostgreSQL failover** | primary down | подтвердить (не сеть) → авто-failover сработал? (реплика promoted, readyz зелёный) → если нет — ручной promote по DR-runbook (3.3) → проверить отсутствие split-brain → re-attach реплик | Sev-2/1 |
| **Перегрузка / load-shedding** | saturation USE / admission-control сработал | подтвердить источник (тенант/endpoint/бот) → усилить rate-limit виновника → scale-out (если ресурс) → проверить, что критический путь защищён | Sev-2 |
| **Утечка кардинальности/стоимость телеметрии** | рост числа серий/стоимости (2.8) | найти метрику-нарушителя (новый лейбл?) → drop/relabel правило → откат инструментации-виновника | Sev-3 |
| **DR — потеря региона** | региональный outage | активировать DR-runbook (3.3): promote/restore во втором регионе → переключить трафик/DNS → верифицировать SLI → коммуникация тенантам | **Sev-1** |

### 5.3 On-call и эскалация

- **Ротация on-call** — недельная, ≥2 человека (primary/secondary), компенсация/защита от выгорания (humane on-call: алертов должно быть мало — каждый actionable; см. «нет алерта без runbook»).
- **Уровни severity:** **Sev-1** (полный/частичный отказ критического потока, нарушение изоляции/ПДн, потеря региона) — page немедленно, war-room, обязательный post-mortem; **Sev-2** (деградация/быстрый burn) — page; **Sev-3** (медленный burn/процессное) — тикет в рабочее время.
- **Эскалация:** primary on-call → secondary (15 мин нет ответа/подтверждения) → инцидент-командир + домен-эксперт (Sev-1) → менеджмент/comms (внешняя коммуникация тенантам по SLA). Для нарушения ПДн — параллельно compliance/DPO (152-ФЗ сроки уведомления).
- **Blameless post-mortem** (Sev-1/Sev-2) — обязателен: таймлайн, root cause, что сработало/нет, **action items** (в т.ч. новый/исправленный runbook, новый алерт, chaos-эксперимент), отслеживание выполнения. Post-mortem — обучение, не наказание.
- **Status-коммуникация:** статус-страница/уведомление тенантам при Sev-1/Sev-2 (SLA-обязательство); запланированные окна — заранее (исключение из бюджета, 1.4).

## 6. Критерии приёмки (AC)

- **AC1 (SLI инструментированы).** Каждый SLO-поток (разд. 1.1) экспонирует RED-метрики с `tenant_id`/`route`/`query_class`; «хорошее событие» определено и считается; индекс-лаг и success-rate мутаций — отдельные SLI. Покрытие: 100% потоков из 1.1.
- **AC2 (SLO-цели как код).** Целевые пороги (1.2/1.3) и error-budget-окна (1.4) заданы декларативно (as-code); SLO-дашборд показывает текущий SLI vs target + остаток бюджета + burn-rate — для каждого потока (закрывает I6/§D AC1, #117 AC2).
- **AC3 (burn-rate-алерты + runbook).** Multi-window multi-burn-rate алерты (14.4×/6×/3×/1×, 1.4) активны; **каждый** алерт ссылается на runbook (5.1); алерт без runbook не проходит CI (DoD-гейт). Алерты actionable (нет флаппинга — оба окна).
- **AC4 (error-budget-политика).** При исчерпании бюджета критического потока деплой в него **блокируется** гейтом релиза; post-mortem обязателен; запланированное окно исключается из знаменателя.
- **AC5 (корреляция трёх сигналов).** Из метрики (exemplar) → трейс → лог того же `trace_id` переход работает для любого сервиса (#117 AC3 как обязательное); ПДн в метриках/лейблах/логах отсутствуют (проверка в CI на запрещённые лейблы).
- **AC6 (кардинальность под контролем).** Лейблы метрик — из разрешённого набора (2.8); запрещённые (reader_id/MFN/запрос) отклоняются; алерт на рост числа серий активен; ретеншен по 2.5 применён.
- **AC7 (health/readiness).** Все сервисы отдают `/healthz`+`/readyz`+startup-probe; `/readyz` различает critical/non-critical зависимости (деградация vs вывод из балансировки); blackbox-probe питает composite-availability-SLI.
- **AC8 (RPO/RTO достигаются на учении).** Фактические RPO/RTO на учениях (3.4) ≤ целевых (3.1): PITR ≤ 5 мин RPO / ≤ 1 ч RTO; PG failover ≤ 5 мин RTO; региональный ≤ 4 ч RTO. Расхождение → action item (закрывает I6/§D AC2).
- **AC9 (бэкап проверен restore’ом).** Ежедневный авто test-restore зелёный = условие «рабочего» бэкача; провал → Sev-2 «бэкап не восстановим»; 3-2-1 + immutable-копия + шифрование at-rest выполнены.
- **AC10 (per-tenant restore).** Восстановление одной схемы тенанта (3.5) проходит ежемесячное учение за ≤ 1 ч без влияния на других тенантов и без нарушения изоляции (тест изоляции I1 #100 зелёный после restore).
- **AC11 (graceful degradation).** Сценарий «книговыдача недоступна → поиск/каталог работают» проходит автотест; матрица деградации (4.1) покрыта тестами для критичных зависимостей; load-shedding защищает критический путь под нагрузкой (закрывает I6/§D AC3).
- **AC12 (circuit breakers + лимиты).** Каждый внешний/межсервисный вызов обёрнут CB + таймаут + bounded-retry-backoff-jitter; CB открывается по порогу и переходит Half-Open; per-tenant rate-limit (429+Retry-After) защищает от noisy-neighbor; 429 не списывается с error-budget.
- **AC13 (chaos / game day).** Полугодовой полный DR + chaos game-day проведён **по runbook’у** (без эскалации к автору-эксперту); зафиксированы фактические RPO/RTO и gaps; edge-хаос (разрыв/часы/дубли) проверен совместно с I4.
- **AC14 (on-call/эскалация).** Ротация on-call с уровнями Sev-1/2/3 и эскалацией (5.3) действует; Sev-1/Sev-2 → blameless post-mortem с отслеживаемыми action items; нарушение изоляции/ПДн → Sev-1 + параллельная компла-нотификация.
- **AC15 (изоляция-инцидент).** Runbook «нарушение изоляции тенанта» (5.2) существует, Sev-1, проверен на учении; аномалия кросс-тенантного доступа детектируется (алерт) и режется немедленно.

## 7. Критический путь

1. **SLI-определения + SLO-цели (1.1–1.3) как код** поверх инструментации #117 → есть «что считать хорошим» и числа. →
2. **SLO-дашборды (RED/USE) + error-budget-панель + корреляция трёх сигналов (2.3/2.4/2.1)** → видно состояние. →
3. **Burn-rate-алерты (1.4) + библиотека runbook’ов (5.1/5.2)** — каждый алерт с плейбуком; on-call-ротация (5.3) → реагируем осмысленно. →
4. **Health/readiness + circuit breakers + rate-limit + graceful-degradation-матрица (4.1–4.4)** → продукт деградирует, не падает; задаём health-контракт для D3. →
5. **Бэкап-политика (3.2) + ежедневный авто test-restore (3.4)** → бэкапы доказуемо восстановимы; задаём целевые RPO/RTO (3.1) для реализации в D3. →
6. **DR-runbook’и (3.3) + per-tenant restore (3.5) + регулярные учения по графику (3.4)** — PITR/failover/региональный → измеряем факт vs цель. →
7. **Chaos / game day (4.5/3.4) + error-budget-политика-гейт (1.4) + edge-DR/sync-каскад runbook’и** — зрелость: проверяем resilience и runbook’и «вживую», блокируем релизы при сожжённом бюджете.

**Выходной критерий D4:** у продукта есть **числовые SLO** (поиск p95≤300мс / доступность 99.9–99.95%, успех мутаций выдачи ≥99.9%) с **error-budget и burn-rate-политикой** (гейт релиза при исчерпании); **наблюдаемость доращена до SLO** (RED/USE-дашборды, burn-rate-алерты, корреляция метрика↔трейс↔лог, контроль кардинальности/стоимости) поверх #117; **DR с проверенными RPO/RTO** (RPO≤5мин/RTO≤1ч логическое, региональный RTO≤4ч), бэкап-политикой (3-2-1+immutable, гео, шифрование) и **регулярными restore-учениями по графику** (ежедн. авто-restore → полугодовой game-day); **resilience** (health/readiness, graceful degradation «поиск живёт без выдачи», circuit breakers, rate-limit); **библиотека runbook’ов** (всплеск латентности, edge офлайн >24ч, sync-каскад, нарушение изоляции тенанта, DR) с on-call/эскалацией/post-mortem — превращая тонкий I6/§D в реализуемую SRE-практику. Граница с D3: SRE задаёт *цели и контракты*, IaC ([SPEC_iac_fleet.md](SPEC_iac_fleet.md)) — *инфраструктурную реализацию*.

## 9. Риски

| Риск | Митигация |
|---|---|
| **SLO-цифры выбраны без прод-baseline** (p95≤300мс/99.9% — целевые, не измеренные на реальном фонде) | Числа — стартовые цели на пилот (СПб ГТБ); калибровка по факту первых 28 дней (baseline → корректировка target); error-budget сам подсветит нереалистичный target |
| **Алерт-усталость** (слишком много/шумных алертов → игнор) | Только actionable-алерты (нет алерта без runbook), multi-window анти-флаппинг, ревью алертов на каждом post-mortem (убрать шум), humane on-call |
| **Стоимость наблюдаемости растёт быстрее сервиса** (кардинальность/трейсы/логи) | Бюджет кардинальности (2.8), sampling трейсов, ретеншен-тиры, ежемесячный обзор top-cost; edge шлёт агрегаты |
| **Бэкапы пишутся, но не восстанавливаются** (классическая ловушка) | Ежедневный авто test-restore = условие «зелёного» бэкапа; график учений (3.4); «бэкап рабочий только после restore» |
| **DR-инфра ещё не готова** (репликация/мульти-регион — D3, пишется параллельно) | SRE задаёт целевые RPO/RTO и health-контракты как требования к D3; стартуем с ежедн. логического дампа + WAL-архива (single-region) → доращиваем до мульти-региона; on-prem/single-node — релакс SLO явно в SLA |
| **Runbook устаревает** (шаги не работают в реальный инцидент) | Game day **по runbook’у** проверяет актуальность; post-mortem обновляет; runbook рядом с кодом/алертами (версионируется); хранится вне восстанавливаемой системы |
| **Деградация не тестируется → в реальный отказ не срабатывает** | Автотесты матрицы деградации (4.1) + chaos game-day; CB/fallback в тест-сьюте, не только в коде |
| **Edge-DR сложен** (узел = БД-на-тенанта, оперативные данные локальны) | Каталог = облако-master (re-sync без потерь); оперативные edge-данные — локальный бэкап + sync-журнал (I4); RPO edge = «последний sync» явно в SLA |
| **Нарушение изоляции тенанта = и SRE-инцидент, и ПДн-инцидент** | Sev-1 runbook с параллельной компла-веткой (152-ФЗ сроки); детект-алерт; тест изоляции I1 #100 как непрерывный гейт |
| **Внешние зависимости (ЕСИА/СМЭВ/SMS) жгут чужой даунтайм в наш бюджет** | Явное исключение из SLI (1.5) + отдельный dependency-SLI + деградация (4.1), не нарушение |

## Сводка

**SRE-спек D4** превращает тонкий I6/§D в числа и процессы. **SLO-цели:** поиск p95 ≤ 300 мс (p99 ≤ 800 мс), просмотр записи p95 ≤ 150 мс, мутация выдачи p95 ≤ 500 мс; доступность платформы **99.9%**, поиска/OPAC **99.95%**, аутентификации 99.9%; успех мутаций книговыдачи ≥ 99.9%; свежесть индекса p95 ≤ 60 с; edge-узел 99.5% локально. **Error budget** с **multi-window multi-burn-rate** алертами (14.4×/6× → page, 3×/1× → ticket) и **budget-policy-гейтом** (исчерпан → feature-freeze критического потока). **Наблюдаемость** доращена поверх #117: RED/USE-дашборды, корреляция метрика↔трейс↔лог (`trace_id`+`tenant_id`), health/readiness/startup + blackbox-probe, sampling трейсов, **контроль кардинальности/стоимости** (запрет ПДн/неогр. ID в лейблах). **DR-модель:** RPO ≤ 1–5 мин / RTO ≤ 5 мин (PG failover) – 1 ч (PITR логическое) – 4 ч (регион); бэкап-политика **3-2-1 + immutable/WORM + гео + шифрование at-rest** (ежедн. база, непрерывный WAL, еженед. per-tenant дамп); **график restore-учений** — ежедневный авто test-restore (условие «зелёного» бэкапа), ежемес. per-tenant restore, ежекв. PITR/региональный failover, **полугодовой полный DR+chaos game-day по runbook’у**; per-tenant restore (≤1ч, без нарушения изоляции). **Resilience:** graceful degradation (поиск/каталог живут при недоступной выдаче — I6/§D AC3), circuit breakers (Closed/Open/Half-Open + таймауты + backoff-jitter + bulkhead), per-tenant rate-limit (анти-noisy-neighbor), load-shedding. **Runbook-модель:** библиотека инцидент-плейбуков (всплеск латентности поиска, edge офлайн >24ч, sync-каскад, **нарушение изоляции тенанта = Sev-1**, DR-потеря региона, бэкап-невосстановим, PG failover, исчерпан бюджет) по единому шаблону — **нет алерта без runbook**; on-call-ротация с Sev-1/2/3, эскалацией и **blameless post-mortem** (action items → новые runbook/алерты/chaos). Граница с **D3 IaC** ([SPEC_iac_fleet.md](SPEC_iac_fleet.md)): SRE задаёт *цели и контракты* (RPO/RTO, health, SLO), IaC реализует *инфраструктуру* (репликация, гео-хранилище, мульти-регион, авто-rollback).
