## 0. Принципы и границы

- **Metering → Rating → Billing → Enforcement — четыре развязанных слоя.** Источник метрики не знает про цену; rating (тарификация) не знает про платёжный провайдер; enforcement не знает про счёт. Контракт между слоями — типизированное **событие потребления** (`usage_event`, разд. 1) и **состояние подписки** (`subscription`, разд. 3). Это позволяет менять прайс-план без перезаписи метеринга и менять провайдера РФ без переписывания rating.
- **Энтайтлмент — не дублируется, потребляется.** `is_enabled(tenant, module)` остаётся в Identity/Licensing I1 #101 (источник истины «что подключено»). Биллинг — **причина** изменения энтайтлмента (оплатили Профи → подключились модули; churn → отключились), но **не хранит** флаг включения сам. D5 пишет в `tenant_module` через Licensing-API, не параллельно. Двойной гейт доступа (грант × энтайтлмент) остаётся инвариантом шлюза ([SPEC_api_contracts §0](../engines/SPEC_api_contracts.md)).
- **Прайс-план — данные, не код.** Тариф (Базовый/Стандарт/Профи/Сеть), его наборы модулей, оси и единицы — версионируемый ресурс `price_plan` (разд. 4). Добавить тариф / изменить лимит / завести региональную цену = новая версия плана, не релиз ядра. **Цифры — placeholder, согласует D7/GTM** (разд. 4.4); D5 фиксирует **механизм** (что измеряем, как тарифицируем, как энфорсим).
- **Деньги — детерминированы и аудируемы.** Rating одного периода из одного набора `usage_event` + одной версии `price_plan` даёт **один и тот же** счёт (воспроизводимость). Каждая денежная проводка (начисление, счёт, платёж, корректировка) — неизменяемая запись в `billing_ledger` с актором и причиной. Никаких «пересчётов на лету» без новой проводки.
- **Идемпотентность сквозная.** Один `usage_event` (дедуп по ключу) → ровно один учёт; один период → ровно один счёт (UNIQUE на `(tenant, period)`); один webhook платежа → ровно одна сверка. Шина даёт *at-least-once*, движок — *effectively-once* на метрику/счёт/платёж (паритет с A6, разд. 1.4).
- **Над-лимит — декларативная политика, не хардкод.** На каждую ось лимита (разд. 2) прайс-план задаёт одну из трёх стратегий: **hard-fail** (блок операции), **деградация** (мягкое ограничение/очередь), **доп-счёт** (overage-тариф). Enforcement-слой (разд. 5) применяет её, не зная бизнес-логики операции.
- **Анти-lock-in — first-class.** Тенант в любой момент выгружает свои данные (каталог/читатели/настройки) и **свою историю потребления/счетов** (разд. 7.3). Churn не уничтожает данные мгновенно — окно ретеншена + экспорт (разд. 3.5). Это явное обещание GTM §2 (импортозамещение без нового lock-in).
- **Границы.** Движок **только считает потребление, тарифицирует, выставляет счета, ведёт lifecycle подписки и энфорсит лимиты**. Он **не** хранит флаг энтайтлмента (это Licensing I1 #101), **не** проводит саму операцию (выдачу делает ws3, поиск — I2; биллинг лишь считает факт и при over-limit разрешает/блокирует), **не** рендерит уведомления о счёте/просрочке (это движок A6 — биллинг эмитит событие `invoice_issued`/`payment_overdue`, A6 доставляет), **не** реализует криптографию ключей провайдера (KMS/PKI — D2/I6). Цены не назначает — берёт из `price_plan` (наполняет D7/GTM).

## 1. Metering engine: события потребления

### 1.1 Модель события потребления

Метрика собирается как поток типизированных **событий потребления** — каждый сервис эмитит факт использования, не зная цены. Событие — неизменяемая запись.

```python
@dataclass(frozen=True)
class UsageEvent:
    tenant_id: str
    metric: str            # 'storage_gb' | 'fulltext_docs' | 'search_query' | 'loan_op' | 'edge_node' | 'ai_token' | ...
    quantity: Decimal      # дельта (для counter) или абсолют (для gauge/snapshot) — см. 1.2
    kind: str              # 'counter' | 'gauge' | 'snapshot'
    occurred_at: datetime  # время факта (UTC); период определяется по нему
    dedup_key: str         # детерминирован: f"{tenant}:{metric}:{source_id}:{occurred_at_bucket}"
    source: str            # сервис-источник (catalog/search/circulation/edge/ai) — для разбора
    labels: dict           # доп. размерности: {db, edge_node_id, model, ...} для группировки/over-limit
```

- **Контракт «источник → биллинг»** — стабильный, узкий: источник публикует `UsageEvent` в шину (outbox/CDC) или `POST /api/metering/ingest` (синхронный путь, разд. 6.1). Источник не знает про тариф/цену.
- `dedup_key` — UNIQUE на `usage_event` → повтор события (ретрай шины, двойной CDC) не учитывается дважды (разд. 1.4).

### 1.2 Каталог метрик (metric catalog)

Метрики делятся по **семантике замера** — это определяет стратегию сбора и агрегацию.

| Метрика (`metric`) | Семантика (`kind`) | Единица | Источник | Стратегия замера | Ось цены (GTM §4) |
|---|---|---|---|---|---|
| `storage_gb` | **gauge** (текущий объём) | ГБ·месяц (peak или avg) | объектное хранилище (MinIO/S3) + каталог-ядро I1 | периодический **snapshot** (раз в час/сутки) → агрегат за период | объём хранилища/полных текстов |
| `fulltext_docs` | **gauge** | число документов с полным текстом | каталог-ядро (поле/флаг полного текста) | snapshot | объём текстов |
| `search_query` | **counter** | запросов/период | поиск I2 (шлюз `/api/search`) | **сэмплирование/агрегация** (разд. 1.3) — высокочастотно | запросы (использование) |
| `loan_op` | **counter** | операций выдачи/возврата/продления | книговыдача ws3, ячейки/постаматы C1 | счётчик событий (батч-агрегация) | число операций |
| `edge_node` | **gauge** | активных edge-узлов | edge/sync I4 (heartbeat → реестр узлов) | snapshot активных за период | число площадок (edge-узлов) |
| `staff_seat` | **gauge** | активных рабочих мест сотрудников | Identity I1 (`account` kind=staff, активные) | snapshot | число рабочих мест |
| `reader_count` | **gauge** | читателей в фонде | книговыдача (RDR) | snapshot | размер (число читателей) |
| `holdings_count` | **gauge** | объём фонда (экземпляры) | каталог-ядро | snapshot | размер фонда |
| `ai_token` | **counter** | токенов LLM/эмбеддингов | AI/RAG C3 ([SPEC_I7 §D](../SPEC_I7_commercialization.md)) | счётчик по вызову (in/out токены в `labels.model`) | ИИ-сервисы (à la carte) |
| `locker_op` | **counter** | операций ячейки/постамата | ячейки C1 ([DESIGN_COVERAGE §C1](../../DESIGN_COVERAGE.md)) | счётчик | модуль ячеек/à la carte |
| `api_call` | **counter** | внешних API-вызовов (интеграции/SQL-реплика) | шлюз / SierraDNA-стиль ([SPEC_I7 §D](../SPEC_I7_commercialization.md)) | сэмплирование/агрегация | à la carte интеграции |

> **Определение метрики — данные, не код.** Каждая строка — запись `metric_def` (разд. 4.2): `code`, `kind`, `unit`, `aggregation`, `sampling`, дефолтная стратегия over-limit. Новая метрика = новая запись + источник начинает её эмитить; rating подхватывает по `price_plan`.

### 1.3 Стратегия сэмплирования и агрегации

Метрики различаются по частоте и стоимости точного учёта; стратегия выбирается per-metric.

- **Counter (search_query/loan_op/ai_token/api_call)** — высокочастотные. Источник **не** шлёт событие на каждый чих в биллинг (нагрузка + дубли). Варианты:
  - **Локальная пре-агрегация:** источник копит счётчик в окне (минута/час) и эмитит **один** `UsageEvent` с суммой за окно (`quantity` = дельта окна, `dedup_key` с бакетом окна). Дёшево, чуть «грубее».
  - **Сэмплирование (опц., для оценочных осей):** для не-биллируемых-но-наблюдаемых метрик допускается выборочный замер с экстраполяцией; **биллируемые counter — всегда полный учёт** (детерминизм денег, разд. 0), сэмплирование только для аналитики/бенчмарка.
- **Gauge/snapshot (storage_gb/edge_node/staff_seat/reader_count/holdings_count)** — состояние, не поток. **Периодический snapshot** (джоба раз в час/сутки, per-tenant) фиксирует текущее значение → агрегат за период (разд. 1.5: peak / average / end-of-period — задаётся в `metric_def`). Snapshot идемпотентен по `(tenant, metric, snapshot_bucket)`.
- **Realtime vs batch:**
  - **Realtime-путь** нужен **только для enforcement над-лимита** (разд. 5): «можно ли ещё одну операцию/ГБ?». Поддерживается **быстрым счётчиком текущего периода** (Redis/материализованный агрегат `usage_current`, разд. 1.5) — обновляется при ингесте, читается enforcement-слоем за O(1). Не для денег — для гейта.
  - **Batch-путь** — для **rating/счёта**: точная агрегация `usage_event` за закрытый период (ночная/по-закрытию-периода джоба, разд. 5.4 billing). Деньги считаются по batch (детерминизм), не по realtime-счётчику (он может быть приблизительным/сброшенным).
- Принцип: **realtime — для решения «пускать ли», batch — для решения «сколько денег»**; они согласуются, но не обязаны совпадать до копейки в моменте.

### 1.4 Идемпотентность и дедуп метеринга

- **`dedup_key`** UNIQUE на `usage_event`: повтор события (двойной CDC, ретрай шины) не учитывается дважды. Для counter-окон ключ несёт бакет окна (`{tenant}:{metric}:{source}:{window_bucket}`) → переотправка того же окна — no-op.
- **Snapshot** идемпотентен по `(tenant, metric, snapshot_bucket)`: повторный snapshot того же часа перезаписывает (gauge — последнее значение в бакете), не суммирует.
- **Эффект:** шина *at-least-once* → метеринг *effectively-once* на метрику (паритет A6 §4.3).

### 1.5 Агрегация за период

- **`usage_rollup`** — материализованный агрегат `(tenant, metric, period, value)` за биллинг-период (месяц/год, разд. 3.1). Считается из `usage_event` по правилу `metric_def.aggregation`:
  - `sum` — для counter (search_query/loan_op/ai_token): сумма дельт за период.
  - `peak` — для gauge с пиковой тарификацией (storage_gb по максимуму).
  - `average` — для gauge со средней (storage_gb·месяц = средний дневной snapshot).
  - `end_of_period` / `max_active` — для seat/edge_node (число на момент закрытия или пик активных).
- **`usage_current`** — быстрый незакрытый агрегат текущего периода для realtime-enforcement (разд. 1.3); сбрасывается/переоткрывается на границе периода; не источник истины для денег (им является `usage_rollup` по закрытым `usage_event`).
- **Изоляция:** `usage_event`/`usage_rollup`/`usage_current` — в схеме тенанта `t_<slug>` (DoD-гейт: тенант A не видит метрик B).

## 2. Тарифы, лимиты и над-лимит (over-limit)

### 2.1 Тариф как пресет модулей + оси

Тариф (GTM §5) — **пресет**: набор включённых модулей (энтайтлментов I1 #101) + набор лимитов по осям + базовая периодическая цена.

| Тариф | Включённые модули (энтайтлменты I1) | Кому (GTM §1) |
|---|---|---|
| **Базовый** | каталог + поиск + читательский веб (ЛК/заказ) | школьные, малые |
| **Стандарт** | + книговыдача + ячейки + уведомления (A6) | публичные |
| **Профи** | + комплектование + книгообеспеченность + аналитика/BI + discovery/OAI | вузы, крупные |
| **Сеть/Энтерпрайз** | + мультиплощадка + edge/офлайн (I4) + RFID/ТСД + SLA + кастом-модули | сети, ведомственные |

- **À la carte** ([GTM §5](../../GO_TO_MARKET.md)): модули докупаются поверх тарифа (RFID/ТСД, оплата, межбибл. обмен, ИИ-сервисы C3). Каждый à la carte — отдельный `tenant_module` + строка цены.
- **Пресет → энтайтлменты:** при активации тарифа D5 **просит Licensing I1** включить соответствующие `tenant_module` (разд. 0: D5 — причина, Licensing — источник истины). Связь с онбордингом #102: пресет модулей по типу библиотеки ([SPEC_I1 #102](../SPEC_I1_foundation.md)) согласуется с тарифом.

### 2.2 Лимиты по осям

На каждый тариф/ось — **лимит** (`quota`) в `price_plan` (разд. 4): сколько включено в базу, что сверх — по over-limit-политике.

| Ось (GTM §4) | Метрика | Лимит-пример (placeholder, число — D7) | Единица |
|---|---|---|---|
| Объём хранилища | `storage_gb` | N ГБ включено | ГБ |
| Полные тексты | `fulltext_docs` | N документов | шт |
| Размер фонда | `holdings_count` | N экземпляров | шт |
| Число читателей | `reader_count` | N читателей | шт |
| Edge-узлы (площадки) | `edge_node` | N узлов | шт |
| Рабочие места | `staff_seat` | N мест | шт |
| Поисковые запросы | `search_query` | N/месяц | шт |
| Операции выдачи | `loan_op` | N/месяц | шт |
| ИИ-токены | `ai_token` | N токенов/месяц (или 0 — à la carte) | токен |

> Числа — `null`/placeholder в D5; GTM/D7 подставляет. Лимит может быть `unlimited` (Сеть/Энтерпрайз по осям SLA).

### 2.3 Над-лимит: три стратегии (декларативно)

На каждую ось `price_plan` задаёт `over_limit_policy` ∈ {`hard_fail`, `degrade`, `surcharge`}:

| Стратегия | Поведение | Типичная ось | Enforcement (разд. 5) |
|---|---|---|---|
| **`hard_fail`** | операция блокируется при достижении лимита; `402 quota_exceeded` / `403` на шлюзе | `staff_seat` (нельзя завести 11-е место на тарифе с 10), `edge_node` | realtime-гейт читает `usage_current` ≥ `quota` → блок |
| **`degrade`** | мягкое ограничение: throttle/очередь/урезанная функция, без блока | `search_query` (rate-limit вместо отказа), `ai_token` (очередь/замедление) | realtime-гейт включает throttle при превышении |
| **`surcharge`** (доп-счёт) | over-limit разрешён, излишек тарифицируется по overage-цене → в следующий счёт | `storage_gb` (платишь за лишние ГБ), `loan_op`, `ai_token` | rating добавляет overage-строку (разд. 5.4) |

- **Комбинируемо:** ось может иметь мягкий порог (`degrade`/warn) + жёсткий потолок (`hard_fail`) — напр. ИИ-токены: до X — включено, X..Y — `surcharge`, >Y — `hard_fail`.
- **Grace:** на границе периода/при оплате — grace-окно (не блокировать мгновенно, дать оплатить/апгрейднуть); связь с suspend-lifecycle (разд. 3.3) и edge grace-лицензией ([DESIGN_COVERAGE §C2](../../DESIGN_COVERAGE.md)).
- **Уведомление:** приближение к лимиту (80%/100%) → событие `quota_warning`/`quota_exceeded` → A6 доставляет (биллинг не шлёт сам, разд. 0).

## 3. Subscription lifecycle

### 3.1 Биллинг-цикл

- **Период** — месяц или год (per-tenant выбор; год обычно со скидкой — GTM §4 «подписка (год)»). `subscription.period` ∈ {`monthly`, `annual`}. Anchor-дата (день старта) фиксирует границы периодов.
- **Модель цены — гибрид** ([SPEC_I7 §C](../SPEC_I7_commercialization.md): «подписка + использование»): **периодическая подписка** (тариф, авансом за период) **+ usage/overage** (по факту за прошедший период, постфактум). Счёт за период N = подписка(N+1 авансом, опц.) + overage(N постфактум).
- **Проводки** — в `billing_ledger` (неизменяемые): `subscription_charge`, `usage_charge`, `overage_charge`, `credit`/`adjustment`, `tax(VAT)`, `payment`, `refund`.

### 3.2 Состояния подписки

```
                  ┌──────────────────────────────────────────────┐
                  ▼                                              │
  [trial] ──оплата──▶ [active/paid] ──неоплата+grace──▶ [past_due] ──▶ [suspended] ──▶ [cancelled/churned]
     │                    ▲   │                              │  оплата       │  оплата          │
     │ истёк trial,       │   │ апгрейд/даунгрейд            └───────────────┘ (восстановление)  │ ретеншен-окно
     │ нет оплаты         │   │ (proration, разд. 3.4)                                          ▼
     └────────────────────┘   └──── меняет price_plan/модули ────                        [data_retention] ──экспорт──▶ purge
```

| Состояние | Смысл | Доступ (enforcement) | Переход |
|---|---|---|---|
| **`trial`** | пробный период (длительность T дней per-plan; для **школ — free-tier**, разд. 3.6) | полный доступ к модулям тарифа, лимиты trial | оплата → `active`; истёк без оплаты → `suspended` (или `free` для школ) |
| **`active`** (paid) | оплачено, в рамках периода | полный по тарифу + over-limit-политики | неоплата след. периода + grace → `past_due` |
| **`past_due`** | счёт не оплачен, grace-окно идёт | полный (grace) + усиленные напоминания (A6) | оплата → `active`; истёк grace → `suspended` |
| **`suspended`** | доступ ограничен (неоплата) | **read-only / деградация** (читатель видит каталог, staff не пишет; данные целы) — не мгновенный кат-офф | **оплата → восстановление в `active`** (разд. 3.3); долгий simpson → `cancelled` |
| **`cancelled`/`churned`** | подписка прекращена (по воле тенанта или после suspend) | доступ закрыт, данные в ретеншене | ретеншен-окно → экспорт → purge (разд. 3.5) |

### 3.3 Suspend и восстановление после оплаты

- **Suspend ≠ удаление.** При неоплате после grace тенант переходит в `suspended`: enforcement переводит модули в **read-only/деградацию** (каталог/ЛК читателя доступны на чтение, staff-запись и платные модули заблокированы), **данные сохраняются полностью**. Это гуманнее хард-катоффа и снижает отток.
- **Восстановление:** оплата в `suspended` → подписка возвращается в `active`, энтайтлменты восстанавливаются (D5 просит Licensing включить модули обратно), доступ открывается без потери данных/настроек. Идемпотентно: повторный платёж-webhook не двоит восстановление.
- **Edge-связь:** edge-узел в `suspended` получает grace-лицензию ([DESIGN_COVERAGE §C2](../../DESIGN_COVERAGE.md)) — продолжает работать офлайн ограниченное окно, не «окирпичивается» при проблеме оплаты/связи.

### 3.4 Апгрейд/даунгрейд и proration

- Смена тарифа/à la carte среди периода → **пропорциональный пересчёт** (proration): неиспользованный остаток текущего тарифа кредитуется, новый тариф начисляется pro-rata до конца периода. Проводки `adjustment`/`credit` в ledger.
- **Апгрейд** — немедленно (доступ к новым модулям сразу, D5 просит Licensing включить). **Даунгрейд** — обычно с конца периода (или немедленно с кредитом, по политике); при даунгрейде проверка: не превышены ли лимиты нового тарифа (иначе предупреждение/блок до приведения в соответствие).

### 3.5 Churn: ретеншен данных и экспорт (анти-lock-in)

- При `cancelled`/`churned` — **окно ретеншена** (per-plan/политика, напр. 30–90 дней): данные не удаляются сразу.
- В окне тенант **экспортирует всё** (разд. 7.3): каталог, читатели, настройки, **история потребления и счетов**. Форматы — машиночитаемые (стандартные обменные: каталог в RUSMARC/MARC, читатели/настройки в JSON/CSV, счета в PDF+структурированный реестр). Это явное обещание GTM §2 (импортозамещение без нового lock-in).
- По истечении окна без реактивации — **purge** (удаление схемы тенанта, как откат онбординга #102; ПДн — право на забвение I6 B4). Финансовые документы (счета/проводки) хранятся по сроку НК РФ отдельно от ПДн (разд. 6.5), затем архивируются/удаляются по налоговому ретеншену.

### 3.6 Free-tier для школ

- **Школьный тариф** ([GTM §1](../../GO_TO_MARKET.md): «дёшево, облако, минимум поддержки»; §10 открытый вопрос «бесплатный тариф для школ») — отдельное состояние/план `free`: подмножество Базового (каталог + поиск + ЛК), жёсткие лимиты по осям, без overage (превышение → `hard_fail`/`degrade`, не `surcharge` — школа не получает счёт за overage).
- Не trial (бессрочен), но с лимитами привлечения; апгрейд до платного — апсейл-путь. Политика (что входит, лимиты, апсейл) — **D7/GTM** (placeholder в D5).

## 5. Enforcement-слой

### 5.1 Двойной гейт + лимиты

Доступ к операции = **(грант функция×база×уровень)** И **`is_enabled(tenant, module)`** (I1 #101 / [SPEC_api_contracts §0](../engines/SPEC_api_contracts.md)) И **в рамках лимита оси** (D5). Первые два — инвариант шлюза (не меняем). Третий — добавляет биллинг:

```
запрос → шлюз
  ├─ грант? нет → 403 forbidden
  ├─ is_enabled(tenant, module)? нет → 404/403 module_disabled   (Licensing I1, биллинг — причина состояния)
  └─ within quota(metric)? (realtime usage_current vs price_plan.quota)
        ├─ под лимитом → пропустить, эмитнуть usage_event
        ├─ over + degrade   → throttle/очередь, пропустить
        ├─ over + surcharge → пропустить, пометить overage (в счёт)
        └─ over + hard_fail → 402 quota_exceeded
```

### 5.2 Где энфорсится

- **Realtime-гейт** (для `hard_fail`/`degrade`) — в шлюзе/middleware, читает `usage_current` (разд. 1.5) за O(1). Не блокирует на медленном rating.
- **Состояние подписки** (`suspended` → read-only) — шлюз проверяет `subscription.state` (кэш с TTL) и режет write/платные-модули (разд. 3.3).
- **Surcharge** — не блокирует в моменте; помечается на ингесте/rating (разд. 5.4) и попадает в следующий счёт.
- **Энтайтлмент** — биллинг **пишет** в `tenant_module` через Licensing-API при смене состояния (оплата/churn/апгрейд), но гейт `is_enabled` остаётся в Licensing (разд. 0).

### 5.3 Деградация и grace

- `degrade` — мягко: rate-limit поиска, очередь/замедление ИИ, без отказа пользователю (он видит «лимит, замедлено»). Лучше отказа для UX публичной библиотеки.
- **Grace-окна** — на границе периода, при первой неоплате (`past_due`), при апгрейде-в-процессе: не катать мгновенно. Edge — grace-лицензия офлайн (разд. 3.3).

### 5.4 Rating (тарификация периода) → счёт

Batch-джоба на закрытии периода (детерминизм, разд. 1.3):

```
закрытие периода P для тенанта T
  → собрать usage_rollup(T, *, P)   из usage_event (закрытые, дедуплицированные)
  → загрузить price_plan версии, действовавшей в P (исторически, не текущей)
  → для каждой оси:
       included = quota.included
       overage  = max(0, rollup.value − included)   (если over_limit=surcharge)
       charge   = overage × overage_unit_price
  → subscription_charge (тариф за след. период, если аванс)
  → сумма line_items → subtotal
  → VAT (разд. 6.4) → total
  → создать invoice (UNIQUE на (tenant, period)) + проводки в billing_ledger
  → эмитнуть событие invoice_issued → A6 уведомляет
```

- **Детерминизм:** один и тот же `(usage_rollup, price_plan_version)` → один и тот же счёт. Пересчёт — только новой версией/корректировкой (`adjustment` в ledger), не перезаписью.
- **Идемпотентность:** UNIQUE `(tenant, period)` на `invoice` → повторный прогон rating того же периода не плодит второй счёт.

## 6. Billing: счета, НДС, платёжный провайдер РФ, сверка

### 6.1 Генерация счёта

- **Счёт** (`invoice`) — `tenant`, `period`, `line_items[]` (подписка/usage/overage/à la carte), `subtotal`, `vat`, `total`, `currency`, `status` (draft/issued/paid/overdue/void), реквизиты плательщика/исполнителя.
- **Форматы:** **PDF** (человекочитаемый, с реквизитами + НДС — рендерится через A1/печать или отдельный шаблон; счёт на оплату + при оплате счёт-фактура/УПД для юрлиц) **+ API/JSON** (машиночитаемый, для бухгалтерии тенанта и закупок). Реестр счетов выгружается (отчётность для закупок, [SPEC_I7 §C](../SPEC_I7_commercialization.md)).
- Нумерация счетов — последовательная per-tenant/per-исполнитель (требование НК РФ), без пропусков, аудируемая.

### 6.2 НДС

- **НДС** ([SPEC_I7 §C](../SPEC_I7_commercialization.md) подразумевает РФ-биллинг) — `vat_mode` на план/строку: `standard` (ставка из справочника `vat_rate`, версионируемого — ставка может меняться законодательно) или `exempt` (если применима льгота, напр. отдельные виды лицензий ПО — **сверить с D7/юристом**, разд. 11).
- Счёт показывает: subtotal без НДС, сумму НДС (ставка), total с НДС. Для юрлиц — **счёт-фактура/УПД** с реквизитами (ИНН/КПП обеих сторон). Ставка НДС — **не хардкод**, ссылка на `vat_rate_ref` (разд. 4.1), значение/льготы — D7.
- Округление и порядок начисления НДС — по правилам НК РФ (детерминированно, в rating разд. 5.4).

### 6.3 Платёжный провайдер РФ — абстракция

Провайдеры РФ не унифицированы (ЮKassa, банк-эквайринг/банк-API, СБП). Поэтому — **адаптер за единым интерфейсом** (паритет с SMS-абстракцией A6 §1.2):

```python
class PaymentProvider(Protocol):
    def create_payment(invoice, return_url) -> PaymentIntent      # {provider_payment_id, pay_url, status}
    def parse_webhook(body, signature) -> PaymentReceipt | None    # succeeded | pending | failed | refunded
    def refund(payment_id, amount) -> RefundAck
    def status(payment_id) -> PaymentStatus
```

- **Реализации:** `YooKassaProvider`, `BankApiProvider` (эквайринг/расчётный счёт юрлица — для гос/бюджетных частый путь: счёт → оплата по реквизитам → подтверждение), опц. СБП. Один активен на тенант + способ оплаты по типу клиента (юрлицо-бюджет часто платит по счёту через банк, не картой).
- Конфиг провайдера — per-tenant `billing_account` ресурс: ключи **в KMS/секретах** (D2 PKI I6, не в БД открыто), реквизиты, способ оплаты.
- Тест: контракт-тест провайдера на mock-API; интеграционный — sandbox-режим провайдера.

### 6.4 Сверка (reconciliation)

- **Webhook платежа** (`POST /api/billing/webhooks/{provider}`) — провайдер шлёт статус (succeeded/failed/refunded) → валидация подписи → `parse_webhook` → проводка `payment`/`refund` в ledger → счёт `issued→paid` → событие `payment_succeeded` (восстановление из suspend, разд. 3.3) или `payment_failed` (→ напоминание A6). **Идемпотентно:** повтор webhook не двоит платёж (UNIQUE на `provider_payment_id`).
- **Reconciliation-джоба:** периодически сверяет внутренний ledger с выписками/реестром провайдера (банк-API/ЮKassa report) — ловит расхождения (платёж был, webhook потерян; или наоборот). Несовпадение → `staff_alert` финансовому оператору.
- **Для бюджетной оплаты по счёту:** оплата приходит на расчётный счёт → сверка по банк-выписке (банк-API) с привязкой к номеру счёта/назначению платежа → ручное/полу­авто подтверждение в админке.

### 6.5 Налоговый/финансовый ретеншен

- Счета/счета-фактуры/проводки — хранятся по сроку **НК РФ** (отдельно от операционных ПДн читателей). При churn-purge (разд. 3.5) финансовые документы переживают purge данных тенанта и удаляются по своему сроку.
- Доступ к финансовым документам — по гранту (admin/finance), под журналом доступа (152-ФЗ/аудит, как ws3/reader).

## 7. Self-serve админка, отчётность для закупок, экспорт

### 7.1 Self-serve (ws5)

- Тенант-админ ([SPEC_ws5_admin](../SPEC_ws5_admin.md)) видит: текущий тариф/подписку/состояние, **дашборд потребления** (по осям, к лимиту), счета (PDF+реестр), способ оплаты, à la carte-модули.
- Действия self-serve: апгрейд/даунгрейд тарифа (proration авто), вкл/выкл à la carte, смена периода (месяц↔год), оплата счёта, скачивание счетов/реестра, экспорт данных (разд. 7.3). «Самообслуживание тарифов в облаке» — прямое требование [GO_TO_MARKET §9](../../GO_TO_MARKET.md).

### 7.2 Отчётность для закупок

- **Реестр счетов/актов** + структурированная выгрузка (CSV/JSON/PDF) — для бухгалтерии и **44/223-ФЗ закупок** ([GTM §8](../../GO_TO_MARKET.md), [DESIGN_COVERAGE §D7](../../DESIGN_COVERAGE.md)): что оплачено, за какой период, реквизиты, акты выполненных работ.
- Связь D7: вхождение в реестр отеч. ПО + чек-лист 44/223 — отчётность D5 должна давать документы под закупочные процедуры.

### 7.3 Экспорт (анти-lock-in)

- `GET /api/billing/export` (admin) — выгрузка **истории потребления (`usage_rollup`) + счетов + проводок** в машиночитаемом виде. Вместе с экспортом данных тенанта (каталог/читатели/настройки — отдельные сервисы) даёт полный «забрать всё и уйти» (GTM §2).
- Доступен в любом состоянии (включая churn-окно, разд. 3.5) — обещание отсутствия lock-in.

## 9. Критерии приёмки (AC)

- **AC1 (metering).** Каждый биллируемый источник (хранилище/тексты/поиск/выдача/edge/seat/AI-токены) эмитит `UsageEvent`; повторная публикация (двойной CDC) **не учитывается дважды** (дедуп по `dedup_key`); gauge-метрики снимаются snapshot-джобой идемпотентно по бакету; counter — пре-агрегированными окнами.
- **AC2 (агрегация/период).** `usage_rollup` за период детерминированно агрегирует `usage_event` по `metric_def.aggregation` (sum/peak/average/end_of_period); `usage_current` даёт realtime-значение для enforcement за O(1).
- **AC3 (тариф→энтайтлмент).** Активация тарифа подключает соответствующие `tenant_module` **через Licensing I1 #101** (не параллельно); выключенный модуль → 404/403 (инвариант I1 AC2); à la carte добавляет отдельный модуль.
- **AC4 (over-limit).** На ось применяется политика плана: `hard_fail` → `402 quota_exceeded` на лимите; `degrade` → throttle/очередь без отказа; `surcharge` → over-limit пропущен и тарифицирован в следующий счёт. Приближение к лимиту → событие `quota_warning` (A6 доставляет).
- **AC5 (lifecycle).** trial→active по оплате; active→past_due→suspended по неоплате+grace; **suspended→active по оплате с полным восстановлением данных/энтайтлментов** (без потери данных); cancelled→ретеншен-окно→экспорт→purge. Free-tier школ — отдельное состояние без overage.
- **AC6 (proration).** Апгрейд среди периода — немедленный доступ + pro-rata начисление; даунгрейд — с конца периода или с кредитом; проводки `adjustment`/`credit` в ledger; даунгрейд при превышении лимитов нового тарифа предупреждает/блокирует.
- **AC7 (rating/счёт).** Закрытие периода → детерминированный счёт из `usage_rollup` + **исторической** версии `price_plan`; UNIQUE `(tenant, period)` → повторный rating не плодит второй счёт; line_items = подписка + usage + overage + à la carte.
- **AC8 (НДС/счёт).** Счёт показывает subtotal/НДС(ставка из `vat_rate`)/total; PDF (счёт + счёт-фактура/УПД для юрлиц с реквизитами) + JSON; нумерация последовательна без пропусков; ставка НДС — из справочника, не хардкод.
- **AC9 (провайдер РФ/сверка).** Оплата через абстракцию `PaymentProvider` (ЮKassa/банк-API); webhook платежа валидируется по подписи, проводится в ledger, переводит счёт в `paid`, восстанавливает из suspend; **идемпотентен** (UNIQUE `provider_payment_id`); reconciliation-джоба ловит расхождения → `staff_alert`.
- **AC10 (self-serve/экспорт).** Тенант-админ видит потребление/счета/подписку и меняет тариф/à la carte/период сам; реестр счетов выгружается для закупок (44/223); `GET /api/billing/export` отдаёт потребление+счета+проводки (анти-lock-in) в любом состоянии включая churn-окно.
- **AC11 (детерминизм/аудит денег).** Один `(usage_rollup, price_plan_version)` → один счёт (воспроизводимость); каждая денежная проводка — неизменяемая запись в `billing_ledger` с актором/причиной; пересчёт только через `adjustment`, не перезаписью.
- **AC12 (изоляция тенанта).** Метрики/подписки/счета/проводки/реквизиты тенанта A недоступны B (тест в CI — DoD-гейт); секреты провайдера per-tenant в KMS.
- **AC13 (наблюдаемость).** Метрики `metering_events_ingested`/`rated_amount`/`invoice_generated`/`payment_reconciled`/`enforcement_block` экспонируются; trace несёт `tenant_id`; `staff_alert` при расхождении сверки/провале платежа/застрявшем rating.

## 10. Критический путь

1. **Контракт `UsageEvent` + `dedup_key` + Ingest** (приём из шины и `POST /api/metering/ingest`) + **idempotency-чек** + `metric_def`-каталог → метрики собираются, ничего не двоится. →
2. **Агрегация** (`usage_rollup` batch + `usage_current` realtime) по 2–3 базовым осям (storage_gb snapshot, search_query/loan_op counter) → потребление видно. →
3. **`price_plan`/`subscription` модель + состояния** (trial/active) + привязка пресета к энтайтлментам **через Licensing I1** → тариф управляет доступом. →
4. **Enforcement-слой** (двойной гейт + лимиты): realtime-гейт `hard_fail`/`degrade` в шлюзе + `suspended`→read-only → лимиты соблюдаются. →
5. **Rating + invoice + НДС + ledger** (детерминированный счёт за период, UNIQUE `(tenant, period)`, PDF+JSON) → счета формируются. →
6. **Платёжный провайдер РФ** (абстракция, ЮKassa/банк-API) + **webhook-сверка** + restore-from-suspend → деньги принимаются и сверяются. →
7. **Lifecycle полностью** (past_due/suspend/churn + proration + ретеншен/экспорт + free-tier школ) + **self-serve-админка** (ws5) + **отчётность для закупок** + over-limit `surcharge` (overage в счёт).

**Выходной критерий движка D5:** библиотека-тенант подключается по тарифу (Базовый/Стандарт/Профи/Сеть, пресет → энтайтлменты I1 #101), её потребление по осям (хранилище/тексты/запросы/операции/edge/seat/AI-токены) метрится per-tenant идемпотентно, лимиты энфорсятся (hard-fail/деградация/доп-счёт), за период формируется детерминированный счёт с НДС (PDF+API), оплата проходит через абстракцию провайдера РФ (ЮKassa/банк-API) и сверяется по webhook/выписке, подписка проходит lifecycle (trial→paid→suspend→churn с восстановлением после оплаты и экспортом при churn), тенант-админ управляет всем self-serve и выгружает отчётность для закупок — заменяя ручное лицензирование desktop-ИРБИС платформенным метерингом+биллингом. **Прайс-точки наполняет D7/GTM**; D5 даёт оси и механизм.

| **НДС/льготы по ПО — юридически тонко** (ставка/exempt для лицензий ПО, изменения НК) | `vat_rate` — версионируемый справочник, не хардкод; `vat_mode` per-plan; ставку/льготы наполняет/сверяет D7+юрист (разд. 11 TODO); rating пересчитываем новой версией |
| **Провайдеры РФ не унифицированы** (ЮKassa vs банк-эквайринг vs СБП; бюджет платит по счёту, не картой) | Адаптер `PaymentProvider` (разд. 6.3), контракт-тест на mock + sandbox; банк-API-путь (оплата по счёту → сверка по выписке) для бюджетных; способ оплаты по типу клиента |
| **Расхождение учёта и реальной оплаты** (потерянный webhook, оплата без webhook) | Reconciliation-джоба сверяет ledger с реестром/выпиской провайдера → `staff_alert`; ручное подтверждение для бюджетной оплаты по счёту |
| **Хард-катофф при неоплате отпугнёт/сорвёт работу библиотеки** | `suspended` = read-only/деградация, не удаление; grace-окна; edge grace-лицензия офлайн; восстановление при оплате без потери данных |
| **152-ФЗ + НК-ретеншен конфликтуют** (право на забвение vs хранение финдоков) | Финансовые документы (счета/проводки) отделены от ПДн читателей; purge данных по 152-ФЗ переживают финдоки до налогового срока (разд. 6.5); координация I6 B4 |
| **Метрики осей не все доступны** (AI-токены/locker-op зависят от ещё-проектируемых C1/C3) | `metric_def.billable=false` для недоступных осей (наблюдаем, не тарифицируем) до готовности источника; включение оси = источник начинает эмитить + план задаёт quota |

## Сводка

**Metering + billing-engine model.** Движок D5 — **четырёхслойный конвейер коммерции per-tenant**: **(1) Metering** — источники (хранилище/тексты/поиск/выдача/edge/seat/AI-токены) эмитят типизированные `UsageEvent` (counter/gauge/snapshot) через шину I7 #107 или `POST /api/metering/ingest`, дедуп по `dedup_key` (*at-least-once → effectively-once*); пре-агрегация окнами для counter, snapshot-джоба для gauge; `usage_current` (realtime, O(1)) для enforcement и `usage_rollup` (batch, детерминизм) для денег. **(2) Тарифы/лимиты** — прайс-план (Базовый/Стандарт/Профи/Сеть + à la carte) как версионируемые данные: пресет → энтайтлменты **через Licensing I1 #101** (не дублируется), лимиты по осям GTM §4, над-лимит-политика декларативно (`hard_fail`/`degrade`/`surcharge`). **(3) Lifecycle** — trial→active→past_due→suspended→cancelled с grace-окнами, **восстановлением после оплаты без потери данных**, proration при апгрейде, free-tier школ, ретеншен+экспорт при churn (анти-lock-in). **(4) Billing** — rating закрытого периода (детерминированный счёт из rollup + исторической версии плана, UNIQUE `(tenant,period)`) → НДС (справочник ставок, счёт-фактура/УПД) → счёт PDF+API → оплата через абстракцию `PaymentProvider` (ЮKassa/банк-API, бюджет по счёту) → webhook-сверка (идемпотентно) + reconciliation-джоба. **Enforcement** — двойной гейт грант × энтайтлмент (инвариант I1) + лимиты осей в шлюзе.
