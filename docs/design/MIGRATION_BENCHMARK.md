# Бенчмарк миграции: ИРБИС64 → собственный стор

Замер скорости переноса библиографических данных из живой БД ИРБИС64 (IBIS)
в наш собственный стор (`server-own`) через адаптер ИРБИС. Данные-first.

- **Дата:** 2026-06-21
- **Харнесс:** `irbis-web/server-own/bench_migrate.py` (переиспользует `migrate.py` transform/placement; читает только, RDR не трогает)
- **Метод:** мигрировался **весь** датасет IBIS (он мал — выборка не нужна), 2–3 прогона на каждый бэкенд, числа стабильны между прогонами.

## Окружение

| Компонент | Значение |
|---|---|
| Хост ИРБИС64 | `127.0.0.1:6666`, server version `2022+`, рабочая станция `A` |
| Протокол адаптера | **connection-per-request** (новый TCP-сокет на каждую команду), сессия по `client_id` |
| Бэкенд sqlite | `OwnStore` (sqlite + FTS5, `tokenize=unicode61`), Python `py` 3.14.2 |
| Бэкенд PostgreSQL | `PgStore` (psycopg 3.3.4, `py -3.12`), контейнер `irbis-pg` = `postgres:16-alpine` (PG 16.14) на `:5433`, DSN `postgresql://postgres:pg@127.0.0.1:5433/irbisweb` |
| ОС | Windows 11, всё локально (loopback) |

PostgreSQL поднимать через `docker-compose.pg.yml` не пришлось — на `:5433` уже
работал контейнер `irbis-pg` с тем же дефолтным DSN (порт был занят; том compose
снят, использован существующий контейнер).

## Размер датасета

- `max_mfn` (команда `O`) по IBIS = **394**
- Поиск `"I=$"` (все непустые записи) → **390** записей (часть MFN удалены/пусты).
- Перенесено по факту: **390 / 390**, ошибок 0.

## Конвейер на запись (что делает миграция на запись)

На каждую запись `migrate.py` выполняет **два round-trip'а к ИРБИС**:
1. `read_record` — команда `C` (сырые поля/подполя),
2. `format_record` — команда `G` (`@brief`, серверный PFT-рендеринг).

Плюс один разовый `search` (`K`). На стороне стора: `transform` →
`upsert` (вставка в `rec` + DELETE/INSERT в FTS5) → `place_holding`, с
**commit на каждую запись**.

## Результаты — sqlite (основной, надёжный путь)

Полная БД (390 записей), типичный прогон с серверным форматированием (`G`):

| Метрика | Значение |
|---|---|
| Перенесено записей | 390 (errors 0) |
| Общее время цикла (wall) | **51.3 с** |
| **Пропускная способность** | **7.6 зап/с** |
| Чтение из ИРБИС (всего) | 40.2 с — **78 %** времени цикла |
| — `search` (K) | 0.07 с (разово) |
| — `read_record` (C) | 19.8 с (≈ 50.8 мс/зап) |
| — `format_record` (G) | 20.3 с (≈ 52.0 мс/зап) |
| Запись в стор (transform+upsert+FTS+commit) | 11.2 с — **22 %** (≈ 28.8 мс/зап) |
| Разовая инициализация стора (`seed_storage`, дерево локаций) | ≈ 2.7–3.2 с (фикс., не на запись) |

Стабильность: прогон 2 → 7.4 зап/с (52.5 с), та же раскладка 79 % / 21 %.
Прогон без `format_record` (`--no-format`) общее время почти не уменьшил
(~53 с): сервер сериализует connection-per-request, и латентность одиночного
round-trip'а «плавает» — снятая нагрузка `G` перетекает в `C`. Чистый сигнал:
**два round-trip'а ≈ 100 мс чтения на запись против ≈ 28 мс записи**.

## Результаты — PostgreSQL (быстро поднялся)

Тот же датасет, `OWN_STORE_BACKEND=pg`, `py -3.12` (схема пересоздавалась перед прогоном):

| Метрика | Прогон 1 | Прогон 2 |
|---|---|---|
| Перенесено | 390 | 390 |
| Время цикла (wall) | 63.4 с | 64.0 с |
| **Пропускная способность** | **6.15 зап/с** | 6.09 зап/с |
| Чтение из ИРБИС (всего) | 44.0 с (69 %) | 45.6 с (71 %) |
| Запись в стор | 19.9 с (≈ 51 мс/зап) | 18.5 с (≈ 47 мс/зап) |

Чтение из ИРБИС идентично sqlite-пути (тот же адаптер). PG-запись медленнее
sqlite (≈ 50 мс против ≈ 29 мс/зап), потому что `PgStore` работает в
`autocommit=True` — **commit + сетевой round-trip к контейнеру PG на каждую
запись**, плюс расчёт `to_tsvector` на стороне сервера.

## Узкое место

**Чтение из ИРБИС по схеме connection-per-request** — 69–79 % общего времени.
Два TCP-round-trip'а на запись (`C` + `G`), каждый ≈ 50 мс на loopback,
суммарно ≈ 100 мс на запись. Сама запись в стор вторична и масштабируется
тривиально (см. ниже). Подтверждающие микро-замеры:

| Изолированный замер | Результат |
|---|---|
| sqlite write, commit-на-запись | 101 зап/с |
| sqlite write, batched commit (раз в 500) | **499 зап/с (×4.9)** |
| PG write, autocommit per-row | 333 зап/с |
| PG write, одна транзакция `executemany` | **5 636 зап/с (×16.9)** |
| PG write, `COPY` + bulk `to_tsvector` | **32 459 зап/с (×97.4)** |
| ИРБИС `read_record` серийно, 1 соединение | 17.6 зап/с |
| ИРБИС `read_record` параллельно, 4 воркера | 42.1 зап/с (×2.4) |
| ИРБИС `read_record` параллельно, 8 воркеров | **51.2 зап/с (×2.9)** |

Вывод: оба стора способны на сотни–десятки тысяч записей в секунду на запись,
если убрать commit-на-запись. Реальный потолок ставит сеть к ИРБИС. Сервер
ИРБИС принимает параллельные соединения и масштабируется сублинейно
(контеншн/одноядерное ядро сервера ограничивает прирост за ~8 воркерами).

## Проекции времени полной миграции

| Сценарий | IBIS (390) | 100k записей | 1M записей |
|---|---|---|---|
| Текущий sqlite (7.5 зап/с, read-bound) | 52 с | 3.7 ч | 37.0 ч |
| Текущий PG autocommit (6.1 зап/с) | 64 с | 4.6 ч | 45.5 ч |
| **Оптимизир.: параллельное чтение ×8 + батч-запись (~50 зап/с)** | **8 с** | **33 мин** | **5.6 ч** |
| Потолок только записи, sqlite batch (500 зап/с) | 1 с | 3.3 мин | 33 мин |
| Потолок только записи, PG COPY (32 000 зап/с) | <1 с | 3 с | 31 с |

Оптимизированная строка — реалистичная цель: миграция остаётся read-bound, но
параллельное чтение поднимает потолок чтения с ~17 до ~50 зап/с, а батч-запись
убирает её из критического пути. 1М записей: с ~45 ч (текущий) до ~5.6 ч.

## Рекомендованные оптимизации (по убыванию эффекта)

1. **Параллельные/пайплайнные чтения из ИРБИС** (×2.9 уже на 8 воркерах) —
   пул из N клиентов, читать `read_record` конкурентно. Самый высокий рычаг,
   т.к. чтение доминирует. Безопасно (read-only).
2. **Убрать второй round-trip (`format_record` G)** — `@brief` можно собрать
   локально из уже прочитанных полей `C` (fallback в `migrate.py` это уже
   умеет). Снимает ~50 % сетевой нагрузки на запись (с 2 round-trip'ов до 1).
   Либо, наоборот, формировать `brief` **server-side** пакетно одним `G` по
   списку MFN, а не по одному.
3. **Переиспользование соединения** — заменить connection-per-request на
   persistent socket (keep-alive) в адаптере для пути миграции: убирает
   TCP-handshake на каждую команду. (Влияет на `core.py`/адаптер — за рамками
   этого замера, но крупный рычаг.)
4. **Батч-коммиты на стороне стора** — sqlite: commit раз в N записей вместо
   на запись (×4.9). PG: одна транзакция + `executemany` (×17).
5. **Bulk `COPY` для PostgreSQL** + отложенная индексация: грузить через `COPY`,
   `search_vector`/GIN строить **после** bulk-load одним `UPDATE`/`CREATE INDEX`
   (×97 на пути записи). Аналогично sqlite — наполнять FTS5 в конце.
6. **Отложить FTS-индексацию** до конца bulk-load, затем переиндексировать одним
   проходом — убирает per-record DELETE/INSERT в FTS из горячего цикла.

## Воспроизведение

```
# sqlite (полная БД)
py irbis-web/server-own/bench_migrate.py

# без серверного форматирования (1 round-trip/зап)
py irbis-web/server-own/bench_migrate.py --no-format

# PostgreSQL
OWN_STORE_BACKEND=pg OWN_PG_DSN=postgresql://postgres:pg@127.0.0.1:5433/irbisweb \
  py -3.12 irbis-web/server-own/bench_migrate.py
```

Стор пишется в `bench.db` (gitignored); харнесс не создаёт прочих артефактов.
