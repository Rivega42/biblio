# Архитектурный план системы (замена САБ ИРБИС64+)

> Целевая архитектура единой веб-системы автоматизации библиотеки, заменяющей сервер ИРБИС64 + 6 desktop-АРМов + J-ИРБИС. Документ — «карта», по которой ведётся разработка; обновляется по мере работы. Опирается на: [API_CONTRACT](../build/API_CONTRACT.md), [OWN_SERVER_ARCHITECTURE](OWN_SERVER_ARCHITECTURE.md), [ACCESS_MODEL](ACCESS_MODEL_web-irbis.md), [COVERAGE_MATRIX](COVERAGE_MATRIX.md), [WIRE_PROTOCOL](../recon/deep/reference/protocol/WIRE_PROTOCOL.md). Пилот — СПб театральная библиотека.

## 1. Цели и принципы
**Цель:** современная веб-нативная библиотечная система, разворачиваемая в защищённом контуре (Astra Linux, без внешних CDN), которая (а) сразу работает поверх существующего ИРБИС64 как замена клиентов, (б) постепенно замещает и сам сервер ИРБИС нашим, с миграцией данных.

**Принципы:**
1. **Шов — один REST-контракт** ([API_CONTRACT](../build/API_CONTRACT.md)). Фронт не знает, что под ним: адаптер ИРБИС или наш сервер.
2. **Strangler / фасад.** Замещаем ИРБИС по доменам и базам, с dual-run и сверкой; откат возможен на каждом шаге.
3. **Доступ по грантам** (функция×база×уровень), а не «по АРМам».
4. **Конфиг из окружения; секретов в коде/образах нет.** Списки баз/поля/РЛ — данными с сервера.
5. **Точность данных при миграции** — round-trip поля/подполя (MARC-совместимая модель), сохранение `mfn_legacy`.
6. **Доступность и i18n — обязательны** (ГОСТ Р 52872 / WCAG).
7. **Модульный монолит сначала, сервисы — по необходимости** (масштаб библиотеки ≠ хайлоад).

## 2. Контекст системы (C4 L1)
- **Акторы:** Гость, Читатель (билет/ЕСИА), Сотрудник (по грантам: каталогизатор/книговыдача/комплектатор/КО/админ).
- **Внешние системы:** существующий **сервер ИРБИС64 :6666** (на время перехода), почта/SMS (уведомления), платёжный шлюз (штрафы), **RFID/ТСД** (Android), OAI-PMH/Z39.50/discovery (внешняя выдача), ЕСИА/СМЭВ (опц.), резервное хранилище.
- **Наша система:** веб-клиент + бэкенд-сервисы + хранилища.

## 3. Контейнеры (C4 L2) — целевая
| Контейнер | Технология | Роль |
|---|---|---|
| **Web-клиент (SPA)** | React + TypeScript + Vite (+ дизайн-система) | читатель + сотрудник; общается ТОЛЬКО по контракту |
| **API-шлюз / BFF** | Python (aiohttp/FastAPI) | контракт, аутентификация/сессии, **strangler-маршрутизация** per-endpoint/per-base на адаптер или ядро |
| **IRBIS-адаптер** | Python (наш `IrbisClient`, Проход Б) | говорит с ИРБИС :6666; реализует контракт поверх существующего сервера |
| **Каталог-ядро (наш сервер)** | Python + **PostgreSQL** | записи (JSONB поля/подполя), поиск, экземпляры, **ячеистое хранение**, каталогизация-запись |
| **Поиск** | PostgreSQL FTS (`tsvector`+`pg_trgm`) → **OpenSearch** при росте | индексация и запросы |
| **Доступ/идентификация** | Python + PostgreSQL | учётки, гранты, роли, аудит; вход reader/staff/guest |
| **Книговыдача** | Python + PostgreSQL | выдача/возврат/очередь/бронеполка/**постамат+ячейки** |
| **Комплектование / Книгообеспеченность** | Python + PostgreSQL | заказ→КСУ→списание; дисциплины/ККО (позже) |
| **Объектное хранилище** | S3-совместимое (MinIO) | обложки, полные тексты/PDF |
| **Файл-прокси** | Python | выдача бинарных ресурсов с проверкой прав (view-only) |
| **Воркер миграции/синхронизации** | Python | ИРБИС→ядро, инкрементальная синхронизация, **сверка** |
| **Уведомления / Платежи / Device-gateway** | Python | email/SMS; платёжный шлюз; RFID/ТСД |
| **Аналитика/BI** | PostgreSQL (+ материализованные представления) | дашборды |
| **Кэш/очередь** | Redis (позже) | сессии/кэш/фоновые задачи |

> **Упаковка на старте:** «Каталог-ядро + Доступ + Книговыдача» — **один модульный монолит** (внутренние модули), отдельно **IRBIS-адаптер** и **воркеры**. Вынос в сервисы — когда появится причина (масштаб/команда/релизные циклы).

## 4. Архитектура данных
**PostgreSQL — system of record.**
- `record(id, db, mfn_legacy, version, status, fields JSONB, + материализованные title/author/year/doctype/isbn/availability, search_vector tsvector)`.
- **Размещение/ячейки:** дерево `location(id, parent_id, kind, code, name, size, address, gx,gy,gw,gh)` (здание→этаж→помещение→стеллаж→полка→ячейка + постамат/станция); `holding(record_id, inv_no, status, location_id, rfid)`.
- **Авторитетные файлы и словари** — отдельные таблицы (миграция ATHR*/MNU/TRE).
- **Доступ:** `staff_account, grant_entry, role*, audit_log` ([ADR-004](../build/ADR-004_access-store-sqlite-postgres.md)).
- **Книговыдача:** `loan, request(RQST), hold, fine`.
- **Бинарь** (обложки/PDF) — объектное хранилище; ссылки в записи. Поиск — `tsvector`/OpenSearch.
- Версионирование записей (история правок) и `mfn_legacy`-маппинг для сверки.

## 5. Миграция и переключение (strangler)
1. **Зеркало (read-only):** воркер читает базу через адаптер → upsert в ядро (поля/подполя, обложки, экземпляры). 2. **Dual-run + сверка:** одинаковые запросы к адаптеру и ядру → дифф (число записей, контрольные выборки, поля). 3. **Переключение чтения** базы на ядро через шлюз. 4. **Переключение записи** (каталогизация/книговыдача) на ядро. 5. **Инкрементальная синхронизация** по дате правки, пока обе системы живы. 6. **Вывод ИРБИС** базы из эксплуатации. Маршрутизация — на шлюзе per-base; откат — переключением маршрута.

## 6. Безопасность и приватность
- Гранты функция×база×уровень + роли + аудит write/admin; денай и `-3338` → 403.
- **152-ФЗ:** минимизация ПДн (вход по билету; RFID/служебные id читателю не видны), сроки хранения, право на забвение, журнал доступа к ПДн.
- Сессии (JWT/подписанные), секреты в env, файлы через прокси с проверкой прав, без внешних CDN/трекеров, gitleaks в CI.

## 7. Сквозные аспекты
i18n (рус базовый), доступность (ГОСТ/WCAG, проверка в CI), темизация под учреждение, единый конверт ошибок (без внутренних деталей), идемпотентность записи, rate-limiting, наблюдаемость (логи/метрики/трейсы), бэкап PostgreSQL и объектного хранилища.

## 8. Технологии (с обоснованием)
React+TS+Vite+дизайн-система (веб-нативно, без CDN) · Python aiohttp/FastAPI (быстрый старт, async, у нас уже есть) · **PostgreSQL** (JSONB+tsvector+pg_trgm — гибкая модель + полнотекст в одном) · OpenSearch (когда нужен масштаб/релевантность) · MinIO (S3 в контуре) · Redis (кэш/очереди позже) · Docker/Compose (деплой; Kubernetes — опционально позже).

## 10. Фазовый план (привязка к эпикам)
- **Фаза 0 (готово):** клиент + адаптер + ядро-PoC + ячейки/план.
- **Фаза 1 — Фундамент/паритет читателя:** Access на PostgreSQL ([#99](https://github.com/Rivega42/biblio/issues/99)/[#92](https://github.com/Rivega42/biblio/issues/92)); шлюз-маршрутизация; объектное хранилище + просмотрщик полного текста ([#83](https://github.com/Rivega42/biblio/issues/83)); фасеты+навигаторы ([#80](https://github.com/Rivega42/biblio/issues/80)); расширенный ЛК ([#85](https://github.com/Rivega42/biblio/issues/85)). → клиент-читатель к паритету ([#97](https://github.com/Rivega42/biblio/issues/97)).
- **Фаза 2 — Сотрудник + ядро-запись:** полная каталогизация (РЛ/словари/ФЛК/импорт-экспорт, [#98](https://github.com/Rivega42/biblio/issues/98)); книговыдача через ячейки/постамат на ядре ([#95](https://github.com/Rivega42/biblio/issues/95)); миграция пилотных баз + сверка ([#94](https://github.com/Rivega42/biblio/issues/94)); редактор плана ([#96](https://github.com/Rivega42/biblio/issues/96)).
- **Фаза 3 — Охват и масштаб:** комплектование/книгообеспеченность/администрирование ([#98](https://github.com/Rivega42/biblio/issues/98)); OpenSearch; мультибиблиотечность/HA; BI.
- **Фаза 4 — Cutover:** переключение баз на ядро, вывод ИРБИС в архив.

## 11. Ключевые решения (ADR)
- **API-контракт как шов** (этот документ + API_CONTRACT). — принято.
- **Модульный монолит ядра**, адаптер и воркеры отдельно. — принято (пересмотр при масштабе).
- **PostgreSQL + JSONB + FTS** как модель/индекс; OpenSearch позже. — принято.
- **Strangler-миграция** по доменам/базам. — принято.
- ADR-003 (рендер: серверный PFT у адаптера / структурные данные у ядра), ADR-004 (sqlite-dev / PG-prod) — действуют.

## 12. Риски
Проводной протокол ИРБИС не полон (добор `IC_*`) · точность миграции (нужна сверка) · защищённый контур (self-hosted всё) · интеграция RFID/ТСД (устройства) · масштаб полнотекста (OpenSearch) · организационные (обучение, параллельная работа двух систем).
