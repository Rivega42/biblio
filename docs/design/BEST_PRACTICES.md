# Лучшие практики и пробелы верхнеуровневого проектирования

> Ревью архитектуры ([ARCHITECTURE](ARCHITECTURE.md)) против практик мирового уровня для **мультиарендного SaaS + edge/офлайн + микросервисы** в библиотечном домене. Цель — что добавить, чтобы продукт был «как у топов». Помечено: 🔴 решить сейчас (дорого ретрофитить) · 🟡 заложить рано · ⚪ позже.

## Вердикт
Скелет верный (контракт-шов, мультиарендность, edge/офлайн, модули, strangler). Но верхний уровень **не закрыт**: ниже — 14 областей практик и пробелов. Самое важное решить **сейчас** (🔴): модель **синхронизации/консистентности**, **уровень изоляции тенанта**, **contract-first (OpenAPI)**, **наблюдаемость и идентификаторы с первого дня**, **подпись/OTA для edge**. Их тяжело внедрять задним числом.

## 1. Мультиарендность и изоляция 🔴
- Модели: **pool** (общая БД + `tenant_id`+RLS), **silo** (БД/схема на тенанта), **bridge** (схема-на-тенанта в общей БД). У нас: схема-на-тенанта (edge — БД-на-тенанта). Зафиксировать политику и **тесты изоляции** (ни один запрос не пересекает тенант).
- Per-tenant ключи шифрования (envelope encryption), лимиты ресурсов (анти-noisy-neighbor), per-tenant миграции схемы (оркестрация апгрейдов сотен схем), tenant-aware пулы соединений (PgBouncer).
- **Entitlements-сервис** (что включено у тенанта) отдельно от биллинга — единый источник «можно ли».

## 2. Офлайн/edge-синхронизация 🔴 (самое рисковое)
- Выбрать модель осознанно: **LWW** (просто, теряет данные) vs **CRDT** (сходимость без потерь, сложнее) vs **event-sourcing + reducer**. Рекомендация: **журнал событий (event log) + явное владение** (каталог = облако-master, оперативные операции = edge-master), конфликты — доменные правила, не «последний победил».
- Практики: **outbox / change-data-capture**, idempotency-ключи, **векторные часы / Lamport** (не доверять wall-clock на edge), дельта-синхронизация, докачка, backpressure, журнал конфликтов с ручным разбором в UI.
- Готовые паттерны для вдохновения: ElectricSQL / PowerSync / Couchbase Lite / CouchDB-репликация. Не изобретать с нуля там, где есть проверенное.
- **Тестировать на хаосе**: обрывы, переупорядочивание, дубликаты, расхождение часов, долгий офлайн (недели).

## 3. Данные и консистентность 🟡
- **Outbox-паттерн** для надёжной публикации событий (без 2PC). CQRS/проекции — только там, где окупается (поиск, аналитика), не везде.
- Версионирование записей + **audit-trail** как данные; soft-delete/tombstone; data-lineage (откуда поле при миграции). Жизненный цикл ПДн (сроки, забвение) — на уровне данных.
- Миграции схемы: версионируемые, обратимые, **протестированные на копии**; оркестрация по тенантам.

## 4. API и контракты 🔴
- **Contract-first: OpenAPI как источник истины** (генерация клиентов/серверных стабов), **версионирование API**, **contract-tests (Pact)** между фронтом/шлюзом/сервисами.
- Идемпотентность мутаций (ключи), **курсорная пагинация**, единый формат ошибок (есть), rate-limiting, ETag/condition. Async — через события + статусные эндпоинты.

## 5. Поиск и релевантность 🟡
- Русская **морфология/стемминг**, синонимы, опечатки (fuzzy), бустинг полей, фасеты, типизация запроса. Порог PG FTS → **OpenSearch** при росте/релевантности. Реиндексация без простоя (alias-swap).
- Семантический поиск (эмбеддинги) — как модуль (issue ИИ).

## 6. Безопасность и комплаенс 🔴
- **Zero-trust**, mTLS между сервисами, секреты в **Vault/SOPS** (не env в проде), ротация. 
- **Supply-chain**: SBOM, **подпись образов (cosign)**, сканирование (Trivy), pinned deps, gitleaks/trufflehog в CI.
- Мультиарендность: автоматические **тесты изоляции** + pen-test. RBAC/ABAC. 
- **152-ФЗ/ФСТЭК**: классификация ИСПДн, журнал доступа к ПДн, аттестация для гос-сегмента; шифрование at-rest/in-transit.

## 7. Edge-парк и OTA-обновления 🔴
- **Подписанные обновления**, поэтапный rollout (canary по узлам), **автооткат**, версии/совместимость edge↔облако (skew-политика), офлайн-валидация лицензии (grace-период), телеметрия с узлов, удалённая диагностика. Это отдельная дисциплина (fleet management) — заложить рано.

## 8. Надёжность / SRE 🟡
- **SLO/SLI** на ключевые потоки (поиск, выдача), error budget. **OpenTelemetry** (трейсы+метрики+логи) и **корреляция запросов (trace-id, tenant-id)** с первого дня 🔴. Health/readiness, graceful degradation, circuit breakers. **Бэкапы + регулярные restore-учения (DR)**, RPO/RTO. Run-book’и.

## 9. Масштабируемость ⚪→🟡
- Пуллинг соединений (PgBouncer), **read-replica**, шардинг по тенанту при росте, кэш (Redis) с инвалидацией, очереди для тяжёлого (миграция/индексация/уведомления), idempotent-воркеры.

## 10. Доставка / DevEx 🟡
- **Trunk-based + CI/CD**, **feature-flags** (раскатка/эксперименты), blue-green/canary, **IaC (Terraform)** + GitOps, **ephemeral preview-окружения на PR**, автоматические contract/migration/изоляция-тесты в пайплайне. Монорепо с чёткими границами.

## 11. Модульность / плагины 🟡
- Стабильный **plugin-SDK** + semver, capability-based права, **sandbox** для сторонних модулей, реестр/маркетплейс, версионирование расширений UI. Контракт расширения = события + слоты UI (не прямой доступ к БД).

## 12. Коммерческие SaaS-операции 🟡
- Точный **metering** (хранилище/тексты/запросы/операции) → биллинг; жизненный цикл тенанта (trial→paid→suspend→churn), self-serve админка, провижининг/деинициализация, экспорт данных тенанта (анти-lock-in), отчёты для закупок.

## 13. Библиотечные стандарты и интероперабельность 🟡 (домен!)
- Форматы: **MARC21/RUSMARC, ISO 2709, MARCXML**; в перспективе **BIBFRAME/Linked Data** (RDA/FRBR) — конкурентное преимущество.
- Протоколы обмена: **Z39.50 / SRU/SRW, OAI-PMH** (выгрузка/харвестинг), **SIP2/NCIP** (самообслуживание/RFID-станции, межбибл.). 
- Гос-интеграции РФ: **ЕСИА** (вход читателя), **СМЭВ**, **НЭБ**, КСУ-отчётность, статистика 6-НК. Это «галочки» для закупок.

## 14. ИИ/ML (как модули) ⚪
- Семантический поиск/эмбеддинги, авто-классификация (ГРНТИ/УДК/ББК), OCR полнотекста, рекомендации, дедуп/авторитетный контроль с ML, ИИ-ассистент. Уже в эпиках; вынести в отдельный модуль с явной приватностью.

## Что добавить в архитектуру (пробелы, не закрытые сейчас)
1. **Стратегия синхронизации** — отдельный design (модель владения, конфликты, событийный журнал). 🔴
2. **Наблюдаемость и сквозные идентификаторы** (OTel, trace-id+tenant-id) — с первого сервиса. 🔴
3. **Contract-first (OpenAPI) + contract-tests**. 🔴
4. **Edge OTA / fleet-management** дисциплина. 🔴
5. **Entitlements-сервис** (отдельно от биллинга). 🟡
6. **Стандарты домена** (MARC/Z39.50/OAI/SIP2/ЕСИА/СМЭВ) в дорожной карте. 🟡
7. **DR/бэкапы + restore-учения**, SLO. 🟡
8. **Supply-chain безопасность** (SBOM/подпись/скан) в CI. 🟡

> Рекомендация: оформить 🔴-пункты как ADR/design-доки до старта Фазы 1–2, остальное — в дорожную карту (issues). Это и есть «практики топов»: контракт-first, наблюдаемость и идентичность с нуля, осознанная модель консистентности, безопасная цепочка поставок и OTA, изоляция тенантов под тестом.
