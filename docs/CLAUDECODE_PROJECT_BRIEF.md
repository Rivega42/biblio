# Веб-ИРБИС — Полный бриф проекта для Claude Code (онбординг и ведение репозитория)

> Это главный вводный документ. Прочитай его целиком перед работой. Он объясняет: **зачем** проект,
> **что** уже сделано, **какие документы** есть и где истина, **чего не хватает**, **что можно делать
> на этом ПК**, и **как организовать и вести репозиторий**. Остальные документы комплекта — детализация;
> точка входа в них — `PROJECT_INDEX.md`. Дата 20.06.2026.

---

## 1. Конечная цель проекта

Создать **полноценную ВЕБ-версию системы автоматизации библиотек ИРБИС64+** — заменить настольные
клиенты (Читатель, Каталогизатор, Книговыдача, Комплектатор, Книгообеспеченность, Администратор)
**единым веб-приложением**, организованным **по учёткам и грантам**, а не по «АРМам».

- **Первая боевая инсталляция** — каталог Санкт-Петербургской театральной библиотеки (СПб ГТБ),
  кодовое имя `irbis-web`: публичный веб-каталог (функции читателя) + часть книговыдачи. Это **пилот P0**.
- **Дальше** — наращивание остальных функций (каталогизация, комплектование, книгообеспеченность,
  администрирование) поверх общего ядра.

**Почему это вообще делается:** разработчика ИРБИС больше нет, исходников сервера нет, вендорский
веб-слой (J-ИРБИС) устарел и имел дефекты. Поэтому **демо-поставка продукта = действующая
спецификация**, а наша работа — восстановить её и построить современную веб-замену.

## 2. Контекст и принципы (читать обязательно)

- **Демо-поставка = спецификация.** Логику продукта мы восстановили из файлов и документации демо-
  установки ИРБИС64+ 2023. Это источник истины о том, «как должно работать».
- **Организация по учёткам/грантам, не по АРМам.** Разрешение = **(функция × база × уровень)**; на
  учётке — набор грантов; роли — опциональные пресеты. «АРМ» — это только **технический контекст
  протокола** (см. ниже), не раздел интерфейса. Подробно — `ACCESS_MODEL_web-irbis.md`.
- **Протокол гейтит по типу АРМа.** На уровне сети сервер ИРБИС требует регистрироваться сессией
  определённого типа (`IRBIS_READER/CATALOG/COMPLECT/BOOKLAND/BOOKPROVD/ADMINISTRATOR`), иначе ошибка
  `-3338`. Бэкенд держит служебные креды по типам (в env) и под грант учётки открывает нужную сессию.
- **Отображение записей — серверным PFT (ADR-003).** Форматы вывода считает сервер ИРБИС (его язык
  PFT), мы их не переписываем в коде — запрашиваем готовый вывод и оформляем поверх.
- **Безопасность — некомпромиссна.** В прошлом был инцидент: утечка боевых кред/ПДн в публичный
  репозиторий. Отсюда железные правила (раздел 8): секреты только в env, бинарь/данные/INI с кредами —
  не в репозиторий, секрет-сканеры в CI.
- **Документы первичны и явны про неизвестное.** Всё незакрытое помечаем `TODO(...)`, факты — со
  ссылкой на источник. Поведенческую и инженерную спецификации держим раздельно.

## 3. Архитектура (целевая)

- **Фронтенд:** React + TypeScript + Tailwind. Тонкий клиент; экраны собираются **по грантам учётки**.
- **Бэкенд:** Python (aiohttp). Посредник между вебом и сервером ИРБИС; держит слой доступа и пул
  серверных сессий по типам АРМ.
- **Протокол ИРБИС:** нативный TCP (порт 6666). Клиент протокола переиспользуется из проекта
  **BookCabinet** (рабочий клиент — эталон). Полная карта команд — `irbis64_client_dll_TEXT.txt` и
  `RECON_FINDINGS_02.md`.
- **Хранилище приложения:** PostgreSQL — учётки сотрудников, гранты, роли, аудит (читатели — в БД `RDR`
  самого ИРБИС, вход по билету).
- **Развёртывание:** в контуре (Astra Linux на боевом), без внешних CDN/трекеров. Доступность — ГОСТ Р 52872-2019.
- **Граница протокола в коде:** интерфейс `IrbisClient` (мок для разработки / адаптер BookCabinet в
  бою) — позволяет собирать и тестировать без живого сервера. См. `P0_BUILDKIT_web-irbis.md`.

## 4. Что уже сделано (хронология)

1. **Картография продукта по архиву** (демо-поставка ИРБИС64+ 2023): сервер, реестр 51 базы, раскладка.
2. **Полный каталог протокола** — из официальной доки API клиентской DLL (не угадано, а документировано).
3. **Проход А (конфиги):** механика префиксов (секция `[SEARCH]` + FST), словарей, форм ввода (WSS),
   форматов отображения (PFT, диспетчер по `v920`, веб-варианты `_H`), путей (PAR п.11), связей.
4. **GUAR Фонд↔Опись = поле 488** — подтверждено конкретно.
5. **Структура всех 6 настольных клиентов** (источник инвентаря функций) и веб-модель вендора.
6. **Проектный слой:** модель доступа (учётки/гранты), привязка функций к протоколу, инженерное ТЗ +
   аддендум, P0 build kit (контракты для кода), схемы экранов читателя и сотрудника.

**Полнота честно:** сильная карта на уровне структуры/механики; **не** исчерпывающая до каждого поля
(PFT/WSS/CHM сэмплированы). Достаточно для архитектуры и каркаса P0; **не** для прямой генерации
боевого кода по каждому полю без сверки на живом сервере.

## 5. Комплект документов (что есть и где истина)

Точка входа и порядок чтения — **`PROJECT_INDEX.md`**. Кратко:

**Проектирование (что строим):**

- `ACCESS_MODEL_web-irbis.md` — модель доступа (учётки × гранты × базы) — организующий стержень.
- `FUNCTION_PROTOCOL_MAP.md` — функция → команды протокола `IC_*` + контекст.

- `P0_BUILDKIT_web-irbis.md` — контракты P0: структура репо, схема БД, API, интерфейс протокола.
- `SCREENMAP_web-reader.md` / `SCREENMAP_web-staff.md` — схемы экранов (по функциям/грантам).

**Доказательная база (откуда выводы, с источниками):**
- `RECON_FINDINGS_01..07.md` — находки по областям (сервер/базы, протокол, GUAR/веб-модель, конфиги,

- `ReadMe_jirbis2_text.txt` — текст мануала J-ИРБИС (поведенческий референс).

**Методика и перепроверка:**
- `TZ_IRBIS_RECON.md` — методика картографии (статика / живой сервер / UI клиента).
- `PROMPT_CLAUDECODE_RECON.md` — промт для **независимой** перепроверки карты (для сверки с нашей).

> Возможно, в проекте есть документы прошлой фазы: `FINAL_TZ_irbis-web.md` (поведенческий эталон

> Если они есть — учитывай их как поведенческий эталон для веб-Читателя.

## 6. Чего не хватает (открытые задачи)

1. **Живой сервер (Проход Б), `:6666`:** точные форматы пакетов и коды результата `IC_*`, поведение
   поиска/словаря, структура прав в БД `RIGHT`, протокол очереди заказов. → раздел 7.
2. **Боевые базы СПб ГТБ** (PLAY/TUAR/EK/ESKIZ/HPO): их `[SEARCH]`-виды/префиксы, форматы PFT, рабочие
   листы — в демо их нет, нужны с сервера библиотеки. Механика к ним применяется та же.
3. **Точные правила полей** Каталогизатора (нормативная база валидаций) — по конкретному полю.
4. **Независимая перепроверка** карты (`PROMPT_CLAUDECODE_RECON.md`) и сверка с нашей.
5. **Уточнения под штатку СПб ГТБ:** финальные уровни/пресеты-роли.

## 7. Что можно сделать на этом ПК (локально)

Установка ИРБИС64+ 2023 развёрнута локально (сервер `192.168.8.19:6666` / `localhost:6666`, рабочая
папка `C:\IRBIS64\DATA`, демо-креды несекретны). Запускать тебя удобнее **локально** (для протокола/
файлов — WSL; для GUI настольных клиентов — нативно Windows). Что реально можно сделать:

1. **Собрать каркас P0 на моке** — не требует сервера: реализуй `IrbisClient` мок + ядро + слой
   доступа + веб-Читатель по `P0_BUILDKIT`. Это лучший старт.
2. **Проход Б — зондировать живой сервер** `:6666`: подключиться служебным логином, прогнать
   `IC_search`/`IC_read`/`IC_readformat`/`IC_nexttrm`, зафиксировать форматы пакетов и коды; проверить
   `RIGHT` и протокол заказа. Подтвердить опкоды по клиенту BookCabinet. Заменить мок на боевой адаптер.
3. **Снять конфиги боевых баз** (если доступен сервер библиотеки): `[SEARCH]`/PFT/WSS для PLAY/TUAR/EK.
4. **Независимая картография** по `PROMPT_CLAUDECODE_RECON.md` и сверка с нашей картой.

⚠️ Демо-сервер «для дилеров», лимит **3 клиента** — береги сессии в пуле.

## 8. Как организовать и вести репозиторий

**Структура** (см. `P0_BUILDKIT_web-irbis.md` §1): `backend/` (aiohttp: irbis/ access/ api/),
`frontend/` (React-TS), `infra/` (docker-compose PG, миграции), `docs/` (вся проектная документация),
`README.md`, `.gitignore`, `.env.example`.

**Документация в репо.** Перенеси сюда, в `docs/`, весь комплект из раздела 5 и поддерживай его
актуальным. Заведи `docs/PROJECT_INDEX.md` как оглавление и `CHANGELOG.md`. Соглашение для незакрытого:
машинно-находимые метки `TODO(irbis-web #N: …)`; собирай их в отчёт. Любой нетривиальный факт — со
ссылкой на источник.

**Git-дисциплина.** Малые осмысленные коммиты; ветки по фазам/фичам; PR с описанием. История должна
читаться как ход работы.

**.gitignore (обязательно):** `.env`, любые `*.ini` с кредами, бинарь (`*.exe/*.dll`), файлы данных
ИРБИС (`*.mst/*.xrf/*.ifp/*.l01/*.n01`), картинки-данные, дампы, локальные распаковки архивов.

**Секреты.** Только в env (`.env` локально, секрет-менеджер в проде). В репо — `.env.example` со
схемой ключей без значений. Подключи **gitleaks/trufflehog** в pre-commit и в CI — это прямое
требование после прошлой утечки.

**Что НЕ кладём в репо:** исходные файлы продукта ИРБИС (конфиги/доки/бинарь) — они локально и в
`.gitignore`; в репо идут только наши производные (код + документация карты).

## 9. План сборки (фазы)

- **P0 — Ядро + Access + Читатель (пилот СПб ГТБ).** Протокол-клиент (мок→BookCabinet), поиск/рендер
  (серверный PFT), слой доступа (учётки/гранты/резолвер/пул/аудит), веб-Читатель (каталог + ЛК + заказ).
- **P1 — Книговыдача** (очередь заказов/выдача/возврат поверх RQST/RDR).
- **P2 — Каталогизация** (формы рабочих листов, правка полей, глоб. корректировка; загрузка файлов —
  файловым доступом, не протоколом).
- **P3 — Комплектование, Книгообеспеченность** (выходные/табличные формы).
- **P4 — Администрирование** (`IC_adm_*`; строжайший контур/аудит).

Внутри каждой фазы — функции включаются по грантам, не пакетом.

## 11. Первые действия

1. Инициализировать репозиторий, `.gitignore`, `.env.example`, `docs/` с комплектом и `PROJECT_INDEX.md`.

5. Запустить независимую перепроверку по `PROMPT_CLAUDECODE_RECON.md`, сверить с картой.
