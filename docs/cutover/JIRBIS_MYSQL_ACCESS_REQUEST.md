# Что нужно по доступу к MySQL J-IRBIS (Joomla-БД `jirbis`) — cutover СПб ГТБ

> Полноформатное описание: ЗАЧЕМ нужен доступ, ЧТО именно запросить, ЧТО в этой
> БД лежит (и чего там НЕТ), КАК передать, режим безопасности (152-ФЗ), и ЧТО мы
> с этим делаем. Связано: [`JIRBIS_CUTOVER_PLAN.md`](JIRBIS_CUTOVER_PLAN.md) (узел 4).
>
> **TL;DR для админа СПб ГТБ:** нужен **read-only доступ к MySQL-базе `jirbis`**
> (или табличный `mysqldump`) текущего сайта `jirbis.sptl.spb.ru` (J-IRBIS 2.0 на
> Joomla). Нужны НЕ читательские формуляры (они в ИРБИС, не в Joomla), а
> Joomla-слой: учётки/соц/CMS/заявки. **Дамп содержит ПДн → по защищённому каналу,
> в git НЕ коммитим.**

## 0. Контекст

СПб ГТБ переходит с легаси `jirbis.sptl.spb.ru` (J-IRBIS 2.0 — модуль `com_irbis`
на Joomla + MySQL, EOL-стек) на **Biblio**. Боевой ИРБИС остаётся источником
библиографии и движком книговыдачи (read-only до отмашки, посыл #222). Joomla —
это **веб-обвязка** поверх ИРБИС: портал читателя, CMS, соц-слой, AI-ассистент,
заявки. Чтобы перенести то, что живёт ТОЛЬКО в Joomla (а не в ИРБИС), нужен
доступ к MySQL-базе `jirbis`.

## 1. Ключевой факт: что в ИРБИС, а что в Joomla

**В ИРБИС (НЕ в MySQL Joomla) — переносится отдельным треком, доступ к MySQL не нужен:**
- Читатель и формуляр — БД **RDR** (поля 10/11 ФИО, 30 билет `RI=`, 40 выдачи, 50 категория, 130 — данные входа).
- Заказы/брони — БД **RQST** (поля 30/40/903/910^A reservstatus).
- Бесшовный вход читателя в Biblio **уже работает** напрямую через ИРБИС (поле 130, plaintext) — формуляр/выдачи мигрируются из ИРБИС, не из Joomla.

**Только в Joomla (MySQL `jirbis`) — ради этого и нужен доступ:**
- **Учётные записи сайта** (`jos_users`): логины/email/хэши паролей (~240 phpass `$P$…`), профили. Нужны для бесшовного переноса входа CMS-учёток (upgrade-on-login: phpass-верификатор уже готов — `access/phpass.py`, #294).
- **Соц-слой / CMS-контент**: отзывы, полки, новости/страницы/меню Joomla, баннеры, настройки портала.
- **Заявки/обращения**, корзина CMS-стороны, история AI-ассистента.
- **AI-чат фидбэк** (`jos_ai_chat_feedback`: `user_message`/`assistant_content`/`tool_calls_json`/`rating`) — на боевом **0 строк** (данных нет, но схему фиксируем).
- Бронирования CMS-стороны (`*reservation*` — ориентир ~34 строки).

## 2. Что именно запросить у админа (чек-лист)

Достаточно ОДНОГО из двух путей; предпочтительно — табличный дамп.

**Путь A (предпочтительно) — `mysqldump`:**
- Дамп БД `jirbis` (или хотя бы таблиц из §3). Команда-ориентир:
  `mysqldump --single-transaction --default-character-set=<charset> jirbis > jirbis_dump.sql`
- Уточнить **charset** (cp1251 или utf8 — критично для кириллицы; ср. баг кодировки поиска #228).

**Путь B — read-only доступ к MySQL:**
- host / port, имя БД (`jirbis`), **read-only** пользователь + пароль (SELECT only).
- Сетевой доступ (туннель/VPN/IP-whitelist), если БД не наружу.

**Сопутствующие данные (по любому пути):**
1. Версия Joomla + версия `com_irbis` (для маппинга схемы `jos_*`).
2. Префикс таблиц (обычно `jos_`, но мог быть изменён).
3. Кодировка БД/таблиц (`SHOW CREATE TABLE`, `SHOW VARIABLES LIKE 'character_set%'`).
4. Размеры: `SELECT COUNT(*)` по ключевым таблицам (§3) — оценить объём миграции.
5. Подтверждение: есть ли в Joomla что-то, чего НЕТ в ИРБИС и что важно сохранить
   (кастомные таблицы `com_irbis`, интеграции, `llm_router`-конфиг AI-ассистента).

## 3. Целевые таблицы (что вытащить и зачем)

| Таблица (ориентир) | Зачем | ПДн? |
|---|---|---|
| `jos_users` | учётки сайта + ~240 phpass-хэшей → upgrade-on-login (#294) | **да** (email/ФИО/хэш) |
| `jos_user_profiles` / `jos_user_usergroup_map` | профили, группы/роли | да |
| `jos_ai_chat_feedback` | история AI-ассистента (схема; прод 0 строк) | да (содержимое диалогов) |
| `*reservation*` / заявки | брони/заявки CMS-стороны (~34) | да |
| соц-таблицы (отзывы/полки/рейтинги `com_irbis`) | соц-слой портала | да |
| `jos_content` / `jos_menu` / `jos_modules` | CMS-контент/навигация/баннеры | нет (обычно) |
| конфиг `com_irbis` / `llm_router` | настройки портала + AI-прокси | секреты (ключи!) |

> Точные имена таблиц подтверждаются по `SHOW TABLES LIKE 'jos_%'` после получения
> доступа — список выше ориентировочный (по реверсу `com_irbis`).

## 4. Безопасность и 152-ФЗ (жёстко)

- **Дамп содержит ПДн** (ФИО, email, хэши паролей, диалоги) → **НЕ коммитить в git**,
  не класть в репо, не светить в чатах/issue.
- Хранить вне репозитория (напр. `D:\Театральная Библиотека\` — там же, где прочие
  не-коммитимые материалы инцидента), передавать по защищённому каналу.
- **240 phpass-хэшей** — переносим/обрабатываем **off-git**; в коде только верификатор
  (`access/phpass.py`), сами хэши в репо не попадают.
- **Секреты `llm_router`** (ключи LLM-провайдера) — не коммитить, ротировать при переносе.
- Пароль/креды боевого jirbis — **никогда** в память/репо.
- Боевой ИРБИС при этом не трогаем (read-only, #222) — это отдельный источник.

## 5. Что мы делаем после получения (план переноса, узел 4)

1. **Интроспекция дампа** — `SHOW TABLES`/`COUNT` → точная карта `jos_*`, кодировка,
   реальные объёмы (наш `access/migrate_introspect`-подход).
2. **Учётки** → own-store: перенос `jos_users` с **upgrade-on-login** (phpass → нативный
   pbkdf2 при первом входе, #294). Сверка с ИРБИС RDR по билету/email, где возможно.
3. **Соц-слой** → `access/social.py` (отзывы/полки/рейтинги).
4. **AI-история** → own-store `chat_message` (аналог `jos_ai_chat_feedback`, спека #293).
5. **CMS-контент** → портал/CMS Biblio (страницы/меню/баннеры).
6. **Заявки/брони CMS** → own-store holds/orders (где не дублируют RQST из ИРБИС).
7. **Сверка**: читатель в ИРБИС (RDR) ↔ учётка Joomla (`jos_users`) — связать по
   билету/email, не задвоить личность.

## 6. Открытые вопросы (уточнить с админом)

- Кодировка БД (cp1251 vs utf8) — влияет на корректность кириллицы при импорте.
- Реальный префикс таблиц и версия Joomla/`com_irbis`.
- Есть ли кастомные таблицы `com_irbis` сверх стандартного Joomla.
- Где живёт конфиг `llm_router` (БД vs файл) и какие там секреты.
- Нужно ли сохранять CMS-контент 1:1 или редизайнить (ср. трек редизайна сайта SPTL).
