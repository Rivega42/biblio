## 6. Расчёт книгообеспеченности (ККО/КМИ) из конфигов

### 6.1 Где и как считается

Расчёт выполняется **АРМ «Книгообеспеченность»** (конфиг `irbisk_user.ini`), не самой БД VUZ как хранилищем.
Ключевые управляющие параметры:

| Параметр (`irbisk_user.ini`) | Значение | Смысл |
|---|---|---|
| `GetKkoBook` | `1` | Способ подсчёта ККО: в таблицах 1/1.1 и `LicenDisc` — **по книгам по полю 693**; `0` — без поля 693 |
| `TotalStudBook` | `1` | Учёт общего числа студентов на книгу |
| `PftForEkzKMI` | `KoStatEkzKmi` | Формат подсчёта экземпляров для КМИ |
| `PftForEkzCom` | `KoKmiEkz` | Экземпляры в ККО + таблицы |
| `PftStatDisc` | `KoStatDisc` | Фильтрация в суммарных по дисциплине |
| `PftElectroName` | `KoElectro` | Формат, определяющий ЭУ (электронный учебник) с ККО=1 |
| `DbNameForKKO` | `DbNameForKKO.mnu` | Доп. БД каталога для подсчёта (DLS/KBOOK/KNIGI/BOOK) |
| `LimPftFull_boko691` | `10` | Порог свёртки повторений поля 691 |
| `MetaQueryKO` | `"!I=$"` | Запрос отбора записей в отчёт по умолчанию |

`DbNameForKKO.mnu`: `DLS`=Электронная библиотечная система, `KBOOK`/`KNIGI`/`BOOK`=Картотека книгообеспеченности.

### 6.2 Формула (из `KoDiscFull.tbg` / `DiscVuz-Cat.tbg` / `lncRDRVuz.pft`)

- **Кол-во наименований** (КМИ-наимен.) — счёт книг каталога, привязанных к дисциплине: поиск
  `IBIS,?DISC=<назв.дисц.>?` → подсчёт MFN (`f(rsum(...),0,0)`), см. `DiscVuz-Cat.tbg` CELL_TAB6.
- **Кол-во экземпляров** — суммирование экземпляров (поле 910/693^экз.) найденных книг.
- **Кол-во студентов** — из БД RDR (если `ACCESSRDR=1`): `&uf('JRDR,LN=<связка>')`,
  иначе из поля 68^Z. Деление чёт/нечёт семестра по `s('0 1 3 5 7 9 11'):v68^F` (нечётн.→осенний `!0=`,
  чётн.→весенний `!9=`), см. `VUZ.PFT:18–31`, `lncRDRVuz.pft:11–24`.
- **ККО** = экземпляры / студенты; **средняя ККО** = Σ(ККО) / N(наименований) (агрегаторы G13/G14/G61
  в `KoDiscDSort.pft`); опц. нормировка к 1 при `v1^^=1`. **КЭИ** (электр.) — доля дисциплин с
  достаточным числом ЭУ (`v1^9` порог, `v1^%` флаг).

### 6.3 Поле каталога 691 — структура (из `VuzSecCat.fst`, `Move691.gbl`)

`VuzSecCat.fst` формирует строку 691 из записи VUZ:
```
^I<id дисц 3^0> ^D<назв 3^A> ^S<цикл v4> ^B<каф v5> ^K<УК v6>
^A<фак> ^V<вид> ^O<форма> ^C<спец> ^N<напр> ^F<сем> ^H<вып.каф> ^L<филиал> ^E<группа>
```
(краткая форма при `MOVECATFULL=0`: без `^D^B^C^N` — только id/цикл/УК/связка.)

`WPtoCat.fst` пишет 991 из 68: `^A^N^C^V^O^F^E^D(вып.каф из 68^H)`.

### 6.4 Поле каталога 692 — архив

`GlobArhiv=Arhiv692.gbl` (`irbisk_user.ini:27`) переносит снятые/исторические привязки 691 в архивное
поле 692. `TaskDel691=VDEL691.gbl` удаляет повторения 691 (физически в каталоге).

## 9. Связи VUZ ↔ IBIS (691) ↔ RDR (90)

Карта заданий из `Link.mnu` (триггер по типу записи → GBL):

| Тип записи / контекст | Задание | Эффект |
|---|---|---|
| `RDR`, `RDRU` | `LINKRDR` | Привязка дисциплин к студентам RDR (поле 69/90) |
| `DISC` | `VADD83` | Добавление контингента (83) |
| `DISC` | `LINKDISC` (`LinkDisc.gbl`) | DISC → VUZ: создаёт/находит контингент VUZ, пишет 69 (^B каф, ^D id), при `ACCESSRDR=1` дёргает LinkRDR |
| `VUZ` | `LINKVUZ` (`LinkVuz.gbl`) | Чистка дублей 69; проверка дублей VUZ; запуск `CreateFak` и `LinkRDR` через `LINKCORRTASK` |
| `FAK` | `LINKFAK` (`LinkFak.gbl`/`CreateFak.gbl`) | Сборка записи FAK из контингентов |
| `PAZK`,`PVK`,`SPEC` | `LinkIbis` | Связь с каталогом IBIS |

**Поток данных (реконструкция):**

```
DISC (дисциплина, поле 83 = контингенты)
   │  LinkDisc.gbl
   ▼
VUZ (контингент, поле 68 = «связка»; поле 69 = изуч. дисциплины ^B/^D)
   │                                   │
   │ Move691.gbl / vAdd691.gbl         │ LinkVuz.gbl → LINKCORRTASK=LinkRDR
   ▼                                   ▼
IBIS (каталог)                      RDR (студенты)
  поле 691 (^I id, ^D назв, ^B каф,    поле 69 (изуч. дисциплины),
   ^S цикл, ^K УК, ^A^N^C^V^O^F связка) поле 90 (доп. контингент студента)
  поле 692 (архив, Arhiv692.gbl)
  поле 693 (сводная ККО по книге, GetKkoBook=1)
  поле 694 (заявка/комплектование)
```

- **VUZ → IBIS**: «связка» (68^A^N^C^V^O^F) переносится в каталог как поле **691** (те же подполя)
  + id дисциплины (691^I = DISC 3^0) + наименование/кафедра/цикл/УК. Реализация: `Move691.gbl`,
  `vAdd691.gbl`, генератор `VuzSecCat.fst`. ККО считается на стороне каталога по полю **693**.
- **VUZ → RDR**: задание `LinkVuz.gbl` пишет `LINKCORRTASK=LinkRDR%DBN=RDR%QUERY="?<связка>"`
  (или `GR=<группа>` при `LinkGroup=1`). В RDR обновляются поле **69** (изучаемые дисциплины) и
  поле **90** (доп. контингент), см. `MoveVuzRdr.gbl`. Число студентов читается обратно:
  `&uf('JRDR,LN=<связка>')` (`VUZ.PFT`, `lncRDRVuz.pft`).
- **Обратная сверка**: студенты считаются из RDR при `ACCESSRDR=1`; при `=0` — из 68^Z в самой VUZ.
- **FAK**: `LinkVuz.gbl` при отсутствии записи FAK для `FFO=<фак>-<форма>` ставит
  `LINKCORRTASK=CREATEFAK` (→ `CreateFak.gbl`).
