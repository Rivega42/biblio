## 1. IMAGE — имидж-каталог (образы карточек)

**Назначение.** Электронный имидж-каталог: хранит сканированные образы каталожных карточек и/или встроенные графические ресурсы, индексирует распознанный текст. Рабочий лист и основная FST-индексация почти идентичны IBIS; отличие — в полях-ссылках на файлы, FST-«шапке» и формате показа графики.

### 1.1 Ключевые отличающиеся поля
| Поле | Подполя (.wss) | Назначение | Источник |
|---|---|---|---|
| **951** | `^A` имя файла, `^I` URL (`951.mnu`: `http://www.`,`ftp://`), `^T` текст ссылки | внешний файл-объект | `951.WSS` |
| **953** | `^A` тип ресурса (`953.mnu`: JPG/GIF/BMP=«Графика»), `^T` название, `^P` характер просмотра (`xp.mnu`: 0 контекстный/1 неконтекстный), **`^B` двоичный ресурс — ввод `!12` (бинарник в записи)** | встроенный двоичный ресурс (графика внутри записи) | `953.WSS` |
| **952** | `^a/^b/^c/^C/^N` | образ карточки / привязка к ящику-разделителю картотеки | `IMAGE.FST` |
| **955** | `^a` | распознанный (полный) текст образа | `IMAGE.FST` |
| **907** | `^C` этап работы (`etr.mnu`: `ОБРНЗ`=не выдаётся), `^A` дата, `^B` ФИО | состояние обработки образа | `907.WSS` |

### 1.2 Отличия FST (`IMAGE.FST`, 458 строк) — только «шапка» (стр. 1–22)
Тело (стр. 24–458) = стандартная IBIS-индексация. Отличия:
| Строка | Метка | Метод | Префикс/термин | Смысл |
|---|---|---|---|---|
| 2 | 955 | **9** | (разбор на слова) | распознанный текст, идёт первым (`/*9 method must be first`) |
| 3–4 | 955 | 0 | `TXT=`, `V=FT` | слова полного текста; вид = full text |
| 12 | 1112 | 8 | сводный «образ карточки» | `&uf('++0907,951,953,955,691,692,693,910,941,900,920,903')` — собирает образ карточки из полей-ссылок |
| 14 | 952 | 9 | `KT=` | слова из имени файла-образа (`mhl,'/KT=/'`) |
| 16 | 952 | 0 | `TXT=` | слова полного текста из 952^b |
| 21 | 903 | 0 | **`I=`** из `952^b` | шифр документа строится из имени файла-образа → СВЯЗЬ С IBIS |
| 22 | 952 | 0 | **`BX=`** | ящики/разделители картотеки |
| 13 | 1113 | 8 | `KT=` из 953 | слова из встроенного ресурса |
| 5,10 | 1001 | 0 | `GUID=` | GUID записи (`&uf('+*')`) |

Доп. `TEXT.fst`: `952^b`→`TXT=`, `952^B` (PDF, стр. `__0001.`)→`TXT1=`, `952^c`→`TXT2=`.

### 1.3 Отличия рабочего листа
`IMAGE.WS` (208 стр.) по первым полям **совпадает с `IBIS.WS`** (упрощённое БО: 700/701/200/923/210/205/215/300/320/225/910/675/908/903/906/964/606/327/610/331…). Отдельного «листа ввода имидж-карточки» нет; поля 951/953/952/955 в `IMAGE.WS` отсутствуют — ввод образов идёт через `.wss`-подлисты этих полей. Источник: `IMAGE.WS`, проверено Grep (951/953/952/955 не выводятся строкой в WS). **920** наследуется из `WS.OPT`/`920.MNU` (тот же набор, что IBIS).

### 1.4 Отличия форматов показа
- **`PFTW.OPT`** — строка по умолчанию `+++++ IMAGE` вместо обычного библиоформата: все «прочие» записи показываются имидж-форматом (а не `KN_H`). Источник: `PFTW.OPT`.
- **`PFTW.MNU`** — отличительный пункт `UNI_H` «Для всех записей с показом графического внешнего и внутреннего файла непосредственно в формате (HTML)».
- **`Uni_H.pft`** (главный имидж-формат): встроенный ресурс (953^B) рендерится как `<IMG SRC="IRBIS:12,,mfn,1">` (**слот доступа 12 = бинарник в записи по mfn**); внешний файл (951) — `<IMG SRC="IRBIS:10,,Ag951#1">` / `PATH=10` (**слот 10 PAR = директория внешних файлов**); закомментированы альтернативы `PATH=1`/`PATH=0`. Статус 907^C:`ОБРНЗ` → «Издание в обработке. Не выдается». Источник: `Uni_H.pft`.

### 1.5 Поисковый профиль (`image.ini`) — отличается от IBIS
`[SEARCH]` `ItemNumb=3`: префиксы **`KT=`** «Слова полного (распознанного) текста», **`BX=`** «Ящики/Разделители», **`I=`** «Шифр документа»; `DebilPrefix=KT=`. Навигатор `WNLink0=irbis:1,,IMAGE,SEARCH_IC_WN`. Источник: `image.ini`. То есть поиск ориентирован на полный текст образов и ящики-разделители, а не на полный библиословарь.

### 1.6 Связь с IBIS
Реализована через **общий шифр документа** (поле 903 / индекс `I=`), формируемый в IMAGE из имени файла-образа `952^b` (`IMAGE.FST` стр. 21). Запись IMAGE «указывает назад» на каталожную запись IBIS по одинаковому шифру `I=`. Прямого кросс-базового вызова `ref(…,'IBIS',…)` в конфигах IMAGE не найдено.

## 2. SK — Сводный каталог (Union Catalog)

**Назначение.** Сводный (union) каталог: агрегирует записи нескольких библиотек-участников; ведётся импортом записей участников с дедупликацией. Движок IBIS; набор полей/920 — тот же. Отличия — в модели «держателя» (902), механизме импорта/дедупликации (`impsvk*`) и ВУЗ-ориентации (книгообеспеченность).

### 2.1 Модель сводного каталога — держатель/сигла (поле 902)
- **902 «Держатель документа»** — `DEFAULT.WS` стр. 844–851, подсказка «Организация — участник Сводного каталога», подлист `902.wss`.
- Подполя 902 (из `Deposit\902.WSS`): `^a` наименование, `^b` почтовый адрес, `^d` телеком-адрес, **`^s` сигла (идентификатор для импорта)**.
- Индекс держателя: `SK.FST` стр. 345 — `902 0 (|X=|v902^a/)` → префикс **`X=`** по наименованию (наследуется из IBIS, не SK-специфичен). В `SK.ini` `[SEARCH]` (23 элемента) держатель **НЕ** вынесен отдельным поисковым префиксом.
- Приоритетная сигла берётся из настройки оператора `SKSG` (`SETPRIV.WSS` стр. 312–319, меню `317.mnu`, подсказка «Если не введено, берётся из входной записи») либо из `902^s` входной записи.

### 2.2 Импорт и дедупликация (`impsvk*`)
Режимы импорта (`IMPORTW.MNU`):
| Режим | Назначение |
|---|---|
| `IMPSVK` | Импорт СИГЛЫ в сводный каталог (формат ИРБИС) |
| `IMPSVK_910` | Импорт НОВЫХ экземпляров в сводный каталог |
| `Impsvk_Sigla` | Импорт в сводный каталог с приоритетом сигл |

**Ключ дедупликации** (`impsvk.fst`/`impsvk_910.fst`, блок поля 907) — три яруса сверки входной записи со словарём БД (`J,…`):
1. **по шифру** — `I=` (v903) → маркер `^ZI=`;
2. **по ISBN** — `B=` (v10^a) → маркер `^ZB=`;
3. **по свёртке БО** — композит библиоданных, своя для типа записи (920): для книг `!v110^b,v11^a,v210^d,v700^a,v710^a,v200^a,v215^a…`; отдельные варианты для 981/463/spec.
Результат сверки пишется в **907^Z** (`^ZI=…`/`^ZB=…`/`!…`); непустой `^z` = «дубль найден», пустой = «новая запись».

**Источник записи (какой участник):** сигла-поставщик `g7` = `&uf('IPRIVATE,SKSG,')` или `&uf('Av902^s#1')` (т.е. **902^s**) → пишется в **907^B** (`impsvk_Sigla.pft` стр. 1,8).

**Приоритетная сигла (ПС):** `impsvk_Sigla.fst` помечает запись `1/0` (врем. поле 920) если `902^s` входит в `SiglP.mnu`. При добавлении полей приоритетной сиглой к существующей записи каждое добавленное поле получает подполе **`^_<сигла>`** (тег источника). Поля-индексы (606,607,929,610,621,675,964,965,605,995,996) добавляются при `SVKADDIND` (умолч. 1); поля-экземпляры (141,316,317,318,391,395–399,910) — при `SVKADDEX` (умолч. 0). `impsvk_Sigla.pft` стр. 17–39.

**Экземпляры (910):** при дубле импортируются только экземпляры, чьи инв.№ (910^b) или штрих-код (910^h) ещё не присутствуют в записи БД (`impsvk_910.fst`).

**Слияние:** `impsvk_Sigla.gbl` — глобальная корректировка: логирует «Корректируется запись … найдена по ISBN/шифру/свёртке … Поставщик (сигла) — v907^b» и DEL-then-ADD накладывает поля импортируемой записи на найденную в БД; 902 добавляется независимо от приоритета.

Серверные параметры (namespace PRIVATE/MAIN): `SKSG` (сигла импорта), `SVKADDIND` (умолч. 1), `SVKADDEX` (умолч. 0).

## 3. Sk_ekz / Sk_new / Sk_user — вспомогательные БД сводного каталога

Все три — **«лёгкие» спутники SK** (отдельные БД, поле 1 = login/идентификатор пользователя; ссылка на издание в SK через шифр `2` и форматор `&uf('DSK,!I=',v2,'!,@brief')`). Это не библиографические каталоги, а реестры экземпляров/заявок/пользователей школьного/учебного контура поверх SK.

### 3.1 Sk_ekz — экземпляры пользователя (личный фонд)
- **Поля:** `1` login пользователя, `2` шифр издания в БД SK, **910** экземпляры (`910.wss`), `907` технология (`^A` дата, `^B` кол-во экз.). Источник: `SK_EKZ.ws`.
- **FST (`Sk_ekz.fst`):** `I=` (v1/v2 связка login/шифр), `RI=` (v1, пользователи), `II=` (v2, шифры), `T=` (заглавие через `&uf('DSK,!I=',v2,…@brief')`), `DISC=`/`SEM=`/`KO=` (книгообеспеченность из 691 записи SK), `IN=` (910^B/910^H). Источник: `Sk_ekz.fst`.
- **Поиск (`SK_EKZ.INI`):** 8 элементов — `I=` Пользователь/Шифр, `RI=` Пользователи, `II=` Шифры, `IN=` Инв.№/Штрих-код, `T=` Издания, `DISC=` Уч.предмет, `SEM=` Параллель, `KO=` Уч.предмет/Параллель.
- **Форматы (`PFTW.MNU`):** `SK_EKZ` (основной — HTML-таблица экземпляров: п/п, инв.№/№ уч.к., дата поступл., цена, штрих-код, кол-во, № КСУ), `BRIEF`, `SK_fullw`, `ALL`, **`SK_EKZ_TXT`** (выгрузка личного каталога пользователя в TXT). Школьно-КО-форматы `KoSchool*`, `SK_KoDisc*`, `SK_ListInv*`. Источник: `Sk_ekz.pft`, `PFTW.MNU`.

### 3.2 Sk_new — реестр новых поступлений / заявок
- **Поля:** `1` login, `2` заглавие, `3` год издания, `4` ISBN, `907` технология (`^A` дата, `^B` login). Источник: `SK_NEW.ws` (поля помечены как «1: Первый автор / 2: Заглавие / 3: Год / 4: ISBN» — данные о заявленном издании).
- **FST (`Sk_new.fst`):** пустой (0 байт) — БД-список без инверсии.
- **Контроль (`dbnflc.pft`):** `if s(v1,v2,v3,v4)='' then '1Нет сведений для ЗАЯВКИ'` — ФЛК на непустоту заявки.
- **Форматы:** `SK_NEW` (основной), `BRIEF`, `ALL`. Источник: `Sk_new.pft`, `PFTW.MNU`.

### 3.3 Sk_user — справочник пользователей (организаций-участников школьного контура)
- **Поля:** `1` login, `2` название пользователя, `3` адрес, **4** тип образовательного учреждения (`school.mnu`), **10** контингент (`contingent.wss`; `^F` параллель, `^V` уровень, `^A` кол-во учащихся), `907` технология. Источник: `sk_user.ws`.
- **FST (`Sk_user.fst`):** `I=` (v1 login), `T=` (v2 название).
- **`school.mnu`** — типология ОУ: дошкольные, нач./осн./средняя школа, профильная, гимназия/лицей, коррекционные, интернаты/детдома, доп. образование, СПО/ВПО педагогические, ИПК. (Школьный/региональный сводный контур.)
- **Форматы:** `Sk_user.pft` (HTML: пользователь, название, адрес, тип ОУ + расшифровка `school.mnu`, таблица контингента), `vselect.pft` (HTML-`<select>` для выбора). Источник: `Sk_user.pft`, `vselect.pft`.
- **Поиск (`SK_USER.INI`):** `I=` Пользователь, `T=` Название.

## 4. VKR — выпускные квалификационные работы

**Назначение.** Каталог ВКР (бакалаврских/магистерских/дипломных работ) с полными текстами, ВУЗ-метаданными, антиплагиатом (% оригинальности) и разграничением доступа к полному тексту. Тонкая надстройка над IBIS (в папке лежат и общие `IBIS.WS`/`IBIS.FST`).

### 4.1 Поле 920 и рабочий лист
- **920 = `VKR`** (`VKR.WS` стр. 552–560, меню `920.mnu`, значение по умолчанию `VKR`). При этом `VKR` **отсутствует** в самом `920.mnu` (там PAZK…BOUNI) — жёстко задан как default РЛ. Следствие: `PFTW.OPT` не имеет строки `VKR` → запись показывается дефолтным **`KN.pft`** (отдельного формата показа ВКР нет).

### 4.2 Специфические поля ВКР (`VKR.WS`)
| Поле | Подпись в VKR.WS | Роль | .wss/.mnu |
|---|---|---|---|

### 4.3 Полные тексты (слот 11 = `…\VKR\TEXTS\`)
- **955** (`955.wss`): `^A` имя файла PDF (выбор файла, маска `*.PDF`, ред.16), `^T` текст ссылки, **`^N` кол-во страниц** (для лимита доступа), **`^B` идентификатор записи права доступа** → join к БД **RIGHT** (`RIGHT,I=,*`).
- `VKR.FST` стр. 2–4: 955 → `V=FT` + `TXT=v955^a` (фильтр «есть полный текст»); показ гейтится `&uf('IFULLTEXT,FULLTEXTACTUAL,1')`.
- **951** (внешний объект): FST `V=EXT`,`TEK=`,`TEKV=`,`NRSU=`; показ «Режим доступа: v951^i (v951^t)».
- **TEXTS-подбаза** (`TEXT.fst`): `952^b`→`TXT=`, `952^B` (PDF `__0001.`)→`TXT1=`, `952^c`→`TXT2=`, `999`→`GUID=`. Физические страницы/файлы текста внутри `DATAI\VKR\TEXTS\`, ключ — GUID. Поле 953 в VKR не используется.

### 4.4 Антиплагиат и разграничение доступа
**Антиплагиат:** отдельного поля «плагиат» нет — моделируется как **215^W «% оригинальности текста»** (`215.WSS` стр. 22–23, дубль в `215D.WSS`). Литералов `antiplagiat`/`ЭБС` в конфигах нет.

**Разграничение доступа** (подсистема RIGHT/RDR/LICH):
- join `955^B` → запись БД **RIGHT** (`right_ft.pft` стр. 3: `&uf('DRIGHT,!I=',v955^B,'!,@RIGHT0')`).
- `right_ft.pft` — главный вычислитель права: читает окно дат (RIGHT `^d`/`^e`) и повторения правил v3; тип правила `g3^a`: `01`=RID читатель, `02`=RKAT категория, `05`=RFAK факультет, `06`=RSEM семестр, `07`=RSPC специальность (сверка с записью читателя в БД **RDR**), `03`=IP-адрес (модель-поле 1100), `04`=домен (1110). Каждое — в окне дат.
- Уровень доступа: **`0` запрет / `1` только просмотр / `2` скачивание**, опц. с лимитом страниц (`#NN`, `%`=процент страниц).
- `right2_ft.pft`/`_G`: для уровня 2 считает разрешённые страницы = `955^N * NN/100` минус уже скачанные читателем (трекинг в БД **LICH**, индекс `I=v1002/v955^A`, поле v4) → `2#MM` (остаток) или `1#NN`.
- `right3_ft_G.pft` — максимум права по повторениям; `right4_ft_G.pft` — минимум.
- Модель-поля: `1002` ID читателя, `1100` IP клиента, `1110` домен клиента.
