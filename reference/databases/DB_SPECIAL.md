# ЧАСТЬ I. ГУМАНИТАРНЫЙ АРХИВ (GUAR + GURF + GuSk)

Трёхбазовая подсистема. **GUAR** — основная иерархическая БД (5 уровней). **GURF** и **GuSk** — два плоских рубрикатора-справочника, на которые ссылается поле 615 записей GUAR.

## I-1. GUAR — назначение и PAR

5-уровневая иерархия архивного описания: **Фонд → Опись → Дело → Документ → Лист**. Каждый уровень — отдельная запись со своим рабочим листом и своим значением поля 920. PAR: `Datai\GUAR.par`, все 11 путей → `.\DATAI\GUAR\` (единый каталог). Идентификатор записи — **поле 903** «Шифр документа в БД», индекс `I=` (`Guar.fst:19`), формируется автоматически из `488^T` (шифр родителя) + `852` (`@!903gua`).

Иерархия связывается **полем 488 (вверх)** + **полями 330/903/852 (вниз)**, материализуется GBL-заданиями операторского режима (ОР), которые читают родителя и порождают дочерние записи.

| GBL | назначение |
|---|---|
| `autoin.gbl` | на сохранении: нормализует 327 (паддинг `^L`, уровень `^C` 1–8, `^K`/`^R`), копирует 327→545, **резолвит 615 живьём против GURF/GuSk** (`&uf('DGURF,…')`/`&uf('DGUSK,…')` → `615^A/^F`), строит 903 из `488^t`+`852` |
| `OR_gua.gbl` (497 стр.) | главный генератор иерархии: по выбранным строкам 392 (листы) / 330 (дела) `NEWMFN '*'` создаёт дочернюю запись и заполняет 920 (`Kor_999_920.mnu`), **488** (`^1`+имя связи `Kor_920_488.mnu` + `^T`=903 родителя + `^C/^J/^G`), 852 (`^J`=номер ребёнка, `^G`=группа родителя), 903 (903 родителя + номер), 251/545 |

**Точный механизм связи:**

Одна БД, два типа объектов: **План** и **Мероприятие**, связь полем **488**. PAR `Datai\event.par`: пути 1–10 → `.\datai\event\`, путь 11 → `.\DATAI\event\texts\` (полнотекст). ## II-5. Объектная модель и связь через 488

**Связь = поле 488 с признаком `^1=2`** («План мероприятий на …»), несущим: `488^W` = **903 родительского ПЛАНА**, `488^A` = название плана, `488^9` = имя БД. Down-указатель на стороне Мероприятия — **`972^0`** (№ пункта внутри плана).

**FST строит обратные индексы** (`EVENT.fst:36`): `II=<шифр_плана>` (каждый ребёнок регистрируется под шифром плана → `&uf('J,II='v903)` = «Всего ссылок» в `Plan.pft`); `III=<шифр_плана>/<№_пункта>` (дети, сгруппированные по пункту плана).

**Навигация:**
- **Вниз (План → Мероприятия):** `Plan.pft` рендерит таблицу пунктов 330; строка со связанными мероприятиями → `IRBIS:2,,,,III=<план>/<пункт>` (поиск в EVENT по `III=`).
- **Вверх (Мероприятие → План):** `EVENT.pft` для `488^1:'2'` → `IRBIS:1,,,,I=<488^W>` (переход к родителю-Плану по `I=`).

**Глобалки связывания:** `mer_VvPlan.gbl` («Выбрать ПЛАН» — двусторонне: ребёнку 488+972^0, родителю CORREC+330); `OR_mer_Plan.gbl` (массовое **порождение Мероприятий из Плана**: `NEWMFN` на каждый пункт 330, заполнение 972/900/903/30/210/211 + обратная запись `488^W`); `OR_mer_ToPlan.gbl` (запись пункта в План от мероприятия). ФЛК `!488ev.pft`: каждое 488 должно иметь `^1`; **только одна** связь `^1:'2'` (один родитель-План).

## II-6. Регистрация посетителей (90/40/41)

Рабочий лист **`Reg.ws`** (вкладка «Регистрация посещений») экспонирует ровно 90/40/41. Все три индексируются в **`R=`** (FST 37/38/39). Три вида посетителя:
- **90 — зарегистрированный читатель** (`90mer.wss`): `^H` id (валидация `RDR,RI=`), `^A` дата (авто), `^D` итог за сутки (авто). Запись через `OR_mer_ChRdr.gbl`.
- **40 — организованная группа** (`40mer.wss`): `^A` дата, `^B` откуда, `^C` число. `OR_mer_ChGr.gbl`.
- **41 — гость/неорганизованная группа** (`41mer.wss`): `^A` дата, `^C` число, `^D` итог за дату. `OR_mer_ChGr.gbl`.

Посетитель регистрируется ОР-заданием: оператор сканирует id читателя (→90) или вводит группу/гостей (→40/41); GBL ставит сегодняшнюю дату (`&uf('3')`), инкрементирует итог дня; FST переиздаёт `R=`-счётчики.

**Ночная актуализация словарей по дате:** дата-зависимые словари строятся **прямо в `EVENT.fst`** (строки 1–7, 43–44), относительно текущей даты через глобальный регистр **`g998`** (и `&uf('3')`=сегодня). Драйверы дат: **30^A/^B** (даты ГГГГММДД), гейт по **900^C** (характер: P постоянное, R однодневное, D…) и **997** (статус Z). `210/211` — это **места, а не даты**. `autoin.gbl`/`VstZ.gbl` дополнительно пишут устойчивый флаг `997^AZ`. `TODO(recon #SPEC-01)`: сам планировщик/.bat ночного реиндекса вне каталога `cfg_txt\Event\`.

# ЧАСТЬ IV. PAY — ПЛАТНЫЕ УСЛУГИ

# ЧАСТЬ V. IRI — ИЗБИРАТЕЛЬНОЕ РАСПРОСТРАНЕНИЕ ИНФОРМАЦИИ

## V-1. Назначение и PAR

Журнал поисков/рассылок ИРИ: одна запись = «постоянный запрос профиля прогнан по целевой БД на дату, получено N документов, отв.лицо, опц. файл результата».
